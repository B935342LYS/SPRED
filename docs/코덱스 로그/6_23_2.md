이해한 현재 상태는 다음과 같습니다.

**현재 작업 내용**

`회귀 코드`는 `regression-code/`를 중심으로 구현 중인 TypeScript + HTML + CSS 웹 앱이고, legacy `이세계 코드`는 참고용입니다. 현재 기준 문서는 `docs/HARNESS.md`와 active spec들이며, YouTube 관련 최신 기준은 `docs/2.7-youtube-sync-ui-spec.md`입니다.

현재 구현은 이미 꽤 진행되어 있습니다.

- parser/analyzer/renderer/audio/playback/edit/layout/track layer의 1차 구현이 연결됨
- Web Audio 기반 자체 재생, seek/pause/resume/stop, score scroll playback reference가 구현됨
- track layer 토글, active track 기반 edit/playback, inactive alpha render가 구현됨
- layout editor, layout preset local/file save/load가 구현됨
- Details metadata 편집, JSON/local score save/load, Clear All/Expand/Trim 등이 연결됨
- YouTube sync 1차 구현 완료 상태
  - `musicData.youtube.videoId`, `offsetMs` 사용
  - 오른쪽 YouTube 패널에서 video/offset 입력
  - `Reload`로 metadata 갱신
  - IFrame API lazy-load
  - playback play/pause/stop/seek가 YouTube player를 follower로 제어
  - URL/offset helper 테스트 존재
- 기본 템플릿 악보는 `regression-code/src/assets/templates/default-score.json`로 이동됨
- 현재 검증 명령도 정리되어 있고 `test:youtube`가 package script에 포함되어 있음

**YouTube 모드의 남은 성격**

문서상 YouTube 모드는 “완성 직전 또는 1차 구현 후 수동 검증 필요” 상태로 보입니다. 코드 구조는 들어가 있지만, 실제 브라우저에서 다음 케이스를 계속 확인해야 합니다.

- embeddable 영상 / embedding 차단 영상
- 빈 video 입력
- offset 조정 후 Reload
- play/pause/resume/stop/seek 동기화
- score load 후 YouTube 입력 동기화
- GitHub Pages 배포 환경에서 iframe/API 동작

즉, 지금은 새 구조를 크게 바꾸기보다 브라우저 검증과 발견된 동기화/UX 문제를 좁게 보완하는 단계로 이해했습니다.

**앞으로 예정된 작업**

가장 가까운 다음 큰 작업은 partial rebuild 쪽입니다. 배경은 현재 edit mutation 경로가 여전히 full rebuild라는 점입니다.

현재 편집 경로는 대략:

`ScoreFile mutation -> createRuntimeDocument() -> buildParsedDocument() -> analyzeDocument() -> canvas item rebuild -> renderCanvasScore()`

이 방식은 작은 악보에서는 단순하고 안전하지만, 테스트용 악보를 길게 만들거나 많은 셀을 빠르게 편집할 때 지연이 커집니다. HARNESS의 성능 로드맵도 이 문제를 명시하고 있습니다.

예정 작업은 단계적으로 나누는 게 맞아 보입니다.

1. **YouTube 모드 마무리 검증**
   - 현재 구현이 spec `2.7`과 어긋나는 부분이 있는지 확인
   - 필요하면 YouTube reload/reset/playback binding의 작은 버그 수정
   - `npm run test:youtube`, `npm run build` 및 브라우저 수동 검증

2. **partial rebuild 전 현 병목 확인**
   - 편집 적용 함수와 rebuild 함수 위치 확인
   - 현재 full rebuild가 어떤 artifact를 매번 다시 만드는지 정리
   - partial 적용 범위를 “테스트 악보 제작을 빠르게 하는 데 필요한 최소 범위”로 제한

3. **partial parse/rebuild 1차 추가**
   - 변경된 note/global cell 범위를 기반으로 parsed document 일부만 갱신
   - 기존 `ParsedScoreDocument`를 source of truth로 유지한다는 `1.8` 원칙 준수
   - 처음부터 모든 analyzer 최적화를 욕심내기보다, 안전한 dirty range 단위부터 시작

4. **partial analyze 또는 bounded render로 확장**
   - analyzer는 hold/gliss/tuplet/timing 영향 때문에 변경 범위 확장이 필요함
   - 그래서 “셀 하나만 재분석”이 아니라 변경 column 주변 또는 track/global kind별 dirty range로 설계해야 함
   - 긴 악보 성능까지 보려면 partial rebuild와 별개로 viewport/tile rendering도 필요

5. **테스트용 악보 제작 UX 개선**
   - 긴 score column 생성, 반복 패턴 입력, batch edit 같은 제작 편의 기능과 연결 가능
   - 다만 이것은 partial rebuild 기반이 어느 정도 안정된 뒤 붙이는 편이 안전함

요약하면, 지금은 “YouTube sync 1차 구현을 실제 사용 기준으로 닫고”, 다음으로 “테스트용 긴 악보를 빠르게 만들 수 있도록 full rebuild 병목을 줄이는 partial rebuild/dirty range 작업”으로 넘어가는 흐름입니다.

> 우선은 유튜브 모드는 제대로 기능하는 것 같습니다. 영상 정상 인식되고, 120BPM 기준으로 최소 30 초범위에서는 지연없이 템포 유지되는 것 확인했습니다.
>
> 2번 병목 확인에 들어갑시다. 우선 전역 시각 효과는 노트 셀 수정시에는 다시 그릴 필요가 없습니다. 반대로 전역 셀 수정 시에도 노트를 다시 그릴 필요가 없습니다. base layer는 두 경우 모두 다시 그릴 필요 없어 보입니다.

확인한 병목 구조는 말씀하신 방향과 맞습니다.

현재 실제 경로는 `applyRawTextBatchEditToState()`에서 note/global 구분 없이 항상:

`ScoreFile clone -> createRuntimeDocument -> buildParsedDocument -> analyzeDocument -> buildCanvas*Items -> renderCanvasScore`

까지 갑니다. 그리고 [renderCanvasScore](/d:/오타마톤악보생성기/regression-code/src/renderer/canvas_score_renderer.ts:34)는 매번 `layout`, `base`, `marker`, `note` canvas를 전부 resize/redraw합니다.

따라서 1차 분리 기준은 이렇게 잡는 게 맞습니다.

- **base/layout layer**: note 셀 수정, global 셀 수정 모두에서 재그릴 필요 없음. column count, layout, row order, zoom, DPR 변화에서만 갱신.
- **note 계열 layer**: note 셀 수정 때만 갱신. note rectangle, mute text, gliss, orphan anchor, tuplet container 포함.
- **global 계열 layer**: global 셀 수정 때만 갱신. beat/bar line, BPM change marker, dynamics guide, global rawText 표시 포함.
- **playback/audio schedule**: note 셀 수정과 global 셀 수정 모두 영향 가능. note는 이벤트 목록이 바뀌고, global은 BPM/dynamics/timing mapping이 바뀌므로 별도 갱신 필요.

주의할 점 하나가 있습니다. 현재 구현에서는 `markerItems` 안에 전역 marker와 note-derived marker가 섞여 있고, `note` canvas에도 `globalTextItems`와 tuplet overlay가 같이 그려집니다. 그래서 단순히 “note만 다시 그리기”를 하려면 먼저 renderer item을 다음처럼 쪼개야 합니다.

- `globalMarkerItems`: beat/bar, bpmChange, dynamicsGuide
- `noteMarkerItems`: gliss, glissOrphanAnchor, tupletContainer
- `globalTextItems`: 가능하면 note layer와 분리하거나, 당장은 note redraw 때 같이 다시 그리는 임시 처리

가장 안전한 다음 작업은 full partial analyzer로 바로 가지 않고, 먼저 **render layer partial redraw**를 만드는 것입니다. 즉 parse/analyze는 일단 기존 full rebuild를 유지하되, 편집 종류에 따라 불필요한 canvas layer redraw를 생략합니다. 그 다음 단계에서 note edit은 global parse/analyze 재사용, global edit은 track note analysis 재사용으로 줄이면 됩니다.

정리 내용을 [2.8 명세](/d:/오타마톤악보생성기/docs/2.8-edit-invalidation-and-partial-rebuild-spec.md)에 반영했습니다.

추가한 핵심은 다음입니다.

- note 사각형: `NoteEvent`의 시간, 행, pitch/display, shape 변경 시 갱신
- display text: `displayTextAnchors` 또는 anchor text 변경 시 갱신
- tuplet container: `TupletGroupEvent`, `TupletExtendGroupEvent` 변경 시에만 갱신
- gliss 연결선: `GlissEvent` 변경 시 갱신
- orphan gliss marker: 연결 실패/해소 상태 변경 시 갱신
- tremolo/vibrato: `NoteEvent.effects`의 trem/vib 구간 변경 시 갱신
- mute text: `MuteEvent` 변경 시 갱신
- fallback: 그룹 판정이 불명확하면 해당 note event 범위의 note-derived visual을 넓게 갱신

추가 고려사항도 명세에 넣었습니다. 특히 hold 연결 변화, pitch modifier 변화, tuplet slot 변화, gliss anchor 좌우 segment 영향, global timing/dynamics가 audio schedule에 미치는 영향, 그리고 논리 invalidation group과 실제 canvas dirty rect가 반드시 1:1은 아니라는 점을 적었습니다.

> 추가 고려사항은 타당합니다. 추가적으로 로직을 효율적으로 개선 가능한 부분이나, 아직 모호한 지점이 있으면 짚고 넘어가고 싶습니다. 참고로 2차 구현 도입시 예상되는 증가 효율은 어느정도가 될까요? 노트와 마커 레이어 전체 아이템을 n개라고 하면, O(n)에서 O(k) 정도로 줄어드는 걸까요?

네, 이상적인 2차 구현의 목표는 대체로 **O(n)에서 O(k + 주변 영향 범위)**로 줄이는 것입니다.

여기서:

- `n`: 전체 note/marker render item 수
- `k`: 이번 편집으로 실제 변경된 event/item 수
- 주변 영향 범위: hold, gliss, tuplet처럼 인접 이벤트를 같이 확인해야 하는 범위

예상 효율은 편집 유형별로 다릅니다.

| 편집 유형 | 현재 1차 | 2차 목표 |
| --- | --- | --- |
| 일반 note text 1개 수정 | 전체 note/marker item 재생성/재draw에 가까움 | 해당 `NoteEvent`의 rectangle/text만 갱신 |
| `@p`, `@m` 수정 | 전체 note group | 해당 note rectangle/audio event |
| `~`, trem 수정 | 전체 note group | 해당 note effect segment |
| gliss anchor 수정 | 전체 note marker group | 관련 gliss segment 1-2개 + orphan marker |
| tuplet 수정 | 전체 note marker group | 해당 tuplet group + slot events + 양옆 연결 |
| global dynamics 수정 | note group 불필요 | dynamics guide + audio automation |
| BPM/박자 수정 | note visual 불필요 | global marker + audio time map |

그래서 일반적인 “한 셀 note 수정”은 긴 악보에서 item이 5000개라면, 이론상 5000개 순회/재생성 대신 1개에서 수십 개 수준으로 줄어들 수 있습니다. gliss/tuplet도 보통은 전체가 아니라 연결된 segment/group만 보면 됩니다.

다만 실제 성능은 세 단계로 나뉩니다.

1. **분석 비용**
   - 바뀐 이벤트를 찾으려면 partial analyze가 필요합니다.
   - 처음에는 `trackId + col range`로 주변을 재분석하고, 그 결과를 diff해야 합니다.
   - 이 부분은 완전 O(k)는 아니고 `O(r)`입니다. `r`은 재분석 범위 안의 셀/이벤트 수입니다.

2. **item 생성 비용**
   - event diff가 있으면 전체 item builder를 돌리지 않고 변경 event만 item으로 바꿀 수 있습니다.
   - 이 단계는 `O(n)`에서 `O(k)`로 줄일 수 있습니다.

3. **canvas redraw 비용**
   - canvas는 기존 그림에서 특정 요소만 “삭제”하기 어렵습니다.
   - 보통 dirty rect를 지우고 그 영역에 겹치는 item을 다시 그려야 합니다.
   - 따라서 실제 redraw는 `O(k)`라기보다 `O(items overlapping dirty rect)`입니다.
   - 그래도 전체 canvas redraw보다는 훨씬 작아질 가능성이 큽니다.

남은 모호점/결정 지점은 몇 가지 있습니다.

1. **event identity 기준**
   - diff를 하려면 `eventId`가 안정적이어야 합니다.
   - 현재 event id가 source cell 기반으로 충분히 안정적인지 확인해야 합니다.
   - hold 병합처럼 이벤트가 합쳐지거나 갈라지는 경우에는 old/new id 매칭이 애매할 수 있어, sourceCells overlap 기반 fallback이 필요합니다.

2. **dirty rect 기준**
   - tick 범위만 dirty로 잡을지, row 범위까지 포함할지 정해야 합니다.
   - gliss는 두 row 사이 대각선이라 y 범위가 넓어질 수 있습니다.
   - `@m` microPitch도 인접 row 방향으로 튀어나올 수 있습니다.

3. **canvas layer 구조**
   - note rectangle, text, trem/vib, tuplet overlay가 같은 note canvas에 있으면 개별 갱신이 어렵습니다.
   - 진짜 O(k)에 가깝게 하려면 layer를 더 쪼개거나 dirty rect 재합성 로직이 필요합니다.
   - 현실적인 2차는 “event group은 O(k), canvas redraw는 dirty rect 기반” 정도가 적절합니다.

4. **audio schedule**
   - note pitch/text/effect 수정은 event 단위 schedule 교체가 가능합니다.
   - 하지만 BPM/박자 변경은 이후 모든 seconds mapping을 바꿀 수 있어 넓은 invalidation이 필요합니다.
   - dynamics도 automation 구간 이후를 다시 봐야 할 수 있습니다.

제안하는 2차 구현 순서는 이렇습니다.

1. 안정적인 event key/source range index 만들기
2. partial analyze 결과와 기존 event diff 계산
3. diff를 `noteRectangle`, `displayText`, `glissMarker`, `tupletContainer`, `noteEffect` 등으로 분류
4. renderer item cache를 event key 기준으로 갱신
5. dirty rect 기반 redraw 적용
6. audio schedule은 처음엔 note/global 분리만 유지하고, 그 다음 event-level schedule cache로 확장

즉 효율은 방향상 `O(n) -> O(k)`가 맞지만, 실제 구현에서는 `O(r + d)`에 가깝습니다. `r`은 재분석 범위, `d`는 dirty rect 안에서 다시 그려야 하는 item 수입니다. 일반 단일 note 수정에서는 `r`과 `d`가 매우 작아서 체감상 O(k)에 가까워질 수 있습니다.

> eventId는 대부분의 이벤트 타입에 이미 존재하고 있습니다. 이게 key가 될 수 있을까요?

<details><summary>1 previous message</summary>

> 먼저 현재 analyzer 타입과 eventId 생성 방식을 확인해 보겠습니다. eventId가 “렌더/오디오 캐시 key로 쓸 만큼 안정적인지”는 생성 규칙과 hold/gliss/tuplet에서 병합/분리 시 어떻게 바뀌는지에 달려 있습니다.
</details>

네. **대부분의 이벤트 캐시 key로 `eventId`를 쓸 수 있습니다.** 다만 “단독으로 모든 것을 해결하는 key”라기보다는, 2차 구현의 1차 식별자로 쓰고 몇 가지 fallback을 같이 두는 게 안전합니다.

현재 기준으로는:

- `NoteEvent.eventId`: cache key로 적합
  - 일반 note: `trackId:note:rowId:col`
  - tuplet slot note: `trackId:note:rowId:col:slot:slotIndex`
- `GlissEvent.eventId`: cache key로 적합
  - `trackId:gliss:glissId:startRowId:startCol:slotIndex`
- `TupletGroupEvent`: `eventId`는 없고 `groupId`가 사실상 key
- `TupletExtendGroupEvent`: `groupId` 사용
- `RestEvent`, `MuteEvent`: 현재 `eventId` 없음
  - 필요하면 `eventKind + trackId + sourceCell(rowId,col,slotIndex)`로 파생 key 생성 가능
- `timingTimeline`, `dynamicsTimeline`: event가 아니라 segment라서 별도 segment key 필요

권장 key 형태는 그냥 `eventId`만 쓰기보다 다음처럼 **kind를 포함한 normalized key**입니다.

```ts
type AnalyzerEntityKey =
  | `note:${string}`
  | `gliss:${string}`
  | `tuplet:${string}`
  | `tupletExtend:${string}`
  | `mute:${string}`
  | `rest:${string}`
  | `timing:${string}`
  | `dynamics:${string}`;
```

주의할 부분은 hold/merge입니다.

예를 들어 `C4 - -`가 하나의 `NoteEvent`라면 `eventId`는 시작 셀 기준입니다. 가운데 `-`를 수정하면 같은 `eventId`의 내용이 바뀌므로 update로 처리하면 됩니다. 하지만 시작 셀을 삭제하거나 hold 연결이 끊겨 이벤트가 둘로 갈라지면 기존 `eventId`가 사라지고 새 `eventId`가 생깁니다. 이 경우는 `eventId`만으로 “같은 이벤트의 변경”이라고 보기 어렵고, `sourceCells` overlap 또는 dirty range 기반 fallback이 필요합니다.

따라서 결론은:

- **NoteEvent/GlissEvent의 cache key는 `eventId`로 시작해도 된다.**
- **Tuplet은 `groupId`를 key로 본다.**
- **Mute/Rest/Timeline/Dynamics는 파생 key를 추가해야 한다.**
- **hold split/merge, tuplet 구조 변경, gliss anchor 재연결은 `sourceCells`/dirty range fallback이 필요하다.**

2차 구현의 첫 단계로는 `getAnalyzedEntityKey(event)` 같은 helper를 만들어서 `eventId`, `groupId`, 파생 source key를 한곳에서 정규화하는 게 좋겠습니다.

> eventId를 쓰는게 아예 새로운 event key를 사용하는 것보다는 비용이 적습니까? 아니면 오히려 이벤트 키를 불러오고 fallback이 이루어지는 과정에서 비용이 늘어나는 선택인가요?

대체로 **기존 `eventId`를 쓰는 쪽이 비용이 적습니다.** 새 event key 체계를 따로 만들면 결국 기존 이벤트의 `trackId`, `rowId`, `col`, `slotIndex`, `glissId`, `groupId`, `sourceCells`를 다시 조합해야 해서 계산과 유지보수 비용이 늘어납니다.

다만 정리하면 이렇게 보는 게 맞습니다.

| 선택 | 계산 비용 | 구현 비용 | 안정성 |
| --- | ---: | ---: | --- |
| 기존 `eventId` 사용 | 낮음 | 낮음 | Note/Gliss에는 충분 |
| 새 key를 매번 생성 | 중간 | 중간~높음 | 설계에 따라 좋지만 중복 가능 |
| `eventId` + fallback | 낮음~중간 | 중간 | 가장 실용적 |
| sourceCells 전체 signature | 높음 | 높음 | 안정적이지만 과함 |

`eventId`를 불러오는 비용은 사실상 무시해도 됩니다. 문자열 필드 하나 읽는 정도입니다. fallback 비용도 모든 이벤트에 매번 적용하는 게 아니라, 다음 상황에서만 쓰면 됩니다.

- `eventId`가 없는 이벤트: `MuteEvent`, `RestEvent`, timeline segment 등
- 기존 key가 사라진 경우: hold split/merge, tuplet group 구조 변경
- 같은 dirty range 안에서 old/new 이벤트 매칭이 애매한 경우

즉 일반 note edit 대부분은:

```text
old eventId 조회 -> new eventId 조회 -> 같으면 diff -> 해당 item만 갱신
```

정도로 끝납니다. 이건 `O(1)` map lookup에 가깝습니다.

새 key를 모든 이벤트에 별도로 만들면 오히려:

```text
event 종류 판별
sourceCells 정렬/직렬화
slotIndex 처리
gliss/group/timeline 예외 처리
key 문자열 생성
```

을 매번 해야 합니다. 결과적으로 기존 `eventId`가 있는 이벤트에도 같은 일을 반복하게 됩니다.

제가 권장하는 방식은 이겁니다.

```ts
function getAnalyzerEntityKey(event: AnalyzedEvent): string {
  if ("eventId" in event) {
    return `${event.eventKind}:${event.eventId}`;
  }

  if (event.eventKind === "tupletGroup" || event.eventKind === "tupletExtendGroup") {
    return `${event.eventKind}:${event.groupId}`;
  }

  return `${event.eventKind}:${event.trackId}:${sourceCellKey(event.sourceCells[0])}`;
}
```

핵심은 **새 key 체계를 만드는 게 아니라, 기존 식별자를 우선 쓰는 얇은 normalization layer**를 두는 것입니다.

성능 관점 결론:

- `eventId` 사용 자체는 비용 증가가 아니라 절감입니다.
- fallback은 모든 이벤트에 적용하지 않으면 부담이 작습니다.
- 진짜 비용은 key 생성이 아니라 partial analyze 범위 계산, event diff, dirty rect redraw 쪽에서 발생합니다.
- 따라서 `eventId 우선 + 필요한 이벤트만 fallback`이 가장 균형 잡힌 선택입니다.

> 일부 이벤트 타입에는 eventId가 존재하지 않습니다. 이 김에 만드는 것이 좋은가요?

네. **2차 partial rebuild를 본격적으로 할 거라면, 이 김에 모든 `AnalyzedEvent`에 안정적인 `eventId`를 부여하는 편이 좋습니다.**

이유는 단순합니다. 이벤트별 캐시, diff, dirty rect, audio schedule 교체를 하려면 “이전 이벤트와 새 이벤트가 같은 논리 단위인지”를 빠르게 판단해야 합니다. 일부 이벤트만 `eventId`가 있고 일부는 `groupId`나 `sourceCells` fallback을 쓰면 구현이 계속 갈라집니다.

권장 방향은:

```ts
type AnalyzedEventBase = {
  eventKind: AnalyzedEventKind;
  eventId: string;
  trackId: TrackId;
  time: TimeRange;
  sourceCells: SourceCellRef[];
};
```

그리고 각 이벤트별 id는 기존 규칙과 맞춥니다.

- `NoteEvent`
  - 기존 유지: `note:${trackId}:${rowId}:${col}`
  - tuplet slot: `note:${trackId}:${rowId}:${col}:${slotIndex}`
- `GlissEvent`
  - 기존 유지: `gliss:${trackId}:${glissId}:${startRowId}:${startCol}:${slotIndex}`
- `RestEvent`
  - `rest:${trackId}:${rowId}:${col}:${slotIndex}`
- `MuteEvent`
  - `mute:${trackId}:${rowId}:${col}`
- `TupletGroupEvent`
  - `tupletGroup:${trackId}:${headRowId}:${headCol}`
  - 기존 `groupId`는 유지하되 `eventId`도 추가
- `TupletExtendGroupEvent`
  - `tupletExtendGroup:${trackId}:${firstExtendRowId}:${firstExtendCol}`
  - 기존 `groupId`도 유지
- timeline/dynamics segment는 `AnalyzedEvent`는 아니지만, 추후 캐시가 필요하면 별도 `segmentId`를 고려

다만 한 가지는 조심해야 합니다. `eventId`는 **저장 JSON에 들어가는 영구 ID가 아니라 런타임 파생 ID**로 유지해야 합니다. 즉 score 파일 포맷에는 영향이 없어야 합니다.

구현 순서는 이렇게 가면 안전합니다.

1. `AnalyzedEventBase`에 `eventId: string` 추가
2. `NoteEvent`, `GlissEvent`의 중복 선언 제거 또는 유지 방식 정리
3. `RestEvent`, `MuteEvent`, `TupletGroupEvent`, `TupletExtendGroupEvent` 생성부에 `eventId` 추가
4. 기존 `groupId`는 tuplet 소속 연결용으로 계속 유지
5. 테스트에서 eventId 안정성 확인
   - 같은 score를 두 번 analyze하면 같은 eventId
   - unrelated cell edit 후 기존 eventId 유지
   - tuplet/gliss source가 같으면 eventId 유지

이렇게 해두면 2차 구현에서 `event.eventId`만 기본 key로 쓰고, hold split/merge 같은 경우에만 source range fallback을 보면 됩니다.

> 코어 모듈의 데이터구조 변경이니 연결된 후속 모듈에 미치는 영향도 세심하게 고려해야 할 것 같습니다.

맞습니다. `AnalyzedEventBase.eventId` 추가는 작아 보여도 core analyzer 계약 변경이라 downstream 전체를 봐야 합니다.

영향 범위는 대략 이렇습니다.

| 영역 | 영향 |
| --- | --- |
| analyzer types | `AnalyzedEventBase`에 `eventId` 추가, 개별 이벤트 타입 중복 정리 |
| analyzer 생성부 | `RestEvent`, `MuteEvent`, `TupletGroupEvent`, `TupletExtendGroupEvent`에 id 생성 추가 |
| renderer item builder | 지금은 `eventId`를 거의 안 쓰지만, 2차 캐시에서는 item에 source event key를 붙일 수 있음 |
| audio schedule builder | 이미 note/gliss `eventId`를 사용함. 모든 event에 id가 생기면 타입 좁히기 단순화 가능 |
| tests | analyzer event snapshot/검증이 있다면 eventId 추가 반영 필요 |
| docs | `1.7`, `2.8`에 “모든 AnalyzedEvent는 runtime eventId를 가진다”로 갱신 필요 |

중요한 설계 결정은 두 가지입니다.

1. **`eventId`는 runtime derived ID**
   - score JSON에 저장하지 않음
   - import/export 포맷 변경 없음
   - analyzer가 매번 deterministic하게 생성

2. **기존 의미 ID는 유지**
   - `TupletGroupEvent.groupId`는 slot membership 연결용으로 계속 필요
   - `GlissEvent.glissId`도 사용자/문법상 gliss 연결 id라 계속 필요
   - `eventId`는 캐시/diff/render/audio 추적용

따라서 구현 전에 먼저 명세에 이렇게 못 박는 게 좋습니다.

```ts
type AnalyzedEventBase = {
  eventKind: AnalyzedEventKind;
  eventId: string;
  trackId: TrackId;
  time: TimeRange;
  sourceCells: SourceCellRef[];
};
```

그리고 downstream 원칙:

- renderer/audio/cache는 이벤트 추적 key로 `eventId`를 우선 사용
- tuplet 소속 관계는 `groupId` 사용
- gliss 문법 연결은 `glissId` 사용
- event split/merge는 `eventId`가 바뀔 수 있으므로 `sourceCells` dirty range fallback 허용

바로 구현하기보다는 다음 순서가 안전합니다.

1. `1.7`과 `2.8` 명세 업데이트
2. analyzer type 변경
3. event 생성부 전체 수정
4. typecheck로 downstream 깨지는 곳 확인
5. analyzer/audio/render tests 실행
6. 그 다음 event diff/cache 설계로 이동

이렇게 하면 core 계약 변경을 깔끔하게 닫고 2차 partial rebuild로 넘어갈 수 있습니다.

> 부분 갱신은 renderer와 audio에 모두 관여합니다. 별도의 모듈로 분리하는 것이 좋을까요?

네. 2차 partial rebuild는 별도 모듈로 분리하는 편이 좋습니다. 이유는 renderer와 audio 양쪽에 걸치지만, 둘 중 어느 한쪽의 책임은 아니기 때문입니다.

추천 구조는 이런 방향입니다.

```text
edit apply
  -> partial rebuild planner
      -> analyzer event diff
      -> renderer invalidation plan
      -> audio invalidation plan
  -> renderer apply
  -> audio schedule apply
```

예상 모듈 위치는 `src/app/partial_rebuild/` 또는 `src/app/rebuild/`가 적절합니다. `core/analyze` 아래에 두기에는 renderer/audio 결정을 알아야 하고, `renderer` 아래에 두기에는 audio까지 침범합니다. 앱 orchestration 계층에 두는 게 맞습니다.

초기 타입은 대략 이렇게 잡을 수 있습니다.

```ts
type PartialRebuildPlan = {
  editKind: "noteCell" | "globalCell" | "mixedCell" | "structure";
  changedEventIds: string[];
  renderer: RendererInvalidationPlan;
  audio: AudioInvalidationPlan;
  fallback: "none" | "fullRender" | "fullRuntime";
};
```

분리 기준은 다음이 좋겠습니다.

- `core/analyze/event_diff.ts`
  - 순수 diff만 담당
  - renderer/audio를 모름

- `app/partial_rebuild/partial_rebuild_plan.ts`
  - edit 종류, changed cols, event diff를 받아 갱신 계획 생성
  - renderer/audio 양쪽 invalidation을 동시에 산출

- `renderer/*`
  - plan을 받아 item 교체 또는 layer redraw 수행
  - audio를 모름

- `audio/*`
  - plan을 받아 schedule 전체/부분 교체 수행
  - renderer를 모름

- `app/main.ts` 또는 `app_ui_sync.ts`
  - 실제 순서 조율만 담당

중요한 점은 “부분 갱신 모듈이 직접 그리거나 직접 스케줄하지 않는 것”입니다. 이 모듈은 판단만 하고, 실행은 renderer/audio 각 모듈에 맡기는 구조가 유지보수에 유리합니다.

제안하는 다음 단계는 `app/partial_rebuild`에 plan 타입과 planner skeleton을 만들고, 현재는 아직 부분 교체 대신 기존 full render/full audio rebuild로 fallback하되, `eventDiff`와 invalidation group만 계산해 테스트로 고정하는 것입니다. 그러면 실제 최적화 구현을 붙일 때 위험이 작습니다.

- global edit은 note event 구조를 건드리지 않으므로, partial 적용을 더 안전하게 좁힐 수 있습니다.
- 현재 사용자가 확인한 병목도 “노트 셀 수정 시 전역 시각 효과 불필요, 전역 셀 수정 시 노트 불필요”였기 때문에 global 쪽을 먼저 완전히 분리하면 체감 개선과 구조 안정성이 좋습니다.

1. **global render item partial**

   - `globalTextItems`, `globalMarkerItems`, `markerItems`만 next에서 교체

핵심은 이겁니다.

```text

  -> dirty columns / dirty tracks
  -> parser partial range
  -> analyzer impact range 확장
  -> old events 제거
  -> new events 삽입
  -> renderer/audio plan
```

- `-`, `~` hold는 앞뒤 note와 병합/분리될 수 있음
- gliss anchor는 좌우 인접 anchor와 연결됨
- tuplet head/extend/slot은 group 전체와 extend run에 영향
- 같은 col의 다른 row note가 gliss 중복 anchor 처리에 영향
- display row와 sound pitch가 `@n`, `@p`, `@m`에 따라 바뀜

그래서 추천 1차 partial parse/analyze는 이렇게 잡는 게 안전합니다.

1. **편집 track 단위 partial**
   - global parser/analysis는 유지
   - 편집된 track만 parse/analyze
   - 다른 trackResults는 이전 결과 재사용
   - 구현 난이도 낮고 효과 큼

2. **편집 track 안에서 range 확장**
   - edited col 기준으로 좌우 확장
   - 처음에는 보수적으로 “해당 track 전체”도 괜찮습니다.
   - 이후 hold/gliss/tuplet 규칙에 따라 range를 줄임

3. **진짜 column range partial**
   - old track event 중 dirty range와 겹치는 event 제거
   - dirty range를 재분석해 new event 삽입
   - eventId diff로 renderer/audio patch
   - 이 단계가 가장 어렵습니다.

즉 바로 “편집점 중심”으로 갈 수는 있지만, 첫 단계는 **edited track 중심**이 좋습니다. 이 정도만 해도 다중 track 악보에서는 큰 효과가 납니다.

현재 note edit에서도 남아 있는 길이 비례 작업이 있습니다.

1. `renderCanvasScorePartial(..., "note")`가 marker layer 전체를 다시 그림
   [canvas_score_renderer.ts](./regression-code/src/renderer/canvas_score_renderer.ts:83)

   note edit이어도 현재는:

   ```ts
   drawScoreMarkers(target.marker.context, layout, markerItems);
   drawScoreNotes(...);
   drawScoreOverlayMarkers(...);
   ```

   를 수행합니다.

2. `drawScoreMarkers()`는 marker canvas 전체를 clear합니다.

   ```ts
   context.clearRect(0, 0, layout.stageWidth, layout.stageHeight);
   ```

   `stageWidth = columnCount * columnWidth`라서 악보가 길면 노트 수와 무관하게 비용이 커집니다.

   [canvas_note_renderer.ts](./regression-code/src/renderer/canvas_note_renderer.ts:40)
