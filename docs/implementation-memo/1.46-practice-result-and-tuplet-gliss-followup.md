## 2. Timing Judge Update

기존 timing 판정은 `80-249ms`를 모두 작은 `early` / `late` 표시로만 처리했다.
이 범위가 너무 넓어 `80ms`와 `249ms`의 체감 차이가 결과에 반영되지 않는 문제가 있었다.

현재 timing 판정:

| Timing offset | Overlay | Final label | Timing accuracy |
| --- | --- | --- | ---: |
| `<80ms` | none | pitch label 유지 | 100% |
| `80-149ms` | `early` / `late` | pitch label 유지 | 66.7% |
| `150-249ms` | `early` / `late` | `Bad`로 강등 | 33.3% |
| `>=250ms` | none | `Miss`로 강등 | 0% |

pitch accuracy도 같은 단계감으로 맞췄다.

```text
Perfect = 1
Ok      = 2 / 3
Bad     = 1 / 3
Miss    = 0
```

`Bad` timing은 timing 전용 label이 아니라 최종 pitch label을 `Bad`로 낮추는 방식이다.
따라서 기존 `Bad` count와 combo reset 정책을 그대로 사용한다.

## 5. Tuplet Gliss `- + Start` Rendering

긴 tuplet slot gliss는 start / mid anchor note를 정사각형으로 보이게 하는 보정이 있었다.
하지만 `-@g(...,S)`처럼 hold와 gliss start가 결합된 경우에는 해당 slot이 이전 slot note에 병합되므로, 정사각형 item을 덧그려도 뒤에 긴 note rectangle이 남아 정사각형으로 보이지 않았다.

현재 처리:

```text
NoteEvent
  -> tuplet gliss start/mid anchor 후보 수집
  -> note event 전체와 anchor 시간이 같으면 기존처럼 base item을 anchorSquare로 표시
  -> hold 병합으로 anchor가 note event 내부에 있으면
       base rect를 anchor 앞/뒤 범위로 분할
       anchor 구간은 별도 anchorSquare item으로 생성
```

이 처리는 draw 중 반복 조건을 늘리지 않고 `CanvasNoteRenderItem` 생성 단계에서 한 번 수행한다.
따라서 viewport draw나 canvas paint 루프에 별도 복잡한 분기를 추가하지 않는다.

검증 fixture:

```text
/3(C@n(60)|-@g(v,S)@n(60)|E@g(v,E)@n(64))
```

기대 결과:

- 기본 merged note rectangle은 `62..63`까지만 남는다.
- `63..64` 구간에는 별도 `anchorSquare` item이 생성된다.
- gliss marker는 held start slot anchor tick에서 시작한다.

## 8. Edit Mode Lock Follow-up

practice mode 진입 시 `AppState.mode`는 이미 `view`로 전환되고 있었지만, edit toggle DOM의 `checked` 상태가 항상 state에서 재동기화되지는 않았다.
그 결과 edit mode가 열린 상태에서 practice mode를 켜면 내부 상태는 view로 바뀌어도 disabled된 edit toggle이 켜진 것처럼 보일 수 있었다.

현재 처리:

- practice mode open 시 기존 mode가 edit이면 `mode: { kind: "view" }`로 자동 전환한다.
- edit mode였던 상태에서 practice를 열면 status에 edit mode가 practice 동안 잠겼음을 표시한다.
- `syncUiControls()`가 `dom.editToggle.checked`와 `aria-pressed`를 `AppState.mode` 기준으로 강제 동기화한다.
- practice lock 중 edit toggle은 disabled이면서 off 상태로 표시된다.

검증:

```text
npm run test:game
npm run typecheck
npm run build
```

## 9. Gliss Effect Bonus Prototype

modifier 가산 판정은 감점 없는 bonus 방식으로 시작했다.
첫 구현은 gliss만 실제 판정하고, vib/trem은 결과 표시와 summary count가 0으로 남을 수 있게 필드만 준비했다.

현재 처리:

- `GameScoreSummary`에 `glissBonusCount`, `vibBonusCount`, `tremBonusCount`, `effectBonusScore`를 추가했다.
- `GameEffectBonusResult`는 `Gliss!`, `Vib!`, `Trem!` 표시를 분리할 수 있게 한다.
- `game_effect_judge.ts`가 active track의 `GlissEvent`를 practice bonus target으로 바꾼다.
- gliss start~end 구간을 `250ms` interval로 나누고 interval별로 한 번만 판정한다.
- 해당 interval의 현재 `GamePitchFrame`이 voiced pitch이면 `Gliss!` bonus를 지급한다.
- 실패한 interval은 감점하지 않고 아무 결과도 만들지 않는다.
- 성공한 gliss interval은 `trackDifficulty * 0.25`를 `effectBonusScore`와 total score에 더한다.
- live panel/result dialog에는 `Gliss!`, `Vib!`, `Trem!`, result의 `Effect Bonus` 표시를 준비했다.
