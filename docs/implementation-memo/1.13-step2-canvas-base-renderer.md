# 1.13. 2단계 Canvas base renderer 구현 메모
> 본 설계 메모 문서 1.13에서는 2단계 구현 중 canvas base renderer 경로의 역할과 구현 판단을 기록한다.
> 본 문서는 active specification을 대체하지 않는다.
> 함수별 세부 설명은 코드 파일의 `/** ... */` 주석을 중심으로 하고, 본 문서는 흐름과 사용 문법, 설계 판단을 보충한다.

## 1. 개요

이번 구현은 기존 `index.html` 안의 임시 inline canvas draw 코드를 제거하고, 실제 score layout JSON을 기반으로 악보 grid를 그리는 첫 renderer 경로를 만든 것이다.

현재 구현 흐름은 아래와 같다.

```text
sample score JSON
-> loadRuntimeDocument()
-> createCanvasRenderInput()
-> renderCanvasScore()
-> buildCanvasScoreLayout()
-> drawLayoutGrid()
-> drawScoreGrid()
```

아직 analyzer 결과를 소비하는 note rectangle 렌더링은 포함하지 않는다.
이번 단계의 목표는 `RuntimeDocument`의 layout 정보를 renderer 전용 DTO로 바꾸고, layout label canvas와 score base canvas가 같은 row 좌표계를 공유하는지 확인하는 것이다.

## 2. 관련 파일

이번 구현에서 추가되거나 변경된 주요 파일은 다음과 같다.

- `regression-code/src/app/main.ts`
  - 브라우저 DOM 진입점이다.
  - sample score 로드, canvas element 조회, status 표시, scroll sync를 담당한다.
- `regression-code/src/app/canvas_renderer_adapter.ts`
  - core score 구조를 renderer 전용 DTO인 `CanvasRenderInput`으로 변환한다.
- `regression-code/src/renderer/canvas_types.ts`
  - renderer 계층의 입력, 좌표, target 타입을 정의한다.
- `regression-code/src/renderer/canvas_coordinate.ts`
  - `CanvasRenderInput`을 `CanvasScoreLayout`으로 변환하고 canvas bitmap 크기를 DPR에 맞춘다.
- `regression-code/src/renderer/canvas_grid_renderer.ts`
  - layout label 영역과 score base 영역의 grid/background를 그린다.
- `regression-code/src/renderer/canvas_score_renderer.ts`
  - renderer 진입점으로, 좌표 계산과 layer draw 순서를 조립한다.
- `regression-code/vite.config.ts`
  - GitHub Pages 같은 하위 경로 배포에서도 asset 경로가 깨지지 않도록 `base: "./"`를 설정한다.

## 3. 모듈 경계

이번 구현에서 가장 중요한 판단은 renderer가 `RuntimeDocument`를 직접 소비하지 않게 한 점이다.

`RuntimeDocument`는 score 로드 계층의 산출물이다.
반면 canvas renderer는 화면을 그리는 데 필요한 값만 받는 편이 유지보수에 유리하다.
따라서 app/controller 쪽 adapter가 아래 변환을 담당한다.

```text
RuntimeDocument
-> CanvasRenderInput
```

`CanvasRenderInput`에는 다음 값만 들어간다.

- 표시할 row 목록
- 각 row의 종류, label, height
- 전체 column 수
- 기본 column width

이렇게 하면 renderer 내부 파일은 score 저장 구조의 세부 필드 변화에서 보호된다.
나중에 score JSON 구조가 바뀌더라도 adapter만 조정하면 base renderer는 같은 계약을 유지할 수 있다.

## 4. app/main.ts의 책임

`main.ts`는 브라우저 앱 진입점이다.
이 파일은 DOM과 직접 만나는 app/controller 경계에 해당한다.

현재 책임은 다음이다.

- canvas element를 찾고 2D context를 만든다.
- sample score JSON을 로드한다.
- `loadRuntimeDocument()`를 호출한다.
- `createCanvasRenderInput()`으로 renderer 입력을 만든다.
- `renderCanvasScore()`를 호출한다.
- renderer 결과의 stage 크기를 CSS 변수에 반영한다.
- score 영역의 세로 스크롤을 layout label 영역에 동기화한다.

반대로 이 파일은 canvas grid를 직접 그리지 않는다.
grid draw는 renderer 계층의 `canvas_grid_renderer.ts`가 담당한다.

## 5. sample JSON 로드 방식

현재 sample score는 아래 import로 읽는다.

```ts
import sampleScoreJson from "../../dev/test_cases/minimal-valid-score.json?raw";
```

`?raw`는 Vite가 제공하는 기능으로, 파일 내용을 문자열로 import한다.
이 방식을 사용한 이유는 다음과 같다.

- `loadRuntimeDocument()`가 JSON 문자열을 입력으로 받는다.
- GitHub Pages 배포 시 `dev/test_cases` 파일을 별도 public asset으로 복사하지 않아도 된다.
- production build 결과에 sample JSON 문자열이 함께 포함된다.

주의할 점은 이 방식이 Vite를 거쳐야 동작한다는 것이다.
따라서 `index.html`을 파일 탐색기에서 직접 열어 `file://`로 실행하면 동작하지 않는다.
개발 중에는 `npm run dev`, 배포 확인은 `npm run build` 후 `npm run preview`로 확인한다.

## 6. Vite를 도입한 이유

현재 브라우저 앱은 TypeScript source file을 직접 HTML에서 module로 연결한다.
일반 브라우저는 `.ts` 파일과 Vite 전용 import 문법을 그대로 실행하지 못한다.

따라서 Vite가 아래 역할을 맡는다.

- TypeScript browser entry를 JavaScript로 변환한다.
- `?raw` import를 문자열 import로 변환한다.
- 개발 서버에서 module import 경로를 해석한다.
- production build 시 `dist/`에 배포 가능한 asset을 생성한다.

`vite.config.ts`에는 아래 설정을 두었다.

```ts
export default defineConfig({
  base: "./",
});
```

이 설정은 GitHub Pages처럼 프로젝트가 domain root가 아닌 하위 경로에 배포될 때도 script와 CSS 경로가 `./assets/...` 형태로 생성되게 한다.

## 7. canvas_coordinate.ts의 책임

`canvas_coordinate.ts`는 좌표 계산과 canvas bitmap resize만 담당한다.
실제 grid나 label은 그리지 않는다.

현재 주요 함수는 다음이다.

- `buildCanvasScoreLayout()`
  - row y 좌표를 누적 계산한다.
  - column width와 stage width를 계산한다.
  - layout label width와 playback boundary x 좌표를 계산한다.
- `columnToX()`
  - column/tick 값을 score canvas의 x 좌표로 바꾼다.
- `resizeCanvasLayers()`
  - layout/base/note/marker canvas의 CSS 크기와 bitmap 크기를 맞춘다.

canvas 좌표는 항상 CSS pixel 기준으로 계산한다.
`devicePixelRatio`는 canvas bitmap 해상도 보정에만 사용하고, score의 논리 좌표에는 섞지 않는다.
이렇게 해야 note 위치, scroll 위치, hit test 위치가 DPR에 따라 두 배로 밀리는 문제를 피할 수 있다.

## 8. 옵션 정규화

`CanvasRenderOptions`는 UI에서 오는 값이므로 잘못된 값이 들어올 수 있다.
좌표 계산 단계에서는 다음 기본값 정책을 적용한다.

- `zoom <= 0` 또는 유한하지 않은 값이면 `1`
- `devicePixelRatio <= 0` 또는 유한하지 않은 값이면 `1`
- `layoutWidth <= 0` 또는 유한하지 않은 값이면 renderer 기본값
- `columnWidth`가 없거나 유효하지 않으면 `CanvasRenderInput.baseColumnWidthPx`
- padding column 값이 음수이거나 유한하지 않으면 `0`

이 정규화는 renderer가 빈 화면이나 `NaN` 좌표를 만들지 않도록 하는 최소 방어선이다.

## 9. layout 영역과 score 영역의 좌표 분리

`2.0`, `2.1` 명세에 따라 layout label 영역에는 playback 기준을 강조하기 위한 padding 표시가 있을 수 있다.
하지만 이 padding은 실제 score column이 아니다.

따라서 현재 구현은 아래처럼 좌표를 분리한다.

- score canvas
  - 실제 column 0은 x=0에서 시작한다.
  - `stageWidth = columnCount * columnWidth`
- layout label canvas
  - row label과 playback boundary 강조를 그린다.
  - right padding overlay는 layout 영역 안에서만 그린다.

현재 playback boundary 관련 값은 다음처럼 해석한다.

```text
layoutPlaybackBoundaryX = layoutWidth
overlayStartX = layoutWidth - layoutRightPaddingWidth / 2
overlayWidth = layoutRightPaddingWidth / 2
```

즉 반투명 red overlay는 layout 영역의 오른쪽 끝 절반에만 표시되고, score canvas 안쪽에는 그리지 않는다.
후속 playback/scroll controller는 별도의 playback boundary 값을 기준으로 `scrollLeft = columnToX(currentTick) - playbackBoundaryX` 형태를 사용할 수 있다.

## 10. canvas_grid_renderer.ts의 draw 순서

base grid draw 순서는 다음이다.

```text
1. 전체 배경을 진한 회색으로 칠한다.
2. note row만 연한 회색으로 덮는다.
3. column grid를 그린다.
4. row grid를 그린다.
```

global row와 gap row는 같은 진한 배경으로 둔다.
현재 시점에서는 global row를 별도 색상으로 다시 칠하지 않는다.

layout label canvas도 같은 `CanvasScoreLayout.rows`를 사용하므로, score base canvas와 row 경계가 같은 y 좌표에 그려진다.
세로 스크롤은 `main.ts`에서 layout stage에 `translateY(-scoreArea.scrollTop)`을 적용해 동기화한다.

## 11. HTML 직접 실행이 안 되는 이유

현재 `index.html`은 Vite 개발 서버 또는 Vite build 결과를 전제로 한다.
파일 탐색기에서 직접 열면 URL이 `file://.../index.html`이 되며, 이 방식은 현재 앱 실행 조건과 맞지 않는다.

직접 실행이 안 되는 주된 이유는 다음이다.

- 브라우저는 `.ts` module을 직접 해석하지 않는다.
- Vite의 `?raw` import는 dev server 또는 build 과정에서만 변환된다.
- module import 경로 해석과 asset 경로 처리가 HTTP server 기준으로 설계되어 있다.

따라서 개발 확인은 아래 명령을 사용한다.

```text
npm run dev
```

배포 산출물 확인은 아래 명령을 사용한다.

```text
npm run build
npm run preview
```

GitHub Pages 역시 `file://`가 아니라 HTTP로 파일을 제공하므로, Vite build 결과를 기준으로 보는 것이 맞다.

## 12. 이번 구현에서 사용한 주요 TypeScript / browser 문법

### 12.1. type-only import

타입만 필요한 import에는 `import type`을 사용한다.

```ts
import type { CanvasRenderTarget } from "../renderer/canvas_types";
```

이 import는 JavaScript output에 남지 않는다.
타입 참조와 런타임 의존성을 구분하기 위해 사용한다.

### 12.2. generic DOM query helper

`main.ts`의 `queryElement<T>()`는 selector로 찾은 DOM 요소가 기대 타입인지 확인한다.

```ts
function queryElement<T extends HTMLElement>(
  selector: string,
  ctor: { new (): T },
): T
```

이렇게 하면 `HTMLCanvasElement`, `HTMLInputElement`처럼 구체 타입이 필요한 곳에서 타입 단언을 반복하지 않아도 된다.

### 12.3. CanvasRenderingContext2D

canvas에 그리기 위해 `getContext("2d")`로 `CanvasRenderingContext2D`를 얻는다.
이 값은 실패 시 `null`일 수 있으므로, target 생성 단계에서 한 번 확인하고 이후 renderer에는 null이 아닌 context만 넘긴다.

### 12.4. devicePixelRatio와 setTransform

고해상도 화면에서 canvas가 흐릿해지는 것을 막기 위해 bitmap 크기는 CSS 크기에 DPR을 곱해 잡는다.
그 뒤 `setTransform(dpr, 0, 0, dpr, 0, 0)`을 적용하여 draw 함수는 계속 CSS pixel 좌표로 그릴 수 있게 한다.

```ts
target.context.setTransform(
  devicePixelRatio,
  0,
  0,
  devicePixelRatio,
  0,
  0,
);
```

다만 score canvas는 가로로 매우 길어질 수 있다.
sample score만 해도 `columnCount = 1000`, `baseColumnWidthPx = 21`이므로 기본 score width가 21000px가 된다.
여기에 DPR 2를 그대로 곱하면 bitmap width가 42000px가 되어 브라우저 canvas bitmap 한계나 GPU 메모리 압박을 일으킬 수 있다.

따라서 현재 구현에서는 layer별 CSS 크기에 따라 effective DPR을 낮춘다.
이 처리는 선명도보다 안정성을 우선하는 임시 완화책이다.
장기적으로는 전체 stage를 하나의 거대한 canvas에 그리지 않고, viewport 기반 visible range 또는 tile 기반 canvas로 전환해야 한다.

### 12.5. CSS custom properties

renderer가 계산한 stage 크기는 CSS 변수로 반영한다.

```ts
document.documentElement.style.setProperty(
  "--score-stage-width",
  `${stageWidth}px`,
);
```

이 방식으로 TypeScript 좌표 계산 결과와 CSS layout 크기를 맞춘다.

## 14. zoom 처리와 성능 판단

초기 구현에서는 zoom range의 `input` 이벤트마다 `renderCanvasScore()`를 호출했다.
이 방식은 slider를 드래그하는 동안 전체 layout/base canvas를 계속 지우고 다시 그리므로 비용이 크다.

현재는 zoom range의 `change` 이벤트에서만 render를 호출하도록 바꾸었다.
즉 사용자가 drag를 끝내고 값을 확정했을 때 한 번만 zoom을 적용한다.

이 결정은 다음 이유로 현재 단계에 적합하다.

- zoom은 매 프레임 실시간 preview가 없어도 사용성이 크게 떨어지지 않는다.
- note/marker/audio scroll이 붙으면 전체 재렌더 비용이 더 커진다.
- 후속 viewport virtualization이 들어가기 전까지는 전체 stage 재렌더 횟수를 줄이는 것이 가장 효과적이다.

장기적으로는 아래 개선을 검토한다.

- slider drag 중에는 CSS transform으로 가벼운 preview만 제공하고, drag 종료 후 실제 canvas를 다시 그린다.
- zoom 변경 시 base layer와 note/marker layer의 dirty 범위를 분리한다.
- visible range 기반으로 보이는 row/column만 다시 그린다.
- 정적인 base grid는 OffscreenCanvas 또는 bitmap cache로 재사용한다.
