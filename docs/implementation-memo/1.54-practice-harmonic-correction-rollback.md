# 1.54 Practice Harmonic Correction Rollback

## 1. 목적

프랙티스 모드에서 일부 마이크가 저음 연주를 한 옥타브 위 배음으로 인식하는 문제가 다시 확인되었다.

대표 증상:

- 실제 연주: A#3
- detector 표시: A#4
- 추정 원인: 마이크 / 브라우저 입력 경로에서 기본음 성분보다 2배음 성분이 더 선명하게 남음

이 문서는 2026-07-11에 시도한 harmonic verifier / low-pass verifier 방식의 결과와 롤백 판단을 기록한다.

## 2. 시도한 접근

기존 pitch 입력 경로는 `pitchy`의 McLeod Pitch Method 결과를 받아 `GamePitchFrame`으로 변환한다.

시도한 보정은 `pitchy`가 반환한 후보 `f`를 그대로 쓰기 전에 다음 검증을 추가하는 방식이었다.

1. `f / 2` 후보를 계산한다.
2. Goertzel 방식으로 `f / 2`, `f`, `3f / 2`, `2f`, `5f / 2` 주변 에너지를 측정한다.
3. `f / 2`의 실제 성분과 홀수 배음 증거가 충분하면 한 옥타브 아래 후보를 채택한다.
4. 이후 보강 시도에서는 `f` 위쪽을 약화하는 low-pass sample을 만든 뒤 보조 `PitchDetector`가 `f / 2`를 다시 잡는지 확인했다.

의도:

- 단순한 target 기반 octave correction을 피한다.
- hysteresis lock처럼 시간축 상태가 새 부작용을 만들지 않게 한다.
- 현재 frame의 harmonic evidence만으로 2배음 오인식을 줄인다.

## 3. 실제 결과

합성 sine partial 테스트에서는 다음 조건을 통과했다.

- `f / 2`와 홀수 배음 증거가 있는 경우 `f / 2`를 선택한다.
- 약한 subharmonic leakage만 있는 경우 실제 `f`를 유지한다.
- low-pass 후 `f / 2`가 살아나는 합성 신호는 `f / 2`를 선택한다.

그러나 실사용 마이크 테스트에서는 핵심 증상인 A#3 -> A#4 오인식이 해결되지 않았다.

## 4. 롤백 판단

제품 코드에서는 해당 시도를 롤백한다.

이유:

- 실제 문제를 해결하지 못했다.
- 합성 테스트가 실제 마이크 입력 조건을 충분히 대표하지 못했다.
- `game_pitch_input.ts`의 복잡도가 크게 증가했다.
- 실패한 보정 로직을 남기면 이후 실험의 기준이 흐려진다.

따라서 harmonic verifier / low-pass verifier 코드는 제거하고, 기존 `pitchy` raw detection 경로로 되돌린다.

## 5. 배운 점

배음은 기본음보다 위쪽에 생기므로, 문제의 핵심은 "위쪽 배음을 기본음으로 잘못 선택하는 것"이다.

하지만 단순히 가장 낮은 spectrum peak를 선택하면 안 된다.

잘못 낮아질 수 있는 원인:

- 50/60Hz hum
- 마찰음이나 충격성 저주파
- 방 울림
- autocorrelation 계열의 subharmonic 착시

따라서 다음 접근은 "가장 낮은 peak"가 아니라 "현재 spectrum을 설명할 수 있는 가장 낮은 그럴듯한 fundamental candidate"를 고르는 방식이어야 한다.

## 6. 다음 접근 후보

다음 실험은 low-pitch 기반 harmonic candidate selection으로 진행한다.

후보 방향:

1. `pitchy` 후보 `f`에서 `f`, `f / 2`, `f / 3`, `f / 4` 후보를 만든다.
2. layout 기반 허용 frequency range 밖의 후보는 버린다.
3. 각 후보 `c`에 대해 `c`, `2c`, `3c`, `4c`, `5c` 주변 harmonic energy를 합산한다.
4. hum / 충격성 저음 방지를 위해 최소 frequency와 최소 harmonic support 조건을 둔다.
5. 충분히 설명 가능한 후보들 중 가장 낮은 후보를 선택한다.

주의:

- 실제 A#4 연주도 A#3의 2배음으로 설명될 수 있으므로, spectrum만으로 완전한 구분은 어렵다.
- 현재 프랙티스 모드는 pitch class 중심 판정이므로, 저음역 배음 오인식 방지를 더 우선하는 정책이 실용적일 수 있다.
- 다음 구현은 디버그 표시 또는 console instrumentation을 먼저 둬서 후보별 score를 실제 마이크에서 확인할 수 있게 하는 편이 좋다.

## 8. 이후 시도한 저음 기반 후보 선택

롤백 후 같은 날 `resolveLowFundamentalPitchCandidate()`를 새로 추가했다.

새 접근:

1. `pitchy` 후보 `f`를 먼저 받는다.
2. `f`, `f / 2`, `f / 3`, `f / 4` 후보를 만든다.
3. layout 기반 frequency range 밖 후보는 버린다.
4. 각 후보에 대해 1~6배음의 Goertzel band energy를 합산한다.
5. 후보의 1배음 또는 2배음 primary energy가 너무 약하면 버린다.
6. 낮은 후보는 detected peak와 다른 위치의 distinct harmonic energy가 일정 이상 있어야 인정한다.
7. 기준 후보 `f`와 비교해 harmonic score가 일정 비율 이상이면 후보로 인정한다.
8. 인정된 후보 중 가장 낮은 frequency를 선택한다.

의도:

- A#3의 기본음이 약하고 A#4가 강하게 잡혀도, A#4를 A#3의 2배음으로 설명할 수 있으면 A#3를 선택한다.
- 단일 고음 peak 때문에 `f / 2`, `f / 3`, `f / 4`로 내려가는 상황은 distinct harmonic energy 조건으로 줄인다.
- layout range를 사용해 악보가 기대하지 않는 지나치게 낮은 hum 후보를 배제한다.

검증:

- A#4 성분이 강한 합성 파형에서 A#3 후보를 선택한다.
- A#3가 layout range 밖이면 A#4를 유지한다.
- 단일 A#4 peak는 A#3로 내리지 않고 A#4를 유지한다.
- `npm run test:game` 통과
- `npm run build` 통과

결과:

- 실제 문제를 충분히 해결하지 못했다.
- 고음 실제 연주가 저음 후보로 내려가는 부작용도 확인되었다.
- 따라서 이 후보 해석 방식은 제품 코드에서 롤백하고, 다음 방식으로 전환한다.

## 9. 실제 에너지 최저 주파수 방식 시도와 롤백

다음 시도는 `resolveLowestEnergyPitchCandidate()`로 전환했다.

기본 전제:

- 잡음 처리는 마이크 하드웨어와 사용 환경이 처리한다고 본다.
- 앱은 spectrum을 harmonic 후보로 해석하기보다, layout 범위 안에서 실제로 일정 이상 에너지를 내는 가장 낮은 주파수를 찾는다.
- `pitchy` 결과는 fallback과 clarity 판정에는 유지한다.

구현 방향:

1. layout 기반 frequency range 안을 50cent 간격으로 훑는다.
2. 각 주파수 주변의 Goertzel band energy를 계산한다.
3. 가장 강한 energy 대비 일정 비율 이상이고, 전체 RMS 기준 절대 하한도 넘는 첫 번째 저주파 후보를 고른다.
4. 선택된 후보 주변은 10cent 단위로 다시 훑어 local peak를 찾는다.
5. 정제 결과가 기존 `pitchy` 후보와 거의 같은 영역이면 `pitchy`의 원래 주파수를 유지해 cent 표시 손실을 줄인다.

검증:

- A#3와 A#4가 함께 있고 A#3 에너지가 threshold 이상이면 A#3를 선택한다.
- 약한 저주파 성분은 threshold 아래이면 무시하고 A#4를 유지한다.
- `npm run test:game` 통과
- `npm run build` 통과

실제 결과:

- 실제 테스트에서 모든 음정을 안정적으로 잡지 못하는 문제가 발생했다.
- 음정 검출의 기준이 `pitchy`의 주기성 판단에서 단일 주파수 에너지 scan으로 이동하면서, 일부 음정의 안정성이 떨어졌다.
- 합성 테스트는 통과했지만 실제 마이크 입력 전체를 대표하지 못했다.

롤백 판단:

- `resolveLowestEnergyPitchCandidate()` 방식은 제품 코드에서 제거한다.
- `game_pitch_input.ts`는 기존 `pitchy` raw detection 경로로 되돌린다.
- `test_game.ts`에 추가했던 최저 에너지 scan 테스트도 제거한다.

남은 판단:

- 단순한 octave 보정, low-pass 보정, 낮은 harmonic candidate 선택, 최저 energy scan 모두 현재 실사용 문제를 안정적으로 해결하지 못했다.
- 다음 접근은 제품 코드에 바로 넣기보다, 실제 마이크 frame의 raw waveform / spectrum / `pitchy` 결과 / 후보 score를 확인할 수 있는 디버그 계측을 먼저 만들어야 한다.
- 실제 입력 데이터를 관찰한 뒤 알고리즘을 고르는 편이 안전하다.

## 10. 다음 실험 브랜치 작업 예정

실험은 안정판 `main`이 아니라 별도 브랜치에서 진행한다.

추천 브랜치명:

- `experiment/practice-pitch-detection`

목표:

- 현재 `pitchy` 기반 제품 경로를 보존한다.
- 새 알고리즘은 교체가 아니라 비교 가능한 backend로 붙인다.
- 실제 마이크 환경에서 raw 입력과 detector 결과를 나란히 보고 판단한다.

### 10.1. 1순위: 디버그 계측

먼저 알고리즘을 바꾸지 않고 관찰 도구를 만든다.

기록할 값:

- `rawFrequencyHz`
- 최종 `frequencyHz`
- `clarity`
- `rms`
- voiced / reject reason
- layout frequency range
- 가능하면 짧은 spectrum 또는 후보별 score

표시 방식 후보:

- `?pitchDebug=1` query로 켜는 console table
- practice panel 내부 작은 debug row
- 일정 시간 동안 frame snapshot을 배열로 모아 copy 가능한 JSON 출력

목적:

- A#3가 A#4로 잡히는 순간에 원본 detector 결과와 입력 세기가 어떻게 변하는지 확인한다.
- 알고리즘을 바꾸기 전에 실제 실패 데이터를 확보한다.

### 10.2. 2순위: 라이브러리 비교 backend

비교 후보:

1. `pitchy` 현재 경로
   - 기준선으로 유지한다.
   - MPM 기반이라 실시간성이 좋고 현재 앱에 이미 통합되어 있다.

2. `aubio` / `aubio.js` 계열
   - 후보 알고리즘: `yin`, `yinfft`, `mcomb`, `fcomb`, `specacf`
   - 목표: 이미 검증된 C/WASM 기반 pitch backend를 브라우저에서 비교한다.
   - 확인할 점: npm 패키지 유지 상태, 브라우저 번들 가능 여부, 라이선스, TypeScript 선언 상태.

3. Essentia.js
   - 후보 알고리즘: `PitchYin`, `PitchYinFFT`, `PitchMelodia`
   - 목표: 주파수 영역 / melody extraction 계열이 A#3 -> A#4 문제에 강한지 비교한다.
   - 주의: 라이선스가 AGPL 계열이면 제품 직접 의존성으로 넣지 않고 비교 실험 또는 보고서 참고용으로 제한한다.

4. Cepstrum analysis
   - 목표: harmonic spacing / quefrency peak가 기본음을 더 안정적으로 가리키는지 확인한다.
   - 직접 구현은 최소 실험으로만 한다.
   - 가능하면 라이브러리 구현을 먼저 확인한다.

5. CREPE / SPICE 계열 ML backend
   - 목표: octave error에 강한지 비교한다.
   - 후순위 이유: 모델 크기, 초기 로딩, 브라우저 성능, 배포 복잡도, TypeScript 통합 부담이 크다.

### 10.3. 비교 기준

실제 테스트 음:

- A#3
- A#4
- A3 ~ C4 저음역 주변
- gliss 중 A#3 근처를 통과하는 입력
- vibrato가 걸린 저음 입력

비교 항목:

- octave error 빈도
- voiced / unvoiced 안정성
- pitch dot 흔들림
- latency
- CPU 사용 체감
- 빠른 gliss 반응성
- vib / trem 판정 부작용

채택 기준:

- A#3 -> A#4 오인식을 줄이면서 실제 A#4를 과하게 A#3로 내리지 않아야 한다.
- 기존 `pitchy`보다 전체 음정 인식 안정성이 나빠지면 채택하지 않는다.
- 제품 경로에 넣기 전 `npm run test:game`, `npm run build`를 통과해야 한다.
- 라이선스가 제품 배포와 충돌하면 직접 의존성으로 채택하지 않는다.

### 10.4. 실험 순서

1. `experiment/practice-pitch-detection` 브랜치를 만든다.
2. pitch debug 계측을 먼저 추가한다.
3. 실제 문제 마이크에서 A#3/A#4 frame을 관찰한다.
4. `aubio` 또는 Essentia.js 중 브라우저 통합이 쉬운 쪽을 하나만 먼저 붙인다.
5. 같은 입력에서 `pitchy`와 새 backend 결과를 나란히 비교한다.
6. 개선이 확인된 경우에만 backend 선택 UI 또는 설정값을 검토한다.
7. 개선이 확인되지 않으면 제품 코드는 유지하고 보고서/메모에 실패 결과만 남긴다.
