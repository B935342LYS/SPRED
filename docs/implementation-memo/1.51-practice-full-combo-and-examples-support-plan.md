## 2. Practice Target Selection Fix

글리산도 자체는 정상 판정되지만, 글리산도 종료 직후 이웃한 같은 pitch note가 잘 판정되지 않는 문제가 있었다.

원인 후보를 검토한 결과, onset 생성 자체보다는 target 선택 tie-break가 문제일 가능성이 높았다.
현재 scoring은 여러 target이 transition grace 구간에서 겹치면 pitch class error가 가장 작은 target 하나를 선택한다.
같은 pitch의 gliss end target과 다음 ordinary note target은 pitch error가 같으므로, 이전 target이 먼저 선택될 수 있었다.

수정 방향:

- pitch error가 더 작으면 기존처럼 그 target을 선택한다.
- pitch error가 같으면 현재 score time이 실제 note 구간 안에 있는 target을 우선한다.
- 따라서 grace로 남은 이전 note보다 새로 시작된 adjacent note가 우선된다.

검증:

- `test_game.ts`에 gliss end F#과 다음 F#이 겹치는 case를 추가했다.
- 같은 pitch gliss end grace target이 다음 active note target의 onset credit을 가져가지 않는지 확인했다.

## 3. FULL COMBO Result Display

실사용 테스트에서 result dialog가 최종 성취 상태를 더 분명히 보여 줄 필요가 있었다.

구현 정책:

- `Perfect`, `Ok`, `Bad`, `Miss`로 실제 반영된 scoring sample 수를 judged sample count로 본다.
- judged sample count가 1개 이상이고 `bestCombo === judgedSampleCount`이면 `FULL COMBO`를 표시한다.
- Standard / Easy Mode에서는 score-eligible `Bad`가 combo를 이어가므로 FULL COMBO가 가능하다.
- Pro Mode에서는 `Bad`가 combo를 끊으므로 같은 Bad 결과라도 FULL COMBO가 아니다.
- effect bonus는 별도 보너스 축이므로 FULL COMBO 조건에 포함하지 않는다.

구현 위치:

- `regression-code/src/app/game/game_types.ts`
  - `isFullComboSummary()` 추가
- `regression-code/src/app/game/game_ui.ts`
  - result dialog 열기 전 `FULL COMBO` 표시 여부 갱신
- `regression-code/index.html`
  - result dialog에 `result-full-combo` 요소 추가
- `regression-code/styles/app.css`
  - result banner 스타일 추가

검증:

- `npm run test:game`
- `npm run typecheck`
- `npm run build`

배포 staging:

- `regression-code-test-publish/`에 런타임 파일을 동기화했다.

## 4. Examples / Supabase Planning

2.11 Examples 지원은 로컬 정적 예제를 두는 방향에서 Supabase Edge Function 기반 방향으로 조정했다.

현재 결정:

- `public/examples/manifest.json`과 로컬 example Score JSON은 첫 구현에 넣지 않는다.
- DB / Edge Function 로드 실패 시 fallback score를 불러오지 않고 현재 악보를 유지한다.
- Examples는 File menu에서 진입한다.
- Examples dialog는 임시 암호 단어 입력을 포함한다.
- 앱은 Supabase table을 직접 REST select하지 않고 Edge Function만 호출한다.
- Edge Function은 서버 환경 변수의 암호 단어를 확인한 뒤 manifest를 반환한다.
- `example_scores` table은 ScoreFile 본문 저장소가 아니라 catalog table이다.
- 첫 table 초안은 최소 표시 metadata만 둔다.
- 라이선스, 설명, 썸네일은 첫 임시 테스트 구현에서 제외한다.
- `created_at`, `updated_at`은 catalog 등록일/갱신일로 사용자에게 표시할 수 있다.

현재 catalog 필드:

- `id`
- `title`
- `artist`
- `difficulty_basic`
- `difficulty_optional`
- `difficulty_extra`
- `supported_tracks`
- `duration_seconds`
- `size_bytes`
- `score_url`
- `display_order`
- `is_published`
- `access_group`
- `created_at`
- `updated_at`

보안 판단:

- `service_role` key는 프론트엔드 코드, GitHub repository, 공개 문서 예시에 넣지 않는다.
- `anon` role에 `example_scores` 직접 select 권한을 주지 않는 임시 테스트 모델을 우선한다.
- SQL injection 방어는 앱이나 Edge Function이 사용자 입력으로 SQL fragment를 조립하지 않는 구조를 기본으로 한다.
- Edge Function도 고정 query와 parameterized / typed client 호출만 사용한다.
- Edge Function 응답 manifest와 선택된 Score JSON은 모두 앱 validator를 다시 통과해야 한다.

## 5. Proposed Module Boundary

예상 구현 위치:

```text
regression-code/src/app/examples/
  example_types.ts
  example_manifest_validate.ts
  example_supabase_provider.ts
  example_dialog.ts
  example_binding.ts
```

역할:

- `example_types.ts`: manifest, item, provider, Supabase row 타입 정의
- `example_manifest_validate.ts`: Edge Function 응답 검증과 optional field 정규화
- `example_supabase_provider.ts`: Edge Function 호출, access word 전달, score text fetch
- `example_dialog.ts`: dialog 렌더링, notice, 목록 선택, load button 상태
- `example_binding.ts`: File > Examples 버튼과 기존 score load pipeline 연결

`src/assets/`에는 예제 score를 두지 않는다.
이번 기능은 정적 asset이 아니라 원격 예제 로드 기능이므로 `src/app/examples/`가 더 적합하다.
