# 1.6. 1단계 전역 셀 Parser 구현 메모
> 본 설계 메모 문서 1.6에서는 1단계 구현 중 `parse_global_cell.ts`의 역할과 구현 판단을 기록한다.
> 본 문서는 active specification을 대체하지 않는다.
> 함수별 세부 설명은 코드 파일의 `/** ... */` 주석을 중심으로 하고, 본 문서는 흐름과 사용 문법, 설계 판단을 보충한다.

## 1. 개요

score 모듈의 로드 경로가 `RuntimeDocument` 생성까지 연결되었으므로, 다음 단계는 parser 구현이다.
이번 작업에서는 note parser보다 범위가 작고 fixture로 즉시 검증 가능한 전역 셀 parser를 먼저 구현하였다.

현재 파이프라인에서 전역 셀 parser의 위치는 아래와 같다.

```text
RuntimeDocument
-> globalLines.cells
-> parseGlobalCell()
-> ParsedGlobalCell
```

전역 셀 parser는 셀 하나의 `rawText`만 해석한다.
같은 kind의 전역 셀들을 열 순서로 연결하고, `<`, `>`, `><`의 실제 구간 의미를 확정하는 일은 analyzer 단계의 책임이다.

## 2. parse_global_cell.ts의 현재 책임

현재 구현은 `parseGlobalCell(input, context)`를 공개 진입점으로 제공한다.

입력은 다음 정보를 가진다.

- `rowId`
  - 어떤 전역 행 셀인지 식별한다.
- `col`
  - 현재 구현에서는 오류 메시지에 직접 쓰지 않지만, parser 입력 계약과 후속 캐시 key를 위해 유지한다.
- `rawText`
  - 사용자가 입력한 원본 문자열이다.

문맥은 `rowById`를 가진다.
parser는 `rowId`를 통해 전역 행의 `GlobalKind`를 유도한다.
이는 JSON 셀에 `kind`를 중복 저장하지 않고, layout 정의를 기준으로 의미를 판정하기 위한 구조이다.

현재 지원하는 전역 행은 다음 네 종류이다.

- `bpm`
  - 양의 정수 또는 양의 실수
  - ramp token 허용
- `dynamics`
  - 0 이상 150 이하의 정수
  - ramp token 허용
- `beatsPerBar`
  - 1 이상의 정수
  - ramp token 금지
- `stepsPerBeat`
  - 1 이상의 정수
  - ramp token 금지

## 3. 구현 판단

### 3.1. global parser를 먼저 구현한 이유

`docs/implementation-memo/1.0-roadmap.md`에는 parse 구현 순서가 note parser, global parser, document parser 순서로 적혀 있다.
다만 이번 작업에서는 전역 셀 parser를 먼저 구현하였다.

이 판단의 이유는 다음과 같다.

- 전역 셀 문법은 note 셀 문법보다 작다.
- 현재 fixture에 전역 셀 4개가 이미 존재한다.
- score load pipeline 이후 parser 연결을 작은 단위로 검증하기 좋다.
- note parser는 hold, escape, modifier, tuplet까지 포함하므로 별도 단계로 나누는 편이 안전하다.

따라서 이번 구현 순서는 task-local 판단이며, active specification의 parser/analyzer 경계를 바꾸지는 않는다.

### 3.2. rowId에서 GlobalKind를 유도하는 이유

전역 셀 JSON에는 `kind` 필드를 직접 저장하지 않는다.
셀은 `rowId`, `col`, `rawText`만 저장하고, `rowId`가 가리키는 `layout.rowDefinitions`에서 kind를 얻는다.

이 구조는 다음 장점이 있다.

- 저장 셀 구조가 단순하게 유지된다.
- 같은 전역 셀의 kind 정보가 JSON에 중복 저장되지 않는다.
- validator와 index builder가 만든 `rowById`를 parser도 그대로 사용할 수 있다.

### 3.3. ramp token 처리 방식

`bpm`과 `dynamics`는 문자열 끝의 ramp token을 허용한다.
토큰은 다음 순서로 판정한다.

```text
"><" -> "endStart"
"<"  -> "start"
">"  -> "end"
없음 -> "none"
```

`><`는 두 글자 토큰이므로 가장 먼저 확인한다.
그렇지 않으면 `100><`가 `100>`와 남은 `<`처럼 잘못 해석될 수 있다.

`beatsPerBar`, `stepsPerBeat`는 ramp token을 허용하지 않는다.
이 두 kind에서 문자열 끝에 `<`, `>`, `><`가 있으면 `invalid_ramp_token`으로 처리한다.

### 3.4. 숫자 파싱 범위

현재 숫자 파싱은 입력 문자열 그대로를 검사한다.
앞뒤 공백을 자동으로 제거하지 않는다.

현재 허용 형태는 다음과 같다.

- 정수: `^[0-9]+$`
- 십진수: `^[0-9]+(?:\.[0-9]+)?$`

따라서 `120`, `87.5`는 허용되지만 ` 120`, `120 `, `1e2`, `.5`, `1.`은 허용하지 않는다.
이는 `1.6-global-cell-parser-spec.md`의 “입력 문자열 그대로 판정” 원칙을 따른다.

## 4. 이번 구현에서 사용한 주요 TypeScript 문법

### 4.1. 타입 가드

`isGlobalRow()`는 row가 global row인지 확인하는 타입 가드이다.

```ts
function isGlobalRow(
  row: RowDefinition | undefined,
): row is Extract<RowDefinition, { type: "global" }>
```

이 함수가 true인 분기에서는 TypeScript가 row를 global row로 좁혀서 `row.kind` 접근을 허용한다.

### 4.2. Extract

선형 변화 token을 허용하는 kind와 허용하지 않는 kind를 분리하기 위해 `Extract`를 사용한다.

```ts
Extract<GlobalKind, "bpm" | "dynamics">
Extract<GlobalKind, "beatsPerBar" | "stepsPerBeat">
```

이를 통해 함수 인수 단계에서 가능한 kind 범위를 좁힐 수 있다.

### 4.3. 정규식 기반 숫자 검사

숫자 문자열은 `Number.parseFloat()`나 `Number.parseInt()`만으로 바로 처리하지 않는다.
JavaScript의 숫자 파싱 함수는 앞부분만 숫자이면 일부 문자열을 받아들이는 경우가 있기 때문이다.

예를 들어 `Number.parseInt("12abc", 10)`은 `12`를 반환한다.
parser에서는 이런 입력을 유효한 숫자로 보지 않아야 하므로, 먼저 정규식으로 전체 문자열 형식을 검사한다.

### 4.4. discriminated union 반환

`ParsedGlobalCell`은 `kind` 필드로 성공/실패와 세부 종류를 구분한다.

```ts
kind: "linearGlobalValue"
kind: "instantGlobalValue"
kind: "invalid"
```

후속 analyzer는 `switch (parsedCell.kind)` 또는 조건문으로 타입을 좁혀 각 필드에 접근할 수 있다.

## 5. 테스트 확인

`dev/test_parse.ts`를 추가하고 `package.json`에 다음 스크립트를 등록하였다.

```json
{
  "test:parse": "tsx ./dev/test_parse.ts"
}
```

현재 테스트는 `minimal-valid-score.json`을 `loadRuntimeDocument()`로 로드한 뒤, `globalLines.cells`의 모든 전역 셀에 `parseGlobalCell()`을 적용한다.
그리고 invalid 결과가 하나라도 있으면 실패로 처리한다.

현재 fixture 기준 성공 출력은 다음과 같다.

```text
Global cell parse completed.
global cells: 4
invalid global cells: 0
```
