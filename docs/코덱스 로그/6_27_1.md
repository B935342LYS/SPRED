**Undo/Redo 구현 방향**

`docs/2.10-undo-redo-edit-history-spec.md` 기준으로 첫 구현은 `ScoreFile` 전체 snapshot이 아니라 cell patch 기반 history입니다.

핵심 모델은 다음입니다.

```ts
UndoHistoryState {
  undoStack
  redoStack
  maxEntries: 50
}

UndoHistoryEntry {
  id
  label
  patches
}

CellHistoryPatch {
  selection
  beforeRawText: string | null
  afterRawText: string | null
}
```

첫 undo 대상은 다음으로 제한합니다.

- 단일 note/global cell edit
- drag edit의 현재 RAF batch 단위
- range delete
- range paste

첫 구현에서 제외할 대상은 다음입니다.

- score file load
- Details metadata edit
- YouTube metadata reload
- Clear All
- Expand Right / Trim Right
- layout apply / layout preset apply
- local save / file download
- track/view/playback 상태 변경
- range selection 자체

> Undo/Redo의 history 저장 한도는 어느정도인가요?

현재 한도는 **50개 history entry**입니다.

구현 위치는 [edit_history.ts](./regression-code/src/app/edit/edit_history.ts)입니다.

```ts
export const DEFAULT_UNDO_HISTORY_MAX_ENTRIES = 50;
```

의미는 “cell 개수 50개”가 아니라 **사용자 조작 단위 50개**입니다.

예:
- 단일 cell edit 1회 = 1 entry
- range delete 1회 = 1 entry
- range paste 1회 = 1 entry
- drag edit은 현재 1차 구현상 RAF batch 단위로 여러 entry가 생길 수 있음

50개를 넘으면 가장 오래된 entry부터 제거됩니다. 저장 JSON에는 포함되지 않고, 앱 세션 안에서만 유지됩니다.

> 50개면 비용 면에서 적절한 수치인가요?

네, 현재 구조에서는 **50개가 꽤 보수적이고 적절한 기본값**입니다.

이 구현은 `ScoreFile` 전체 스냅샷을 저장하지 않고, 변경된 cell의 `beforeRawText` / `afterRawText`만 저장합니다. `rawText`도 최대 100자 제한이 있으므로 메모리 비용은 대략 다음에 비례합니다.

```text
history 비용 ~= entry 수 * entry당 변경 cell 수 * rawText 길이
```

일반 단일 입력은 거의 비용이 작습니다. range paste/delete처럼 한 번에 수십~수백 cell을 바꾸는 경우가 더 크지만, 그래도 전체 악보 JSON 50벌을 저장하는 방식과는 비교가 안 될 정도로 가볍습니다.

다만 현재 drag edit은 “드래그 전체 1회”가 아니라 RAF batch 단위로 entry가 생길 수 있어서, 빠르게 드래그하면 50개가 생각보다 빨리 찰 수 있습니다. 그래서 다음 개선으로는 drag pointerdown~pointerup 전체를 1 entry로 묶는 게 UX와 비용 면에서 모두 좋습니다.

결론:
- 단기 기본값으로는 `50` 적절
- range paste가 매우 큰 악보에서 반복되면 `30~50` 사이가 안전
- drag transaction grouping을 넣으면 `100`까지 늘려도 큰 부담은 낮음

고려는 하되, **그 사양을 최저 기준으로 강하게 맞출 필요는 낮아 보입니다.**

i5-10210U / RAM 8GB는 2019년형 저전력 노트북으로, 지금 기준으로는 확실히 낮은 편입니다. 특히 Chrome + Canvas + DevTools + CPU throttling/계측까지 켜면 체감이 더 나빠질 수 있습니다. 그 환경에서 “약간의 프레임 드랍” 정도라면 현재 MVP 기준으로는 수용 가능한 상태에 가깝습니다.

다만 이 프로젝트 특성상 사용자가 학교/개인 노트북에서 쓸 가능성이 있으니, 아래 정도는 기준에 넣는 게 좋습니다.

권장 기준:
- 데스크톱: 부드러워야 함
- 2019년 저전력 i5급 노트북: 재생/스크롤이 약간 드랍될 수 있으나 사용 가능해야 함
- edit 입력: 눈에 띄는 멈춤 없이 반응해야 함
- 긴 악보 9000+ column: 완전한 60fps보다 조작 가능성과 음 끊김 방지가 우선

즉, **이 노트북은 “최적화 참고 기기”로 쓰되, 모든 시각 효과를 거기에 맞춰 낮추지는 않는 것**이 좋습니다.

다음에 할 만한 현실적인 개선은:
- 재생 중 seek label/BPM 갱신 빈도 낮추기
- perf profiler가 꺼졌을 때 overhead가 충분히 낮은지 확인
- 긴 악보에서 scroll redraw가 발생하는지 재확인
- 필요하면 저사양용 옵션 추가
  - 예: `Smooth Follow: On/Light`
  - Light에서는 seek label 갱신 빈도만 낮추고 scroll은 RAF 유지

지금 판단으로는 노래방 모드 전에 “추가 대형 최적화”까지 들어갈 상황은 아니고, **현재 노트북에서 사용 가능한 수준인지 몇 곡으로 실사용 테스트**해보는 정도가 적절합니다.

판단은 이렇습니다: **현재 단계에서는 안정화가 충분히 되었다고 봐도 됩니다.**

근거:
- 데스크톱에서 재생이 충분히 부드러움
- Chrome 4x/6x throttling에서 병목 후보를 확인했고, seek UI 쪽 overhead를 줄임
- 30fps follow scroll은 수치상 가벼워도 체감이 나빠서 폐기했고, RAF 기반 follow scroll을 유지
- 2019년 저전력 i5-10210U / RAM 8GB 노트북에서도 악보 재생이 정상 가능
- 약간의 frame drop은 있지만 오디오 재생과 사용성은 유지됨

따라서 추가 최적화는 지금 더 밀어붙이기보다, **이후 기능 추가 중 회귀가 보이거나 실제 사용자 테스트에서 문제가 확인될 때 진행**하는 게 맞습니다.

다음 단계로 넘어가도 됩니다. 다음 후보는 이전 판단대로 `loop playback 연결` 또는 그 다음 큰 기능인 `노래방 모드 MVP 설계`입니다.
