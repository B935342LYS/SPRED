# 1.12. 2단계 Canvas API 기초 구현 메모
> 본 설계 메모 문서 1.12에서는 다음 canvas layer 전환 구현 전에 알아야 할 Canvas API 개념을 정리한다.
> 본 문서는 active specification을 대체하지 않는다.
> 실제 renderer 계약은 `docs/1.9-mvp-analyzer-renderer-ui-spec.md`와 `docs/2.0-ui-mvp-spec.md`를 우선 기준으로 한다.

## 1. 개요

다음 작업은 `score-viewer`의 임시 DOM grid를 canvas layer 구조로 바꾸는 것이다.
canvas는 HTML 요소이지만, 내부 그림은 DOM node가 아니라 bitmap에 직접 그려진다.
따라서 일반 HTML/CSS와 다른 사고방식이 필요하다.

기본 흐름은 아래와 같다.

```text
HTML
-> <canvas>
-> getContext("2d")
-> size 설정
-> clear
-> draw grid / notes / markers
-> scroll 또는 상태 변경 시 다시 draw
```

## 2. canvas 요소의 두 가지 크기

canvas에는 서로 다른 두 크기가 있다.

```html
<canvas width="800" height="400"></canvas>
```

```css
canvas {
  width: 800px;
  height: 400px;
}
```

- HTML 속성 `width`, `height`
  - 실제 bitmap 해상도이다.
  - JavaScript에서 `canvas.width`, `canvas.height`로 다룬다.
- CSS `width`, `height`
  - 화면에 보이는 크기이다.

두 값이 맞지 않으면 브라우저가 bitmap을 늘리거나 줄여서 표시하므로 선과 글자가 흐릿해질 수 있다.
고해상도 화면에서는 `devicePixelRatio`까지 반영해야 한다.

## 3. 2D context

canvas에 그림을 그리려면 2D context를 얻는다.

```js
const canvas = document.querySelector("canvas");
const ctx = canvas.getContext("2d");
```

`ctx`는 실제 그리기 명령을 가진 객체이다.

주요 명령은 다음과 같다.

```js
ctx.clearRect(x, y, width, height);
ctx.fillRect(x, y, width, height);
ctx.strokeRect(x, y, width, height);
ctx.fillText(text, x, y);
ctx.beginPath();
ctx.moveTo(x, y);
ctx.lineTo(x, y);
ctx.stroke();
```

canvas는 retained mode가 아니라 immediate mode이다.
즉 한 번 그린 rectangle이나 text는 DOM 요소처럼 남아 있는 객체가 아니다.
다음 화면에서 바뀐 내용이 필요하면 지우고 다시 그려야 한다.

## 4. 좌표계

canvas 좌표는 기본적으로 좌상단이 원점이다.

```text
(0, 0) -----> x 증가
  |
  |
  v
 y 증가
```

악보 renderer에서는 다음 변환이 필요하다.

```text
column index -> x
row index    -> y
tick         -> x
note event   -> rectangle
```

예를 들어 column width가 48px이면 다음처럼 계산한다.

```js
const x = columnIndex * columnWidth;
```

row height가 36px이면 다음처럼 계산한다.

```js
const y = rowIndex * rowHeight;
```

이 좌표 계산은 renderer와 playback controller가 공유해야 한다.
재생 중 현재 tick이 화면 기준선에 오도록 scroll을 움직일 때도 같은 `tick -> x` 계산을 써야 한다.

## 5. 선명한 canvas를 위한 DPR 처리

브라우저의 CSS pixel과 실제 device pixel은 다를 수 있다.
고해상도 화면에서는 `window.devicePixelRatio`가 2 이상일 수 있다.

권장 초기화 흐름은 아래와 같다.

```js
function resizeCanvas(canvas, cssWidth, cssHeight) {
  const dpr = window.devicePixelRatio || 1;
  canvas.style.width = `${cssWidth}px`;
  canvas.style.height = `${cssHeight}px`;
  canvas.width = Math.floor(cssWidth * dpr);
  canvas.height = Math.floor(cssHeight * dpr);

  const ctx = canvas.getContext("2d");
  ctx.setTransform(dpr, 0, 0, dpr, 0, 0);
  return ctx;
}
```

`setTransform(dpr, 0, 0, dpr, 0, 0)`을 사용하면 이후 그리기 좌표는 CSS pixel 기준으로 쓸 수 있다.
예를 들어 `fillRect(0, 0, 48, 36)`은 화면 기준 48px x 36px 크기로 그려진다.

## 6. draw 순서

canvas는 나중에 그린 것이 앞에 보인다.
따라서 한 canvas 안에서 그릴 때는 아래 순서를 지킨다.

```text
1. background
2. grid
3. note rectangle
4. note text
5. marker
6. selection / hover / playhead
```

하지만 모든 것을 한 canvas에 그리면 일부만 바뀌어도 전체를 다시 그려야 한다.
그래서 본 프로젝트는 layer를 나누는 방식을 사용한다.

```text
base layer        grid, row background
note layer        note rectangle, note text
marker layer      beat/bar/loop/playback markers
```

자주 바뀌는 layer와 거의 바뀌지 않는 layer를 분리하면 redraw 비용을 줄일 수 있다.
선택, hover, drag preview 같은 조작 overlay를 실제로 보여주기로 결정하기 전까지 interaction layer는 만들지 않는다.

## 7. canvas layer 구조

여러 canvas를 같은 위치에 겹치려면 부모 stage를 `position: relative`로 두고, 각 canvas를 absolute로 겹친다.

```html
<div class="score-canvas-stage">
  <canvas class="score-layer base-layer"></canvas>
  <canvas class="score-layer note-layer"></canvas>
  <canvas class="score-layer marker-layer"></canvas>
</div>
```

```css
.score-canvas-stage {
  position: relative;
  width: 2400px;
  height: 1400px;
}

.score-layer {
  position: absolute;
  inset: 0;
  width: 100%;
  height: 100%;
}
```

DOM tree는 layer를 구분하지만, 각 canvas 내부의 note나 grid는 DOM 요소가 아니다.
hover나 click 처리는 좌표를 직접 계산해야 한다.

## 8. scroll과 viewport

긴 악보를 전부 DOM으로 만들지 않고 canvas로 그리더라도, scroll container는 DOM을 그대로 사용할 수 있다.

```text
score-area
  overflow: auto
  score-canvas-stage
    canvas layers
```

초기 구현에서는 stage 크기를 전체 악보 크기로 두고 스크롤한다.
이후 성능 최적화가 필요해지면 viewport virtualization으로 전환할 수 있다.

가상화 방식은 아래와 같다.

```text
scrollLeft / scrollTop 확인
-> 보이는 column / row 범위 계산
-> 그 범위만 canvas에 그림
-> 좌표에는 viewport offset을 반영
```

MVP에서는 먼저 전체 stage canvas를 만들고, 나중에 악보가 커졌을 때 가상화를 도입한다.

## 9. 그리기 상태와 `save()` / `restore()`

2D context에는 현재 fillStyle, strokeStyle, font, lineWidth 같은 상태가 있다.

```js
ctx.fillStyle = "#46a6b8";
ctx.strokeStyle = "#0f5664";
ctx.font = "12px Arial";
```

복잡한 draw 함수 안에서 상태가 섞이는 것을 막으려면 `save()`와 `restore()`를 사용할 수 있다.

```js
ctx.save();
ctx.fillStyle = "#46a6b8";
ctx.fillRect(x, y, width, height);
ctx.restore();
```

`save()`는 현재 상태를 stack에 저장하고, `restore()`는 직전 상태로 되돌린다.
작은 MVP에서는 필수는 아니지만, layer draw 함수가 늘어나면 실수를 줄여준다.

## 10. text 그리기

노트 텍스트는 `fillText()`로 그린다.

```js
ctx.font = "700 12px Arial";
ctx.textAlign = "center";
ctx.textBaseline = "middle";
ctx.fillText(label, x + width / 2, y + height / 2);
```

`textAlign = "center"`는 x 좌표를 텍스트 중앙 기준으로 해석한다.
`textBaseline = "middle"`은 y 좌표를 텍스트 중앙 기준으로 해석한다.

canvas text는 DOM text처럼 자동 줄바꿈이나 ellipsis를 제공하지 않는다.
텍스트가 rectangle보다 길면 직접 생략하거나 작게 그려야 한다.
MVP에서는 note label이 짧다는 전제로 시작하고, 이후 필요하면 text 측정(`measureText`)을 추가한다.

## 11. clear와 redraw

canvas를 다시 그리기 전에 보통 해당 layer를 지운다.

```js
ctx.clearRect(0, 0, width, height);
```

전체 layer를 매번 지우는 방식은 단순하지만, 악보가 커지면 비용이 커질 수 있다.
나중에는 dirty rectangle 또는 viewport 단위 redraw를 고려할 수 있다.

MVP에서는 다음 원칙으로 충분하다.

- base layer는 layout / zoom / score 변경 시 다시 그림
- note layer는 analysis result 또는 active track 변경 시 다시 그림
- marker layer는 beat/bar/loop/playback marker 변경 시 다시 그림
- interaction layer는 후속 edit overlay 구현 시 추가하고, hover / selection 변경 때 자주 지우는 layer로 둔다

## 12. hit testing

canvas 내부 그림은 DOM 요소가 아니므로, 특정 note rectangle을 클릭했는지 브라우저가 알려주지 않는다.
마우스 좌표를 직접 악보 좌표로 바꾼 뒤 hit test를 해야 한다.

```js
const rect = canvas.getBoundingClientRect();
const x = event.clientX - rect.left;
const y = event.clientY - rect.top;
```

scroll container가 있다면 scroll offset도 함께 고려한다.

```js
const scoreX = x + scoreArea.scrollLeft;
const scoreY = y + scoreArea.scrollTop;
```

그 뒤 note rectangle 목록에서 `x/y`가 범위 안에 들어가는지 확인한다.
edit mode의 hover, selection, drag는 이 원리를 사용한다.

## 13. 접근성 주의

canvas 내부 그림은 screen reader가 직접 읽을 수 없다.
따라서 상단 메뉴와 편집 패널은 DOM으로 유지하는 것이 중요하다.

악보 canvas에는 다음 정도의 접근성 보조를 둔다.

```html
<section class="score-area" aria-label="Score canvas">
```

나중에 선택된 셀 정보나 현재 재생 위치는 별도 status text 또는 inspector DOM으로 제공하는 편이 좋다.

## 14. 본 프로젝트의 다음 구현 기준

첫 canvas 전환 작업은 아래 범위까지만 진행한다.

1. `score-area` 안에 canvas stage와 base/note/marker canvas를 만든다.
2. 좌측 `layout-area`는 layout label canvas slot으로 교체한다.
3. CSS로 canvas layer를 겹친다.
4. 작은 inline script 또는 후속 TypeScript 모듈에서 canvas 크기와 DPR을 맞춘다.
5. 임시 grid와 C/D/E note rectangle을 canvas로 다시 그린다.

이 단계에서는 analyzer 결과 연결, audio sync, hit testing, 가상화는 구현하지 않는다.
먼저 canvas layer의 DOM/CSS/초기 draw 구조를 안정화한다.
interaction layer는 hover나 selection 표시가 필요해진 뒤 추가한다.

## 15. 임시 draw 구현 기록

현재 임시 draw는 `index.html` 하단의 작은 inline script로 작성했다.
이는 정식 renderer 모듈이 아니라 canvas layer 구조와 디자인을 확인하기 위한 임시 구현이다.

현재 script가 그리는 항목은 다음이다.

- layout label canvas
  - `BPM`
  - `BeatsPerBar`
  - `StepsPerBeat`
  - `Dynamics`
  - note/gap row label
- base layer
  - 흰 배경
  - global row 배경
  - gap row 배경
  - column grid
  - row grid
- note layer
  - C/D/E 임시 note rectangle
  - note text
- marker layer
  - layout/score 경계 쪽 playback boundary 확인용 red overlay

inline script를 선택한 이유는 다음과 같다.

- 현재 단계는 정식 TypeScript web build 구조를 다시 도입하는 단계가 아니다.
- `index.html`을 직접 열어 canvas 표시를 확인하는 흐름을 유지한다.
- canvas layer의 DOM/CSS/시각 디자인이 안정된 뒤 renderer TypeScript 모듈로 옮기는 편이 낫다.

다만 이 script는 장기 구조가 아니다.
다음 단계에서 실제 score layout과 연결하기 시작하면 아래 모듈로 옮긴다.

```text
src/ui/canvas/
  score_canvas_view.ts
  layout_canvas_view.ts

src/core/render/
  build_score_view_model.ts
```

현재 임시 draw는 row/column 상수를 직접 가진다.

```text
rowHeight = 36
columnWidth = 48
stageWidth = 2400
stageHeight = 1400
```

이 값들은 다음 단계에서 score layout 또는 renderer layout config에서 온 값으로 바뀐다.
