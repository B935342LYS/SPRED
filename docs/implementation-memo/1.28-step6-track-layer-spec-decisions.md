# 1.28. Step6 트랙 레이어 명세 결정 메모

## 1. 목적

이 메모는 track layer 구현 착수 전 확정한 1차 정책을 기록한다.

기준 문서:

- `docs/2.6-track-layer-ui-spec.md`
- `docs/3.1-extension-roadmap.md`

이 문서는 구현 해설용 메모이다.
Track 저장 포맷과 analyzer 계약은 active spec 문서를 따른다.

## 2. 확정된 상태 모델

Track layer 상태는 `activeTrackIds` 하나로 단순화한다.

```ts
type TrackLayerState = {
  activeTrackIds: TrackId[];
};
```

첫 구현에서 active track은 다음 의미를 모두 가진다.

- 편집 대상
- 정상 opacity 표시 대상
- 재생 대상

편집 대상, 표시 대상, 재생 대상을 별도 필드로 분리하지 않는다.
마지막 active track을 끄는 조작은 차단하거나 무시한다.

## 3. 입력 정책

Note row 입력은 모든 active track의 같은 좌표에 batch 적용한다.

```text
pointer hit
-> activeTrackIds 각각에 대해 ScoreSelection 생성
-> ScoreTextEdit[] 생성
-> applyScoreCellRawTextBatch()
```

Global row 입력은 track에 속하지 않으므로 기존처럼 1회만 적용한다.

반복 클릭 cycle은 track별 기존 rawText를 따로 보지 않는다.
사용자의 한 번의 action이 만든 rawText를 모든 active track에 동일하게 덮어쓴다.

따라서 active track 중 일부에 다른 값이 이미 있어도 확인 없이 같은 rawText로 갱신한다.

## 4. Renderer 정책

Inactive track은 숨기지 않고 legacy처럼 반투명하게 표시한다.

- active track alpha: `1`
- inactive track alpha: `0.10`

겹침 표시 우선순위는 다음과 같다.

```text
top    basic
middle optional
bottom extra
```

Canvas draw order는 반대로 `extra -> optional -> basic`이다.

Opacity 정책은 note rectangle뿐 아니라 gliss, vib, trem, tuplet, mute text 등 track에서 나온 복합 표시 전체에 적용한다.

## 5. Audio 정책

Inactive track은 audio schedule event를 만들지 않는다.
즉 inactive track event를 schedule에 넣고 volume만 `0`으로 두는 방식은 첫 구현에서 사용하지 않는다.

이유:

- 현재 scheduler/backend 구조에서는 schedule event가 실제 oscillator/gain node 예약으로 이어진다.
- volume `0` event를 많이 만들면 active 전환 속도 개선보다 event queue와 예약 비용 증가가 먼저 생긴다.
- 무음 oscillator를 나중에 살리는 방식은 track별 gain bus나 dynamic mute automation이 필요해 backend 범위를 넓힌다.

후속 후보:

- 전체 track schedule을 유지하되 scheduler가 active track만 예약한다.
- track별 gain bus를 두고 active/inactive를 gain automation으로 전환한다.

## 6. 재생 중 조작

첫 구현에서는 playback state가 `playing`일 때 active track toggle을 비활성화한다.

정지 상태에서는 active track toggle을 조작할 수 있다.
Edit mode에서도 playback이 `playing`이 아니면 active track toggle을 조작할 수 있다.

Paused 상태는 구현 시 playback controller 상태 보존 비용을 확인한다.
첫 구현에서 paused position 보존과 schedule 재생성이 복잡하면 paused 상태에서도 toggle을 비활성화해도 된다.

## 7. 공용 Track Policy 모듈 위치

Track 순서, inactive alpha, track gain 등 renderer/audio/app이 공유하는 순수 정책은 다음 위치에 둔다.

```text
regression-code/src/track/track_control.ts
```

`src/track/`을 선택한 이유:

- `shared`보다 track domain이 분명하다.
- `core/score`는 저장 포맷 중심 계층이므로 renderer opacity나 audio gain 정책을 넣지 않는다.
- renderer/audio 내부에 각각 track helper를 두면 draw order와 gain 정책이 어긋날 수 있다.
- 후속 부분 갱신에서 track 단위 영향 범위 판단이나 track 단위 실행 정책을 같은 폴더 아래로 확장할 수 있다.

이 모듈은 UI 상태를 소유하지 않는다.
순수 helper만 제공한다.
