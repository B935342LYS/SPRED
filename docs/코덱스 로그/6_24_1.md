`pause 후 start로 점프` 방식은 권장하지 않습니다. 동작은 만들 수 있지만 루프 경계마다 `stopAll -> reset -> playFromSeconds(start)`가 일어나서 짧은 공백, 클릭 노이즈, scheduler lookahead 지연, YouTube sync 흔들림이 생기기 쉽습니다.

구현 방향은 “재생을 멈췄다가 다시 시작”이 아니라 **score time만 loop end에서 start로 wrap하고, end를 넘는 이벤트는 scheduler/backend에 넘기기 전에 잘라내는 방식**이 맞습니다.

핵심은 2곳입니다.

1. **PlaybackController가 loop time을 modulo로 계산**
   - 현재 재생 시간이 `loop.endSeconds`를 넘으면 stopped 처리하지 않고:
     - `loop.startSeconds + ((rawSeconds - loop.startSeconds) % loopDuration)`
   - 동시에 `loopCycleIndex = floor((rawSeconds - loop.startSeconds) / loopDuration)`를 계산해서 scheduler 중복 예약 key에 넘깁니다.
   - UI seek bar/현재 시간도 이 wrapped score time을 기준으로 갱신합니다.

2. **AudioScheduler가 loop end에서 이벤트를 clip**
   - 이벤트가 `loop.endSeconds`를 넘어가면 원본 이벤트를 그대로 예약하면 안 됩니다.
   - 예: 루프가 10s-20s이고 18s-24s note가 있으면, 이번 cycle에서는 18s-20s까지만 예약해야 합니다.
   - 다음 cycle에서는 10s부터 다시 시작하고, 18s-24s 이벤트의 20s-24s 나머지는 재생하지 않습니다. 사용자가 말한 “중간에 끊고 start에서 재시작”이 이 의미입니다.
   - 따라서 scheduler에서 backend 호출 전 `clipScheduleEventToWindow(event, queryStart, queryEnd)` 같은 함수를 둬서 `endSeconds`, `effects`, `automation`, `glissChain.segments`를 window 끝으로 잘라야 합니다.

현재 코드 기준으로는 `audio_scheduler.ts`에 이미 loop window 분할은 들어가 있지만, 시작 이벤트를 예약할 때 `event`를 그대로 넘깁니다. 이러면 end 경계를 넘는 긴 note/gliss가 loop 이후까지 울릴 수 있습니다. 여기에 **window end 기준 event clipping**을 추가해야 합니다.

대략 형태는 이렇습니다.

```ts
const clippedEvent = clipScheduleEventToWindow(
  event,
  window.queryStartSeconds,
  window.queryEndSeconds,
);

if (clippedEvent === null) {
  continue;
}

input.backend.scheduleEvent(
  clippedEvent,
  clippedEvent.startSeconds + window.offsetBaseSeconds,
);
```

그리고 `clipScheduleEventToWindow()`는:
- `event.endSeconds <= windowStart` 또는 `event.startSeconds >= windowEnd`면 `null`
- `startSeconds = max(event.startSeconds, windowStart)`
- `endSeconds = min(event.endSeconds, windowEnd)`
- 중간 진입은 기존 `createResumedScheduleEvent()` 로직 재사용 가능
- 끝 자르기는 automation/effects/segments의 `endSeconds`도 `windowEnd` 이하로 잘라야 함

결론적으로, 루프는 “pause 후 start 점프”가 아니라 **controller는 시간만 wrap, scheduler는 boundary-crossing event를 hard clip**으로 가는 게 맞습니다. 이렇게 해야 end 경계에 걸친 긴 음, gliss, trem/vib effect도 루프 경계에서 정확히 끊기고 다음 cycle이 start에서 새로 시작됩니다.

구현 전 확인할 모호점은 아래 4개입니다.

1. **end 경계 포함 여부**
   - 루프 범위를 `[start, end)`로 보면 될까요?
   - 즉 `end` column/tick에 정확히 시작하는 이벤트는 이번 루프에서 재생하지 않고, wrap 후 start부터 재생합니다.

2. **end를 넘는 긴 이벤트 처리**
   - 긴 note/gliss/vib/trem이 `end`를 넘으면 `end`에서 즉시 끊고, 다음 루프에서 그 이벤트의 남은 부분은 이어서 재생하지 않는 것으로 이해했습니다. 맞습니까?

3. **start보다 앞에서 시작해 start를 걸치는 이벤트**
   - 루프 재생을 시작했을 때 `start` 이전부터 이어져 `start`를 걸치는 이벤트는 현재 코드의 resume clipping처럼 `start` 지점부터 중간 진입 재생하면 될까요?
   - 예: 루프 10s-20s, note 8s-12s이면 루프 시작 시 10s-12s만 재생.

4. **loop on 상태에서 Play 버튼 동작**
   - loop가 켜져 있으면 Play는 항상 loop start에서 시작하는 게 맞습니까?
   - 아니면 현재 seek 위치가 loop 범위 안이면 그 위치부터 시작하고, 범위 밖이면 loop start로 보정할까요?

제안하는 기본값은 `1: [start, end)`, `2: end에서 hard cut`, `3: start에서 중간 진입 재생`, `4: seek 위치가 범위 안이면 거기서 시작, 밖이면 start로 보정`입니다.

> 1. [start, end]입니다. end column에 존재하는 셀이 있으면 그것을 연주합니다.
> 2. end 경계를 넘는 순간, 즉 emd+1에 돌입하는 순간 끊습니다.
> 3. 네. 그렇습니다.
> 4. loop 범위 안이면 그 위치부터 시작합니다.

  - gliss start token을 붙임
- 셀 입력 후 `Bridge`를 자동으로 끌지 여부는 선택 가능한데, 반복 작업이 많다면 유지하는 편이 좋습니다.

제 판단으로는 **2번 Mode 방식**이 가장 낫습니다. UI가 많이 늘지 않고, 사용자의 실제 작업 단위인 “글리산도 연결점 만들기”를 직접 표현합니다. 구현도 parser/analyzer를 건드릴 필요 없이 edit rawText composition 계층에서 처리할 수 있습니다.

먼저 작게 간다면:
- Gliss 패널에 `Bridge` toggle 추가
- `Bridge + start`일 때만 `- + gliss start` 자동 합성
- 기존 Default/Long 값은 변경하지 않음

이렇게 하면 “세 패널을 조작하고 다시 되돌리는 문제”는 대부분 사라집니다.

> 단축키를 두는 것이로군요. 그런데 gliss에서 "-"의 입력까지 담당하게 만들 수 있나요? 경계가 흐려지지는 않습니까?

가능은 합니다. 다만 말씀대로 **모듈/패널 의미 경계가 흐려질 위험**이 있습니다.

정리하면:

- `Long` 패널의 `-`는 “이 셀이 앞 음을 이어받는 hold cell인가”를 정하는 일반 기능입니다.
- `Gliss` 패널의 `S/M/E`는 “이 셀이 gliss anchor인가”를 정하는 기능입니다.
- `Bridge Start`는 사실상 두 기능을 조합한 **작업 매크로**입니다.

그래서 `Gliss`가 `-` 자체를 담당한다고 해석하면 경계가 흐립니다. 대신 UX상으로는 이렇게 정의하면 안전합니다.

**Gliss 패널이 hold 기능을 소유하지 않는다.**
**다만 Gliss 패널은 자주 쓰는 입력 조합을 실행하는 shortcut을 제공한다.**

즉 내부 구현/개념은:
- `Long`의 의미는 그대로 유지
- `Gliss`의 의미도 그대로 유지
- `Bridge Start`는 `rawText` 합성 시 `holdToken = "-"`를 임시로 주입하는 shortcut

사용자에게 보이는 UI 문구도 `Bridge Start`보다 더 명확하게:
- `Start after hold`
- `Hold + Start`
- `- + S`
같은 식이 낫습니다.

제가 추천하는 방식은 **Gliss 패널 안에 “shortcut row”를 두는 것**입니다.

예:
- Gliss kind: `(none) / start / curve / end`
- id: `a`
- shortcut button: `- + Start`

동작:
- `- + Start` 버튼/토글이 켜진 상태에서 셀 클릭
- 입력 rawText는 `-`와 gliss start token을 함께 생성
- 하지만 Long 패널의 현재 값은 바꾸지 않음
- 클릭 후 shortcut은 유지 또는 1회성으로 선택 가능

경계 관점에서 더 엄격하게 하려면 버튼 이름을 **“Preset”** 또는 **“Combo”**로 표시할 수 있습니다.

예:
- `Combo: - + S`
- `Preset: Hold Start`

이렇게 하면 “Gliss가 hold를 담당한다”가 아니라 “Gliss 작업에서 자주 쓰는 조합을 빠르게 넣는다”로 해석됩니다.

개인적으로는 `Gliss` 패널에 `Combo -+S` 토글 하나를 두는 정도는 괜찮다고 봅니다. parser/analyzer 계약을 바꾸지 않고, edit composer에서만 rawText 조합을 보조하는 UX 계층 기능이기 때문입니다.

> 그렇기는 합니다. 그렇다면 "-@g(a,S)"를 입력하는 단축 프리셋을 Gliss 패널의 드롭다운에 추가할까요?

네, 그 방식이 가장 깔끔합니다. 별도 버튼을 늘리는 것보다 **Gliss 패널의 kind 드롭다운에 프리셋 항목을 추가**하는 편이 현재 UI 구조에 잘 맞습니다.

예를 들면:

```text
(none)
start
curve
end
hold start
```

내부 값은 이런 식이 좋습니다.

```ts
""          // none
"S"         // start
"M"         // curve
"E"         // end
"holdStart" // -@g(a,S) preset
```

동작은:

- `hold start` 선택 시 rawText 합성에서 `-`와 `@g(id,S)`를 함께 넣음
- `id`는 기존 gliss id select 값을 그대로 사용
- Long 패널의 hold 선택값은 변경하지 않음
- Default 패널 값도 변경하지 않음
- 저장되는 rawText만 `-@g(a,S)` 형태로 생성

이렇게 하면 의미 경계도 괜찮습니다. Gliss 드롭다운에 들어가긴 하지만 “gliss start를 위한 shortcut preset”으로 해석할 수 있고, Long 패널 상태를 직접 바꾸지 않으니 패널 간 상태 오염도 없습니다.

표시 이름은 `hold start`보다 조금 더 명확하게는 `- + start`가 좋겠습니다. 실제 생성 결과와 거의 1:1이라 사용자가 예측하기 쉽습니다.

> 다만 이 경우 Default의 드롭다운은 어떻게 되나요? 사용자가 기존에 AUTO#로 두고 있었다면 그대로 유지되면서 동시에 입력 내용에서는 제외되는 건가요?

네. 제가 권하는 동작은 **Default 드롭다운 상태는 그대로 유지하되, `- + start` 프리셋 입력 1회에 한해서 Default 출력을 무시**하는 방식입니다.

즉 사용자가 Default를 `AUTO♯`로 둔 상태에서 Gliss kind를 `- + start`로 선택하고 셀을 클릭하면:

- UI 상태:
  - Default 드롭다운: `AUTO♯` 그대로
  - Long 드롭다운: 기존 값 그대로
  - Gliss 드롭다운: `- + start`
  - Gliss id: 예를 들어 `a`

- 실제 입력 rawText:
  - `-@g(a,S)`
  - AUTO♯로 계산된 note text는 넣지 않음
  - Long 패널 값도 별도로 참조하지 않음

이렇게 해야 “프리셋”의 의미가 분명합니다. `- + start`는 기존 세 패널 상태를 조합하는 게 아니라, **그 셀에 넣을 rawText 조합을 직접 지정하는 shortcut**입니다.

구현상으로는 `composeEditRawText()`나 그 근처에서:

```ts
if (input.gliss.kind === "holdStart") {
  return `-@g(${input.gliss.id},S)`;
}
```

처럼 최우선 분기로 처리하면 됩니다.

다만 한 가지 UX 표시는 필요합니다. 사용자가 Default가 켜져 있는데 왜 음이 안 들어갔는지 헷갈릴 수 있으니, current rawText preview에 반드시 실제 결과가 보여야 합니다.

예:
```text
current: -@g(a,S)
```

그러면 “AUTO♯가 선택되어 있어도 이번 프리셋은 이 rawText를 넣는구나”가 명확해집니다.

> 1회보다는 "-" + start 드롭다운이 유지되는 동안은 계속 "-@g(a,S)" 입력이 들어가게 하는 것도 좋겠습니다.

그게 더 좋습니다. 반복 작업용 프리셋이 목적이니 **드롭다운이 `- + start`로 유지되는 동안 계속 `-@g(a,S)`가 입력**되게 하는 편이 맞습니다.

정책을 이렇게 잡으면 됩니다.

- Gliss kind dropdown에 `- + start` 추가
- 선택값이 `holdStart`인 동안:
  - 실제 입력 rawText는 항상 `-@g(id,S)`
  - Default 패널 상태는 그대로 보존하지만 입력에는 반영하지 않음
  - Long 패널 상태도 그대로 보존하지만 입력에는 반영하지 않음
  - Gliss id 변경은 즉시 반영됨
  - current preview는 `current: -@g(a,S)`처럼 표시
- 사용자가 Gliss kind를 `start/curve/end/(none)`으로 바꾸면 기존 조합 방식으로 복귀

이 방식이면 경계도 크게 흐려지지 않습니다. `holdStart`는 Long 상태를 바꾸는 기능이 아니라, Gliss 입력의 반복 프리셋으로 동작합니다.
구현은 edit rawText compose 단계에서 `glissKind === "holdStart"`를 최우선 처리하면 충분합니다.

> 다음은 성능 개선 작업입니다. 들어가기 앞서 태블릿에서 만든 악보를 실행해봤는데, 유튜브 영상의 싱크가 데스크탑이나 노트북이랑 다릅니다. 이유가 무엇일까요?

가능성이 큰 원인은 **YouTube iframe과 Web Audio/score playback이 같은 clock을 쓰지 않기 때문**입니다.

현재 구조는 대략 이렇습니다.

- 악보 재생: Web Audio `AudioContext.currentTime` 기준
- 악보 스크롤/seek: 앱의 score seconds 기준
- YouTube: iframe player의 내부 재생 clock 기준
- 앱은 YouTube를 “follower”로 두고, offset을 적용해서 가끔 seek 보정

데스크탑에서는 이 차이가 작게 느껴지지만, 태블릿에서는 다음 요인 때문에 싱크가 달라질 수 있습니다.

1. **YouTube iframe 재생 지연**
   - 태블릿 브라우저는 iframe video start/pause/seek 반응이 데스크탑보다 늦을 수 있습니다.
   - 특히 iPad Safari/Chrome은 실제로는 WebKit 기반이라 media 재생 정책과 iframe 타이밍이 다릅니다.

2. **영상 디코딩/버퍼링 지연**
   - 같은 `seekTo(10.0)`을 호출해도 실제 화면/소리가 10.0초에서 즉시 시작하지 않습니다.
   - 기기 성능, 네트워크, YouTube 화질, 하드웨어 디코더 상태에 따라 지연이 달라집니다.

3. **브라우저 timer와 background throttling**
   - 현재 YouTube drift check가 interval 기반이면, 태블릿에서 timer 해상도나 실행 주기가 흔들릴 수 있습니다.
   - 화면 터치, 스크롤, 주소창 변화, 절전 모드에서도 타이밍이 달라질 수 있습니다.

4. **오디오 출력 경로 차이**
   - 태블릿 스피커, 블루투스 이어폰, AirPods 같은 무선 출력은 추가 latency가 큽니다.
   - 이 경우 Web Audio와 YouTube가 둘 다 늦어지는 것이 아니라, 출력 경로나 브라우저 내부 mixing 방식에 따라 체감 offset이 달라질 수 있습니다.

5. **YouTube offset이 기기 공통값이 아닐 가능성**
   - 현재 `musicData.youtube.offsetMs`는 score 데이터에 저장되는 단일 값입니다.
   - 하지만 실제로는 “영상 자체 offset” + “기기/브라우저 재생 latency”가 섞입니다.
   - 그래서 데스크탑에서 맞춘 offset이 태블릿에서 그대로 맞지 않을 수 있습니다.

제 판단으로는 가장 가능성이 큰 것은 **YouTube iframe seek/play latency가 기기별로 다르고, 현재 offset이 전역 score 값 하나라서 태블릿 보정값을 따로 담지 못하는 문제**입니다.

해결 방향은 두 단계가 좋아 보입니다.

1. **진단**
   - 태블릿에서 같은 영상으로 offset을 다시 맞췄을 때 몇 ms 정도 차이가 나는지 기록
   - 블루투스 없이 내장 스피커로 테스트

네, 진단은 거의 맞습니다. 현재 구조상 글리산도는 별도 `gliss`/`glissChain` schedule event로 만들어지고, 앵커 셀의 `NoteEvent`도 별도 schedule event로 남을 수 있습니다. 그러면 앵커 지점에서 같은 pitch가 **note oscillator + gliss oscillator**로 겹쳐서 “두 번 울림” 또는 “강조된 attack”처럼 들립니다.

다만 말씀대로 단순히 앵커 note를 mute하는 건 좋지 않습니다. 특히 롱노트에서:

```text
C4 - - @g(a,S) ...
```

같은 경우, gliss 시작 전까지의 롱노트 발음은 반드시 필요합니다. 없애야 하는 것은 “노트 전체”가 아니라 **gliss와 겹치는 아주 짧은 접점/중복 구간**입니다.

현실적인 개선 방향은 이렇습니다.

1. **note event를 제거하지 않는다**
   - 롱노트의 본래 발음은 유지합니다.

2. **gliss 시작 anchor에서는 note를 gliss 시작 직전까지만 자연스럽게 fade out**
   - 지금도 schedule builder에 `buildGlissStartNoteClipEndTickMap()` 비슷한 clipping 경로가 있습니다.
   - 다만 hard cut이면 클릭/재어택 느낌이 생길 수 있으므로, backend envelope에서 짧은 release/crossfade가 자연스럽게 먹는지 봐야 합니다.

3. **gliss event는 시작점에서 강한 attack을 만들지 않도록 fade in**
   - `AudioGlissScheduleEvent.crossfadeSeconds`가 이미 있으므로, fallback gliss의 시작 gain을 `0 -> target`으로 올리는 방향은 맞습니다.
   - 문제는 crossfade 시간이 너무 짧거나, note 쪽과 합산되어 체감 attack이 남는 경우입니다.

4. **앵커 중복 구간에만 note 또는 gliss gain ducking 적용**
   - note 전체 mute가 아니라, anchor 근처 예를 들어 10-30ms만 gain을 낮추는 방식입니다.
   - 시작 anchor: note가 빠르게 내려가고 gliss가 올라감
   - 종료 anchor: gliss가 내려가고 도착 note가 올라감
   - 이러면 롱노트는 보존하면서 “두 번 울림”만 줄일 수 있습니다.

제일 안전한 구현 후보는 **gliss fallback/chain과 anchor note 사이에 짧은 crossfade 정책을 명시적으로 넣는 것**입니다. 즉 audio schedule 단계에서 앵커 note를 삭제하지 않고, gliss 접점에 걸친 note의 `gainScale` 또는 별도 `gainRamp`를 추가하는 방식입니다.

우선 확인할 지점은:
- 단일 gliss fallback에서 start anchor note가 실제로 어디까지 schedule되는지
- `crossfadeSeconds = 0.02`가 충분한지
- start anchor note와 gliss event가 같은 시점에 둘 다 attack envelope을 갖는지

수정한다면 “앵커 note mute”보다 **anchor crossfade/ducking** 쪽으로 가는 게 맞습니다.

검토 결과, 방향은 맞지만 비용은 중간 이상입니다. 단순한 “앵커 note mute”가 아니라 오디오 모델을 `note + gliss`에서 `note/gliss를 모두 segment chain으로 합성`하는 쪽으로 조금 옮기는 작업입니다.

핵심 판단은 이렇습니다.

`anchor가 단일 셀인 경우 note event 묵음`은 도입할 만합니다. 단일 셀의 `@g(a,S)`나 `@g(a,E)`는 독립 note attack이 실제로 중복 발음의 원인이므로, audio schedule 단계에서 해당 note를 빼는 정책이 자연스럽습니다. renderer/analyzer는 유지하고 audio schedule에서만 처리하면 됩니다.

`롱노트와 연결된 start/end를 gliss chain으로 포함`하는 정책도 음악적으로는 더 맞습니다. 예를 들어:

```text
- - - @g(a,S) ... @g(a,E) - - -
```

같은 경우, 시작 전 롱노트는 같은 pitch의 constant segment, 실제 gliss는 pitch ramp segment, 도착 후 롱노트도 같은 pitch의 constant segment로 보는 방식입니다. 즉 내부적으로는:

```text
constant pitch -> gliss ramp -> constant pitch
```

가 됩니다. 이러면 anchor note를 따로 울릴 필요가 없어져 중복 발음이 사라집니다.

다만 비용이 생기는 지점은 `비브라토/트레몰로 보존`입니다. 현재 구조상 note event의 vibrato는 note backend에서만 처리되고, gliss/glissChain 쪽은 주로 tremolo 일부만 이어받는 구조입니다. 롱노트를 gliss chain으로 흡수하면, 그 롱노트에 있던 vib/trem 효과를 chain segment에도 적용할 수 있어야 합니다. 특히 vibrato는 현재 gliss backend에 적용 경로가 없어서 추가 구현이 필요합니다.

도입 비용을 나누면:

1. 낮은 비용: 단일 셀 anchor note mute
   `audio_schedule_builder.ts`에서 gliss start/end anchor에 해당하고 롱노트 병합이 없는 짧은 note를 schedule에서 제외하면 됩니다.

2. 중간 비용: 롱노트를 constant-pitch gliss chain segment로 흡수
   기존 `buildConnectedGlissChains()`를 확장하거나, audio 전용으로 `buildGlissPlaybackChains()` 같은 별도 경로를 만드는 편이 좋습니다. analyzer event 자체를 바꾸기보다는 audio schedule builder 내부 정책으로 두는 것이 안전합니다.

3. 중간-높은 비용: vib/trem 효과 보존
   `AudioGlissChainScheduleEvent`가 segment별 또는 구간별 effect를 받을 수 있어야 하고, backend에서 glissChain에도 vibrato/tremolo를 적용해야 합니다. tremolo는 기존 구조를 확장하면 되지만, vibrato는 gliss frequency ramp와 detune LFO를 같이 써야 하므로 테스트가 필요합니다.

제 의견은 “도입할 만하지만 한 번에 다 하지 않는 것”입니다.

추천 순서:
1. 직전 `MIN_NOTE_SECONDS` 제거 정책은 되돌립니다.
2. 먼저 단일 셀 anchor note mute만 구현해서 중복 발음이 줄어드는지 확인합니다.
3. 그 다음 롱노트 흡수형 gliss chain을 추가합니다.
4. 마지막으로 gliss/glissChain 효과 보존, 특히 vibrato 적용을 확장합니다.

이 정책의 장점은 명확합니다. “겹치는 두 발음의 gain을 조절한다”가 아니라 “음악적으로 하나의 연결된 발음원으로 재해석한다”는 방식이라, 근본 원인에 더 가깝습니다. 다만 기존 오디오 스케줄 구조를 조금 키우는 작업이라 테스트 케이스를 충분히 붙여야 합니다.

> 글리산도의 시적 시점은 anchor의 startTick과 동일하죠?

네. 현재 구현상 **글리산도의 실제 재생 시작 시점은 `startAnchorTick`**입니다.

정확히는:

- analyzer가 `GlissEvent.startAnchorTick`을 만듭니다.
- audio schedule은 `createAudioGlissScheduleEvent()`에서 그 값을 `startSeconds`로 변환합니다.
- 단독 gliss fallback은 `startAnchorTick -> endAnchorTick` 구간을 재생합니다.
- glissChain도 각 ramp segment는 `GlissEvent.startAnchorTick -> endAnchorTick`을 그대로 segment로 씁니다.

다만 anchor의 `startTick` 자체와는 약간 다릅니다. 일반 셀 anchor의 경우 `createGlissAnchorTick()`이 셀의 **중심 tick**을 씁니다. 예를 들어 1칸짜리 셀이 `[2, 3)`이면 gliss anchor tick은 `2.5`입니다.

tuplet slot은 예외가 있습니다:
- slot 길이가 1 tick보다 짧으면 slot 중심 tick
- 아니면 slot start + 0.5 tick

그래서 “gliss 시작 시점 = anchor cell의 시작 tick”은 아니고, 현재 기준으로는 **gliss 시작 시점 = analyzer가 계산한 anchor 기준 tick**, 일반 셀에서는 **셀 중심 tick**입니다.

> 단일 셀 anchor에서 시작되는 gliss는 셀 시작 tick으로 맞추는게 좋겠습니다.

> 다음은 배음에 대한 UX 개선입니다. AbsolutePitch로 현에서 다른 음이 울리게 함으로서 배음(하모닉스)을 구현하고 있습니다. 하지만 하모닉스가 일정 패턴을 지닌다면 AUTO 옵션으로 자동화할수도 있어 보입니다. 예를 들어 AUTO HARMONICS 버튼을 누르고 특정 행을 선택하면 그 행에 기록된 셀에서는 1~5배음까지 동시에 재생되도록 하는 것이죠.

가능합니다. 다만 이건 Tone.js보다 훨씬 실용적인 확장 후보입니다. 현재 구조에서도 비교적 자연스럽게 들어갑니다.

핵심은 “표시 pitch”와 “실제 발음 pitch”를 분리하는 기존 `AbsolutePitch @p(...)` 개념을, 단일 pitch override가 아니라 **harmonic voice expansion**으로 확장하는 겁니다.

예상 설계는 이렇습니다.

```text
사용자 선택:
AUTO HARMONICS ON
→ 특정 note row 선택

그 row의 note cell:
C4
→ audio schedule에서 C4 단음 대신
   C4 + 2배음 + 3배음 + 4배음 + 5배음
```

다만 여기서 “1~5배음”은 MIDI semitone으로 단순히 `+12, +19, +24, +28`처럼 근사해야 합니다. 실제 배음은 정수배 주파수이므로 3배음, 5배음은 평균율과 약간 다릅니다. 현재 오디오 백엔드가 `midi + centOffset`을 지원하므로 더 정확하게 할 수 있습니다.

배음 변환 예시:

| 배음 | 주파수 비 | 평균율 근사 | cent 보정 |
|---|---:|---:|---:|
| 1 | 1x | 원음 | 0 |
| 2 | 2x | +12 semitone | 0 |
| 3 | 3x | +19 semitone | +1.96 cents |
| 4 | 4x | +24 semitone | 0 |
| 5 | 5x | +28 semitone | -13.69 cents |

구현 방식은 세 가지가 있습니다.

1. 저장 포맷에 명시 modifier 추가
   예: `@h(5)` 또는 `@harm(1-5)`
   장점: 저장/공유 시 재현 가능
   단점: parser/analyzer/UI 문법 추가 필요

2. 트랙/행 단위 playback overlay로 처리
   선택한 rowId를 앱 설정에 두고, audio schedule build 때 해당 row note를 여러 schedule event로 확장
   장점: 빠르게 구현 가능, 기존 악보 rawText를 안 바꿈
   단점: JSON 저장/공유 정책을 정해야 함

3. 편집 보조 기능으로 rawText를 자동 입력
   버튼을 누르고 row를 선택하면 해당 행의 셀에 `@p(...)`류를 실제로 써 넣음
   장점: 현재 parser와 호환 쉬움
   단점: 여러 배음을 한 셀에서 표현하는 문법이 아직 없음

제안은 2번입니다. 우선 **행 단위 playback overlay**로 시작하는 게 좋습니다.

초기 UX:
- `AUTO HARMONICS` 토글 버튼
- 켠 상태에서 note row 클릭
- 선택된 row는 renderer에서 얇은 표시 또는 상태 텍스트로 표시
- 해당 row의 note events는 audio schedule에서 1~5배음으로 확장
- 저장은 나중에 결정

구현 난이도는 중간입니다. parser는 건드리지 않아도 되고, analyzer도 거의 유지 가능합니다. 대신 `audio_schedule_builder` 입력에 harmonic row 설정을 전달하거나, app state에 harmonic playback setting을 두는 확장이 필요합니다.

주의할 점:
- 1~5배음을 모두 같은 volume으로 울리면 너무 큽니다. 배음별 gain을 줄여야 합니다.
- trem/vib/gliss가 있는 note에서 배음도 같이 따라갈지 정책이 필요합니다.
- 하모닉스 행 선택이 트랙별인지 전체 rowId 기준인지 정해야 합니다.
- 저장할지, 세션 옵션으로만 둘지 정해야 합니다.

현 단계 추천 MVP:
- 단일 선택 row
- playback 전용
- 1~5배음 고정
- 배음 gain은 예: `[1, 0.45, 0.3, 0.22, 0.18]`
- gliss에는 일단 적용하지 않고 note event에만 적용
- 저장은 하지 않고 UI 상태로만 유지

이 정도면 UX 개선으로 체감이 크고, 리스크는 제한적입니다.

> AbsolutePitch 토큰의 특수 패턴으로서 인식하게 하면 어떻습니까? "@p(h)"로 하고 이 경우 배음 노트로 인식하여 배음까지 발음하게끔요.
> 하지만 구현이 복잡할 것 같네요. 비용에 비해 효용이 클지 모르겠습니다.

`@p(h)`를 특수 패턴으로 쓰는 건 설계상 꽤 괜찮습니다. 기존 `AbsolutePitch`의 의미가 “표시 위치와 실제 발음 음을 분리한다”이므로, `h`를 “이 row의 기본 pitch를 기준으로 harmonic voicing을 만든다”로 확장하는 건 문법적으로도 크게 어색하지 않습니다.

다만 말씀대로 비용은 중간 이상입니다. 단순 UI 기능이 아니라 parser/analyzer/audio 계약을 모두 조금씩 건드립니다.

필요 변경 범위:

- parser
  현재 `@p(...)`는 숫자 MIDI pitch로 검증될 가능성이 큽니다. `h`를 허용하려면 `absolutePitch` 타입을 `number | "harmonics"` 같은 식으로 넓혀야 합니다.

- analyzer
  `@p(h)`일 때 `sound.midi`를 단일 pitch로 확정하지 않고, “기준 pitch + harmonic expansion”이라는 정보를 보존해야 합니다. 단순히 `sound.midi` 하나에 넣기 어렵습니다.

- audio schedule
  `NoteEvent` 하나를 여러 `AudioNoteScheduleEvent`로 확장해야 합니다. 1~5배음 각각에 MIDI/cent/gain을 부여해야 합니다.

- renderer
  표시는 기존 row pitch 그대로 둘지, `@p(h)` 셀을 특수 표시할지 정해야 합니다. 최소 구현이면 renderer는 건드리지 않아도 됩니다.

- edit UI
  AbsolutePitch 드롭다운에 `Harmonics` 옵션을 추가할 수 있습니다. 직접 `@p(h)` 입력도 허용해야 합니다.

효용은 꽤 명확합니다.

장점:
- 악보 JSON에 저장됩니다.
- 다른 사용자/기기에서도 같은 소리가 재현됩니다.
- `AUTO HARMONICS` 같은 별도 세션 상태보다 명확합니다.
- 셀 단위로 필요한 곳에만 적용할 수 있습니다.
- 기존 `@p(...)` 개념과 연결됩니다.

단점:
- “행 전체 자동 적용” UX보다는 셀마다 토큰이 필요합니다.
- `@p`가 단일 pitch override에서 “pitch override 또는 voicing mode”로 의미가 넓어집니다.
- 나중에 `@p(60)`과 `@p(h)` 타입 분기가 계속 따라다닙니다.

비용 대비로 보면, **하모닉스를 실제 악보 기능으로 오래 쓸 생각이면 도입 가치가 있습니다.** 반대로 “가끔 편하게 듣기 위한 실험”이면 세션 overlay가 더 싸고 빠릅니다.

제가 추천하는 절충안은 이겁니다.

1. 먼저 `@p(h)` 문법만 parser/analyzer/audio에 최소 도입
2. UI는 당장은 직접 입력만 허용하거나 AbsolutePitch 드롭다운에 `Harmonics` 하나만 추가
3. 적용 범위는 note event만
4. gliss/trem/vib와의 복잡한 상호작용은 보수적으로 처리
   - trem/vib는 배음 note들에도 복제
   - gliss는 일단 harmonic expansion 제외 또는 start pitch 기준 고정 배음
5. 1~5배음과 gain table은 고정

이 정도면 “새 시스템”이라기보다 `@p`의 작은 확장으로 끝낼 수 있습니다. 구현 비용은 parser/analyzer/audio/edit UI까지 가므로 작지는 않지만, 기능 자체는 깨끗합니다.

`@p(1005)` 방식은 저는 비추천입니다. 이유는 명확합니다.

현재 명세상 `@p(midi_num)`은 `0~127` MIDI 정수입니다. `1005`를 “5배음”으로 쓰면 숫자인데 pitch가 아닌 예외값이 됩니다. 구현은 가능하지만, 나중에 코드를 읽을 때 `1005`가 잘못 입력된 MIDI 값인지, 의도된 하모닉스인지 계속 특별 지식이 필요합니다.

즉 이런 분기가 됩니다.

```ts
if (midiNum >= 1001 && midiNum <= 1005) {
  // harmonics sentinel
} else {
  // real midi pitch
}
```

이건 비용이 적어 보이지만 의미가 불투명합니다. 사용자 입장에서도 `@p(1005)`가 “5배음”이라는 건 기억하기 어렵습니다.

차라리 `@p(h)`가 낫습니다. 이쪽은 이례적이긴 해도 의미가 드러납니다.

```text
@p(64)  -> absolute pitch MIDI 64
@p(h)   -> harmonic voicing
```

만약 배음 수를 지정하고 싶다면 확장 가능하게:

```text
@p(h)   -> 기본 1~5배음
@p(h5)  -> 1~5배음
@p(h3)  -> 1~3배음
```

정도가 낫습니다. `@p(h)`만 먼저 넣고, `h3`, `h5`는 나중에 열어도 됩니다.

구현은 이렇게 잡는 게 안전합니다.

1. branch 생성
2. parser의 `ParsedAbsolutePitch`를 union으로 변경
   - `{ kind: "midi"; midiNum: number }`
   - `{ kind: "harmonics"; harmonicCount: 5 }`
3. analyzer의 `NoteEvent`에 단일 `sound`는 유지하되, 별도 `voicing` 또는 `harmonics` 정보를 추가
4. audio schedule에서 note event를 1~5개 schedule event로 확장
5. edit UI의 AbsolutePitch 드롭다운에 `Harmonics` 옵션 추가
6. gliss에는 처음에는 적용하지 않거나, note event에만 제한

`@p(1005)`는 단기 구현량은 조금 줄 수 있지만, 장기 유지보수 비용이 더 큽니다. 이 프로그램이 현악기 전용이라는 명분이 있다면 오히려 `@p(h)`처럼 도메인 의미를 드러내는 쪽이 더 낫습니다.

> 음악 이론적으로 하모닉스의 배음이 들리는 방식이 현재 저희의 구현인 "5배음까지 동시에 오실레이터로 재생"과 맞는 편인가요?

음악 이론적으로는 현재 구현이 “정확한 하모닉스 주법 모델”이라고 보기는 어렵습니다.

현악기 하모닉스는 보통 **1~5배음이 같은 세기로 동시에 울리는 코드**가 아니라, 손가락으로 특정 node를 만들어 **특정 배음 하나가 주된 sounding pitch로 부각**되는 주법입니다. 예를 들어 4배음 artificial harmonic이면 실제로는 기준음의 4배 주파수, 즉 두 옥타브 위 음이 주된 소리로 납니다. 물론 실제 악기 음색 안에는 그 소리의 상위 partial들이 섞이지만, “기준음 + 2배음 + 3배음 + 4배음 + 5배음”을 모두 독립 oscillator로 동시에 치는 것과는 다릅니다.

그래서 현재 구현은 이론적으로는 다음에 가깝습니다.

- 정확한 하모닉스 주법: 아님
- 배음열을 덧입히는 additive synth 효과: 맞음
- “현악기 전용 프로그램에서 하모닉스 느낌을 빠르게 내는 실용적 효과”: 어느 정도 가능

특히 glissando에서도 마찬가지입니다. 실제 harmonic glissando는 특정 harmonic touch/partial이 이동하면서 고음 배음 계열이 들리는 느낌이지, 1~5 partial gliss가 모두 병렬로 움직이는 것이 기본은 아닙니다. 현재 구현은 더 풍성하고 합성음적인 “spectral gliss”에 가깝습니다.

제가 보기엔 방향을 둘 중 하나로 정해야 합니다.

1. 현재 구현 유지
   `AUTO◇`를 “하모닉스 느낌의 배음 보강 효과”로 취급합니다. 소리는 풍성하지만 실제 주법과는 다소 다릅니다.

2. 더 이론적인 모델로 수정
   `@p(h4)`처럼 특정 partial을 지정하고, 그 partial 하나를 주된 pitch로 발음합니다. 예: `h2`, `h3`, `h4`, `h5`.
   이 경우 `AUTO◇` 기본값은 현악 artificial harmonic에서 흔한 `h4`로 둘 수 있습니다.

실제 주법 구현을 목표로 한다면 2번이 더 맞습니다. 현재의 1~5배음 동시 재생은 실험 효과로는 괜찮지만, “하모닉스 주법”이라고 부르기에는 과장된 모델입니다.

> 2번에서 단일 음만 재생하여도 충분히 하모닉스로서의 의미가 사용자에게 전달될 수 있나요?
> 그렇다면 @p(h) 특수 토큰 자체가 필요없어질 수도 있겠습니다. AUTO 선택시 내부적으로 @p(n) 내부의 숫자 n에 +7, +5 등만 더해서 적용하면 되니까요.

네. **단일 음만 재생해도 하모닉스로서의 의미는 충분히 전달될 수 있습니다.** 실제 현악 하모닉스에서 연주자가 주로 인식하고 기보하는 것은 “배음열 전체”가 아니라 **울려 나오는 특정 높은 음**입니다. 음색 차이는 실제 악기에서는 중요하지만, 이 앱의 현재 oscillator 기반 음원에서는 “어떤 높은 pitch가 나는가”가 사용자에게 더 직접적으로 전달됩니다.

다만 `@p(n)`에 단순히 `+7`, `+5`를 더하는 방식은 조심해야 합니다. 배음 간격은 반음 단위로 정확히 떨어지지 않는 경우가 있습니다.

대표 partial 기준:
- 2배음: `+12` semitones
- 3배음: 약 `+19.02` semitones
- 4배음: `+24` semitones
- 5배음: 약 `+27.86` semitones

즉 3배음과 5배음은 `@m(...)` cent 보정까지 같이 있어야 이론적으로 맞습니다. 단순 `+7`, `+5`는 “기존 음에서 완전5도/완전4도 위” 같은 음악적 편의 입력으로는 쓸 수 있지만, harmonic partial 자체와는 다릅니다.

제안은 이렇습니다.

- `AUTO◇`는 UI 프리셋으로 유지합니다.
- 저장 형식은 특수 토큰 없이 기존 토큰 조합으로 만듭니다.
- 예를 들어 기본값을 4배음으로 정하면:
  - row MIDI 기준 `@p(rowMidi + 24)`
  - `@m(0)`은 생략
- 3배음/5배음까지 옵션화한다면:
  - 3배음: `@p(rowMidi + 19)@m(2.0)` 근사
  - 5배음: `@p(rowMidi + 28)@m(-13.7)` 근사

이러면 parser/analyzer/audio에 `@p(h)`라는 예외가 필요 없어지고, 현재 구조의 `@p(n)`/`@m(c)`만으로 하모닉스 sounding pitch를 표현할 수 있습니다. 렌더러의 마름모 표시는 별도 문제가 남는데, 저장 토큰으로 구분하려면 `@p(n)`만으로는 “사용자가 하모닉스 의도로 만든 absolutePitch인지” 알 수 없습니다.

그래서 선택지는 두 가지입니다.

1. **소리만 자동화**
   `AUTO◇`는 `@p(n)`/`@m(c)`를 자동 입력하는 편의 버튼입니다.
   장점: 저장 문법 단순.
   단점: 다시 열었을 때 renderer가 마름모를 자동 표시할 근거가 없음.

2. **의도 표시용 토큰 유지**
   `@p(h4)` 같은 토큰은 “하모닉스 표시와 sounding pitch 계산”을 위한 의미 토큰으로 둡니다.
   장점: 마름모 표시, gliss 하모닉스, 향후 partial 선택이 명확.
