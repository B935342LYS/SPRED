## 2. 용어와 위치

본 프로젝트에서 `노래방 모드`와 `게임 모드`는 같은 기능을 가리킨다.

사용자에게 보이는 이름은 후속 UI 문구에서 정하되, 내부 문서에서는 다음처럼 구분한다.

- `game mode` : 내부 앱 상태, 모듈명, 런타임 상태명 후보
- `practice mode` : 사용자에게 표시하는 기본 UI 문구
- `karaoke mode` : 기능 설명용 별칭

확정 결정:

- 내부 구현명은 `game` 또는 `gameMode`로 둔다.
- UI 버튼 문구는 `practice mode`를 기본으로 한다.
- 종료 버튼 문구는 초안의 `exit practice mode`를 우선 후보로 둔다.

## 3. 핵심 원칙

노래방 / 게임 모드는 기존 parser, analyzer, renderer, audio generator의 공식 의미를 바꾸지 않는다.

기본 흐름은 다음과 같다.

```text
ScoreFile
  -> parser
  -> analyzer
  -> AnalysisResult
  -> Game target events
  -> Microphone pitch frames
  -> Judge results / score runtime
  -> Game overlay UI
```

원칙:

- 판정 기준의 진실 원천은 score JSON rawText가 아니라 `AnalysisResult`이다.
- 게임 모드는 `rawText`를 다시 해석하지 않는다.
- pitch detection 결과, 판정 상태, 점수, 랭크는 score JSON에 저장하지 않는 런타임 데이터이다.
- score load, stop, seek, jump, edit, layout apply처럼 시간축 또는 악보 구조를 불연속으로 바꾸는 동작은 게임 세션을 종료하거나 초기화한다.
- GitHub Pages 정적 배포를 전제로 하며, 서버 저장소나 음원 업로드 기능을 요구하지 않는다.

## 5. UI 범위

### 5.1. 진입 버튼

플레이어 영역 우하단에 `practice mode` 버튼을 둔다.

첫 구현 정책:

- 브라우저가 `navigator.mediaDevices.getUserMedia`를 지원하지 않으면 버튼을 disabled 처리한다.
- 실제 마이크 권한과 장치 사용 가능 여부는 버튼 클릭 후 권한 요청 결과로 판단한다.
- 마이크 권한이 거부되면 게임 모드로 진입하지 않고 상태 메시지를 표시한다.
- 마이크 권한이 허용되면 `AppState` 또는 그 하위 runtime state를 `game` 상태로 전환한다.

주의:

- 페이지 로드 시점에 “마이크가 있다 / 없다”를 완전히 확정하기 어렵다.
- 브라우저 권한 정책상 실제 장치 목록은 권한 허용 전 제한될 수 있다.
- 따라서 “마이크가 없으면 버튼 비활성화”는 기능 탐지 수준으로만 적용하고, 최종 판단은 클릭 후 권한 요청에서 한다.

### 5.2. Broadcast Window

게임 모드 진입 시 `broadcast window`를 표시한다.

위치 후보:

- 기존 player 영역 위 또는 내부의 빈 영역
- player 영역 위에 덧씌우는 투명 overlay layer
- 점수와 안내 문구를 표시하는 compact panel

표시 정보:

- 기존 player 영역의 재생 버튼, 정지 버튼, seek bar
- oscillator volume bar
- 작곡가, 곡 제목, 현재 BPM
- 점수 정보: 평균 음정 정확도, Perfect / Ok / Bad / Miss 카운트, Combo, 총점
- `exit practice mode` 버튼
- 판정 기준과 환경 안내

환경 안내 문구에 포함할 내용:

- 이 기능은 연습 보조용이며 정밀한 채점 시스템이 아니다.
- 마이크와 조용한 환경을 권장한다.
- 스피커 출력, 연결된 재생 장치, 입력 장치 설정, 주변 소음은 판정 안정성에 영향을 줄 수 있다.
- 음정 판정은 장치, 브라우저, 주변 소음의 영향을 받는다.

개선 제안:

- broadcast window는 처음부터 큰 modal로 만들기보다 player 주변 overlay로 시작한다.
- 점수 표시와 안내 문구는 접거나 축약할 수 있게 한다.
- 재생 중에는 점수와 현재 판정 표시가 먼저 보이고, 긴 안내 문구는 시작 전 상태에서만 자세히 보이게 한다.

## 6. 게임 상태와 허용 동작

### 6.1. 상태 모델 초안

게임 모드는 앱의 일반 edit/view 상태와 별도 런타임 세션으로 둔다.

```ts
type GameModeState =
  | { kind: "off" }
  | { kind: "preparing" }
  | { kind: "ready"; session: GameSession }
  | { kind: "playing"; session: GameSession }
  | { kind: "paused"; session: GameSession }
  | { kind: "finished"; result: GameResult }
  | { kind: "error"; message: string };
```

상태 의미:

- `off` : 게임 모드가 꺼져 있다.
- `preparing` : 마이크 권한 요청, pitch detector 초기화, target event 준비 중이다.
- `ready` : 시작 전 설정이 가능하다.
- `playing` : 재생과 판정을 진행한다.
- `paused` : 재생과 판정을 일시중지하고 현재 점수를 유지한다.
- `finished` : 더 이상 재생할 판정 대상이 없어 최종 점수를 산정했다.
- `error` : 마이크, pitch detector, 권한, audio graph 문제로 진입 또는 진행에 실패했다.

### 6.2. 허용 동작

게임 모드에서 허용하는 동작:

- YouTube 동시 재생
- 처음부터 재생
- 일시정지
- 일시정지 후 다시 재생
- 처음으로 돌아가기
- oscillator volume 조절
- 시작 전 YouTube mode on/off와 offset 조절
- 시작 전 view speed 조절

게임 모드에서 금지하는 동작:

- edit mode
- score jump
- seek bar 임의 이동
- loop playback
- layout apply
- score load
- Clear All
- range selection, paste, undo/redo 등 편집 동작
- active track 변경

현재 판단:

- `ready` 상태에서는 volume, speed, YouTube 설정 같은 환경 설정을 허용한다.
- `playing`과 `paused` 상태에서는 판정 공정성을 위해 대부분의 환경 설정을 고정한다.
- `stop` 또는 `처음으로 돌아가기`는 점수를 리셋하고 `ready`로 돌아간다.
- `pause`와 `resume`은 점수를 유지하고 판정만 중단/재개한다.
- `finished`에서는 최종 결과를 표시하고, restart 또는 exit만 허용한다.

## 7. 마이크 입력과 Pitch Detection

### 7.1. 입력 경로

브라우저 `getUserMedia({ audio: true })`로 마이크 입력을 받는다.

첫 구현 후보 구조:

```text
MediaStream
  -> AudioContext
  -> MediaStreamAudioSourceNode
  -> AnalyserNode 또는 AudioWorklet
  -> PitchDetector
  -> GamePitchFrame
```

첫 구현은 단순성과 브라우저 호환성을 위해 main thread의 `AnalyserNode.getFloatTimeDomainData()` 기반으로 시작할 수 있다.
성능 문제가 확인되면 `AudioWorklet` 또는 worker 분리를 검토한다.

### 7.2. Pitch Frame

```ts
type GamePitchFrame = {
  capturedAtMs: number;
  scoreSeconds: number;
  frequencyHz: number | null;
  midi: number | null;
  centOffset: number | null;
  clarity: number;
  rms: number;
  isVoiced: boolean;
};
```

필드 의미:

- `capturedAtMs` : `performance.now()` 기준 캡처 시각
- `scoreSeconds` : 해당 프레임을 비교할 score time
- `frequencyHz` : 인식된 주파수
- `midi` : 가장 가까운 MIDI note number
- `centOffset` : 가까운 MIDI note와의 cent 차이
- `clarity` : pitch detector가 제공하거나 앱이 계산한 신뢰도
- `rms` : 입력 음량
- `isVoiced` : 판정에 사용할 만큼 유효한 음성/연주 입력인지 여부

### 7.3. 필터링

첫 구현에서 pitch frame은 다음 조건을 모두 만족할 때 판정에 사용한다.

- `frequencyHz`가 유한한 양수이다.
- `clarity`가 threshold 이상이다.
- `rms`가 threshold 이상이다.
- 너무 낮거나 높은 비현실적 frequency가 아니다.

초기 후보값:

- `clarity >= 0.75`
- `rms >= 0.01`
- `frequencyHz` 범위는 현재 layout의 min/max MIDI에서 한 옥타브 정도 여유를 둔 범위

위 값은 고정 명세가 아니라 브라우저 수동 테스트로 조정한다.

### 7.4. 라이브러리 후보

첫 후보는 `pitchy`이다.

이유:

- JavaScript 브라우저 환경에서 사용 가능하다.
- 실시간 tuner 수준의 사용을 목표로 한다.
- pitch와 clarity를 함께 반환한다.
- MIT license로 직접 의존 위험이 낮다.
- ESM 패키지라 현재 Vite/TypeScript 구조와 잘 맞는다.

비교 후보:

- `cwilso/PitchDetect` : MIT license, Web Audio autocorrelation 예제로 개념 검토에 적합하다.
- `Meyda` : MIT license, real-time feature extraction이 가능하며 RMS, spectral feature 등 보조 필터 후보로 검토할 수 있다.

확정 결정:

- 첫 구현은 `pitchy`를 정식 npm 의존성으로 추가해 pitch detection 품질을 먼저 검증한다.
- 마이크 권한, pitch 안정성, clarity/RMS threshold, latency는 브라우저 수동 검증으로 계속 조정한다.
- 1차 최소 필터는 별도 feature extraction 라이브러리 없이 직접 구현한다.
- 최소 필터는 `pitchy`의 clarity, 직접 계산한 RMS, layout 기반 frequency range를 조합한다.
- pitch 튐이나 잡음 문제가 prototype에서 확인되면 최근 frame median/smoothing, `AudioWorklet`, Meyda 같은 보조 feature extraction 라이브러리 도입을 후속으로 검토한다.
- Meyda는 pitch detector라기보다 보조 feature extraction 후보로 본다.
- GPL 계열 pitch detection 코드는 비교 전용으로만 둔다.

## 8. Pitch Visualization

악보 영역의 가장 왼쪽 재생 기준선 인근에 현재 인식된 음정을 작은 형광초록색 원으로 표시한다.

정책:

- `GamePitchFrame.isVoiced === true`일 때만 표시한다.
- `midi + centOffset`을 renderer row coordinate로 변환한다.
- pitch dot은 target note에 맞춰 보정하지 않고 detector가 반환한 생 pitch를 그대로 표시한다.
- target 기반 octave correction, 배음 후보 보정, hysteresis lock은 현재 구현에서 제거한다.
- 표시 위치는 playback 기준선 근처의 고정 x 좌표를 사용한다.
- layout pitch 범위를 벗어나면 최상단 또는 최하단 note row 중심에 clamp한다.
- pitch가 불안정하거나 무성 구간이면 원을 숨기거나 흐리게 표시한다.

구현 경계:

- renderer의 공식 note item에는 넣지 않는다.
- game overlay canvas 또는 DOM overlay에 그린다.
- viewport bounded rendering과 독립적으로 현재 보이는 화면 기준으로 갱신한다.

개선 제안:

- 형광초록색 원만으로는 정확도 상태를 알기 어려우므로, 후속으로 색상 또는 테두리로 `Perfect / Ok / Bad / Miss`를 표현할 수 있다.
- 단, 1차 구현에서는 pitch dot과 판정 text를 분리한다.
- pitch dot은 현재 입력 상태 확인용이므로, 점수 판정용 timing `Sync` 보정도 적용하지 않는다.

## 9. 판정 대상 생성

게임 모드는 현재 활성 track의 `NoteEvent`를 판정 대상으로 사용한다.

입력:

- `AnalysisResult`
- `activeTrackIds`
- `TickTimeMapper`

출력:

```ts
type GameJudgeTarget = {
  eventId: string;
  trackId: TrackId;
  startSeconds: number;
  endSeconds: number;
  targetMidi: number;
  targetCentOffset: number;
  sourceEventKind: "note";
  effectKinds: GameEffectKind[];
};
```

기본 정책:

- `NoteEvent` 중심으로 target을 만든다.
- `RestEvent`, `MuteEvent`는 1차 판정 대상에서 제외한다.
- tuplet은 group 자체가 아니라 내부 note event를 통해 간접 처리한다.
- gliss, vib, trem은 1차에서는 기본 note 판정 위에 effect marker만 붙인다.
- 동일 시점에 여러 active track note가 있으면, 해당 시점의 후보 target 중 사용자 pitch와 가장 가까운 target 하나를 선택한다.
- 1차 구현에서는 동시 이벤트를 사전에 묶는 `judge group` 구조를 만들지 않는다.
- 1차 구현에서는 전체 판정 대상 대비 coverage를 계산하지 않는다.

개선 제안:

- target 선택은 canvas 좌표 겹침보다 `midi/cent` 수치 비교가 더 단순하고 테스트하기 쉽다.
- renderer 좌표 기반 판정은 시각 효과와 zoom/view 상태에 영향을 받으므로 판정 로직의 진실 원천으로 쓰지 않는다.
- 좌표 겹침은 디버그 overlay 또는 사용자 설명용 보조 자료로만 둔다.
- 같은 시간대에 여러 track 이벤트가 겹치는 악보를 불리하게 만들지 않기 위해, 점수는 현재 scoring sample에서 선택된 target 하나만 기준으로 계산한다.

## 10. 기본 NoteEvent 판정

### 10.1. 음정 오차

기본 오차는 cent 단위로 계산한다.

```text
pitchErrorCent = abs(userPitchCent - targetPitchCent)
```

단, 초안의 “옥타브가 다르더라도 동일 계이름이면 정답” 정책을 적용하려면 pitch class 기반 오차를 사용한다.

```text
pitchClassErrorCent = min distance in cents modulo 1200
```

확정 결정:

- 첫 구현 기본값은 pitch class 허용을 켠다.
- 같은 계이름이면 옥타브가 달라도 정답으로 인정한다.
- “옥타브까지 정확히 맞추기”는 후속 옵션 후보로만 둔다.

이유:

- 오타마톤 계열 악기나 연습 환경에서는 옥타브 차이가 쉽게 발생할 수 있다.
- 게임 모드의 1차 목표가 정밀 채점보다 실용적 연습 보조이기 때문이다.

### 10.2. 정확도 등급

초기 판정표:

| 오차 범위 | 정확도 | 표시 문자 | 점수 처리 |
| --- | ---: | --- |
| 50 cent 미만 | 100% | Perfect | 가점 |
| 50 cent 이상 100 cent 미만 | 66.7% | Ok | 가점 |
| 100 cent 이상 200 cent 미만 | 33.3% | Bad | 낮은 가점 |
| 200 cent 이상 | 0% | Miss | 점수 변화 없음 |

주의:

- `Miss`는 유효 pitch 입력이 있었지만 현재 후보 target과 충분히 가깝지 않은 경우를 의미한다.
- 무음, 불인식, 신뢰도 부족은 `Miss`가 아니며 점수 변화도 없다.
- `Miss`는 사용자가 실제로 다른 음을 낸 상황을 구분하기 위한 피드백이며, 1차 점수식에서는 음수 감점을 적용하지 않는다.

### 10.3. 판정 주기

초안은 일정 시간마다 현재 pitch와 현재 재생 중인 NoteEvent를 비교하는 방식을 제안한다.

첫 구현 후보:

- pitch detection frame rate: 약 30fps 이하
- scoring sample interval: 80ms에서 120ms
- target window: 현재 `scoreSeconds`가 `[startSeconds, endSeconds)` 안에 있는 NoteEvent
- 너무 짧은 note는 최소 1회 판정되도록 보정

개선 제안:

- `0.5 initialTick + n currentTick` 방식은 BPM과 `stepsPerBeat`에 따라 실제 시간이 크게 달라질 수 있다.
- 판정 샘플링은 ms 기준으로 두고, target lookup만 score seconds를 사용한다.
- pitch detection은 가능한 주기로 실행하되, 점수 계산은 고정 scoring interval에서만 수행한다.
- 기기 성능이나 pitch detection frame rate 차이가 점수에 직접 영향을 주지 않도록, detection frame마다 점수를 더하지 않는다.
- 1차 구현은 이벤트별 결과를 반드시 확정하지 않고, scoring sample 단위로 점수를 누적한다.

### 10.4. 인접 note 전환부 완화

빠른 솔로에서 사용자가 이전 note를 아주 조금 늦게 끝내거나 다음 note를 아주 조금 일찍 시작하면, 인접 note 전환부의 scoring sample이 반대쪽 note 기준 `Miss`를 만들 수 있다.

확정 결정:

- `NoteEvent.startSeconds` 이전 약 `120ms` 동안 다음 note를 판정 후보에 미리 포함한다.
- `NoteEvent.endSeconds` 이후 약 `120ms` 동안 이전 note를 판정 후보로 유지한다.
- 이 값은 현재 practice scoring cadence가 약 `100ms`인 점을 감안해, 전환부가 한 번의 scoring sample 간격 때문에 보정창 밖으로 밀려나는 상황을 줄이기 위한 값이다.
- 이 grace는 새로운 note를 강제로 맞게 처리하는 기능이 아니라, 현재 scoring sample의 후보 target 목록에 인접 note를 잠깐 함께 두는 기능이다.
- 동시에 현재 note와 인접 note가 후보가 되면 기존 정책처럼 사용자 입력 pitch와 가장 가까운 target 하나를 선택한다.
- 따라서 이전 note를 늦게 release한 입력은 직전 note 기준으로, 다음 note를 일찍 attack한 입력은 다음 note 기준으로 `Perfect` / `Ok`가 될 수 있다.
- 무음 또는 불명확한 입력은 기존처럼 scoring sample을 만들지 않는다.
- 이 완화는 연습 보조 목적의 forgiving 판정이며, 엄격한 리듬게임식 timing 판정은 후속 기능으로 분리한다.

## 11. 점수 산정

### 11.1. 표시 점수

게임 모드는 최소 세 가지 점수를 표시한다.

- 전체 평균 음정 정확도
- 전체 평균 타이밍 정확도
- Perfect
- Ok
- Bad
- Miss
- Early
- Late
- Bad (timing)
- Miss (timing)
- Combo
- 총점

### 11.2. scoring sample 결과

```ts
type GameScoringSampleResult = {
  targetEventId: string;
  trackId: TrackId;
  scoreSeconds: number;
  targetMidi: number;
  targetCentOffset: number;
  pitchAccuracy: number;
  label: "Perfect" | "Ok" | "Bad" | "Miss";
  status: "hit" | "miss";
  scoreEligible: boolean;
  scoreBlockedReason: "attackRequired" | null;
  scoreContribution: number;
  timing: GameTimingJudgeResult;
  timingOnsetId: number | null;
  timingJudgedEventId: string | null;
};
```

scoring sample 결과 정책:

- 현재 score time의 active track 후보 중 사용자 pitch와 가장 가까운 target 하나만 선택한다.
- detector가 반환한 생 pitch를 target으로 이동시키지 않는다.
- pitch class 허용 모드에서는 생 pitch와 target의 pitch class 오차를 직접 비교한다.
- 유효 pitch frame이 없거나 신뢰도/음량 기준을 만족하지 못하면 scoring sample을 만들지 않고 점수 변화도 없다.
- 선택된 target과의 오차가 `200 cent` 이상이면 `Miss` sample로 기록하되, scoreContribution은 `0`으로 둔다.
- timing 오차가 `150ms` 이상 `250ms` 미만이면 pitch 오차가 `Perfect` 또는 `Ok` 범위여도 최종 label을 `Bad`로 강등한다.
- timing 오차가 `250ms` 이상이면 pitch 오차가 `Perfect` 범위여도 최종 label을 `Miss`로 강등하고 pitchAccuracy와 scoreContribution을 `0`으로 둔다.
- `Perfect`, `Ok`, `Bad`는 `hit`으로 본다.
- score eligibility가 true인 sample만 pitchAccuracy와 track weight에 따라 scoreContribution을 더한다.
- score eligibility가 false인 sample은 pitch label을 유지하되 scoreContribution은 `0`으로 둔다.
- 사용자가 일부 이벤트를 놓치는 상황은 track 분리와 한 열 다중 이벤트 허용 구조상 피하기 어렵기 때문에, 놓친 이벤트 자체를 별도 coverage 감점으로 처리하지 않는다.

### 11.3. 총점 공식

확정 결정:

- 1차 구현은 coverage 기반 감점 또는 보정을 사용하지 않는다.
- 동시 이벤트 `judge group` 사전 생성도 1차 구현 범위에서 제외한다.
- 점수는 고정 scoring interval마다 현재 후보 target 중 사용자 pitch와 가장 가까운 target 하나를 기준으로 가산한다.
- `Miss`는 유효 pitch 입력이 있었지만 허용 오차를 초과한 상태로 기록하되, 음수 감점 없이 `0`점으로 처리한다.
- 무음, 불인식, 신뢰도 부족은 `Miss`가 아니며 점수 변화가 없다.
- `Perfect`, `Ok`, `Bad`, `Miss`는 각각 sample 수를 카운트해 결과 화면에 표시한다.
- timing 판정은 `Early`, `Late`, `Bad (timing)`, `Miss (timing)`을 별도 카운트로 표시한다.
- Standard `Combo`는 scoreEligible인 `Perfect`, `Ok`, `Bad`가 연속된 sample 수로 계산한다.
- Pro Mode `Combo`는 별도 Pro 정책에 따라 scoreEligible인 `Perfect`, `Ok`만 이어지고 `Bad`에서 초기화된다.
- `Combo`는 1차 구현에서 점수 공식에 넣지 않고, 재생 중 피드백과 결과 표시용 값으로 둔다.
- 평균 정확도는 `Perfect`, `Ok`, `Bad`, `Miss` sample 전체를 분모로 계산한다.
- `Miss` sample은 정확도 `0%`로 포함되어 전체 평균을 낮춘다.

기본 공식:

```text
articulationBonus =
  1 when articulation detection is not enabled

attackCredit =
  1 when score eligibility is satisfied
  0 when a required attack has not been detected

sampleScore =
  selectedTrackWeight * pitchAccuracy * articulationBonus * attackCredit

totalScore =
  sum(sampleScore)
```

표시값:

- `Accuracy` : `Perfect`, `Ok`, `Bad`, `Miss` sample의 평균 pitchAccuracy
- `Timing Accuracy` : timing onset이 매칭된 sample의 평균 timing accuracy
- `Perfect` : `Perfect` sample 수
- `Ok` : `Ok` sample 수
- `Bad` : `Bad` sample 수
- `Miss` : 유효 pitch 입력이 있었지만 허용 오차를 초과했거나 timing miss로 강등된 sample 수
- `Early` : timing 오차가 `80ms` 이상 `150ms` 미만인 빠른 입력 수
- `Late` : timing 오차가 `80ms` 이상 `150ms` 미만인 늦은 입력 수
- `Bad (timing)` : timing 오차가 `150ms` 이상 `250ms` 미만이라 `Bad`로 강등된 sample 수
- `Miss (timing)` : timing 오차가 `250ms` 이상이라 `Miss`로 강등된 sample 수
- `Combo` : 현재 mode 기준으로 이어진 현재 또는 최고 combo sample 수
- `Score` : 가점 sample의 누적 점수

현재 판단:

- 점수는 경쟁용 정밀 채점이 아니라 연습 보조용 누적 점수로 본다.
- 긴 note를 안정적으로 유지하면 여러 scoring sample에서 점수를 얻을 수 있다.
- 악보의 이벤트 겹침 여부를 사전에 검사하지 않고, 현재 시점의 후보 target 중 가장 가까운 하나만 자유롭게 선택한다.
- coverage, 전체 target 수 대비 처리율, event별 상세 결과는 필요하면 후속 분석용 디버그 값으로만 추가한다.
- Combo는 사용자가 흐름을 유지하고 있는지 보여 주는 UI 피드백으로는 유용하지만, 1차 점수식에 넣으면 점수 변동 원인이 불투명해질 수 있다.

### 11.4. Attack Credit / Score Eligibility

한 번 누른 같은 pitch 입력이 별도 note event들을 모두 `Perfect`로 통과하는 문제는 최종 label을 `Bad` 또는 `Miss`로 강등하는 방식으로 해결하지 않는다.

이유:

- gliss end anchor, gliss 직후 long/vib note, hold, vibrato hold, 같은 pitch legato에서는 새 onset이 없는 것이 정상일 수 있다.
- no-onset 자체를 `Bad` 또는 `Miss`로 보면 정상적인 legato 구조가 과도하게 실패한다.
- 이 경우 사용자의 pitch는 맞을 수 있으므로 label을 강제로 틀린 음정처럼 보이게 만드는 것은 피드백 의미가 맞지 않는다.

따라서 현재 timing anti-cheat는 `label`이 아니라 `scoreEligible`을 분리해 처리한다.

현재 구현 정책:

- pitch label은 기존대로 계산한다.
- matched onset이 없더라도 내부 pitch 판정 label은 유지할 수 있다.
- 일반 새 note event가 attack을 요구하는데 attack이 확인되지 않으면 `scoreEligible = false`로 둔다.
- `scoreEligible = false`이면 `scoreContribution = 0`이고 combo도 증가하지 않는다.
- `scoreEligible = false`이면 judge overlay를 표시하지 않고 live/result pitch count에도 더하지 않는다.
- 필요하면 후속으로 별도 `hold` / `no attack` 보조 텍스트를 붙인다.

초기 attack requirement:

| 구조 | attackRequired |
| --- | --- |
| 일반 새 `NoteEvent` | true |
| 같은 `NoteEvent` 내부 hold sustain | true로 시작하되, 한 번 attack credit을 받은 뒤 같은 event의 후속 sample은 scoreEligible 유지 |
| gliss destination note | false |
| gliss destination 직후 같은 pitch long/vib neighbor | false 후보 |
| 같은 pitch의 반복 별도 note event | true |
| tuplet 내부 짧은 same-pitch note | 보류 |

주의:

- `attackRequired` 판단은 `AnalysisResult`와 `NoteEvent`/`GlissEvent` 문맥에서 만든 runtime target 속성으로 둔다.
- score JSON에는 저장하지 않는다.
- 첫 구현에서는 결과 화면에 별도 attack count를 추가하지 않고, total score와 combo에만 반영한다.
- 실제 마이크 테스트 전까지 `Bad`/`Miss` 강등보다 보수적으로 적용한다.

현재 runtime 타입:

```ts
type GameJudgeTarget = {
  eventId: string;
  trackId: TrackId;
  startSeconds: number;
  endSeconds: number;
  targetMidi: number;
  targetCentOffset: number;
  attackRequired: boolean;
};

type GameScoringSampleResult = {
  ...
  scoreEligible: boolean;
  scoreBlockedReason: "attackRequired" | null;
};
```

초기 score 처리:

```text
if sample.label is Perfect/Ok/Bad and attackRequired and attack is missing:
  scoreEligible = false
  scoreContribution = 0
  combo does not increase
else:
  scoreEligible = true
  use existing pitch/timing score
```

### 11.5. Combo

Combo는 현재 judge mode의 combo policy에 따라 끊기지 않고 이어진 scoreEligible scoring sample 수이다.

정책:

- `Perfect`, `Ok`, `Bad` sample이고 scoreEligible이면 combo를 1 증가시킨다.
- `Perfect`, `Ok`, `Bad` sample이어도 scoreEligible이 false이면 combo를 증가시키지 않는다.
- `Miss` sample이면 combo를 0으로 초기화한다.
- 무음, 불인식, 신뢰도 부족처럼 scoring sample이 생성되지 않은 경우에는 combo를 유지할지 끊을지 후속 prototype에서 결정한다.
- 1차 구현에서는 현재 combo와 최고 combo를 표시할 수 있다.
- 1차 구현에서는 combo bonus를 적용하지 않는다.

## 12. 효과 판정 후보

효과 판정은 실패 감점을 만들지 않고, 성공한 경우에만 별도 bonus를 더한다.
마이크, 브라우저, 연주 방식에 따라 gliss/vib/trem 인식률이 달라질 수 있으므로, 인식되지 않은 효과를 pitch/timing 점수 감점으로 처리하지 않는다.

점수 집계는 공통 `effectBonusScore`를 두되, 표시와 후속 확장을 위해 효과별 count를 분리한다.

효과별 표시:

| 이벤트 | 표시 문자 | summary count |
| --- | --- | ---: |
| gliss | `Gliss!` | `glissBonusCount` |
| vib | `Vib!` | `vibBonusCount` |
| trem | `Trem!` | `tremBonusCount` |

통합 bonus 정책:

- 성공한 effect target은 한 번만 bonus를 지급한다.
- 실패한 effect target은 아무 결과도 만들지 않는다.
- `Total Score`에는 기존 pitch/timing 기반 score와 `effectBonusScore`를 함께 더한다.
- 첫 구현의 gliss interval bonus는 `trackDifficulty * 0.25`를 사용한다.
- 결과 화면에는 `Gliss!`, `Vib!`, `Trem!`, `Effect Bonus`를 별도로 표시할 수 있게 둔다.
- effect 판정은 별도 실시간 분석 루프를 만들지 않고 기존 scoring tick에서 함께 처리한다.
- pitch detector frame마다 모든 effect를 검사하지 않고, `100ms` 내외 scoring cadence에서 필요한 target만 조회한다.
- effect target 목록은 practice playback 시작 시 analyzer 결과에서 한 번 만들고, playback 중에는 작은 배열과 set 조회만 수행한다.

공식:

```text
score =
  previousScore + pitchOrTimingContribution + effectBonusContribution

glissIntervalBonusContribution =
  trackDifficulty * 0.25
```

현재 실제 effect 판정은 gliss와 vib가 1차 구현되었고, trem은 count와 표시 필드만 준비한다.

성능 판단:

- 현재 gliss 구현은 `GlissEvent` target 수만큼 단순 interval index와 voiced 여부를 검사하므로, 일반 악보 규모에서는 pitch detection 자체보다 훨씬 가볍다.
- vib/trem도 같은 원칙으로 구현한다. 즉, audio thread나 pitch input loop에 무거운 계산을 넣지 않고, 최근 frame ring buffer를 scoring tick에서만 요약한다.
- 후속 구현에서 효과 target이 많아지는 악보가 문제를 만들면, 현재 score time 근처 target만 보는 time-window index를 추가한다.

### 12.1. Gliss

1차 구현:

- analyzer의 `GlissEvent`에서 start/end anchor sound와 anchor tick을 practice bonus target으로 만든다.
- 별도 pitch detector나 FFT를 추가하지 않고 현재 `GamePitchFrame`만 사용한다.
- gliss start~end 구간을 `250ms` 간격 interval로 나눈다.
- 각 interval은 한 번만 판정한다.
- 짧은 gliss의 실제 연주 지연을 완화하기 위해 gliss duration의 `50%`, 최대 `150ms`를 앞뒤 grace window로 둔다.
- grace window 안의 판정은 가장 가까운 gliss interval로 clamp하고, target pitch 보간은 원래 gliss start/end 범위 안으로 clamp한다.
- 해당 interval의 scoring tick에서 현재 frame이 voiced pitch이면 `Gliss!` bonus를 지급한다.
- 성공하면 `Gliss!` overlay를 표시하고 `glissBonusCount`와 `effectBonusScore`를 증가시킨다.
- 실패 frame은 interval을 소비하지 않는다. 같은 interval 안에서 이후 voiced frame이 들어오면 다시 판정할 수 있다.
- overlay 표시 위치는 마이크 입력 pitch가 아니라 gliss start/end target pitch를 현재 score time 기준으로 선형 보간한 위치를 사용한다.
- 따라서 사용자가 다른 옥타브로 연주해도 `Gliss!` 텍스트는 악보 target note 근처에 뜬다.
- `Gliss!`는 기본 판정/Combo 텍스트를 지우지 않고 그 우측에 별도 effect 텍스트로 표시한다.
- 실패해도 감점하지 않는다.

후속 후보:

- 필요하면 start/end pitch, 전체 방향성, continuity 조건을 다시 약하게 추가한다.
- 선형 pitch ramp와의 평균 오차 계산은 오탐/미탐이 많을 때만 재검토한다.

### 12.2. Vibrato

1차 구현:

- 대상은 `NoteEvent.effects` 중 `vib: true`인 구간으로 제한한다.
- 최근 pitch frame ring buffer에서 target 구간 안의 voiced frame만 사용한다.
- 평균 pitch는 target pitch class 근처여야 한다.
- 선형 drift를 제거한 뒤 흔들림 amplitude, direction-change count, rate를 검사한다.
- 일반 target은 최소 `0.3s`, 최소 4 voiced frame, 최소 방향 전환 2회를 요구한다.
- `0.25s` 이하 short target은 최소 window를 `max(0.1s, targetDuration * 0.55)`, 최소 voiced frame을 3, 최소 방향 전환을 1로 완화한다.
- 160 BPM, `stepsPerBeat = 4` 기준 2 tick, 약 `187.5ms` vib target을 인식하는 회귀 테스트가 있다.
- 조건을 만족하면 `Vib!` 표시와 `vibBonusCount`, `effectBonusScore` 증가를 붙인다.

주의:

- 틀린 음정과 vibrato를 구분하려면 평균 pitch와 흔들림 특성을 함께 봐야 한다.
- short target 완화는 실제 짧은 vib 인식을 돕지만 false positive 가능성이 있으므로 실제 마이크 테스트로 계속 조정한다.
- `25 cent`는 단일 오차 허용값이 아니라, 평균 pitch 주변으로 실제로 흔들리는 폭의 기준이다.
- detector 튐이 많은 환경에서는 median smoothing 또는 outlier 제거를 먼저 적용한다.

### 12.3. Tremolo

현재 판단:

- tremolo는 pitch detection만으로 판정하기 어렵다.
- 별도 FFT를 추가하지 않고 현재 `GamePitchFrame.rms`와 timing onset 후보를 우선 재사용한다.
- 대상은 trem modifier가 적용된 note effect segment로 제한한다.
- 구간 내 RMS envelope가 반복적으로 올라갔다 내려가는지, 또는 voiced onset 후보가 반복되는지 본다.
- 1차 구현은 기대 trem division과 완전히 맞추기보다 반복 에너지 변화가 있는지만 약하게 본다.
- 조건을 만족하면 `Trem!` 표시와 `tremBonusCount`, `effectBonusScore` 증가를 붙인다.
- 실패해도 감점하지 않는다.

## 13. 타이밍 판정 후보

타이밍 판정은 별도 큰 판정 label을 만들지 않고, 음정 판정 label을 보조하거나 강등하는 조건으로 둔다.

확정 결정:

- onset 후보는 `unvoiced -> voiced` 전환 또는 voiced 상태에서 pitch class가 충분히 바뀐 순간으로 만든다.
- onset 후보와 `NoteEvent.startSeconds`의 차이를 ms 단위로 비교한다.
- `Sync` 보정이 적용된 score time을 타이밍 후보와 target 조회에 사용한다.
- 각 `NoteEvent`는 timing 판정을 최대 1회만 받는다.
- 한 onset 후보는 여러 note에 중복 배정하지 않는다.
- 같은 pitch의 아주 짧은 연속 note는 onset 분리가 불확실하므로 timing 강등을 무리하게 적용하지 않는다.
- onset 후보가 없는 긴 note를 직접 `Bad` 또는 `Miss`로 강등하는 보정은 gliss/long/vib legato 구조에서 너무 엄격해져 롤백했다.
- no-onset anti-cheat는 후속 설계에서 sustain/attack 분리, note-start-only overlay gating, gliss/hold/vib context 처리를 함께 검토한다.

타이밍 판정표:

| 타이밍 오차 | 표시 | 최종 음정 판정 |
| --- | --- | --- |
| 80ms 미만 | 없음 | 음정 판정 유지 |
| 80ms 이상 150ms 미만 | 작은 `early` / `late` 텍스트 | 음정 판정 유지 |
| 150ms 이상 250ms 미만 | 작은 `early` / `late` 텍스트 | `Bad`로 강등 |
| 250ms 이상 | 별도 early/late 텍스트 없음 | `Miss`로 강등 |

주의:

- 여기서 `Miss`는 timing 전용 label이 아니라 기존 음정 판정의 최종 `Miss` label이다.
- 여기서 `Bad` 역시 timing 전용 label이 아니라 기존 음정 판정의 최종 `Bad` label이다.
- timing accuracy는 `<80ms = 100%`, `80-149ms = 66.7%`, `150-249ms = 33.3%`, `250ms 이상 = 0%`의 등분 보정치로 계산한다.
- pitch accuracy도 `Perfect = 100%`, `Ok = 66.7%`, `Bad = 33.3%`, `Miss = 0%`로 맞춘다.
- 따라서 음정이 `Perfect` 범위여도 timing 오차가 250ms 이상이면 최종 sample은 `Miss`, accuracy 0%, score 0으로 처리한다.
- 무음, 불인식, 신뢰도 부족처럼 scoring sample이 생성되지 않는 경우에는 기존 정책처럼 `Miss`를 만들지 않는다.

### 13.1. Sync 입력 지연 보정

practice mode는 마이크 입력, 브라우저 audio graph, pitch detector 처리로 인해 사용자가 실제로 낸 음정보다 늦은 pitch frame을 받을 수 있다.

확정 결정:

- 사용자-facing 이름은 `Sync`로 둔다.
- `Sync` 값은 score JSON에 저장하지 않는 앱 런타임 값이다.
- `Sync` 값은 브라우저 localStorage에 사용자 환경값으로 저장한다.
- localStorage를 사용할 수 없는 환경에서는 현재 페이지 세션 동안만 유지한다.
- UI 범위는 `-200ms`부터 `+200ms`까지, `10ms` 단위로 조정한다.
- 기본값은 `+90ms`이다.
- `Sync +N ms`는 입력이 N ms 늦게 앱에 도착한다고 보고, 판정 target 조회 시 `scoreSeconds - N / 1000`을 사용한다.
- `Sync -N ms`는 입력이 N ms 빠르게 판정된다고 보고, 판정 target 조회 시 `scoreSeconds + N / 1000`을 사용한다.
- 보정된 score time은 0초보다 작아지지 않게 clamp한다.
- pitch dot 시각화는 현재 입력 상태 표시이므로 첫 구현에서는 Sync를 적용하지 않는다.
- 점수 판정과 practice 종료 target 확인에는 Sync 보정된 score time을 사용한다.

Sync 팝업:

- `Sync` 버튼으로 calibration 팝업을 연다.
- 팝업에는 반복 beep, 가운데 기준선, 입력 marker, `-10ms` / `+10ms`, `Reset`, `Apply`를 둔다.
- 반복 beep는 사용자가 입력 marker가 기준선에 가까워지는 방향으로 수동 조정하기 위한 보조 기능이다.
- 첫 구현은 자동 추천값을 적용하지 않는다.
- onset detection은 후속 연구 항목으로 남기며, 현재는 voiced pitch frame이 들어온 시점을 marker로 표시한다.

## 14. Pro Mode

Pro Mode는 기본 practice mode보다 엄밀한 판정을 원하는 사용자를 위한 세션 단위 강화 모드이다.

목표:

- 기본 practice mode의 연습 친화적 판정은 유지한다.
- 사용자가 직접 선택했을 때만 더 엄격한 pitch/timing 기준을 적용한다.
- 점수 배율이나 저장 포맷을 바꾸지 않고, 판정 threshold와 combo reset 정책만 바꾼다.

### 14.1. 상태와 저장 정책

내부 상태 후보:

```ts
type PracticeJudgeMode = "standard" | "pro";
```

정책:

- 기본값은 `standard`이다.
- Pro Mode 값은 score JSON에 저장하지 않는다.
- Pro Mode 값은 localStorage에도 저장하지 않는다.
- 브라우저 페이지 세션 동안만 유지한다.
- score load, Clear All, layout 변경은 Pro Mode 값을 직접 초기화하지 않는다.
- 새로고침하면 기본값 `standard`로 돌아간다.

### 14.2. UI

배치:

- practice panel head의 `Sync` 오른쪽에 `Pro` toggle button을 둔다.
- `Pro`가 켜진 동안 practice panel은 red accent를 사용한다.
- error 상태와 혼동되지 않도록 전체 배경을 강한 빨강으로 바꾸지 않고, border/accent/status 중심으로 표시한다.

상호작용:

- practice mode off 상태와 `ready` 상태에서만 Pro toggle을 바꿀 수 있다.
- `preparing`, `countdown`, `playing`, `paused`, `finished` 상태에서는 Pro toggle을 disabled 처리한다.
- `stop`으로 practice score가 reset되어 `ready`로 돌아오면 다시 변경할 수 있다.
- result dialog에는 가능하면 `Mode: Standard` 또는 `Mode: Pro`를 표시한다.

### 14.3. Pitch Threshold

Standard pitch threshold는 현재 구현 기준을 유지한다.

| Mode | Perfect | Ok | Bad | Miss |
| --- | ---: | ---: | ---: | ---: |
| Standard | `< 50 cent` | `< 100 cent` | `< 200 cent` | `>= 200 cent` |
| Pro | `< 30 cent` | `< 60 cent` | `< 100 cent` | `>= 100 cent` |

정책:

- Pro Mode에서도 pitch class 허용은 유지한다.
- target 기반 octave correction, harmonic correction, hysteresis lock은 Pro Mode에서도 사용하지 않는다.
- Pro Mode는 반음 이상 차이 나는 입력을 바로 `Miss`로 처리한다.

### 14.4. Timing Threshold

Standard timing threshold는 현재 구현 기준을 유지한다.

| Mode | No timing text | Early/Late 표시 | Bad 강등 | Miss 강등 |
| --- | ---: | ---: | ---: | ---: |
| Standard | `< 80ms` | `80-149ms` | `150-249ms` | `>= 250ms` |
| Pro | `< 50ms` | `50-99ms` | `100-149ms` | `>= 150ms` |

정책:

- timing 판정 구조는 그대로 유지한다.
- matched onset이 있을 때만 timing early/late/bad/miss를 적용한다.
- no-onset 자체를 직접 `Bad` 또는 `Miss`로 강등하지 않는다.
- `Sync` 보정은 Standard와 Pro 모두 동일하게 적용한다.

### 14.5. Combo Policy

Standard:

```text
scoreEligible Perfect / Ok / Bad -> combo +1
Miss -> combo reset
scoreEligible false / no input -> combo unchanged
```

Pro:

```text
scoreEligible Perfect / Ok -> combo +1
Bad / Miss -> combo reset
scoreEligible false / no input -> combo unchanged
```

이유:

- Standard는 프렛리스 악기와 마이크 환경을 고려해 Full Combo 목표를 현실적으로 유지한다.
- Pro Mode는 사용자가 스스로 선택한 엄격한 연습 기준이므로 `Bad`에서 combo를 끊어도 의도가 분명하다.
- Pro Mode에서도 combo는 점수 배율에 영향을 주지 않는다.

### 14.6. 유지되는 정책

Pro Mode에서도 다음 정책은 유지한다.

- score formula는 변경하지 않는다.
- combo multiplier는 추가하지 않는다.
- effect bonus 정책은 변경하지 않는다.
- gliss grace, vib 판정, 이후 trem 판정은 우선 Standard와 동일하게 둔다.
- no input은 `Miss`를 만들지 않는다.
- scoreEligible이 false인 sample은 judge overlay와 score summary에 반영하지 않는다.

후속 후보:

- Pro result badge 또는 `Pro Clear` 표시
- Pro Mode에서 gliss/vib/trem effect threshold 축소
- Pro Mode 전용 Rules 문구 추가

## 15. 시각 판정 효과

판정 문자는 재생 기준선과 note event가 만나는 지점 근처에 표시한다.

정책:

- 복잡한 CSS animation은 1차 범위에서 제외한다.
- 짧게 나타났다 사라지는 DOM 또는 overlay canvas text로 구현한다.
- label별 색상만 다르게 둔다.

색상 후보:

- `Perfect` : green
- `Ok` : blue
- `Bad` : orange
- `Miss` : red
- `Gliss! / Vib! / Trem!` : 별도 accent color

구현 경계:

- 판정 text는 renderer의 score item이 아니라 game overlay item이다.
- zoom/scroll에 맞춰 위치를 계산하되, score JSON이나 renderer base layer에는 영향을 주지 않는다.

## 16. YouTube Sync와의 관계

게임 모드는 YouTube sync와 공존할 수 있다.

정책:

- YouTube player는 기존 `2.7` 정책처럼 score time을 따라가는 follower이다.
- 게임 모드는 score time을 기준으로 판정한다.
- YouTube offset 조절은 `ready` 상태에서만 허용한다.
- `playing` 또는 `paused` 상태에서는 YouTube video/offset reload를 막는다.
- YouTube embed가 실패해도 local oscillator playback만으로 게임 모드를 진행할 수 있다.

확정 결정:

- 더 안정적인 쪽을 선택한다.
- 따라서 첫 구현에서는 YouTube video/offset 수정과 reload를 `ready` 상태에서만 허용한다.
- `paused` 상태는 점수와 판정 세션을 보존하는 중단 상태이므로, YouTube 설정 변경으로 score time 동기화가 흔들리지 않게 한다.

주의:

- 스피커로 YouTube 음원이 재생되면 마이크에 반주가 들어가 pitch detection을 방해할 수 있다.
- 다만 헤드셋 연결 자체도 OS/브라우저의 입출력 장치 선택, 드라이버 처리, echo cancellation / noise suppression 경로에 영향을 줄 수 있다.
- 현재 practice UI 안내는 헤드셋 권장 문구를 제거하고, 마이크와 조용한 환경 권장만 남긴다.

## 17. Loop / Seek / Speed와의 관계

Loop:

- 게임 모드에서는 loop playback을 금지한다.
- 이유는 반복 구간에서 처리 이벤트 수와 최종 점수 산정 기준이 모호해지기 때문이다.

Seek:

- 게임 모드에서는 임의 seek와 score jump를 금지한다.
- pause/resume은 허용하지만 score time은 불연속으로 바뀌지 않아야 한다.

Stop:

- stop 또는 처음으로 돌아가기는 점수를 리셋하고 `ready` 상태로 돌아간다.

Speed:

- 현재 View Speed는 column width 변경 기능이며 audio seconds를 바꾸지 않는다.
- 따라서 게임 점수에는 영향을 주지 않는다.
- 시작 전 시각적 연습 편의를 위해 허용할 수 있다.

미래의 playback rate:

- 실제 재생 속도를 바꾸는 기능이 생기면 game target seconds와 pitch frame mapping에도 반영해야 한다.
- 이 기능은 1차 게임 모드 범위에서 제외한다.

## 18. Track Layer와 난이도

게임 모드는 현재 active track을 판정 대상으로 사용한다.

정책:

- `ready` 상태에서는 active track 변경을 허용할 수 있다.
- `playing`과 `paused` 상태에서는 active track 변경을 금지한다.
- 동일 시점에 여러 active track note가 있으면 사용자 pitch와 가장 가까운 target 하나를 선택한다.

track 난이도 가중치:

```ts
type GameTrackWeight = {
  basic: number;
  optional: number;
  extra: number;
};
```

확정 결정:

- track 난이도 가중치는 `ScoreFile.musicData.scoreDifficulty`에서 가져온다.
- `scoreDifficulty[trackId]`가 비어 있거나 `0`이면 아래 기본값을 사용한다.
- 사용자가 말한 `ScoreFile.MusicData`의 difficulty는 현재 타입명 기준 `musicData.scoreDifficulty`로 해석한다.

기본값:

- `basic = 1.0`
- `optional = 1.2`
- `extra = 1.5`

multi-track 판정 정책:

- active track이 여러 개일 때도 사용자 pitch와 가장 가까운 target 하나를 선택한다.
- 해당 target의 `trackId`에 대응하는 난이도 가중치를 점수 계산에 사용한다.
- 한 열에 여러 이벤트가 놓이는 구조를 제한하지 않으므로, 사용자가 일부 이벤트를 놓치는 상황은 허용 가능한 구조적 한계로 본다.
- 이 구조적 한계 때문에 1차 점수식은 전체 `NoteEvent` coverage를 계산하지 않는다.

주의:

- score JSON validation 단계에서 `scoreDifficulty`는 트랙별 number를 가지지만, 게임 모드 계산에서는 0과 누락값을 별도로 fallback 처리한다.
- 음수나 유한하지 않은 값이 들어온 경우는 저장 검증 또는 게임 target build 단계에서 방어적으로 기본값으로 대체한다.

## 18.1. 결과 저장과 결과 화면

게임 모드 결과는 score JSON에 저장하지 않는다.

확정 결정:

- 게임 결과는 `ScoreFile`에 저장하지 않는다.
- localStorage에도 자동 저장하지 않는다.
- 별도 결과 파일 저장 기능도 1차 범위에서 제외한다.
- 결과를 남기고 싶은 사용자는 finished 결과 화면을 직접 스크린샷으로 저장한다.
- 이벤트별 상세 결과는 표시하거나 저장하지 않는다.

이유:

- 텍스트 기반 결과 파일은 사용자가 쉽게 수정하거나 다시 작성할 수 있으므로 보상으로서의 상징성이 약하다.
- 공식 점수 기록이나 공유 랭킹 기능을 제공하지 않는 현재 범위에서는 파일 export가 오히려 기능의 의미를 흐릴 수 있다.
- 스크린샷은 사용자가 직접 남기는 결과 공유 방식으로 충분하다.

결과 화면 표시 항목:

- 곡 제목과 아티스트
- `Pitch Accuracy`와 `Timing Accuracy`
- `Total Score`와 `Max Combo`
- pitch 판정군: `Perfect`, `Ok`, `Bad`, `Miss`
- timing 판정군: `Early`, `Late`, `Bad (timing)`, `Miss (timing)`
- 판정표 요약

결과 화면 배치:

```text
Pitch Accuracy | Timing Accuracy
Total Score    | Max Combo
Perfect        | Early
Ok             | Late
Bad            | Bad (timing)
Miss           | Miss (timing)
```

표시 정책:

- 판정별 결과 박스 텍스트 색상은 재생 중 judge overlay의 `Perfect`, `Ok`, `Bad`, `Miss`, `early`, `late` 색상 계열과 맞춘다.
- active track 목록은 결과 화면 첫 구현에서 제외한다.

후속 후보:

- 공식 랭킹, 서명된 결과, 서버 저장 같은 별도 운영 계획이 생기기 전까지 결과 파일 export는 추가하지 않는다.
- 보고서나 발표용으로 필요하면 사용자가 결과 화면을 스크린샷으로 첨부한다.

## 18.2. 종료 조건과 모바일 지원

종료 조건:

- 더 이상 재생할 `NoteEvent`가 없는 시점을 게임 세션 종료 시점으로 본다.
- 기준은 현재 active track에서 생성된 게임 판정 대상 `NoteEvent`이다.
- 마지막 target의 `endSeconds`를 지나면 `finished` 상태로 전환하고 최종 점수를 산정한다.
- 전체 score duration이 더 길더라도 남은 판정 대상 note가 없으면 게임 결과를 확정한다.

모바일 지원:

- 모바일 지원은 가능하면 좋지만 필수 사항은 아니다.
- 첫 구현은 데스크톱 브라우저를 기준으로 한다.
- 모바일은 view/load/play 중심이라는 기존 프로젝트 결정을 유지한다.
- 모바일 마이크 입력, latency, 성능, 브라우저 권한 정책이 충분히 안정적이라고 확인되면 후속 지원 대상으로 둔다.

## 21. 검증 기준

단위 검증:

- frequency to midi 변환이 A4 440Hz를 MIDI 69로 계산한다.
- cent error 계산이 반음과 옥타브 차이를 올바르게 처리한다.
- pitch class mode에서 C4와 C5가 같은 pitch class로 비교된다.
- `GameJudgeTarget` 생성이 active track만 포함한다.
- 동일 시점 target 후보 중 pitch가 가장 가까운 target을 선택한다.
- scoring 함수가 평균 정확도, Perfect / Ok / Bad / Miss 카운트, Combo, 총점을 기대대로 계산한다.
- Miss sample은 정확도 평균 분모에 포함되어 전체 평균을 낮춘다.
- 유효 pitch 입력이 없으면 점수와 Miss가 변하지 않는다.
- 허용 오차를 초과한 유효 pitch 입력은 Miss를 증가시키지만 총점을 낮추지 않는다.
- timing 오차가 `150ms` 이상 `250ms` 미만이면 음정이 맞아도 최종 `Bad`가 된다.
- timing 오차가 `250ms` 이상이면 음정이 맞아도 최종 `Miss`가 되며 scoreContribution은 0이다.
- `80ms` 이상 `150ms` 미만 timing 오차는 작은 `early` / `late` 텍스트만 표시하고 pitch label은 유지한다.
- pitch/timing accuracy는 각각 `100% / 66.7% / 33.3% / 0%` 단계로 계산된다.
- Standard에서는 scoreEligible인 Perfect, Ok, Bad sample이 combo를 증가시키고, Miss sample은 combo를 초기화한다.
- Pro Mode에서 pitch threshold가 `<30 / <60 / <100 / >=100 cent`로 적용된다.
- Pro Mode에서 timing threshold가 `<50 / 50-99 / 100-149 / >=150ms`로 적용된다.
- Pro Mode에서 Bad sample은 combo를 초기화한다.
- Pro Mode 설정은 새로고침 뒤 유지되지 않는다.

브라우저 수동 검증:

- 권한 거부 시 게임 모드에 진입하지 않는다.
- 권한 허용 시 broadcast window가 표시된다.
- 무음 상태에서는 pitch dot이 표시되지 않거나 inactive 상태로 보인다.
- 마이크에 일정 음을 넣으면 pitch dot이 detector 생 pitch의 row 근처에 표시된다.
- pitch 범위를 벗어나면 dot이 상단 또는 하단으로 clamp된다.
- 재생 중 note event와 pitch가 가까우면 `Perfect` 또는 `Ok`가 표시된다.
- 재생 중 note event와 pitch가 크게 다르면 `Miss`가 표시되지만 총점은 음수로 내려가지 않는다.
- 노트 시작보다 `80ms` 이상 빠르거나 늦은 입력은 작은 `early` / `late` 텍스트가 표시된다.
- 노트 시작보다 `250ms` 이상 어긋난 입력은 음정이 맞아도 `Miss`로 표시된다.
- stop은 점수를 리셋한다.
- pause/resume은 점수를 유지한다.
- game mode 중 edit, loop, seek, score load, layout apply가 차단된다.
- YouTube mode와 함께 사용할 때 score playback과 판정이 같은 score time을 기준으로 유지된다.
- Pro toggle은 off/ready 상태에서만 변경되고 playback 중에는 잠긴다.
- Pro Mode가 켜져 있으면 practice panel이 red accent로 표시된다.

- timing 판정은 별도 큰 label이 아니라 기존 pitch label을 보조하거나 `Miss`로 강등하는 조건으로 둔다.
- track 난이도 가중치는 `musicData.scoreDifficulty`에서 가져오며, 0 또는 누락값은 기본값으로 대체한다.
- YouTube 설정 변경은 안정성을 위해 `ready` 상태에서만 허용한다.
- 게임 결과는 score JSON, localStorage, 별도 파일로 저장하지 않는다.
- 이벤트별 상세 결과는 필요하지 않으므로 표시/저장하지 않는다.
- 결과를 남기고 싶은 사용자는 결과 화면을 직접 스크린샷으로 저장한다.
- 종료 조건은 더 이상 재생할 `NoteEvent`가 없는 시점이다.
- 모바일 지원은 필수 범위가 아니며, 첫 구현은 데스크톱 기준으로 둔다.
- Pro Mode는 기본 practice보다 엄격한 판정 강화 모드로 두며, 페이지 세션 동안만 유지한다.
- Pro Mode의 score formula와 effect bonus 정책은 Standard와 동일하게 둔다.
- Pro Mode에서는 Bad sample이 combo를 초기화한다.

## 23. 현재 결론

노래방 / 게임 모드는 현재 구조 위에 별도 런타임 평가 계층으로 얹는 방향이 적절하다.

가장 안전한 1차 목표는 다음이다.

- 게임 모드 진입/종료 상태
- 마이크 권한과 pitch frame 생성
- pitch dot 시각화
- active track `NoteEvent` 중심 음정 판정
- average accuracy / Perfect / Ok / Bad / Miss / Combo / score 표시
- 생 pitch 기반 판정 유지와 target 기반 보정 제거
- timing `early` / `late` 표시 및 timing miss 강등
- edit, seek, loop 등 판정 공정성을 깨는 동작 차단

gliss, vibrato, tremolo는 1차 구현 이후의 확장으로 둔다.
