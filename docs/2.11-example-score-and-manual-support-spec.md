# 2.11. Example Score and Manual Support 명세

## 1. 문서 목적

본 문서는 `회귀 코드` 안정판의 후반 보조 기능인 예제 악보 지원과 Manual 지원을 정의한다.

목표는 다음이다.

- 테스터가 별도 파일을 준비하지 않아도 앱에서 예제 악보를 선택해 실사용 테스트를 시작할 수 있게 한다.
- 예제 악보가 여러 개로 늘어나도 초기 페이지 로드와 JavaScript bundle 크기를 불필요하게 키우지 않는다.
- GitHub Pages 단독 배포에서도 예제 기능이 동작하고, Supabase가 연결되면 원격 예제 목록으로 확장할 수 있게 한다.
- Help 메뉴의 Manual을 실제 사용자 안내 경로로 활성화한다.
- 실사용 테스트 피드백 수집에 필요한 기본 기록 항목을 정한다.

본 문서는 `docs/2.0-ui-mvp-spec.md`의 File / Help 메뉴 후속 명세이며, `docs/1.3-score-json-format.md`의 Score JSON 저장 포맷을 바꾸지 않는다.

## 2. 기본 원칙

예제 악보 JSON은 앱의 TypeScript source bundle에 직접 포함하지 않는다.

```text
app boot
  -> bundled default template 1개만 즉시 로드
  -> example manifest는 필요 시 fetch
  -> selected example score JSON은 사용자가 선택했을 때 fetch
  -> validate / parse / analyze / render
```

3~4분 곡의 Score JSON이 200~300 KiB가 될 수 있으므로, 여러 예제 악보를 `src/assets`에서 `?raw` import하는 방식은 사용하지 않는다.
예제 악보 파일을 여러 개 제공하더라도 초기 앱 로드는 작은 manifest만 추가로 읽거나, 사용자가 Examples 메뉴를 열 때까지 manifest 읽기도 미룬다.

Supabase는 앱의 핵심 저장소가 아니라 읽기 전용 예제 공급원으로 취급한다.
Supabase 장애, 네트워크 실패, CORS 실패, 응답 검증 실패가 있어도 로컬 정적 예제 기능은 유지되어야 한다.

Manual은 정적 HTML 문서를 기본 안내로 두고, 사용법 녹화 영상 또는 YouTube 재생목록은 보조 링크로 둔다.
정적 Manual은 GitHub Pages 앱과 함께 배포되어 네트워크 정책, 영상 비공개, 영상 삭제, 광고, 외부 페이지 차단에 덜 의존한다.

## 3. 범위

2.11 첫 구현 범위:

- File 메뉴 또는 별도 Help-adjacent 메뉴에서 Examples 진입점 추가
- 로컬 정적 example manifest 로드
- 로컬 정적 example Score JSON 지연 로드
- 예제 선택 dialog 또는 panel
- 예제 악보 metadata, 라이선스, 난이도 표시
- 예제 악보 로드 전 현재 악보 교체 확인
- Supabase 원격 manifest 읽기 전용 연결을 위한 provider 경계 정의
- Help 메뉴 Manual 버튼 활성화
- 정적 Manual 페이지 또는 dialog 추가
- Manual 안에서 YouTube 재생목록 링크 제공
- 실사용 테스트 피드백 기록 항목 정의

2.11 첫 구현에서 제외한다:

- Supabase 로그인
- 사용자 악보 업로드
- 사용자 악보 공개 공유
- 예제 악보 편집본 원격 저장
- Supabase DB 쓰기
- 저작권이 불명확한 상용곡 예제 포함
- MIDI / audio import
- Manual 영상 제작 자체

## 4. 예제 악보 저장 위치

로컬 정적 예제는 Vite public asset으로 둔다.

```text
regression-code/public/examples/
  manifest.json
  example-score-1.json
  example-score-2.json
```

`public/examples/*.json`은 빌드 시 JavaScript bundle에 포함되지 않고 정적 파일로 복사된다.
앱은 `fetch("./examples/manifest.json")`로 목록을 읽고, 사용자가 특정 예제를 선택하면 해당 `scoreUrl`을 다시 `fetch()`한다.

초기 boot용 template score는 현재처럼 매우 작은 기본 악보 1개만 source asset으로 유지할 수 있다.
단, 실제 곡 예제는 source import가 아니라 public 정적 파일 또는 Supabase URL로 제공한다.

## 5. Example Manifest 형식

로컬 manifest와 원격 manifest는 같은 의미 구조를 사용한다.

```ts
type ExampleScoreManifest = {
  version: 1;
  generatedAt?: string;
  examples: ExampleScoreManifestItem[];
};

type ExampleScoreManifestItem = {
  id: string;
  title: string;
  artist: string;
  description?: string;
  difficulty?: {
    basic?: number;
    optional?: number;
    extra?: number;
  };
  supportedTracks: ("basic" | "optional" | "extra")[];
  durationSeconds?: number;
  sizeBytes?: number;
  license: ExampleScoreLicense;
  scoreUrl: string;
  thumbnailUrl?: string;
};

type ExampleScoreLicense = {
  label: string;
  url?: string;
  note?: string;
};
```

Manifest 검증 규칙:

- `version`은 `1`만 허용한다.
- `examples`는 배열이어야 한다.
- `id`, `title`, `artist`, `scoreUrl`, `license.label`은 빈 문자열이면 안 된다.
- `supportedTracks`는 하나 이상의 track id를 가져야 한다.
- `scoreUrl`은 로컬 상대 경로 또는 HTTPS URL만 허용한다.
- 원격 manifest의 `scoreUrl`이 상대 경로이면 해당 manifest URL 기준으로 해석한다.
- manifest가 깨져 있으면 Examples UI에 오류를 표시하고, 앱 전체 boot는 실패시키지 않는다.

Manifest는 Score JSON 저장 포맷이 아니다.
예제 목록 표시와 score fetch 위치를 위한 별도 카탈로그 데이터이다.

## 6. Supabase 연동 정책

Supabase 첫 연결은 읽기 전용 example provider로 제한한다.

허용 범위:

- public read 가능한 table, view, edge function, 또는 storage JSON에서 example manifest 읽기
- public read 가능한 storage object 또는 HTTPS URL에서 Score JSON 읽기
- anon key 사용
- 실패 시 로컬 manifest로 fallback

금지 범위:

- Supabase auth 사용
- insert / update / delete
- 사용자별 개인 악보 저장
- 악보 편집본 remote save
- Supabase 응답을 검증 없이 ScoreFile로 취급

Provider 경계는 다음 형태를 권장한다.

```ts
type ExampleProvider = {
  providerId: "local" | "supabase";
  displayName: string;
  loadManifest(): Promise<ExampleScoreManifest>;
  loadScoreText(item: ExampleScoreManifestItem): Promise<string>;
};
```

앱은 provider가 어디인지와 무관하게 `loadScoreText()` 결과를 기존 Score JSON 로드 경로에 전달한다.
따라서 예제 악보 로드는 반드시 `loadScoreJsonText()` 또는 같은 검증 파이프라인을 통과해야 한다.

### 6.1. Supabase 테이블 초안

Supabase table은 ScoreFile 본문을 저장하는 구조가 아니라, 예제 악보 목록을 표시하고 실제 Score JSON 위치를 찾기 위한 카탈로그이다.
따라서 `thumbnail_url`, `display_order`, `is_published`처럼 ScoreFile에는 없는 UI/운영용 metadata가 포함될 수 있다.

첫 구현에서는 앱에서 insert/update/delete를 호출하지 않는다.
Supabase dashboard에서 관리자가 데이터를 올리고, GitHub Pages 앱은 `anon` role로 공개된 row만 읽는다.
`service_role` key는 앱 코드, GitHub repository, 문서 예시에 넣지 않는다.

```sql
create table public.example_scores (
  id text primary key,
  title text not null,
  artist text not null,
  description text,

  difficulty_basic numeric,
  difficulty_optional numeric,
  difficulty_extra numeric,

  supported_tracks text[] not null,

  duration_seconds integer,
  size_bytes integer,

  license_label text not null,
  license_url text,
  license_note text,

  score_url text not null,
  thumbnail_url text,

  display_order integer not null default 0,
  is_published boolean not null default false,

  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),

  constraint example_scores_title_not_blank
    check (length(trim(title)) > 0),

  constraint example_scores_artist_not_blank
    check (length(trim(artist)) > 0),

  constraint example_scores_license_label_not_blank
    check (length(trim(license_label)) > 0),

  constraint example_scores_score_url_not_blank
    check (length(trim(score_url)) > 0),

  constraint example_scores_supported_tracks_not_empty
    check (array_length(supported_tracks, 1) >= 1),

  constraint example_scores_supported_tracks_allowed
    check (supported_tracks <@ array['basic', 'optional', 'extra']::text[]),

  constraint example_scores_duration_positive
    check (duration_seconds is null or duration_seconds > 0),

  constraint example_scores_size_positive
    check (size_bytes is null or size_bytes > 0),

  constraint example_scores_difficulty_basic_positive
    check (difficulty_basic is null or difficulty_basic > 0),

  constraint example_scores_difficulty_optional_positive
    check (difficulty_optional is null or difficulty_optional > 0),

  constraint example_scores_difficulty_extra_positive
    check (difficulty_extra is null or difficulty_extra > 0)
);

alter table public.example_scores enable row level security;

grant select on public.example_scores to anon;

create policy "Public can read published example scores"
on public.example_scores
for select
to anon
using (is_published = true);

create index example_scores_public_order_idx
on public.example_scores (is_published, display_order, title);
```

### 6.2. Supabase 컬럼 의미

| 컬럼 | 앱 manifest 대응 | null 허용 | 의미 |
|---|---|---:|---|
| `id` | `id` | No | 예제 항목의 안정적인 식별자. URL slug처럼 사람이 읽을 수 있는 값을 권장한다. |
| `title` | `title` | No | Examples UI에 표시할 곡 또는 악보 제목. ScoreFile의 `musicData.musicTitle`과 같을 필요는 없지만 보통 동일하게 둔다. |
| `artist` | `artist` | No | Examples UI에 표시할 아티스트/작곡자. 모르면 `Unknown`처럼 명시 문자열을 넣는다. |
| `description` | `description?` | Yes | 예제 설명. 없으면 manifest 변환 시 생략한다. |
| `difficulty_basic` | `difficulty.basic?` | Yes | basic track 표시 난이도. ScoreFile 내부 `musicData.scoreDifficulty.basic`과 별도 catalog 표시값이며, 없으면 생략한다. |
| `difficulty_optional` | `difficulty.optional?` | Yes | optional track 표시 난이도. 없으면 생략한다. |
| `difficulty_extra` | `difficulty.extra?` | Yes | extra track 표시 난이도. 없으면 생략한다. |
| `supported_tracks` | `supportedTracks` | No | 예제 악보에서 테스트 대상으로 안내할 track 목록. `basic`, `optional`, `extra` 중 하나 이상이다. |
| `duration_seconds` | `durationSeconds?` | Yes | 목록에 표시할 예상 곡 길이. 정확한 재생 엔진 값이 아니라 catalog metadata이다. |
| `size_bytes` | `sizeBytes?` | Yes | Score JSON 파일 크기 표시값. 앱은 실제 fetch 이후에도 ScoreFile 크기 제한을 별도로 검사한다. |
| `license_label` | `license.label` | No | 예제 악보 또는 원곡/편곡 사용 조건을 짧게 표시하는 문자열. |
| `license_url` | `license.url?` | Yes | 라이선스 또는 출처 안내 URL. 없으면 생략한다. |
| `license_note` | `license.note?` | Yes | 저작권/출처/사용 제한에 대한 보조 설명. 없으면 생략한다. |
| `score_url` | `scoreUrl` | No | 실제 Score JSON을 fetch할 URL. Supabase storage public URL 또는 HTTPS URL을 권장한다. |
| `thumbnail_url` | `thumbnailUrl?` | Yes | Examples UI에서 썸네일을 표시할 때 사용할 이미지 URL. 첫 UI에서 사용하지 않더라도 후속 확장을 위해 optional로 둔다. |
| `display_order` | provider 내부 정렬값 | No | Examples 목록 정렬 순서. manifest로 변환할 때 직접 노출하지 않아도 된다. |
| `is_published` | provider 필터 | No | 공개 여부. RLS policy는 `true`인 row만 `anon` role에 노출한다. |
| `created_at` | 없음 | No | Supabase catalog row 생성 시각. 앱 manifest로 노출하지 않는다. |
| `updated_at` | 없음 | No | Supabase catalog row 수정 시각. 필요하면 manifest `generatedAt` 또는 provider debug 정보에 사용할 수 있으나 첫 구현에서는 노출하지 않는다. |

`NULL` 허용 컬럼은 앱 manifest로 변환할 때 `undefined`처럼 취급하고, JSON 응답에는 해당 속성을 넣지 않는다.
반대로 `not null` 컬럼이 빈 문자열이거나 누락되면 DB 제약 또는 앱 manifest validator에서 실패시킨다.

### 6.3. Supabase 보안 규칙

- RLS는 반드시 켠다.
- `anon` role에는 `select`만 허용한다.
- `insert`, `update`, `delete`는 앱에서 사용하지 않는다.
- 앱은 Supabase 응답을 그대로 ScoreFile로 취급하지 않는다.
- 앱은 Supabase row를 먼저 `ExampleScoreManifestItem`으로 변환/검증하고, 사용자가 선택한 `scoreUrl`의 JSON을 다시 기존 ScoreFile validation pipeline에 통과시킨다.
- SQL injection 방어는 앱에서 임의 SQL 문자열을 만들지 않는 구조를 기본으로 한다. Supabase REST의 고정 `select` 요청만 사용하고, 사용자 입력을 SQL fragment로 조립하지 않는다.
- `score_url`, `thumbnail_url`, `license_url`은 DB 제약만 믿지 않고 앱 validator에서도 HTTPS URL 또는 허용된 상대 경로인지 확인한다.

## 7. Examples UI

Examples 진입점은 File 메뉴에 둔다.

```text
File
  JSON Load
  JSON Download
  Local Save
  Local Load
  Examples
```

Examples UI는 dialog 또는 오른쪽 panel 형태로 구현한다.
첫 구현은 dialog를 우선한다.

표시 항목:

- 곡 제목
- 아티스트
- 난이도
- 지원 track
- 예상 길이
- 파일 크기
- 라이선스 label
- 설명
- provider 표시

동작:

- Examples 버튼 클릭 시 manifest를 로드한다.
- manifest 로드 중에는 busy 상태와 left status line을 갱신한다.
- manifest 로드 실패 시 fallback provider를 시도하고, 최종 실패하면 오류를 표시한다.
- 예제 선택 후 Load Example을 누르면 현재 score 교체 확인을 띄운다.
- 확인 후 score JSON을 fetch하고 기존 score load pipeline으로 전달한다.
- 로드 성공 시 edit mode, selection, range selection, undo/redo history는 score file load 정책과 동일하게 초기화한다.
- 로드 실패 시 현재 score는 유지한다.

현재 score가 기본 template에서 변경되지 않았는지 자동 판별하는 기능은 첫 구현에서 필수로 두지 않는다.
첫 구현은 항상 교체 확인을 띄우는 보수적 정책을 사용한다.

## 8. Manual 지원

Help 메뉴의 Manual 버튼을 활성화한다.

기본 정책:

- Manual은 정적 HTML 문서 또는 앱 내부 dialog로 제공한다.
- 정적 HTML 경로를 사용할 경우 `regression-code/public/manual/index.html`을 우선한다.
- Manual은 앱 사용을 막는 modal이 아니라 별도 탭 또는 큰 dialog로 열 수 있다.
- YouTube 사용법 재생목록은 Manual 안의 보조 링크로 둔다.

Manual 첫 내용:

- 앱 목적과 저장 데이터 범위
- JSON Load / Download
- Local Save / Load
- 재생, 정지, seek
- View: Zoom, Speed, Fit Height, Reverse, Theme, Text off
- Loop
- Edit Mode 기본 입력
- Track layer
- Layout
- YouTube sync
- Practice mode beta
- Examples 로드
- 저작권과 예제 악보 라이선스 주의
- 실사용 테스트 피드백 작성법

Manual은 기능 구현과 함께 갱신되어야 한다.
영상 재생목록은 실제 조작 흐름을 보여 주는 보조 자료로 좋지만, 앱의 기본 도움말 계약을 대체하지 않는다.

## 9. 실사용 테스트 피드백 항목

테스터 피드백은 다음 항목을 기준으로 수집한다.

- 테스트 일시
- 앱 배포 URL 또는 commit
- 브라우저와 버전
- OS와 기기 종류
- 사용한 예제 악보 또는 JSON 파일명
- 수행한 작업 순서
- 기대 결과
- 실제 결과
- 재현 가능 여부
- 콘솔 오류 여부
- 저장 후 다시 불러왔을 때 동일 증상 여부
- 화면 캡처 또는 녹화 여부

앱 내부에 피드백 전송 기능을 넣는 것은 2.11 첫 구현 범위가 아니다.
첫 구현에서는 Manual 또는 테스트 안내 문서에 위 항목을 안내한다.

## 10. 오류 처리

Examples 관련 오류는 앱 전체를 중단시키지 않는다.

오류 유형:

- manifest fetch 실패
- manifest 구조 검증 실패
- score fetch 실패
- score JSON 크기 초과
- score JSON parse 실패
- ScoreFile validation 실패
- 예제 score의 라이선스 정보 누락

오류 표시는 left status line과 Examples dialog notice를 함께 사용한다.
score 로드 실패 시 기존 score와 playback state를 가능한 보존한다.
단, fetch 중 사용자가 명시적으로 load를 시작한 경우에는 busy 상태에서 중복 load를 막는다.

## 11. 검증

구현 후 확인할 항목:

- 앱 초기 JavaScript bundle에 실제 예제 악보 JSON이 포함되지 않는다.
- Examples 메뉴를 열기 전에는 개별 예제 Score JSON을 fetch하지 않는다.
- manifest만 로드했을 때 score는 교체되지 않는다.
- 예제 선택 후 로드한 Score JSON이 기존 JSON Load와 같은 검증 경로를 통과한다.
- 깨진 manifest는 현재 score를 망가뜨리지 않는다.
- 깨진 예제 Score JSON은 현재 score를 유지한 채 오류를 표시한다.
- Supabase provider 실패 시 local provider fallback이 가능하다.
- Manual 버튼이 비활성 상태로 남지 않는다.
- Manual에서 YouTube 재생목록 링크가 없어도 기본 안내 문서는 열 수 있다.

프로젝트의 정식 build 명령이 유지되는 경우 `npm run build`를 통과해야 한다.
예제 manifest/parser helper를 추가한다면 해당 helper의 최소 검증 테스트를 추가한다.

## 12. 구현 순서

권장 구현 순서:

1. `public/examples/manifest.json`과 작은 예제 1개를 추가한다.
2. example manifest type과 validator를 만든다.
3. local example provider를 만든다.
4. Examples dialog와 File 메뉴 버튼을 연결한다.
5. 선택 예제 로드를 기존 score load pipeline에 연결한다.
6. Manual 정적 문서를 추가하고 Help 메뉴 Manual 버튼을 활성화한다.
7. Manual 안에 YouTube 재생목록 링크 placeholder를 둔다.
8. Supabase provider는 local provider가 안정화된 뒤 선택적으로 추가한다.

Supabase provider는 2.11의 구조 목표에 포함되지만, 첫 안정화 패치에서 반드시 동시에 배포할 필요는 없다.
