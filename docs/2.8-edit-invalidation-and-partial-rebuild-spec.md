# 2.8. Edit Invalidation / Partial Rebuild 명세

## 1. 문서 목적

본 문서는 `회귀 코드`에서 score cell 편집 후 어떤 런타임 산출물과 canvas layer를 다시 계산하거나 다시 그려야 하는지 정의한다.

첫 목표는 긴 테스트 악보 제작 중 발생하는 지연을 줄이기 위해, note cell 편집과 global cell 편집의 갱신 집합을 분리하는 것이다.

본 문서는 `docs/1.8-parser-analyzer-pipeline-spec.md`, `docs/2.1-canvas-renderer-module-spec.md`, `docs/2.2-ui-state-edit-mode-spec.md`, `docs/2.3-audio-playback-module-spec.md`의 후속 구현 명세이다.

## 2. 기본 원칙

score 편집 후 모든 산출물을 항상 다시 만들 필요는 없다.

```text
ScoreFile
  -> RuntimeDocument / ScoreIndexes
  -> ParsedScoreDocument
  -> AnalysisResult
  -> Canvas render input/items
  -> Canvas layers
  -> Audio schedule / playback runtime
```

이 파이프라인에서 변경 종류별로 영향을 받는 범위는 다르다.

- note cell 편집은 track note event, note-derived marker, note layer, audio note schedule에 직접 영향을 준다.
- global cell 편집은 timing/dynamics timeline, global marker, global text, audio time mapping/dynamics schedule에 직접 영향을 준다.
- base grid/layout layer는 score 구조가 바뀔 때만 영향을 받는다.
- `ScoreFile`은 계속 진실 원천이며, partial rebuild는 공식 저장 구조를 바꾸지 않는다.

## 3. 편집 종류

본 문서에서 다루는 cell 편집은 score 구조를 바꾸지 않는 `rawText` upsert/delete이다.

```ts
type EditInvalidationKind =
  | "noteCell"
  | "globalCell"
  | "mixedCell"
  | "structure";
```

구분 기준:

- `noteCell`
  - 모든 편집 대상의 `rowKind`가 `"note"`이다.
  - `track.cells`만 바뀐다.
- `globalCell`
  - 모든 편집 대상의 `rowKind`가 `"global"`이다.
  - `globalLines.cells`만 바뀐다.
- `mixedCell`
  - 같은 내부 batch 안에 note와 global 편집이 섞여 있다.
  - 일반 pointer drag UI에서는 만들지 않는다.
- `structure`
  - `globalLines.columnCount`, `layout.rowDefinitions`, track 목록, rowId 참조, row height, row order, reverseRows, zoom, DPR처럼 좌표 또는 구조가 바뀐다.

`gap` row 편집은 score cell 편집 대상이 아니며, 현재 정책에서는 blocked edit으로 처리한다.

사용자 pointer drag 편집은 drag 시작 시점의 `rowKind`를 batch 종류로 잠근다.
따라서 note row에서 시작한 drag는 note row hit만 edit batch에 포함하고, global/gap row로 넘어간 hit는 무시한다.
global row에서 시작한 drag도 같은 방식으로 global row hit만 batch에 포함한다.
이 정책은 드래그 중 의도치 않게 전역 셀과 노트 셀을 동시에 수정하는 상황을 막고, partial rebuild invalidation을 단순하게 유지하기 위한 것이다.

## 4. 산출물 분류

런타임 산출물은 아래 그룹으로 나눈다.

| 그룹 | 내용 | 주 영향 편집 |
| --- | --- | --- |
| Score structure | `RuntimeDocument`, `ScoreIndexes`, renderer base rows/columns | structure |
| Parsed note cells | `ParsedScoreDocument.cellsByTrackAndCol` | noteCell |
| Parsed global cells | `ParsedScoreDocument.globalCellsByKindAndCol` | globalCell |
| Track analysis | `AnalysisResult.trackResults` | noteCell |
| Timing analysis | `AnalysisResult.timingTimeline` | globalCell 중 bpm/beatsPerBar/stepsPerBeat |
| Dynamics analysis | `AnalysisResult.dynamicsTimeline` | globalCell 중 dynamics |
| Note render items | note, mute, note text, trem/vib | noteCell |
| Note marker items | gliss, gliss orphan anchor, tuplet container | noteCell |
| Global render items | global rawText text | globalCell |
| Global marker items | beat/bar, BPM change, dynamics guide | globalCell |
| Base/layout layer | layout label, score grid background, row/column grid | structure |
| Audio schedule | note schedule, tick-second map, dynamics automation | noteCell 또는 globalCell |
| Playback runtime | controller/scheduler/backend 상태 | audio schedule 변경 또는 structure reset |
| YouTube sync | iframe player, video id, offset | YouTube metadata edit만 직접 영향 |

## 5. Canvas layer 갱신 정책

현재 canvas layer는 `layout`, `base`, `marker`, `note`로 구성된다.

### 5.1. Base / layout layer

다음 경우에만 `layout`과 `base` layer를 다시 그린다.

- score load
- Clear All
- Expand Right / Trim Right
- layout apply 또는 layout preset apply
- reverse row view 변경
- zoom 변경
- DPR 또는 canvas target 크기 변경
- renderer row/column 좌표가 바뀌는 구조 변경

다음 경우에는 다시 그리지 않는다.

- note cell `rawText` upsert/delete
- global cell `rawText` upsert/delete
- active track toggle
- YouTube video/offset reload
- 일반 musicData edit

### 5.2. Note layer

note layer는 note-derived visual을 그린다.

포함 대상:

- note rectangle
- note display text anchor
- mute text
- tremolo chop
- vibrato wave
- tuplet container overlay
- global rawText text는 1차 구현에서는 기존 layer 배치를 유지할 수 있으나, 최종 분리 목표에서는 global overlay로 분리한다.

note cell 편집 시 note layer를 다시 그린다.
global cell 편집 시 note event 자체는 다시 그릴 필요가 없다.

단, 현재 구현에서 global rawText text가 note layer에 함께 그려지는 동안에는 global cell 편집 시 note layer를 다시 그리거나, global text 전용 overlay drawing 함수를 분리해야 한다.
1차 최적화 목표는 global text drawing을 note event drawing과 분리해 global cell 편집이 note event redraw를 유발하지 않게 하는 것이다.

### 5.3. Marker layer

marker layer는 두 종류로 나눈다.

```ts
type CanvasMarkerInvalidationGroup =
  | "globalMarker"
  | "noteMarker";
```

global marker:

- beat line
- bar line
- BPM change line
- dynamics guide

note marker:

- gliss line
- gliss orphan anchor
- tuplet container

global cell 편집 시 global marker를 다시 그린다.
note cell 편집 시 note marker를 다시 그린다.

현재 구현처럼 두 marker 종류가 `markerItems` 하나에 섞여 있으면, 1차 구현에서는 marker layer 전체 redraw를 허용한다.
다만 renderer item builder는 다음 단계에서 `globalMarkerItems`와 `noteMarkerItems`를 분리해야 한다.

## 6. 편집별 갱신 집합

### 6.1. Note cell 편집

note cell 편집 후 갱신 대상:

| 대상 | 갱신 여부 | 이유 |
| --- | --- | --- |
| `ScoreFile.tracks[].cells` | 예 | 원본 rawText 변경 |
| `RuntimeDocument` / `ScoreIndexes` | 1차 구현 예, 후속 선택 | 현재 인덱스 구조는 score clone 후 재생성이 단순함 |
| parsed note cells | 예 | note parser 결과 변경 |
| parsed global cells | 아니오 | global rawText 불변 |
| `trackResults` | 예 | note/hold/gliss/tuplet 구조 변경 가능 |
| `timingTimeline` | 아니오 | BPM/박자/step 불변 |
| `dynamicsTimeline` | 아니오 | dynamics 불변 |
| note render items | 예 | note 표시 변경 |
| note marker items | 예 | gliss/tuplet/mute 등 변경 가능 |
| global marker items | 아니오 | timing/dynamics visual 불변 |
| global text items | 아니오 | global rawText 불변 |
| layout/base layer | 아니오 | 좌표와 grid 불변 |
| audio schedule | 예 | 발음 이벤트 변경 |
| playback runtime | 예 | schedule 교체 필요 |
| YouTube player | 아니오 | score time follower일 뿐 직접 영향 없음 |

주의:

- hold, gliss, tuplet은 주변 cell 또는 group 전체에 영향을 줄 수 있으므로 track analysis의 실제 partial 범위는 analyzer가 확장한다.
- active track filter는 score mutation 범위와 별도이다. 편집은 active track 정책을 따르되, 렌더 item alpha만 바뀌는 경우 parse/analyze는 다시 하지 않는다.

### 6.2. Global cell 편집

global cell 편집 후 갱신 대상:

| 대상 | 갱신 여부 | 이유 |
| --- | --- | --- |
| `ScoreFile.globalLines.cells` | 예 | 원본 rawText 변경 |
| `RuntimeDocument` / `ScoreIndexes` | 1차 구현 예, 후속 선택 | 현재 global index 재생성이 단순함 |
| parsed global cells | 예 | global parser 결과 변경 |
| parsed note cells | 아니오 | note rawText 불변 |
| `trackResults` | 아니오 | note/gliss/tuplet 구조 자체는 불변 |
| `timingTimeline` | bpm/beatsPerBar/stepsPerBeat 편집 시 예 | tick-second mapping과 beat/bar visual 변경 |
| `dynamicsTimeline` | dynamics 편집 시 예 | dynamics guide와 volume automation 변경 |
| note render items | 아니오 | note 표시 구조 불변 |
| note marker items | 아니오 | gliss/tuplet 구조 불변 |
| global marker items | 예 | timing/dynamics visual 변경 가능 |
| global text items | 예 | global rawText 표시 변경 |
| layout/base layer | 아니오 | 좌표와 grid 불변 |
| audio schedule | 예 | tick-second map 또는 dynamics automation 변경 가능 |
| playback runtime | 예 | schedule 또는 mapper 교체 필요 |
| YouTube player | 위치 재정렬만 선택 | scoreSeconds는 유지하되 audio duration/mapper가 바뀔 수 있음 |

주의:

- BPM, beatsPerBar, stepsPerBeat는 timing 계열로 묶는다.
- dynamics는 dynamics timeline만 갱신할 수 있다.
- global 편집은 note event 구조를 다시 만들 필요가 없지만, audio schedule은 note event와 새 timing/dynamics를 결합해 다시 만들어야 한다.

### 6.3. Mixed cell batch 편집

같은 batch에 note와 global 편집이 섞이면 noteCell과 globalCell 갱신 집합의 합집합을 적용한다.

일반 사용자 pointer drag에서는 mixed batch를 만들지 않는다.
mixed batch는 다음과 같은 방어적 내부 경로에서만 발생할 수 있다.

- 테스트 코드가 여러 종류의 edit을 한 번에 호출한 경우
- 향후 macro edit 또는 script edit이 여러 rowKind를 묶어 전달한 경우
- 기존 코드 경로가 note/global edit을 분리하지 못한 상태로 batch를 넘긴 경우

1차 구현에서는 mixed batch를 기존 full rebuild 경로로 보내도 된다.
후속 partial rebuild에서는 mixed batch를 UI 단계에서 금지하거나, 내부에서 note batch와 global batch로 분리한 뒤 마지막에 audio schedule과 필요한 canvas layer를 한 번만 갱신한다.

### 6.4. Structure 변경

structure 변경은 전체 rebuild 대상이다.

예:

- score load
- Clear All
- Expand Right / Trim Right
- layout apply
- layout preset apply
- row order 또는 row height 변경
- column width 또는 zoom 변경

structure 변경 후 갱신 대상:

- `RuntimeDocument`
- `ParsedScoreDocument`
- `AnalysisResult`
- `CanvasRenderInput`
- 모든 render item
- 모든 canvas layer
- audio schedule
- playback runtime reset
- YouTube 입력 동기화 또는 player reset은 해당 기능 정책을 따른다.

## 7. 1차 구현 단계

1차 구현은 partial parse/analyze까지 한 번에 완성하지 않는다.

우선순위:

1. 편집 batch를 `noteCell`, `globalCell`, `mixedCell`, `structure`로 분류한다.
2. pointer drag batch는 시작 `rowKind`와 다른 row hit를 포함하지 않도록 제한한다.
3. renderer item을 global group과 note group으로 나눌 수 있는 타입/빌더를 추가한다.
4. `renderCanvasScore()`를 전체 redraw 진입점으로 유지하면서, layer별 redraw 진입점을 추가한다.
5. note cell 편집 시 base/layout redraw를 생략한다.
6. global cell 편집 시 base/layout redraw와 note event redraw를 생략한다.
7. audio schedule은 안정성을 위해 note/global cell 편집 모두에서 다시 만든다.
8. parse/analyze는 초기에는 full rebuild를 유지할 수 있으나, render layer redraw 집합은 편집 종류에 맞춰 줄인다.

이 단계의 목표는 “계산량 최적화”보다 “무효화 집합 분리”이다.
분리가 끝나면 partial parse/analyze가 들어갈 위치가 명확해진다.

## 8. 후속 partial parse / analysis 단계

### 8.1. Note partial

note partial은 다음 순서로 확장한다.

```text
note edit
  -> changed track/col set
  -> parse note range
  -> merge ParsedScoreDocument.cellsByTrackAndCol
  -> analyze affected track range
  -> replace affected track events
  -> rebuild note render item group
  -> rebuild audio schedule
```

범위 확장은 `1.7`의 hold/gliss/tuplet 규칙을 따른다.

1차 구현에서는 note cell 편집이 발생하면 해당 track의 note-derived render group을 넓게 갱신해도 된다.
즉 note rectangle, mute text, trem/vib, gliss marker, tuplet container를 하나의 note visual group으로 취급한다.

2차 구현에서는 재분석 결과의 event 종류에 따라 더 구체적인 invalidation group을 만들 수 있다.

```ts
type NoteEventInvalidationGroup =
  | "noteBody"
  | "noteEffect"
  | "muteText"
  | "glissMarker"
  | "tupletContainer";
```

분류 기준:

- `noteBody`
  - `NoteEvent`의 time, display row, pitch, text anchor, display shape가 바뀐 경우
  - note rectangle과 note text를 다시 만든다.
- `noteEffect`
  - `NoteEvent.effects`의 trem/vib 구간이 바뀐 경우
  - note rectangle 자체는 유지 가능하지만, 1차 구현에서는 note item 전체를 다시 그려도 된다.
- `muteText`
  - `MuteEvent`가 추가, 삭제, 변경된 경우
  - mute text item만 다시 만들 수 있다.
- `glissMarker`
  - `GlissEvent` 또는 orphan gliss anchor가 추가, 삭제, 변경된 경우
  - gliss marker item만 다시 만들 수 있다.
- `tupletContainer`
  - `TupletGroupEvent` 또는 `TupletExtendGroupEvent`가 추가, 삭제, 변경된 경우
  - tuplet container item만 다시 만들 수 있다.

예를 들어 수정 범위 안에 `GlissEvent` 변화가 없고 일반 `NoteEvent`의 text만 바뀌었다면 gliss marker rebuild를 생략할 수 있다.
반대로 수정 범위가 gliss anchor만 바꾸는 경우에는 note body와 gliss marker가 함께 바뀔 수 있으므로 두 group을 갱신한다.

이 2차 최적화는 다음 조건이 갖춰진 뒤에 적용한다.

- analyzer partial 결과가 “삭제된 이벤트 / 추가된 이벤트 / 변경된 이벤트” 또는 동등한 diff 정보를 제공한다.
- renderer item builder가 event kind별 item group을 독립적으로 만들 수 있다.
- layer redraw 함수가 같은 canvas layer 안에서도 필요한 group을 다시 합성할 수 있다.

이 조건이 충족되기 전에는 note edit을 note visual group 전체 갱신으로 처리하는 편이 안전하다.

### 8.2. Global partial

global partial은 다음 순서로 확장한다.

```text
global edit
  -> changed global kind/col set
  -> parse global range
  -> merge ParsedScoreDocument.globalCellsByKindAndCol
  -> rebuild timingTimeline and/or dynamicsTimeline
  -> rebuild global marker/text item group
  -> rebuild audio schedule
```

global partial은 `trackResults`를 유지한다.
단, analyzer 구조가 향후 global 값에 따라 note event 자체를 바꾸는 기능을 도입하면 해당 기능은 별도 invalidation rule을 추가해야 한다.

## 9. Renderer item 분리 목표

후속 구현의 renderer input은 아래처럼 분리하는 방향을 목표로 한다.

```ts
type CanvasAnalyzedRenderInput = CanvasRenderInput & {
  noteItems: CanvasNoteRenderItem[];
  muteItems: CanvasMuteRenderItem[];
  globalTextItems: CanvasGlobalTextRenderItem[];
  globalMarkerItems: CanvasMarkerItem[];
  noteMarkerItems: CanvasMarkerItem[];
};
```

`CanvasMarkerItem` 자체를 유지하더라도 builder 단계에서 global/note group을 분리한다.

분류:

- `beat`, `bar`, `bpmChange`, `dynamicsGuide` -> `globalMarkerItems`
- `gliss`, `glissOrphanAnchor`, `tupletContainer` -> `noteMarkerItems`

이 분리는 draw order를 바꾸기 위한 것이 아니라 invalidation scope를 줄이기 위한 것이다.
시각적 z-order가 필요하면 renderer 진입점에서 같은 순서로 여러 group을 그린다.

## 10. 검증 기준

단위 검증:

- note-only edit batch는 invalidation kind가 `noteCell`이어야 한다.
- global-only edit batch는 invalidation kind가 `globalCell`이어야 한다.
- pointer drag가 시작 rowKind와 다른 row hit를 batch에 포함하지 않아야 한다.
- 내부 note/global mixed batch는 invalidation kind가 `mixedCell`이거나 note/global batch로 분리되어야 한다.
- active track toggle은 parse/analyze 없이 render item alpha만 갱신해야 한다.
- YouTube reload는 parser/analyzer/render item을 갱신하지 않아야 한다.

브라우저 수동 검증:

- note cell 편집 후 base grid가 흔들리거나 다시 깜빡이지 않는다.
- global cell 편집 후 note rectangle이 다시 깜빡이지 않는다.
- BPM/박자/step 수정 후 beat/bar marker와 seek/playback timing이 갱신된다.
- dynamics 수정 후 dynamics guide와 playback volume automation이 갱신된다.
- layout apply, expand, trim은 여전히 전체 redraw와 playback reset을 수행한다.

성능 확인:

- 1000열 이상 테스트 악보에서 note drag edit 시 기존보다 canvas redraw 시간이 줄어드는지 확인한다.
- 1000열 이상 테스트 악보에서 global row 수정 시 note event redraw가 생략되는지 확인한다.
- 이 문서의 1차 구현만으로 부족하면 `1.8`과 `1.7` 기준에 따라 partial parse/analyze를 추가한다.

## 11. 현재 구현과의 차이

현재 구현은 `buildRuntimeArtifacts()`와 `renderCanvasScore()`가 각각 parse/analyze/render item 생성과 모든 canvas layer redraw를 통합 수행한다.

따라서 이 명세를 반영하려면 다음 분리가 필요하다.

- edit batch invalidation kind 계산
- render item group 분리
- layer별 redraw 함수 추가
- app state에 마지막 정상 `CanvasScoreLayout`을 보존하고 재사용하는 경로 추가
- audio schedule reset과 canvas redraw를 별도 단계로 분리
