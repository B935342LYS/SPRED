# 1.50 Practice Judge Modes Implementation and Trem Plan

## 1. Scope

이번 메모는 practice mode의 최근 구현 상태와 다음 Trem bonus 판정 설계 방향을 기록한다.

기록 대상:

- practice mode 진입 시 자동 Fit Height UX 개선
- Practice Pro/Easy Mode 1차 구현
- Rules dialog 폭 조정
- Trem bonus 1차 구현 후보

## 2. Current Implementation Status

### 2.1. Practice Fit Height

practice mode 진입 시 score가 현재 화면 높이에 맞도록 자동 Fit Height를 실행한다.

구현 위치:

- `regression-code/src/app/app_view_binding.ts`
  - `fitScoreHeight()`에 `silent` option을 추가했다.
  - 수동 Fit Height 버튼은 기존처럼 status message를 표시한다.
  - practice mode 내부 호출은 status message를 덮지 않도록 `silent: true`를 사용한다.
- `regression-code/src/app/game/game_binding.ts`
  - practice panel을 연 직후와 microphone permission이 허용되어 ready가 된 직후 `fitScoreHeight()`를 호출한다.

목적:

- 작은 화면이나 노트북 화면비에서 practice panel이 열리면 score 영역이 줄어들 수 있다.
- 사용자가 매번 수동 Fit Height를 누르지 않아도 note row 전체가 화면에 들어오게 한다.

### 2.2. LG Gram Microphone Issue

LG Gram 노트북에서 초록 pitch dot은 보이지만 score가 오르지 않는 현상이 있었다.

처음에는 내장 마이크 특성 때문에 onset 감지가 실패한다고 보고 RMS 급상승을 attack 후보로 보는 보강을 시도했다.

이후 사용자 확인 결과, 문제는 microphone/pitch 판정 자체가 아니라 순간적인 lag 상승으로 발생한 것으로 판단했다.

따라서 RMS attack 보강은 롤백했고, 현재 timing onset 정책은 다시 다음 두 조건 중심이다.

- unvoiced -> voiced 전환
- pitch class가 충분히 바뀐 voiced frame

### 2.3. Practice Pro Mode

Pro Mode 1차 구현이 완료되었다.

구현 내용:

- `PracticeJudgeMode = "standard" | "pro"` 세션 상태를 추가했다.
- 기본값은 `standard`이다.
- score JSON과 localStorage에는 저장하지 않는다.
- 새로고침하면 `standard`로 돌아간다.
- score load, Clear All, layout 변경은 현재 Pro/Standard 선택을 직접 초기화하지 않는다.
- practice panel의 `Sync` 오른쪽에 `Pro` toggle button을 추가했다.
- `off` / `ready` 상태에서만 toggle 가능하다.
- `preparing` / `countdown` / `playing` / `paused` / `finished` 상태에서는 toggle을 잠근다.
- Pro Mode가 켜져 있으면 practice panel에 red accent를 적용한다.
- result dialog에 `Mode: Standard` 또는 `Mode: Pro`를 표시한다.
- Rules dialog에 Standard/Pro pitch/timing threshold를 함께 표시한다.

판정 정책:

| Mode | Perfect | Ok | Bad | Miss |
| --- | ---: | ---: | ---: | ---: |
| Standard | `< 50 cent` | `< 100 cent` | `< 200 cent` | `>= 200 cent` |
| Pro | `< 30 cent` | `< 60 cent` | `< 100 cent` | `>= 100 cent` |

| Mode | No timing text | Early/Late | Bad downgrade | Miss downgrade |
| --- | ---: | ---: | ---: | ---: |
| Standard | `< 80ms` | `80-149ms` | `150-249ms` | `>= 250ms` |
| Pro | `< 50ms` | `50-99ms` | `100-149ms` | `>= 150ms` |

Combo 정책:

- Standard: scoreEligible `Perfect` / `Ok` / `Bad`가 combo를 이어간다.
- Pro: scoreEligible `Perfect` / `Ok`만 combo를 이어가며, `Bad` / `Miss`는 combo를 초기화한다.
- 두 mode 모두 combo는 score multiplier로 쓰지 않는다.

검증:

- `npm run test:game`
- `npm run typecheck`
- `npm run build`

### 2.4. Practice Easy Mode

실사용자 테스트에서 음정을 끊어 치는 것이 어렵다는 피드백이 들어와 Easy Mode를 추가했다.

구현 내용:

- `PracticeJudgeMode = "easy" | "standard" | "pro"`로 확장했다.
- practice panel head의 `Pro` toggle을 `Easy` / `Std` / `Pro` segmented buttons로 바꿨다.
- `off` / `ready` 상태에서만 mode를 바꿀 수 있고, playback 중에는 잠근다.
- Easy Mode가 켜져 있으면 practice panel에 blue accent를 적용한다.
- result dialog에 `Mode: Easy`를 표시한다.
- Rules dialog에 Easy pitch-only 정책을 추가했다.

판정 정책:

- Easy pitch threshold는 Standard와 동일하게 둔다.
- Easy에서는 timing judge result를 만들지 않고 pitch label을 그대로 사용한다.
- Easy에서는 attack-required score block을 적용하지 않는다.
- 따라서 같은 음을 끊지 않고 이어서 연주해도 현재 target의 pitch class가 맞으면 scoreEligible sample로 인정한다.
- Easy와 Standard는 `Perfect` / `Ok` / `Bad`가 combo를 이어가고, `Miss`에서 combo가 초기화된다.

### 2.5. Rules Dialog Width

Rules dialog의 표 문장이 줄바꿈되어 읽기 불편했으므로 dialog 폭을 키웠다.

구현 내용:

- `practice-rules-dialog` 폭을 `min(920px, calc(100vw - 32px))`로 조정했다.
- `practice-sync-dialog`는 기존 폭을 유지한다.
- Rules table cell에 `white-space: nowrap`을 적용했다.
- 좁은 화면에서는 dialog form 내부 가로 스크롤을 허용한다.

## 3. Trem Bonus Direction

현재 gliss와 vib는 effect bonus가 연결되어 있고, trem은 UI count와 result field만 준비된 상태이다.

사용자 제안:

> trem 구간 내 일정 시간 동안 연속적인 attack을 감지하면 attack 시도마다 가산점을 준다.

이 방향은 현재 practice architecture와 잘 맞는다.

이유:

- tremolo는 pitch contour만으로 판정하기 어렵다.
- 이미 playback binding에는 timing 판정용 onset 후보가 있다.
- trem 판정은 별도의 무거운 실시간 분석보다 기존 onset 후보와 최근 pitch frame buffer를 재사용하는 편이 안전하다.
- effect bonus는 실패 감점을 만들지 않는다는 기존 정책과도 맞는다.

## 4. Proposed Trem Runtime Target

`collectGameEffectBonusTargets()`에 trem target을 추가한다.

후보 타입:

```ts
type GameTremBonusTarget = {
  kind: "trem";
  targetId: string;
  trackId: TrackId;
  startSeconds: number;
  endSeconds: number;
  targetMidi: number;
  targetCentOffset: number;
  source: "explicit" | "rapidRepeat";
};
```

생성 기준:

- active track의 `NoteEvent.effects`를 순회한다.
- `effect.trem !== null`인 segment를 trem target으로 만든다.
- 첫 구현은 병합 없이 segment 단위로 target을 만든다.
- active track에서 explicit trem modifier가 없는 1tick 이하 note가 같은 pitch class로 gap 없이 3개 이상 연속되면 `source: "rapidRepeat"` trem target을 만든다.
- rapid repeat target 수집은 practice 시작 전 target collection에서 한 번만 수행한다.
- 2개 반복이나 pitch가 섞인 1tick 선율은 rapid repeat trem으로 보지 않는다.
- `division`은 target에 포함하지 않고, interval matching에도 사용하지 않는다.

`targetId` 후보:

```text
${noteEvent.eventId}:trem:${effectIndex}
```

## 5. Proposed Trem Judge Rule

### 5.1. Valid Trem Attack

trem attack 후보는 다음 조건을 만족해야 한다.

- onset 후보가 target의 `[startSeconds, endSeconds]` 안에 있다.
- onset 주변 최신 pitch frame이 voiced이다.
- 음정은 안정적이라고 가정하고 trem 전용 pitch tolerance 분기를 만들지 않는다.
- 같은 target에서 이미 처리한 onset id가 아니다.
- 직전 accepted trem attack과 너무 가깝지 않다.

현재 판단:

- Pro Mode에서도 effect bonus 정책은 동일하게 유지한다.
- trem은 pitch 정확도보다 반복 attack 여부를 보는 보너스이므로, 별도 pitch 흔들림 보정이나 mode별 pitch 조건을 만들지 않는다.

### 5.2. Continuous Attack Requirement

단일 attack 하나만으로 바로 `Trem!`을 주면 일반 note 시작 attack과 구분이 약하다.

따라서 target별 streak을 둔다.

후보 조건:

- 최소 accepted attack count: `2`
- 직전 accepted attack과의 간격이 `0.06s ~ 0.45s` 안에 있으면 같은 streak으로 본다.
- 간격이 너무 길면 streak을 reset한다.
- 짧은 trem segment도 최소 2회 attack 조건을 유지한다.

첫 구현 추천:

```text
acceptedAttackCount >= 2부터 Trem! 표시와 bonus 지급
그 이후 같은 streak의 accepted attack마다 bonus 지급
```

이렇게 하면 사용자가 실제로 반복 attack을 시작했다는 근거가 생긴 뒤부터 각 attack 시도에 보상을 줄 수 있다.

### 5.3. Bonus Amount

gliss는 interval당 `trackDifficulty * 0.25`, vib는 segment당 `trackDifficulty * 0.5`를 사용한다.

trem은 attack마다 지급하므로 너무 크면 score가 과도하게 오른다.

확정 후보:

```text
tremAttackBonus = trackDifficulty * 0.12
```

현재 판단:

- 첫 구현은 `0.12`를 단일 상수로 둔다.
- mode, target 길이, division별 bonus 분기는 만들지 않는다.
- 실제 result screen에서 비중이 과하거나 약하면 단일 상수만 조정한다.

## 6. Playback Binding Integration

현재 `app_playback_binding.ts`는 다음 runtime 상태를 이미 가진다.

- `timingOnsetCandidates`
- `consumedTimingOnsetIds`
- `effectBonusTargets`
- `effectFrames`
- `judgedEffectIntervalIds`
- `failedEffectTargetIds`

trem 구현에는 별도 runtime state가 필요하다.

후보:

```ts
type TremRuntimeState = {
  acceptedOnsetIds: Set<number>;
  lastAcceptedScoreSeconds: number | null;
  streakCount: number;
  rewardedOnsetIds: Set<number>;
};

const tremRuntimeByTargetId = new Map<string, TremRuntimeState>();
```

흐름:

1. `collectGameEffectBonusTargets()`가 trem targets를 포함한다.
2. scoring RAF에서 `updatePracticeTimingInput()`이 onset 후보를 계속 만든다.
3. `updatePracticeEffectBonus()`가 현재 score time 근처의 trem target을 확인한다.
4. target 안의 아직 보상하지 않은 onset 후보를 검사한다.
5. accepted attack streak 조건을 만족하면 `GameEffectBonusResult(kind: "trem")`을 만든다.
6. `applyGameEffectBonus()`와 `showGameEffectBonusOverlay()`는 기존 구조를 그대로 사용한다.

주의:

- timing 판정에서 onset을 consume하더라도 trem bonus에서는 같은 onset id를 별도로 참조할 수 있어야 한다.
- 일반 note score와 trem bonus는 서로 다른 보상 축이므로 같은 onset이 note attack credit과 trem bonus 모두에 쓰일 수 있다.
- 대신 trem target별 `rewardedOnsetIds`로 중복 보상만 막는다.

## 7. Edge Cases

### 7.1. Long Sustain With No Attack

사용자가 trem 구간에서 음을 계속 유지하기만 하면 `Trem!`을 주지 않는다.

이유:

- trem은 반복 attack 또는 반복 energy change가 핵심이다.
- sustain은 기본 pitch scoring에서 이미 보상받을 수 있다.

### 7.2. Very Short Trem Segment

짧은 trem 구간도 최소 2회 attack 조건을 유지한다.

이유:

- 사용자 결정에 따라 attack은 반드시 여러 번 있어야 한다.
- 첫 valid attack만으로 보상하면 일반 note 시작 attack과 trem attack의 구분이 약해진다.
- 짧은 구간 예외를 만들지 않아 runtime state와 테스트 조건을 단순하게 유지한다.

### 7.3. Division Matching

첫 구현은 trem `division`을 정확히 맞추지 않는다.

이유:

- 브라우저 microphone onset latency와 RMS fluctuation 때문에 division 정확도 판정은 불안정할 가능성이 높다.
- 현재 practice mode는 정밀 리듬게임보다 연습 보조 성격이 강하다.

사용자 결정에 따라 division은 후속 rough matching 후보로도 올리지 않고, 첫 구현에서는 완전히 사용하지 않는다.

### 7.4. Pro Mode

현재 정책은 Pro Mode에서도 effect bonus threshold를 동일하게 둔다.

따라서 trem 1차 구현도 Standard/Pro 동일 조건으로 시작한다.

첫 구현에서는 Pro Mode 전용 trem 분기를 만들지 않는다.

## 8. Proposed Implementation Order

1. `GameTremBonusTarget` 타입 추가
2. `collectGameEffectBonusTargets()`에서 `effect.trem` segment와 rapid repeat run을 target으로 수집
3. `judgeTremAttackBonus()` 함수 추가
4. playback binding에 target별 trem runtime state 추가
5. `updatePracticeEffectBonus()`에서 trem target 분기 연결
6. `test_game.ts`에 다음 케이스 추가
   - active track trem target만 수집
   - 1tick 이하 같은 pitch class note 3개 이상 연속은 rapid repeat trem target 생성
   - 2개 반복과 mixed-pitch 1tick run은 rapid repeat trem target 미생성
   - 첫 attack 하나만으로는 bonus 없음
   - 두 번째 연속 attack부터 `Trem!` bonus 생성
   - 짧은 trem segment도 첫 attack 하나만으로는 bonus 없음
   - 같은 onset은 중복 보상하지 않음
   - 긴 간격 후 streak reset
   - Pro Mode에서도 effect bonus amount 동일
   - division 값은 bonus 여부에 영향 없음
7. `npm run test:game`
8. `npm run typecheck`
9. `npm run build`

## 9. Current Recommendation

Trem 1차 구현은 사용자의 제안처럼 attack 기반으로 가는 것이 좋다.

다만 “구간 내 일정 시간 동안 연속적인 attack”을 바로 시간 길이로만 보지 말고, 다음 두 조건을 함께 쓰는 편이 안정적이다.

- accepted attack count가 최소 2개 이상
- accepted attack 사이 간격이 너무 짧거나 너무 길지 않음

그 뒤부터는 attack 시도마다 작은 bonus를 지급한다.

초기 추천값:

- min attack interval: `60ms`
- max attack interval: `450ms`
- min streak count: `2`
- bonus: `trackDifficulty * 0.12`

이 값들은 실제 microphone 테스트 후 조정하되, 조정이 필요하면 단일 상수를 바꾸고 mode, target 길이, division별 분기는 만들지 않는다.
