# 2.11. Example Score and Manual Support 명세

## 1. 문서 목적

본 문서는 `회귀 코드` 안정판의 후반 보조 기능인 예제 악보 지원과 Manual 지원을 정의한다.

목표는 다음이다.

- 테스터가 별도 파일을 준비하지 않아도 앱에서 예제 악보를 선택해 실사용 테스트를 시작할 수 있게 한다.
- 예제 악보가 여러 개로 늘어나도 초기 페이지 로드와 JavaScript bundle 크기를 불필요하게 키우지 않는다.
- GitHub Pages 배포 앱에서 Supabase Edge Function을 통해 허락된 테스터에게 예제 목록을 제공한다.
- Help 메뉴의 Manual을 실제 사용자 안내 경로로 활성화한다.
- 실사용 테스트 피드백 수집에 필요한 기본 기록 항목을 정한다.

본 문서는 `docs/2.0-ui-mvp-spec.md`의 File / Help 메뉴 후속 명세이며, `docs/1.3-score-json-format.md`의 Score JSON 저장 포맷을 바꾸지 않는다.

## 2. 기본 원칙

예제 악보 JSON은 앱의 TypeScript source bundle에 직접 포함하지 않는다.

```text
app boot
  -> bundled default template 1개만 즉시 로드
  -> example manifest는 필요 시 fetch
  -> selected example score JSON은 사용자가 선택했을 때 fetch
  -> validate / parse / analyze / render
```

3~4분 곡의 Score JSON이 200~300 KiB가 될 수 있으므로, 여러 예제 악보를 `src/assets`에서 `?raw` import하는 방식은 사용하지 않는다.
예제 악보 파일을 여러 개 제공하더라도 초기 앱 로드는 예제 데이터를 읽지 않고, 사용자가 Examples 메뉴를 열 때까지 manifest 읽기를 미룬다.

Supabase는 앱의 핵심 저장소가 아니라 읽기 전용 예제 공급원으로 취급한다.
Supabase 장애, 네트워크 실패, CORS 실패, 응답 검증 실패가 있으면 현재 악보 상태를 유지하고 Examples dialog에 오류를 표시한다.

Manual은 정적 HTML 문서를 기본 안내로 두고, 사용법 녹화 영상 또는 YouTube 재생목록은 보조 링크로 둔다.
정적 Manual은 GitHub Pages 앱과 함께 배포되어 네트워크 정책, 영상 비공개, 영상 삭제, 광고, 외부 페이지 차단에 덜 의존한다.

## 3. 범위

2.11 첫 구현 범위:

- File 메뉴 또는 별도 Help-adjacent 메뉴에서 Examples 진입점 추가

- Supabase 원격 manifest 읽기 전용 연결을 위한 provider 경계 정의
- Examples 접근을 위한 임시 암호 단어 입력 UI
- Help 메뉴 Manual 버튼 활성화

- Manual 영상 제작 자체

## 4. 예제 악보 저장 위치

첫 임시 테스트 구현에서는 로컬 정적 예제 Score JSON을 저장하지 않는다.
예제 악보는 Supabase Storage public URL 또는 별도 HTTPS URL에 두고, 앱은 Supabase Edge Function이 반환한 manifest의 `scoreUrl`을 사용해 선택된 Score JSON만 지연 로드한다.

```text
Supabase
  example_scores table
  example-scores storage bucket 또는 외부 HTTPS score URL
  get-example-manifest Edge Function
```

초기 boot용 template score는 현재처럼 매우 작은 기본 악보 1개만 source asset으로 유지할 수 있다.
실제 곡 예제는 source import나 `public/examples` 정적 파일로 두지 않는다.
DB/Edge Function 로드 실패 시 fallback score를 따로 불러오지 않고 현재 악보 상태를 그대로 유지한다.

## 5. Example Manifest 형식

Edge Function 응답 manifest는 다음 의미 구조를 사용한다.

```ts
type ExampleScoreManifest = {
  version: 1;
  generatedAt?: string;
  examples: ExampleScoreManifestItem[];
};

type ExampleScoreManifestItem = {
  id: string;
  title: string;
  artist: string;
  difficulty?: {
    basic?: number;
    optional?: number;
    extra?: number;
  };
  supportedTracks: ("basic" | "optional" | "extra")[];
  durationSeconds?: number;
  sizeBytes?: number;
  scoreUrl: string;
  createdAt?: string;
  updatedAt?: string;
};
```

Manifest 검증 규칙:

- `version`은 `1`만 허용한다.
- `examples`는 배열이어야 한다.
- `id`, `title`, `artist`, `scoreUrl`은 빈 문자열이면 안 된다.
- `supportedTracks`는 하나 이상의 track id를 가져야 한다.
- `supportedTracks`는 `basic`, `optional`, `extra`만 허용하며 중복 track id가 있으면 실패한다.
- `scoreUrl`은 HTTPS URL만 허용한다.
- `createdAt`, `updatedAt`은 예제 ScoreFile 자체의 생성/갱신 시각 표시값이다.
- `createdAt`, `updatedAt`, `generatedAt`이 있으면 `Date.parse()` 가능한 ISO 계열 문자열이어야 한다.
- `difficulty.basic`, `difficulty.optional`, `difficulty.extra`가 있으면 finite number이고 `0 < value <= 99`여야 한다.
- `durationSeconds`가 있으면 finite integer이고 `0 < value`여야 한다.
- `sizeBytes`가 있으면 finite integer이고 `0 < value`여야 한다.
- manifest가 깨져 있으면 Examples UI에 오류를 표시하고, 앱 전체 boot는 실패시키지 않는다.

Manifest는 Score JSON 저장 포맷이 아니다.
예제 목록 표시와 score fetch 위치를 위한 별도 카탈로그 데이터이다.
라이선스, 설명, 썸네일은 공개 예제 배포 단계에서 다시 검토한다.
첫 임시 테스트 구현에서는 저작권 리스크와 UI 복잡도를 줄이기 위해 manifest 필수 필드에서 제외한다.

## 6. Supabase 연동 정책

Supabase 첫 연결은 읽기 전용 example provider로 제한한다.

허용 범위:

- Supabase Edge Function에서 example manifest 읽기
- public read 가능한 storage object 또는 HTTPS URL에서 Score JSON 읽기
- Edge Function 호출에 필요한 공개 `anon` key 사용

`anon` key는 GitHub Pages 앱에서 Edge Function을 호출하기 위한 공개 key로만 사용한다.
앱은 `anon` key로 `example_scores` table을 직접 REST select하거나 private storage object를 직접 읽지 않는다.

금지 범위:

- Supabase auth 사용
- insert / update / delete
- 사용자별 개인 악보 저장
- 악보 편집본 remote save
- Supabase 응답을 검증 없이 ScoreFile로 취급

Provider 경계는 다음 형태를 권장한다.

```ts
type ExampleProvider = {
  providerId: "supabase";
  displayName: string;
  loadManifest(accessWord: string): Promise<ExampleScoreManifest>;
  loadScoreText(item: ExampleScoreManifestItem): Promise<string>;
};
```

앱은 provider가 어디인지와 무관하게 `loadScoreText()` 결과를 기존 Score JSON 로드 경로에 전달한다.
따라서 예제 악보 로드는 반드시 `loadScoreJsonText()` 또는 같은 검증 파이프라인을 통과해야 한다.

### 6.1. Supabase 임시 테스트 보안 모델

Supabase table은 ScoreFile 본문을 저장하는 구조가 아니라, 예제 악보 목록을 표시하고 실제 Score JSON 위치를 찾기 위한 카탈로그이다.
첫 사용자 테스트 단계에서는 공개 예제 배포가 아니라 허락된 테스터에게만 임시로 예제를 제공하는 것을 목표로 한다.
따라서 DB table을 `anon` role에 직접 공개하지 않고, Edge Function이 가벼운 암호 단어를 확인한 뒤 manifest를 반환하는 구조를 우선 검토한다.

```text
GitHub Pages app
  -> File > Examples
  -> 사용자가 임시 암호 단어 입력
  -> Supabase Edge Function 호출
  -> Edge Function이 서버 환경 변수의 암호와 비교
  -> Edge Function이 example_scores에서 공개 대상 목록 조회
  -> 앱은 받은 manifest를 검증
  -> 사용자가 선택한 scoreUrl을 fetch
  -> 기존 ScoreFile validation pipeline 통과 후 로드
```

이 모델의 목적은 다음이다.

- GitHub Pages 앱 code에 DB service key나 비밀값을 넣지 않는다.
- 앱에 입력한 암호 단어는 UI 편의 장치가 아니라 Edge Function에서 확인한다.
- `example_scores` table은 `anon` role이 직접 읽지 못하게 한다.
- Edge Function은 고정 query만 사용하고 사용자 입력으로 SQL fragment를 조립하지 않는다.
- Score JSON 자체도 fetch 이후 기존 ScoreFile validator를 반드시 통과한다.

암호 단어 방식은 정식 인증이 아니다.
테스트 진입 장벽을 낮추기 위한 임시 접근 제한이며, 장기적으로 공개 예제 전환 또는 Supabase Auth 초대제 전환을 검토할 수 있다.

앱은 암호 단어를 localStorage/sessionStorage에 저장하지 않는다.
암호 단어는 console log, status line, dialog notice, telemetry 또는 오류 메시지에 출력하지 않는다.

Edge Function 요청/응답 첫 계약은 다음을 기준으로 한다.

```text
POST /functions/v1/get-example-manifest
Content-Type: application/json

request body:
  { "accessWord": string }

success response:
  ExampleScoreManifest

error response:
  { "error": { "code": string, "message": string } }
```

`accessWord`가 빈 문자열이면 앱은 네트워크 요청 전에 dialog notice로 입력 필요 오류를 표시할 수 있다.
Edge Function이 `401`, `403`, `429`, `500` 계열 응답을 반환하면 앱은 현재 score를 유지하고 Examples dialog notice와 left status line에 오류를 표시한다.
첫 구현의 앱 provider는 응답 body가 success manifest인지 error object인지 구분하되, error object를 ScoreFile이나 manifest로 취급하지 않는다.

첫 구현에서 사용하는 error code는 다음을 기준으로 한다.

- `INVALID_ACCESS_WORD` : 암호 단어가 비어 있거나 일치하지 않는다.
- `MANIFEST_UNAVAILABLE` : DB 조회, catalog 변환, 또는 manifest 생성에 실패했다.
- `RATE_LIMITED` : 임시 접근 제한 또는 요청 빈도 제한에 걸렸다.
- `INTERNAL_ERROR` : Edge Function 내부 오류이다.
- `INVALID_RESPONSE` : 앱이 받은 응답이 success manifest나 표준 error object가 아니다.

앱은 error code가 알 수 없는 값이어도 현재 score를 유지하고 일반 Examples 오류로 표시한다.

Supabase Dashboard에서 Edge Function을 직접 만들 때는 `get-example-manifest` 이름을 사용한다.
Dashboard 기본 template의 `name` / `Hello` 예제 코드는 사용하지 않고, 다음 역할을 수행하도록 바꾼다.

- request body에서 `accessWord`를 읽는다.
- `Deno.env.get("EXAMPLE_ACCESS_WORD")`와 비교한다.
- 불일치하면 `INVALID_ACCESS_WORD`와 HTTP `401`을 반환한다.
- 일치하면 서버 측 Supabase client로 `example_scores`를 조회한다.
- `is_published = true`, `access_group = "tester"` row만 manifest에 포함한다.
- `display_order`, `title` 순으로 정렬한다.
- DB row의 `snake_case` 필드를 앱 manifest의 `camelCase` 필드로 변환한다.
- 모든 응답에 CORS header를 붙인다.
- `OPTIONS` preflight 요청에는 빈 성공 응답을 반환한다.

Edge Function 환경 변수에는 다음 값을 둔다.

```text
EXAMPLE_ACCESS_WORD=privateRoom
```

`EXAMPLE_ACCESS_WORD`는 GitHub Pages 앱 코드, Vite 환경 변수, URL query, Score JSON, 공개 문서 예시에 넣지 않는다.
프론트엔드의 `VITE_EXAMPLE_MANIFEST_FUNCTION_URL`은 access word나 API key가 아니라 Edge Function 전체 URL만 의미한다.

```text
VITE_EXAMPLE_MANIFEST_FUNCTION_URL=https://<project-ref>.supabase.co/functions/v1/get-example-manifest
```

첫 구현에서 Edge Function은 다음 형태를 기준 template으로 삼을 수 있다.
Supabase Dashboard template 또는 CLI 환경에 따라 import path와 auth 설정은 조정될 수 있으나, access word 확인, service-role/server-side 조회, CORS, manifest 변환 책임은 유지한다.

```ts
import "jsr:@supabase/functions-js/edge-runtime.d.ts";
import { withSupabase } from "jsr:@supabase/server@^1";

type ExampleScoreRow = {
  id: string;
  title: string;
  artist: string;
  difficulty_basic: number | null;
  difficulty_optional: number | null;
  difficulty_extra: number | null;
  supported_tracks: string[];
  duration_seconds: number | null;
  size_bytes: number | null;
  score_url: string;
  created_at: string;
  updated_at: string;
};

type RequestPayload = {
  accessWord?: string;
};

const corsHeaders = {
  "Access-Control-Allow-Origin": "*",
  "Access-Control-Allow-Headers": "authorization, x-client-info, apikey, content-type",
  "Access-Control-Allow-Methods": "POST, OPTIONS",
};

export default {
  fetch: withSupabase({ auth: ["publishable", "secret"] }, async (req, ctx) => {
    if (req.method === "OPTIONS") {
      return new Response(null, { headers: corsHeaders });
    }

    if (req.method !== "POST") {
      return jsonError("METHOD_NOT_ALLOWED", "POST request is required.", 405);
    }

    let payload: RequestPayload;
    try {
      payload = await req.json();
    } catch {
      return jsonError("INVALID_REQUEST", "Request body must be JSON.", 400);
    }

    const expectedAccessWord = Deno.env.get("EXAMPLE_ACCESS_WORD");
    if (!expectedAccessWord || payload.accessWord !== expectedAccessWord) {
      return jsonError("INVALID_ACCESS_WORD", "Access word is invalid.", 401);
    }

    const { data, error } = await ctx.supabaseAdmin
      .from("example_scores")
      .select(
        [
          "id",
          "title",
          "artist",
          "difficulty_basic",
          "difficulty_optional",
          "difficulty_extra",
          "supported_tracks",
          "duration_seconds",
          "size_bytes",
          "score_url",
          "created_at",
          "updated_at",
        ].join(","),
      )
      .eq("is_published", true)
      .eq("access_group", "tester")
      .order("display_order", { ascending: true })
      .order("title", { ascending: true });

    if (error) {
      return jsonError("MANIFEST_UNAVAILABLE", "Example manifest is unavailable.", 500);
    }

    const examples = (data as ExampleScoreRow[]).map((row) => ({
      id: row.id,
      title: row.title,
      artist: row.artist,
      difficulty: buildDifficulty(row),
      supportedTracks: row.supported_tracks,
      durationSeconds: row.duration_seconds ?? undefined,
      sizeBytes: row.size_bytes ?? undefined,
      scoreUrl: row.score_url,
      createdAt: row.created_at,
      updatedAt: row.updated_at,
    }));

    return Response.json(
      {
        version: 1,
        generatedAt: new Date().toISOString(),
        examples,
      },
      { headers: corsHeaders },
    );
  }),
};

function buildDifficulty(row: ExampleScoreRow) {
  const difficulty: Record<string, number> = {};

  if (row.difficulty_basic !== null) difficulty.basic = row.difficulty_basic;
  if (row.difficulty_optional !== null) difficulty.optional = row.difficulty_optional;
  if (row.difficulty_extra !== null) difficulty.extra = row.difficulty_extra;

  return Object.keys(difficulty).length > 0 ? difficulty : undefined;
}

function jsonError(code: string, message: string, status: number) {
  return Response.json(
    {
      error: { code, message },
    },
    {
      status,
      headers: corsHeaders,
    },
  );
}
```

Supabase Edge Function auth/key 정책은 Supabase의 최신 platform 설정에 따라 차이가 날 수 있다.
publishable key를 사용하는 경우 JWT 검증 방식과 맞지 않을 수 있으므로, 배포 시 Dashboard/CLI의 function auth 설정과 request header 요구사항을 실제 호출 테스트로 확인한다.
이 프로젝트 명세의 핵심 보안 경계는 프론트엔드에 service role/secret key/access word를 저장하지 않고, Edge Function 서버 환경에서만 catalog 조회와 access word 확인을 수행하는 것이다.

### 6.2. Supabase 테이블 축소 초안

초기 catalog table은 UI 표시에 필요한 최소 필드만 둔다.
라이선스, 설명, 썸네일은 보류한다.
특히 `thumbnail_url`은 앨범아트나 영상 썸네일 사용 시 저작권 리스크가 있으므로 첫 구현에 포함하지 않는다.

`created_at`, `updated_at`은 예제 ScoreFile 자체의 생성/갱신일을 catalog에 복사해 둔 표시값이다.
UI에서는 각각 `생성일`, `갱신일`처럼 표시한다.

```sql
create table public.example_scores (
  id text primary key,
  title text not null,
  artist text not null,

  difficulty_basic numeric,
  difficulty_optional numeric,
  difficulty_extra numeric,

  supported_tracks text[] not null,

  duration_seconds integer,
  size_bytes integer,

  score_url text not null,

  display_order integer not null default 0,
  is_published boolean not null default false,
  access_group text not null default 'tester',

  created_at timestamptz not null,
  updated_at timestamptz not null,

  constraint example_scores_title_not_blank
    check (length(trim(title)) > 0),

  constraint example_scores_artist_not_blank
    check (length(trim(artist)) > 0),

  constraint example_scores_score_url_not_blank
    check (length(trim(score_url)) > 0),

  constraint example_scores_supported_tracks_not_empty
    check (array_length(supported_tracks, 1) >= 1),

  constraint example_scores_supported_tracks_allowed
    check (supported_tracks <@ array['basic', 'optional', 'extra']::text[]),

  constraint example_scores_duration_positive
    check (duration_seconds is null or duration_seconds > 0),

  constraint example_scores_size_positive
    check (size_bytes is null or size_bytes > 0),

  constraint example_scores_difficulty_basic_positive
    check (difficulty_basic is null or difficulty_basic > 0),

  constraint example_scores_difficulty_optional_positive
    check (difficulty_optional is null or difficulty_optional > 0),

  constraint example_scores_difficulty_extra_positive
    check (difficulty_extra is null or difficulty_extra > 0)
);

alter table public.example_scores enable row level security;

-- 첫 임시 테스트 모델에서는 anon role에 table 직접 select 권한을 주지 않는다.
-- Edge Function이 서버 측 권한으로 고정 query를 실행하고 검증된 manifest만 반환한다.

create index example_scores_public_order_idx
on public.example_scores (access_group, is_published, display_order, title);
```

### 6.3. Supabase 컬럼 의미

| 컬럼 | 앱 manifest 대응 | null 허용 | 의미 |
|---|---|---:|---|
| `id` | `id` | No | 예제 항목의 안정적인 식별자. URL slug처럼 사람이 읽을 수 있는 값을 권장한다. |
| `title` | `title` | No | Examples UI에 표시할 곡 또는 악보 제목. ScoreFile의 `musicData.musicTitle`과 같을 필요는 없지만 보통 동일하게 둔다. |
| `artist` | `artist` | No | Examples UI에 표시할 아티스트/작곡자. 모르면 `Unknown`처럼 명시 문자열을 넣는다. |
| `difficulty_basic` | `difficulty.basic?` | Yes | basic track 표시 난이도. ScoreFile 내부 `musicData.scoreDifficulty.basic`과 별도 catalog 표시값이며, 없으면 생략한다. |
| `difficulty_optional` | `difficulty.optional?` | Yes | optional track 표시 난이도. 없으면 생략한다. |
| `difficulty_extra` | `difficulty.extra?` | Yes | extra track 표시 난이도. 없으면 생략한다. |
| `supported_tracks` | `supportedTracks` | No | 예제 악보에서 테스트 대상으로 안내할 track 목록. `basic`, `optional`, `extra` 중 하나 이상이다. |
| `duration_seconds` | `durationSeconds?` | Yes | 목록에 표시할 예상 곡 길이. 정확한 재생 엔진 값이 아니라 catalog metadata이다. |
| `size_bytes` | `sizeBytes?` | Yes | Score JSON 파일 크기 표시값. 앱은 실제 fetch 이후에도 ScoreFile 크기 제한을 별도로 검사한다. |
| `score_url` | `scoreUrl` | No | 실제 Score JSON을 fetch할 URL. Supabase storage public URL 또는 HTTPS URL을 권장한다. |
| `display_order` | provider 내부 정렬값 | No | Examples 목록 정렬 순서. manifest로 변환할 때 직접 노출하지 않아도 된다. |
| `is_published` | provider 필터 | No | 목록 반환 여부. Edge Function은 `true`인 row만 manifest에 포함한다. |
| `access_group` | provider 필터 | No | 임시 테스트 그룹 구분값. 첫 구현 기본값은 `tester`이다. |
| `created_at` | `createdAt?` | No | 예제 ScoreFile 내부 생성일. |
| `updated_at` | `updatedAt?` | No | ScoreFile 내부 수정일. |

`NULL` 허용 컬럼은 앱 manifest로 변환할 때 `undefined`처럼 취급하고, JSON 응답에는 해당 속성을 넣지 않는다.
반대로 `not null` 컬럼이 빈 문자열이거나 누락되면 DB 제약 또는 앱 manifest validator에서 실패시킨다.

난이도 값은 소수를 허용한다.
앱의 Details UI와 ScoreFile validator가 이미 난이도 입력을 제한하므로, catalog table은 첫 구현에서 `difficulty_* > 0` 이상의 추가 상한 제약을 두지 않는다.
`supported_tracks`의 허용 값과 중복 여부는 앱 manifest validator에서도 다시 확인한다.

### 6.4. 내부 Score JSON 값과 catalog 동기화

`example_scores` table의 `title`, `artist`, `difficulty_*`, `duration_seconds`, `size_bytes`는 ScoreFile 내부 JSON과 완전히 같은 저장 구조가 아니다.
ScoreFile은 `score_url`이 가리키는 외부 JSON 파일이므로, Postgres check constraint만으로 해당 파일을 열어 내부 값을 검증할 수 없다.

초기 운영 방법은 다음 중 1단계를 사용한다.

1. 관리자가 Supabase dashboard에서 catalog 값을 수동 입력한다.
2. 예제 수가 늘어나면 로컬 Node 관리 스크립트가 Score JSON을 읽고 기존 validator로 검증한 뒤 metadata를 추출해 Supabase row를 upsert한다.
3. 장기적으로 필요하면 Supabase Edge Function이 Score JSON fetch, 검증, metadata 추출, upsert를 맡는다.

첫 구현에서는 1번을 사용한다.
2번 관리 스크립트는 예제 수가 늘어나거나 catalog와 ScoreFile 내부 metadata 불일치가 반복될 때 추가한다.

### 6.5. Supabase 보안 규칙

- RLS는 반드시 켠다.
- 임시 테스트 모델에서는 `anon` role에 `example_scores` 직접 select 권한을 주지 않는다.
- 앱은 Supabase Edge Function만 호출한다.
- Edge Function은 암호 단어를 서버 환경 변수와 비교한다.
- Edge Function은 고정 query만 사용하며 사용자 입력으로 SQL fragment를 조립하지 않는다.
- `insert`, `update`, `delete`는 앱에서 사용하지 않는다.
- 앱은 Supabase 응답을 그대로 ScoreFile로 취급하지 않는다.
- 앱은 Edge Function 응답을 먼저 `ExampleScoreManifestItem`으로 변환/검증하고, 사용자가 선택한 `scoreUrl`의 JSON을 다시 기존 ScoreFile validation pipeline에 통과시킨다.
- `score_url`은 DB 제약만 믿지 않고 앱 validator에서도 HTTPS URL인지 확인한다.
- Edge Function은 관리자가 catalog에 등록한 `score_url`만 반환한다.
- 공개 테스트 중 실제 score host가 고정되면 앱 validator 또는 provider에 허용 origin / Supabase Storage bucket prefix allowlist를 추가할 수 있다.
- 암호 단어 방식은 임시 접근 제한이다. 정식 비공개 운영이 필요해지면 Supabase Auth 기반 초대제 또는 별도 backend 인증으로 전환한다.

### 6.6. 예상 모듈 구조

Examples 기능은 score load, Supabase 통신, manifest 검증, UI dialog를 분리한다.
Score JSON 본문 검증과 runtime rebuild는 기존 score load pipeline을 재사용한다.

```text
regression-code/src/app/examples/
  example_types.ts
  example_manifest_validate.ts
  example_supabase_provider.ts
  example_dialog.ts
  example_binding.ts
```

모듈별 책임:

- `example_types.ts`
  - `ExampleScoreManifest`, `ExampleScoreManifestItem`, `ExampleProvider` 타입을 둔다.
  - Supabase row 원본 타입과 앱 manifest 타입을 구분한다.
- `example_manifest_validate.ts`
  - Edge Function 응답을 신뢰하지 않고 manifest 구조를 검증한다.
  - `scoreUrl`은 HTTPS URL만 허용한다.
  - `NULL` 또는 누락된 optional 값은 manifest 속성 생략으로 정규화한다.
  - `supportedTracks`는 허용 track id만 포함하고 중복이 없어야 한다.
- `example_supabase_provider.ts`
  - Edge Function endpoint와 통신한다.
  - 암호 단어를 request body로 보내고 manifest 응답을 받아 validator에 넘긴다.
  - Supabase table에 직접 REST select를 하지 않는다.
- `example_dialog.ts`
  - Examples dialog 렌더링, 암호 입력, 목록 선택, 오류 notice 표시를 담당한다.
  - 현재 score 교체 확인은 이 UI 경계에서 수행한다.
- `example_binding.ts`
  - File > Examples 버튼과 app state/session callback을 연결한다.
  - 선택된 예제의 Score JSON text를 기존 score load pipeline에 전달한다.

DOM은 `index.html`의 File menu와 Examples dialog에 추가한다.
CSS는 `styles/app.css`에 두되, dialog 내부 표/버튼/notice 스타일만 좁게 추가한다.
Supabase URL과 Edge Function path는 별도 설정 상수 또는 Vite 환경 변수로 분리하고, `service_role` key는 어떠한 프론트엔드 파일에도 넣지 않는다.
첫 구현은 배포 환경 교체를 쉽게 하기 위해 Vite build-time 환경 변수를 우선 사용한다.
권장 변수명은 `VITE_EXAMPLE_MANIFEST_FUNCTION_URL`이며, 값은 `https://.../functions/v1/get-example-manifest`처럼 Edge Function 전체 URL로 둔다.
값이 비어 있으면 Examples dialog는 manifest load를 시도하지 않고 설정 누락 오류를 표시한다.
이 URL은 공개되어도 되는 endpoint이며, Supabase `service_role` key나 암호 단어는 포함하지 않는다.

## 7. Examples UI

Examples 진입점은 File 메뉴에 둔다.

```text
File
  JSON Load
  JSON Download
  Local Save
  Local Load
  Examples
```

Examples UI는 dialog 또는 오른쪽 panel 형태로 구현한다.
첫 구현은 dialog를 우선한다.

표시 항목:

- 곡 제목
- 아티스트
- 난이도
- 지원 track
- 예상 길이
- 파일 크기
- 등록일
- 갱신일
- provider 표시

첫 임시 테스트 구현에서는 라이선스 label, 긴 설명, 썸네일을 표시하지 않는다.
공개 예제 배포로 전환할 때 라이선스 표시와 출처 표기를 별도 UI로 추가한다.

동작:

- Examples 버튼 클릭 시 암호 단어 입력이 포함된 dialog를 연다.
- 사용자가 Load List를 누르면 Supabase Edge Function에서 manifest를 로드한다.
- manifest 로드 중에는 busy 상태와 left status line을 갱신한다.
- manifest 로드 실패 시 현재 score를 유지하고 Examples dialog notice와 left status line에 오류를 표시한다.
- 예제 선택 후 Load Example을 누르면 현재 score 교체 확인을 띄운다.
- 확인 후 score JSON을 fetch하고 기존 score load pipeline으로 전달한다.
- 로드 성공 시 edit mode, selection, range selection, undo/redo history는 score file load 정책과 동일하게 초기화한다.
- 로드 실패 시 현재 score는 유지한다.

현재 score가 기본 template에서 변경되지 않았는지 자동 판별하는 기능은 첫 구현에서 필수로 두지 않는다.
첫 구현은 항상 교체 확인을 띄우는 보수적 정책을 사용한다.

암호 단어는 dialog session 동안 input 값으로만 유지한다.
Examples dialog를 닫으면 암호 단어 input과 loaded manifest selection은 초기화한다.
암호 단어는 localStorage, sessionStorage, ScoreFile, URL query, console log에 저장하지 않는다.

예제 score load는 transaction처럼 처리한다.
score fetch, size check, JSON parse, ScoreFile validation, runtime rebuild가 모두 성공한 뒤에만 현재 score를 교체한다.
중간 단계에서 실패하면 기존 score, selection, edit history, playback state는 가능한 한 유지한다.

practice mode 중에는 Examples 진입점과 예제 load를 비활성화한다.
일반 playback 중 예제 load가 확인되면 기존 score file load 정책과 동일하게 playback을 정지하고 새 score load 초기화 흐름을 따른다.

## 10. 오류 처리

Examples 관련 오류는 앱 전체를 중단시키지 않는다.

오류 유형:

- manifest fetch 실패
- manifest 구조 검증 실패
- 암호 단어 불일치 또는 Edge Function 접근 거부
- score fetch 실패
- score JSON 크기 초과
- score JSON parse 실패
- ScoreFile validation 실패

오류 표시는 left status line과 Examples dialog notice를 함께 사용한다.
score 로드 실패 시 기존 score와 playback state를 가능한 보존한다.
단, fetch 중 사용자가 명시적으로 load를 시작한 경우에는 busy 상태에서 중복 load를 막는다.
로컬 정적 fallback 예제는 첫 임시 테스트 구현에 두지 않는다.

## 11. 검증

구현 후 확인할 항목:

- 앱 초기 JavaScript bundle에 실제 예제 악보 JSON이 포함되지 않는다.
- Examples 메뉴를 열기 전에는 개별 예제 Score JSON을 fetch하지 않는다.
- manifest만 로드했을 때 score는 교체되지 않는다.
- 예제 선택 후 로드한 Score JSON이 기존 JSON Load와 같은 검증 경로를 통과한다.
- 예제 Score JSON은 fetch, size check, parse, validation, runtime rebuild가 모두 성공한 뒤에만 현재 score를 교체한다.
- 깨진 manifest는 현재 score를 망가뜨리지 않는다.
- 깨진 예제 Score JSON은 현재 score를 유지한 채 오류를 표시한다.
- Supabase Edge Function 실패 또는 암호 불일치 시 현재 score를 유지한다.
- practice mode 중 Examples 진입점 또는 load가 비활성화된다.
- Manual 버튼이 비활성 상태로 남지 않는다.
- Manual에서 YouTube 재생목록 링크가 없어도 기본 안내 문서는 열 수 있다.

프로젝트의 정식 build 명령이 유지되는 경우 `npm run build`를 통과해야 한다.
예제 manifest/parser helper를 추가한다면 해당 helper의 최소 검증 테스트를 추가한다.

## 12. 구현 순서

권장 구현 순서:

1. Supabase `example_scores` table과 Edge Function 계약을 확정한다.
2. example manifest type과 validator를 만든다.
3. Supabase Edge Function provider를 만든다.
4. Examples dialog와 File 메뉴 버튼을 연결한다.
5. 암호 단어 입력, manifest load, 오류 notice를 연결한다.
6. 선택 예제 로드를 기존 score load pipeline에 연결한다.
7. Manual 정적 문서를 추가하고 Help 메뉴 Manual 버튼을 활성화한다.
8. Manual 안에 YouTube 재생목록 링크 placeholder를 둔다.

로컬 정적 example manifest와 example Score JSON은 첫 임시 테스트 구현에 포함하지 않는다.
필요하면 공개 예제 배포 단계에서 별도 fallback 정책으로 다시 검토한다.
