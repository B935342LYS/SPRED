## 3. Current Limits

The current implementation uses these limits:

| Target | Limit | Boundary |
| --- | ---: | --- |
| layout preset display name | 30 characters | UI input and preset creation/save validation |
| layout preset JSON | 256 KiB, UTF-8 byte length | file load/save and local slot save/load validation |
| score JSON file load | 8 MiB, UTF-8 byte length | file import parsing boundary |
| local score save | 3 MiB, UTF-8 byte length | localStorage score save boundary |
| score cell `rawText` | 1..100 characters | score validation and edit composition |
| tuplet slot input | 30 characters | UI input boundary |
| composed tuplet `rawText` | 100 characters | edit composition boundary |
| local layout preset slots | 3 slots per `instrumentPresetId` | localStorage key policy |

The local score save limit is lower than the file-load limit because browser `localStorage` quota is shared by score data, layout presets, and any other same-origin data.

Loading a large JSON file into runtime memory does not itself reduce `localStorage` quota.
It can still increase runtime memory pressure while the app parses, validates, analyzes, and renders the score.

## 4. Validation Responsibility

Validation is intentionally split by data boundary.

- `src/core/score/score_limits.ts`
  - owns shared score/file byte helpers and score-oriented limit constants
- `src/core/score/json_load.ts`
  - rejects oversized score JSON text before parsing
- `src/core/score/score_validate.ts`
  - validates persisted `ScoreFile` structure and stored cell `rawText`
- `src/infra/score_local_storage.ts`
  - rejects oversized local score saves before `localStorage.setItem()`
- `src/app/edit/edit_apply.ts`
  - blocks note/global edit commands that would write oversized `rawText`
- `src/app/edit/edit_tuplet.ts`
  - blocks finalized tuplet text above the stored cell limit
- `src/app/layout/layout_draft.ts`
  - validates layout draft mutation rules before a preset or score mutation is produced
- `src/app/layout/layout_preset.ts`
  - validates `UserLayoutPresetData`, preset display name length, preset JSON byte size, and preset file names
- `src/infra/layout_preset_storage.ts`
  - owns fixed local slot keys and localStorage read/write error conversion

The division keeps score-file integrity checks in the core score layer while keeping layout-preset-specific rules in the app layout layer.

## 5. Reuse and Encapsulation Decision

The byte-length helper and score-oriented constants are placed in `core/score` because score file import, score local storage, and edit write guards all depend on the same persisted score limits.

`app/layout` may import these small pure helpers when it needs consistent UTF-8 byte measurement.
The reverse direction is not allowed: `core` must not depend on `app`, `infra`, DOM, or layout dialog code.

Similar-looking validations are not always shared.
For example, `ScoreFile` row/cell validation and `UserLayoutPresetData` row validation are separate because they protect different persisted formats.
Sharing only the low-level byte helper avoids coupling those domain contracts.

## 6. Layout Apply Decision

Layout apply now creates the next `ScoreFile` through structural sharing.

The apply path replaces only:

- `instData`
- `layout.rowDefinitions`
- affected track cell arrays when deleted row references must be removed

Unchanged metadata, global lines, and unaffected tracks are reused.
This avoids whole-score deep clone cost while preserving immutable app-state replacement.

After layout apply, the app still rebuilds runtime artifacts:

```text
ScoreFile
-> RuntimeDocument
-> ParsedScoreDocument
-> AnalysisResult
-> CanvasRenderInput
-> render
-> playback runtime reset
```

This is appropriate for the current stage because layout changes affect renderer coordinates, hit testing, and audio scroll reference behavior.
Fine-grained parse/analyze/render invalidation remains a later performance extension.
