# 1.8. 1단계 Note Modifier Parser 구현 메모
> 본 설계 메모 문서 1.8에서는 `parse_note_cell.ts`의 일반 note modifier 구현 범위를 기록한다.
> 본 문서는 active specification을 대체하지 않는다.
> 함수별 세부 설명은 코드 파일의 `/** ... */` 주석을 중심으로 하고, 본 문서는 흐름과 구현 판단을 보충한다.

## 1. 개요

직전 note parser 골격은 mute, pletExtend, hold, defaultText까지만 처리했다.
이번 단계에서는 일반 note 셀 뒤에 붙는 modifier 4종을 구현했다.

현재 일반 note 파싱 흐름은 아래와 같다.

```text
parseNoteCell()
-> parseNoteText()
-> scanDefaultText()
-> parseModifiers()
   -> parseGliss()
   -> parseTrem()
   -> parseAbsolutePitch()
   -> parseMicroPitch()
```

## 2. 구현 범위

현재 구현한 modifier는 다음과 같다.

- `@g(id,kind)`
  - `id`는 소문자 알파벳 한 글자만 허용한다.
  - `kind`는 `S`, `M`, `E`만 허용한다.
  - 각각 start, mid, end를 의미한다.
- `@t(divNum)`
  - `divNum`은 `2`, `3`, `4`만 허용한다.
- `@p(midiNum)`
  - 정수만 허용한다.
  - 범위는 `0`부터 `127`까지이다.
- `@m(centNum)`
  - `-100`부터 `100`까지 허용한다.
  - 소수점 이하 최대 1자리까지만 허용한다.

modifier 순서는 명세에 따라 `@g -> @t -> @p -> @m`으로 고정했다.
각 modifier는 최대 1회만 허용한다.

## 3. 구현 판단

### 3.1. 순서 고정 방식

`parseModifiers()`는 정해진 순서대로 `startsWith()` 검사를 수행한다.
예를 들어 `@p`가 먼저 나오면 `@g`, `@t`는 건너뛰고 `@p`를 파싱한다.
이후에 `@t`가 남아 있으면 이미 canonical order 처리 위치를 지난 것이므로 `modifier_order`로 분류한다.

### 3.2. duplicate 판정

각 modifier를 성공적으로 읽으면 `seenModifiers`에 종류를 기록한다.
canonical order 처리 뒤 남은 토큰이 이미 읽은 종류라면 `duplicate_modifier`를 반환한다.

예를 들어 `E3@t(2)@t(3)`은 첫 번째 `@t`를 읽은 뒤 두 번째 `@t`가 남는다.
이때 `seenModifiers.has("trem")`이 true이므로 중복으로 판정한다.

### 3.3. `~` hold와 tremolo 충돌

`~` hold는 vibrato 성격을 가지므로 `@t(...)` tremolo modifier와 같은 note 셀에서 함께 사용할 수 없다.
따라서 `parseModifiers()`에서 `@t(`를 읽기 직전에 hold가 `~`인지 먼저 검사하고, 충돌 시 `vib_and_trem`을 반환한다.

### 3.4. 인수 읽기 공통화

`readModifierArgs()`는 `@g(`, `@t(`, `@p(`, `@m(` 뒤의 닫는 괄호를 찾아 내부 문자열만 반환한다.
괄호가 닫히지 않으면 `unterminated_paren`, 내부가 비어 있으면 `missing_argument`를 반환한다.

각 modifier별 인수 개수, 숫자 형식, 범위 검사는 개별 하위 parser가 담당한다.

## 4. 이번 구현에서 사용한 주요 TypeScript 문법

### 4.1. `as const`

modifier 시작 토큰 목록은 `as const`를 사용해 문자열 리터럴 튜플로 보존했다.

```ts
const KNOWN_MODIFIER_TOKENS = ["@g(", "@t(", "@p(", "@m("] as const;
type ModifierToken = (typeof KNOWN_MODIFIER_TOKENS)[number];
```

이렇게 하면 `ModifierToken`은 일반 `string`이 아니라 `"@g(" | "@t(" | "@p(" | "@m("` 유니언 타입이 된다.
`readModifierArgs()`의 token 인수에 잘못된 문자열이 들어가는 실수를 줄일 수 있다.

### 4.2. 제네릭 결과 타입

modifier 하위 parser들은 같은 성공/실패 형태를 공유하지만 성공 값의 타입은 서로 다르다.
이를 위해 `ParsePartResult<T>`를 재사용했다.

```ts
type ParseModifierPartResult<T> = ParsePartResult<T>;
```

예를 들어 `parseGliss()`는 `ParseModifierPartResult<ParsedGliss>`를 반환하고, `parseTrem()`은 `ParseModifierPartResult<ParsedTrem>`을 반환한다.

### 4.3. 타입 가드

`isGlissKind()`와 `isTremDivision()`은 런타임 검사를 하면서 TypeScript 타입도 좁힌다.

```ts
function isGlissKind(value: string): value is ParsedGliss["glissKind"]
function isTremDivision(value: number): value is ParsedTrem["divNum"]
```

검사 통과 후에는 객체를 만들 때 별도 타입 단언 없이 `glissKind`, `divNum`에 값을 넣을 수 있다.

## 5. 테스트 확인

`dev/test_parse.ts`에 modifier 직접 샘플을 추가했다.

valid 샘플은 다음 경로를 확인한다.

- gliss
- trem
- absolute pitch
- micro pitch
- 여러 modifier 조합

invalid 샘플은 다음 오류 코드가 유지되는지 확인한다.

- `unknown_modifier`
- `modifier_order`
- `duplicate_modifier`
- `vib_and_trem`
- `invalid_number`

현재 성공 출력은 다음 항목을 포함한다.

```text
Note modifier sample parse completed.
valid modifier samples: 5
invalid valid-modifier samples: 0
Invalid note modifier sample parse completed.
invalid modifier samples: 5
unexpected invalid-modifier results: 0
```
