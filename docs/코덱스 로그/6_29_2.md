현재 이해한 진행 상황은 다음과 같습니다.

**큰 방향**

노래방 모드는 내부적으로 `game mode`, UI에서는 `practice mode`로 구현 중입니다. 기존 parser/analyzer/renderer/audio 구조를 바꾸지 않고, analyzer가 만든 `AnalysisResult`와 현재 active track의 `NoteEvent`를 판정 대상으로 삼는 별도 런타임 계층으로 얹는 방향입니다. 점수, 마이크 입력, 판정 결과는 score JSON에 저장하지 않는 세션 데이터입니다.

**구현 완료된 핵심**

- `practice mode (beta)` 버튼과 practice panel UI가 들어가 있습니다.
- 마이크 권한 요청, `pitchy` 기반 pitch detection, RMS/clarity/frequency range 필터링이 연결되어 있습니다.
- 입력 pitch dot overlay가 구현되어 있고, target 보정 없이 detector의 생 pitch를 표시합니다.
- practice 진입 시 edit mode, loop, range selection, paste preview 등을 정리하고 view 상태로 잠급니다.
- active track은 `ready/off/error`에서는 변경 가능하고, `playing/paused/finished/countdown/preparing`에서는 잠깁니다.
- `Sync` 보정 UI가 있습니다. 기본값은 `+90 ms`, 범위는 `-200~+200 ms`, 10ms 단위입니다.
- Rules dialog, Result dialog, Mic/Hz/Clarity/RMS 진단 표시가 있습니다.

**판정 로직 상태**

기본 음정 판정은 구현되어 있습니다.

- 현재 score time에 걸친 active track `NoteEvent`를 target으로 수집합니다.
- 여러 target이 동시에 있으면 pitch class 오차가 가장 작은 하나를 고릅니다.
- 같은 계이름이면 옥타브가 달라도 맞는 것으로 처리합니다.
- 기준은 `Perfect < 50 cent`, `Ok < 100 cent`, `Bad < 200 cent`, 그 이상 `Miss`입니다.
- `Miss`는 0점 sample이며 정확도 평균 분모에 포함됩니다.
- 무음/불명확 입력은 sample 자체를 만들지 않아 Miss로 세지 않습니다.
- 인접 note 전환부는 앞뒤 약 `120ms` grace를 둬서 늦은 release/빠른 attack을 완화합니다.

타이밍 판정도 1차 구현되어 있습니다.

- onset 후보를 note 시작점과 매칭합니다.
- `<80ms`는 별도 timing text 없음.
- `80~149ms`는 early/late 표시만 하고 pitch label은 유지합니다.
- `150~249ms`는 최종 label을 Bad로 강등합니다.
- `250ms+`는 Miss로 강등합니다.
- 일반 note는 attack credit이 없으면 내부 pitch feedback은 유지하되 점수/combo에는 반영하지 않습니다.
- gliss end처럼 legato 성격인 target은 no-onset만으로 막지 않습니다.

**효과 판정 상태**

효과 보너스는 부분 구현 상태입니다.

- `Gliss!` 보너스: 구현됨. gliss 구간을 250ms interval로 나누고, 입력이 끊기거나 pitch가 크게 틀리면 실패/잠금 처리합니다. 짧은 gliss에는 grace가 들어가 있습니다.
- `Vib!` 보너스: 구현됨. target 근처에서 pitch가 충분히 흔들리는지, 평균 pitch가 target 근처인지, direction change/rate/window 조건을 봅니다. 짧은 vib target 완화도 들어가 있습니다.
- `Trem!` 보너스: UI 카운트와 타입 자리는 있지만 실제 RMS envelope 기반 판정은 아직 후속입니다.

**검증 상태**

`test:game`이 따로 있고, 다음 범위를 다룹니다.

- frequency to MIDI, RMS 계산
- Sync offset 정규화/저장/로드
- active track target 수집
- pitch class 판정
- 전환부 grace
- timing early/late/bad/miss
- attack credit / score eligibility
- gliss 보너스
- vib 보너스

**아직 남은 주요 작업**

- Trem effect bonus 실제 구현
- Pro Mode 구현: 명세는 있으나 코드에는 아직 반영되지 않은 상태로 보입니다.
- 실제 마이크/브라우저 수동 검증 계속 필요
- timing anti-cheat 추가 튜닝: no-onset 직접 Bad/Miss 강등은 이전에 롤백된 상태
- YouTube sync와 practice mode의 실사용 조합 검증
- loop playback 연결은 별도 후속으로 남아 있음
- 하위 성능 기기에서 긴 악보 + practice mode 조합 재검증 필요

요약하면, 노래방 모드는 “UI skeleton” 단계를 넘어 실제 마이크 pitch 입력, 기본 음정/타이밍 판정, 결과 표시, gliss/vib 보너스까지 들어간 상태입니다. 현재 남은 중심 과제는 trem, Pro Mode, 실제 마이크 환경에서의 판정 안정화입니다.

> 네. 그런데 프로 모드를 추가하기 전에 심각한 UX 믄제를 두가지 개선해야 하겠습니다.
>
>  - 다른 화면 비율에서의 UX 개선 : 프랙티스 모드를 열 시 자동으로 fit height가 동작하도록 해서 작은 화면에서도 악보가 정상적으로 잘 보이도록 맞춥니다.
>
>   - 노트북에서 음정 인식 및 표시 초록 원은 발생하는데 점수 산정이 되지 않는 버그가 있습니다. 노트북은 2019년에 산 LG gram입니다.

정리된 Trem 1차 로직은 다음입니다.

- trem target은 active track의 `effect.trem !== null` segment에서 수집
- `division`은 target에도 넣지 않고 판정에도 사용하지 않음
- pitch는 안정적이라고 가정하고 trem 전용 pitch tolerance 분기 없음
- 같은 trem target 안에서 accepted attack이 최소 2회 이상이어야 함
- 첫 attack은 streak 시작만 하고 보너스 없음
- 두 번째 attack부터 attack마다 `Trem!` 및 bonus 지급
- 짧은 trem segment도 예외 없이 최소 2회 attack 필요
- Standard/Pro 분기 없이 동일한 Trem 규칙 사용
- bonus는 단일 상수 `trackDifficulty * 0.12`
- 남은 조정 여지는 `min attack interval = 60ms`, `max attack interval = 450ms`, bonus 상수 정도이며, 조정하더라도 분기는 만들지 않는 방향

이번 변경은 문서 정리만이라 빌드/테스트는 실행하지 않았습니다. 현재 루트 작업 상태는 Trem 문서 수정 1개와 새 구현 메모 1개가 남아 있습니다.

> 그러고보니 1tick 노트의 반복도 trem으로 인식하게 할지 고민입니다. 초기엔 별 생각 없었는데 직접 플레이해보니 그 속도가 트레몰로에 버금갑니다.

좋은 포인트입니다. 직접 플레이 기준이면 “1tick 반복 note”는 사실상 trem처럼 다뤄도 타당합니다. 다만 명세상으로는 두 경우를 구분하는 게 좋겠습니다.

추천 방향은 **explicit trem + implicit rapid repeat trem** 두 경로입니다.

- **explicit trem**: 악보에 `trem` modifier가 붙은 구간
- **implicit rapid repeat trem**: `1tick` 또는 매우 짧은 동일 계열 note가 연속된 구간

구현은 단순하게 유지하려면 이렇게 잡는 게 좋습니다.

```text
같은 active track에서
짧은 note가 3개 이상 연속되고
note 사이 gap이 거의 없으며
각 note가 attackRequired이면
하나의 implicit trem target처럼 취급한다.
```

판정은 기존 Trem과 동일하게 둡니다.

- 첫 attack은 보너스 없음
- 두 번째 연속 attack부터 `Trem!`
- attack마다 `trackDifficulty * 0.12`
- division 미사용
- Pro/Standard 분기 없음
- pitch 불안정 보정 없음

다만 **2개 반복만으로는 trem으로 보지 않는 것**을 추천합니다. 2개는 일반 반복음이나 장식음과 구분이 약합니다. 3개 이상이면 플레이 감각상 “빠른 반복 주법”으로 인정해도 자연스럽습니다.

명세상 이름은 `implicit trem target` 또는 `rapid repeat trem target` 정도가 적절합니다. 실제 UI에는 그대로 `Trem!`을 띄워도 됩니다. 사용자는 modifier가 붙었는지보다 “빠른 반복 attack을 잘 넣었는지”를 체감하니까요.

> 맞습니다. 이 "1tick 트레몰로"를 감지하는 로직을 추가하는 것이 어느 정도 크기의 연산이죠? 부담이 가지 않았으면 좋겠습니다.

부담은 작게 만들 수 있습니다. **매 frame마다 악보 전체를 훑지만 않으면** 거의 무시 가능한 수준입니다.

추천 구조는 이렇습니다.

1. score/analyzer 결과가 바뀔 때 한 번만 rapid repeat target을 수집
2. practice 시작 시 `effectBonusTargets`에 explicit trem target과 함께 넣음
3. 실시간 scoring loop에서는 기존 trem target처럼 “현재 score time 주변 target만” 확인

즉, 비싼 작업은 실시간이 아니라 준비 단계에서 끝냅니다.

대략 연산량은:

```text
target 수집: O(note event count)
실시간 판정: O(current nearby trem targets + recent onset candidates)
```

일반 악보 규모라면 `note event`가 수백~수천 개여도 target 수집은 한 번 도는 정도라 부담이 작습니다. 실시간 쪽도 이미 onset 후보 ring buffer를 보고 있으니, implicit trem target만 조금 늘어나는 수준입니다.

실제 구현은 다음처럼 제한하면 안전합니다.

- active track만 검사
- 같은 pitch class 또는 같은 note 계열 반복만 대상
- note duration이 1tick 또는 짧은 threshold 이하
- 연속 note 3개 이상일 때만 target 생성
- gap이 거의 없는 경우만 인정
- 생성된 target에는 `startSeconds`, `endSeconds`, `targetMidi`, `targetCentOffset`, `targetId`만 저장
- division은 계속 미사용

이 정도면 “1tick 트레몰로” 감지는 성능보다 **오탐 기준 설계**가 더 중요한 작업입니다. 연산 부담 자체는 낮습니다.

볼륨 차이를 보는 보조 판정은 넣을 수 있지만, **주 판정으로 바꾸는 것은 비추천**입니다.

이유는 2가지입니다.

첫째, 마이크가 attack을 못 잡는 수준이면 RMS의 미세한 점멸도 안정적으로 잡힌다는 보장이 약합니다. 특히 노트북 내장 마이크는 자동 gain control, noise suppression, echo cancellation이 걸려 있으면 작은 볼륨 차이를 평탄화할 수 있습니다.

둘째, RMS 기반 판정은 오탐이 늘기 쉽습니다. 손 움직임, 숨, 방 소음, 오타마톤 음량 흔들림, 마이크 거리 변화가 모두 trem처럼 보일 수 있습니다.

그래도 넣는다면 방향은 **attack 대체가 아니라 explicit trem 전용 보조 조건**이 좋습니다.

```text
Trem success =
  repeated attack streak
  OR
  explicit trem 구간에서 RMS가 일정 횟수 이상 위아래로 흔들림
```

제안 기준은 보수적으로:

- rapid repeat 1tick trem에는 RMS 보조를 쓰지 않음
- `@t(...)` explicit trem segment에만 RMS 보조 적용
- pitch frame은 voiced여야 함
- target 구간 안에서만 RMS 분석
- RMS peak/trough 변화가 최소 2회 이상 있어야 함
- bonus는 attack과 동일하게 `Trem!`로 주되, 같은 target/onset 또는 같은 window 중복 보상 방지
- 실패해도 감점 없음

성능 부담은 작습니다. 이미 `effectFrames`에 `rms`가 들어 있으니 최근 frame window를 훑는 정도입니다. 문제는 성능보다 **판정 신뢰도**입니다.

제 결론은 이렇습니다.

- 지금 1tick trem은 attack 기반으로 유지
- 2 division explicit trem은 “RMS 보조 판정”을 실험 기능처럼 작게 추가
- 오탐이 많으면 바로 끌 수 있게 상수/함수 경계를 분리

즉, 넣어볼 가치는 있습니다. 다만 “마이크가 못 듣는 것을 점수 시스템이 확실히 복구한다”기보다는, 잡히는 환경에서는 보너스를 조금 더 인정해주는 보조 장치로 보는 게 맞습니다.

> 그 전에, UI에 이지 모드를 추가합시다.
> 실 사용자 테스트 결과가 들어왔습니다. 음정을 끊어 치는 것이 너무 어렵다는 제보가 들어왔습니다.
> 홀드했을 때 이후 연속적으로 오는 노트를 전부 Perfect로 판정하지 않도록 보정하는 함수를 넣었었죠. 그 기능을 의도적으로 빼고 뭉개진 연주도 옳다고 인식하는, 말 그대로 음정만 보는 easy mode를 만들어야 할 것 같습니다. 일반적인 사람들이 이렇게까지 프렛리스 악기를 못 다룰 줄은 몰랐습니다.

> 다음은 잠시 다른 작업을 해봅시다. 혹시 모바일 환경인 경우 특별히 감지가 가능한가요?

네. 웹 앱에서는 모바일 환경을 어느 정도 감지할 수 있습니다. 다만 **완벽한 “모바일 판별”은 어렵고**, 보통은 여러 신호를 조합합니다.

가장 실용적인 기준은 이 정도입니다.

```ts
const isCoarsePointer = window.matchMedia("(pointer: coarse)").matches;
const isSmallViewport = window.matchMedia("(max-width: 768px)").matches;
const isTouchDevice = navigator.maxTouchPoints > 0;

const isLikelyMobile = isCoarsePointer && (isSmallViewport || isTouchDevice);
```

추가로 User-Agent Client Hints가 있으면 더 직접적입니다.

```ts
const isMobileUa = navigator.userAgentData?.mobile === true;
```

하지만 `navigator.userAgentData`는 브라우저별 지원 차이가 있어서 보조 신호로 보는 게 좋습니다.

현재 프로젝트 기준으로는 목적에 따라 이렇게 나누는 게 적절합니다.

- **UI 레이아웃 조정**: viewport width, height, `pointer: coarse`
- **모바일 edit 제한**: `pointer: coarse` + `maxTouchPoints`
- **practice/game mode 제한 또는 경고**: `pointer: coarse`, `maxTouchPoints`, 작은 viewport, iOS/Android UA 보조 확인
- **성능 경량화**: 실제 FPS/latency 측정값 기반이 더 정확

즉, “모바일이라서 차단”보다는 `isLikelyTouchMobile()` 같은 함수로 **모바일 가능성이 높으면 안내/제한 UI를 켜는 방식**이 안전합니다.

> 가볍게 할 수 있는 UI 레이아웃 조정만 조금 해보고자 합니다.
> 비단 모바일만 아니어도 세로로 긴 화면에서 메뉴 영역 + 에딧 패널을 전부 열더라도  악보 영역을 충분히 잘 볼 수 있도록, 메뉴 영역의 세로 최대 비율을 설정하고 메뉴 영역의 요소들이 그에 맞게 축소될 수 있도록 테스트해보고 싶습니다.

완전 자동은 어렵습니다. 브라우저에서 YouTube iframe의 실제 오디오 출력 시점이나 디코딩 지연을 직접 측정할 수 없고, 기기/브라우저/네트워크/블루투스 오디오 지연까지 섞이면 앱이 관측할 수 있는 정보가 부족합니다.

가능한 수준은 세 단계입니다.

1. **기기별 offset 저장**
   가장 현실적입니다. 같은 곡의 `score.musicData.youtube.offsetMs`는 악보 기준값으로 두고, 별도로 `localStorage`에 “이 기기 보정값”을 저장해서 합산합니다.

   ```text
   effectiveOffsetMs = scoreYoutubeOffsetMs + deviceYoutubeOffsetMs
   ```

   예: 악보 offset `-500`, 이 휴대폰 보정 `+220`이면 실제 적용은 `-280`.

2. **수동 캘리브레이션 UI**
   “YouTube sync +100ms / -100ms” 버튼을 두고, 사용자가 들으면서 맞춘 값을 기기 보정으로 저장합니다. 현재 offset 입력창보다 UX가 좋아집니다.

3. **반자동 보정**

네. **MIDI → ScoreFile import를 먼저 구현하는 편이 좋습니다.** 오디오 자동 악보 생성보다 훨씬 통제 가능하고, 이후 오디오 분석 결과를 넣을 “목표 포맷”을 검증하는 역할도 합니다.

이유는 간단합니다.

1. MIDI는 이미 note on/off, pitch, duration 정보가 구조화되어 있습니다.
   즉, “외부 음악 데이터 → 회귀 코드 ScoreFile” 변환 문제만 풀면 됩니다.

2. 오디오 분석은 훨씬 불확실합니다.
   pitch detection, onset detection, tempo 추정, quantization, 단선율/다성 구분, 노이즈 제거가 모두 얽힙니다. 분석 결과를 바로 ScoreFile로 넣으려 하면, 버그가 “분석 문제인지 변환 문제인지” 분리하기 어렵습니다.

3. MIDI import가 있으면 오디오 분석의 중간 산출물을 MIDI-like note event로 정규화할 수 있습니다.
   이후 흐름을 이렇게 잡을 수 있습니다.

```text
MIDI file
→ normalized note events
→ ScoreFile

Audio file / YouTube audio
→ pitch/onset/tempo analysis
→ normalized note events
→ ScoreFile
```

그래서 추천 순서는:

1. 새 브랜치 생성: `feature/midi-import-experiment` 정도
2. MIDI parser 라이브러리 선정
3. MIDI를 내부 중간 구조로 변환
4. 중간 구조를 `ScoreFile`로 변환
5. import preview 또는 간단한 file input 연결
6. 그 다음 오디오 분석 라이브러리 실험

초기 MIDI import는 범위를 좁히는 게 좋습니다.

- 단일 track 또는 사용자가 선택한 track만 import
- note pitch와 duration만 우선 변환
- tempo는 MIDI tempo event를 읽되, 복잡한 tempo change는 1차에서 제한 가능
- velocity는 나중에 dynamics로 연결
- pitch bend, sustain pedal, percussion, chord/polyphony는 후순위

결론적으로, **오디오 자동 악보 생성의 전제 작업으로 MIDI import를 먼저 만드는 것이 맞습니다.** 이걸 먼저 해두면 이후 오디오 분석 기능은 “MIDI 비슷한 note event를 얼마나 잘 만들 수 있느냐”의 문제로 줄어듭니다.

> 1~3, 4~6씩 묶어서 진행합시다.
> MIDI파일의 데이터 구조가 어떻게 되나요? 변환하기 쉬운 형태인가요?

MIDI 파일은 변환하기 꽤 쉬운 편입니다. “오디오”가 아니라 “연주 이벤트 목록”이라서, 이미 악보에 가까운 구조를 갖고 있습니다.

기본 구조는 대략 이렇습니다.

```text
MIDI file
├─ header chunk
│  ├─ format type
│  ├─ track count
│  └─ ticks per quarter note
└─ track chunks[]
   ├─ delta time
   ├─ event
   ├─ delta time
   ├─ event
   └─ ...
```

핵심은 각 track 안의 이벤트입니다.

주요 이벤트:
- `note on`: 어떤 채널에서 어떤 pitch의 음을 누름
- `note off`: 해당 pitch의 음을 뗌
- `set tempo`: BPM 정보
- `time signature`: 박자 정보
- `program change`: 악기 음색
- `control change`: sustain pedal 등
- `pitch bend`: 연속 pitch 변화

MIDI의 시간은 초가 아니라 **tick**입니다. 예를 들어 header에 `ticksPerQuarter = 480`이면 4분음표 하나가 480 tick입니다. `delta time`은 “이전 이벤트 이후 몇 tick 뒤인가”를 뜻합니다.

변환은 보통 이렇게 합니다.

```text
MIDI track events
→ delta time 누적해서 absolute tick 계산
→ note on/off 짝 맞추기
→ { pitch, startTick, endTick, velocity, channel, track } 목록 생성
→ ScoreFile의 cells/rawText 구조로 배치
```

쉬운 부분:
- pitch는 MIDI note number라서 `C4`, `D#4` 같은 음 이름으로 바로 변환 가능
- note 길이는 `endTick - startTick`으로 계산 가능
- tempo/BPM도 meta event로 들어 있음
- track/channel 구분도 있음

어려운 부분:
- 겹치는 음, 즉 chord/polyphony를 현재 ScoreFile 구조에 어떻게 넣을지 정해야 함
- quantization이 필요함. MIDI는 480 tick 같은 세밀한 단위인데, 회귀 코드 score column은 `stepsPerBeat` 기반 격자라서 반올림 정책이 필요함
- sustain pedal은 note off보다 실제 울림이 길어질 수 있음
- pitch bend는 단순 note로 표현하기 어렵고 gliss/cent 변화 후보가 됨
- 드럼 channel 10은 일반 음정 악보로 바꾸기 애매함
- 복잡한 tempo change는 ScoreFile global row에 어떻게 배치할지 정해야 함

그래서 1차 MIDI import는 충분히 현실적입니다. 단, 범위를 이렇게 좁히는 게 좋습니다.

- 단일 melody track 우선
- note on/off만 변환
- tempo/time signature는 기본 global row로 반영
- velocity, program change, pitch bend, sustain은 일단 보존하지 않거나 warning 처리
- polyphony는 첫 구현에서 skip 또는 최고음/최저음 선택

결론: MIDI는 ScoreFile 변환의 좋은 첫 대상입니다. 오디오 분석보다 훨씬 쉽고, 이후 오디오 분석 결과도 MIDI-like note list로 정규화하면 같은 변환 파이프라인을 재사용할 수 있습니다.
