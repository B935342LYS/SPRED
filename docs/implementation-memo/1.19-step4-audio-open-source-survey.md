# Step 4 Audio Open Source Survey

## 1. 문서 목적

본 문서는 `회귀 코드`의 audio generator / playback controller 구현에 앞서, 참고하거나 직접 활용할 수 있는 오픈소스 및 공개 문서를 조사한 결과를 정리한다.

이번 조사의 목적은 다음과 같다.

1. 이미 검증된 Web Audio 기반 구현 사례를 확인한다.
2. `AnalysisResult`를 오디오 스케줄 입력으로 변환할 때 참고할 알고리즘을 선별한다.
3. 직접 의존, 부분 참고, 개념 참고, 비채택을 분리한다.
4. 라이선스와 외부 음원 asset 사용 위험을 구현 전에 기록한다.

본 문서는 구현 메모이며, active spec의 parser / analyzer / renderer / audio 계약을 변경하지 않는다.

## 2. 회귀 코드의 오디오 요구사항

현재 analyzer 출력 기준으로 audio generator가 직접 소비해야 할 핵심 입력은 다음이다.

- `AnalysisResult.timingTimeline`
- `AnalysisResult.dynamicsTimeline`
- `AnalysisResult.trackResults[].events`
- `NoteEvent.sound.midi`
- `NoteEvent.sound.centOffset`
- `NoteEvent.effects`
- `GlissEvent.startSound`, `GlissEvent.endSound`
- `TupletMembership`이 붙은 note/rest event

따라서 audio module은 parser의 `rawText`를 다시 읽지 않고, analyzer가 확정한 의미 이벤트만 사용해야 한다.

초기 구현에서 필요한 변환은 다음이다.

1. tick range를 초 단위 time range로 변환한다.
2. MIDI + cent offset을 frequency 또는 detune 값으로 변환한다.
3. note start / stop을 Web Audio clock 기준으로 예약한다.
4. gliss, vibrato, tremolo는 후속 단계에서 AudioParam automation으로 확장한다.
5. playback controller는 play / pause / stop / seek와 score scroll 동기화를 담당한다.

## 3. 조사 기준

오픈소스 및 공개 자료는 아래 기준으로 평가했다.

- 브라우저 Web Audio API와 직접 관련이 있는가
- 음악 이벤트 scheduling 또는 transport 개념을 제공하는가
- MIDI pitch, cent detune, sample playback, synth voice 중 회귀 코드에 필요한 기능을 제공하는가
- TypeScript / npm / GitHub Pages 환경과 맞는가
- MIT, Apache-2.0, BSD 등 직접 의존에 적합한 라이선스인가
- 음원 sample asset의 라이선스가 코드 라이선스와 분리되어 있는가

## 5. Web Audio API / MDN

- URL: https://developer.mozilla.org/en-US/docs/Web/API/Web_Audio_API
- 관련 페이지:
  - https://developer.mozilla.org/en-US/docs/Web/API/BaseAudioContext/currentTime
  - https://developer.mozilla.org/en-US/docs/Web/API/AudioScheduledSourceNode
  - https://developer.mozilla.org/en-US/docs/Web/API/OscillatorNode/detune
  - https://developer.mozilla.org/en-US/docs/Web/API/AudioParam
- 라이선스 / 성격: 웹 표준 및 MDN 문서. 코드 의존성은 아니며, 구현 기준 문서로 사용한다.

### 참고한 점

`BaseAudioContext.currentTime`은 오디오 스케줄과 timeline 시각화에 사용할 수 있는 초 단위 hardware clock이다. 회귀 코드의 playback controller는 UI의 `Date.now()`나 `setTimeout()` 시간을 기준으로 note를 직접 울리지 않고, `AudioContext.currentTime`을 기준으로 note start / stop을 예약해야 한다.

`AudioScheduledSourceNode` 계열은 `start()`와 `stop()`을 지정 시각에 예약할 수 있다. 초기 oscillator 방식 구현에서는 note마다 source node를 만들고 `start(audioTime)`, `stop(audioTime + duration)`을 호출하는 구조가 적합하다.

`OscillatorNode.detune`은 cent 단위 `AudioParam`이다. 회귀 코드의 `FinalSoundPitch.centOffset`은 `frequency`를 직접 다시 계산하는 방식과 `detune`을 쓰는 방식 모두 가능하다. gliss / vibrato 확장을 고려하면 `frequency` 또는 `detune`의 AudioParam automation을 한 곳에서 관리하는 편이 좋다.

`AudioParam`은 `setValueAtTime()`, `linearRampToValueAtTime()`, `setValueCurveAtTime()` 같은 timeline 기반 automation을 제공한다. 이는 후속 gliss, dynamics, tremolo, vibrato 구현의 기본 기법이 된다.

### 회귀 코드에 활용할 부분

- `midiToFrequency(midi, centOffset)` 유틸을 만들되, oscillator에는 `frequency.setValueAtTime()` 또는 `detune.setValueAtTime()`으로 반영한다.
- note scheduling은 `AudioContext.currentTime + offsetSeconds` 기준으로 수행한다.
- gliss는 `frequency.linearRampToValueAtTime()` 또는 cent 기반 `detune.linearRampToValueAtTime()` 후보로 둔다.
- vibrato는 OscillatorNode 또는 PeriodicWave LFO를 GainNode/AudioParam에 연결하는 방식으로 후속 검토한다.

### 채택하지 않을 부분

MDN 예제 코드를 그대로 복사하지 않는다. 필요한 API 사용 방식만 표준 근거로 삼고, 회귀 코드의 `AnalysisResult` 중심 구조에 맞춰 별도 TypeScript 모듈로 작성한다.

## 6. Lookahead Scheduler 참고

- URL: https://ircam-ismm.github.io/webaudio-tutorials/scheduling/timing-and-scheduling.html
- 관련 원문 배경: Chris Wilson, A Tale of Two Clocks
- 라이선스 / 성격: 공개 튜토리얼. 직접 코드 복사보다 알고리즘 구조 참고에 적합하다.

### 참고한 점

브라우저의 JavaScript timer는 음악 재생 기준으로 충분히 정확하지 않다. 그러나 모든 이벤트를 한 번에 예약하면 편집, seek, tempo 변경에 반응하기 어렵다. lookahead scheduler는 짧은 주기로 다음 구간의 이벤트를 검색하고, 실제 발음은 Web Audio clock에 예약하는 절충 구조이다.

회귀 코드도 편집기와 playback이 결합된 앱이므로 다음 조건을 만족해야 한다.

- 재생 중 UI thread가 약간 지연되어도 이미 예약된 가까운 note는 정확히 울려야 한다.
- stop / seek / 재분석 이후에는 아직 예약하지 않은 미래 이벤트를 새 상태 기준으로 다시 찾을 수 있어야 한다.
- score scroll은 UI frame 기준으로 움직이되, audible note는 audio clock 기준으로 예약해야 한다.

### 회귀 코드에 활용할 부분

초기 playback controller는 아래 구조를 후보로 둔다.

```text
PlaybackController
  -> AudioScheduler
    -> lookahead window 계산
    -> AudioEventQueue에서 다음 note 검색
    -> SynthVoiceFactory에 schedule 요청
  -> scroll/render UI update는 requestAnimationFrame으로 별도 처리
```

`lookaheadSeconds`는 0.10-0.20초 범위, `schedulerIntervalMs`는 25-50ms 범위를 초기값으로 둘 수 있다. 실제 값은 note 밀도와 브라우저 상태를 보면서 조정한다.

### 채택하지 않을 부분

튜토리얼의 metronome 예제 구조를 그대로 가져오지는 않는다. 회귀 코드는 이미 analyzer event list가 있으므로, scheduler는 metronome tick 생성기가 아니라 `AudioScheduleEvent[]`를 시간 범위로 조회하는 역할을 맡는다.

## 7. Tone.js

- URL: https://github.com/Tonejs/Tone.js
- 문서: https://tonejs.github.io/
- 라이선스: MIT
- 성격: Web Audio 기반 음악 프레임워크

### 참고한 점

Tone.js는 browser music application에서 transport, scheduling, synth, effect를 한꺼번에 제공한다. README 기준으로 global transport와 event scheduling을 제공하고, 내부 처리는 native Web Audio node를 활용한다. 또한 npm 의존성으로 쉽게 설치할 수 있다.

Tone.js의 핵심 참고 지점은 다음이다.

- `Transport`처럼 재생 위치와 이벤트 scheduling을 분리하는 구조
- user gesture 후 audio context를 시작해야 하는 브라우저 제약 처리
- synth / gain / effect graph를 모듈 조합으로 구성하는 방식
- 많은 Web Audio boilerplate를 감싸는 API

### 회귀 코드에 활용할 부분

직접 의존으로 채택할 경우 다음 기능을 우선 검토한다.

- `Tone.start()` 또는 Tone context 시작 흐름을 play 버튼 안에서 처리
- `Transport` 또는 `Part`를 사용해 analyzer event를 예약
- `Synth` / `PolySynth`를 초기 음원으로 사용
- `frequency`, `detune`, envelope, gain 관련 API를 gliss / dynamics 확장 후보로 검토

다만 회귀 코드의 시간 구조는 `AnalysisResult.timingTimeline`과 column/tick 기반이므로, Tone.js의 notation time string에 과도하게 의존하지 않는 편이 낫다. 먼저 회귀 코드 내부에서 `AudioScheduleEvent`를 초 단위로 만든 뒤, Tone.js에는 이미 계산된 시각을 전달하는 구조가 더 안전하다.

### 채택 판단

- 직접 의존 가능성: 높음
- 즉시 채택 여부: 조건부
- 판단: 교수님이 기존 코드 활용을 강조했다는 맥락에서는 가장 설명하기 쉬운 직접 의존 후보이다. 다만 회귀 코드의 analyzer 중심 계약을 보존하려면 Tone.js가 parser/analyzer를 대체하지 않고, audio backend 중 하나로만 들어와야 한다.

## 8. smplr

- URL: https://github.com/danigb/smplr
- 라이선스: MIT
- 성격: Web Audio API용 sampled instrument library

### 참고한 점

smplr는 Web Audio API 위에서 sample 기반 악기를 제공한다. README 기준으로 General MIDI soundfont, piano, drum machine, mallet 등 여러 sampled instrument를 제공하며, note start에는 MIDI note number와 초 단위 time/duration을 사용할 수 있다. 또한 scheduler 옵션, sample load progress, offline rendering, stop 함수 등을 제공한다.

### 회귀 코드에 활용할 부분

초기 oscillator 음색이 충분하지 않을 경우, smplr를 sampled backend 후보로 둘 수 있다.

```text
AudioBackend
  - OscillatorBackend
  - ToneBackend
  - SmplrBackend
```

이렇게 backend interface를 두면 초기에는 oscillator로 검증하고, 보고서 또는 후속 버전에서 smplr 기반 악기 음색을 추가할 수 있다.

활용 후보:

- MIDI note number 기반 `start({ note, time, duration })`
- instrument load progress를 UI status에 표시
- sample cache / shared scheduler 개념 참고
- piano / marimba 등 기본 악기 preset을 optional dependency로 검토

### 주의할 점

smplr의 코드 라이선스는 MIT이지만, 실제 sample asset은 별도 출처를 가진다. README도 온라인 sample 저장소를 사용한다고 설명한다. 따라서 GitHub Pages에 sample을 직접 bundle하거나 보고서에 “모든 음원 자산을 MIT로 사용했다”고 쓰면 안 된다. 직접 포함 전에는 각 instrument sample의 출처와 라이선스를 별도로 확인해야 한다.

### 채택 판단

- 직접 의존 가능성: 중간
- 즉시 채택 여부: 후속 후보
- 판단: 음색 품질을 높이는 데 유용하지만, 초기 오디오 모듈의 핵심 난제는 scheduler와 analyzer event 변환이므로 첫 구현에서는 optional backend로 남기는 편이 안전하다.

## 9. soundfont-player

- URL: https://github.com/danigb/soundfont-player
- 성격: Web Audio API용 soundfont loader/player
- 상태: 저장소가 archived 상태이며, README에서 더 나은 대안으로 smplr를 안내한다.

### 참고한 점

soundfont-player는 MIDI note number, detune, polyphony, note list scheduling 등 회귀 코드와 관련 있는 API 아이디어를 제공한다. 특히 “soundfont loader와 note player를 분리한다”는 구조는 sampled backend를 설계할 때 참고할 만하다.

### 채택하지 않는 이유

저장소가 archived 상태이므로 신규 구현의 직접 의존으로는 적합하지 않다. 또한 기본 soundfont URL이 외부 pre-rendered soundfont package에 의존하므로, 코드 라이선스와 sample asset 라이선스를 분리해 관리해야 한다.

### 채택 판단

- 직접 의존 가능성: 낮음
- 활용 구분: 비교 참고
- 판단: 보고서에는 “이전 세대 Web Audio soundfont loader 사례”로 언급하고, 실제 구현 후보는 smplr 쪽을 우선한다.

## 10. standardized-audio-context

- URL: https://github.com/chrisguttandin/standardized-audio-context
- 라이선스: MIT
- 성격: Web Audio API cross-browser wrapper

### 참고한 점

standardized-audio-context는 Web Audio API를 브라우저별 차이 없이 쓰기 위한 wrapper이다. README 기준으로 global scope를 patch하지 않는 ponyfill 방식을 사용하며, Web Audio 표준에 가깝게 동작하는 API를 제공한다.

### 회귀 코드에 활용할 부분

초기 target이 desktop-class browser 중심이면 native `AudioContext`만으로 시작해도 된다. 그러나 모바일 view/load/play 범위까지 고려하거나 Safari 계열 문제가 반복되면, 아래 영역에 도입을 검토할 수 있다.

- `AudioContext` 생성과 resume/suspend 처리
- browser compatibility issue 최소화
- 테스트 가능한 audio context wrapper 제공

### 채택 판단

- 직접 의존 가능성: 중간
- 즉시 채택 여부: 보류
- 판단: 초기 구현 복잡도를 줄이기 위해 native Web Audio API로 시작하되, 호환성 문제가 실제로 발생하면 도입한다.

## 11. midi-js-soundfonts / FluidR3 GM 계열

- URL: https://github.com/gleitz/midi-js-soundfonts
- 라이선스: README 기준 Creative Commons Attribution 3.0
- 성격: MIDI.js에서 사용할 수 있는 pre-rendered General MIDI soundfont package

### 참고한 점

회귀 코드가 sampled instrument를 제공하려면, 코드 라이브러리뿐 아니라 음원 asset의 라이선스를 별도로 다뤄야 한다. midi-js-soundfonts는 Web Audio 기반 soundfont loader들이 자주 참조하는 asset source이며, FluidR3 GM 같은 General MIDI 음원 접근 방식을 보여준다.

### 주의할 점

CC BY 3.0은 attribution 요구가 있다. 또한 soundfont package는 원본 sample의 권리와 배포 조건을 확인해야 한다. 따라서 이 저장소의 asset을 그대로 `regression-code/`에 포함하는 것은 초기 구현에서 피한다.

### 채택 판단

- 직접 bundle 가능성: 낮음
- 활용 구분: sample asset 라이선스 검토 사례
- 판단: 보고서에서는 “음원 asset은 코드 라이선스와 별도로 검토해야 한다”는 근거로 사용한다.

## 12. 회귀 코드 적용 설계안

오픈소스 조사 결과를 반영한 audio module 구조 후보는 다음이다.

```text
src/audio/
  audio_types.ts
  tick_time_mapper.ts
  note_event_scheduler.ts
  audio_event_queue.ts
  oscillator_backend.ts
  playback_controller.ts
```

### 12.1. `audio_types.ts`

`AnalysisResult`를 직접 재생하지 않고, audio module 내부 schedule DTO로 변환한다.

```ts
type AudioScheduleEvent = {
  eventId: string;
  trackId: string;
  startSeconds: number;
  endSeconds: number;
  midi: number;
  centOffset: number;
  velocity: number;
  effects: AudioScheduleEffect[];
};
```

이 구조를 두는 이유는 renderer item builder와 비슷하다. analyzer 결과는 공식 의미 구조이고, audio backend는 초 단위 예약에 최적화된 DTO를 소비하는 편이 테스트하기 쉽다.

### 12.2. `tick_time_mapper.ts`

`AnalyzedTimeSegment`를 기준으로 tick을 seconds로 변환한다.

초기 구현:

- `stepsPerBeat`
- `startBpm`
- `endBpm`
- `bpmCurve`

을 읽되, 첫 단계에서는 constant BPM segment부터 검증한다.

후속 구현:

- linear tempo segment
- seek position에서 seconds -> tick 역변환
- score scroll 동기화를 위한 playhead mapping

### 12.3. `audio_event_queue.ts`

lookahead scheduler가 `nowSeconds`, `lookaheadSeconds`로 다음 이벤트를 조회할 수 있게 정렬 배열 또는 cursor를 가진다.

초기 구현은 전체 `AudioScheduleEvent[]`를 `startSeconds` 오름차순으로 정렬한다. 부분 갱신이 붙기 전에는 단순 배열과 cursor가 충분하다.

### 12.4. `oscillator_backend.ts`

초기 backend는 외부 sample asset 없이 oscillator + gain envelope로 구현한다.

채택 이유:

- GitHub Pages 배포가 쉽다.
- sample license 문제가 없다.
- `@m`, gliss, vibrato, tremolo를 AudioParam automation으로 검증하기 쉽다.

오픈소스 활용 지점:

- MDN Web Audio API의 source scheduling 기준
- IRCAM lookahead scheduler 구조
- Tone.js의 Transport / Synth 책임 분리 개념

### 12.5. `playback_controller.ts`

play / pause / stop / seek 상태를 관리한다.

초기 상태 모델 후보:

```ts
type PlaybackState =
  | { kind: "stopped" }
  | { kind: "playing"; startedAtAudioTime: number; startedAtScoreSeconds: number }
  | { kind: "paused"; pausedAtScoreSeconds: number };
```

UI scroll은 audio scheduling과 분리해 `requestAnimationFrame`에서 현재 score time을 계산해 갱신한다.

## 13. 최종 채택 제안

초기 오디오 구현은 다음 순서가 적절하다.

1. native Web Audio API + oscillator backend로 첫 재생을 구현한다.
2. IRCAM / Chris Wilson 방식의 lookahead scheduler를 회귀 코드 event queue에 맞게 구현한다.
3. Tone.js는 직접 의존 후보로 유지하되, 첫 구현에서는 backend interface 뒤에 둔다.
4. smplr는 음색 개선 단계에서 optional sampled backend로 검토한다.
5. soundfont asset은 코드와 별도 라이선스 검토가 끝나기 전까지 repository에 포함하지 않는다.

이 판단은 “오픈소스 활용을 했다”는 설명과 “회귀 코드의 analyzer 중심 계약을 보존했다”는 설명을 동시에 만족한다.

보고서에는 다음처럼 정리할 수 있다.

> 오디오 모듈은 Web Audio API의 표준 scheduling 구조와 공개 lookahead scheduler 알고리즘을 참고하여 설계하였다. Tone.js, smplr 등 기존 Web Audio 오픈소스의 transport, synth, sampled instrument 구조를 비교 검토했으며, 초기 구현은 analyzer가 생성한 event list를 초 단위 schedule event로 변환한 뒤 Web Audio clock에 예약하는 방식으로 진행하였다. 외부 sample asset은 코드 라이선스와 별도 권리 검토가 필요하므로 초기 구현에는 포함하지 않았다.
