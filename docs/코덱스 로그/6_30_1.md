1~2번은 우려가 맞습니다. **예제 악보 JSON을 TypeScript import로 번들에 넣으면 불리합니다.** 지금 기본 악보처럼 `import ... ?raw`로 가져오면 JS 번들에 포함될 수 있어서, 예제 10개만 있어도 초기 로드가 2~3MB 가까이 커질 수 있습니다.

- 앱 시작 시 번들 포함: 아주 작은 기본 템플릿 1개만 유지
- 예제 목록: 작은 `manifest.json`만 로드
- 실제 예제 악보 JSON: 사용자가 선택했을 때 `fetch()`로 지연 로드
- Supabase: manifest 또는 JSON URL을 원격에서 가져오는 선택 경로

| DB 컬럼 | 앱 타입 | NULL 권장 | 이유 |

| `score_url` | `string` | No | manifest 필수 |
| `thumbnail_url` | `string?` | Yes | 선택 필드 |
| `display_order` | 내부 정렬용 | No | 기본값 0 |
| `is_published` | 공개 여부 | No | RLS 필터용 |

중요한 정책은 **DB의 `NULL`을 앱 manifest에서는 `undefined`로 정규화**하는 것입니다. 반대로 필수 필드가 `null`이면 앱 validator에서 거부해야 합니다.

추천 제약:
- `title`, `artist`, `license_label`, `score_url`은 `trim(...) <> ''`
- `supported_tracks`는 빈 배열 금지
- `supported_tracks` 원소는 `basic`, `optional`, `extra`만 허용
- `score_url`, `thumbnail_url`, `license_url`은 HTTPS URL 또는 허용한 로컬 상대 경로만 앱에서 재검증
- `duration_seconds`, `size_bytes`는 양수만 허용
- `is_published = true`인 row만 anon select 허용

SQL injection 관점에서는 앱이 임의 SQL을 만들지 않고 Supabase REST의 고정 `select`만 쓰면 위험이 낮습니다. 그래도 DB 제약 + RLS + 앱 validator를 모두 둬야 합니다.

제외 또는 보류:
- `thumbnail_url`: 보류 권장. 앨범아트/영상 썸네일 저작권 리스크가 큽니다.
- `description`: 보류 또는 아주 짧은 `note`로 대체. UI 표에는 부담됩니다.
- `license_label/url/note`: 공개 예제 배포 단계에서는 중요하지만, 현재 “허락된 사람만 임시 테스트” 단계에서는 보류 가능. 다만 내부 관리용 `rights_note` 정도는 나중에 둘 수 있습니다.

**2. `created_at`, `updated_at`**
사용자에게 보여줄 값이라면 유지하면 됩니다. 다만 의미를 분명히 해야 합니다.

- `created_at`: 예제 catalog 등록일
- `updated_at`: 예제 catalog 또는 score 파일 갱신일

ScoreFile 내부의 `musicData.createdAt/updatedAt`과는 다릅니다. UI에서는 “등록일”, “갱신일”처럼 표시해야 혼동이 적습니다.

**3. 내부 JSON 값을 스키마에 자동 대입하는 방법**
가능은 하지만, DB만으로는 어렵습니다.

`score_url`이 외부 JSON 파일을 가리키는 구조라면 Postgres check constraint가 그 JSON을 열어 `musicTitle`, `scoreDifficulty`, `columnCount` 등을 검증할 수 없습니다.

현실적인 방법은 셋입니다.

1. **수동 입력**
   가장 단순합니다. 사용자가 JSON을 올리고, title/artist/difficulty 등을 직접 DB에 넣습니다.

2. **관리용 스크립트**
   로컬 Node script가 Score JSON을 읽고 기존 validator로 검증한 뒤, metadata를 추출해서 Supabase row를 upsert합니다. 장기적으로 가장 안정적입니다.

3. **Supabase Edge Function**
   업로드/등록 요청을 Edge Function으로 보내면, 함수가 JSON을 fetch/검증/metadata 추출/upsert합니다. 보안은 좋지만 구현량이 늘어납니다.

현재 단계에서는 1번으로 시작하고, 예제 수가 늘어나면 2번 관리 스크립트를 추가하는 쪽이 좋습니다.

**4. “허락된 사람만 접근”**
중요합니다. File -> Examples에서 암호 단어를 입력하게 하는 것만으로는 실제 DB 보호가 되지 않습니다.

GitHub Pages 앱은 프론트엔드 코드가 모두 공개됩니다. Supabase URL과 anon key, REST 요청 방식도 브라우저 devtools에서 볼 수 있습니다. 따라서 클라이언트 쪽 암호 입력은 “UI 숨김”일 뿐, DB 접근 제한은 아닙니다.

임시 보안 수준별 선택지는 이렇습니다.

- **가벼운 UI 잠금**
  File -> Examples 클릭 시 단어 입력.
  장점: 구현 쉬움.
  단점: 실제 DB 보호 아님.

- **Edge Function 암호 게이트**
  앱이 Supabase table을 직접 읽지 않고, Edge Function에 암호 단어를 보냅니다. 함수가 맞는 경우에만 manifest를 반환합니다.
  장점: DB table은 public read를 막을 수 있음. 암호는 서버 env에 둠.
  단점: Edge Function 구현 필요.

- **Supabase Auth 초대제**
  로그인한 특정 사용자만 read 가능.
  장점: 가장 정석적.
  단점: 테스트 진입 장벽이 커짐.

현재 요구에는 **Edge Function 암호 게이트**가 가장 균형이 좋습니다. “가벼운 암호” UX를 유지하면서도, DB를 public read로 열지 않을 수 있습니다.

정리하면 스키마는 더 줄이고, 보안 모델은 이렇게 바꾸는 게 좋습니다.

```text
GitHub Pages app
  -> File > Examples
  -> 암호 단어 입력
  -> Supabase Edge Function 호출
  -> 함수가 암호 확인
  -> 함수가 example_scores에서 is_published=true 목록 조회
  -> 앱은 받은 manifest를 검증
  -> 선택한 score_url fetch
  -> 기존 ScoreFile validation 통과 후 로드
```

이 방향이면 SQL injection도 줄어듭니다. 앱은 SQL을 만들지 않고, Edge Function도 고정 query만 실행하면 됩니다.
