# UI MVP 요구사항 명세

## 1. 문서 목적

본 문서는 `회귀 코드`의 첫 번째 웹 UI 구현 범위를 정의한다.

`1.9-mvp-analyzer-renderer-ui-spec.md`가 analyzer, renderer, audio, UI의 전체 MVP 연결 범위를 정했다면, 본 문서는 그중 UI 화면 구조와 사용자 조작 흐름을 별도로 구체화한다.

본 문서의 목표는 다음과 같다.

- legacy `이세계 코드`의 화면 구조를 참고하되, `회귀 코드`의 웹 앱 구조에 맞는 UI 영역을 정의한다.
- 메뉴 영역, 레이아웃 라벨 영역, 악보 영역의 책임을 분리한다.
- MVP에서 실제 동작할 기능과 후속 구현을 위한 비활성 확장 지점을 구분한다.
- 구현 전 필요한 최소 상태 모델과 event-action table을 고정한다.

## 2. 기본 해석

사용자가 말한 3개 영역은 다음처럼 해석한다.

1. 메뉴 영역
   - 실제 `<canvas>`가 아니라 HTML DOM 기반 UI 영역으로 구현한다.
   - 버튼, 입력창, select, toggle, 접이식 패널, 파일 입력 등은 DOM 요소로 둔다.
2. 레이아웃 캔버스 영역
   - 악보 행 라벨과 행 간격을 보여주는 고정 좌측 영역이다.
   - 실제 canvas 또는 canvas와 DOM의 조합으로 구현할 수 있다.
3. 악보 캔버스 영역
   - 실제 note grid와 note rectangle, overlay를 표시하는 주 악보 영역이다.
   - legacy `이세계 코드`처럼 canvas 기반으로 구현한다.

상단 메뉴를 canvas가 아니라 DOM으로 두는 이유는 다음과 같다.

- 버튼과 입력 요소의 접근성 및 키보드 조작이 쉽다.
- 상태 변경, disabled 처리, 파일 선택, 드롭다운 구현이 단순하다.
- 반응형 배치와 CSS 스타일링이 canvas보다 안정적이다.

## 3. 전체 화면 구조

MVP 화면은 아래 구조를 가진다.

```text
App Root
  ├─ Menu Area
  │   ├─ Left Tool Group
  │   ├─ Center Player Group
  │   └─ Right YouTube Group
  ├─ Edit Panel
  └─ Score Viewer
      ├─ Layout Label Area
      └─ Score Canvas Area
```

`Edit Panel`은 메뉴 영역 아래에 접히고 펼쳐지는 패널로 둔다.
보기 모드에서는 접혀 있고, edit mode가 켜졌을 때만 표시된다.

Edit Panel이 펼쳐질 때 레이아웃 캔버스 영역과 악보 캔버스 영역이 화면 밑으로 밀려 내려가서는 안 된다.
메뉴 영역 아래의 사용 가능 높이 안에서 edit panel과 score viewer가 함께 배치되어야 하며, edit panel이 열리면 score viewer는 남은 높이에 맞춰 줄어들고 내부 scroll을 유지한다.
즉 전체 문서 스크롤이 아니라 앱 내부 viewport 조정으로 처리한다.

## 4. 메뉴 영역

메뉴 영역은 상단 고정 UI이다.

### 4.1. 왼쪽 기능 그룹

왼쪽에는 음악 재생 자체보다 앱 조작, 보기 설정, 도움말에 가까운 기능을 모은다.
개별 버튼이 많아지므로 legacy `이세계 코드`의 visual menu처럼 hover menu 또는 click menu로 묶는다.

MVP에서 표시할 메뉴 그룹은 다음이다.

- File
- View
- Tools
- Help

각 메뉴의 소속 항목은 다음을 우선안으로 둔다.

| Menu | Items | MVP 동작 |
| --- | --- | --- |
| File | JSON file load, JSON file download, reload | load/reload는 동작, download는 disabled |
| View | zoom, fullscreen, fit height, reverse mode, UI theme, text scale | zoom만 동작, 나머지는 placeholder |
| Tools | edit mode, expand, track menu | track menu는 동작, edit/expand는 placeholder 중심 |
| Help | Info, Manual | Info는 popup, Manual은 YouTube hyperlink |

MVP에서 실제 동작할 항목은 다음이다.

- initial sample auto load
- JSON file load
- JSON file download placeholder
- reload
- visual menu 중 zoom
- track menu 중 active track 표시/선택
- Info popup
- Manual hyperlink

MVP에서 비활성 또는 placeholder로 둘 수 있는 항목은 다음이다.

- edit mode 내부의 실제 셀 편집 적용
- expand에 의한 실제 열 추가 저장
- fullscreen
- reverse mode
- UI theme
- text scale
- fit height

sample score는 별도 버튼으로 두지 않고 앱 시작 시 자동 로드한다.
JSON file download는 향후 JSON 파일 목록을 모아 둔 Google Drive, Google Sheets, 자체 DB, 또는 정적 링크 페이지로 연결될 수 있는 자리로 둔다.
MVP에서는 disabled 버튼으로 표시한다.

Info는 법적 분쟁을 피하기 위한 안내 popup을 호출한다.
popup에는 최소 다음 내용을 포함한다.

- 본 프로그램은 비영리 목적으로 제작되었다.
- 본 프로그램은 음악 파일 다운로드를 지원하지 않는다.
- 모든 음악 저작물의 권리는 적법한 저작권자에게 있다.
- 저작권자의 요청이 있는 경우 관련 악보 데이터는 DB 또는 공개 목록에서 내려갈 수 있다.
- 사용자는 저작권을 존중해야 한다.

Manual은 사용법 안내 영상으로 연결되는 YouTube hyperlink이다.
MVP에서 실제 안내 영상 URL이 아직 없으면 disabled 또는 준비 중 상태로 표시한다.

### 4.2. 중앙 플레이어 그룹

중앙에는 음악 정보와 재생 조작을 묶어 음악 플레이어 UI처럼 배치한다.

표시 정보는 다음이다.

- artist
- title
- 재생 상태
- 현재 tick 또는 column
- 전체 길이

artist/title은 `musicData.musicArtist`와 `musicData.musicTitle`을 사용한다.
두 필드는 validate 단계에서 존재가 보장된다.
다만 공문자열이면 각각 `"unknown"` 기본값으로 표시한다.

조작 요소는 다음이다.

- play/pause toggle button
- stop
- seek bar
- volume
- waveform

MVP에서는 play, pause, stop, seek bar를 실제 구현한다.
play 버튼은 일반적인 음악 플레이어처럼 재생 중에는 pause 역할로 전환된다.
stop 버튼은 현재 위치 정지가 아니라 곡의 처음으로 되돌리는 정지 동작이다.

seek bar는 악보 영역의 horizontal scroll 위치와 연결한다.
사용자가 seek bar를 움직이면 악보 캔버스 영역의 스크롤 위치가 바뀌고, 악보를 직접 스크롤하면 seek bar 위치도 갱신되어야 한다.
seek bar 단위는 column 정수 단위로 둔다.

후속 확장 지점은 다음이다.

- loop
- 자세한 정보 보기
- 음악 정보 수정

자세한 정보 보기와 음악 정보 수정은 후속 단계에서 metadata popup으로 구현할 수 있다.
MVP에서는 버튼 위치만 만들고 disabled 상태로 둔다.

### 4.3. 오른쪽 YouTube 그룹

오른쪽에는 YouTube 동기화 관련 UI를 둔다.

MVP에서는 YouTube 재생 연동을 구현하지 않는다.
다만 후속 확장을 위해 다음 UI 위치를 확보한다.

- YouTube mode toggle
- URL 또는 video id 입력
- offset 입력
- re-sync 버튼
- 연결 상태 표시

초기 구현에서는 전체 YouTube 그룹을 disabled 또는 준비 중 상태로 표시할 수 있다.

## 5. Edit Panel

Edit Panel은 기존 `이세계 코드`처럼 접었다 펼 수 있는 구조를 따른다.

MVP에서는 edit mode UI의 배치와 상태만 먼저 잡고, 실제 score mutation은 후속 단계로 둘 수 있다.

### 5.1. 기존 구조에서 유지할 영역

다음 영역은 legacy 구조를 참고하여 유지한다.

- Number / global value 입력 영역
- Default note 입력 영역
- Long / hold 입력 영역
- Gliss 입력 영역
- Trem 입력 영역
- Tuplet 입력 영역
- Edit Actions 영역

### 5.2. Tuplet 영역 추가 요구사항

Tuplet 영역에는 currentText를 `pletExtend`로 전환하는 버튼을 추가한다.

이 버튼의 의도는 다음과 같다.

- 사용자가 현재 선택한 note 입력값 또는 currentText를 tuplet head가 아니라 pletExtend 입력으로 전환할 수 있게 한다.
- tuplet head와 pletExtend가 다른 parser kind로 해석되는 점을 UI에서 명확히 구분한다.

MVP에서는 버튼을 배치하되 실제 tuplet 편집 기능은 비활성으로 둘 수 있다.

### 5.3. Pitch Modifier 영역 추가 요구사항

Edit Panel에는 단일 pitch modifier 영역을 추가한다.

이 영역에는 다음 입력을 둔다.

- absolutePitch
  - 계이름 드롭다운으로 입력한다.
  - 후속 구현에서는 선택값을 `@p(midi_num)` 또는 equivalent internal value로 변환한다.
- microPitch
  - 숫자 전용 입력창으로 입력한다.
  - 허용 범위는 기존 프로젝트 결정에 따라 `-100`부터 `100`까지이다.
  - 소수점 이하 1자리까지 허용한다.

MVP analyzer는 `@p`, `@m`을 아직 소비하지 않지만, UI 배치는 후속 확장을 위해 미리 반영한다.

### 5.4. Edit Actions

Edit Actions에는 다음 버튼을 둔다.

- undo
- apply edits
- clear edits

MVP에서는 실제 편집 저장이 구현되지 않을 수 있으므로 disabled 상태로 둘 수 있다.

## 6. 레이아웃 캔버스 영역

레이아웃 캔버스 영역은 악보 좌측에 고정되어 row label과 행 간격을 보여준다.

기본 동작은 legacy `이세계 코드`와 유사하게 둔다.

- note row label 표시
- global row label 항상 표시
- score canvas와 세로 스크롤 동기화
- row height를 반영한 라벨 위치 표시

상단에는 레이아웃 조작 버튼을 둔다.

- modify
- save
- reset

MVP에서는 layout modify/save/reset을 구현한다.
기본 저장소는 `localStorage`로 한다.
후속 단계에서는 layout 설정을 별도 JSON 파일로 export/import할 수 있는 경로를 추가할 수 있다.

사용자가 레이아웃을 변경해 저장하는 순간, 원본 `instData.presetId`와 구분되는 사용자 layout preset id를 새로 지정해야 한다.
즉 `instData.presetId`는 악기/기본 프리셋의 식별자로 유지하고, 사용자 레이아웃 선택에는 별도의 `layoutPresetId`를 사용한다.
사용자는 저장된 layout preset을 드롭다운으로 전환할 수 있어야 한다.
사용자에게 보이는 layout preset 이름과 내부 저장 id는 분리한다.
예를 들어 `layoutPresetDisplayName`은 `"내 DX용"`처럼 표시명으로 쓰고, `layoutPresetId`는 `"layout-20260527-001"`처럼 저장 key에 안전한 내부 id로 둔다.

localStorage에는 layout preset 목록용 index key와 개별 layout data key를 분리해 저장한다.

- 목록 index key
  - `layout-index:{scoreWriter}:{instrumentPresetId}`
  - 드롭다운에 표시할 layout preset 목록을 저장한다.
- 개별 layout data key
  - `layout:{scoreWriter}:{instrumentPresetId}:{layoutPresetId}`
  - 실제 row height 등 사용자 layout customization 값을 저장한다.

목록 index 값은 아래 형태를 권장한다.

```ts
type UserLayoutPresetIndexItem = {
  layoutPresetId: string;
  layoutPresetDisplayName: string;
  updatedAt: string;
};
```

레이아웃 변경은 score JSON에 직접 저장하지 않고, 사용자별 layout customization으로 분리한다는 기존 프로젝트 결정을 따른다.

## 7. 악보 캔버스 영역

악보 캔버스 영역은 실제 악보 테이블을 표시한다.

MVP에서 표시할 항목은 다음이다.

- column grid
- row grid
- note rectangle
- hold로 병합된 긴 note rectangle
- note display text

후속 확장 항목은 다음이다.

- 100 column 단위 좌표 숫자 표시
- beat/bar line
- gliss line
- vibrato wave
- trem pattern
- tuplet bracket / slot
- dynamics curve
- edit selection overlay
- hover inspector

100 column 단위 좌표 숫자는 후속 기능으로 둔다.
다만 renderer 좌표 계산에서 column index 표시를 추가할 수 있는 overlay layer를 고려한다.

별도 playhead 선은 MVP에서 그리지 않는다.
레이아웃 캔버스 영역과 악보 캔버스 영역의 경계가 현재 재생 기준선 역할을 한다.
재생 중에는 playback controller가 악보 영역의 horizontal scroll을 갱신하여 현재 tick이 이 경계에 오도록 한다.
따라서 renderer는 별도 playhead drawing보다 scroll-position 기반 재생 위치 동기화를 우선 구현한다.

## 8. Canvas Layer 구조

legacy 구조를 참고하여 score viewer는 역할별 layer를 분리할 수 있다.

MVP 권장 layer는 다음이다.

- base layer
  - grid, row background, column background
- note layer
  - `NoteEvent` rectangle과 display text
- marker layer
  - 향후 loop marker, beat/bar marker
- interaction layer
  - hover, selection, edit overlay

MVP에서는 base, note layer를 우선 구현한다.
marker layer는 beat/bar line 또는 loop marker 구현 시 활성화한다.
interaction layer는 후속 edit mode 구현 시 활성화한다.

레이아웃 라벨 영역도 별도 canvas 또는 DOM/canvas 혼합 영역으로 둔다.
중요한 점은 라벨 영역과 악보 영역의 세로 위치가 항상 같은 row layout을 기준으로 계산되어야 한다는 것이다.

## 9. MVP 상태 모델

MVP UI 상태는 아래 필드를 가진다.

```ts
type UiMode = "view" | "edit";
type PlaybackState = "stopped" | "playing" | "paused";
type YoutubeState = "off" | "placeholder";
type LayoutEditState = "off" | "editing";
type DirtyState = "clean" | "dirty";

type UiState = {
  uiMode: UiMode;
  playbackState: PlaybackState;
  youtubeState: YoutubeState;
  layoutEditState: LayoutEditState;
  dirtyState: DirtyState;
  activeTrackIds: string[];
  layoutPresetId: string;
  layoutPresetDisplayName: string;
  zoom: number;
  volume: number;
  waveform: OscillatorType;
  seekTick: number;
};
```

기본 active track은 `"basic"` 하나로 둔다.
`basic`, `optional`, `extra` track은 validate 단계에서 모두 존재가 보장되므로 fallback track 정책은 두지 않는다.
optional/extra는 기본 비활성 표시 상태로 시작하지만, 사용자가 활성화하면 이미 존재하는 빈 track에 셀을 추가한다.

## 10. MVP Event-Action Table

| Event | Guard | Action | Result |
| --- | --- | --- | --- |
| app boot | 항상 가능 | bundled/sample fixture 자동 로드, validate, parse, analyze, render, prepareAudioEvents | sample score 표시 |
| JSON file load | 파일 선택됨 | 파일 읽기, validate, parse, analyze, render, prepareAudioEvents | score 표시 |
| JSON file download click | disabled | 아무 동작 없음 | 준비 중 상태 유지 |
| Info click | 항상 가능 | 법적 안내 popup 열기 | Info popup 표시 |
| Manual click | URL 설정됨 | 사용법 안내 YouTube URL 열기 | 새 탭 또는 현재 탭 이동 |
| reload click | score loaded | 현재 score를 다시 parse/analyze/render | 화면 갱신 |
| play click | score analyzed, playback stopped | audio context 준비, playback start, 악보 scroll 시작 | `playbackState = "playing"` |
| play click | playback playing | scheduled audio 중지, 현재 seek column 보존 | `playbackState = "paused"` |
| play click | playback paused | 현재 seek column에서 audio 재스케줄링 후 playback resume | `playbackState = "playing"` |
| stop click | playback playing 또는 paused | 모든 scheduled audio 정지, seekTick을 0으로 변경, 악보 scroll을 시작 위치로 복귀 | `playbackState = "stopped"` |
| seek bar input | score rendered | seek column 갱신, 악보 horizontal scroll 동기화 | 현재 위치 변경 |
| score horizontal scroll | score rendered | scroll 위치를 column으로 변환, seek bar 갱신 | 현재 위치 변경 |
| zoom change | score rendered | renderer scale 갱신, viewer 재배치 | zoom 반영 |
| volume change | audio ready | master gain 값 갱신 | 음량 반영 |
| waveform change | audio ready | 다음 재생부터 oscillator type 반영 | wave 반영 |
| active track toggle | score analyzed | activeTrackIds 갱신, render/audio 대상 갱신 | 선택 track만 표시/재생 |
| edit mode toggle | playback stopped 또는 paused | edit panel open/close, score viewer 높이 재계산 | `uiMode` 변경 |
| layout modify click | score loaded | layout edit controls 활성화 | `layoutEditState = "editing"` |
| layout save click | layout editing, layout preset name set | 새 내부 `layoutPresetId`를 생성하고 사용자 layout customization을 localStorage에 저장 | `layoutEditState = "off"` |
| layout preset select change | score loaded | 선택한 layout preset을 localStorage에서 읽어 row layout에 적용 | layout preset 전환 |
| layout reset click | score loaded | localStorage layout customization 초기화 | 기본 layout 복원 |
| YouTube toggle click | 항상 가능 | MVP에서는 준비 중 상태 표시 | 실제 연동 없음 |

MVP에서 edit mode는 playback 중 켜지지 않도록 한다.
재생 중 편집을 허용하면 audio/scroll/render 상태 동기화가 복잡해지므로 후속 단계에서 별도 정책을 정한다.
paused 상태에서는 edit panel을 열 수 있으나, 실제 편집 적용은 후속 단계로 둔다.

## 13. 확정된 구현 결정과 후속 확인

아래 항목은 MVP 구현 기준으로 확정한다.

1. JSON file download의 초기 동작
   - MVP에서는 disabled 버튼으로 둔다.
2. 중앙 플레이어의 자세한 정보 보기 / 수정 버튼
   - MVP에서는 버튼만 만들고 disabled 상태로 둔다.
3. seek bar 단위
   - MVP에서는 column 정수 단위로 둔다.
4. pause 구현 방식
   - MVP에서는 scheduled audio를 중지하고 seek column을 보존한 뒤 resume 시 재스케줄링한다.
5. layout JSON export/import
   - MVP에서는 localStorage 저장만 먼저 구현한다.
6. metadata 필드
   - 중앙 플레이어는 `musicData.musicArtist`, `musicData.musicTitle`을 사용하고, 공문자열이면 `"unknown"`으로 표시한다.
7. active track
   - validate 단계에서 `basic`, `optional`, `extra` track 존재가 모두 보장된다.
   - 기본 active track은 항상 `"basic"`이다.
8. layout storage key
   - 사용자 저장 레이아웃은 원본 `instData.presetId`와 구분되는 `layoutPresetId`를 가진다.
   - 사용자 표시명 `layoutPresetDisplayName`과 내부 id `layoutPresetId`는 분리한다.
   - 목록 key는 `layout-index:{scoreWriter}:{instrumentPresetId}`를 사용한다.
   - 개별 저장 key는 `layout:{scoreWriter}:{instrumentPresetId}:{layoutPresetId}`를 사용한다.
