# 1.3. 1단계 Score 검증 및 테스트 환경 메모
> 본 설계 메모 문서 1.3에서는 1단계 구현 중 `score_validate.ts`와 최소 테스트 환경 조성 과정을 기록한다.
> 본 문서는 active specification을 대체하지 않는다.
> 구현 중 생긴 판단 근거와 개발 환경 설정 이유만 보충한다.

## 1. 개요

`json_load.ts`를 구현하는 과정에서 JSON 파싱과 ScoreFile 구조 검증을 어디까지 분리할지 다시 검토하였다.
초기에는 `json_load.ts`에서 `parsed as ScoreFile`로 타입 단언을 하고 후속 구현을 진행하는 방식을 고려하였다.
그러나 이 방식은 후속 모듈인 `build_score_indexes.ts`와 parser 구현 단계에서 다음 문제가 생길 수 있다.

- 실제로는 `ScoreFile` 구조가 아닌 값이 후속 모듈로 전달될 수 있다.
- `rowId`, `trackId`, `stringId`의 존재 여부를 후속 모듈에서 반복 방어해야 한다.
- 나중에 validator를 추가할 때 `json_load.ts`의 반환 타입과 호출부를 다시 수정해야 한다.
- 테스트 fixture가 valid한지 확인하지 못한 채 인덱스 생성 로직을 구현하게 된다.

따라서 1단계에서 완성형 validator를 만들지는 않더라도, 최소 구조 검증을 담당하는 `score_validate.ts`를 먼저 추가하기로 하였다.

현재 score 모듈의 로드 흐름은 아래와 같다.

```text
JSON text
-> loadScoreJson()
-> unknown candidate
-> validateScoreFile()
-> ScoreFile
-> buildScoreIndexes()
-> RuntimeDocument
```

## 2. score_validate.ts의 현재 책임

`score_validate.ts`는 JSON 파싱이 끝난 값을 `ScoreFile`로 사용할 수 있는지 최소한으로 검증한다.
이 단계의 목표는 모든 세부 규칙을 확정하는 것이 아니라, 후속 score/parse/analyze 모듈이 기본 구조를 신뢰할 수 있게 만드는 것이다.

현재 검사 범위는 다음과 같다.

- 최상위 값이 object인지 확인
- 필수 최상위 필드 존재 여부 확인
  - `format`
  - `musicData`
  - `instData`
  - `layout`
  - `globalLines`
  - `tracks`
- 주요 배열 필드의 형태 확인
  - `instData.strings`
  - `layout.rowDefinitions`
  - `globalLines.cells`
  - `tracks`
- `stringId` 중복 확인
- `rowId` 중복 확인
- `trackId` 중복 및 허용값 확인
- global cell의 `rowId`가 global row를 참조하는지 확인
- track cell의 `rowId`가 존재하는지 확인
- track cell이 `gap` row를 참조하지 않는지 확인
- global/track cell 좌표 중복 확인
- `col`이 `0 <= col < columnCount` 범위 안에 있는지 확인
- `bpm`, `beatsPerBar`, `stepsPerBeat`, `dynamics` 전역 시작 셀이 `col = 0`에 존재하는지 확인

반대로 현재 단계에서 검사하지 않는 항목은 다음과 같다.

- note cell의 `rawText` 문법
- global cell의 숫자/ramp 문법
- 날짜 문자열의 엄밀한 ISO 형식
- 모든 숫자 필드의 세부 범위
- layout과 instData 음역대의 세부 충돌
- import 복구 또는 reconciliation

이 항목들은 parser 또는 후속 validator 확장 단계에서 다루는 편이 적절하다.

## 2-1. score_validate.ts 함수 해설 위치

함수별 상세 설명은 `score_validate.ts` 내부의 `/** ... */` 블록 주석으로 옮긴다.
설계 메모에는 파일 전체 흐름과 구현 판단, 사용된 TypeScript 문법만 남겨 코드와 문서의 중복을 줄인다.

현재 `score_validate.ts`의 함수 구성은 다음과 같다.

- `validateScoreFile()`
  - 외부에서 호출하는 공개 진입점
- `validateRequiredTopLevelFields()`, `validateBasicShapes()`
  - 구조 검증의 첫 단계
- `validateUniqueStringIds()`, `validateRows()`, `validateTrackIds()`
  - ID 중복과 지원 범위 검증
- `validateGlobalCells()`, `validateTrackCells()`, `validateGlobalStartCells()`
  - 셀 참조와 좌표 무결성 검증
- `validateCellCol()`, `unknownRowId()`, `invalidShape()`, `isRecord()`
  - 반복되는 검증과 오류 생성을 위한 helper

구현 의도가 필요한 세부 분기는 코드 내부의 `//` 주석으로 둔다.
예를 들어 `ScoreFile` 타입 단언 시점, global row 참조 제한, gap row 입력 금지, `isRecord()`의 null/array 제외 이유는 코드 옆에서 바로 확인하는 편이 이해하기 쉽다.

## 2-2. 이번 구현에서 사용한 주요 TypeScript 문법

### 2-2-1. unknown 입력

`validateScoreFile()`의 입력 타입은 `ScoreFile`이 아니라 `unknown`이다.
JSON 파싱 결과는 런타임 입력이며, 아직 `ScoreFile` 구조라고 신뢰할 수 없기 때문이다.

```ts
export function validateScoreFile(value: unknown): ScoreValidationResult
```

이후 `isRecord()`, 필수 필드 검사, 기본 shape 검사를 차례로 통과한 뒤에야 내부 검증용으로 `ScoreFile` 타입 단언을 수행한다.

### 2-2-2. 타입 가드와 is 반환 타입

`isRecord()`의 반환 타입은 `boolean` 대신 타입 가드 형태를 사용한다.

```ts
function isRecord(value: unknown): value is Record<string, unknown>
```

런타임에서는 여전히 `true` 또는 `false`를 반환한다.
다만 TypeScript는 이 함수가 `true`를 반환한 분기 안에서 `value`를 `Record<string, unknown>`으로 좁혀서 해석한다.
이 덕분에 `unknown` 값에 바로 필드 접근을 하지 않고, 일반 객체인지 확인한 뒤 다음 검증으로 넘어갈 수 있다.

### 2-2-3. Record<string, unknown>

`Record<string, unknown>`은 문자열 key로 접근 가능한 객체를 뜻한다.

```ts
type AnyRecord = Record<string, unknown>;
```

이는 대략 아래 구조와 비슷하게 이해할 수 있다.

```ts
type AnyRecord = {
  [key: string]: unknown;
};
```

각 필드 값은 여전히 `unknown`이므로, `score.format`처럼 접근한 뒤에도 그 값이 object인지 array인지 다시 확인해야 한다.

### 2-2-4. Partial<ScoreFile>

`Partial<ScoreFile>`은 `ScoreFile`의 모든 필드를 optional로 바꾼 타입이다.

```ts
const score = value as Partial<ScoreFile>;
```

최상위 값이 object라는 점만 확인한 시점에는 필수 필드가 모두 있는지 아직 모른다.
따라서 바로 `ScoreFile`로 다루지 않고, 필드가 빠져 있을 수 있는 후보 객체라는 의미로 `Partial<ScoreFile>`을 사용한다.

### 2-2-5. Set

중복 검사는 `Set`으로 처리한다.

```ts
const trackIds = new Set<TrackId>();
const coords = new Set<string>();
```

`Set`은 이미 본 값을 저장하고 `has()`로 중복 여부를 빠르게 확인할 수 있다.
`stringId`, `trackId`, cell 좌표 key 중복 검사가 이 방식이다.

중요한 점은 `has()`가 원본 배열 전체를 검사하는 것이 아니라, 현재까지 순회하며 `Set`에 넣어 둔 값만 검사한다는 것이다.
따라서 아래 패턴에서 `has()`가 `true`를 반환한다는 것은 "현재 값이 이번 순회 이전에 이미 등장했다"는 뜻이다.

```ts
const stringIds = new Set<string>();

for (const stringInfo of score.instData.strings) {
  if (stringIds.has(stringInfo.stringId)) {
    return duplicateStringIdError;
  }

  stringIds.add(stringInfo.stringId);
}
```

예를 들어 `["s1", "s2", "s1"]`을 검사하면 다음 순서로 동작한다.

```text
첫 번째 s1:
  Set = {}
  has("s1") -> false
  add("s1")

두 번째 s2:
  Set = {"s1"}
  has("s2") -> false
  add("s2")

세 번째 s1:
  Set = {"s1", "s2"}
  has("s1") -> true
  이전에 나온 값이 다시 나온 것이므로 중복
```

즉 중복 검사에서 `Set`은 "지금까지 본 값 목록"이고, `has()`는 "이번 값이 그 목록에 이미 있는가"를 묻는 역할을 한다.

### 2-2-6. Map

`validateRows()`는 rowId 중복을 확인하면서 `Map<RowId, RowDefinition>`도 함께 만든다.
이 Map은 이후 global cell과 track cell이 참조하는 `rowId`가 실제로 존재하는지 확인하는 데 사용된다.

배열을 매번 순회해서 찾을 수도 있지만, 검증 과정에서는 같은 rowId 조회가 반복되므로 Map으로 변환해두는 편이 명확하다.

### 2-2-7. Discriminated union 좁히기

`RowDefinition`은 `row.type` 값에 따라 global, note, gap row로 나뉜다.

```ts
if (row.type !== "global") {
  // global cell이 붙을 수 없는 행
}
```

TypeScript는 `row.type` 조건문을 보고 해당 분기에서 가능한 row 타입을 좁혀서 해석한다.
`validateGlobalStartCells()`에서 `row?.type === "global"` 조건을 통과하면 `row.kind`에 접근할 수 있는 이유도 이 때문이다.

### 2-2-8. 조기 반환

현재 validator는 오류를 모두 모아서 반환하지 않고, 처음 발견한 오류를 즉시 반환한다.

```ts
if (trackIdError) {
  return { ok: false, error: trackIdError };
}
```

1단계 구현의 목표는 전체 오류 리포트 생성보다 후속 모듈로 잘못된 구조가 넘어가지 않게 막는 것이다.
따라서 조기 반환 방식이 구현이 단순하고 실패 지점도 명확하다.

## 3. 테스트 환경 조성 필요성

현재 프로젝트는 TypeScript 파일을 작성하고 있지만, 브라우저 UI와 번들러 환경이 아직 준비되지 않았다.
따라서 지금 단계의 검증 목표는 화면 테스트가 아니라 다음 두 가지이다.

1. TypeScript 타입 구조가 깨지지 않았는지 확인한다.
2. 로컬 JSON fixture를 읽어 `loadScoreFile()`까지 정상 실행되는지 확인한다.

이를 위해 Jest 같은 테스트 프레임워크를 바로 도입하지 않고, 최소 개발 스크립트 기반으로 시작하였다.

현재 필요한 도구는 다음과 같다.

- `typescript`
  - `tsc --noEmit`으로 타입 검사를 수행한다.
- `tsx`
  - TypeScript dev script를 별도 빌드 없이 실행한다.
- `@types/node`
  - `fs`, `process`, `import.meta.url` 등 Node.js API의 타입을 제공한다.

작업 위치는 `regression-code/`이다.
우선 `package.json`에 개발 의존성과 실행 스크립트를 작성한 뒤, 다음 명령으로 패키지를 설치하였다.

```powershell
cd regression-code
npm install
```

현재 `package.json`의 개발 의존성은 다음과 같다.

```json
{
  "devDependencies": {
    "@types/node": "^20.12.0",
    "tsx": "^4.19.0",
    "typescript": "^5.4.0"
  }
}
```

`npm install` 실행 결과 `node_modules/`와 `package-lock.json`이 생성되었다.
`node_modules/`는 로컬 설치 패키지 실체이며, `package-lock.json`은 설치된 패키지 버전을 고정하기 위한 파일이다.

설치 후 `package.json`에는 다음 스크립트를 추가하였다.

```json
{
  "scripts": {
    "typecheck": "tsc --noEmit",
    "test:score": "tsx ./dev/test_score.ts"
  }
}
```

각 명령의 역할은 다음과 같다.

- `npm run typecheck`
  - `tsc --noEmit`을 실행한다.
  - TypeScript 타입 검사를 수행하되 JavaScript 출력 파일은 만들지 않는다.
- `npm run test:score`
  - `tsx ./dev/test_score.ts`를 실행한다.
  - 로컬 JSON fixture를 읽고 `loadScoreFile()` 경로를 실제로 호출한다.

## 4. tsconfig.json

TypeScript 검사를 수행하기 위해 `regression-code/tsconfig.json`을 추가하였다.

```json
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "ESNext",
    "moduleResolution": "Bundler",
    "strict": true,
    "noEmit": true,
    "skipLibCheck": true,
    "types": ["node"]
  },
  "include": ["src/**/*.ts", "dev/**/*.ts"]
}
```

현재 설정의 의미는 다음과 같다.

- `target: "ES2022"`
  - 비교적 최신 JavaScript 문법을 기준으로 타입 검사한다.
- `module: "ESNext"`
  - `import` / `export` 기반 ESM 코드를 사용한다.
- `moduleResolution: "Bundler"`
  - 브라우저 번들러나 `tsx` 기반 개발 실행에 맞춘 모듈 해석 방식을 사용한다.
- `strict: true`
  - TypeScript의 엄격 검사를 켠다.
- `noEmit: true`
  - 타입 검사만 수행하고 JavaScript 파일을 생성하지 않는다.
- `skipLibCheck: true`
  - 외부 라이브러리 타입 정의 파일 검사를 생략하여 초기 개발 속도를 높인다.
- `types : ["node"]`
  - Node.js 전용 타입을 사용한다고 명시한다.

## 5. test_score.ts

`dev/test_score.ts`는 현재 Jest 테스트가 아니라 score 로드 경로를 확인하는 간단한 실행 스크립트이다.

```ts
import { readFileSync } from "node:fs";

import { loadScoreFile } from "../src/core/score/json_load";

const fixtureUrl = new URL("./test_cases/minimal-valid-score.json", import.meta.url);
const jsonText = readFileSync(fixtureUrl, "utf8");
const result = loadScoreFile(jsonText);
```

이 스크립트는 다음을 확인한다.

- `minimal-valid-score.json` 파일을 읽을 수 있는가
- `json_load.ts`의 `loadScoreFile()`이 JSON 파싱에 성공하는가
- `score_validate.ts`의 최소 검증을 통과하는가
- 통과한 경우 제목, 행 개수, 트랙 개수, 전역 셀 개수를 출력할 수 있는가

현재 성공 출력은 다음과 같다.

```text
Score load succeeded.
title: testScore
rows: 72
tracks: 1
global cells: 4
```

## 6. 실행 명령과 결과

타입 검사는 다음 명령으로 수행한다.

```powershell
cd regression-code
npm run typecheck
```

현재 결과는 통과이다.

score fixture 실행은 다음 명령으로 수행한다.

```powershell
cd regression-code
npm run test:score
```

현재 결과는 통과이다.

다만 현재 환경에서는 `tsx`가 내부적으로 사용하는 `esbuild` helper process 실행이 샌드박스에서 `EPERM`으로 막힐 수 있었다.
따라서 같은 환경에서는 `npm run test:score` 실행에 권한 상승이 필요할 수 있다.

## 7. 현재 단계의 판단

지금 단계에서 Jest 같은 정식 테스트 프레임워크를 바로 도입하지 않은 이유는 다음과 같다.

- 아직 core 함수 수가 적다.
- UI와 bundler 구성이 확정되지 않았다.
- 우선 필요한 것은 자동화된 단위 테스트보다 타입 검사와 fixture 기반 실행 확인이다.
- `build_score_indexes.ts`와 parser 구현이 진행되면 테스트 케이스 구조가 더 명확해진다.

따라서 현재는 아래 수준을 1차 테스트 환경으로 둔다.

```text
npm run typecheck
npm run test:score
```

이후 `build_score_indexes.ts`, `create_runtime_document.ts`, parser 구현이 진행되면 Jest 또는 Vitest 도입을 다시 검토한다.

## 8. 다음 작업

다음 구현 대상은 `build_score_indexes.ts`이다.

`score_validate.ts`를 통과한 `ScoreFile`을 입력으로 받아 다음 런타임 인덱스를 생성해야 한다.

- `rowById`
- `rowsInDisplayOrder`
- `noteRowIdsByStringId`
- `noteRowIdByStringMidi`
- `trackById`
- `cellMapByTrackId`
- `cellsByTrackAndCol`
- `globalCellMapByCoord`
- `globalCellsByKindAndCol`
- `globalCellsInColOrder`

그 다음 `create_runtime_document.ts`에서 `ScoreFile`과 `ScoreIndexes`를 묶어 `RuntimeDocument`를 생성한다.
