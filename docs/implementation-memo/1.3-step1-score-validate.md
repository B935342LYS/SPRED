# 1.3. 1단계 Score 검증 및 테스트 환경 메모
> 본 설계 메모 문서 1.3에서는 1단계 구현 중 `score_validate.ts`와 최소 테스트 환경 조성 과정을 기록한다.
> 본 문서는 active specification을 대체하지 않는다.
> 구현 중 생긴 판단 근거와 개발 환경 설정 이유만 보충한다.

## 1. 개요

`json_load.ts`를 구현하는 과정에서 JSON 파싱과 ScoreFile 구조 검증을 어디까지 분리할지 다시 검토하였다.
초기에는 `json_load.ts`에서 `parsed as ScoreFile`로 타입 단언을 하고 후속 구현을 진행하는 방식을 고려하였다.
그러나 이 방식은 후속 모듈인 `build_score_indexes.ts`와 parser 구현 단계에서 다음 문제가 생길 수 있다.

- 실제로는 `ScoreFile` 구조가 아닌 값이 후속 모듈로 전달될 수 있다.
- `rowId`, `trackId`, `stringId`의 존재 여부를 후속 모듈에서 반복 방어해야 한다.
- 나중에 validator를 추가할 때 `json_load.ts`의 반환 타입과 호출부를 다시 수정해야 한다.
- 테스트 fixture가 valid한지 확인하지 못한 채 인덱스 생성 로직을 구현하게 된다.

따라서 1단계에서 완성형 validator를 만들지는 않더라도, 최소 구조 검증을 담당하는 `score_validate.ts`를 먼저 추가하기로 하였다.

현재 score 모듈의 로드 흐름은 아래와 같다.

```text
JSON text
-> loadScoreJson()
-> unknown candidate
-> validateScoreFile()
-> ScoreFile
-> buildScoreIndexes()
-> RuntimeDocument
```

## 2. score_validate.ts의 현재 책임

`score_validate.ts`는 JSON 파싱이 끝난 값을 `ScoreFile`로 사용할 수 있는지 최소한으로 검증한다.
이 단계의 목표는 모든 세부 규칙을 확정하는 것이 아니라, 후속 score/parse/analyze 모듈이 기본 구조를 신뢰할 수 있게 만드는 것이다.

현재 검사 범위는 다음과 같다.

- 최상위 값이 object인지 확인
- 필수 최상위 필드 존재 여부 확인
  - `format`
  - `musicData`
  - `instData`
  - `layout`
  - `globalLines`
  - `tracks`
- 주요 배열 필드의 형태 확인
  - `instData.strings`
  - `layout.rowDefinitions`
  - `globalLines.cells`
  - `tracks`
- `stringId` 중복 확인
- `rowId` 중복 확인
- `trackId` 중복 및 허용값 확인
- global cell의 `rowId`가 global row를 참조하는지 확인
- track cell의 `rowId`가 존재하는지 확인
- track cell이 `gap` row를 참조하지 않는지 확인
- global/track cell 좌표 중복 확인
- `col`이 `0 <= col < columnCount` 범위 안에 있는지 확인
- `bpm`, `beatsPerBar`, `stepsPerBeat`, `dynamics` 전역 시작 셀이 `col = 0`에 존재하는지 확인

반대로 현재 단계에서 검사하지 않는 항목은 다음과 같다.

- note cell의 `rawText` 문법
- global cell의 숫자/ramp 문법
- 날짜 문자열의 엄밀한 ISO 형식
- 모든 숫자 필드의 세부 범위
- layout과 instData 음역대의 세부 충돌
- import 복구 또는 reconciliation

이 항목들은 parser 또는 후속 validator 확장 단계에서 다루는 편이 적절하다.

## 2-1. score_validate.ts 함수별 해설

이 절은 `score_validate.ts`를 읽고 유지보수하기 위한 함수 단위 해설이다.
현재 파일은 "JSON 파싱은 끝났지만 아직 신뢰할 수 없는 값"을 받아서, 후속 모듈이 최소한의 `ScoreFile` 구조와 참조 관계를 신뢰할 수 있게 만드는 데 초점이 있다.

### validateScoreFile()

`score_validate.ts`의 공개 진입점이다.
외부 모듈은 이 함수만 호출하고, 내부의 세부 검증 함수들은 파일 내부에서만 사용한다.

입력 타입을 `ScoreFile`이 아니라 `unknown`으로 둔 이유는 JSON 파싱 결과를 아직 신뢰하지 않기 때문이다.
`json_load.ts`는 JSON 문법과 최상위 object 여부만 확인하고, 구체적인 ScoreFile 구조 검증은 이 함수에 위임한다.

처리 순서는 다음과 같다.

1. 최상위 값이 일반 object인지 확인한다.
2. 필수 최상위 필드가 존재하는지 확인한다.
3. 후속 검증에서 접근할 주요 필드가 object 또는 array인지 확인한다.
4. `ScoreFile`로 타입 단언한 뒤, ID 중복과 참조 무결성을 확인한다.
5. 모든 검증을 통과하면 `{ ok: true, score }`를 반환한다.

중간에 하나라도 실패하면 즉시 `{ ok: false, error }`를 반환한다.
이 방식은 한 번에 모든 오류를 수집하지는 않지만, 1단계 구현에서는 실패 지점을 명확히 확인하고 후속 모듈을 보호하는 데 충분하다.

### validateRequiredTopLevelFields()

`ScoreFile`에 반드시 있어야 하는 최상위 필드가 존재하는지 확인한다.
현재 필수 필드는 다음 여섯 개이다.

- `format`
- `musicData`
- `instData`
- `layout`
- `globalLines`
- `tracks`

이 함수는 필드의 세부 타입까지 보지는 않는다.
예를 들어 `layout`이라는 필드가 있으면 이 단계는 통과하고, `layout`이 실제 object인지와 `layout.rowDefinitions`가 array인지는 `validateBasicShapes()`에서 확인한다.

역할을 이렇게 나눈 이유는 오류 원인을 단순하게 분리하기 위해서이다.
필드 자체가 없으면 `missing_required_field`, 필드는 있지만 형태가 맞지 않으면 `invalid_shape`로 보고한다.

### validateBasicShapes()

후속 검증 함수들이 안전하게 접근해야 하는 기본 형태를 확인한다.
여기서 말하는 "기본 형태"는 상세 스키마 전체가 아니라, 다음 단계에서 바로 접근하는 object와 array 여부이다.

현재 확인하는 대표 항목은 다음과 같다.

- `format`, `musicData`, `instData`, `layout`, `globalLines`가 일반 object인지
- `instData.strings`가 array인지
- `layout.rowDefinitions`가 array인지
- `globalLines.columnCount`가 integer인지
- `globalLines.cells`가 array인지
- `tracks`가 array인지

이 검사를 먼저 수행해야 이후 함수들이 `score.instData.strings`, `score.layout.rowDefinitions`, `score.globalLines.cells` 등에 접근할 때 런타임 예외가 나지 않는다.
다만 배열 내부 원소의 모든 필드 타입까지는 아직 엄밀히 검사하지 않는다.
현재 구현은 "후속 인덱스 생성과 parser 진입 전에 필요한 최소 방어선"에 가깝다.

### validateUniqueStringIds()

`instData.strings` 안의 `stringId`가 중복되지 않는지 확인한다.

`stringId`는 이후 `noteRowIdsByStringId`, `noteRowIdByStringMidi` 같은 인덱스의 기준 키가 된다.
중복된 `stringId`를 허용하면 한 현에 속해야 할 행과 셀을 안정적으로 분류할 수 없으므로, 인덱스 생성 전에 실패시키는 편이 안전하다.

내부적으로는 `Set<string>`을 사용한다.
이미 본 `stringId`가 다시 나오면 `duplicate_string_id` 오류를 반환하고, 아니면 set에 추가한다.

### validateRows()

`layout.rowDefinitions`를 순회하면서 `rowId` 중복 여부를 확인하고, 동시에 `rowById` map을 만든다.

반환 타입이 단순한 `ScoreValidationError | null`이 아닌 이유는 성공 시 후속 검증에서 사용할 `rowById`가 필요하기 때문이다.
`rowById`는 global cell과 track cell이 참조하는 `rowId`가 실제로 존재하는지, 그리고 그 행의 종류가 무엇인지 빠르게 확인하는 데 쓰인다.

이 함수가 성공하면 다음 형태를 반환한다.

```ts
{
  ok: true,
  rowById: Map<RowId, RowDefinition>
}
```

중복 행 ID가 발견되면 `duplicate_row_id` 오류를 반환한다.
`rowId`는 셀 좌표와 행 참조의 기준이므로, 중복을 허용하면 같은 `rowId`가 서로 다른 행 정의를 가리키는 문제가 생긴다.

### validateTrackIds()

`tracks` 배열의 `trackId`가 허용된 값인지, 그리고 중복되지 않는지 확인한다.

현재 허용되는 트랙 ID는 코드 상단의 `TRACK_IDS` 상수에 정의되어 있다.

```ts
["basic", "optional", "extra"]
```

지원하지 않는 값이 들어오면 `invalid_shape`로 실패한다.
이미 등장한 트랙 ID가 다시 나오면 `duplicate_track_id`로 실패한다.

이 검사가 필요한 이유는 `ScoreIndexes.trackById`, `cellMapByTrackId`, `cellsByTrackAndCol` 같은 런타임 인덱스가 `TrackId`를 기준으로 만들어지기 때문이다.
트랙 ID가 중복되면 어떤 트랙의 셀인지 명확하지 않고, 허용되지 않은 값이 들어오면 이후 UI와 parser/analyzer가 처리할 수 있는 범위를 벗어난다.

### validateGlobalCells()

`globalLines.cells`에 들어 있는 전역 셀들의 참조와 좌표를 검증한다.

이 함수가 확인하는 내용은 다음과 같다.

- 셀의 `rowId`가 `layout.rowDefinitions`에 존재하는가
- 해당 `rowId`가 global row를 가리키는가
- `col`이 `0 <= col < columnCount` 범위 안에 있는가
- 같은 전역 좌표 `rowId|col`이 중복되지 않는가

global cell은 `bpm`, `beatsPerBar`, `stepsPerBeat`, `dynamics` 같은 전역 시간/연주 정보를 담는다.
따라서 note row나 gap row에 전역 셀이 붙으면 의미가 깨진다.
이 경우 `invalid_global_row_type`으로 실패시킨다.

좌표 중복 검사는 `Set<string>`에 `${cell.rowId}|${cell.col}` 형태의 키를 저장하는 방식으로 처리한다.
같은 전역 행과 같은 열에 셀이 두 개 있으면 어떤 값을 적용해야 할지 결정할 수 없기 때문에 `duplicate_global_cell_coord`로 실패한다.

### validateTrackCells()

각 트랙의 note cell들이 참조하는 행과 좌표를 검증한다.

이 함수가 확인하는 내용은 다음과 같다.

- 셀의 `rowId`가 존재하는가
- 셀이 gap row를 참조하지 않는가
- `col`이 `0 <= col < columnCount` 범위 안에 있는가
- 같은 트랙 안에서 같은 좌표 `rowId|col`이 중복되지 않는가

현재 구현은 track cell이 global row를 참조하는 경우까지 막지는 않는다.
코드상으로는 `row.type === "gap"`만 실패 조건으로 둔다.
이는 최소 validator 단계의 구현 상태이며, 더 엄밀한 정책이 필요하면 후속 validator 확장에서 `row.type !== "note"` 조건으로 강화할 수 있다.
다만 현재 문서의 책임 범위에서는 "gap row는 입력 셀이 붙을 수 없는 시각적 간격 행"이라는 점을 우선 막는 데 초점을 둔다.

좌표 중복은 트랙별로 별도의 `Set<string>`을 만들어 검사한다.
즉, basic 트랙과 optional 트랙이 같은 `rowId|col` 좌표를 갖는 것은 허용되지만, 같은 트랙 내부에서 같은 좌표가 중복되는 것은 허용하지 않는다.

### validateGlobalStartCells()

필수 전역 행 종류가 `col = 0`에 시작 셀을 가지고 있는지 확인한다.

현재 필수 전역 종류는 코드 상단의 `GLOBAL_KINDS` 상수에 정의되어 있다.

```ts
["bpm", "beatsPerBar", "stepsPerBeat", "dynamics"]
```

이 검사는 첫 열에서 기본 전역 상태를 알 수 있게 하기 위한 최소 조건이다.
예를 들어 `bpm` 시작 셀이 없으면 재생 시작 시점의 템포를 결정할 수 없다.
`beatsPerBar`, `stepsPerBeat`, `dynamics`도 같은 이유로 첫 열의 기준값이 필요하다.

구현은 `globalLines.cells` 중 `col === 0`인 셀만 보고, 그 셀이 참조하는 row가 global row이면 해당 `kind`를 `startKinds`에 추가한다.
마지막으로 `GLOBAL_KINDS`를 순회하면서 빠진 종류가 있으면 `missing_global_start_cell`을 반환한다.

### validateCellCol()

global cell과 track cell이 공통으로 사용하는 열 좌표 검증 함수이다.

검사 조건은 다음과 같다.

```text
Number.isInteger(cell.col)
cell.col >= 0
cell.col < columnCount
```

이 조건을 만족하지 않으면 `col_out_of_range` 오류를 반환한다.
`path`를 인수로 받는 이유는 같은 열 좌표 오류라도 전역 셀에서 발생했는지, 특정 트랙 셀에서 발생했는지를 호출자가 구분해서 전달하기 위해서이다.

### unknownRowId()

존재하지 않는 `rowId` 참조 오류를 만드는 작은 helper 함수이다.

global cell과 track cell 모두 `rowId` 참조 검사를 하므로, 오류 객체 생성 코드를 중복하지 않기 위해 분리되어 있다.
항상 `unknown_row_id` 코드를 가진 `ScoreValidationError`를 반환한다.

### invalidShape()

필드 형태가 맞지 않을 때 사용하는 `ScoreValidationError` 생성 helper이다.

`validateScoreFile()`의 최상위 object 검사와 `validateBasicShapes()`의 object/array/integer 검사에서 주로 사용한다.
항상 `invalid_shape` 코드를 가진 오류 객체를 반환하며, 선택적으로 `path`를 포함할 수 있다.

이 helper를 사용하면 `code: "invalid_shape"`를 반복해서 쓰지 않아도 되고, 오류 객체 형식도 한 곳에서 유지할 수 있다.

### isRecord()

입력값이 일반 object인지 확인하는 타입 가드이다.

검사 조건은 다음과 같다.

```ts
typeof value === "object" && value !== null && !Array.isArray(value)
```

JavaScript에서 `typeof null`은 `"object"`이므로 `value !== null` 검사가 필요하다.
배열도 object의 일종이지만 ScoreFile의 최상위나 주요 nested object로는 맞지 않으므로 `Array.isArray(value)`로 제외한다.

반환 타입이 `value is Record<string, unknown>`인 이유는 TypeScript가 이 함수 통과 후 해당 값을 일반 객체처럼 좁혀서 다룰 수 있게 하기 위해서이다.
이 덕분에 `score.format`, `score.layout` 같은 필드 접근 전에 최소한 "객체 형태"라는 조건을 코드로 표현할 수 있다.

## 3. 테스트 환경 조성 필요성

현재 프로젝트는 TypeScript 파일을 작성하고 있지만, 브라우저 UI와 번들러 환경이 아직 준비되지 않았다.
따라서 지금 단계의 검증 목표는 화면 테스트가 아니라 다음 두 가지이다.

1. TypeScript 타입 구조가 깨지지 않았는지 확인한다.
2. 로컬 JSON fixture를 읽어 `loadScoreFile()`까지 정상 실행되는지 확인한다.

이를 위해 Jest 같은 테스트 프레임워크를 바로 도입하지 않고, 최소 개발 스크립트 기반으로 시작하였다.

현재 필요한 도구는 다음과 같다.

- `typescript`
  - `tsc --noEmit`으로 타입 검사를 수행한다.
- `tsx`
  - TypeScript dev script를 별도 빌드 없이 실행한다.
- `@types/node`
  - `fs`, `process`, `import.meta.url` 등 Node.js API의 타입을 제공한다.

작업 위치는 `regression-code/`이다.
우선 `package.json`에 개발 의존성과 실행 스크립트를 작성한 뒤, 다음 명령으로 패키지를 설치하였다.

```powershell
cd regression-code
npm install
```

현재 `package.json`의 개발 의존성은 다음과 같다.

```json
{
  "devDependencies": {
    "@types/node": "^20.12.0",
    "tsx": "^4.19.0",
    "typescript": "^5.4.0"
  }
}
```

`npm install` 실행 결과 `node_modules/`와 `package-lock.json`이 생성되었다.
`node_modules/`는 로컬 설치 패키지 실체이며, `package-lock.json`은 설치된 패키지 버전을 고정하기 위한 파일이다.

설치 후 `package.json`에는 다음 스크립트를 추가하였다.

```json
{
  "scripts": {
    "typecheck": "tsc --noEmit",
    "test:score": "tsx ./dev/test_score.ts"
  }
}
```

각 명령의 역할은 다음과 같다.

- `npm run typecheck`
  - `tsc --noEmit`을 실행한다.
  - TypeScript 타입 검사를 수행하되 JavaScript 출력 파일은 만들지 않는다.
- `npm run test:score`
  - `tsx ./dev/test_score.ts`를 실행한다.
  - 로컬 JSON fixture를 읽고 `loadScoreFile()` 경로를 실제로 호출한다.

## 4. tsconfig.json

TypeScript 검사를 수행하기 위해 `regression-code/tsconfig.json`을 추가하였다.

```json
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "ESNext",
    "moduleResolution": "Bundler",
    "strict": true,
    "noEmit": true,
    "skipLibCheck": true,
    "types": ["node"]
  },
  "include": ["src/**/*.ts", "dev/**/*.ts"]
}
```

현재 설정의 의미는 다음과 같다.

- `target: "ES2022"`
  - 비교적 최신 JavaScript 문법을 기준으로 타입 검사한다.
- `module: "ESNext"`
  - `import` / `export` 기반 ESM 코드를 사용한다.
- `moduleResolution: "Bundler"`
  - 브라우저 번들러나 `tsx` 기반 개발 실행에 맞춘 모듈 해석 방식을 사용한다.
- `strict: true`
  - TypeScript의 엄격 검사를 켠다.
- `noEmit: true`
  - 타입 검사만 수행하고 JavaScript 파일을 생성하지 않는다.
- `skipLibCheck: true`
  - 외부 라이브러리 타입 정의 파일 검사를 생략하여 초기 개발 속도를 높인다.
- `types : ["node"]`
  - Node.js 전용 타입을 사용한다고 명시한다.

## 5. test_score.ts

`dev/test_score.ts`는 현재 Jest 테스트가 아니라 score 로드 경로를 확인하는 간단한 실행 스크립트이다.

```ts
import { readFileSync } from "node:fs";

import { loadScoreFile } from "../src/core/score/json_load";

const fixtureUrl = new URL("./test_cases/minimal-valid-score.json", import.meta.url);
const jsonText = readFileSync(fixtureUrl, "utf8");
const result = loadScoreFile(jsonText);
```

이 스크립트는 다음을 확인한다.

- `minimal-valid-score.json` 파일을 읽을 수 있는가
- `json_load.ts`의 `loadScoreFile()`이 JSON 파싱에 성공하는가
- `score_validate.ts`의 최소 검증을 통과하는가
- 통과한 경우 제목, 행 개수, 트랙 개수, 전역 셀 개수를 출력할 수 있는가

현재 성공 출력은 다음과 같다.

```text
Score load succeeded.
title: testScore
rows: 72
tracks: 1
global cells: 4
```

## 6. 실행 명령과 결과

타입 검사는 다음 명령으로 수행한다.

```powershell
cd regression-code
npm run typecheck
```

현재 결과는 통과이다.

score fixture 실행은 다음 명령으로 수행한다.

```powershell
cd regression-code
npm run test:score
```

현재 결과는 통과이다.

다만 현재 환경에서는 `tsx`가 내부적으로 사용하는 `esbuild` helper process 실행이 샌드박스에서 `EPERM`으로 막힐 수 있었다.
따라서 같은 환경에서는 `npm run test:score` 실행에 권한 상승이 필요할 수 있다.

## 7. 현재 단계의 판단

지금 단계에서 Jest 같은 정식 테스트 프레임워크를 바로 도입하지 않은 이유는 다음과 같다.

- 아직 core 함수 수가 적다.
- UI와 bundler 구성이 확정되지 않았다.
- 우선 필요한 것은 자동화된 단위 테스트보다 타입 검사와 fixture 기반 실행 확인이다.
- `build_score_indexes.ts`와 parser 구현이 진행되면 테스트 케이스 구조가 더 명확해진다.

따라서 현재는 아래 수준을 1차 테스트 환경으로 둔다.

```text
npm run typecheck
npm run test:score
```

이후 `build_score_indexes.ts`, `create_runtime_document.ts`, parser 구현이 진행되면 Jest 또는 Vitest 도입을 다시 검토한다.
