> 전체 평균 계산에 miss의 0%는 분자로 가산되지 않나요?

현재 구현에서는 `Miss`의 0%가 평균에 반영되지 않습니다.

[game_judge.ts](./regression-code/src/app/game/game_judge.ts:235)에서 평균 분모가:

```ts
const hitSampleCount = perfectCount + okCount + badCount;
```

이라서 `missCount`가 빠져 있습니다. 그리고 `Miss`일 때는 분자에도 더하지 않습니다:

```ts
const nextAccuracySum = sample.label === "Miss"
  ? previousAccuracySum
  : previousAccuracySum + sample.pitchAccuracy;
```

즉 현재 `accuracyPercent`는 “맞춘/부분적으로 맞춘 샘플 중 평균 정확도”에 가깝고, 사용자가 낸 유효 pitch가 틀려서 `Miss`가 된 샘플은 평균을 낮추지 않습니다.

일반적인 “전체 평균 음정 정확도”라면 `Miss = 0%`가 분모에는 들어가야 하므로 `perfect + ok + bad + miss`를 분모로 쓰는 쪽이 맞습니다. 이건 문구와 직관상 수정하는 게 좋아 보입니다.

> 네. MISS를 틀린 샘플로 판정하여 평균을 낮춰야 하겠습니다.

> 문제가 해결됐는데 이유를 모르겠습니다.
> 설마 마이크가 새거라서 길드는데 시간이 걸린 걸까요?

마이크가 “길들어서” 좋아졌을 가능성은 거의 낮습니다. 전자 마이크는 악기 줄이나 스피커처럼 사용하면서 물성이 안정화되는 장치가 아니라서, 몇 분~몇 시간 뒤 pitch 인식이 갑자기 안정되는 원인으로 보기는 어렵습니다.

더 가능성 높은 원인은 이쪽입니다.

1. 우리가 제거한 보정 로직이 실제 문제였을 가능성
   특히 target 기반 octave fold, hysteresis, 배음 후보 역산이 “도와주는” 척하면서 특정 순간에 잘못된 후보로 붙었을 수 있습니다. 지금 생 pitch 경로가 안정적이면 이 가능성이 큽니다.

2. 마이크 자동 처리 상태 변화
   브라우저/OS/장치가 echo cancellation, noise suppression, auto gain control을 켜고 입력 레벨을 조정하면서 초반 몇 초~수십 초 동안 안정화될 수는 있습니다. 이건 마이크 길들이기라기보다 소프트웨어 입력 처리 적응입니다.

3. 입력 음량/거리/각도 변화
   테스트하면서 자연스럽게 마이크 위치, 연주 세기, 스피커 음량, 헤드셋 사용 여부가 바뀌면 pitchy 결과가 크게 달라질 수 있습니다.
