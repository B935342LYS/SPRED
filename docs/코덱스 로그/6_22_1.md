현재 이해한 내용은 다음과 같습니다.

**현재 작업 위치**
`회귀 코드`는 legacy `이세계 코드`를 직접 이식하는 프로젝트가 아니라, `TypeScript + HTML + CSS + GitHub Pages` 기반으로 새로 만드는 웹 앱입니다. 구현 대상은 `regression-code/`이고, legacy 폴더와 `1. 개발문서`는 필요할 때만 참고하는 자료입니다.

현재 구현은 parser/analyzer/renderer/edit/playback/layout editor까지 1차 연결이 된 상태입니다. 특히 최근에는 layout editor와 layout preset 저장/불러오기, layout apply 시 구조 공유 방식, 삭제 확인, playback reset 등이 정리되었고, 그 다음 단계로 **트랙 레이어 기능 구현 전 명세 작성과 효율성 검토**가 진행 중입니다.

**현재 트랙 레이어 명세의 핵심**
`docs/2.6-track-layer-ui-spec.md` 기준으로, 첫 구현은 저장 포맷을 바꾸지 않고 기존 고정 트랙 `basic`, `optional`, `extra`만 다룹니다.

핵심 정책은 다음과 같습니다.

- `activeTrackIds`는 복수 선택 상태이며 초기값은 `["basic"]`
- active track은 편집 대상, 정상 표시 대상, 재생 대상
- inactive track은 편집/재생 대상은 아니지만 renderer에서 `0.10` alpha로 반투명 표시
- 마지막 active track은 끌 수 없음
- note row 편집은 모든 active track의 같은 `rowId / col`에 batch 적용
- global row는 track 소속이 아니므로 기존처럼 1회만 적용
- draw order는 `extra -> optional -> basic`, 결과적으로 `basic`이 가장 위에 보임
- audio schedule은 active track만 포함하고 inactive track은 아예 schedule에 넣지 않음
- 첫 구현에서는 track별 gain은 모두 `1.0`
- 재생 중에는 active track toggle을 비활성화하는 쪽이 1차 정책

효율성 판단도 비교적 명확합니다. inactive track을 schedule에 넣고 volume만 0으로 두는 방식은 현재 Web Audio scheduler/backend 구조에서 불필요한 예약 비용을 만들 수 있으므로, 1차 구현은 **active track filtering 후 schedule 재생성**으로 가는 것이 맞습니다. 즉 지금 단계에서는 track별 gain bus나 dynamic mute automation은 보류하는 판단입니다.

**앞으로 예정된 가까운 작업**
가장 가까운 작업 순서는 다음으로 이해했습니다.

1. 트랙 레이어 구현
   `src/track/track_control.ts` 같은 공용 정책 모듈을 만들고, app state/UI/edit/renderer/audio/playback 경로에 `activeTrackIds`를 연결합니다.

2. 트랙 레이어 검증
   basic만 표시/재생되는 초기 상태, multi-track batch edit/delete, inactive opacity, audio filtering, draw order, playing 상태 toggle 차단 등을 확인합니다.

3. 기본 오디오 검증
   vibrato, tremolo, gliss fallback, seek, pause/resume, layout 변경 후 playback reset 등을 청감/동기화 기준으로 점검합니다.

4. YouTube 반주 싱크 준비
   local/general audio 기준선이 안정된 뒤 YouTube iframe API와 playback controller 동기화를 검토합니다.

5. 성능 확장 검토
   긴 악보나 반복 편집에서 병목이 확인되면 partial parse/analyze/render, viewport virtualization, audio schedule 부분 재생성 등을 단계적으로 검토합니다.

문서 우선순위는 `HARNESS.md`가 현재 상태를 가장 잘 반영하고 있고, 트랙 레이어는 `2.6-track-layer-ui-spec.md`가 직접 기준 문서입니다. `1.0-development-spec.md`에는 일부 이전 단계 표현이 남아 있지만, 현재 작업 판단은 HARNESS의 최신 진행 요약과 2.6 명세를 우선하는 것이 맞습니다.

> 마지막 active 트랙을 끌 수 없음 제약은 없애는 편이 더 단순할까요?
> 또한 더 효율적으로 개선 가능한 부분 또는 모호한 부분이 남아있습니까?

네. **“마지막 active 트랙을 끌 수 없음” 제약은 없애는 편이 구현은 더 단순할 가능성이 큽니다.** 다만 UX 의미를 어떻게 둘지 하나만 정하면 됩니다.

추천은 다음입니다.

- `activeTrackIds = []` 허용
- 의미: “모든 track 비활성”
- renderer: 모든 track을 inactive alpha로 표시하거나, 아예 track note layer를 숨김
- playback: schedule event 0개, 재생 버튼은 비활성화하거나 눌러도 무음 재생
- edit: note row 편집은 적용 대상이 없으므로 차단하고 상태 메시지 표시
- global row 편집은 track과 무관하므로 계속 허용

이렇게 하면 “마지막 하나는 못 끈다” 같은 예외 UI가 사라지고, toggle 로직도 단순해집니다. 대신 빈 active 상태에서 편집/재생이 어떻게 동작하는지만 명확히 해야 합니다. 개인적으로는 **빈 active 허용 + note edit/playback만 no-op 또는 차단**이 가장 예측 가능합니다.

남아 있는 모호한 부분은 몇 가지 있습니다.

1. **active track 0개일 때 renderer 정책**
   - 모든 track을 `0.10` alpha로 보여줄지
   - 아무 note도 표시하지 않을지
   현재 명세의 “inactive track은 반투명 표시” 원칙을 유지하면 전자가 자연스럽습니다.

2. **active track 0개일 때 edit 정책**
   - note row 편집을 조용히 무시할지
   - “활성 트랙이 없습니다” 상태 메시지를 띄울지
   실사용 기준으로는 메시지를 띄우는 편이 낫습니다.

3. **paused 상태에서 track toggle 허용 여부**
   - 현재 명세는 구현 비용 확인 후 결정으로 남아 있습니다.
   - 단순성을 우선하면 `playing`뿐 아니라 `paused`도 toggle 비활성화하고, `stopped`에서만 허용하는 편이 안전합니다.
   - 더 나은 UX는 paused에서 toggle 허용 후 같은 score time으로 schedule rebuild지만, 구현 범위가 약간 넓어집니다.

4. **track toggle 후 playback runtime reset 범위**
   - active track 변경 시 audio schedule은 반드시 재생성해야 합니다.
   - renderer는 단순 opacity 변경만 필요하므로 full parse/analyze는 불필요합니다.
   - 가능하면 `activeTrackIds` 변경은 `analyze` 재실행 없이 `canvas item 재빌드 또는 draw 옵션 변경 + schedule rebuild`로 제한하는 것이 효율적입니다.

5. **renderer opacity 적용 위치**
   - 명세는 item builder 또는 draw 직전 단계를 권장합니다.
   - 더 효율적인 방향은 `AnalysisResult`는 그대로 두고, `CanvasRenderInput` 또는 render options에 `activeTrackIds`를 넘겨 draw 단계에서 alpha를 결정하는 방식입니다.
   - 다만 gliss/tuplet/vib/trem 등 복합 item 정렬까지 고려하면, track draw order 정렬은 item builder 쪽에서 한 번 해두는 편이 안정적입니다.

6. **동일 좌표 multi-track 편집의 기존 값 처리**
   - 명세는 “track별 기존 rawText를 따로 비교하지 않고 action 결과를 동일하게 쓴다”고 되어 있습니다.
   - 이 정책은 단순하지만, active track들이 서로 다른 값을 가진 상태에서 cycle 클릭을 하면 일부 track의 기존 내용을 덮어씁니다.
   - 의도된 정책이라면 괜찮고, 문서에 “multi-active edit은 overwrite 성격”이라고 더 명확히 적는 것이 좋습니다.

효율성 측면에서 가장 개선 여지가 큰 부분은 **active track 변경 시 full rebuild를 피하는 것**입니다. track toggle은 score data, parsed document, analysis result를 바꾸지 않습니다. 따라서 이상적인 경로는:

```text
activeTrackIds 변경
-> app state 갱신
-> renderer opacity/draw order 반영
-> audio schedule만 active track 기준으로 재생성
-> parse/analyze는 재실행하지 않음
```

정리하면, 제안은 이렇습니다.

- 마지막 active track 보호 제약은 제거
- `activeTrackIds = []`를 합법 상태로 정의
- 빈 active 상태에서는 inactive renderer 표시 유지, note edit/playback 차단
- track toggle은 parse/analyze 없이 renderer/audio만 갱신
- paused toggle 정책은 1차 구현에서는 stopped only로 단순화

이 방향이 명세도 단순하고, 구현 비용과 예외 처리를 줄입니다.

> 제안들을 수용하겠습니다. 추가 제안은 있나요?

추가 제안은 있습니다. 지금 수용한 방향과 잘 맞는 것들만 추리면 다음 정도입니다.

1. **track toggle은 “view/play/edit target”이 아니라 `track activity filter`로 명명**
   UI 문구는 `Tracks` / `Basic Optional Extra`로 충분하지만, 코드 쪽에서는 `activeTrackIds`가 “선택된 편집 track”만 뜻하지 않고 renderer/audio/edit 공통 필터이므로 함수명도 `isTrackActive`, `filterActiveTrackIds`, `getTrackRenderAlpha`처럼 공통 의미로 두는 편이 좋습니다.

2. **빈 active 상태의 renderer는 “모두 반투명 표시”로 고정**
   아무것도 안 보이게 하면 사용자가 어떤 track에 데이터가 있는지 알 수 없습니다. 빈 active 상태는 “전체 미선택 preview”에 가깝게 두고, 모든 track을 inactive alpha로 보여주는 쪽이 낫습니다.

3. **note edit 차단은 입력 전에 처리**
   `activeTrackIds = []`일 때는 edit command 생성 단계에서 차단하고 `edit_apply`까지 보내지 않는 편이 좋습니다. 그러면 `edit_apply`는 “비어 있는 batch를 어떻게 볼 것인가” 같은 정책을 몰라도 됩니다.

4. **track toggle 시 parse/analyze 생략을 명세에 명시**
   이건 성능상 중요합니다. active track 변경은 score rawText나 analyzer 결과를 바꾸지 않으므로:
   ```text
   parse/analyze 재실행 없음
   renderer 재표시
   audio schedule 재생성
   playback runtime reset
   ```
   으로 고정하는 편이 좋습니다.

5. **paused 상태도 1차에서는 toggle 비활성화**
   `playing`만 막고 `paused`를 허용하면 “재개 시 같은 위치에서 새 track 조합으로 이어지는가”를 처리해야 합니다. 1차 구현은 `stopped`에서만 toggle 허용이 가장 단순합니다.

6. **track order와 track list는 하드코딩 분산 금지**
   `["extra", "optional", "basic"]`, `["basic", "optional", "extra"]`, alpha, gain을 여러 파일에 흩뿌리면 나중에 틀어질 가능성이 큽니다. `src/track/track_control.ts`에 다음 정도를 모으는 게 좋습니다.
   ```ts
   TRACK_IDS
   TRACK_UI_ORDER
   TRACK_DRAW_ORDER
   INACTIVE_TRACK_ALPHA
   getTrackRenderAlpha()
   getTrackGain()
   filterActiveTrackResults()
   ```

7. **multi-active edit은 destructive overwrite임을 명세화**
   여러 active track에 같은 rawText를 쓰는 것은 단순하고 예측 가능하지만, 서로 다른 기존 값을 덮어씁니다. 이건 버그가 아니라 정책이어야 하므로 “multi-active note edit은 active tracks의 같은 좌표를 동일 rawText로 덮어쓴다”고 명시하는 게 좋습니다.

8. **검증 항목에 active 0개 케이스 추가**
   기존 검증 항목에서 “마지막 active track 차단”은 제거하고 아래로 바꾸면 됩니다.
   - 모든 track을 inactive로 만들 수 있다.
   - active track이 0개일 때 모든 track note가 inactive alpha로 표시된다.
   - active track이 0개일 때 note edit은 차단되고 global edit은 가능하다.
   - active track이 0개일 때 audio schedule은 비어 있다.

결론적으로, 지금 방향은 `activeTrackIds`를 “최소 1개 선택해야 하는 상태”가 아니라 “0개 이상 선택 가능한 필터”로 재정의하는 것이 핵심입니다. 이 정의만 문서와 구현에 일관되게 반영하면, 예외 처리가 줄고 성능 최적화 경로도 더 깔끔해집니다.

> 명세에 제안하신 사항들을 반영하겠습니다.
> 5번의 경우 나중에는 paused 상태에서 토글 조작이 가능하게 해야 하는데, 2차 구현으로 넘겨도 무리는 없는 것이 확실하지요?

네, **paused 상태 track toggle 허용은 2차 구현으로 넘겨도 무리 없습니다.** 오히려 1차에서는 `stopped`에서만 toggle 허용으로 제한하는 편이 더 안전합니다.

이유는 명확합니다.

- track toggle은 `AnalysisResult`는 바꾸지 않지만, audio schedule은 다시 만들어야 합니다.
- paused 상태는 “현재 score time을 보존한 채 재개”해야 하므로, toggle 후 새 active track 조합으로 schedule을 재생성하고 같은 위치에서 이어 재생하는 처리가 필요합니다.
- 현재 1차 목표는 track 표시, 편집 대상, 재생 필터링을 안정화하는 것이므로 paused resume semantics까지 같이 묶으면 검증 범위가 커집니다.
- 사용자는 paused 상태에서 toggle이 막혀도 stop 후 toggle하면 동일 기능을 사용할 수 있으므로 기능상 막히는 것은 아닙니다.

명세에는 이렇게 정리하면 좋겠습니다.

```text
1차 구현에서는 active track toggle은 playback state가 stopped일 때만 허용한다.
playing 또는 paused 상태에서는 toggle을 비활성화한다.

2차 구현에서는 paused 상태에서 active track toggle을 허용할 수 있다.
이 경우 activeTrackIds 변경 후 audio schedule을 재생성하고,
기존 paused score time을 유지한 채 resume 시 새 active track 조합으로 재생한다.
```

그리고 2차 구현 조건도 같이 적어두면 좋습니다.

```text
paused toggle 2차 구현 조건:
- 현재 score time 보존
- schedule rebuild 후 playback controller의 paused position 유지
- renderer opacity는 즉시 반영
- resume 시 새 activeTrackIds 기준 schedule만 재생
- 이미 예약된 audio node가 남지 않도록 pause 시 scheduler/backend 정리 보장
```

따라서 지금은 2차로 넘기는 판단이 맞습니다. 1차 범위에서는 `stopped only toggle`이 명세, 구현, 검증 모두 가장 단순합니다.

> 네. 명세를 직접 수정 부탁드립니다. 추가적으로 더 검토해 볼 사항이 없다면 ㄱ이후 구현에 들어가도 되겠습니다.

> 두가지 질문이 있습니다.
>  - DEFAULT_GLISS_CROSSFADE_SECONDS랑 DEFAULT_GLISS_INTERNAL_OVERLAP_SECONDS는 역할이 어떻게 다른가요?
>
>  - 타입 GlissOverLapInfo는 start와 end의 overlapSeconds를 나눴는데, 둘은 동일한 시간 간격을 가지는게 일반적일 것으로 보입니다. 하나로 통일해도 무방하지 않을까요?

두 값의 역할은 다릅니다.

`DEFAULT_GLISS_CROSSFADE_SECONDS`는 **gliss segment 자체의 외곽 fade 길이**입니다. 연결되지 않은 단독 gliss나 S-E gliss에서 시작할 때 서서히 켜지고, 끝날 때 서서히 꺼지는 envelope 용도입니다. 즉 “이 gliss oscillator를 켜고 끄는 기본 fade”입니다.

`DEFAULT_GLISS_INTERNAL_OVERLAP_SECONDS`는 **연결된 gliss segment 사이의 내부 anchor 연결부 겹침 길이**입니다. S-M-E에서 S-M segment와 M-E segment가 M anchor에서 만날 때, 앞 segment를 조금 늦게 끄고 뒤 segment를 조금 일찍 켜서 중간이 꺼지는 느낌을 줄이는 용도입니다. 즉 “두 oscillator를 이어 붙이는 crossfade 폭”입니다.

두 번째 질문은 방향이 맞습니다. 현재 `GlissOverlapInfo`가 `startOverlapSeconds`, `endOverlapSeconds`를 나눈 이유는 각 segment 입장에서 “앞에서 겹치는가”와 “뒤에서 겹치는가”가 다르기 때문입니다.

예를 들면 S-M-E 두 segment는 이렇게 됩니다.

```text
S-M: start 0, end 0.02
M-E: start 0.02, end 0
```

그래서 각 event에는 start/end 방향성이 필요합니다. 다만 “겹치는 시간 값 자체”는 말씀대로 일반적으로 같아도 됩니다. 지금도 실제 값은 둘 다 `DEFAULT_GLISS_INTERNAL_OVERLAP_SECONDS` 하나를 사용합니다.

정리하면:

- `startOverlapSeconds` / `endOverlapSeconds` 필드는 유지하는 게 안전합니다.
- 하지만 “몇 초 겹칠지”를 나타내는 기본 상수는 하나면 충분합니다.
- 이미 현재 구현도 `DEFAULT_GLISS_INTERNAL_OVERLAP_SECONDS` 하나로 양쪽 값을 채우고 있어서 이 부분은 적절합니다.

더 단순화한다면 타입을 이렇게 바꿀 수는 있습니다.

```ts
type GlissOverlapInfo = {
  hasStartOverlap: boolean;
  hasEndOverlap: boolean;
};
```

그리고 backend에서 boolean이면 `DEFAULT_GLISS_INTERNAL_OVERLAP_SECONDS`를 적용하게 할 수 있습니다. 하지만 그러면 event DTO가 “실제 예약 시간”을 덜 명시적으로 갖게 됩니다. 나중에 짧은 segment에서는 overlap을 줄이거나, tempo/길이에 따라 overlap을 다르게 주려면 현재처럼 seconds 값을 내려보내는 편이 확장성이 좋습니다.

따라서 지금 구조는 유지해도 괜찮고, 이름만 더 분명하게 바꾸는 정도가 좋겠습니다. 예를 들면 `internalStartOverlapSeconds`, `internalEndOverlapSeconds`처럼요.

> overlap 수치를 조정했습니다. 0.1 이상은 볼륨이 튀는 문제가 발생하고, 0.02는 큰 변화가 없어 0.05로 맞췄습니다. 다만 근본적으로 끊김을 숨길 수는 없는 듯해 보이네요.

두 항목 모두 가능하지만, 첫 번째는 정책 선택이 필요합니다.

`트랙 레이어간 볼륨 총합 일정`은 해석이 두 가지입니다.

1. **active track 수 기준 정규화**
   - 예: active track이 3개면 각 트랙 gain을 `1/3` 또는 `1/sqrt(3)`로 낮춤
   - 구현이 단순하고 안정적입니다.
   - 단점: 실제로 한 트랙만 소리나는 순간도 작게 들립니다.

2. **실제로 동시에 울리는 event 수 기준 정규화**
   - 예: 같은 시각에 basic+optional이 겹치면 각각 낮추고, basic만 울리면 원래 볼륨
   - 사용자가 말한 “겹치는 상황”에는 이쪽이 더 정확합니다.
   - 단점: 긴 note 중간에 다른 트랙이 들어오고 나가면 gain automation을 추가해야 해서 구현이 더 복잡합니다.

제 판단으로는 1차는 `1/sqrt(n)` 계열보다는 **실제 동시 발음 event 수 기준**이 맞습니다. 단, 완전한 실시간 compressor가 아니라 schedule 단계에서 overlap 구간을 계산해 `gainRamp` 또는 step gain automation을 추가하는 방식이 좋습니다. 그러면 절대 합은 줄이되, 너무 납작해지는 느낌은 피할 수 있습니다.

두 번째, `일시정지 후 재개 시 이벤트 중간부터 재생`은 방향이 명확합니다. 이미 `getEventsOverlappingRange()`가 있어서 기반은 있습니다. 필요한 변경은:

- resume/playFromSeconds 직후, 시작 시각 이전에 시작했지만 아직 끝나지 않은 event도 예약
- note는 `resumeTime -> event.endSeconds`만 울림
- gliss는 resume 지점의 보간 pitch에서 시작해 원래 end pitch까지 ramp
- vibrato/tremolo/dynamics automation도 resume 이후 구간으로 clamp
- 기존 start 기준 envelope 대신 “중간 진입용 짧은 fade-in” 적용

여기서 확인할 점은 하나입니다.

재개 시 중간 event는 **원래 envelope 상태를 최대한 복원**하기보다, 짧은 fade-in으로 자연스럽게 들어오게 하면 될까요?
저는 이 방식이 현재 oscillator backend 구조에 맞고, pause/resume 청감도 안정적이라고 봅니다.

> 1. 네. 실제 이벤트 기준이 맞습니다.
> 2. 네. fade in이 없으면 볼륨이 튈 위험이 있으므로, 그렇게 처리합니다.

> 전체 악보의 모든 이벤트를 한번에 모아 시간 경계 목록을 만들지는 않죠? lookahead를 따라 일정 시간 간격마다 다시 모으니까요.

lookahead마다 다시 계산하지 않습니다.

정확히는 [audio_schedule_builder.ts](./regression-code/src/audio/audio_schedule_builder.ts)에서 `buildAudioScheduleWithMapper()`가 전체 `events`를 만든 뒤:

```ts

```

를 호출합니다. 이 함수가 전체 event의 `startSeconds`, `endSeconds`를 모두 모아 `boundaries`를 만들고, 각 event에 `gainScale` automation을 붙입니다.

그 뒤 lookahead scheduler는 이미 만들어진 schedule을 시간 범위로 조회해서 예약만 합니다. 즉:

```text
analyzer result
-> 전체 AudioSchedule 생성
-> 전체 event 기준 gainScale automation 계산
-> playback 중 lookahead는 조회/예약만 수행
```

입니다.

다만 질문하신 방식, 즉 **lookahead 구간마다 동시 이벤트 수를 다시 계산하는 방식**도 가능합니다. 하지만 현재 구조에서는 그 방식보다 전체 schedule 생성 시점 계산이 더 안정적입니다.

이유는:
- 같은 score + active tracks에서 항상 같은 gain 결과가 나옵니다.
- lookahead window 경계에서 gainScale이 달라지는 문제를 피합니다.
- pause/resume/seek 시에도 동일한 automation을 재사용할 수 있습니다.
- 현재 MVP 규모에서는 전체 schedule event 수가 아주 크지 않으면 비용이 감당 가능합니다.

대신 단점도 있습니다. 매우 큰 악보에서는 전체 event 경계 계산이 `O(n²)`에 가까워질 수 있습니다. 이후 가상화/부분 schedule 캐시로 넘어가면 이 부분은 최적화 대상입니다.

그래서 구현 메모의 “schedule 생성 시점의 결정적 정규화”라는 설명은 현재 코드와 일치합니다.제가 직전 답변에서 “일정 시간 간격마다 다시 모은다”는 식으로 들리게 말했다면 그건 현재 구현과 다릅니다. 현재는 **전체 schedule 생성 시 1회 계산**입니다.

> 말씀대로 악보가 커지면 병목이 생길 것이 우려됩니다.

맞습니다. 현재 방식은 큰 악보에서 병목 후보입니다.

현재 알고리즘은 대략:

```text
events 전체 수 = n
boundary 수 <= 2n

각 event마다 전체 boundary 구간을 순회
각 구간마다 전체 events를 다시 filter/count
```

라서 최악에 가깝게 보면 `O(n^3)` 성향까지 갈 수 있습니다. 실제로는 boundary merge와 event 길이 때문에 덜하겠지만, 큰 악보에서는 불안한 구조입니다.

개선 방향은 명확합니다. 전체 구간을 sweep-line으로 한 번 훑으면 됩니다.

```text
1. 모든 event의 start/end를 time point로 만든다.
2. 시간순 정렬한다.
3. active event Set을 유지하며 start에서는 추가, end에서는 제거한다.
4. 인접 time span마다 active count를 기록한다.
5. 각 event에는 자신이 포함된 span의 gainScale만 붙인다.
```

복잡도는 대략:

```text
정렬: O(n log n)
sweep: O(n + k)
event별 automation 부여: 구현 방식에 따라 O(n + totalOverlaps)
```

현재 `countOverlappingEvents()`처럼 매 구간마다 전체 event를 다시 세는 방식은 없앨 수 있습니다.

더 나은 실용 구현은 다음 형태입니다.

```ts
type GainScaleSpan = {
  startSeconds: number;
  endSeconds: number;
  gainScale: number;
};
```

먼저 전체 `GainScaleSpan[]`을 만든 뒤, 각 event에는 자기 구간과 겹치는 span만 붙입니다. `spans`와 `events`가 모두 시간순이므로 포인터를 전진시키면 전체 비용을 줄일 수 있습니다.

제안은 이겁니다.

- 지금 당장 sweep-line으로 바꾸는 편이 좋습니다.
- 기능 의미는 그대로 유지됩니다.
- 테스트 기대값도 거의 그대로 유지됩니다.
- 나중에 partial schedule이나 chunk schedule로 갈 때도 `GainScaleSpan` 구조가 재사용됩니다.

즉, 현재 구현은 청감 검증용으로는 충분하지만, 큰 악보를 고려하면 바로 sweep-line으로 교체하는 게 낫습니다.

> n^3은 치명적이네요. sweep-line으로 진행합시다.

현재 구조는 대략 이렇습니다.

```text
앞 segment: anchor까지 full gain, anchor 이후 fade-out
뒤 segment: anchor 이전 fade-in, anchor부터 full gain
```

그래서 anchor 순간에는:

```text
앞 full + 뒤 full
```

이 되어 볼륨이 튈 수 있습니다. 글리산도 연결부도 말씀대로 오디오 overlap인데, 현재 `gainScale`은 schedule event의 원래 `startSeconds/endSeconds` 기준으로 계산하므로 backend에서 추가로 늘린 overlap 구간까지는 모릅니다.

복잡한 예외 처리를 피하려면 두 선택지가 있습니다.

1. **gliss overlap을 gainScale에 포함**
   - schedule event에 실제 audible start/end까지 반영해야 합니다.
   - gliss용 예외가 schedule 정규화까지 퍼집니다.
   - 로직이 커집니다.

2. **gliss 내부 crossfade 자체를 constant-sum 형태로 수정**
   - 앞 segment와 뒤 segment가 anchor에서 동시에 full gain이 되지 않게 합니다.
   - 예를 들어 연결부 중심을 anchor로 두고:
     ```text
     앞 segment: anchor - overlap/2 부터 anchor + overlap/2 까지 1 -> 0
     뒤 segment: anchor - overlap/2 부터 anchor + overlap/2 까지 0 -> 1
     ```
   - 두 oscillator의 합이 대략 일정해집니다.
   - schedule/gainScale 로직은 건드리지 않아도 됩니다.

저는 2번이 맞다고 봅니다. gliss 연결부는 이미 backend 내부에서만 처리하는 “세그먼트 접합 완화”이므로, 그 안에서 crossfade envelope을 제대로 잡는 편이 경계가 깔끔합니다.

다만 이전에도 확인했듯이 gliss 끊김 자체를 완전히 없애기는 어렵습니다. 지금 목표는 “숨기기”보다 “튀지 않게 완화”가 맞습니다. 다음 수정은 overlap 시간을 유지하되, anchor에서 두 음이 동시에 full이 되지 않도록 crossfade를 anchor 중심으로 바꾸는 게 좋겠습니다.

이세계 코드 시절에는 지금처럼 “gliss segment별 별도 oscillator를 crossfade로 잇는 방식”이 주된 해결책이 아니었습니다.

핵심은 legacy의 `playLegatoVibSpan_()`입니다.
[parts_analyze_audio.html](<./0.%20이세계%20코드%20(legacy>)/parts_analyze_audio.html)에서 이 함수는 **하나의 oscillator를 만들고**, 그 oscillator의 `frequency`에 여러 gliss segment를 순서대로 automation했습니다.

요약하면:

```text
하나의 legato span oscillator
-> gain envelope는 span 전체에 1회 적용
-> gliss segment들은 같은 oscillator.frequency에 setValue / curve 예약
```

그래서 S-M-E 연결부에서도 oscillator가 교체되지 않습니다. 즉:
- oscillator start/stop 없음
- phase reset 없음
- 두 oscillator overlap 없음
- crossfade도 필요 없음

legacy의 관련 흐름은 이렇습니다.

- `glissLegatoSegs`를 따로 모음
- `startHoldLike`인 gliss는 별도 gliss event가 아니라 legato span에 붙임
- `playLegatoVibSpan_(..., glissSegs)`에서 같은 oscillator에 각 gliss segment를 적용
- `applyGlissCurve_()`는 `setValueCurveAtTime()`으로 smoothstep curve를 사용

반면 별도 함수 `playGlissAt_()` / `playGlissWithTremAt_()`도 있었습니다. 이쪽은 독립 gliss oscillator를 만들고 `headSec`, `tailSec`, `attackSec` 같은 값을 받습니다. 하지만 연결된 legato gliss에서 끊김을 피하는 더 중요한 경로는 **단일 oscillator span 안에 gliss를 병합하는 방식**이었습니다.

현재 회귀 코드에서 겪는 문제는 구조적으로 자연스럽습니다.

```text
현재:
gliss segment A oscillator
gliss segment B oscillator
-> 둘을 fade/crossfade로 연결하려고 함
-> phase 차이, gain 합, automation 경계 때문에 튐/치직거림 발생

legacy:
하나의 oscillator
-> frequency만 시간에 따라 이동
-> 연결부 oscillator 교체 없음
```

그래서 결론은 이렇습니다.

- 지금의 crossfade 보정은 임시 완화책으로 한계가 있습니다.
- 이세계 코드가 더 안정적으로 들렸다면 이유는 **voice-span 병합**에 가깝습니다.
- 회귀 코드에서도 근본 해결은 “같은 gliss chain을 하나의 oscillator frequency automation으로 재생”하는 쪽입니다.
- 다만 이건 현재의 “앵커 한 쌍별 segment 구조 유지”와 충돌하지는 않습니다. analyzer/schedule DTO는 segment를 유지하되, backend 예약 직전에 같은 chain을 하나의 playback voice로 묶을 수 있습니다.

즉, 지금 단계에서 더 crossfade를 만지는 것보다, “연결된 gliss segment를 재생 단계에서만 voice span으로 병합”하는 후속 작업으로 넘기는 게 맞아 보입니다.

> 결국 체인 방식인가요... 부분 갱신 구현때 범위가 얼마나 늘어날지 모르겠습니다. 다만 청감은 중요하므로 해결 방식이 이것밖에 없다면 어쩔 수 없기는 하겠습니다. 큰 변경이 되겠네요.

네, 근본 해결은 사실상 **체인/voice-span 방식**입니다. 지금 겪은 현상이 그 근거입니다.

crossfade 방식은 작은 보정으로는 가능하지만, oscillator가 바뀌는 순간의 phase reset과 envelope 경계는 완전히 숨기기 어렵습니다. 특히 S-M-E처럼 같은 pitch anchor에서 연속되는 gliss는 사용자가 “하나의 손가락 움직임”처럼 듣기를 기대하는데, 현재 구조는 “두 개의 소리를 이어 붙이는” 방식이라 한계가 드러납니다.

다만 “큰 변경”의 범위를 줄이는 방법은 있습니다.

추천 방향은 analyzer를 바꾸는 게 아니라, **audio schedule 단계에만 chain grouping을 추가**하는 것입니다.

```text
Analyzer:
GlissEvent segment 구조 유지

Audio schedule builder:
같은 trackId + glissId + 연결된 시간/pitch를 가진 GlissEvent들을 chain으로 그룹화

Backend:
chain 하나를 하나의 oscillator로 재생
각 segment는 같은 oscillator.frequency automation으로 예약
```

이렇게 하면 부분 갱신 관점에서도 경계가 비교적 명확합니다.

- analyzer event는 그대로 segment 단위
- renderer도 그대로 segment 단위
- audio schedule만 “segment event”와 “chain playback event”를 구분
- 부분 갱신 시에는 변경된 gliss anchor 주변의 chain만 재생성하면 됨

문제는 “어디까지 chain으로 묶을 것인가”입니다. 최소 구현은 다음 조건만 묶으면 됩니다.

```text
same trackId
same glissId
prev.endAnchorTick === next.startAnchorTick
prev.endSound === next.startSound
```

이 조건이면 현재 내부 overlap 판정과 거의 같습니다. 즉 기존 로직을 확장해 chain event를 만들 수 있습니다.

가장 안전한 단계적 구현은:

1. 기존 fallback gliss event는 유지
2. 연결된 2개 이상 gliss segment만 `AudioGlissChainScheduleEvent`로 대체
3. 단독 S-E gliss는 기존 fallback 그대로 유지
4. tremolo가 섞인 gliss chain은 일단 fallback 유지하거나, chain 전체에 gate 적용
5. 테스트로 S-M-E가 chain 1개가 되는지만 검증

이러면 변화 범위는 `audio_types`, `audio_schedule_builder`, `oscillator_backend`, `test_audio`, 구현 메모 정도로 제한됩니다. 그래도 작은 변경은 아니지만, parser/analyzer/renderer까지 흔드는 대형 변경은 피할 수 있습니다.

청감이 중요하다면 이 방향으로 가는 게 맞습니다. Crossfade를 더 만지는 건 시간을 더 쓰고도 애매할 가능성이 높습니다.

> crossfade와 오버랩 관련 로직은 완전히 폐기되겠네요?

대부분은 폐기하는 편이 맞습니다. 특히 **연결된 gliss segment를 잇기 위한 overlap/crossfade 로직**은 chain 방식이 들어가면 필요 없어집니다.

정리하면:

폐기 대상:
- `startOverlapSeconds`
- `endOverlapSeconds`
- `DEFAULT_GLISS_INTERNAL_OVERLAP_SECONDS`
- 연결된 gliss segment를 찾아 overlap 값을 주는 로직
- backend에서 gliss oscillator 시작/끝을 overlap만큼 늘리는 로직
- anchor 중심 crossfade envelope

유지 가능 대상:
- `crossfadeSeconds` 자체는 단독 gliss fallback의 fade-in/out 용도로 남길 수 있습니다.
- 단독 S-E gliss나 chain으로 묶지 않는 예외 gliss는 여전히 별도 oscillator로 재생되므로, 시작/끝 click 방지를 위한 짧은 fade는 필요합니다.
- 다만 이름은 `crossfadeSeconds`보다 `fadeSeconds` 또는 `edgeFadeSeconds`가 더 정확해집니다. 실제로는 다른 oscillator와 crossfade하는 게 아니라 단독 event의 edge fade니까요.

chain 방식 이후 구조는 이렇게 되는 게 깔끔합니다.

```text
단독 gliss fallback:
- 하나의 gliss oscillator
- edge fade in/out만 적용

연결 gliss chain:
- 하나의 oscillator
- chain 전체 edge fade in/out만 적용
- segment 경계에서는 gain 변화 없음
- frequency automation만 이어서 예약
```

그래서 “완전히 폐기?”에 대한 답은:

- **연결부 overlap/crossfade는 폐기**
- **단독 gliss의 시작/끝 fade는 유지**
- `crossfadeSeconds`라는 이름은 `fadeSeconds` 계열로 바꾸는 것이 좋음

그리고 현재까지 추가한 overlap 관련 구현 메모도 chain 도입 시에는 “시도했으나 phase/envelope 문제로 폐기”로 남기거나, 별도 문단으로 정리하는 편이 보고서에도 좋습니다.
