# 1.0. 첫 인크리먼트 설계 로드맵
> 상위 경로에 존재하는 md파일 전반은 specification이다.
> 본 경로에 존재하는 md파일들은 세부 구현 일지 및 해설 문서로 그 성격이 명확히 구분된다.
> specification의 넘버링 1.x에서 각 모듈의 기능 정의 / 필요한 클래스 정의 / 모듈 사이의 인터페이스 정의를 완료하였다.
> 이제 실제 ts 파일 구현에 들어가 클래스를 선언하고, 각 모듈의 input에서 output을 도출하는 알고리즘의 구현에 들어간다.
> 본 설계 메모 문서 세트에서는 코드의 구상부터 실제 제작 과정, 이후 제작된 코드의 해설까지 구현 전반에 대해 세부 설명으로 보충하는 내용을 담는다. 

## 1. 목표

- 첫 인크리먼트 전체 목표
    - JSON loader -> parser -> analyzer 까지의 입력과 출력 과정을 구현한다.
    - loader와 parser 사이에는 악보 파일의 유효성과 안전성을 검사할 validator가 낄 예정이다. 초기 테스트 단계에서 벌써 공격자를 고려할 것은 없어 보이므로 현 구현 목표에서는 제외한다.

- 4월 5주차 부분 목표
    - JSON -> parser output 도출까지의 단계를 완성하는 것을 목표로 한다.

### 3.1. index.ts 배럴 파일에 대하여

core 쪽에 index.ts 파일을 만들어 거대한 흐름을 export를 통해 간략하게 정리할 수도 있을 것이다. 이걸 배럴 파일이라고 부른다.
배럴
대형 프로젝트에서는 하위 폴더에도 하나하나 index.ts 파일을 만들면 연쇄적으로 배럴 파일이 import되어 overhead가 발생하는 문제가 있다고 한다. 본 프로그램은 학부생 규모의 1인 프로젝트니 걱정하지 않아도 될 것 같지만 그래도 검토는 해보기로 하였다.

index.ts의 남용이 불러일으키는 문제를 요약하면 다음과 같다.

- 성능 오버헤드
    - export로 하위 배럴 파일을 import한다. 배럴 파일은 많은 export 문을 포함하므로 그 중 또 다른 하위 배럴 파일을 포함할 수도 있다. 연쇄적으로 수많은 파일 묶음이 불러와지면서 성능 오버헤드가 발생할 수 있다.

- 모듈간 경계 흐려짐
    - 위처럼 연쇄적으로 수많은 파일을 불러오게 되면 모든 흐름을 파악하고 통제하기 어렵다. 하나의 모듈에서 필요한 파일만 부르려고 했는데 다른 모듈의 파일이 불러와질 가능성도 높아지며 따라서 모듈 경계가 흐려져 버린다.

AI의 평가는 다음과 같다.
- 본 프로젝트의 규모는 성능 오버헤드를 걱정할 정도는 아니다.
- 하지만 모듈간 경계 침범은 하나의 index.ts가 수십개의 파일을 포함하게 되거나 계층이 2단 이상만 늘어나도 신경써야 하는 문제가 된다.

수십 export를 포함하는 대규모 index.ts를 생성하게 될지는 아직 모르겠다. 다만 폴더가 2계층 이상으로 늘어나는 일은 충분히 발생할 수 있다. 이때 core와 같은 아키텍처 단위의 상위 폴더에서만 index.ts를 가지고 나머지 내부 파일들은 일단 import문으로 우선 처리하는 방향으로 가기로 한다. 정말 필요한 경우에만 신중히 배럴 파일의 사용을 결정한다.
