이해한 현재 상태는 다음과 같습니다.

현재 `회귀 코드`는 `regression-code/`를 중심으로 구현 중이며, `이세계 코드`는 참고용 legacy입니다. 지금 초점은 긴 악보, 특히 4000열 이상에서 편집 시 체감되는 지연을 줄이는 것입니다.

현재까지 진행된 핵심 작업은 `docs/2.8-edit-invalidation-and-partial-rebuild-spec.md` 기준의 부분 갱신 1차/2차 일부 구현입니다.

- note/global 편집을 `noteCell`, `globalCell`, `mixedCell`, `structure`로 구분
- note 편집 시 base/layout/global marker redraw를 피하고 note 계열만 갱신
- global 편집 시 note event redraw를 피하고 global text/marker 중심으로 갱신
- renderer item을 note/global, marker 계열로 분리
- canvas layer를 layout/base/global marker/note marker/note 등으로 분리
- note 편집은 dirty tick range 기반으로 note row 일부 redraw
- note marker는 gliss/orphan/tuplet 영향 때문에 아직 넓게 redraw
- `ScoreFile` 전체 JSON deep clone을 제거하고, edited track/global cells만 복제
- `ParsedScoreDocument`는 note/global 그룹을 재사용
- analyzer는 note 편집 시 edited track 중심, global 편집 시 timing/dynamics 중심으로 재생성
- drag 편집은 `requestAnimationFrame` 단위로 batch 처리해 pointermove마다 rebuild가 반복되는 것을 줄임
- audio schedule은 아직 note/global 편집 모두에서 full rebuild 유지
- `RuntimeDocument` / `ScoreIndexes`도 아직 매 edit마다 재생성되는 쪽에 가까움
- 4000열 수준에서는 이전보다 병목은 줄었지만, 입력 지연이 아직 남아 있음

현재 병목 후보는 대략 세 가지로 보입니다.

1. `RuntimeDocument` / `ScoreIndexes` 재생성 비용
   편집 셀은 일부인데 런타임 인덱스가 아직 전체 범위 기준으로 다시 만들어지는 부분이 남아 있습니다.

2. audio schedule / playback reset 비용
   note/global rawText 편집 후 audio schedule은 아직 full rebuild라서 긴 악보에서 누적 비용이 날 수 있습니다.

3. full-width canvas 정책
   5000열 전후에서는 브라우저 canvas 한계와 redraw 폭 문제가 커집니다. 현재 DPR/bitmap cap은 임시 안정화 조치이고, 근본 해결은 viewport/tile rendering입니다.

앞으로 예정된 작업은 우선순위상 다음 흐름으로 이해했습니다.

1. 현재 edit path를 더 계측해서 실제 병목을 분리
   `apply raw text`, runtime/index rebuild, partial artifact rebuild, render patch, canvas redraw, playback reset 중 어디가 4000열 지연의 주 원인인지 숫자로 확인합니다.

2. `RuntimeDocument` / `ScoreIndexes` 부분 갱신 검토
   ordinary rawText 편집은 구조 변경이 아니므로 전체 인덱스 재생성 없이 edited track/global cell index만 갱신할 수 있는지 확인합니다.

3. audio schedule 부분 갱신 또는 reset 비용 축소
   당장 완전한 partial audio가 어렵다면, 편집 중 반복 reset을 더 줄이거나 schedule rebuild 타이밍을 늦추는 방식도 후보입니다.

4. viewport rendering 도입
   전체 score width를 매번 canvas에 들고 가는 구조를 줄이고, 현재 보이는 column 범위와 overscan만 draw하는 구조로 전환합니다. 장악보 성능 개선의 가장 큰 축입니다.

5. tile / dirty-range rendering 확장
   viewport rendering 이후에는 column tile 캐시나 더 정교한 dirty range를 붙여 scroll/edit/playback에서 전체 폭 redraw를 피합니다.

6. analyzer range partial은 후속 단계
   현재는 edited track 전체 재분석에 가깝고, hold/gliss/tuplet 때문에 영향 범위 계산이 복잡합니다. viewport/tile과 runtime/audio 쪽 병목을 먼저 줄인 뒤, 필요하면 dirty column 주변 range partial로 들어가는 순서가 안전합니다.

즉, 현재 작업은 “전체 rebuild에서 부분 갱신으로 넘어가는 중간 단계”이고, 이미 parser/analyzer/render item 재사용과 layer 분리는 들어갔습니다. 남은 큰 과제는 `runtime index`, `audio schedule`, `canvas full-width rendering` 세 영역이며, 4000열 지연은 이 세 축을 계측한 뒤 가장 큰 비용부터 줄이는 방향이 맞습니다.

`canvas render`는 순수하게 “그리기 함수 하나”만이 아니라, `renderAppPartial()` 전체입니다. 즉 아래가 포함됩니다.

note edit의 일반 경로는 [canvas_score_renderer.ts](./regression-code/src/renderer/canvas_score_renderer.ts:75)의 `renderCanvasScorePartial()`입니다.

1. `buildCanvasScoreLayout()`
   - [canvas_coordinate.ts](./regression-code/src/renderer/canvas_coordinate.ts:107)
   - 모든 row의 y/height, 전체 stage width/height, column width 계산
   - 5000열이면 `stageWidth = columnCount * columnWidth`

2. 기존 canvas 크기 재사용 가능 여부 확인
   - [canvas_score_renderer.ts](./regression-code/src/renderer/canvas_score_renderer.ts:122)
   - row 목록을 비교해서 full render fallback이 필요한지 확인

3. note edit이면 동적 layer redraw
   - [canvas_score_renderer.ts](./regression-code/src/renderer/canvas_score_renderer.ts:95)
   - dirty range가 있으면 현재는:
     - `drawScoreMarkers(noteMarker, ..., noteMarkerItems)`
     - `drawScoreNotesInRange(note, ..., dirtyTickRange)`
     - `drawScoreOverlayMarkersInRange(note, ..., dirtyTickRange)`

여기서 가장 의심되는 부분은 첫 번째입니다.

`drawScoreMarkers()`는 [canvas_marker_renderer.ts](./regression-code/src/renderer/canvas_marker_renderer.ts:29)에서 시작하고, 맨 처음에:

```ts
context.clearRect(0, 0, layout.stageWidth, layout.stageHeight);
```

를 실행합니다. 즉 note edit마다 `noteMarker` canvas 전체를 지웁니다. 5000열이면 실제 marker가 0개거나 적어도, `stageWidth * stageHeight` 크기의 canvas clear가 발생합니다. 사용자가 본 `canvas render 2.8~5.4ms`의 주범 후보입니다.

그 다음 `drawScoreNotesInRange()`는 [canvas_note_renderer.ts](./regression-code/src/renderer/canvas_note_renderer.ts:84)입니다. 이 함수는 전체 note canvas를 지우지는 않고 dirty tick 범위만 지웁니다. 다만 [clearNoteRowsInRange](./regression-code/src/renderer/canvas_note_renderer.ts:183)에서 모든 note row를 순회하며 같은 x 범위를 `clearRect`합니다. 그리고 note/mute item 배열 전체를 훑으면서 dirty range와 겹치는 item만 다시 그립니다.

마지막으로 `canvas render` 계측에는 `renderAppPartial()` 안의 후처리도 포함됩니다.

- CSS 변수 갱신: `--score-stage-width`, `--score-scroll-width`, `--score-stage-height`, `--label-width`
- label scroll sync
- 하단 status 텍스트 갱신

이 부분은 [app_ui_sync.ts](./regression-code/src/app/app_ui_sync.ts:280)의 `renderAppPartial()`에 들어 있습니다.

정리하면 현재 note edit의 `canvas render` 비용은 주로:

1. layout 재계산
2. noteMarker layer 전체 clear/redraw
3. note layer dirty x range clear/redraw
4. tuplet overlay dirty range redraw
5. stage CSS 변수/status 갱신

입니다.

지금 수치상 runtime/analyze/audio는 거의 문제가 아니고, 다음 계측은 `canvas render` 내부를 `layout`, `noteMarker full clear/redraw`, `note dirty redraw`, `overlay dirty redraw`, `CSS/status sync`로 쪼개는 게 맞습니다. 특히 `noteMarkerItems`가 거의 없는 상황에서도 시간이 크면, 전체 `noteMarker` canvas `clearRect`가 병목이라고 봐도 됩니다.

중요한 점은 JS 내부 계측에서 `drawScoreNotesInRange()`가 1~2ms여도, 브라우저는 그 이후 큰 canvas bitmap/raster/composite 작업을 frame 전에 처리합니다. 이 비용은 `performance.now()`로 감싼 JS 함수 안에는 잘 안 잡히고, 지금처럼 `syncToFirstRafMs`에서 드러납니다.

따라서 다음 최적화 방향은 명확합니다.

1. 작은 최적화로 해결 불가
   `clearRect` 범위를 줄이거나 item loop를 줄이는 것만으로는 한계가 있습니다. 이미 dirty redraw 자체는 작게 하고 있는데도, 큰 canvas surface에 draw했다는 사실이 문제입니다.

2. 핵심 해결은 viewport/tile rendering
   note/noteMarker 같은 자주 바뀌는 dynamic layer를 전체 4000열 canvas가 아니라 현재 viewport 폭 + overscan 정도의 작은 canvas에 그려야 합니다.

3. 단기 우회책은 drag 중 dynamic canvas redraw 지연
   드래그 중에는 score data만 반영하고 실제 canvas redraw는 pointerup 또는 낮은 빈도로 하는 방식입니다. 반응성은 좋아지지만 시각 피드백이 늦어지는 tradeoff가 있습니다.

제가 보기엔 다음 작업은 `dynamic viewport canvas` 실험이 맞습니다. 즉 base grid는 일단 기존 full canvas를 유지하더라도, `note` / `noteMarker` layer부터 viewport 폭으로 제한해서 그려보고 frame 지연이 사라지는지 확인하는 겁니다. 이게 성공하면 이후 base/marker도 tile 또는 viewport 방식으로 확장하면 됩니다.

> 로직에 대한 설명이 가상화처럼 들립니다. 다른 알고리즘인가요?

네, 사실상 **가상화(virtualization)** 계열입니다. 다만 적용 대상이 DOM 리스트가 아니라 **canvas score surface**라는 점이 다릅니다.

현재 구조는 “전체 악보를 하나의 긴 canvas bitmap에 그려두고, scroll container로 일부만 보여주는 방식”입니다. 4000열이면 보이는 영역은 몇십 열이어도 `note-layer` canvas 자체는 4000열 폭을 가집니다. 그래서 작은 dirty range만 그려도 브라우저가 큰 canvas surface를 frame에 반영하면서 비용이 튑니다.

제가 말한 viewport/tile rendering은 다음 중 하나입니다.

1. **Viewport rendering**
   - canvas bitmap 자체를 현재 보이는 화면 폭 + overscan 정도로만 둡니다.
   - scrollLeft를 보고 “현재 보이는 column 범위”만 canvas에 그립니다.
   - canvas는 화면 위에 고정된 크기이고, 내부 좌표만 scroll offset을 빼서 그립니다.
   - DOM 리스트 가상화와 가장 비슷합니다.

2. **Tile rendering**
   - 전체 악보를 예를 들어 256열/512열 단위 tile canvas들로 나눕니다.
   - 보이는 tile만 만들고, dirty tile만 다시 그립니다.
   - 긴 악보 전체를 하나의 거대한 canvas로 유지하지 않습니다.
   - 지도 앱이나 대형 이미지 뷰어 방식에 가깝습니다.

3. **Hybrid**
   - edit 중에는 viewport dynamic layer만 사용
   - base grid나 global marker는 tile/cache로 점진 전환
   - 구현 리스크를 줄이기 좋습니다.

지금 병목은 “분석/파싱 알고리즘”이 아니라 “렌더링 표면이 너무 큼”입니다. 그래서 알고리즘 개선이라기보다는 **렌더링 데이터 구조와 화면 반영 방식 변경**에 가깝습니다.

현재 단계에서 가장 작은 실험은:

- `note` / `noteMarker` layer만 viewport canvas로 바꾸기
- `base` / `marker`는 기존 full canvas 유지
- scroll 시 viewport dynamic layer를 다시 그림
- edit 시 현재 viewport canvas만 갱신

> 잘 동작합니다. 이러면 지금까지 구현한 partial build는 별로 의미가 없게 된 건가요...?

아닙니다. 의미가 있습니다. 다만 **병목의 1순위가 바뀐 것**에 가깝습니다.

partial build가 해결한 것은 이쪽입니다.

- score edit 후 전체 parse/analyze를 매번 돌리지 않기
- edited track/global만 재계산하기
- renderer item group을 note/global로 나누기
- audio/render invalidation 범위를 분리하기
- drag 중 edit batch를 묶기

이번에 드러난 병목은 그 다음 단계였습니다.

- JS 계산은 이미 2~10ms 수준으로 내려옴
- 그런데 full-width dynamic canvas에 draw하면 브라우저 frame 반영이 70~300ms까지 튐
- 그래서 dynamic viewport canvas가 필요해짐

즉 partial build가 없었다면 viewport rendering을 넣어도 parse/analyze/audio 쪽에서 다시 막혔을 가능성이 큽니다. 지금은 partial build 덕분에 “남은 병목이 canvas surface 크기”라는 걸 명확히 분리할 수 있었습니다.

그리고 앞으로도 partial build는 계속 필요합니다.

- viewport rendering은 **화면에 그리는 범위**를 줄입니다.
- partial build는 **데이터를 다시 만드는 범위**를 줄입니다.

둘은 다른 층입니다. 긴 악보에서 안정적으로 빠르려면 둘 다 필요합니다.

다만 우선순위는 조정됩니다. 지금 당장 체감 성능에는 `dynamic viewport rendering`이 훨씬 큽니다. 그래서 다음 작업은 partial build를 더 미세하게 파는 것보다, 지금 실험 코드를 정식 viewport dynamic renderer로 정리하는 게 맞습니다. Runtime index partial이나 audio schedule partial은 그 다음 병목이 다시 드러날 때 진행하면 됩니다.

> 테스터가 클로드로 8MB 제한을 풀었다고 합니다. 아래는 채팅 내용입니다.
> 8MB 제한이 어디서 걸리는지 소스를 봐야 콘솔로 풀 수 있는지 알 수 있어. 파일 로드 쪽을 확인할게.`readTextFile`(score_file_io.ts)에서 크기를 막는 것 같아. 거길 볼게.여기엔 크기 제한이 없네 (그냥 `file.text()`). 제한은 JSON 파싱 단계(`json_load.ts`)나 `score_limits.ts`에 있을 거야. 확인할게.제한 위치 찾았어 — `loadScoreJson`에서 `getUtf8ByteLength(jsonText) > MAX_SCORE_JSON_BYTES`로 막아. 이게 패치 가능한지(어떻게 바이트를 재는지) `score_limits.ts`를 봐야 해.제한 위치/방식 다 확인했어.
>
> - 한도: **`MAX_SCORE_JSON_BYTES = 8 * 1024 * 1024`** (8 MiB), `loadScoreJson`에서 파싱 전에 검사.
> - 측정: `getUtf8ByteLength` = `new TextEncoder().encode(text).length`.
>
> 이 상수는 window에 노출 안 돼서 직접 못 바꾸지만, **측정에 쓰는 `TextEncoder.prototype.encode`는 전역이라 콘솔에서 가로챌 수 있어.** 큰 입력일 때 길이를 0으로 위장하면 8MB 검사를 통과해.
>
> 근데 먼저 — **더 나은 해결책**이 있어서 변환기 출력을 **압축(minify)** 으로 바꿨어. 지금까지 들여쓰기된 pretty JSON으로 내보내서 용량이 2배 가까이 부풀어 있었거든. 압축하면 보통 40~50% 줄어서 **8MB 안에 들어올 가능성이 커.** (셀 하나가 90바이트 → 52바이트 수준) 변환기 출력을 압축으로 바꿨어 — 이것만으로도 8MB 넘던 게 안으로 들어올 수 있어. 그래도 넘으면 콘솔로 제한을 우회해.
>
> **8MB 제한 우회 콘솔 패치** (JSON Load **누르기 전에** 실행):
>
> ```js
> (() => {
>   const NE = TextEncoder.prototype.encode;
>   TextEncoder.prototype.encode = function (s) {
>     if (typeof s === 'string' && s.length > 4_000_000) return { length: 0 };  // 큰 입력은 크기 0으로 위장
>     return NE.call(this, s);
>   };
>   console.log('SPRED JSON 8MB 제한 우회 ON — 이제 JSON Load 하세요.');
> })();
> ```
>
> 작동 원리: 8MB 검사가 `TextEncoder().encode(text).length`로 크기를 재는데, 큰 문자열일 때 길이를 0으로 돌려줘서 `0 > 8MB = false` → 통과시켜. 그 다음 `JSON.parse`는 전체를 정상 처리해.
>
> **주의 2가지:**
> - 제한만 푼다고 끝이 아니야 — 거대한 악보는 SPRED가 **파싱·렌더링에서 매우 느려지거나 멈출 수** 있어 (앱 자체 경고 "Large scores may lag"). 우회보다 **압축 + 가로 폭/행 수 조절**로 용량을 줄이는 걸 우선해.
> - 새로고침하면 패치는 풀려 (원상복구). 그래서 매 로드 전에 한 번씩 실행하면 돼.
>
> 요약: 먼저 **압축된 새 출력**으로 다시 받아서 8MB 안에 드는지 보고, 그래도 크면 위 패치 쓰고 로드해줘.

네, 분석은 맞습니다. 다만 중요한 점은 이겁니다.

**8MB 제한은 보안 장벽이 아니라 안정성 가드입니다.**
GitHub Pages 정적 앱이라 사용자가 자기 브라우저 콘솔에서 전역 객체를 패치하면, 어떤 클라이언트 제한도 원천적으로 막을 수 없습니다. 서버가 있는 서비스라면 서버 측 제한이 진짜 장벽인데, 지금 앱은 서버가 없으니 “사용자 본인 브라우저를 멈추지 않게 하는 안전장치”에 가깝습니다.

클로드가 제안한 우회는 실제로 통할 가능성이 큽니다. 현재 제한이 `TextEncoder().encode(text).length`에 의존하므로 `TextEncoder.prototype.encode`를 바꾸면 크기 검사만 속일 수 있습니다.

다만 저 우회 사용은 권장하기 어렵습니다.

- `JSON.parse`가 메인 스레드를 오래 점유할 수 있음
- parser/analyzer/render/audio schedule 생성이 한 번에 큰 부하를 받을 수 있음
- 브라우저 탭이 멈추거나 메모리 사용량이 급증할 수 있음
- 제한을 통과해도 정상 편집 성능을 보장하지 못함

변환기 출력을 pretty JSON에서 minified JSON으로 바꾼 것은 좋은 방향입니다. 이건 우회가 아니라 실제 파일 크기를 줄이는 것이므로 우선 적용할 만합니다.

앱 쪽에서 할 수 있는 개선은 두 단계입니다.

1. `TextEncoder` 의존을 없애고 직접 UTF-8 byte 길이를 계산
   이렇게 하면 저 특정 콘솔 패치는 막을 수 있습니다.

2. 파일 로드 직후 `file.size`도 먼저 확인
   큰 파일을 `file.text()`로 읽기 전에 차단할 수 있어 브라우저 부담이 줄어듭니다.

그래도 콘솔을 쓰는 사용자가 악의적으로 우회하는 것까지 막을 수는 없습니다. 목표는 “실수로 너무 큰 파일을 넣는 일반 사용자를 보호”하는 쪽입니다.

제 판단은 이렇습니다. **8MB 제한을 그냥 풀지는 않는 게 좋습니다.** 대신 minified JSON을 권장하고, 앱 제한 검사는 `TextEncoder` 패치에 덜 취약하게 바꾸는 게 맞습니다. 그리고 더 큰 악보를 공식 지원하려면 Web Worker 파싱, 단계별 로딩, base/grid tile rendering까지 간 뒤에 제한 상향을 검토하는 게 안전합니다.

> 큰 파일을 넣어서 프로그램을 뻗게 만들어도 정적 웹 앱이니까 사용자 본인만 손해겠죠.
> 저희 쪽에서 제한을 더 단단하게 걸 필요는 없다고 봐도 되나요?

네. **보안 관점에서는 더 단단하게 걸 필요가 크지 않습니다.** 정적 웹 앱이라 큰 파일로 앱을 멈추게 해도 피해 범위는 기본적으로 그 사용자의 브라우저 탭입니다. 서버 비용, DB 손상, 다른 사용자 영향, 계정 탈취 같은 쪽으로 번지지 않습니다.

다만 **제품 안정성 관점의 제한은 유지하는 게 좋습니다.** 이유는 보안이 아니라 사용자 경험입니다.

- 실수로 큰 파일을 넣었을 때 탭이 멈추는 걸 줄임
- “앱이 고장났다”는 피드백을 줄임
- 공개 테스트 중 재현 어려운 성능 불만을 줄임

> JSON에 script를 넣어도 현재는 stringfy로 문자열로만 인식하여 동작하지 않죠?

네, 현재 구조라면 JSON 안에 `<script>...</script>` 같은 문자열을 넣어도 **코드로 실행되지 않고 문자열 데이터로만 취급됩니다.**

이유는 크게 두 가지입니다.

- JSON 로드는 `JSON.parse()`로 값으로만 파싱합니다.
- 화면 표시 쪽은 빠르게 확인한 범위에서 `innerHTML`이 아니라 `textContent`, canvas text drawing 등을 사용합니다.


앱에서는 셀 텍스트/표시 텍스트 후보로 처리될 뿐, 브라우저가 script 태그로 해석하지 않습니다.

단, 이 전제는 계속 지켜야 합니다.

```text
사용자 입력/JSON에서 온 문자열을 innerHTML, insertAdjacentHTML, document.write 등에 넣지 않는다.
```

canvas에 그리거나 `textContent`, `value`, `title`에 넣는 것은 일반적으로 괜찮습니다. 현재 상태에서는 JSON script injection으로 실행될 가능성은 낮습니다.

- Score JSON import: 상한 없음.
  - [score_validate.ts](./regression-code/src/core/score/score_validate.ts:238)에서 `integer` 여부만 확인합니다.
  - `col 0` 필수 전역 셀 검증 때문에 `0` 이하 값은 사실상 실패하지만, `1_000_000` 같은 큰 양수는 막지 않습니다.
- Expand UI: HTML에는 `max="960"`이 있습니다.
  - [index.html](./regression-code/index.html:99)

  - 즉 DOM 조작/직접 입력/브라우저 차이에 따라 큰 추가 열이 들어갈 수 있습니다.

**row.height**
- Layout editor UI: `1..500` 제한이 있습니다.
  - [layout_draft.ts](./regression-code/src/app/layout/layout_draft.ts:20)
- Score JSON import: 상한 없음.
  - [score_validate.ts](./regression-code/src/core/score/score_validate.ts:342)에서 `positive integer`만 확인합니다.
- Layout preset import: 상한 없음.
  - [layout_preset.ts](./regression-code/src/app/layout/layout_preset.ts:387)에서 `positive integer`만 확인합니다.
  - 이 값은 draft로 들어간 뒤 apply 시에도 현재 `validateScoreFile()` 기준으로 통과할 수 있습니다.

정리하면 현재 남아 있는 입력점은 다음입니다.

```text
1. Score JSON의 globalLines.columnCount
2. Expand Columns 입력값
3. Score JSON의 layout.rowDefinitions[].height
4. Layout preset JSON의 rowDefinitions[].height
```

특히 `columnCount`가 과도하게 크면 base grid render에서 column line을 전부 순회하므로 브라우저가 쉽게 멈출 수 있습니다. `row.height`가 과도하게 크면 stage height/CSS layout/canvas 크기 계산 쪽에 부담이 갑니다.

따라서 제한을 걸려면 HTML `max`가 아니라 검증/적용 경계에 상수를 두는 게 맞습니다. 예를 들면:

```text
MAX_SCORE_COLUMN_COUNT
MAX_ROW_HEIGHT
MAX_EXPAND_COLUMNS_PER_ACTION
```

그리고 적용 위치는 `score_validate.ts`, `layout_preset.ts`, `app_runtime.ts`가 핵심입니다.
