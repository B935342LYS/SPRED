# 1.39. Range selection, paste preview, Undo 계획 메모

## 1. 목적

이 메모는 edit mode의 영역 선택, 선택 영역 삭제/복사/붙여넣기, 붙여넣기 미리보기, 그리고 다음 단계 Undo / Redo 설계 결정을 기록한다.

기준 문서:

- `docs/2.8-edit-invalidation-and-partial-rebuild-spec.md`
- `docs/2.9-range-selection-edit-spec.md`
- `docs/2.10-undo-redo-edit-history-spec.md`

이 문서는 구현 해설용 메모이다.
저장 포맷, parser/analyzer 공식 계약, audio timing 계약은 active spec 문서를 따른다.

## 2. Range selection 구현

edit mode에서 `Ctrl + left drag`를 영역 선택 gesture로 사용한다.

주요 상태:

- `AppState.rangeSelection`
- `AppState.rangeClipboard`
- `AppState.pastePreview`

`ScoreRangeSelection`은 note/global 중 하나의 row kind만 가진다.
gap row는 실제 bulk 대상에서 제외한다.
다만 화면의 선택 사각형은 선택된 note/global row 범위를 감싸는 하나의 큰 DOM rectangle로 표시한다.
이렇게 하면 내부 계산은 gap row를 제외하면서도 사용자는 하나의 선택 영역으로 인식할 수 있다.

선택 상태와 overlay는 score JSON에 저장하지 않는다.
parser, analyzer, audio 계약도 바꾸지 않는다.

## 3. Bulk delete / copy / paste

선택 영역 삭제는 기존 `ScoreTextEdit[]` batch edit 경로를 사용한다.
note range는 선택 당시의 `activeTrackIds`를 `trackIds`로 저장하고, 삭제/복사 시 그 track 목록을 사용한다.
global range는 col 0 필수 global cell 삭제를 보호한다.

복사/붙여넣기는 app runtime 내부 clipboard로 구현했다.
clipboard에는 실제 존재하는 cell만 저장한다.
빈 cell은 저장하지 않는다.
Ctrl+C 이후에는 range selection을 해제한다.
clipboard와 paste preview는 유지하지만, 영역 선택 모드는 한 번의 복사로 끝낸다.
Ctrl+V 첫 붙여넣기가 성공하면 clipboard와 paste preview를 함께 비운다.
단일 cell edit이나 range delete로 일반 편집 흐름에 들어가도 clipboard와 paste preview를 비워, 이후 pointermove에서 preview가 다시 나타나지 않게 한다.

붙여넣기 row 정책:

- clipboard의 원본 `sourceRowIds`를 유지한다.
- 마우스 y 좌표나 단일 선택 row는 붙여넣기 row 위치를 바꾸지 않는다.
- 현재 layout에 없는 rowId나 rowKind가 맞지 않는 row는 건너뛴다.

붙여넣기 column 정책:

- `pastePreview.anchorCol`
- range selection 시작 column
- 단일 selection column

위 순서로 시작 column을 결정한다.

## 4. Paste preview

`Ctrl+C` 이후 clipboard가 있으면 paste preview를 표시할 수 있다.
마우스가 score stage 위를 움직일 때 pointer x 좌표만으로 column을 계산해 `pastePreview.anchorCol`을 갱신한다.
row hit이 없어도 x 좌표가 score content 안이면 preview column을 정할 수 있다.

preview 갱신은 canvas 재렌더를 호출하지 않는다.
`paste-preview-overlay` DOM layer의 자식 rectangle만 교체해 비용을 작게 유지한다.

시각 정책:

- selection overlay와 별도 DOM layer를 사용한다.
- 실제 clipboard cell footprint만 반투명 사각형으로 표시한다.
- preview rectangle 높이는 renderer의 `CANVAS_METRICS.baseNoteRenderHeight` 기준 21px에 현재 zoom 배율을 곱해 노트 사각형 높이와 맞춘다.
- preview rectangle은 대상 row의 중앙 y 좌표를 기준으로 세로 중앙 정렬한다.
- pointer가 score stage 밖으로 나가면 preview만 숨긴다.
- `Esc`, paste 완료, range delete, 단일 edit, score/load/layout 구조 변경 뒤 clipboard와 preview를 해제한다.

## 5. Undo / Redo 설계 결정

초기 논의에서는 `ScoreFile` snapshot 기반 Undo도 검토했지만, 현재 명세는 cell patch 기반으로 정리했다.

선택한 방식:

```ts
type CellHistoryPatch = {
  selection: ScoreSelection;
  beforeRawText: string | null;
  afterRawText: string | null;
};
```

의미:

- `beforeRawText: null`은 command 전 cell이 없었다는 뜻이다.
- `afterRawText: null`은 command 후 cell이 삭제되었다는 뜻이다.
- undo는 `beforeRawText`를 적용한다.
- redo는 `afterRawText`를 적용한다.

이 방식의 장점:

- 메모리 사용량이 전체 악보 크기가 아니라 변경된 cell 수에 비례한다.
- 단일 edit, drag batch, range delete, range paste에 같은 구조를 쓸 수 있다.
- paste가 기존 cell을 덮어쓴 경우도 `beforeRawText`로 복원할 수 있다.
- 기존 score text edit batch와 partial rebuild 경로를 재사용할 수 있다.

첫 undo 대상은 악보 영역 편집으로 제한한다.
Details metadata edit와 YouTube metadata reload는 undo 대상에서 제외한다.

## 6. 구현상 주의점

cell patch history를 만들 때는 mutation 전에 대상 cell의 기존 rawText를 읽어야 한다.
같은 command 안에서 같은 cell이 여러 번 등장하면 하나의 patch로 병합하고 최종 after 값을 사용한다.
`rawText: ""` edit은 삭제 적용 방식이지만, history에는 `null`로 정규화해 의미를 명확히 한다.

drag edit은 현재 RAF batch 단위로 score edit이 적용된다.
첫 구현에서는 batch 단위 undo group을 허용하고, 후속으로 pointerdown~pointerup 전체를 하나의 transaction으로 묶는 개선을 둔다.
