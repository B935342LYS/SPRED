# 2.6. 트랙 레이어 UI / Renderer / Playback 명세

## 1. 문서 목적

이 문서는 `회귀 코드`의 첫 track layer 확장 범위를 정의한다.

대상은 이미 score JSON에 존재하는 고정 track인 `basic`, `optional`, `extra`이다.
이 문서는 track 저장 포맷을 바꾸지 않고, 활성 track 선택과 활성 track 대상 편집/표시/재생을 연결하는 1차 구현을 다룬다.

관련 문서:

- `docs/1.3-score-json-format.md`
- `docs/1.7-analyzer-event-list-spec.md`
- `docs/1.8-parser-analyzer-pipeline-spec.md`
- `docs/2.0-ui-mvp-spec.md`
- `docs/2.1-canvas-renderer-module-spec.md`
- `docs/2.2-ui-state-edit-mode-spec.md`
- `docs/2.3-audio-playback-module-spec.md`

## 2. 기본 원칙

- track은 저장 포맷상 `basic`, `optional`, `extra` 3개가 모두 존재해야 한다.
- parser와 analyzer는 track 간 이벤트 결합을 하지 않는다.
- track layer 기능은 analyzer 결과를 다시 해석하지 않고, renderer/audio/app 계층에서 활성 track만 소비하는 방식으로 구현한다.
- 첫 구현에서는 편집 대상, 화면 표시 대상, 재생 대상을 분리하지 않는다.
- active track은 편집 대상이고, 재생 대상이며, renderer에서 정상 opacity로 표시되는 track을 뜻한다.
- inactive track은 편집/재생 대상은 아니지만, renderer에서 반투명하게 표시한다.
- 첫 구현에서는 track 추가/삭제, track 이름 변경, 사용자 정의 track 수 확장을 하지 않는다.

## 3. 상태 모델

App 상태에는 다음 개념을 둔다.

```ts
type TrackLayerState = {
  activeTrackIds: TrackId[];
};
```

기존 단일 `activeTrackId`는 복수 선택인 `activeTrackIds`로 확장한다.

초기값:

```text
activeTrackIds = ["basic"]
```

사용자가 optional 또는 extra를 켜면 해당 track은 이미 존재하는 빈 track에 cell을 추가하거나, 이미 존재하는 cell을 정상 opacity로 표시하고 재생한다.
fallback track 생성 정책은 두지 않는다.

`activeTrackIds`는 0개 이상을 허용한다.
빈 배열은 모든 track이 inactive인 상태를 뜻한다.

```text
activeTrackIds = []
```

이 상태에서는 note row 편집 대상과 audio schedule 대상이 없다.
다만 renderer는 track 데이터를 완전히 숨기지 않고, 모든 track을 inactive alpha로 표시한다.
Global row는 track에 속하지 않으므로 `activeTrackIds`가 비어 있어도 기존처럼 편집할 수 있다.

## 4. UI 범위

첫 구현 UI는 다음 조작을 제공한다.

- track active on/off
- 현재 active track 목록을 상단 메뉴 영역의 tracks 패널 안에 표시

권장 UI:

- 상단 메뉴 영역의 tracks 패널 안에 track별 row 또는 segmented control을 둔다.
- 각 track에 active toggle을 둔다.
- `basic`, `optional`, `extra`는 고정 순서로 표시한다.
- score status line이나 edit panel에 active track 목록을 중복 표시하지 않아도 된다.

첫 구현에서 제외한다.

- solo button 또는 유사한 단독 활성화 shortcut
- track별 volume slider
- track별 색상 편집
- track 이름 변경
- 사용자 정의 track 추가/삭제

Solo button은 특정 track 하나만 남기고 다른 track을 임시로 inactive 처리하는 단독 활성화 shortcut을 뜻한다.
현재 1차 구현에서는 track active toggle만 제공하고 solo 개념은 두지 않는다.

Track별 volume과 색상 편집도 현재 범위에 포함하지 않는다.
색상은 `renderer/canvas_note_colors.ts`와 renderer theme에 준비된 기본 set만 사용한다.

## 5. Edit 동작

Note row edit input은 항상 모든 `activeTrackIds`의 같은 note cell 좌표에 적용한다.

```text
pointer hit
-> rowId / col 계산
-> activeTrackIds 각각에 대해 ScoreSelection(trackId, rowId, col) 생성
-> ScoreTextEdit[] batch 생성
-> edit_apply batch
-> runtime rebuild
-> render
```

`activeTrackIds`가 비어 있으면 note row edit command를 생성하지 않는다.
이 경우 score mutation과 runtime rebuild를 실행하지 않고, UI 상태 메시지로 활성 track이 없음을 알린다.

Global row edit input은 track에 속하지 않으므로 기존처럼 1회만 적용한다.
`activeTrackIds`가 비어 있어도 global row edit은 허용한다.

우클릭 삭제 또는 eraser 삭제도 모든 `activeTrackIds`의 같은 note cell 좌표에 batch 적용한다.

같은 cell 반복 클릭 cycle은 track별 기존 rawText를 따로 비교하지 않는다.
사용자의 한 번의 action이 만든 rawText를 모든 active track에 동일하게 쓴다.

예:

```text
첫 클릭      -> 모든 active track에 현재 입력 rawText
반복 클릭 1 -> 모든 active track에 "-"
반복 클릭 2 -> 모든 active track에 "~"
반복 클릭 3 -> 모든 active track에 현재 입력 rawText
```

이 방식은 각 track의 기존 값이 달라도 사용자 action의 결과를 예측 가능하게 만든다.
여러 active track을 대상으로 한 note edit은 active track들의 같은 좌표 cell을 동일 rawText로 덮어쓰는 destructive overwrite 동작으로 취급한다.

## 6. Renderer 정책

Renderer는 `AnalysisResult.trackResults`의 모든 track을 canvas item으로 변환할 수 있다.
다만 `activeTrackIds`에 포함된 track은 정상 opacity로, 포함되지 않은 inactive track은 낮은 opacity로 그린다.
`activeTrackIds`가 비어 있으면 모든 track을 inactive track으로 취급하여 낮은 opacity로 그린다.

첫 구현에서는 item builder 또는 draw 직전 단계에서 각 item에 active 여부를 반영하는 것을 권장한다.
그 이유는 note, mute text, gliss, tuplet label 등 복합 표시가 같은 track opacity 정책을 공유해야 하기 때문이다.

겹침 표시 순서:

```text
top    basic
middle optional
bottom extra
```

Canvas는 나중에 그린 item이 이전 item을 덮는다.
따라서 같은 위치에 note rectangle이 겹치면 `basic`이 가장 위에 보이고, `optional`이 그 아래, `extra`가 가장 아래에 놓인다.

이 순서는 legacy `이세계 코드`의 `trackPriority_()` 흐름을 참고한 것이다.
Legacy는 `basic = 0`, `optional = 1`, `extra = 2` priority를 두고 track별 note를 정렬했다.
`회귀 코드`에서는 사용자 확인 기준에 맞춰 화면 위쪽 우선순위를 `basic > optional > extra`로 고정한다.
Canvas draw order는 이와 반대로 `extra -> optional -> basic` 순서가 된다.

Track별 기본 시각 표현:

| track | 기본 표현 |
| --- | --- |
| `basic` | 기존 pitch class 색상 |
| `optional` | optional pitch class 색상 |
| `extra` | 검은 note fill, 밝은 stroke/text |

Inactive track은 legacy처럼 반투명하게 표시한다.
첫 구현 기준 inactive alpha는 `0.10`으로 둔다.
이 값은 legacy `INACTIVE_TRACK_ALPHA = 0.10` 흐름을 참고한 초기값이며, 실제 시인성 테스트 후 조정할 수 있다.

## 7. Marker와 복합 표시 순서

Active/inactive opacity 정책은 note rectangle뿐 아니라 해당 track에서 나온 복합 표시에도 적용한다.

포함 대상:

- note rectangle
- mute text
- gliss line
- gliss anchor marker
- vib sine wave
- trem chop line
- tuplet container
- tuplet slot label
- tuplet head label

Draw pass의 큰 순서는 기존 renderer 구조를 유지한다.

```text
base grid
-> global timeline guide / beat / bar / BPM marker
-> note rectangles
-> gliss / tuplet / vib / trem 등 marker와 effect
-> note text / mute text / global rawText
```

같은 draw pass 안에서 track끼리 겹치면 `extra -> optional -> basic` 순서로 그린다.
결과적으로 화면에서는 `basic`이 가장 위에 보인다.

## 8. Playback 정책

Audio schedule 생성은 `activeTrackIds`에 포함된 track만 소비한다.

```text
AnalysisResult.trackResults
-> activeTrackIds filtering
-> AudioScheduleEvent[]
-> scheduler
-> backend
```

첫 구현의 track별 gain은 모두 `1.0`으로 둔다.

```text
basic gain = 1.0
optional gain = 1.0
extra gain = 1.0
```

이는 legacy `이세계 코드`의 `TRACK_GAIN_BASIC`, `TRACK_GAIN_OPTIONAL`, `TRACK_GAIN_EXTRA`가 모두 `1.0`인 흐름을 참고한 것이다.

Inactive track은 audio schedule event를 만들지 않는다.
즉 inactive track event를 schedule에 넣고 volume만 `0`으로 두는 방식은 첫 구현에서 사용하지 않는다.
`activeTrackIds`가 비어 있으면 audio schedule은 빈 event list가 된다.
이 상태에서는 playback button을 비활성화하거나, playback 시작 요청을 무음/no-op으로 처리한다.

이 판단의 이유는 다음과 같다.

- 현재 scheduler/backend 구조에서는 schedule에 들어간 event가 lookahead 범위에 들어오면 실제 oscillator/gain node 예약으로 이어진다.
- volume `0` event를 많이 포함하면 전환 속도보다 event queue 크기와 예약 비용이 먼저 증가한다.
- 이미 예약된 무음 oscillator를 active 전환 시점에 자연스럽게 되살리려면 track별 gain bus나 dynamic mute automation이 필요하며, 이는 현재 oscillator backend 범위를 넓힌다.
- active 전환 속도 병목이 확인되기 전에는 `activeTrackIds` 변경 시 schedule을 다시 만드는 편이 단순하다.

후속으로 재생 중 track toggle의 즉시성이 중요해지면 다음 구조를 검토한다.

- 전체 track schedule은 유지하되 scheduler가 active track만 예약한다.
- 또는 track별 gain bus를 두고 active/inactive를 gain automation으로 전환한다.

여러 track의 소리가 같은 시점에 겹치면 첫 구현에서는 모두 재생한다.
특정 track을 자동으로 ducking하거나 priority로 음을 끊지 않는다.

다만 Web Audio 출력 clipping을 줄이기 위해 다음 중 하나를 적용한다.

- 현재 oscillator backend의 master volume 기본값을 유지한다.
- 후속 검증에서 clipping이 확인되면 master compressor 또는 limiter를 audio backend에 추가한다.

Legacy에는 master gain 뒤에 dynamics compressor를 연결하는 흐름이 있었다.
`회귀 코드` 첫 구현에서는 track layer 자체의 필수 기능으로 compressor를 도입하지 않고, audio 검증 단계의 후보로 둔다.

## 9. 상태 변경과 Playback Reset

다음 상태 변경은 playback schedule을 다시 만들어야 한다.

- `activeTrackIds` 변경
- score edit으로 analyzer result 변경
- layout apply로 runtime artifact 변경
- score load

`activeTrackIds` 변경은 score rawText, parser result, analyzer result를 바꾸지 않는다.
따라서 track toggle 처리에서는 full parse/full analysis를 다시 실행하지 않는다.
첫 구현은 다음 흐름을 사용한다.

```text
activeTrackIds 변경
-> app state 갱신
-> renderer opacity / draw order 반영
-> audio schedule 재생성
-> playback runtime reset
```

첫 구현에서는 playback state가 `stopped` 또는 `paused`일 때 active track toggle을 허용한다.
`playing` 상태에서는 active track toggle을 비활성화한다.
Edit mode에서도 playback state가 `stopped` 또는 `paused`이면 active track toggle을 조작할 수 있다.

Paused 상태에서 track toggle을 조작하면 `activeTrackIds` 변경 후 audio schedule을 재생성하되, 기존 paused score time을 유지하고 resume 시 새 active track 조합으로 재생한다.
이를 위해 다음 조건을 만족한다.

- 현재 score time 보존
- schedule rebuild 후 playback controller의 paused position 유지
- renderer opacity 즉시 반영
- resume 시 새 `activeTrackIds` 기준 schedule만 재생
- 이미 예약된 audio node가 남지 않도록 pause 시 scheduler/backend 정리 보장

## 10. Legacy 참고 요약

참고한 legacy 파일:

- `0. 이세계 코드 (legacy)/parts_analyze_core.html`
- `0. 이세계 코드 (legacy)/parts_analyze_renderNote_core.html`
- `0. 이세계 코드 (legacy)/parts_analyze_renderNote_orchestrator.html`
- `0. 이세계 코드 (legacy)/parts_analyze_audio.html`

참고한 흐름:

- `trackIdForRowIdx_()`는 special row를 `optional` 또는 `extra` track으로 분류하고, 나머지는 `basic`으로 둔다.
- `trackPriority_()`는 `basic = 0`, `optional = 1`, `extra = 2` 우선순위를 둔다.
- note render는 track priority를 기준으로 순서를 정한다.
- `trackGain_()`은 세 track 모두 gain `1.0`을 반환한다.
- render는 inactive track도 낮은 alpha로 표시한다.
- audio 재생은 inactive track을 건너뛰고 active track event를 모두 예약한다.
- master chain에는 master gain과 expression gain, 가능하면 dynamics compressor가 연결된다.

`회귀 코드`는 legacy code를 직접 이식하지 않는다.
위 흐름은 정책 판단의 참고 자료로만 사용한다.

## 11. 구현 후보 파일

예상 수정 파일:

- `regression-code/src/app/app_types.ts`
  - track layer 상태 확장
- `regression-code/src/app/app_ui_sync.ts`
  - track UI 상태 동기화
- `regression-code/src/app/edit/edit_controller.ts`
  - active track 전체 대상 batch edit 반영
- `regression-code/src/track/track_control.ts`
  - track 순서, inactive alpha, track gain 등 renderer/audio/app이 공유하는 순수 정책 helper
- `regression-code/src/renderer/canvas_item_builder.ts`
  - active/inactive 표시 상태와 draw order 정렬 반영
- `regression-code/src/renderer/canvas_note_renderer.ts`
  - item별 alpha 적용
- `regression-code/src/audio/audio_schedule_builder.ts`
  - active track filtering 유지 및 track gain helper 사용
- `regression-code/src/app/playback/app_playback.ts`
  - `activeTrackIds`를 schedule builder에 전달

`src/track/track_control.ts`는 UI 상태를 소유하지 않는다.
이 모듈은 renderer와 audio에 중복되기 쉬운 track 정책만 제공한다.
첫 구현에서는 track policy가 여러 파일에 분산되지 않도록 다음 순수 값을 이 모듈에 모은다.

```ts
TRACK_IDS
TRACK_UI_ORDER
TRACK_DRAW_ORDER
INACTIVE_TRACK_ALPHA
getTrackRenderAlpha()
getTrackGain()
filterActiveTrackResults()
```

`core/score`는 저장 포맷 중심 계층이므로, renderer opacity나 audio gain 같은 표시/재생 정책을 `core/track` 아래에 두지 않는다.
따라서 renderer/audio 내부에 각각 track 관리 파일을 두는 것보다 순서와 gain 정책이 어긋날 위험이 작고, core 계층에 UI/audio 정책이 섞이는 문제도 피할 수 있다.
또한 `src/track/` 폴더는 후속 부분 갱신에서 track 단위 영향 범위 판단이나 track 단위 실행 정책을 둘 수 있는 위치로 확장할 수 있다.

첫 구현에서는 새 저장 타입을 추가하지 않는다.
Track layer state는 app runtime state로만 관리한다.

## 12. 검증 항목

필수 검증:

- 기본 load 직후 `basic`만 표시/재생된다.
- `basic`, `optional`을 active로 켠 뒤 입력하면 두 track의 같은 좌표 cell이 함께 변경된다.
- `basic`, `optional`, `extra`를 active로 켠 뒤 삭제하면 세 track의 같은 좌표 cell이 함께 삭제된다.
- inactive track의 note, gliss, tuplet, mute text가 낮은 opacity로 표시된다.
- inactive track은 audio schedule에 포함되지 않는다.
- 모든 track을 inactive로 만들 수 있다.
- `activeTrackIds`가 비어 있으면 모든 track note가 inactive alpha로 표시된다.
- `activeTrackIds`가 비어 있으면 note row edit은 차단되고 global row edit은 허용된다.
- `activeTrackIds`가 비어 있으면 audio schedule은 빈 event list가 된다.
- playback state가 `playing`일 때 active track toggle이 비활성화된다.
- playback이 정지 또는 일시정지된 상태와 edit mode에서는 active track toggle을 조작할 수 있다.
- paused 상태에서 active track toggle 후 resume하면 기존 paused score time에서 새 active track 조합으로 재생된다.
- active track toggle은 full parse/full analysis를 다시 실행하지 않는다.
- 같은 row/col에 basic, optional, extra note가 겹치면 `basic`이 가장 위에 보인다.
- 같은 시점에 여러 track이 켜져 있으면 모두 재생된다.

권장 명령:

```text
npm run typecheck
npm run test:score
npm run test:edit
npm run test:layout
npm run build
```

Track filtering test가 추가되면 별도 `test:track` 또는 기존 renderer/audio test에 포함한다.
