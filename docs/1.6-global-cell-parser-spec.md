# 파서 중간 산출물 데이터 구조 설계 2 : 전역 행

## 1. 문서 목적

- 본 문서는 회귀 코드의 parser가 전역 행 셀 문자열 `rawText`를 해석하여 생성하는 1차 산출물의 구조를 정의한다.
- 이 산출물은 전역 행 문자열의 문법 해석 결과이며, 아직 인접 셀 관계나 시간축 의미가 확정된 상태는 아니다.
- parser 출력은 analyzer가 `timingTimeline`, `dynamicsTimeline`을 구성하는 입력으로 사용한다.
- validator는 parser 출력이 아니라 JSON 문서 구조와 참조 무결성을 주로 검증한다.
- 본 문서는 `1.3-score-json-format.md`, `1.4-note-string-spec.md`를 직접 이어받으며, deleted transitional document였던 옛 `1.6`의 핵심 내용은 현재 `1.3`, `1.7`, `1.8`에 흡수된 상태를 전제로 한다.

## 2. 전역 행 문자열 문법

### 2.1. 문법 및 제약조건

전역 행 문자열의 문법은 다음과 같다.

```text
<global_text> ::= number
<linear_global_text> ::= <global_text> [<ramp_token>]

<ramp_token> := "<" | ">" | "><"
```

세부 규칙은 다음과 같다.

- 오직 bpm과 dynamics만 linear_text를 사용할 수 있다.
- beatsPerBar와 stepsPerBeat는 자연수만 허용한다.
- `><`는 두 글자 조합이지만 하나의 선형 변화 토큰으로 취급한다.
- 선형 변화 토큰은 문자열 끝에만 올 수 있다.
- 선형 변화 토큰은 최대 1개만 허용한다.
- 숫자 뒤에 다른 문자나 공백이 섞여 있으면 문법 오류이다.
- 빈 문자열은 허용하지 않는다.
- 초기 버전에서는 앞뒤 공백을 자동 제거하지 않고, 입력 문자열 그대로를 기준으로 판정한다.

### 2.2. 선형 변화 토큰 의미

- `none`
  - 상수값 셀이다.
  - 해당 열부터 값이 즉시 적용된다.
- `start`
  - 현재 값에서 시작하는 선형 변화의 시작점 후보이다.
- `end`
  - 현재 값에서 끝나는 선형 변화의 종료점 후보이다.
- `endStart`
  - 이전 선형 변화를 현재 값에서 끝내고, 같은 열에서 새로운 선형 변화를 다시 시작하는 경계점 후보이다.

중요:

- 위 의미는 어디까지나 셀 단위의 “후보 의미”이다.
- 실제로 어떤 셀과 연결되는지는 analyzer가 열 순서 문맥을 보고 결정한다.

## 3. parser의 위치와 책임

회귀 코드의 데이터 흐름은 아래 순서를 따른다.

1. JSON 파일 로드
2. 문서 구조 검증
3. 셀 문자열 파싱
4. 셀 간 관계 분석
5. 렌더/오디오용 이벤트 생성

각 단계의 책임은 다음과 같이 고정한다.

### 3.1. validator가 책임지는 것

- JSON 구조 검증
- 필수 필드 누락 검증
- `rowId`, `trackId`, `stringId` 참조 무결성 검증
- 같은 좌표 중복 검증
- `globalLines.cells` 좌표 중복 검증
- `gap` 행에 셀이 저장된 경우 검증
- 전역 행 row가 실제 `global` 타입인지 검증

### 3.2. parser가 책임지는 것

- 전역 행 문자열에서 숫자 부분과 선형 변화 토큰 판정
- 숫자 형식 검증
- `rowId`에 대응하는 전역 행 종류를 기준으로 허용 범위 검증
- 셀 단독으로 판정 가능한 문법 오류 검출
- analyzer가 소비할 `ParsedGlobalCell` 생성

### 3.3. analyzer가 책임지는 것

- 같은 kind의 전역 셀을 열 순서로 연결하여 구간 의미 확정
- `<`, `>`, `><`의 연결 관계 해석
- BPM, 박자, step, dynamics 변화를 시간 세그먼트로 정규화
- 충돌, 모순 구간 진단 생성
- renderer/audio가 소비할 `timingTimeline`, `dynamicsTimeline` 생성

즉 parser는 전역 셀 하나의 문자열만 해석하며, 열 간 관계와 실제 시간 의미는 analyzer로 넘긴다.

## 4. 설계 원칙

- 전역 행도 note 행과 마찬가지로 JSON에는 `rawText`만 저장한다.
- parser 산출물은 JSON으로 직렬화 가능한 순수 데이터 구조여야 한다.
- parser는 셀 단위 처리만 수행한다.
- parser는 손상된 문자열에 대해 오류 복구를 시도하지 않고 invalid 결과를 반환한다.
- parser 출력은 TypeScript에서 안전하게 다루기 쉬운 discriminated union 구조를 사용한다.
- 전역 행 문자열은 단순한 형태를 가지더라도, 런타임에서는 반드시 parsed 구조를 거쳐 analyzer로 전달한다.

## 5. 설계 해설

### 5.1. 왜 전역 행도 숫자 필드가 아니라 rawText로 저장하는가?

이전 설계에서는 전역 이벤트가 이미 해석된 형태인 `value`, `ramp` 구조로 저장되는 방향을 상정하였다. 그러나 이는 저장 구조와 런타임 해석 구조를 섞는 문제가 있었다.

전역 행도 `rawText` 기반으로 저장해야 하는 이유는 다음과 같다.

1) 저장 구조와 해석 구조를 분리하기 위해
- 저장 포맷은 사용자 입력을 보존하는 것이 우선이다.
- 런타임 산출물은 parser와 analyzer가 소비하기 좋은 형태로 재구성하면 된다.
- 둘을 분리해야 설계 변경이 생겨도 저장 포맷이 덜 흔들린다.

2) 원본 문자열 보존을 위해
- `120<`, `80>`, `100><` 같은 전역 문자열은 숫자값과 선형 변화 토큰이 결합된 입력이다.
- 이를 JSON에 `value: 120`, `ramp: "start"`로만 저장하면 사용자가 실제로 입력한 문자열 정보가 사라진다.
- 디버깅, UI 복원, 재직렬화, 문법 테스트를 위해 원본 문자열이 필요하다.

3) note 행과 저장 규칙을 통일하기 위해
- note 행은 `rawText`를 저장하고 parser가 중간 산출물을 만든다.
- 전역 행만 예외적으로 해석 결과를 저장하면 문서 전체의 구조가 비대칭이 된다.
- 전역 행 역시 “행 + 열 + 문자열 셀” 구조로 두는 편이 전체 파이프라인과 더 잘 맞는다.

### 5.2. 왜 전역 행 parser를 note 행 parser와 별도 문서로 분리하는가?

전역 행 문자열 자체는 숫자와 선형 변화 토큰만 결합된 단순한 구조이다. 그러나 다음 이유로 note 행 parser 문서에 흡수하기보다는 별도 문서로 분리하는 편이 적절하다.

1) 책임 경계가 다르기 때문
- note 행 parser는 hold, gliss, trem, tuplet 등 다양한 토큰을 해석한다.
- 전역 행 parser는 숫자와 선형 변화 토큰만 다루지만, `rowId`가 가리키는 전역 행 종류별 값 범위를 반드시 함께 고려해야 한다.
- 두 구조는 겉보기 난이도는 달라도 parser가 받는 문맥과 출력의 사용처가 다르다.

2) analyzer로 넘기는 의미가 다르기 때문
- note 행 parser 출력은 주로 발음 이벤트 분석의 입력이 된다.
- 전역 행 parser 출력은 timing/dynamics timeline 분석의 입력이 된다.
- 따라서 중간 산출물의 용도와 후속 모듈 경계가 분리되어 있는 편이 문서상 명확하다.

3) 이후 확장에 대비하기 위해
- 초기 문법은 단순하더라도, 전역 행에 새로운 제어 토큰이 추가될 수 있다.
- 별도 문서로 유지하면 향후 확장 시 note 행 문서의 구조를 흔들지 않고 발전시킬 수 있다.

## 6. ParsedGlobalCell이 담아야 할 정보

전역 행 parser의 출력은 단순히 숫자를 읽는 데서 끝나지 않는다. analyzer가 열 순서로 이어 붙여 time segment를 만들 수 있도록, 셀 단위에서 확정 가능한 최소한의 정보를 함께 담아야 한다.

현재 설계에서 `ParsedGlobalCell`이 담아야 할 핵심 정보는 다음과 같다.

### 6.1. 원본 문자열 `rawText`

- JSON 파일에 저장된 셀의 원본 문자열을 그대로 유지해야 한다.
- 이는 디버깅, 오류 보고, UI 복원, 재직렬화에 필요하다.

### 6.2. 숫자값 `value`

- parser는 문자열 숫자를 실제 number 값으로 변환한다.
- analyzer는 이후 이 값을 바로 사용하여 세그먼트를 구성한다.
- 단, 의미 해석은 analyzer가 하더라도 숫자 형식 검증은 parser 단계에서 끝내는 편이 적절하다.

### 6.3. 선형 변화 토큰 정보

- `bpm`, `dynamics` 셀은 문자열 끝에 `<`, `>`, `><` 중 어떤 토큰이 있는지 기록해야 한다.
- 이 정보는 셀 단독으로 확정 가능하므로 parser 산출물에 포함하는 것이 자연스럽다.
- 반면 `beatsPerBar`, `stepsPerBeat` 셀은 선형 변화 토큰을 가질 수 없다.
- 실제 연결 관계와 세그먼트 성립 여부는 analyzer가 판단한다.

### 6.4. 오류 정보

- 문자열이 문법적으로 잘못된 경우, 단순히 실패 여부만이 아니라 어떤 이유로 잘못되었는지를 함께 기록해야 한다.
- 따라서 invalid 결과에는 오류 코드와 가능하면 오류 위치 정보도 포함하는 것이 바람직하다.

### 6.5. 셀 단위 정보와 셀 간 관계 정보의 경계

- 위 정보들은 모두 “전역 셀 하나만 보고 판단 가능한 정보”에 해당한다.
- 반대로 램프 시작과 끝의 연결 성립, 세그먼트 길이, 충돌 판정, 시간 맵 구축은 문서 전체 문맥이 필요하다.
- 이러한 정보는 `ParsedGlobalCell`에 직접 넣지 않고 analyzer 단계에서 처리한다.

즉 `ParsedGlobalCell`은 전역 셀 하나만 보고 확정 가능한 정보만 담고, 열 간 관계나 시간축 의미는 담지 않는 구조를 목표로 한다.

## 7. ParsedGlobalCell 최상위 구조

```ts
type ParsedGlobalCell =
  | ParsedLinearGlobalCell
  | ParsedInstantGlobalCell
  | ParsedGlobalInvalidCell;

type ParsedGlobalCellKind =
  | "linearGlobalValue"
  | "instantGlobalValue"
  | "invalid";

type ParsedGlobalCellBase = {
  kind: ParsedGlobalCellKind;
  rawText: string;
};
```

필드 설명:
- `kind`
  - 1차 파싱 결과로 알 수 있는 셀의 종류이다.
  - 현재 전역 행은 선형 변화 가능 셀, 즉시 적용 셀, invalid 셀의 3분기만 둔다.
- `rawText`
  - 저장된 원본 문자열이다.
  - 재직렬화와 디버깅의 기준이 된다.

## 8. 파싱 오류 구조

전역 행은 note 행보다 문법이 단순하지만, 오류 원인을 구분할 수 있는 구조는 동일하게 필요하다.

```ts
type ParsedGlobalInvalidCell = ParsedGlobalCellBase & {
  kind: "invalid";
  error: GlobalParseError;
};

type GlobalParseError = {
  code: GlobalParseErrorCode;
  charIndex: number | null;
  message?: string;
};

type GlobalParseErrorCode =
  | "empty_global"
  | "invalid_number"
  | "invalid_ramp_token"
  | "more_than_expected"
  | "invalid_bpm_range"
  | "invalid_beats_per_bar_range"
  | "invalid_steps_per_beat_range"
  | "invalid_dynamics_range";
```

오류 구조는 역할에 따라 3계층으로 나눈다.

1. `ParsedGlobalInvalidCell`
   - 해당 셀이 invalid 타입임을 명시한다.
2. `GlobalParseError`
   - 구체적인 오류 정보이다.
3. `GlobalParseErrorCode`
   - 오류 종류를 제한된 문자열 목록으로 고정한다.

필드 설명:
- `code`
  - 발생한 문법 또는 셀 단독 검증 오류의 종류이다.
- `charIndex`
  - 해당 문자열에서 오류가 일어난 위치이다.
  - 위치를 특정하기 어렵다면 null을 허용한다.
- `message`
  - 사람이 읽는 보조 설명 텍스트이다.
  - 디버깅과 테스트에 사용한다.

## 9. 종류별 ParsedGlobalCell 구조

전역 행 종류는 아래와 같이 `GlobalKind`로 고정한다.

```ts
type GlobalKind =
  | "bpm"
  | "beatsPerBar"
  | "stepsPerBeat"
  | "dynamics";
```

이 타입은 전역 행이 어떤 규칙으로 파싱되고 이후 analyzer에서 어떤 스트림으로 해석될지를 구분하는 기준이다.

```ts
type ParsedLinearGlobalCell = ParsedGlobalCellBase & {
  kind: "linearGlobalValue";
  globalKind: "bpm" | "dynamics";
  value: number;
  ramp: ParsedGlobalRamp;
};

type ParsedInstantGlobalCell = ParsedGlobalCellBase & {
  kind: "instantGlobalValue";
  globalKind: "beatsPerBar" | "stepsPerBeat";
  value: number;
};

type ParsedGlobalRamp =
  | "none"
  | "start"
  | "end"
  | "endStart";
```

### 9.1. 선형 변화 가능 셀

`ParsedLinearGlobalCell`은 `bpm`, `dynamics` 행에 사용한다.

필드 설명:
- `globalKind`
  - 선형 변화 토큰을 허용하는 전역 행 종류이다.
- `value`
  - 숫자 변환이 끝난 실제 값이다.
- `ramp`
  - 문자열 끝에 붙은 선형 변화 토큰을 나타낸다.
  - `<`는 `start`, `>`는 `end`, `><`는 `endStart`, 토큰이 없으면 `none`이다.

### 9.2. 즉시 적용 셀

`ParsedInstantGlobalCell`은 `beatsPerBar`, `stepsPerBeat` 행에 사용한다. 이 행들은 선형 변화가 불가능하며 항상 값이 즉시 적용된다.

필드 설명:
- `globalKind`
  - 선형 변화 토큰을 허용하지 않는 전역 행 종류이다.
- `value`
  - 숫자 변환이 끝난 실제 값이다.
- 즉시 적용 셀은 `ramp` 필드를 가지지 않는다.

## 10. parser 입력 구조

전역 행 parser는 셀 좌표 단위 입력을 유지하되, 숫자 검증 규칙은 `rowId`에 대응하는 전역 행 정의에서 유도해야 한다.
같은 문자열이라도 `rowId`가 어떤 전역 행을 가리키는지에 따라 허용 범위와 숫자 형식이 달라질 수 있기 때문이다.
즉 parser 입력에는 `kind`를 중복해서 넣지 않고, `rowId`를 기준으로 문맥에서 `kind`를 유도하는 편이 더 정합적이다.

```ts
type GlobalCellParserInput = {
  rowId: string;
  col: number;
  rawText: string;
};

type GlobalCellParserContext = {
  rowById: Map<string, RowDefinition>;
};
```

필드 설명:
- `rowId`
  - 어떤 전역 행 셀인지 식별하기 위한 값이다.
- `col`
  - 디버깅, 캐시 키, 에러 추적에 사용되는 열 인덱스이다.
- `rawText`
  - 사용자가 입력한 원본 문자열이다.

`GlobalCellParserContext`
- `rowById`
  - `rowId`에 대응하는 row definition을 찾기 위한 런타임 인덱스이다.
  - parser는 여기서 전역 행 종류를 유도한 뒤 검증 규칙을 결정한다.

## 11. kind별 제약조건

전역 행 parser는 숫자 형식과 값 범위를 `rowId`가 가리키는 전역 행 종류에 따라 다르게 해석한다.

### 11.1. BPM 행

- 양의 실수 또는 양의 정수를 허용한다.
- 0 이하는 허용하지 않는다.
- 추후 구현 단계에서 상한값을 정할 수 있다.
- 예: `120`, `87.5`, `132<`

### 11.2. beatsPerBar 행

- 1 이상의 정수만 허용한다.
- 실수는 허용하지 않는다.
- 추후 구현 단계에서 상한값을 정할 수 있다.
- 선형 변화 토큰을 허용하지 않는다.
- 예: `3`, `4`

### 11.3. stepsPerBeat 행

- 1 이상의 정수만 허용한다.
- 실수는 허용하지 않는다.
- 추후 구현 단계에서 상한값을 정할 수 있다.
- 선형 변화 토큰을 허용하지 않는다.
- 예: `4`, `6`

### 11.4. dynamics 행

- 0 이상의 정수만 허용한다.
- 상한은 `1.3` 문서 기준을 따라 0~150이다.
- 실수는 허용하지 않는다.
- 예: `100`, `80<`, `120>`

## 12. 파싱 절차 수도코드

### 12.1. 선형 변화 토큰 분리 규칙

1. 입력 `rowId`로 row definition을 찾는다.
2. row definition의 `kind`가 bpm 또는 dynamics이면 문자열 끝에서 "><", "<", ">" 순으로 판정한다.
3. row definition의 `kind`가 beatsPerBar, stepsPerBeat이면 선형 변화 토큰을 해석하지 않는다.
4. beatsPerBar, stepsPerBeat에서 문자열 끝에 "><", "<", ">"가 있으면 invalid_ramp_token이다.

### 12.2. 최상위 수도코드

```text
parseGlobalCell(input, context):

  rawText = input.rawText
  row = context.rowById.get(input.rowId)

  if row is missing
    return invalid("more_than_expected", null)

  kind = row.kind

  if rawText.length == 0
    return invalid("empty_global", 0)

  numberText = rawText

  // bpm, dynamics만 선형 변화 토큰을 해석한다.
  if kind == "bpm" or kind == "dynamics"
    ramp = "none"

    // "><"는 두 글자 토큰이므로 가장 먼저 판정한다.
    if rawText endsWith "><"
      ramp = "endStart"
      numberText = rawText without last 2 chars

    else if rawText endsWith "<"
      ramp = "start"
      numberText = rawText without last 1 char

    else if rawText endsWith ">"
      ramp = "end"
      numberText = rawText without last 1 char

  else
    // beatsPerBar, stepsPerBeat는 선형 변화 토큰을 허용하지 않는다.
    if rawText endsWith "><"
      return invalid("invalid_ramp_token", rawText.length - 2)

    if rawText endsWith "<" or rawText endsWith ">"
      return invalid("invalid_ramp_token", rawText.length - 1)

  // 숫자 부분이 비어 있으면 invalid
  if numberText.length == 0
    return invalid("invalid_number", 0)

  // 숫자 뒤에 허용되지 않은 문자가 섞여 있으면 invalid
  // 앞뒤 공백 자동 제거는 하지 않는다.
  if kind == "bpm"
    if numberText is not valid positive decimal number
      return invalid("invalid_number", first invalid char index)

    value = parse decimal number

    if value <= 0
      return invalid("invalid_bpm_range", 0)

    return {
      kind: "linearGlobalValue",
      rawText: rawText,
      globalKind: kind,
      value: value,
      ramp: ramp,
    }

  else if kind == "beatsPerBar"
    if numberText is not valid positive integer
      return invalid("invalid_number", first invalid char index)

    value = parse integer

    if value < 1
      return invalid("invalid_beats_per_bar_range", 0)

    return {
      kind: "instantGlobalValue",
      rawText: rawText,
      globalKind: kind,
      value: value,
    }

  else if kind == "stepsPerBeat"
    if numberText is not valid positive integer
      return invalid("invalid_number", first invalid char index)

    value = parse integer

    if value < 1
      return invalid("invalid_steps_per_beat_range", 0)

    return {
      kind: "instantGlobalValue",
      rawText: rawText,
      globalKind: kind,
      value: value,
    }

  else if kind == "dynamics"
    if numberText is not valid non-negative integer
      return invalid("invalid_number", first invalid char index)

    value = parse integer

    if value < 0 or value > 150
      return invalid("invalid_dynamics_range", 0)

    return {
      kind: "linearGlobalValue",
      rawText: rawText,
      globalKind: kind,
      value: value,
      ramp: ramp,
    }
```

## 13. parser 출력 이후 런타임 사용 방식

전역 행 parser 출력은 독립적으로 끝나지 않는다. 런타임에서는 아래처럼 사용한다.

### 13.1. 셀 단위 캐시

```ts
type ParsedGlobalCellCacheKey = `${string}|${number}`;
```

예:

- `${rowId}|${col}`

용도:

- 전역 셀 편집 후 해당 셀만 재파싱
- 동일 셀의 반복 해석 방지

### 13.2. analyzer 입력 단위

analyzer는 단일 `ParsedGlobalCell`이 아니라 아래 문맥을 함께 받는다.

```ts
type AnalyzerGlobalCellInput = {
  rowId: string;
  kind: GlobalKind;
  col: number;
  parsedCell: ParsedGlobalCell;
};
```

추가로 analyzer는 다음 문서 전역 정보에 접근할 수 있어야 한다.

- `rowById`
- `globalCellsByKindAndCol`
- `globalCellsInColOrder`
- `columnCount`

즉 parser 출력은 완전한 최종 해석 결과가 아니라 analyzer의 재료이다.

## 14. parser가 하지 말아야 할 것

아래 항목은 parser에서 처리하지 않는다.

- `<`와 `>`의 실제 짝 연결 성립 여부 판정
- `><`가 이전 구간 종료와 다음 구간 시작을 동시에 만족하는지 판정
- 선형 변화 구간의 실제 범위 계산
- `tick -> ms` 시간 맵 계산
- dynamics 세그먼트 보간값 계산
- 전역 세그먼트 충돌 판정
- timing/dynamics timeline 생성

이 원칙을 지켜야 parser가 단순하고 안정적으로 유지된다.

## 15. 구현 우선순위

이 문서 기준으로 실제 구현 우선순위는 다음과 같다.

1. `ParsedGlobalCell`, `ParsedLinearGlobalCell`, `ParsedInstantGlobalCell`, `ParsedGlobalRamp`, `GlobalParseErrorCode` 확정
2. kind별 숫자 판정 유틸 설계
3. `parseGlobalCell` 구현
4. 전역 셀 단위 테스트 작성
5. analyzer의 timing/dynamics 해석과 연결

## 16. 결론

회귀 코드의 전역 행 parser는 셀 문자열을 안정적으로 해석하는 작은 모듈이어야 한다. 이를 위해 다음 원칙을 고정한다.

- 전역 행도 JSON에는 `rawText`만 저장한다.
- parser는 셀 문자열을 `ParsedGlobalCell`로 변환한다.
- `bpm`, `dynamics`는 선형 변화 가능 셀로, `beatsPerBar`, `stepsPerBeat`는 즉시 적용 셀로 분리한다.
- 숫자 형식과 row kind별 허용 범위는 parser가 검증한다.
- 열 간 관계와 시간축 의미는 analyzer가 처리한다.
- renderer/audio는 전역 행 `rawText`를 직접 해석하지 않고 analyzer 산출물을 소비한다.

이 구조를 기준으로 구현하면, 전역 행 역시 note 행과 같은 저장 원칙 아래에서 관리하면서도 timing/dynamics 분석에 필요한 경계를 명확히 유지할 수 있다.
