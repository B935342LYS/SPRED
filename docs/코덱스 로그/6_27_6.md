> 마이크 입력과 관련하여 검토해야 하는 보안 이슈가 있습니까?

큰 보안 문제는 현재 구조에서는 보이지 않습니다. 현재 마이크 입력은 브라우저 `getUserMedia()`로 받은 `MediaStream`을 로컬 `AudioContext -> AnalyserNode -> pitchy`에만 연결하고, 업로드/저장/녹음/서버 전송 경로는 없습니다.

검토할 만한 이슈는 이 정도입니다.

1. **마이크가 켜져 있는 상태 범위**
   - [game_binding.ts](./regression-code/src/app/game/game_binding.ts)에서 practice mode 진입 후 exit 전까지 stream이 유지됩니다.
   - `ready`, `playing`, `paused`, `finished`에서도 pitch runtime이 살아 있을 수 있습니다.
   - 보안 취약점이라기보다는 privacy UX 이슈입니다. `finished` 상태에서는 자동으로 마이크를 끄는 것도 고려할 만합니다.

2. **정리 로직**
   - `exitPracticeMode()`에서 `pitchInputRuntime.dispose()`와 `track.stop()`을 호출하므로 정상 종료 경로는 괜찮습니다.
   - 다만 score load, fatal error, 페이지 내부 상태 초기화 같은 다른 경로에서 game mode를 강제로 끄는 코드가 생기면 반드시 같은 정리 함수를 타야 합니다.

3. **XSS가 생기면 마이크 권한도 위험해짐**
   - 마이크 자체 코드보다 더 중요한 건 앱 전체 XSS 방어입니다.
   - 사용자가 불러온 score metadata/title/comment 등을 `innerHTML`로 넣는 코드가 있으면 위험합니다. 현재 보이는 game path는 `textContent` 중심이라 괜찮습니다.
   - 향후 CSP를 걸고 싶다면 `index.html`의 inline `onclick` 같은 부분은 제거해야 합니다.

4. **의존성 공급망**
   - `pitchy`는 클라이언트 번들에 포함되는 npm dependency입니다.
   - 보안상 큰 문제는 아니지만 배포 전 `npm audit` 또는 dependency lock 유지 정도는 권장됩니다.

5. **사용자 고지**
   - Info 문구에 “앱은 마이크 입력을 서버로 전송하거나 저장하지 않는다”는 문장을 명시하면 좋습니다.
   - 브라우저 자체 마이크 표시가 있더라도 앱 UI에서도 practice mode 중 마이크 사용 중임을 계속 보여주는 편이 안전합니다.

현재 우선순위로는 **finished 상태에서 마이크 자동 종료**와 **Info/Rules에 마이크 privacy 문구 추가** 정도가 가장 실용적인 보강입니다.

> 큰 문제가 없으면 일단은 넘어가도 될 듯 합니다. 노래방 기능에 대해 다음 구현 또는 보강 사항이 무엇이 남아있죠?

현재 상태 기준으로 노래방 / practice mode에서 남은 큰 항목은 아래 순서가 적절합니다.

1. **마이크 입력 안정화 추가 튜닝**
   - 이번에 큰 문제는 완화했지만, 실제 악기/마이크/브라우저별로 `clarity`, `RMS`, `unclear pitch` 패턴을 더 봐야 합니다.
   - 필요하면 threshold를 UI 옵션 또는 내부 preset으로 분리할 수 있습니다.

2. **결과 화면 / 종료 흐름 정리**
   - 현재 점수 집계와 result dialog는 들어가 있지만, 마지막 판정 대상이 끝났을 때 finished 전환, restart, exit 흐름을 더 매끄럽게 다듬을 여지가 있습니다.
   - finished 상태에서 마이크를 계속 유지할지, 자동 종료할지도 이 단계에서 결정하면 됩니다.

3. **판정 체감 개선**
   - 지금은 pitch sample 기반 `Perfect / Ok / Bad / Miss`가 중심입니다.
   - 짧은 음에서 판정이 너무 적거나, 긴 음에서 점수가 과하게 누적되는지 확인해야 합니다.
   - sample interval, target lookup window, combo 유지/끊김 기준을 조정할 수 있습니다.

4. **시각 피드백 강화**
   - pitch dot과 판정 텍스트는 연결되어 있습니다.
   - 다음 보강 후보는 pitch dot 색상 변화, 현재 target note 강조, 인식 실패 사유를 더 읽기 쉽게 표시하는 것입니다.
   - 다만 화면이 복잡해지기 쉬워서 실제 테스트 후 필요한 것만 넣는 편이 좋습니다.

5. **YouTube 동시 사용 검증**
   - 노래방 모드는 YouTube sync와 공존하도록 설계되어 있습니다.
   - 실제 배포판에서 YouTube on/off, offset, pause/resume, stop, finished 전환이 score time 기준으로 안정적인지 확인해야 합니다.

6. **효과 판정은 후순위**
   - gliss, vibrato, tremolo 판정은 아직 연구/확장 항목입니다.
   - 당장은 기본 음정 판정이 안정된 뒤에 gliss anchor 정도부터 보는 게 현실적입니다.

제 판단으로 다음 구현 우선순위는 **“finished/restart/exit 흐름 정리 + 실제 배포판 수동 테스트에서 나온 threshold 조정”**입니다. 기능을 더 늘리기 전에 세션 종료와 점수 확정 흐름을 단단하게 만드는 쪽이 좋습니다.

다음 안정화 방향은:
- 고정 시간 hold 제거
- raw pitch는 빠르게 따라가게 유지
- 대신 score target이 있는 경우에만 target 근처 후보를 우선
- detector 후보가 target의 2배/3배 배음처럼 보이면 fundamental 후보로 접기
- dot 표시와 scoring에 서로 다른 smoothing 강도 적용

즉, 바로 CREPE로 갈아타기보다는 먼저 **현재 pitchy + harmonic/target-aware 후보 선택**으로 고치는 게 가장 현실적입니다. CREPE는 그 다음 비교 실험 후보가 맞습니다.

입력 지연 보정 방식은 아래 선택지가 있습니다.

1. **고정 latency offset**
   - 마이크 입력을 실제 score time보다 예를 들어 `80ms ~ 150ms` 늦게 들어온 것으로 보고, 판정 시 `scoreSeconds - inputLatencySeconds` 기준 target을 찾습니다.
   - 구현이 가장 단순하고 부작용도 예측 가능합니다.
   - 단점은 장치/브라우저마다 적정값이 다릅니다.

2. **판정 target window 확장**
   - 현재 시점 하나만 보지 않고, `scoreSeconds - 120ms`부터 `scoreSeconds + 40ms` 사이 target을 후보로 둡니다.
   - 빠른 솔로에서 다음 노트가 먼저 잡히는 문제를 줄일 수 있습니다.
   - 단점은 너무 넓히면 실제로 늦게 친 음도 맞는 것으로 처리됩니다.

3. **이전 note grace window**
   - note가 끝난 직후 `80ms ~ 120ms` 동안은 이전 note도 후보로 유지합니다.
   - “정확한 타이밍에 연주했는데 detector/브라우저 입력이 늦게 들어와 다음 note에서 Miss”가 나는 케이스에 직접적입니다.
   - 빠른 곡용으로 가장 체감이 좋을 가능성이 큽니다.

4. **가장 가까운 target 선택**
   - 판정 시점 근처의 이전/현재/다음 note를 모두 보고, 입력 pitch와 가장 가까운 target을 고릅니다.
   - 현재 multi-track target 선택과 비슷한 방향이라 자연스럽습니다.
   - 단점은 사용자가 다음 음을 너무 빨리 낸 경우도 맞게 처리될 수 있습니다.

제 추천은 **1 + 3 조합**입니다. 기본값으로 `inputLatencyMs = 90` 정도를 두고, note 종료 후 `previousTargetGraceMs = 100` 정도를 유지하는 방식이 가장 단순하고 빠른 솔로 문제에 직접 맞습니다. UI 옵션은 나중에 `Practice latency` 슬라이더로 빼면 됩니다.

> latency를 비전문가 사용자가 측정하여 직접 기입할 수 있을까요?

가능은 하지만, 비전문가에게 “정확히 측정해서 숫자를 입력”하게 하는 방식은 추천하기 어렵습니다. 마이크 입력 지연은 장치, 브라우저, OS 오디오 설정, Bluetooth 여부, 노이즈 처리, 현재 부하에 따라 달라지고, 사용자가 체감으로 `ms` 값을 맞추기 어렵습니다.

현실적인 방식은 셋 중 하나입니다.

1. **간단 보정 슬라이더**
   - UI 문구: `Input Timing`
   - 값: `Early ← 0ms → Late`, 예: `0~200ms`
   - 사용자는 숫자보다 “판정이 늦게 따라오면 오른쪽으로” 식으로 조정합니다.
   - 가장 구현이 쉽고 설명도 쉽습니다.

2. **테스트 박자 보정**
   - 앱이 짧은 beep 또는 클릭음을 내고, 사용자가 그 타이밍에 맞춰 소리를 냅니다.
   - 앱이 target 시각과 인식된 pitch/onset 시각 차이를 평균내서 latency를 추정합니다.
   - 다만 현재는 onset detection이 약하므로 pitch가 안정적으로 잡히는 악기 입력에서는 가능하지만, 정확도는 제한적입니다.

3. **자동 보정 없이 넓은 grace**
   - 사용자 입력 없이 기본 `previous note grace`를 둡니다.
   - 연습 참고용 프로그램이라면 이쪽이 가장 부담이 적습니다.
   - 단점은 아주 엄격한 게임 판정에는 부적합하지만, 현재 목적에는 맞습니다.

제안은 **초기에는 직접 ms 입력을 숨기고**, 내부 기본값으로 `inputLatencyMs = 80~100`, `previousNoteGraceMs = 100` 정도를 적용하는 겁니다. 이후 필요하면 고급 설정에만 `Input Timing` 슬라이더를 넣으면 됩니다.
비전문가에게 숫자를 직접 기입하게 하기보다는 “판정이 늦으면 +, 빠르면 -” 형태가 낫습니다.

> 맞습니다. 리듬게임에 해당 기능이 있는 것을 본 적이 있습니다. + -로 직관적으로 latency를 조절하게 만드는 기능이 확실히 좋을 듯 합니다.

그 방식이 가장 맞습니다. 구현 방향은 이렇게 잡으면 됩니다.

- UI 이름: `Input Timing`
- 표시값: `0 ms`, `+20 ms`, `-20 ms`
- 조작: `-` / `+` 버튼 또는 짧은 range slider
- 범위: `-200ms ~ +200ms`
- step: `10ms`
- 기본값: `+90ms` 또는 `+100ms`

의미는 사용자에게 이렇게 잡는 게 직관적입니다.

- `+` : 내 입력이 늦게 판정되는 것 같을 때 올림
- `-` : 내 입력이 너무 빨리 판정되는 것 같을 때 내림

내부 계산은 판정 target 조회 시점만 바꾸면 됩니다.

```text
judgeScoreSeconds = playbackScoreSeconds - inputTimingOffsetMs / 1000
```

즉 `+100ms`면 “마이크 입력이 실제보다 100ms 늦게 들어온다”고 보고, 100ms 전 악보 위치와 비교합니다.

추천 구현 순서는:
1. `GameModeState` 또는 별도 app runtime option에 `inputTimingOffsetMs` 추가
2. practice panel에 `-`, 값 표시, `+` 추가
3. scoring target lookup에서 보정된 score seconds 사용
4. result/score JSON에는 저장하지 않고 세션 runtime 값으로만 유지
5. 나중에 필요하면 local user preference로 저장

다만 pitch dot까지 같이 보정할지는 분리하는 게 좋습니다. 먼저 **점수 판정에만 latency offset**을 적용하고, dot은 현재 입력 표시로 두는 편이 혼란이 적습니다.

> 이름은 sync면 충분할 것 같습니다.

좋습니다. UI 표기는 `Sync`가 짧고 리듬게임식 보정 느낌도 납니다.

권장 문구:
- 라벨: `Sync`
- 값: `+90 ms`
- 버튼: `-` / `+`
- 범위: `-200 ms` to `+200 ms`
- step: `10 ms`

내부 의미:
- `Sync +100 ms`: 마이크 입력이 늦게 들어오는 것으로 보고, 판정을 100ms 이전 target과 비교
- `Sync -50 ms`: 입력이 너무 앞서 판정되는 것으로 보고, 50ms 이후 target과 비교

다음 구현 때는 practice panel 안에 작게 넣고, 점수 판정 경로에만 먼저 적용하는 방식으로 가면 됩니다.

> 사용자가 지연을 어떻게 느끼게 할지도 관건이네요.
> sync를 맞추기 위해 팝업창을 하나 더 띄우도록 할까요? 일정한 박자의 beep 음을 반복하게 한 다음 입력을 넣으면 시각 효과를 보여주어 어느 정도의 지연이 있는지 감각적으로 느끼게 하는 것입니다.

네, 그 방식이 좋습니다. 비전문가에게 `ms` 숫자를 바로 맞추게 하는 것보다, **Sync calibration 팝업**에서 반복 beep와 시각 피드백으로 “내 입력이 앞/뒤로 얼마나 밀리는지”를 느끼게 하는 쪽이 훨씬 낫습니다.

구성 제안:

- 팝업 이름: `Sync`
- 반복 기준: 4박 또는 8박 beep loop
- UI:
  - `Start`
  - `-10 ms`
  - 현재값 `+90 ms`
  - `+10 ms`
  - `Apply`
  - `Reset`
- 시각 효과:
  - 가운데 기준선 또는 원 표시
  - beep 시점에 기준선 flash
  - 사용자의 pitch가 감지된 순간에 점 또는 막대 표시
  - 입력이 늦으면 기준선 오른쪽, 빠르면 왼쪽에 표시
- 안내 문구:
  - `Adjust Sync until your input marker lands near the center.`
  - 너무 길게 설명하지 않고 팝업 안에 한 줄만.

구현 방식은 두 단계가 좋습니다.

1. **수동 calibration**
   - beep를 일정 간격으로 재생합니다.
   - 사용자가 맞춰 연주하면, 앱은 감지된 voiced frame을 기준선 주변에 표시합니다.
   - 사용자가 `- / +`로 조정합니다.
   - 가장 안전하고 구현이 단순합니다.

2. **반자동 추천값**
   - 나중에 voiced frame이 beep 이후 몇 ms에 처음 잡혔는지 여러 번 평균내서 `Suggested +80 ms`처럼 보여줍니다.
   - onset detection이 아직 약하므로 처음부터 자동 적용은 피하는 게 좋습니다.

초기 구현은 **수동 calibration 팝업 + Sync 값 조절**이 적절합니다.
이 값을 실제 practice scoring에는 `judgeScoreSeconds = scoreSeconds - syncMs / 1000`으로 적용하면 됩니다.

추천안은 이렇습니다.

1. **Pitch 판정은 기존 sample 누적 유지**
   - `Perfect / Ok / Bad / Miss`
   - 현재처럼 음정 정확도 중심으로 점수 누적

2. **Time 판정은 note event당 1회만 기록**
   - 각 `NoteEvent`의 시작 시점 기준
   - 사용자가 해당 note에 처음 유효한 pitch를 낸 시각을 `firstHitSeconds`로 저장
   - `timingErrorMs = firstHitSeconds - note.startSeconds`
   - 음수면 early, 양수면 late

3. **점수 반영은 event timing multiplier로 처리**
   - `<= 33.34ms`: `1.0`
   - `<= 66.67ms`: `0.5`
   - `<= 100ms`: `0.25`
   - `> 100ms`: `0`

   - `<= 33.34ms`: `100%`
   - `<= 100ms`: `50%`
   - `> 100ms`: `0%`

```text
0 ~ 33.34ms     : timingAccuracy = 1.0
33.34 ~ 100ms   : timingAccuracy = 0.5
100ms 초과      : timingAccuracy = 0
```

```text
eventScore = pitchScoreForEvent * timingAccuracy
```

다만 바로 전체 점수 구조를 event-finalization 방식으로 바꾸기보다는, 1차 구현은 다음처럼 하는 게 안전합니다.

sampleScore = difficulty * pitchAccuracy * timingAccuracyForTargetEvent

인터벌만 완화하는 것보다는 먼저 **Early 확정 조건을 고치는 것**이 맞아 보입니다.

현재 이상 현상의 핵심 원인은 이쪽일 가능성이 큽니다.

```text
note.startSeconds - 100ms 부터 다음 note를 후보로 허용
→ 아직 다음 note가 시작하지 않았는데도 pitch가 가까우면 그 note의 첫 입력으로 확정
→ timingLabel = Early
```

악보 오실레이터를 그대로 마이크에 대면 물리적으로는 보통 `Late`가 떠야 자연스럽습니다. 그런데 `Early`도 뜬다면, 입력 지연 문제가 아니라 **다음 노트를 너무 일찍 후보로 열어 둔 부작용**입니다. 특히 빠른 곡, 반복음, 같은 계이름, pitch-class 허용이 겹치면 쉽게 발생합니다.

권장 수정 방향은 이렇습니다.

1. **scoring interval은 33.34ms 유지**
   - 이건 판정 해상도라서 유지하는 편이 좋습니다.

2. **Early 후보는 열어두되, timing 확정은 보류**
   - `start - 100ms` 구간에서 pitch가 맞아도 즉시 Early로 확정하지 않습니다.
   - note 시작 시점 근처 또는 시작 이후 첫 유효 sample에서 확정합니다.
   - 즉, early 입력은 “후보”로만 보고, 너무 이른 pre-trigger를 막습니다.

3. **timing 점수 창은 약간 완화**
   - 현재 `33.34ms / 100ms`는 pitch detector + Web Audio + 마이크 + RAF jitter까지 고려하면 꽤 빡빡합니다.
   - 연습 보조용이면 아래가 더 현실적입니다.

```text
<= 66.67ms     : 100%
<= 150ms       : 50%
> 150ms        : 0%
```

4. **Early/Late 표시는 보수적으로**
   - `66.67ms` 안이면 아무 배지도 표시하지 않음
   - 그 밖에서만 작은 `Early` / `Late`
   - `Miss`에는 계속 표시하지 않음

정리하면, “인터벌 완화”가 아니라:

```text
scoring interval: 33.34ms 유지
timing perfect window: 33.34ms → 66.67ms 완화
timing half window: 100ms → 150ms 완화
pre-start 입력은 즉시 timing 확정하지 않도록 수정
```

이 조합이 맞겠습니다. 특히 세 번째보다 **pre-start 확정 방지**가 더 중요합니다.

네. 일단 방향은 **생 입력 pitch로 복귀**가 맞습니다. target을 보고 pitch를 보정하면 “사용자가 낸 음”이 아니라 “악보에 맞춰진 음”을 판정하게 되어, 지금처럼 자동으로 맞는 판정이 나옵니다.

현실적인 선택지는 세 가지입니다.

1. **판정은 raw pitch만 사용**
   - target 기반 octave/harmonic 보정 없음
   - pitchy가 낸 `midi + centOffset` 그대로 판정
   - 가장 정직하고 디버깅이 쉽습니다.
   - 단점은 배음 오인식이 그대로 `Miss`가 됩니다.

2. **보정은 판정이 아니라 “신뢰도 필터”로만 사용**
   - 예: pitch가 갑자기 C4 → C6로 튀면 C4로 고치지 않고, 그 frame을 `unclear pitch`로 버림
   - 즉, “맞게 만들어주기”가 아니라 “수상하면 채점하지 않기”
   - 이게 부작용이 가장 적습니다.

3. **탐지기 자체 교체/비교**
   - 현재 `pitchy`는 McLeod Pitch Method 기반입니다. 실시간 튜너용으로 적합하지만 배음/악기 특성에서 튈 수 있습니다.
   - aubio 쪽은 `yin`, `yinfft`, `fcomb`, `mcomb` 등 여러 방식이 있고, 특히 comb 계열은 harmonic fundamental 추정에 초점이 있습니다.
   - CREPE는 논문상 강력한 monophonic pitch tracker지만, 브라우저 실시간 앱에서는 모델 크기와 성능 부담이 큽니다.

제가 추천하는 다음 단계는 이렇습니다.

```text
1. scoring에서 모든 target 기반 pitch 보정 제거
2. pitch dot 표시도 일단 raw pitch로 통일
3. 배음 의심 frame은 보정하지 말고 reject 처리
4. 이후 aubio/YIN 또는 CREPE를 실험 브랜치로 비교
```

배음을 “정확히 필터링”하는 완벽한 휴리스틱은 사실상 어렵습니다. 대신 안정적인 구조는 **raw detector 결과를 믿되, 의심스러운 frame은 점수에 넣지 않는 것**입니다.

- pitchy: McLeod Pitch Method 기반 JS pitch detector, MIT

- CREPE: CNN 기반 monophonic pitch estimation 논문/오픈소스 구현
