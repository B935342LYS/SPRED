# 1.26. Step5 레이아웃 Draft Apply와 Preset 구현 메모

## 1. 목적

이 메모는 layout draft, Apply, 로컬 슬롯 프리셋, 파일 프리셋, toolbar 프리셋 흐름이 연결된 뒤의 레이아웃 편집기 구현 상태를 기록한다.

이 메모는 다음 문서를 보완한다.

- `docs/2.4-layout-edit-ui-spec.md`
- `docs/2.5-layout-preset-format-spec.md`

이 문서는 구현 해설용 메모이다.
저장 포맷과 활성 동작 규칙은 active spec 문서를 따른다.

## 2. 구현된 범위

구현된 범위는 다음과 같다.

- 현재 `ScoreFile` 기반 `LayoutDraftBundle` 생성
- layout dialog의 선택 string과 선택 row 상태
- 편집 가능한 note/gap row 목록 렌더링
- 단일 `Note Height` 입력을 통한 공통 note row 높이 편집
- gap별 입력과 preview drag handle을 통한 gap row 높이 편집
- note row 추가/삭제 draft mutation
- gap row 추가/삭제 draft mutation
- 실제 row 높이를 반영한 draft preview
- draft를 `ScoreFile`에 적용하는 Apply 흐름
- 삭제된 note row를 참조하는 track cell 삭제 요약
- layout apply가 track cell을 삭제하기 전 사용자 확인
- layout apply 후 구조 공유 방식의 `ScoreFile` 갱신
- layout apply 후 runtime rebuild와 playback runtime reset
- 로컬 layout preset Slot 1..3 저장/불러오기
- layout preset JSON 파일 저장/불러오기
- toolbar dropdown의 Default Layout과 local slot 연결

## 4. Draft 편집 판단

Note row 높이는 공통값으로 다룬다.
이는 note row 높이가 string별 물리 거리라기보다 손가락 접촉 지점의 표시 간격에 가깝다는 판단에 따른 것이다.
최종 row definition에는 각 note row가 자기 `height` 값을 계속 저장하지만, UI에서는 모든 note row를 함께 편집한다.

Note row MIDI 선택은 자동으로 처리한다.
사용자가 선택 row의 위나 아래에 note row를 삽입하면, 새 MIDI는 선택 row와 삽입 방향에서 계산한다.
따라서 UI는 row를 추가할 때마다 pitch를 직접 고르게 하지 않는다.

Gap row는 더 이상 두 note row 사이에만 제한하지 않는다.
최상단과 최하단 gap row도 허용한다.
Boundary MIDI 범위는 `-1..128`이므로 MIDI 0 아래나 MIDI 127 위의 gap도 필요한 경우 표현할 수 있다.

Gap row는 서로 인접할 수 없다.
이 규칙은 layout 구조를 가볍게 유지하고, 중복 gap row에 대한 suffix 기반 rowId 생성을 피하기 위한 것이다.

Note row 삭제는 선택 string의 최상단 또는 최하단 note row에만 허용한다.
중간 note row 삭제는 실수로 파괴적인 layout 편집이 발생하기 쉬우므로 차단한다.
Gap row는 직접 삭제할 수 있다.

## 5. Apply 판단

Apply 흐름은 구조 공유 기반 replacement model이다.

```text
current ScoreFile
-> 변경되지 않은 하위 객체를 공유하는 next ScoreFile 생성
-> 기존 global row 유지
-> draft의 note/gap row로 교체
-> 가능한 경우 변경되지 않은 track 재사용
-> 영향을 받은 track만 복사하고 note row를 더 이상 가리키지 않는 cell 삭제
-> validateScoreFile(nextScore)
-> createRuntimeDocument()
-> buildParsedDocument()
-> analyzeDocument()
-> createCanvasRenderInput()
-> render
-> playback runtime reset
```

Cell 삭제는 core validation error로 보지 않는다.
이는 note row가 제거된 layout을 적용할 때 app 계층에서 발생하는 정상적이지만 파괴적인 결과이다.
따라서 `layout_apply.ts`에서 삭제 대상을 요약하고, UI 경계에서 사용자 확인을 받은 뒤 명시적으로 허용된 경우에만 적용한다.

`ScoreFile.musicData.createdAt`과 `ScoreFile.musicData.updatedAt`은 layout preset metadata에서 복사하지 않는다.
Preset timestamp는 preset 자체를 설명하며, score timestamp가 아니다.
Score timestamp 갱신은 score JSON 저장 정책의 영역으로 남긴다.

## 6. Preset 판단

로컬 저장/불러오기와 파일 저장/불러오기는 같은 `UserLayoutPresetData` 구조를 사용한다.

로컬 preset은 고정 slot을 사용한다.

```text
layout-slot:{instrumentPresetId}:1
layout-slot:{instrumentPresetId}:2
layout-slot:{instrumentPresetId}:3
```

이는 이전의 index 기반 local preset 구상을 대체한다.
고정 slot 방식을 선택한 이유는 다음과 같다.

- localStorage 공간은 main score JSON 저장을 위해 보호되어야 한다.
- 이 기능은 대규모 preset library가 아니라 자주 쓰는 개인 preset을 위한 기능이다.
- slot UI는 overwrite 동작을 명확하게 보여준다.
- MVP에서는 별도 delete flow가 없어도 된다.

파일 저장/불러오기에는 slot 제한을 두지 않는다.
파일 저장은 backup, 공유, 브라우저/기기 간 preset 이동에 사용한다.

## 7. Toolbar 동작

Layout toolbar dropdown은 다음 항목을 표시한다.

```text
Default Layout
Slot 1: ...
Slot 2: ...
Slot 3: ...
```

`Default Layout`을 선택하면 현재 score를 load했을 때 캡처한 layout을 다시 적용한다.
값이 있는 local slot을 선택하면 dialog의 Apply 버튼과 같은 삭제 확인, 구조 공유 apply, runtime rebuild, playback reset 경로로 preset을 즉시 적용한다.
빈 slot을 선택하면 Default로 fallback하고 warning message를 표시한다.

Save/Reset 버튼을 outer toolbar에서 제거한 뒤 toolbar 폭을 줄였다.
해당 action들은 draft context가 분명한 layout dialog 내부에 남겨 둔다.
