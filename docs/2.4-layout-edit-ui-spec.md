# 2.4. 레이아웃 편집 UI 명세

## 1. 목적

본 문서는 `회귀 코드`의 레이아웃 편집 UI를 정의한다.

여기서의 레이아웃 편집은 현재 악보의 실제 행 구조인
`ScoreFile.layout.rowDefinitions`와 해당 레이아웃이 참조하는 악기 정보인
`ScoreFile.instData`를 수정하는 기능이다.
사용자가 팝업창에서 행 구성을 편집한 뒤 `Apply`를 누르면, 앱은 현재 악보를 다시 빌드하여 변경된 레이아웃을 실제 렌더링과 편집 좌표에 반영한다.

즉 이 기능은 단순 화면 설정이 아니다.
적용된 레이아웃은 악보의 기본 레이아웃으로 취급된다.

## 2. 핵심 결정

- 전역 행은 고정한다.
- 사용자는 레이아웃 프리셋 이름과 악기 정보를 함께 수정할 수 있다.
- 별도 Instrument Name 입력은 두지 않고, layout preset 표시명을 `instData.instName`에도 적용한다.
- 수정 가능한 악기 정보는 악기 종류/표시명, 개방현 지원 여부, 현별 min/max/open MIDI이다.
- 사용자는 전역 행을 제외한 note 행과 gap 행을 추가하거나 삭제할 수 있다.
- 사용자는 기존 note/gap 행의 높이를 수정할 수 있다.
- 행 추가 위치는 사용자가 선택한 행의 위 또는 아래로 지정한다.
- note 행 삭제 시 그 행을 참조하던 악보 셀도 삭제한다.
- 셀이 삭제되는 경우, 적용 전 사용자에게 반드시 고지하고 확인을 받는다.
- note MIDI 중복은 허용하지 않는다.
- MIDI 값 범위는 `0..127`이다.
- note 행의 MIDI 입력은 번호 직접 입력이 아니라 전체 MIDI `0..127` 계이름 드롭다운으로 처리한다.
- rowId는 앱이 자동 생성한다.
- 사용자는 팝업에서 편집 대상 `stringId`를 먼저 선택한다.
- note/gap 행 추가 시 `stringId`는 현재 선택된 string을 자동으로 사용한다.
- gap 행은 note 행 사이의 빈 공간이며, 사용자는 높이만 직접 입력한다.
- gap 행의 `fromMidi/toMidi`는 프로그램이 인접 note 행을 기준으로 자동 생성한다.
- 같은 note 행 사이에는 gap 행을 하나만 둘 수 있다.
- note 행만 존재하는 레이아웃은 허용한다.

## 3. 편집 대상 저장 구조

첫 구현은 레이아웃 변경을 행 단위 patch나 mutation log로 저장하지 않는다.
팝업에서는 현재 score에서 만든 draft bundle을 편집하고, `Apply` 시 현재 score clone 위에 draft bundle을 통째로 덮어쓴다.

draft bundle의 핵심 구성은 다음 두 가지이다.

- `instData`
- 전역 행을 제외한 note/gap `rowDefinitions`

전역 행은 draft bundle에 포함하지 않는 것을 원칙으로 한다.
파일 불러오기 등 외부 입력에 전역 행이 포함되어 있더라도 첫 구현에서는 무시한다.

### 3.1. 악보 내부 레이아웃

편집 대상은 다음 필드이다.

```ts
ScoreFile.layout.rowDefinitions
```

기존 저장 타입은 유지한다.

```ts
type RowDefinition =
  | GlobalRowDefinition
  | NoteRowDefinition
  | GapRowDefinition;

type NoteRowDefinition = {
  rowId: string;
  type: "note";
  stringId: string;
  midi: number;
  height: number;
  displayLabel: string;
};

type GapRowDefinition = {
  rowId: string;
  type: "gap";
  stringId: string;
  fromMidi: number;
  toMidi: number;
  height: number;
};
```

### 3.2. 악기 정보 편집

레이아웃 팝업은 `ScoreFile.instData`도 draft로 편집한다.

첫 구현에서 사용자가 수정할 수 있는 필드:

```ts
type EditableInstrumentLayoutData = {
  presetId: string;
  family: string;
  instName: string;
  supportsOpen: boolean;
  strings: Array<{
    stringId: string;
    stringName: string;
    openMidi?: number;
    minMidi: number;
    maxMidi: number;
  }>;
};
```

주의:

- 기준이 되는 현 MIDI 입력은 기존 저장 구조의 `openMidi`를 사용한다.
- `supportsOpen === true`일 때만 `openMidi`를 저장한다.

검증 규칙:

- MIDI 값은 모두 정수이며 `0..127` 범위여야 한다.
- 각 string은 `minMidi <= maxMidi`를 만족해야 한다.
- `supportsOpen === true`이면 개방현 MIDI 값이 필요하다.
- `supportsOpen === false`이면 저장되는 `openMidi`는 제거한다.
- 적용하려는 note row의 MIDI는 해당 row의 `stringId`가 가리키는 string의 `minMidi..maxMidi` 안에 있어야 한다.
- note row 드롭다운은 전체 `0..127`을 보여주지만, `Apply` 시 string 범위를 벗어난 note row가 있으면 오류로 막는다.

### 3.3. 레이아웃 프리셋 저장

사용자가 만든 `rowDefinitions`는 별도 레이아웃 프리셋으로 저장할 수 있어야 한다.

저장 경로는 두 가지를 제공한다.

1. 브라우저 로컬 저장
2. 파일 저장 및 파일 불러오기

로컬 저장은 사이트에 다시 접속해도 사용자가 이전에 만든 레이아웃을 다시 불러올 수 있게 하기 위한 기능이다.
파일 저장/불러오기는 브라우저 로컬 저장과 별개로 레이아웃을 백업하거나 다른 환경으로 옮기기 위한 기능이다.

#### 3.2.1. 로컬 저장 key

기존 `2.0` 문서의 방향을 유지해, 악보 작성자와 악기 프리셋별로 레이아웃 프리셋 목록을 분리한다.

```text
layout-index:{scoreWriter}:{instrumentPresetId}
layout:{scoreWriter}:{instrumentPresetId}:{layoutPresetId}
```

- `scoreWriter`는 `ScoreFile.musicData.scoreWriter`를 사용한다.
- `instrumentPresetId`는 `ScoreFile.instData.presetId`를 사용한다.
- `layoutPresetId`는 앱이 생성하는 내부 식별자이다.
- 악보 제목은 localStorage key에 포함하지 않는다.

#### 3.2.2. 로컬 저장 index 값

```ts
type UserLayoutPresetIndexItem = {
  layoutPresetId: string;
  layoutPresetDisplayName: string;
  updatedAt: string;
};
```

#### 3.2.3. 개별 레이아웃 프리셋 값

첫 구현의 저장값은 layout diff가 아니라 적용 가능한 layout bundle 전체를 보관한다.
즉, 사용자가 편집한 `instData`와 전역 행을 제외한 note/gap `rowDefinitions`를 함께 저장한다.

```ts
type UserLayoutPresetData = {
  formatVersion: "1";
  layoutPresetId: string;
  layoutPresetDisplayName: string;
  instrumentPresetId: string;
  createdAt: string;
  updatedAt: string;
  instData: EditableInstrumentLayoutData;
  rowDefinitions: RowDefinition[];
};
```

파일 저장도 같은 구조를 JSON으로 내려받는다.
따라서 레이아웃 프리셋 파일은 rowDefinitions만이 아니라 악기 정보 draft도 함께 복원한다.

저장하지 않는 값:

- score title
- zoom level
- scroll position
- 전역 행
- track/global cell 데이터

프리셋은 score 전체 백업이 아니다.
프리셋을 적용한 뒤 삭제된 cell을 복구하려면 별도 score 파일 저장이나 후속 undo 기능이 필요하다.

## 4. 전역 행 정책

전역 행은 레이아웃 편집 대상에서 제외한다.

고정 전역 행:

- `bpm`
- `beatsPerBar`
- `stepsPerBeat`
- `dynamics`

팝업창의 행 목록에는 전역 행을 표시하지 않거나, 표시하더라도 잠금 상태로 보여준다.
첫 구현에서는 단순화를 위해 전역 행을 목록에서 제외한다.

`Apply` 시에는 기존 score의 전역 행들을 그대로 유지하고, 사용자가 편집한 note/gap 행 목록을 그 뒤에 붙인다.

```text
기존 global rows
-> 사용자가 편집한 note/gap rows
```

## 5. rowId 생성 규칙

rowId는 사용자가 직접 입력하지 않는다.
앱이 행 타입과 입력값으로 생성한다.

```text
note row: {stringId}-note-{midi}
gap row:  {stringId}-gap-{fromMidi}-{toMidi}
```

note row는 같은 `stringId + midi` 조합이 중복될 수 없으므로 rowId도 자연스럽게 고유해야 한다.

gap row는 같은 note 행 사이에 하나만 둘 수 있다.
따라서 같은 `stringId + fromMidi + toMidi` 조합의 gap row도 중복될 수 없다.

```text
s1-gap-60-61
```

첫 구현에서는 rowId가 유일하면 된다.
다만 gap 행은 프로그램이 인접 note 행을 기준으로 만들기 때문에, 저장된 `fromMidi/toMidi`와 실제 배치의 인접 note 행이 일치하도록 유지한다.

## 6. 값 제약

### 6.1. 공통

- `height`는 정수이다.
- `height`는 1 이상이어야 한다.
- `stringId`는 `ScoreFile.instData.strings[].stringId` 중 하나여야 한다.
- rowId는 `layout.rowDefinitions` 안에서 유일해야 한다.

### 6.2. 높이 상한과 압축 정책

첫 구현에서는 전체 레이아웃 높이 압축을 포함하지 않는다.
사용자가 입력한 px 값은 실제 row height로 저장되고 렌더링된다.

임시 높이 입력 상한은 `1100px`이다.
이는 현재 테스트 악보 파일의 전체 layout 높이 `1026px`보다 큰 100의 최소 배수이다.

```text
height range: 1..1100
```

전체 높이가 커졌을 때 화면에 맞추는 동작은 `Fit Height`, zoom, scroll 같은 보기 기능에서 처리한다.
레이아웃 편집 저장값 자체를 비율로 압축하지 않는다.

### 6.3. note 행

- `midi`는 내부 저장값으로는 정수이다.
- `midi` 범위는 `0..127`이다.
- 같은 `stringId` 안에서 같은 `midi`를 가진 note 행은 중복될 수 없다.
- `displayLabel`은 MIDI 값에서 자동 생성한다.
- 첫 구현의 표기 방식은 sharp 계열을 사용한다.
- UI에서는 MIDI 번호 직접 입력 대신 전체 MIDI `0..127` 계이름 드롭다운을 사용한다.
- 드롭다운에서 선택한 MIDI가 현재 string의 `minMidi..maxMidi` 밖일 수 있으므로, `Apply` 시 최종 검증을 수행한다.
- 드롭다운 정렬은 높은 음이 위, 낮은 음이 아래에 오도록 한다.
- 추후 `Reverse Mode` 기능에서 자동 역순 정렬을 지원할 수 있다.

예:

```text
60 -> C4
61 -> C#4
```

### 6.4. gap 행

gap 행은 빈 공간 이상의 음악적 의미를 갖지 않는다.
사용자는 gap 행의 높이만 입력한다.

기존 저장 타입에는 `fromMidi`와 `toMidi`가 있으므로, 프로그램은 gap 행이 놓이는 위치의 인접 note 행을 찾아 자동으로 값을 채운다.

자동 생성 규칙:

1. 사용자가 gap 행을 삽입할 위치를 고른다.
2. 프로그램은 그 위치 위쪽에서 가장 가까운 note 행과 아래쪽에서 가장 가까운 note 행을 찾는다.
3. 두 note 행이 모두 존재하고 같은 `stringId`에 속해야 gap 행을 만들 수 있다.
4. `fromMidi`는 두 note MIDI 중 낮은 값이다.
5. `toMidi`는 두 note MIDI 중 높은 값이다.
6. 이미 같은 두 note 행 사이에 gap 행이 있으면 추가하지 않는다.

따라서 gap 행은 note 행 사이에만 삽입할 수 있다.
note 행 위쪽 끝이나 아래쪽 끝처럼 한쪽 note 행만 있는 위치에는 gap 행을 추가할 수 없다.

## 7. 행 추가 위치

사용자는 현재 행 목록에서 기준 행을 선택한다.

행 추가 시 위치 선택은 다음 중 하나이다.

- 선택 행 위에 삽입
- 선택 행 아래에 삽입

기준 행이 선택되지 않은 경우 기본값은 마지막 note/gap 행 아래 삽입이다.

전역 행은 편집 목록에서 제외되므로, 삽입 기준은 note/gap 행만 사용한다.

## 8. 삭제 정책

note 행 또는 gap 행 삭제는 layout row 삭제로 처리한다.

### 8.1. note 행 삭제

note 행을 삭제하면 해당 rowId를 참조하는 모든 track cell이 삭제된다.

삭제 전 확인 메시지는 반드시 보여준다.
메시지에는 최소 다음 정보를 포함한다.

- 삭제할 rowId
- 삭제할 행 종류
- 삭제될 track cell 개수

사용자가 취소하면 아무것도 변경하지 않는다.

### 8.2. gap 행 삭제

현재 저장 규칙상 track cell은 gap 행을 참조할 수 없다.
따라서 gap 행 삭제는 보통 셀 삭제를 동반하지 않는다.

그래도 첫 구현에서는 동일한 삭제 확인 흐름을 사용한다.

## 9. 팝업창 UI

레이아웃 영역의 `Modify` 버튼을 누르면 레이아웃 설정 팝업창을 연다.

팝업창 구성:

- 제목: `Layout`
- 현재 layout preset 선택 드롭다운
- 새 layout preset 생성 버튼
- layout preset 표시명 입력
- 악기 정보 입력 영역
- 편집 대상 string 선택 드롭다운
- row 목록
- row 추가 입력 영역
- 적용/취소 버튼
- 로컬 저장/불러오기 버튼
- 파일 저장/불러오기 버튼

### 9.1. 악기 정보 입력

악기 정보 입력 영역은 현재 layout draft와 함께 적용되는 `instData` draft를 편집한다.

입력 항목:

- layout preset 표시명
- 악기 종류 또는 family
- 개방현 지원 여부
- string 목록
- string별 이름
- string별 min MIDI
- string별 max MIDI
- 개방현 지원 시 string별 open MIDI

첫 구현에서는 string 추가/삭제를 허용하지 않는다.
기존 string 목록의 값을 수정하는 범위로 시작한다.

오타마톤 대상 첫 구현에서는 개방현 개념을 실제 기능으로 사용하지 않는다.
따라서 `supportsOpen` 체크박스와 string별 open MIDI 드롭다운은 UI에 표시하되 disabled 상태로 둔다.
저장/적용 로직 연결은 후속 악기 확장 단계에서 다룬다.

### 9.2. row 목록

row 목록은 전역 행을 제외한 note/gap 행 중 현재 선택된 `stringId`의 행을 표시한다.
다른 string의 행은 선택된 string을 바꾸면 별도로 표시한다.

각 행 표시 정보:

- 선택 radio 또는 클릭 가능한 row
- 종류: note/gap
- rowId
- height 입력값
- note 행: midi, displayLabel
- gap 행: fromMidi, toMidi
- delete 버튼

기존 행에서 첫 구현에 허용하는 직접 수정값:

- note 행 height
- gap 행 height

기존 note 행의 MIDI와 stringId 변경은 첫 구현에서 직접 수정하지 않는다.
음정이나 string을 바꾸고 싶을 때는 새 note 행을 추가하고 기존 행을 삭제하는 흐름으로 처리한다.
이는 기존 rowId를 참조하는 cell 삭제/이동 판단을 단순하게 유지하기 위한 결정이다.

### 9.3. row 추가 입력

입력 항목:

- row type: note 또는 gap
- height
- note 선택 시: 계이름 드롭다운
- gap 선택 시: 별도 MIDI 입력 없음. 삽입 위치의 인접 note 행으로 자동 판정

`stringId`는 별도 행 입력값으로 받지 않는다.
팝업 상단의 편집 대상 string 드롭다운에서 선택한 값을 사용한다.

기본값:

- note row 기본 높이: `7px`
- gap row 기본 높이: `21px`

버튼:

- 선택 행 아래 삽입
- 선택 행 위에 삽입

두 삽입 버튼은 팝업 내부의 draft row 목록만 바꾼다.
실제 악보에는 아직 반영하지 않는다.

### 9.4. 적용과 취소

- `Apply`: draft instData와 draft row 목록을 현재 score에 반영하고 악보를 reload한다.
- `Cancel`: 팝업을 닫고 변경을 버린다.

삭제될 cell이 있으면 `Apply` 과정에서 확인 dialog를 띄운다.

## 10. 적용 흐름

첫 구현의 적용 방식은 단순 덮어쓰기 모델이다.

- 현재 `ScoreFile`을 clone한다.
- clone의 `instData`를 draft `instData`로 교체한다.
- clone의 `layout.rowDefinitions`는 기존 global rows와 draft note/gap rows를 합쳐 새로 만든다.
- 새 rowDefinitions에 존재하지 않거나 note row가 아닌 rowId를 참조하는 track cell은 제거한다.
- 제거될 cell이 있으면 실제 반영 전에 사용자 확인을 받는다.
- 마지막에 기존 score 검증 함수를 최종 권위로 사용한다.

이 방식에서는 “어떤 행을 추가/삭제했는지”를 별도 patch로 보관하지 않는다.
팝업 내부에서만 draft 목록을 수정하고, `Apply` 시에는 완성된 결과를 한 번에 score에 반영한다.

```text
Modify 클릭
-> 현재 ScoreFile.layout.rowDefinitions에서 note/gap rows를 draft로 복사
-> 현재 ScoreFile.instData를 draft로 복사
-> 팝업에서 draft 수정
-> Apply 클릭
-> draft 전용 사전 검증
-> 기존 global rows + draft note/gap rows로 새 rowDefinitions 생성
-> 삭제될 track cell 계산
-> 삭제될 셀이 있으면 사용자 확인
-> ScoreFile clone에 instData, layout, cell 삭제 반영
-> validateScoreFile(nextScore)
-> createRuntimeDocument()
-> buildParsedDocument()
-> analyzeDocument()
-> createCanvasRenderInput()
-> render
-> playback runtime reset
```

검증 실패 시:

- 기존 score는 유지한다.
- 팝업은 닫지 않는다.
- 오류 메시지를 status line 또는 팝업 내부에 표시한다.

### 10.1. 검증 책임 분리

검증은 두 단계로 나눈다.

1. draft 전용 사전 검증
2. `ScoreFile` 최종 검증

draft 전용 사전 검증은 UI 조작 즉시 안내해야 하는 레이아웃 편집 규칙만 다룬다.

- note row MIDI 중복 여부
- gap row 삽입 가능 위치
- 같은 note 행 사이 gap row 중복 여부
- 선택된 `stringId` 존재 여부
- 높이 입력 범위
- layout preset 표시명 중복 여부

`ScoreFile` 최종 검증은 기존 `src/core/score/score_validate.ts`의 `validateScoreFile()`을 사용한다.
레이아웃 편집용 validator가 score validator와 같은 저장 형식 검증을 중복해서 구현하지 않는다.

필요하다면 `score_validate.ts`의 검증 범위를 확장한다.
예를 들어 다음 규칙은 저장/import 관점에서도 유효하므로 `validateScoreFile()` 쪽에 두는 것이 적절하다.

- `instData.strings[].minMidi/maxMidi/openMidi` 값 범위
- `supportsOpen`과 `openMidi`의 관계
- note/gap row의 `stringId`가 `instData.strings`에 존재하는지
- note row의 MIDI가 해당 string의 `minMidi..maxMidi` 안에 있는지
- note row의 `stringId + midi` 중복 여부
- row height가 양의 정수인지

### 10.2. cell 삭제 판정

`Apply` 후 새 layout에서 유효한 note rowId 집합을 만든다.
track cell은 다음 조건을 만족할 때만 유지한다.

- cell의 `rowId`가 새 layout에 존재한다.
- 해당 row가 note row이다.

나머지 track cell은 삭제 대상으로 계산한다.
삭제 대상이 하나 이상이면 사용자에게 삭제될 cell 개수를 표시하고 확인을 받는다.

global cell은 전역 행이 유지되므로 layout 편집으로 삭제하지 않는다.

## 11. 로컬 저장 흐름

### 11.1. Local Save

팝업의 현재 draft bundle을 레이아웃 프리셋으로 저장한다.

필수 입력:

- layout preset 표시명

저장 결과:

- index key에 프리셋 목록 갱신
- data key에 `UserLayoutPresetData` 저장

같은 `layoutPresetDisplayName`은 허용하지 않는다.
동일한 표시명이 이미 있으면 경고 후 저장을 취소한다.

### 11.2. Local Load

현재 score의 `scoreWriter + instData.presetId`에 해당하는 프리셋 목록을 불러온다.
사용자가 선택한 프리셋의 `instData`와 `rowDefinitions`를 팝업 draft에 적용한다.

이 시점에도 실제 악보에는 반영하지 않는다.
실제 반영은 `Apply`를 눌렀을 때 수행한다.

## 12. Reset 정책

`Reset`은 저장 비용을 줄이기 위해 두 단계 의미를 구분한다.

### 12.1. Dialog Reset

팝업이 열린 상태에서의 `Reset`은 현재 팝업 draft를 마지막으로 로드/적용된 score 상태로 되돌린다.

이 동작은 별도 저장 공간이 필요 없다.
팝업을 열 때 현재 `ScoreFile.instData`와 `ScoreFile.layout.rowDefinitions`에서 draft를 다시 만들면 된다.

복구 대상:

- draft instData
- draft note/gap row 목록
- 선택된 layout preset 표시명

복구하지 않는 대상:

- 이미 `Apply`되어 score에 반영된 이전 layout
- 삭제 확인 후 이미 제거된 track cell

### 12.2. 원래 레이아웃 복구

사용자가 생각한 “원래 레이아웃과 셀 복구”는 별도 snapshot이 있어야 정확히 가능하다.
하지만 layout 적용 전의 모든 rowDefinitions, instData, 삭제될 track cells를 계속 저장하면 저장 비용과 구현 복잡도가 커진다.

따라서 첫 구현의 `Reset`은 원래 score 전체 복구가 아니라, 팝업 draft를 현재 score 상태로 되돌리는 기능으로 제한한다.

원래 layout을 보존하고 싶다면 다음 기능으로 해결한다.

- layout preset File Save
- layout preset Local Save
- 사용자가 적용 전 현재 레이아웃을 저장해둔 뒤 필요 시 다시 불러오기

후속 기능으로 undo stack을 만들 경우, layout apply 직전 score snapshot 1개만 메모리에 보관하는 방식은 검토할 수 있다.
단, 이 snapshot은 앱 세션 중 undo용이며 localStorage에 자동 저장하지 않는다.

## 13. 파일 저장/불러오기 흐름

### 13.1. File Save

현재 draft bundle을 `UserLayoutPresetData` JSON 파일로 다운로드한다.
파일에는 `instData`와 note/gap `rowDefinitions`가 포함된다.

### 13.2. File Load

사용자가 선택한 layout preset JSON 파일을 읽어 draft bundle에 반영한다.
이 시점에는 실제 score를 바꾸지 않는다.
실제 반영은 `Apply`에서 수행한다.

파일 불러오기 검증:

- JSON parse 가능
- `formatVersion === "1"`
- `instData`가 존재하고 값 제약을 통과
- `rowDefinitions`가 배열
- 전역 행은 포함하지 않거나, 포함되어 있어도 무시한다.
- note/gap row 값 제약을 통과해야 한다.

## 14. 확정 상태

현재 구현 전 추가로 확정해야 할 항목은 없다.

## 15. 첫 구현 권장 범위

첫 구현은 다음 순서가 안전하다.

1. 팝업 UI와 draft row 편집
2. draft instData 편집과 검증
3. Apply로 `ScoreFile.instData`와 `ScoreFile.layout.rowDefinitions` 반영
4. note row 삭제 시 cell 삭제 확인
5. localStorage 저장/불러오기
6. 파일 저장/불러오기

이 순서라면 실제 레이아웃 반영 경로를 먼저 안정화한 뒤, 저장/불러오기 기능을 붙일 수 있다.

## 16. 구현 파일 구조

첫 구현은 모듈 수를 과도하게 늘리지 않고, 기존 기능을 재사용한다.

권장 구조:

```text
regression-code/src/app/layout/
  layout_types.ts
  layout_draft.ts
  layout_apply.ts
  layout_dialog_binding.ts
  layout_preset.ts

regression-code/src/infra/
  layout_preset_storage.ts
```

역할:

- `layout_types.ts`: 팝업 draft와 layout preset 저장 타입
- `layout_draft.ts`: 현재 score에서 draft bundle 생성, draft row 추가/삭제, rowId/displayLabel 자동 생성
- `layout_apply.ts`: draft bundle을 score clone에 덮어쓰기, 삭제 cell 계산, `validateScoreFile()` 호출
- `layout_dialog_binding.ts`: DOM 생성, 이벤트 연결, 팝업 상태 표시
- `layout_preset.ts`: preset id 생성, 표시명 중복 확인, 저장 데이터 조립
- `layout_preset_storage.ts`: localStorage index/data 저장과 불러오기

재사용할 기존 기능:

- `validateScoreFile()`을 최종 score 검증으로 사용한다.
- `createRuntimeDocument()`와 기존 빌드/분석/렌더 파이프라인을 재사용한다.
- score clone은 기존 edit 적용 코드의 helper를 재사용하거나, 필요하면 `src/core/score/` 쪽 공용 helper로 이동한다.
- score 파일 입출력에서 이미 사용하는 JSON 직렬화/파일 읽기 helper를 재사용한다.
- pitch label 생성 helper가 있으면 note row `displayLabel`과 드롭다운 표시에 재사용한다.

별도의 대형 `layout_validate.ts`는 만들지 않는다.
UI 조작에 필요한 가벼운 draft 규칙은 `layout_draft.ts` 또는 작은 helper 안에 두고, 저장 형식의 최종 판단은 `score_validate.ts`에 둔다.

### 16.1. 기존 모듈 재사용 계획

다음 기능은 새 layout 전용 모듈을 만들기보다 기존 모듈을 직접 사용하거나, 기존 모듈을 작게 일반화해 사용한다.

| 필요한 기능 | 재사용 대상 | 처리 방향 |
| --- | --- | --- |
| ScoreFile 깊은 복사 | `src/app/edit/edit_apply.ts`의 `cloneScoreFile()` | 우선 재사용한다. layout 적용에서도 공용으로 쓰이게 되면 후속으로 `src/core/score/score_clone.ts` 이동을 검토한다. |
| ScoreFile 최종 검증 | `src/core/score/score_validate.ts`의 `validateScoreFile()` | 최종 저장 구조 검증 권위로 사용한다. 부족한 row/string/MIDI 검증은 이 모듈에 확장한다. |
| 런타임 인덱스 재생성 | `src/core/score/create_runtime_document.ts`의 `createRuntimeDocument()` | layout apply 후 새 RuntimeDocument 생성에 사용한다. |
| parser/analyzer/renderer 입력 재생성 | `src/app/app_runtime.ts`의 `buildRuntimeArtifacts()` | layout apply 후 기존 full rebuild 흐름을 재사용한다. |
| 계이름 표시 | `src/app/pitch_label.ts`의 `formatPitchName()` | note row `displayLabel` 생성과 MIDI 드롭다운 표시 문자열에 사용한다. |
| 정수 입력 읽기 | `src/app/app_view_actions.ts`의 `readIntegerInput()` | height, minMidi, maxMidi, openMidi 입력 읽기에 재사용할 수 있다. |
| 파일 텍스트 읽기 | `src/infra/score_file_io.ts`의 `readTextFile()` | layout preset file load에서 그대로 사용한다. |
| JSON 다운로드 | `src/infra/score_file_io.ts` | `downloadJsonFile(fileName, value)` 같은 공용 helper로 작게 일반화한 뒤 score JSON과 layout preset JSON이 함께 사용한다. |
| localStorage 저장/조회 패턴 | `src/infra/score_local_storage.ts` | key가 여러 개인 preset 저장은 `layout_preset_storage.ts`로 분리하되, 예외 처리와 문자열 저장 방식은 기존 패턴을 따른다. |
| 파일/localStorage 버튼 배선 패턴 | `src/app/app_file_binding.ts` | layout dialog 내부 버튼 binding에서 동일한 session callback 패턴을 따른다. |
| DOM 조회 | `src/app/app_dom.ts`의 `queryElement()` | 정적 DOM 요소를 추가할 경우 재사용한다. 동적으로 만드는 dialog 내부 요소는 생성 시점에 직접 참조를 보관한다. |

### 16.2. 새로 만들 모듈의 최소 책임

`layout_types.ts`:

- `LayoutDraftBundle`
- `UserLayoutPresetData`
- `UserLayoutPresetIndexItem`
- layout apply 결과 타입

`layout_draft.ts`:

- 현재 score에서 draft bundle 생성
- note/gap row 추가와 삭제
- rowId 생성
- gap row의 `fromMidi/toMidi` 자동 계산
- note row `displayLabel` 생성
- draft 단계에서 즉시 판단 가능한 오류 반환

`layout_apply.ts`:

- draft bundle을 `ScoreFile` clone에 덮어쓰기
- 기존 global rows 보존
- 삭제될 track cell 계산
- 사용자가 확인한 뒤 cell 제거
- `validateScoreFile()` 호출
- 성공 시 next `ScoreFile` 반환

`layout_preset.ts`:

- preset id 생성
- preset 표시명 중복 확인
- draft bundle과 metadata에서 `UserLayoutPresetData` 생성
- layout preset JSON parse와 형식 확인
- layout preset 파일명 생성

`layout_preset_storage.ts`:

- localStorage key 생성
- index 저장/조회
- 개별 preset 저장/조회/삭제

`layout_dialog_binding.ts`:

- layout dialog DOM 생성 또는 조회
- draft 값을 UI에 표시
- UI 입력을 draft에 반영
- local/file save/load 버튼 연결
- apply/cancel/reset 흐름을 app 상태 변경 callback에 연결

### 16.3. 기존 모듈 소폭 수정 후보

첫 구현 중 실제 코드 중복이 생기면 다음 정도만 기존 모듈을 수정한다.

- `score_file_io.ts`
  - `serializeScoreFile()`은 유지한다.
  - `downloadScoreJson()` 내부의 Blob 다운로드 생성을 `downloadJsonFile(fileName, jsonText)` 또는 `downloadTextFile(fileName, text, mimeType)`으로 분리한다.
  - layout preset 파일 저장은 이 공용 다운로드 helper를 사용한다.
- `score_validate.ts`
  - 레이아웃 편집에서 새로 필요한 저장 무결성 검증을 추가한다.
  - layout 전용 검증 모듈로 같은 규칙을 중복 구현하지 않는다.
- `app_runtime.ts`
  - layout apply 후 full rebuild가 필요하므로, 현재 `buildRuntimeArtifacts()`를 그대로 사용한다.
  - 필요하다면 `applyLayoutDraftToState()` 같은 app 상태 반영 함수만 추가한다.

이 범위를 넘어서는 공용화는 첫 구현에서 보류한다.
예를 들어 모든 localStorage 접근을 generic storage 모듈로 재설계하거나, dialog framework를 새로 만드는 작업은 레이아웃 편집 기능 완성에 비해 범위가 크므로 하지 않는다.

2026-06-19 기준 1차 UI shell 구현이 완료되었다.

완료된 범위:

- `layout_types.ts`에 layout draft, preset, apply 결과 타입 정의
- 레이아웃 영역 `Modify` 버튼과 `Layout` dialog 연결
- preset / instrument / rows / add row 영역의 정적 UI 구성
- 현재 `ScoreFile.instData` 기반 instrument/string shell 표시
- MIDI 계이름 dropdown helper 연결
- Add Row 영역의 note/gap, height, pitch, 위/아래 삽입 버튼 배치
- Open string 관련 UI는 표시하되 disabled 처리
- New Preset 버튼의 임시 dropdown 항목 추가 UI 연결
- Apply / Add Row / Local Save / Local Load / File Save / File Load는 placeholder 안내만 표시

아직 구현하지 않은 범위:

- 실제 `LayoutDraftBundle` 생성
- 선택 string별 note/gap row 목록 렌더링
- 기존 note/gap row height 수정 반영
- note/gap row 추가/삭제 draft mutation
- draft 검증
- `ScoreFile` apply
- 삭제될 track cell 확인 dialog
- localStorage layout preset 저장/불러오기
- layout preset JSON 파일 저장/불러오기

다음 구현 단계는 `layout_draft.ts`를 추가해 현재 score에서 draft bundle을 만들고, row 목록에 기존 note/gap 행과 height 입력을 표시하는 것이다.
