# 2차 구현 보고서 작성 기준

이 문서는 2026-06-19 이후 `회귀 코드` 2차 구현 보고서를 작성할 때 사용할 목차 구조와 서술 원칙을 고정한다.

## 1. 문서 목적

- 이후 대화 context가 요약되거나 리셋되더라도 보고서 작성 방향을 유지한다.
- `docs/report-prep/`의 기계 추출 자료, git log, 구현 메모를 외부 제출용 보고서 문장으로 정리할 때 사용할 기준을 제공한다.
- 구현 명세를 새로 정의하지 않는다. 기능 동작과 데이터 계약은 기존 active spec과 실제 구현을 우선한다.

## 2. 기본 서술 방향

2차 구현 보고서는 단순 기능 목록이 아니라, 1학기 최종 구현 이후의 안정화와 확장 과정을 설명하는 개발 기록으로 작성한다.

핵심 관점:

- Google Sheets / Apps Script 기반 `이세계 코드`를 TypeScript 웹 앱 `회귀 코드`로 리메이크한 뒤, 2차 구현에서 사용자 테스트 가능한 안정판을 구축했다.
- 2차 구현은 새 기능 추가뿐 아니라 긴 악보 성능, 재생 안정성, 편집 UX, Practice Mode, 예제/매뉴얼/배포 준비를 함께 다룬 안정화 작업이다.
- 시행착오, rollback, 실사용 테스트 결과를 숨기지 않고 설계 판단의 근거로 정리한다.
- 과장된 완성 표현보다 현재 안정판의 의미와 남은 검증 항목을 명확히 구분한다.

## 3. 권장 목차

### 3.1 작업 개요

포함할 내용:

- 2026-06-19 이후 작업의 목적
- 1학기 최종 구현 상태에서 2차 확장으로 넘어간 이유
- 테스트와 사용자 피드백 준비를 위한 안정판 구축

### 3.2 2차 구현 요약

포함할 기능군:

- 레이아웃 편집 UI
- 트랙 레이어
- YouTube 동기화
- 부분 갱신 및 장악보 성능 개선
- 영역 선택, 복사/붙여넣기, Undo/Redo
- Practice Mode / 노래방 모드
- 예제 악보, 매뉴얼, 배포 준비

서술 초점:

- 기능군별 목표와 최종 달성 상태를 짧게 정리한다.
- “무엇이 추가되었는가”를 중심으로 쓰고, 상세한 시행착오와 구조 설명은 다음 장으로 넘긴다.
- 각 항목은 지나치게 길게 쓰지 않고 보고서 전체의 지도 역할을 하게 한다.

### 3.3 주요 구현 내용

기능별 상세 설명의 본문이다.

각 기능 절은 가능한 한 아래 순서를 따른다.

```text
문제
설계 판단
구현 방식
검증
후속 문제 발견
개선 / 재검증
```

서술 원칙:

- 기능별로 “문제에서 출발해 어떤 판단으로 구현했고, 테스트 후 무엇을 고쳤는지”를 설명한다.
- 시간 순서가 중요한 경우 기능 절 내부에서 검증, 문제 발견, 개선, 재검증 흐름을 반영한다.
- 각 기능의 테스트 결과는 별도 장으로 분리하지 않고 해당 기능 설명 안에 포함한다.
- 관련 모듈, 상태 흐름, renderer/audio/parser/analyzer 경계가 중요한 경우 파일명과 역할을 간단히 언급한다.
- commit hash는 필요한 곳에 근거로 붙이되, 본문을 commit 나열로 만들지 않는다.

기능별 강조 후보:

- 레이아웃 편집 UI: layout draft, row 추가/삭제/높이 조정, renderer 좌표계와의 연결
- 트랙 레이어: active/inactive track, 편집/재생 대상 분리, Practice Mode 대상 track 기반
- YouTube 동기화: iframe mode, offset, seek/pause/resume, 음수 offset 보정
- 성능 개선: partial rebuild, layer 분리, viewport bounded rendering, RAF batching, profiler
- 편집 UX: range selection, paste preview, undo/redo history, drag edit transaction
- Practice Mode: microphone pitch input, pitch/timing judge, combo/result UI, effect bonus, Pro/Easy 정책
- 예제/매뉴얼/배포: Supabase Examples, Manual, GitHub Pages staging, 사용자 테스트 준비

### 3.4 구현상 난점

이 장은 기능 목록을 반복하지 않고, 여러 기능을 관통하는 어려운 문제를 중심으로 정리한다.

포함할 난점 후보:

- 부분 갱신과 렌더링 성능
- 글리산도, 셋잇단, hold 결합 시각화와 오디오 연결
- 마이크 기반 pitch 판정 안정화
- Practice Mode의 공정성, 관대함, 실사용성 균형
- Supabase 예제 목록과 정적 GitHub Pages 배포 구조의 절충

서술 원칙:

- “어떤 기능을 만들었다”보다 “왜 어려웠고 어떤 제약 때문에 현재 구조를 택했는가”를 설명한다.
- 실패한 접근이나 rollback은 최종 판단의 근거로만 압축해서 쓴다.
- 구현 난점과 미완성 항목을 구분한다. 난점은 해결 과정 중심, 한계는 다음 장에서 다룬다.

### 3.5 현재 완성도와 한계

포함할 내용:

- 2차 구현 목표 달성 여부
- 현재 안정판의 의미
- 장기 테스트가 필요한 항목
- 2학기 공식 보고서에서 보강할 항목

서술 초점:

- 2차 구현 목표는 달성했지만, 장기간 실사용 테스트와 다양한 환경 검증이 필요하다고 정리한다.
- 저사양 환경, 다양한 마이크/브라우저, 실제 예제 악보, YouTube embedding edge case, Supabase 운영 상태를 후속 검증 항목으로 둔다.
- 후속 확장 후보와 현재 안정판의 범위를 분리한다.

## 4. 자료 사용 원칙

사용할 자료:

주의할 점:

- 구현 메모는 active spec을 대체하지 않는다. 구현 흐름과 판단 근거를 보강하는 자료로만 사용한다.
- 날짜와 commit은 최종 본문 작성 전 실제 git log와 다시 대조한다.

## 5. 해설용 코드 스냅샷 원칙

보고서 설명을 위해 `regression-code/`를 복사한 해설용 코드 스냅샷을 만들 수 있다.

권장 원칙:

- active 구현인 `regression-code/`에 설명용 장문 주석을 직접 대량 추가하지 않는다.
- 해설용 스냅샷은 별도 로컬 폴더로 만들고 `.gitignore`에 추가한다.
- `regression-code-2026-06-19/`의 세부 주석 방식을 참고하되, 기능 수정은 하지 않는다.
- 보고서에서 실제로 설명할 대표 모듈 위주로 주석을 단다.
- 해설 주석은 함수의 역할, 상태 흐름, 모듈 경계, 비직관적 분기 설명에 집중한다.

## 6. 해설용 코드 스냅샷 현재 작업 현황

현재 해설용 스냅샷은 `regression-code-report-commentary/`이다.

상태:

- 루트 `.gitignore`에서 제외된다.
- active 구현인 `regression-code/`와 분리된 보고서 설명용 복사본이다.
- `node_modules`, `dist`, Vite 로그, 로컬 환경 파일은 포함하지 않는다.
- 기능 수정이 아니라 코드 읽기와 보고서 설명을 위한 주석 추가만 한다.
- 타입체크는 이 복사본이 의존성 제외 상태이므로 기본적으로 실행하지 않는다. 동작 검증은 active `regression-code/`에서 수행한다.

이미 주석을 보강한 영역:

- `src/app/main.ts`
- `src/app/app_runtime.ts`
- `src/app/app_types.ts`
- `src/orchestration/partial_rebuild/`
  - `partial_rebuild_app_plan.ts`
  - `partial_rebuild_artifacts.ts`
  - `partial_rebuild_plan.ts`
  - `partial_rebuild_render_patch.ts`
  - `partial_rebuild_types.ts`
- `src/renderer/`
  - `canvas_score_renderer.ts`
  - `canvas_viewport.ts`
  - `canvas_visible_range.ts`
  - `canvas_coordinate.ts`
  - `canvas_grid_renderer.ts`
  - `canvas_item_builder.ts`
  - `canvas_note_renderer.ts`
  - `canvas_marker_renderer.ts`
  - `canvas_note_colors.ts`
  - `canvas_theme.ts`
  - `canvas_types.ts`

현재 주석 밀도:

- 함수와 주요 타입에는 기존 JSDoc을 유지하거나 보강한다.
- 함수 내부에는 모든 줄에 주석을 달기보다, 순회, 분기, 좌표 계산, 캐시/부분 갱신, canvas draw order, 조기 종료 조건, 모듈 경계 호출 앞에 동작 단위 `//` 주석을 단다.
- 주석은 설계 의도만 추상적으로 쓰지 않고 “어떤 값을 어떤 기준으로 계산하고 어디에 전달하는지”를 설명한다.
- active 구현으로 역이식하지 않는다.

이어받을 때 우선 후보:

1. `src/app/playback/`
   - app 상태와 playback controller 연결
   - seek/pause/resume/YouTube follower 경계
2. `src/audio/`
   - schedule builder
   - playback controller
   - Web Audio backend
   - gliss/vib/trem/dynamics 처리
3. `src/app/game/`
   - pitch frame 수집
   - practice judge
   - effect bonus judge
   - result/live UI 상태 흐름
4. `src/app/examples/`
   - Supabase manifest 호출
   - remote score load pipeline
   - 실패 시 현재 score 보존 정책
5. `src/app/edit/`, range selection, undo/redo 관련 모듈
   - rawText 작성
   - batch edit
   - selection clipboard
   - history patch

노트북에서 이어갈 때의 시작 순서:

1. `docs/HARNESS.md`의 Current Paths, Path Roles, Current Working Mode, Current Progress Summary를 확인한다.
2. 이 문서의 5장과 6장을 확인한다.
3. `git status --short --ignored`로 `regression-code-report-commentary/`가 ignored 상태인지 확인한다.
4. active 구현 수정이 아니라면 `regression-code/`는 열람만 하고, 주석 추가는 `regression-code-report-commentary/`에서만 한다.
5. 한 번에 전부 끝내려 하지 말고 모듈 단위로 나누어 진행한다.

주의할 점:

- 노트북에서 이어받을 때 해설 복사본이 없으면 다시 `regression-code/`를 복사하되, 기존 주석 작업을 덮어쓰지 않도록 먼저 현재 작업 폴더를 백업하거나 동기화 상태를 확인한다.
- 해설 주석을 추가한 복사본은 보고서 원자료이며 배포 산출물이 아니다.
- 보고서 본문에는 주석을 그대로 붙이지 말고, 주석이 설명하는 모듈 흐름을 기능별 서술로 압축한다.

## 7. 다음 작성 순서

권장 순서:

1. 이 문서의 목차 기준으로 보고서 skeleton을 만든다.
2. `daily-timeline.md`와 git log를 기능군별로 재분류한다.
3. 각 기능군마다 문제, 설계 판단, 구현 방식, 검증, 개선 기록을 bullet로 정리한다.
4. bullet을 외부 제출용 문장으로 확장한다.
5. 필요한 경우 대표 모듈만 해설용 코드 스냅샷에 주석으로 보강한다.
6. 마지막에 현재 완성도와 한계를 과장 없이 정리한다.
