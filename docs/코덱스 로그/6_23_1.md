현재 이해한 내용은 다음과 같습니다.

**현재 작업 상태**

`회귀 코드`는 `regression-code/`를 중심으로 구현 중인 TypeScript + HTML + CSS 웹 앱이고, 기존 `이세계 코드`는 Google Sheets / Apps Script 기반 참고 자료일 뿐 현재 호환성 대상은 아닙니다.

최근 구현의 중심은 다음입니다.

- 트랙 레이어 1차 구현 완료
  - `basic`, `optional`, `extra` 고정 트랙 사용
  - active track은 편집/재생/정상 opacity 대상
  - inactive track은 재생 제외, 화면에는 낮은 alpha로 표시
  - 여러 active track에 같은 좌표 편집을 batch 적용
  - 재생 중 track toggle 비활성화, paused 상태에서는 toggle 후 같은 위치에서 resume

- 오디오 1차 구현 완료
  - `AnalysisResult -> AudioSchedule -> Web Audio` 흐름
  - note, hold, vibrato, tremolo, dynamics 일부, gliss fallback/chain 재생 구현
  - seek/pause/resume/scroll 동기화 연결
  - active track만 audio schedule에 포함

- 글리산도 오디오 개선 진행 중
  - 기존에는 글리산도 구간별로 나뉜 오디오 스케줄을 만들던 방식
  - 현재는 연결된 gliss를 하나의 chain으로 보고 단일 oscillator와 segment-level frequency ramp로 재생하는 방향으로 바뀐 상태
  - 하지만 anchor 위치, 즉 글리산도 시작/중간/끝 연결점 근처에서 여전히 “치직” 하는 artifact가 남아 있음

**문제의 핵심으로 보이는 지점**

현재 남은 문제는 parser/analyzer 문법 자체보다는 오디오 backend 쪽 가능성이 큽니다.

검토해야 할 후보는 다음입니다.

- gliss chain 안에서 frequency automation 경계가 불연속으로 들어가는지
- anchor tick 주변에서 일반 note event와 gliss chain oscillator가 중복 발음되는지
- tremolo gain gate와 gliss frequency ramp가 같은 시점에서 충돌하는지
- note clipping, attack/release envelope, gainScale normalization이 anchor 근처에서 click을 만드는지
- 같은 anchor를 기준으로 이전 segment stop/release와 다음 segment start/attack이 겹치거나 너무 급격한지

**앞으로 예정된 작업**

우선순위는 이렇게 잡는 것이 맞아 보입니다.

1. 현재 connected gliss chain의 치직거림 원인 분리
   - tremolo disabled
   - frequency ramp 단순화
   - gainScale disabled
   - note clipping variant 비교
   - anchor 주변 중복 oscillator 여부 확인

2. 오디오 수동 검증 확대
   - vibrato
   - tremolo
   - gliss fallback
   - connected gliss chain
   - pause/resume
   - seek
   - layout 변경 시 playback reset

3. track layer 브라우저 검증
   - inactive alpha 시인성
   - empty active track 상태
   - multi-track edit overwrite
   - playing 중 toggle lock
   - paused toggle 후 resume

4. 오디오 후속 기능
   - dynamics automation 정교화
   - tuplet timing audio 반영
   - loop playback range
   - 필요 시 Tone.js 또는 sampled backend 검토

5. 장기 확장
   - YouTube/accompaniment sync는 이후 단계
   - sampled instrument나 더 오타마톤에 가까운 음색 개선
   - game mode는 가장 뒤쪽 확장

즉, 지금 바로 이어갈 핵심 작업은 “트랙 레이어 자체 구현”보다는 “트랙 레이어가 적용된 상태에서 gliss chain 오디오 artifact를 분리 진단하고, anchor 주변 scheduling/automation을 안정화하는 것”으로 이해했습니다.

> 트랙 레이어 구현은 꽤 안정화되었다고 판단했습니다.
> 글리산도의 연결이 문제입니다. 트레몰로는 문제가 아닌 듯 합니다. 테스트케이스에서는 트레몰로를 이용하지 않았거든요. 글리산도가 연속적으로 단일 오실레이터로 제대로 변환되었는지, anchor 위치에서 의도하지 않은 gain 변경이 있는지 위주로 점검 부탁드립니다.

현재 열화의 원인은 “가로 CSS px 제한”이라기보다 `canvas_coordinate.ts`의 bitmap DPR clamp입니다.

현재 설정:
- `MAX_CANVAS_BITMAP_DIMENSION = 16_384`
- `MAX_CANVAS_BITMAP_AREA = 67_108_864`
- column width 기본값 `21px`
- 기본 1000 columns면 score CSS width가 `21,000px`

즉 1000열 악보는 이미 `16,384px`를 넘기 때문에 DPR이 자동으로 낮아지고, 그 결과 화질이 흐려집니다.

대략 기준은 이렇습니다.

| 조건 | 선명도 유지 가능 폭 |
| --- | ---: |
| 현재 코드, DPR 2 화면 | 약 `8,192 CSS px` |
| 현재 코드, DPR 1 화면 | 약 `16,384 CSS px` |
| Chrome 계열에서 더 공격적으로 잡을 경우 | 약 `32,000 CSS px` 근처까지 가능할 수 있음 |
| Safari/저사양 환경 포함 안전권 | `16,384px` 근처가 보수적 |

`21px` 열 기준으로 환산하면:
- 현재 DPR 2 선명 유지: 약 `390 columns`
- 현재 DPR 1 선명 유지: 약 `780 columns`
- Chrome 공격 설정 DPR 1 기준: 약 `1,560 columns`

제한을 완전히 제거하는 것은 권하지 않습니다. 브라우저 canvas 최대 bitmap dimension을 넘으면 흐려지는 정도가 아니라 canvas가 비거나 렌더 실패할 수 있습니다.

1차 배포용 임시 조정이라면 두 선택지가 현실적입니다.

1. **보수적 추천**
   - clamp는 유지
   - 대신 사용자 테스트용으로 “너무 긴 악보는 흐릴 수 있음”을 감수
   - 안정성 우선

2. **Chrome/Edge 데스크톱 우선 임시 개선**
   - `MAX_CANVAS_BITMAP_DIMENSION`을 `16_384`에서 `32_767`로 올림
   - area limit도 함께 상향
   - 1000열, 21px 폭의 기본 악보는 훨씬 덜 흐려질 가능성이 큼
   - Safari/저사양 브라우저에서는 리스크 있음

가상화 전 임시 배포라면 2번을 “데스크톱 Chrome/Edge 권장” 조건으로 적용하는 게 가장 체감 개선이 큽니다. 단, 완전 제거는 피하는 쪽이 맞습니다.

> 사용량 9% 안에서 긴 악보(약 5000열정도)의 화질을 확실하게 개선 가능한 방법이 없겠습니까?

5000열을 **확실하게** 선명하게 하려면 단일 canvas 한 장으로는 어렵습니다.

계산상 기본 `21px * 5000열 = 105,000 CSS px`입니다. DPR 2 화면에서는 bitmap 폭이 `210,000px`가 필요합니다. 이건 Chrome/Edge/Safari 모두 단일 canvas 한계 밖입니다. 제한값을 올리는 방식은 1000~1500열 정도에는 도움이 될 수 있지만, 5000열에는 확실한 해결책이 아닙니다.

확실한 방법은 한 가지입니다.

**가로 분할 렌더링**
- 전체 악보를 한 canvas에 그리지 않음
- 예: 300~380열 단위로 tile/page를 나눔
- 현재 화면 근처 tile만 canvas로 생성
- 스크롤하면 필요한 tile만 다시 그리거나 교체
- 각 tile canvas는 `~8000px` 이하로 유지해서 DPR 2에서도 선명도 유지

즉 사실상 “가상화의 최소 버전”입니다. 다만 전체 가상화보다 훨씬 좁게 잡을 수 있습니다.

1차 배포용으로 가능한 최소안은:

1. canvas stage 전체 폭은 그대로 `105000px`로 둔다.
2. 실제 canvas layer는 viewport 폭 + 좌우 buffer 정도만 만든다.
3. scroll 위치가 바뀌면 현재 보이는 column range만 다시 렌더한다.
4. renderer에는 `visibleStartColumn`, `visibleEndColumn`, `xOffset`만 추가한다.

이 방식이면 5000열도 선명도는 유지됩니다. 대신 스크롤 중 재렌더 최적화가 필요해서 “완전 가벼운 조정”은 아닙니다.

반대로 사용량을 아끼기 위한 임시 조정으로는:
- `MAX_CANVAS_BITMAP_DIMENSION`을 키우기: 5000열에는 불충분
- zoom 낮추기: 선명도 개선 아님
- column width 줄이기: 사용감/악보 비율 훼손
- canvas image smoothing 끄기: 텍스트/선명도 문제 해결 안 됨

결론은 이렇습니다.
5000열 화질을 확실히 개선하려면 제한 제거가 아니라 **부분 렌더링/타일 렌더링**이 필요합니다. 다만 지금 9% 안에서 착수한다면 설계만 정하고, 구현은 다음 주에 하는 편이 안전합니다.

> 부분 렌더링은 원리와 알고리즘이 어떻게 되나요? 현재 사용량에서 착수하기엔 아무래도 구현 내용이 많겠습니까?

원리는 단순합니다. **전체 악보 폭은 스크롤 가능한 가상 공간으로만 두고, 실제 canvas는 현재 화면에 보이는 구간만 그리는 방식**입니다.

현재 방식은 대략 이렇습니다.

```text
5000 columns * 21px = 105000px

[ canvas width 105000px ]
전체를 한 번에 그림
```

문제는 canvas bitmap이 너무 커져서 브라우저 한계에 걸리고 DPR이 낮아져 흐려집니다.

부분 렌더링은 이렇게 바꿉니다.

```text
scrollWidth는 105000px 유지

실제 canvas width는 viewport + buffer만 사용
예: 1600px 화면 + 좌우 800px buffer = 3200px

[ 가상 스크롤 공간 105000px ]
             [ 실제 canvas 3200px ]
             현재 보이는 부분만 그림
```

알고리즘은 이렇습니다.

1. 전체 score width 계산
   - `totalWidth = columnCount * columnWidth`
   - 이 값은 scroll 영역의 가상 폭으로만 사용합니다.

2. 현재 scroll 위치 읽기
   - `scrollLeft = scoreArea.scrollLeft`
   - `viewportWidth = scoreArea.clientWidth`

3. 렌더링할 column 범위 계산
   - `bufferPx = viewportWidth` 또는 고정값 `1000px`
   - `renderLeftPx = max(0, scrollLeft - bufferPx)`
   - `renderRightPx = min(totalWidth, scrollLeft + viewportWidth + bufferPx)`
   - `startColumn = floor(renderLeftPx / columnWidth)`
   - `endColumn = ceil(renderRightPx / columnWidth)`

4. 실제 canvas 위치 지정
   - canvas CSS `left = renderLeftPx`
   - canvas CSS `width = renderRightPx - renderLeftPx`
   - canvas bitmap도 이 폭만큼만 생성

5. 그릴 때 좌표 보정
   - 기존 x 좌표는 전체 악보 기준입니다.
   - canvas 내부에서는 `drawX = globalX - renderLeftPx`
   - 즉 `xOffset = renderLeftPx`를 빼고 그립니다.

6. 스크롤 시 재렌더
   - scroll event마다 바로 그리면 과하니 `requestAnimationFrame`으로 throttle
   - 현재 render 범위 안에서 조금 움직인 정도면 다시 그리지 않음
   - buffer 바깥으로 나가면 새 범위로 다시 렌더

핵심 변화는 renderer가 “전체 score”가 아니라 “현재 보이는 window”를 받아야 한다는 점입니다.

필요한 타입은 대략 이런 모양입니다.

```ts
type RenderViewport = {
  leftPx: number;
  widthPx: number;
  startColumn: number;
  endColumn: number;
};
```

그리고 renderer 내부에서:

```ts
const localX = columnToX(col, layout) - viewport.leftPx;
```

현재 사용량 안에서 착수하기엔 구현량이 꽤 있습니다. 이유는 단순히 canvas width만 줄이는 문제가 아니라 다음 경계를 건드립니다.

> 악보 해상도는 과하게 선명할 필요는 없겠습니다. 세로가 720 정도만 되어도 충분합니다. 이러면 렌더 가능 길이가 조금 더 늘어날까요?

네, 세로 해상도를 `720px` 정도로 제한하면 렌더 가능 길이는 꽤 늘어납니다. 다만 핵심 한계는 여전히 **가로 bitmap 폭**입니다.

현재 기본 template은:

- CSS 크기: `21000 x 1033px`
- DPR 2 bitmap: `42000 x 2066px`

세로 bitmap을 720px로 제한하면 effective scale은:

```text
720 / 1033 = 약 0.697
```

이때 1000열은:

```text
21000 * 0.697 = 약 14637px
```

즉 1000열은 안정권에 들어갑니다.

5000열은:

```text
105000 * 0.697 = 약 73185px
```

이건 여전히 브라우저에 따라 위험합니다. Chrome/Edge도 환경에 따라 canvas 최대 가로가 `32767px` 또는 `65535px` 근처에서 막힐 수 있기 때문입니다.

대략 계산하면:

| 기준 | 세로 720px 스케일에서 가능한 열 수 |
| --- | ---: |
| 가로 한계 `32767px` | 약 `2200열` |
| 가로 한계 `65535px` | 약 `4400열` |
| 5000열 필요 폭 | 약 `73185px` |

그래서 세로 720 제한은 **1000~4000열대까지는 체감 개선 가능**하지만, **5000열을 확실히 보장하지는 못합니다.**

임시 배포용으로는 현재 “제한 완전 해제”보다 다음 정책이 더 낫습니다.

```text
effectiveDpr = min(
  devicePixelRatio,
  720 / cssHeight,
  65535 / cssWidth
)
```

이러면:
- 세로는 과하게 선명하지 않게 제한
- 가로는 브라우저 실패 가능성을 조금 줄임
- 5000열도 Chrome/Edge에서는 약 `65535 x 645` 정도로 시도 가능
- Safari/저사양에서는 여전히 실패 가능

결론: 세로 720 제한은 좋은 절충안입니다. 5000열 완전 보장은 아니지만, 현재보다 훨씬 현실적인 임시 정책입니다.

> 로딩은 됐는데 악보와 레이아웃 영역이 비었네요. 무슨 작업을 수행해야 할까요?  Actions 항목으로 들어가 보니까 Webpack이나 Publish Node.js Package to GitHub Packages 등을 Configure하는 버튼이 뜨는데, 이게 필요한가요?

Webpack이나 `Publish Node.js Package to GitHub Packages`는 필요 없습니다.
이 앱은 **Vite 정적 사이트**라서 해야 할 일은 “빌드된 `dist/`를 GitHub Pages에 배포”입니다.

지금 “로딩은 됐는데 악보/레이아웃 영역이 비어 있음”이면 가장 먼저 의심할 것은 **소스 파일을 그대로 Pages에 올린 상태**입니다. `index.html`은 열리지만 브라우저가 TypeScript/Vite import를 그대로 실행하지 못해서 앱 일부가 깨질 수 있습니다.

해야 할 작업은 둘 중 하나입니다.

**방법 A: 로컬에서 빌드해서 dist만 배포**
1. `regression-code-test-deploy`에서:
   ```bash
   npm install
   npm run build
   ```
2. 생성된 `dist/` 안의 파일들을 GitHub Pages 배포 대상으로 올립니다.
3. Pages가 `dist/index.html`을 제공하게 합니다.

**방법 B: GitHub Actions로 자동 빌드/배포**
이게 더 추천입니다. Actions에서 Webpack 템플릿을 고르는 게 아니라, 직접 Vite용 workflow를 추가해야 합니다.

`.github/workflows/deploy.yml` 파일을 만들고 이런 내용으로 둡니다.

```yaml
name: Deploy GitHub Pages

on:
  push:
    branches: [main]
  workflow_dispatch:

permissions:
  contents: read
  pages: write
  id-token: write

concurrency:
  group: pages
  cancel-in-progress: true

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: npm

      - name: Install
        run: npm ci

      - name: Build
        run: npm run build

      - name: Upload Pages artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: dist

  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    runs-on: ubuntu-latest
    needs: build
    steps:
      - name: Deploy
        id: deployment
        uses: actions/deploy-pages@v4
```

그 다음 GitHub에서:

1. Repository `Settings`
2. `Pages`
3. `Build and deployment`
4. Source를 **GitHub Actions**로 선택
5. `main`에 push

정리하면:
- Webpack configure 필요 없음
- Node package publish 필요 없음
- Vite build 필요
- GitHub Pages는 `dist/`를 배포해야 함

빈 화면이 계속되면 브라우저 개발자도구 Console에 빨간 에러가 있을 가능성이 큽니다. 그 메시지를 보면 경로 문제인지, 빌드 누락인지 바로 확인할 수 있습니다.

> 타입스크립트라서 생긴 문제군요. 오류도 Failed to load module script: Expected a JavaScript-or-Wasm module script but the server responded with a MIME type of "video/mp2t". Strict MIME type checking is enforced for module scripts per HTML spec.입니다.

네, 정확히 그 문제입니다.

브라우저가 `src/app/main.ts` 같은 TypeScript 파일을 직접 module script로 불러오고 있고, GitHub Pages는 `.ts` 확장자를 JavaScript가 아니라 다른 MIME type으로 응답하고 있습니다. 그래서 이 오류가 납니다.

```text
Failed to load module script...
MIME type of "video/mp2t"
```

해결은 TypeScript 소스를 직접 배포하지 않고, **Vite로 빌드한 결과물인 `dist/`를 배포**하는 것입니다.

필요한 작업:
1. 로컬 또는 GitHub Actions에서:
   ```bash
   npm install
   npm run build
   ```
2. GitHub Pages가 `dist/index.html`과 `dist/assets/*.js`를 제공하게 설정

소스의 `index.html`에는 이런 식으로 되어 있습니다.

```html
<script type="module" src="./src/app/main.ts"></script>
```

이건 개발 서버용입니다.
Vite build 후 `dist/index.html`에서는 자동으로 이런 식으로 바뀝니다.

```html
<script type="module" crossorigin src="./assets/index-xxxxx.js"></script>
```

GitHub Pages에는 이 `dist` 결과물이 올라가야 합니다.

따라서 지금 필요한 것은 Webpack 설정이 아니라 **Vite GitHub Pages 배포 workflow**입니다.
원하시면 제가 `regression-code-test-deploy/.github/workflows/deploy.yml` 파일을 바로 만들어드리면 됩니다.

**발견된 문제**

1. 긴 악보 편집 지연

편집이 발생할 때 현재 흐름은 대체로 다음 전체 재계산을 수행합니다.

```text
ScoreFile mutation
-> createRuntimeDocument()
-> buildParsedDocument()
-> analyzeDocument()
-> renderer item rebuild
-> renderCanvasScore()
```

짧은 악보에서는 단순하고 안정적인 장점이 있지만, 열 수가 커질수록 셀 수, analyzer 이벤트 수, canvas draw 범위가 함께 커져 편집 1회마다 큰 지연이 발생합니다.

해결 방향은 단계적으로 나누는 것이 좋습니다.

- 1차: 편집 중 full rebuild 호출 빈도 줄이기
- 2차: 변경된 column 주변만 parse/analyze/render하는 제한적 partial update
- 3차: row/track/column 단위 dirty range 관리
- 4차: viewport/tile renderer와 결합

2. 5000열 이상 렌더링 실패

현재 임시 방책은 canvas bitmap 크기를 제한하거나 DPR을 낮춰서 너무 큰 bitmap 생성을 피하는 방식입니다. 이 방식은 화질과 렌더 가능 길이 사이의 타협일 뿐, 근본 해결은 아닙니다.

문제의 본질은 브라우저 canvas가 무한히 큰 bitmap을 안정적으로 만들 수 없다는 점입니다. 긴 악보를 하나의 거대한 canvas로 그리면 다음 문제가 발생합니다.

- 브라우저별 최대 canvas 크기 제한
- GPU/메모리 한계
- DPR 적용 시 실제 bitmap 폭/높이 폭증
- 전체 redraw 비용 증가
- 실패 시 빈 화면, 깨짐, 심한 렉 발생

해결 방향은 **전체 악보 canvas 하나**에서 벗어나는 것입니다.

- 1차: 현재 viewport 주변 열만 그리는 bounded viewport rendering
- 2차: 열 범위를 tile 단위로 나누어 필요한 tile만 렌더
- 3차: 스크롤 위치에 따라 tile 생성/폐기/cache
- 4차: hit-test, playback scroll, edit preview가 viewport/tile 좌표계를 공유하도록 정리

**정리**

지금 임시 제한은 “첫 테스트 배포를 가능하게 하는 안전장치”이고, 다음 본 작업은 사실상 다음 두 축입니다.

- 편집 성능: full rebuild를 줄이는 partial update
- 렌더 안정성: 거대 단일 canvas를 없애는 viewport/tile rendering
