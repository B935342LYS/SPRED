# 1.10. 1단계 Build Parsed Document 구현 메모
> 본 설계 메모 문서 1.10에서는 `build_parsed_document.ts`의 첫 구현 범위를 기록한다.
> 본 문서는 active specification을 대체하지 않는다.
> 함수별 세부 설명은 코드 파일의 `/** ... */` 주석을 중심으로 하고, 본 문서는 흐름과 구현 판단을 보충한다.

## 1. 개요

단일 셀 parser 구현 이후, 문서 전체의 rawText 셀을 analyzer가 소비할 `ParsedScoreDocument` 구조로 묶는 단계를 추가했다.
이 단계는 parser의 마지막 조립 단계이며, 셀 간 의미 분석은 수행하지 않는다.

현재 흐름은 아래와 같다.

```text
RuntimeDocument
-> score + indexes
-> buildParsedDocument(score, indexes)
   -> buildParsedNoteCells()
   -> buildParsedGlobalCells()
-> ParsedScoreDocument
```

## 2. 구현 범위

현재 구현한 공개 진입점은 `buildParsedDocument(score, indexes)`이다.

반환 구조는 명세의 `ParsedScoreDocument`를 따른다.

```text
noteCellsByTrackAndCol
  TrackId -> col -> ParsedCellEntry[]

globalCellsByKindAndCol
  GlobalKind -> col -> ParsedGlobalCellEntry
```

note 셀은 같은 track과 col에 여러 row의 셀이 동시에 있을 수 있으므로 배열로 누적한다.
global 셀은 같은 kind와 col에 하나만 존재한다는 전제를 validator가 보장하므로 단일 entry로 저장한다.

## 3. 구현 판단

### 3.1. 원본 배열 순회 기준

문서 단위 parser는 `ScoreFile`의 원본 배열을 기준으로 순회한다.
`ScoreIndexes`는 조회 보조 구조이므로, parser 결과의 진실 원천을 인덱스에 두지 않는다.

이 방식은 원본 문서에 담긴 셀 순서와 좌표 정보를 그대로 유지한다.

### 3.2. analyzer 입력 축과 맞춘 Map 구조

note 결과는 `track -> col -> entries` 축으로 만든다.
이는 track 단위 분석과 이후 partial parse 범위의 기본 축이 된다.

global 결과는 `kind -> col -> entry` 축으로 만든다.
이는 전역 timing/dynamics 분석이 kind별 스트림으로 진행될 것을 전제로 한다.

### 3.3. 의미 분석 제외

`buildParsedDocument()`는 다음 작업을 하지 않는다.

- hold 연결 판정
- gliss 연결 판정
- pletHead와 `/&` 결합
- tuplet slot의 실제 시간 계산
- `@n(midi_num)`의 실제 rowId 연결

이 작업들은 analyzer 단계의 책임이다.

## 4. 이번 구현에서 사용한 주요 TypeScript 문법

### 4.1. 인덱스 접근 타입

함수 반환 타입은 `ParsedScoreDocument["noteCellsByTrackAndCol"]`처럼 기존 타입의 필드 타입을 직접 참조한다.

```ts
function buildParsedNoteCells(
  score: ScoreFile,
  indexes: ScoreIndexes,
): ParsedScoreDocument["noteCellsByTrackAndCol"]
```

이 방식은 `ParsedScoreDocument` 필드 타입이 바뀌었을 때 helper 함수 타입도 함께 따라가게 만든다.

### 4.2. Map 배열 누적

같은 col에 여러 note 셀이 들어올 수 있으므로 `appendMapArray()`로 배열 누적을 공통화했다.

```text
Map<K, V[]>
```

key가 이미 있으면 기존 배열에 push하고, 없으면 새 배열을 만들어 넣는다.

### 4.3. 전역 행 타입 좁히기

global entry에는 `GlobalKind`가 필요하므로, `rowById`에서 찾은 row가 전역 행인지 확인한다.
현재 validator가 전제를 보장하지만, 문서 parser 내부에서도 `asGlobalRow()`를 통해 전역 행만 사용한다.

### 4.4. `Map` 중첩 구조

문서 parser 결과는 object literal이 아니라 `Map`을 중첩해서 만든다.

```ts
Map<TrackId, Map<number, ParsedCellEntry[]>>
Map<GlobalKind, Map<number, ParsedGlobalCellEntry>>
```

`Map`은 key 타입이 문자열로 강제되지 않고, 숫자 key를 그대로 사용할 수 있다.
악보 column은 숫자 좌표이므로 `Map<number, ...>` 형태가 자연스럽다.
또한 analyzer 단계에서 특정 track, global kind, column을 기준으로 빠르게 조회할 수 있다.

### 4.5. 배열 tuple로 `Map` 초기화

전역 kind map은 네 종류의 key가 항상 있어야 하므로 배열 tuple로 초기화한다.

```ts
return new Map<GlobalKind, Map<number, ParsedGlobalCellEntry>>([
  ["bpm", new Map()],
  ["beatsPerBar", new Map()],
  ["stepsPerBeat", new Map()],
  ["dynamics", new Map()],
]);
```

`new Map([...])`은 `[key, value]` 쌍의 배열을 받아 초기 값을 만든다.
이 방식은 이후 `result.get(row.kind)`가 항상 존재한다는 전제를 만들기 쉽다.

### 4.6. 선택적 체이닝

global cell 누적에는 선택적 체이닝을 사용한다.

```ts
result.get(row.kind)?.set(cell.col, entry);
```

`?.`는 왼쪽 값이 `null` 또는 `undefined`이면 뒤의 호출을 실행하지 않는다.
현재 구현에서는 `createEmptyGlobalKindMap()`이 모든 global kind를 만들기 때문에 실제로는 `undefined`가 나오지 않아야 한다.
그래도 타입 관점에서는 `Map.get()`이 `undefined` 가능성을 가지므로, 선택적 체이닝으로 호출을 안전하게 표현했다.

### 4.7. `void` 연산자

`buildParsedNoteCells()`에는 아래 코드가 있다.

```ts
void indexes;
```

현재 note cell 파싱 경로에서는 `indexes`를 직접 사용하지 않는다.
하지만 공개 helper의 인수 구조를 `buildParsedDocument(score, indexes)` 흐름과 맞춰 두기 위해 인수를 유지했다.
`void indexes`는 이 값이 의도적으로 사용되지 않는다는 표시이며, unused variable 경고를 피하는 데도 쓰인다.

### 4.8. 제네릭 helper 함수

`appendMapArray<K, V>()`는 key 타입과 value 타입을 호출 지점에서 추론하게 만든다.

```ts
function appendMapArray<K, V>(
  map: Map<K, V[]>,
  key: K,
  value: V,
): void
```

이 함수는 `Map<number, ParsedCellEntry[]>`뿐 아니라 다른 `Map<K, V[]>`에도 재사용할 수 있다.
현재는 note cell 누적에 사용하지만, 같은 패턴이 다른 parser 결과 누적에도 필요해지면 그대로 사용할 수 있다.

### 4.9. `unknown`에서 타입 좁히기

`asGlobalRow(row: unknown)`은 알 수 없는 값을 바로 type assertion하지 않고 먼저 구조를 검사한다.

```ts
if (
  typeof row === "object" &&
  row !== null &&
  "type" in row &&
  row.type === "global"
) {
  return row as GlobalRowDefinition;
}
```

`unknown`은 어떤 값인지 모르는 타입이므로 속성 접근 전에 object 여부와 null 여부를 확인해야 한다.
`"type" in row`는 해당 속성이 있는지 검사하면서 이후 분기에서 `row.type` 접근을 가능하게 만든다.
마지막 `as GlobalRowDefinition`은 런타임 검사를 통과한 뒤 전역 행으로 취급하겠다는 타입 단언이다.
