진단 결과입니다. 원인은 threshold보다 구조 쪽이 큽니다.

**1. Gliss 문제**

현재 Gliss bonus는 실제 음정 오차를 보지 않습니다.

[game_effect_judge.ts](./regression-code/src/app/game/game_effect_judge.ts:123)의 `judgeGlissIntervalBonus()` 조건은 사실상:

- 현재 frame이 voiced인가
- 현재 시간이 gliss 구간 안인가
- 현재 interval 안에 기록된 unvoiced frame이 있는가

까지만 봅니다. `frame.midi / centOffset`이 gliss 예상 음정과 가까운지는 검사하지 않습니다. 그래서 anchor note 판정에서 `Miss`가 나도, 마이크 입력이 voiced이기만 하면 `Gliss!`가 나올 수 있습니다.

또 끊김 검사가 잘 안 되는 이유는 [app_playback_binding.ts](./regression-code/src/app/playback/app_playback_binding.ts:302)에서 practice scoring이 `0.1초`마다만 실행되고, effect frame history도 그때만 쌓이기 때문입니다. 즉 100ms 사이에 음이 끊겼다가 다시 voiced가 되면, 끊긴 frame 자체가 history에 안 들어갈 수 있습니다.

**2. Vib 문제**

Vib도 같은 이유가 핵심입니다.

Vib 판정은 [game_effect_judge.ts](./regression-code/src/app/game/game_effect_judge.ts:169)에서 최근 frame들의 흔들림 폭과 방향 전환을 봅니다. 그런데 frame history가 실제 pitch input의 60fps/RAF 기록이 아니라, playback scoring의 10Hz 기록입니다.

10Hz 샘플링은 vibrato 감지에 너무 낮습니다. 특히 5-7Hz 정도로 흔들면 샘플링 타이밍에 따라 항상 비슷한 위상만 잡히거나, 방향 전환이 사라지는 aliasing이 발생합니다. 그래서 `+-10cent`로 낮춰도 안 잡힐 가능성이 큽니다.

**수정 방향**

- Gliss:
  - effect frame history를 pitch input callback에서 직접 쌓아야 합니다.
  - Gliss interval 내부 frame마다 예상 gliss pitch와의 pitch-class error를 검사해야 합니다.
  - anchor 근처는 더 엄격하게 보거나, anchor note sample이 `Miss`면 해당 gliss interval/segment bonus를 막아야 합니다.

- Vib:
  - Vib 판정용 frame history를 10Hz scoring tick이 아니라 pitch input RAF 기준으로 쌓아야 합니다.
  - 그 다음 `+-10cent` 기준으로 `min <= -10`, `max >= +10`, 방향 전환 2회 이상 같은 조건을 적용해야 합니다.

현재 상태에서는 Gliss/Vib 모두 `effect 판정 입력 데이터가 너무 듬성듬성하다`가 1차 원인이고, Gliss는 추가로 `음정 정확도 조건이 아예 없다`가 직접 원인입니다.

> 직접 원인을 먼저 확실하게 고치고, 판정 입력 데이터는 연산에 부담이 안 가는 수준에서 끊김을 확실히 인식할 수 있는 기준을 먼저 결정해 봅시다.

  https://github.com/tiagolbc/vibratoscope
- **VoceVista**: 상용/검증 프로그램 쪽입니다. vibrato를 “fundamental pitch의 주기적 fluctuation”으로 보고, rate와 extent를 실시간/평균값으로 표시합니다. 안내 문서도 sustained single pitch에서 가장 정확하다고 합니다.

현재 우리 로직의 의심점:
- 평균 음정 일치율은 넣었지만, **느린 drift/portamento와 빠른 vibrato 성분을 분리하지 않습니다.**
- 방향 전환 카운트는 잡음이나 detector 튐에 취약하고, 실제 vibrato의 “주기성”을 충분히 보지 못합니다.
- 검증 프로그램들은 보통 pitch contour에서 **평균 pitch 또는 저주파 drift를 제거한 뒤**, 남은 `3-9Hz` 근처 흔들림의 extent/rate를 봅니다.

다음 구현 기준은 이렇게 바꾸는 게 맞겠습니다.

1. Vib 구간 frame 수집
- 최근 `0.8-1.0초`
- voiced ratio 최소 `70%`
- target에서 너무 먼 frame은 제외 또는 실패 처리

2. 평균 음정 확인
- signed cent error의 평균 또는 median이 target 근처인지 확인
- 예: median abs error `<= 45cent`, in-tune ratio `>= 70%`

3. drift 제거
- 단순 평균만 빼지 말고, window 안의 선형 추세를 제거합니다.
- 이렇게 해야 천천히 올라가거나 내려가는 음정 이동을 Vib로 오인하지 않습니다.

4. 진폭 측정
- detrended contour의 half peak-to-peak 또는 RMS 기반 extent 측정
- 사용자 요청대로 우선 `>= 10cent`를 성공 기준으로 둡니다.

5. 주기성 확인
- 방향 전환 대신 zero-crossing 또는 peak/trough 간격으로 `3-9Hz` 범위인지 봅니다.
- 최소 1.5-2 cycle 정도만 요구하면 게임 판정용으로 충분합니다.

즉 방향은 맞지만, 현재 구현은 “진폭 + 평균 음정”에 **drift 제거와 rate 범위 확인**이 빠져 있어서 실제 Vib가 안 잡힐 가능성이 큽니다. 다음 패치에서는 VibratoScope/VoceVista식으로 `pitch contour -> target 기준 cent contour -> trend 제거 -> extent/rate 판정` 흐름으로 바꾸는 게 좋습니다.

추천은 **레이아웃 영역 또는 별도 고정 HUD로 옮기는 방식**입니다.

세 선택지 중에서는:

1. **레이아웃 영역으로 표시 위치 옮기기**
   가장 낫습니다. 악보 note cell을 가리지 않고, 충돌 계산도 필요 없습니다. row label을 잠깐 가리는 정도는 practice 중 허용 가능하고, `Perfect / Ok / Bad / Miss`, `COMBO` 같은 정보는 음표 자체보다 “피드백 HUD” 성격이 강합니다.

2. **판정 표시 위치를 악보 영역 우측 y중앙으로 고정**
   구현은 쉽지만, 긴 악보에서 앞으로 올 note를 가릴 수 있습니다. 특히 오른쪽은 사용자가 곧 볼 영역이라 별로 좋지 않습니다.

3. **글자 오프셋 조절 + 반투명**
   보조책으로는 좋지만 근본 해결은 아닙니다. 반투명이어도 시선 방해가 남습니다.

제가 추천하는 최소 구현안은:

- `COMBO`, `Perfect/Ok/Bad/Miss`, `early/late`는 **layout label 영역 쪽 고정 HUD**로 이동
- `Gliss! / Vib! / Trem!` 같은 effect bonus는 지금처럼 note 근처 또는 약간 오른쪽에 작게 유지
- HUD는 practice mode일 때만 표시
- 위치는 layout label 영역의 세로 중앙 또는 score viewport 상단 좌측에 고정
- 글자는 살짝 반투명 + text shadow 유지

즉, “정확히 어느 note에서 났는지”보다 “지금 판정이 무엇인지”가 중요한 main judge는 악보 밖으로 빼고, effect bonus만 note 근처에 남기는 하이브리드가 좋겠습니다. 구현 비용도 낮고, 노트 가림 문제도 거의 사라집니다.
