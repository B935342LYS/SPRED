# 파서 중간 산출물 데이터 구조 설계 : 노트 행

## 1. 문서 목적

- 본 문서는 회귀 코드의 parser가 노트 행의 셀 문자열 `rawText`를 해석하여 생성하는 1차 산출물의 구조를 정의한다.
- 이 산출물은 문자열의 문법 해석 결과이며, 아직 인접 셀 관계나 시간축 의미가 확정된 상태는 아니다.
- parser 출력은 주로 analyzer가 소비하는 공통 기반 자료구조이며, renderer와 audio generator는 analyzer의 후속 산출물을 소비한다.
- validator는 parser 출력이 아니라 JSON 문서 구조와 참조 무결성을 주로 검증한다.
- 본 문서는 `1.4-note-string-spec.md`를 직접 이어받으며, deleted transitional document였던 옛 `1.6`의 핵심 내용은 현재 `1.3`, `1.7`, `1.8`에 흡수된 상태를 전제로 한다.

## 2. parser의 위치와 책임

회귀 코드의 데이터 흐름은 아래 순서를 따른다.

1. JSON 파일 로드
2. 문서 구조 검증
3. 셀 문자열 파싱
4. 셀 간 관계 분석
5. 렌더/오디오용 이벤트 생성

각 단계의 책임은 다음과 같이 고정한다.

### 2.1. validator가 책임지는 것

- JSON 구조 검증
- 필수 필드 누락 검증
- `rowId`, `trackId`, `stringId` 참조 무결성 검증
- `globalLines.cells` 좌표 중복 검증
- 같은 좌표 중복 검증
- `gap` 행에 셀이 저장된 경우 검증

### 2.2. parser가 책임지는 것

- 셀 문자열 종류 판별
- escape 해석
- 토큰 순서 검증
- 예약문자 충돌 검증
- 괄호/인수 개수 검증
- 셀 단독으로 판정 가능한 중복 modifier 검증
- 표시용 문자열 `displayText` 생성

### 2.3. analyzer가 책임지는 것

- hold 연결 성립 여부 판정
- gliss 연결 상대 탐색
- tuplet head와 extend 결합
- `@n(midi_num)`를 실제 행으로 해석
- `@p(midi_num)`와 현재 문맥의 관계 검증
- 전역 셀을 시간/셈여림 세그먼트로 확정
- renderer/audio가 소비할 이벤트 리스트 생성

즉 parser는 "문자열 해석"까지만 담당하고, 셀 간 관계는 analyzer로 넘긴다.

## 3. 설계 원칙

- parser 산출물은 JSON으로 직렬화 가능한 순수 데이터 구조여야 한다.
- parser는 셀 단위 처리만 수행한다.
- parser는 손상된 문자열에 대해 오류 복구를 시도하지 않고 invalid 결과를 반환한다.
- parser 출력은 TypeScript에서 안전하게 다루기 쉬운 discriminated union 구조를 사용한다.
- JSON 파일에는 `parsedCell`을 저장하지 않고, 런타임에서만 생성한다.
- UI 입력은 장기적으로 `parsedCell`에 가까운 내부 상태를 직접 다루고, 저장 시점에만 `rawText`를 합성하는 방향을 전제한다.

## 4. 설계 해설

### 4.1. 왜 처음부터 JSON에 parsedCell 구조를 저장하지 않는가?
 다음 이유에서 JSON 내부에는 rawText가 필요하다고 판단된다.

1) 저장 포맷과 런타임 해석 구조의 분리를 위해서
- 두 구조를 분리하면 저장 포맷은 안정성/호환성/직렬화 난이도에, 런타임 산출물은 조회 속도/부분 갱신/캐시 효율에 맞추어 각각 최적화할 수 있다.
- 하나의 구조로 저장과 실행을 동시에 만족시키려고 할 때보다 전체 구현 복잡도와 유지보수 비용을 낮출 수 있을 것으로 기대된다.

2) 파서 구조 변경시 호환이 깨질 위험성을 감소시키기 위해
- 파서의 세부 구조, 오류 코드, 필드 등은 개발 과정에서 바뀔 수도 있는 부분이다.
- parsedCell 자체를 저장하면 이전 버전 JSON의 호환이 깨지는 문제가 발생할 수 있다.
- rawText만 저장하면 새 버전의 파서에서 이전 버전의 rawText에도 대응하여 해석할 수 있다.

3) JSON 파일 크기와 중복 정보를 줄이기 위해
- parsedCell은 rawText로부터 유추 가능하다.
- 따라서 정보의 중복이며, parsedCell은 포함하고 있는 정보가 많아 파일 크기가 수 배로 늘어난다. 


### 4.2. 왜 어휘 분석기(Lexer)나 정석적인 파싱 알고리즘을 사용하지 않는가?
 노트 문자열 parser는  lexer -> parser 2단 구조 대신, 문자 단위 스캔과 해석을 결합한 방식으로 설계하였다. 이유는 다음과 같다.

1) 셀 단위 입력의 길이가 매우 짧기 때문
-  본 프로그램의 파서는 악보의 셀 하나에 들어가는 짧은 문자열을 처리한다.
-  따라서 일반적인 프로그래밍 언어 처리처럼 복잡한 토큰 스트림이나 대규모 구문 트리를 구성할 필요성이 낮다.

2) 문법 구조가 선형적이며 canonical order가 정해져 있다.
- 노트 문자열은 재귀, 우선순위, 결합법칙을 가지지 않으며, 고정 순서의 선택적 토큰을 순차적으로 읽는 형태이다.
- 따라서 재귀 하강 파서나 일반  문법 파서보다 선형 스캐너 방식이 더 단순하고 구현하기 쉬울 것으로 판단된다.

3) 현재 목표 범위에 비해 구조 복잡도를 너무 높일 필요가 없기 때문이다.
- lexer를 사용하면 이론적으로 더 체계적인 구조를 확립할 수는 있겠으나 현재 노트 문자열 문법은 아주 단순한 구조이다.
- 따라서 현 단계에서는 단순한 문자 스캔식 파서가 구현 난이도와 유지보수 측면에서 적합하리라 생각된다.


### 4.3. 왜 nullable wrapper 대신 union type을 쓰는가
기본 초안은 아래와 같은 구조를 가지고 있었다.
```ts
type ParsedCell = {
  kind: "mute" | "note" | "pletHead" | "pletExtend" | "invalid";
  rawText: string;


  mute: ParsedMuteCell | null;
  note: ParsedNoteCell | null;
  pletHead: ParsedPletHeadCell | null;
  pletExtend: ParsedPletExtendCell | null;


  parseErrorCode: ParseErrorCode | null;
};
```
 이를 바탕으로 수도 코드를 작성해본 결과, 이는 다소 불편한 방식이라고 판단했다.

1) 분기 후에도 불필요한 null 검사 코드를 반복한다.
- 예를 들어 `kind == "note"`라고 이미 판정한 뒤에도 `note != null` 확인이 다시 필요해질 수 있다.
- 이로 인해 파서 이후의 analyzer, renderer 코드가 장황해지고 가독성이 떨어진다.

2) 논리적으로 불가능한 상태를 타입 수준에서 막기 어렵다.
- 예를 들어 `kind == "mute"`인데 동시에 `note != null`인 구조는 설계상 허용되면 안 된다.
- 이러한 논리적 불가능 상황을 코드 작성 과정에서 수동으로 일일이 막아주어야 한다.

3) 후속 모듈의 사용성이 떨어진다.
- analyzer나 renderer는 `ParsedCell`의 종류에 따라 서로 다른 필드를 사용한다.
- 이때 union type을 사용하면 `switch (parsedCell.kind)`와 같은 분기만으로 해당 타입이 자동으로 좁혀진다.
- 반면 위의 방식은 종류 분기 이후에도 여러 필드가 여전히 nullable로 남아 사용 코드가 번거로워진다.

4) 설계 의도를 직접적으로 반영하지 못한다.
- 본 프로그램의 ParsedCell은 “다섯 종류 중 하나”의 속성을 가지지, “복수의 속성을 동시에 가질 수 있는”  객체는 아니다.
- 이 배타적 관계를 표현하기에 위의 구조는 다소 적합하지 못하다.
- discriminated union이 이러한 관계를 표현하기에 적합하다고 판단되었다.

 이러한 이유로 위의 방식 대신 discriminated union 방식을 채택한다. 예를 들면 `kind: "mute"`인 경우에는 `ParsedMuteCell`, `kind: "note"`인 경우에는 `ParsedNoteCell`만 존재하도록 정의하는 방식이다.

 이 선택을 통해 이후의 analyzer, renderer, audio generator의 구현을 더 안전하고 단순하게 할 수 있을 것으로 기대하고 있다.

## 5. ParsedCell이 담아야 할 정보

Parser의 출력인 `ParsedCell`은 단순히 문자열을 종류별로 분류하는 것에서 끝나지 않는다. 이후 analyzer, renderer, audio generator가 공통으로 활용할 수 있도록, 셀 단위에서 확정 가능한 최소한의 정보를 함께 담아야 한다.  
즉 `ParsedCell`은 “문자열을 문법적으로 해석한 1차 결과”이자, 후속 단계가 재사용할 수 있는 셀 단위 중간 산출물이다.

현재 설계에서 `ParsedCell`이 담아야 할 핵심 정보는 다음과 같다.

### 5.1. 셀의 종류

- 해당 셀이 `mute`, `note`, `pletHead`, `pletExtend`, `invalid` 중 무엇인지 나타내야 한다.
- 이후 처리 로직은 셀 종류에 따라 크게 달라지므로, 최상위 분기 정보가 반드시 필요하다.
- 예를 들어 `mute`는 음악적 발음을 가지지 않지만, `note`는 hold와 modifier를 가질 수 있고, `pletHead`는 내부 slot 배열을 가져야 한다.

### 5.2. 원본 문자열 `rawText`

- JSON 파일에 저장된 셀의 원본 문자열을 그대로 유지해야 한다.
- 이는 디버깅, 오류 보고, UI 복원, 재직렬화에 필요하다.
- 또한 parser 구조가 바뀌더라도 원본 입력을 기준으로 다시 해석할 수 있어야 하므로, `rawText`는 중요한 기준 정보이다.

### 5.3. 표시용 문자열 `displayText`

- renderer가 셀에 무엇을 표시해야 하는지 판단할 수 있도록 표시용 문자열이 필요하다.
- 예를 들어 note 셀의 기본 텍스트, mute 셀의 주석 텍스트, tuplet slot 내부의 표시 텍스트가 이에 해당한다.
- 이 값을 parser 단계에서 미리 구해두면, 후속 단계가 문자열을 다시 해석하지 않아도 된다.

### 5.4. hold 정보

- note 셀 또는 tuplet slot이 `-` 또는 `~`를 포함하는지 기록해야 한다.
- hold 표식의 존재 자체는 셀 단독으로 판정 가능하므로 parser가 기록할 수 있다.
- 다만 이 hold가 실제로 앞 셀과 연결되는지는 주변 셀과의 관계를 봐야 하므로 analyzer가 판단한다.

### 5.5. modifier 정보

- gliss, trem, absolute pitch, micro pitch와 같은 modifier들은 후속 분석과 렌더링의 공통 입력이 되므로 구조적으로 기록해야 한다.
- parser는 modifier의 존재 여부와 인수 값을 기록하고, 실제 의미 확정은 analyzer에 넘긴다.
- 예를 들어 `@g(id, glissKind)`의 연결 상대 탐색이나 `@p(midi_num)`의 실제 의미 해석은 parser가 아니라 analyzer의 역할이다.

### 5.6. tuplet 관련 정보

- `pletHead`인 경우 몇 분할인지(`divNum`)와 각 slot의 내용을 담아야 한다.
- 각 slot은 쉼표인지, 노트인지, 위치 지정 토큰 `@n(midi_num)`을 가지는지 등의 정보를 포함해야 한다.
- 반면 `pletExtend`는 자체 정보가 많지 않지만, 이후 analyzer가 앞선 head와 결합하기 위해 셀 종류 정보가 필요하다.

### 5.7. 오류 정보

- 문자열이 문법적으로 잘못된 경우, 단순히 실패 여부만이 아니라 어떤 이유로 잘못되었는지를 함께 기록해야 한다.
- 따라서 `invalid` 결과에는 오류 코드와 가능하면 오류 위치 정보도 포함하는 것이 바람직하다.
- 이는 UI 경고 표시, import 오류 보고, 테스트 케이스 작성에 모두 필요하다.

### 5.8. 셀 단위 정보와 셀 간 관계 정보의 경계

- 위 정보들은 모두 “셀 하나만 보고 판단 가능한 정보”에 해당한다.
- 반대로 앞뒤 셀과의 연결 여부, gliss의 실제 대응 관계, tuplet의 시간 길이 확정, 전역 셀 해석 결과와의 결합 등은 `ParsedCell`에 직접 넣지 않는다.
- 이러한 정보는 문서 전체 문맥을 참조해야 하므로 analyzer 단계에서 처리하는 것이 적절하다.

즉 `ParsedCell`은 셀 단독으로 확정 가능한 정보만 담고, 셀 간 관계나 시간축 의미는 담지 않는 구조를 목표로 한다.  
이 경계를 명확히 해야 parser는 단순하게 유지되고, analyzer는 문맥 해석에 집중할 수 있다.


## 6. ParsedCell 최상위 구조

```ts
type ParsedCell =
  | ParsedMuteCell
  | ParsedNoteCell
  | ParsedPletHeadCell
  | ParsedPletExtendCell
  | ParsedInvalidCell;

type ParsedCellKind =
  | "mute"
  | "note"
  | "pletHead"
  | "pletExtend"
  | "invalid";

type ParsedCellBase = {
  kind: ParsedCellKind;
  rawText: string;
};
```

필드 설명:
- kind
	- 1차 파싱 결과로 알 수 있는 셀의 종류(구분자)
- rawText
	- 저장된 원본 문자열
	- 재직렬화용


## 7. 파싱 오류 구조

기존 초안에서는 `parseErrorCode: string | null` 형태를 상정했으나, 실제 구현에서는 객체형이 더 적절하다.

```ts
type ParsedInvalidCell = ParsedCellBase & {
  kind: "invalid";
  error: ParseError;
};

type ParseError = {
  code: ParseErrorCode;
  charIndex: number | null;
  message?: string;
};

type ParseErrorCode =
  | "empty_note"
  | "invalid_escape"
  | "unexpected_reserved_char"
  | "unknown_modifier"
  | "modifier_order"
  | "duplicate_modifier"
  | "unterminated_paren"
  | "missing_argument"
  | "invalid_number"
  | "invalid_gliss_id"
  | "invalid_gliss_kind"
  | "invalid_trem_division"
  | "invalid_midi_range"
  | "invalid_cent_range"
  | "invalid_tuplet_division"
  | "tuplet_slot_count_mismatch"
  | "tuplet_nested_forbidden"
  | "tuplet_position_required"
  | "vib_and_trem"
  | "more_than_expected";
```
오류 구조는 역할에 따라 3계층으로 나눈다.

1) ParsedInvalidCell : 셀이 invalid 타입임을 명시
2) ParseError : 구체적인 오류 정보
3) ParseErrorCode : 오류 종류를 문자열의 목록으로 명시&제한

필드 설명:
- code
	- 발생한 문법 오류의 종류
	- 실패 원인을 후속 단계와 UI에서 구분해서 처리를 진행할 수 있다.
-  charIndex
	- 해당 문자열에서 오류가 일어난 부분의 위치
	- 위치를 특정하기 어려우면 null
- message
	- 사람이 읽는 용도의 보조 설명 텍스트
	- 비필수, 디버깅에서 사용


## 8. 종류별 ParsedCell 구조

### 8.1. mute 셀

```ts
type ParsedMuteCell = ParsedCellBase & {
  kind: "mute";
  displayText: string;
};
```
설명:
- 음악적 속성은 가지지 않는다.
- “//” 뒤의 문자열은 토큰으로 재해석하지 않는다.

필드 설명:
- displayText
- renderer가 셀에 표시할 기본 문자열이다. “//” 뒤의 문자열을 의미한다.


### 8.2. 일반 note 셀

```ts
type ParsedNoteCell = ParsedCellBase & {
  kind: "note";
  hold: HoldKind | null;
  displayText: string;
  modifiers: ParsedModifiers;
};

type HoldKind = "-" | "~";

type ParsedModifiers = {
  gliss: ParsedGliss | null;
  trem: ParsedTrem | null;
  absolutePitch: ParsedAbsolutePitch | null;
  microPitch: ParsedMicroPitch | null;
};
```
필드 설명:
- hold
- 셀 문자열 맨 앞에 “-” / “~”이 존재하는지 기록한다.
	- 실제 연결은 analyzer의 담당
- displayText
	- renderer가 셀에 표시할 기본 문자열이다.
	- note셀에서는 default text가 있으면 그 값을 사용하고, 없을 경우 hold 등에 따라 보조적으로 결정된다.
- modifiers
 	- gliss, trem, absolute pitch, micro pitch와 같은 수식어 정보를 묶어서 담는다.
  - parser는 존재 여부와 인수 값을 기록하고, 실제 의미 확정은 analyzer가 담당한다.


### 8.3. tuplet head 셀

```ts
type ParsedPletHeadCell = ParsedCellBase & {
  kind: "pletHead";
  divNum: number;
  slots: ParsedPletSlot[];
};

type ParsedPletSlot = {
  slotIndex: number;
  isRest: boolean;
  note: ParsedPletSlotNote | null;
};

type ParsedPletSlotNote = {
  hold: HoldKind | null;
  displayText: string;
  modifiers: ParsedPletSlotModifiers;
  position: ParsedTupletPosition;
};

type ParsedPletSlotModifiers = {
  gliss: ParsedGliss | null;
  trem: ParsedTrem | null;
  absolutePitch: ParsedAbsolutePitch | null;
  microPitch: ParsedMicroPitch | null;
};
```

“ParsedPletHeadCell” 필드 설명:
- divNum
- 해당 tuplet이 몇 개의 분할음으로 구성되는지를 나타낸다.
- slots.length와 반드시 일치해야 한다.
- slots
- 각 분할음의 내용을 순서대로 담는 배열이다.
- 빈 slot도 쉼표로서 의미를 가지므로 배열에서 생략하지 않는다.


“ParsedPletSlot” 필드 설명:
- slotIndex
	-  잇단음 노트 내부에서 몇번째 slot인지를 나타내는 인덱스. 0부터 시작한다. 
- isRest
	- 해당 slot이 쉼표면 true
	-  true면 소리를 가지지 않는다.
- note
	- 쉼표가 아닌 경우 해당 slot 문자열이 가진 노트 정보를 담는다.
	- isRest == true면 null이다.


“ParsedPletSlotNote” 필드 설명:
- hold
	- 해당 slot 내부에 hold가 존재하는가
- displayText
	- slot에 표시할 문자열이다.
- modifiers
  - 해당 slot에 붙은 수식어들을 묶어 저장한다.
- position
  - `@n(midi_num)`에서 읽은 위치 지정 정보를 담는다.
  - 실제 행과의 매핑은 analyzer가 수행한다.

- 잇단음 내부 잇단음 (nested tuplet) 구조는 허용하지 않는다.


### 8.4. tuplet extend 셀

```ts
type ParsedPletExtendCell = ParsedCellBase & {
  kind: "pletExtend";
};
```

필드 설명:
- `rawText`
  - 원본 문자열 `/&`를 유지한다.

## 9. modifier 구조

```ts
type ParsedGliss = {
  id: string;
  glissKind: "S" | "M" | "E";
};

type ParsedTrem = {
  divNum: 2 | 3 | 4;
};

type ParsedAbsolutePitch = {
  midiNum: number;
};

type ParsedMicroPitch = {
  centNum: number;
};

type ParsedTupletPosition = {
  midiNum: number;
};
```

`ParsedGliss`
- `id`
  - 같은 gliss 연결에 속하는 셀들을 식별하기 위한 식별자이다.
  - 저장 문자열에서는 소문자 알파벳 한 글자만 허용한다.
- `glissKind`
  - 해당 셀이 gliss의 시작, 중간, 끝 중 어느 역할인지를 나타낸다.
  - 저장 문자열에서는 `S`, `M`, `E`를 사용하며 각각 start, mid, end를 의미한다.
  - 실제 연결 완성 여부는 analyzer가 판단한다.

`ParsedTrem`
- `divNum`
  - 셀 길이를 몇 개의 짧은 음으로 분할할지를 나타낸다.
  - 허용 범위는 문법 명세에서 정한 값만 사용한다.

`ParsedAbsolutePitch`
- `midiNum`
  - 실제 발음 음정을 MIDI 번호로 저장한 값이다.
  - 표시 위치와 실제 음정이 다를 수 있음을 나타내는 핵심 정보이다.

`ParsedMicroPitch`
- `centNum`
  - 기본 음정에서 몇 cent만큼 미세 조정할지를 나타낸다.
  - 값의 유효 범위는 `-100~100` 범위의 유한 실수이며 parser가 검증한다.
  - 소수점 이하 자릿수는 최대 1자리까지 허용한다.
  - 소수점 이하 2자리 이상이면 범위 문제가 아니라 숫자 형식 문제로 보고 `invalid_number`로 처리한다.

`ParsedTupletPosition`
- `midiNum`
  - tuplet 내부 분할음의 표시 위치를 지정하는 MIDI 번호이다.
  - analyzer는 이를 현재 문맥의 행 구조와 대응시켜 실제 위치를 확정한다.

## 10. note 셀 파싱

### 10.1. 문법 및 제약조건

`note_text`는 다음 순서로 읽는다.

1. 최전방 hold
2. defaultText
3. gliss
4. trem
5. absolutePitch
6. microPitch

즉 canonical order는 아래와 같다.

```text
[hold] [defaultText] [gliss] [trem] [absolutePitch] [microPitch]
```

세부 규칙은 다음과 같다.

- 문자열 맨 앞의 `-` 또는 `~`만 hold로 인정한다.
- 이후 위치에서 등장하는 unescaped `-`, `~`는 예약문자 충돌로 처리한다.
- modifier 시작 전까지의 문자열을 `defaultText`로 읽는다.
- escape된 예약문자는 일반 문자로 복원한다.
- unescaped `/`, `@`, `|`, `(`, `)`, `-`, `~`는 문법 오류이다.
- modifier 순서는 `@g -> @t -> @p -> @m`으로 고정한다.
- 각 modifier는 최대 1회만 허용한다.
- 알 수 없는 `@x(...)` 형식은 `unknown_modifier`로 처리한다.
- `hold`, `defaultText`, `modifier` 중 하나 이상은 반드시 존재해야 한다.
- 아무것도 없으면 `empty_note`이다.

`displayText`는 아래 규칙으로 결정한다.

```text
1. defaultText가 있으면 그것을 사용
2. defaultText가 없고 hold가 "-"이면 "-"
3. 그 외는 ""
```

주의:

- `~`는 hold 표식이지만 표시 텍스트로 그대로 쓰지 않는다.
- 수식어만 있는 셀은 빈 `displayText`를 가질 수 있다.

### 10.2. 파싱 절차 수도코드

```text
parseNoteText(rawText):

  // 문자열을 순차적으로 읽기 위해 필요한 상태를 초기화한다.
  index = 0
  hold = null
  defaultText = ""
  modifiers = empty modifiers


  // 첫 글자가 hold 토큰이면 hold에 기록하고 index를 증가시킨다.
  if rawText[index] is "-" or "~"
    hold = rawText[index]
    index++


  // defaultText를 읽는다.
  // 이 과정에서 escape 복원과 예약문자 충돌 검사를 함께 수행한다.
  // 오류가 발생하면 invalid 결과를 반환한다.
  defaultTextResult = scanDefaultText(rawText, index)
  if defaultTextResult is error
    return invalid(...)

  defaultText = defaultTextResult.value
  index = defaultTextResult.nextIndex

  // @g, @t, @p, @m 순서로 토큰 검사
  // 순서가 어긋나거나 중복이 있으면 오류가 반환된다.

  if rawText startsWith "@g(" at index
    parse gliss

  if rawText startsWith "@t(" at index
    if hold === "~"
      return invalid("vib_and_trem", charIndex)
    parse trem

  if rawText startsWith "@p(" at index
    parse absolutePitch

  if rawText startsWith "@m(" at index
    parse microPitch

  if index != rawText.length
    if rawText[index] startsWith "@"
      return invalid("unknown_modifier", charIndex)
    else
      return invalid("more_than_expected", charIndex)

  // 모든 토큰이 비었다면 오류를 반환한다.
  if hold == null and defaultText == "" and no modifiers
    return invalid("empty_note", 0)

  // 토큰의 상태에 따라 화면 표시용 문자열을 결정한다.
  displayText = decideDisplayText(defaultText, hold)

  return ParsedNoteCell
```

## 11. mute 셀 파싱

### 11.1. 문법 및 제약조건

mute 셀은 아래 규칙을 따른다.

1. `//`를 제거한다.
2. 남은 문자열을 표시용 텍스트로 처리한다.
3. escape가 있으면 복원한다.
4. 허용되지 않은 escape면 invalid를 반환한다.

허용 escape 문자:

```text
\  /  @  |  (  )  -  ~
```

중요:

- mute 셀 내부는 modifier를 다시 읽지 않는다.
- `//abc@g(x,S)` 같은 문자열도 전부 표시용 텍스트로만 본다.

### 11.2. 파싱 절차 수도코드

```text
parseMuteText(rawText):
  content = rawText.slice(2)
  displayText = unescapeMuteText(content)
  if error
    return invalid("invalid_escape", charIndex)

  return ParsedMuteCell
```

## 12. tuplet head 셀 파싱

### 12.1. 문법 및 제약조건

기본 구조는 아래와 같다.

```text
/n(slot_1|slot_2|...|slot_n)
```

제약조건은 다음과 같다.

- `n`은 2 이상이어야 한다.
- 실제 slot 개수는 `n`과 정확히 일치해야 한다.
- 빈 slot은 허용한다.
- 내용이 있는 slot은 반드시 `@n(midi_num)`을 포함해야 한다.
- nested tuplet은 금지한다.

### 12.2. slot 구조와 해석 규칙

각 slot은 아래 순서를 따른다.

```text
[hold] [defaultText] [gliss] [trem] [absolutePitch] [microPitch] @n(midi_num)
```

즉 일반 note 셀과 거의 같지만, 마지막에 위치 토큰 `@n(...)`이 필수이다.

slot의 표현 방식은 다음과 같다.

- 쉼 slot이면 `isRest: true`
- 노트 slot이면 `isRest: false`, `note != null`
- slot index는 0-based 정수로 기록한다.

### 12.3. 파싱 절차 수도코드

```text
parsePletHead(rawText):

  // pletHead 토큰에 포함된 n을 숫자로 해석한다.
  // n이 2보다 작거나 정수가 아니라면 invalid를 반환한다. 
  parse division number n
  
  if n is not integer
    return invalid("invalid_tuplet_division", charIndex)

  if n < 2
    return invalid("invalid_tuplet_division", charIndex)


  // 괄호가 정상적으로 닫혔는지 확인한다.
  parse "(" ... ")" body
  if closing paren not found
    return invalid("unterminated_paren", charIndex)

  // 구분자 "|"를 기준으로 slot을 분할한다.
  // 이때 슬롯의 개수가 n개가 아니라면 invalid를 반환한다.
  split body by top-level "|"
  if slot count != n
    return invalid("tuplet_slot_count_mismatch", charIndex)


  // 분할된 slot을 순서대로 분석하고 배열에 삽입한다.
  for each slotText

    // 빈 slot은 쉼표 슬롯으로 기록한다.
    if slotText is empty
      push rest slot
      continue

    // 내용이 있는 slot은 note 문법으로 파싱한다.
    parse slot note
    
    // nested tuplet은 허용하지 않는다.
    if nested tuplet found
      return invalid("tuplet_nested_forbidden", charIndex)
      
    // 내용이 있는 slot은 반드시 위치 지정 토큰 @n을 가져야 한다.
    if @n missing
      return invalid("tuplet_position_required", charIndex)

    push note slot

  // 결과를 종합하여 반환한다.
  return ParsedPletHeadCell
```

## 14. 공통 하위 파서 인터페이스

수식어 파서는 동일한 인터페이스를 갖는 것이 좋다.

```ts
type ParsePartResult<T> =
  | {
      ok: true;
      value: T;
      nextIndex: number;
    }
  | {
      ok: false;
      error: ParseError;
    };
```

적용 대상:

- `parseGliss`
- `parseTrem`
- `parseAbsolutePitch`
- `parseMicroPitch`
- `parsePletPosition`

<T>는 generic 타입 변수이다. 해당 자리에 들어갈 타입은 미정이지만 실제 사용 시점에 특정 타입 중 하나를 선언할 수 있는 자리이다.
이 프로그램의 하위 파서들은 처리 대상과 반환값의 종류는 서로 다르지만 반환 방식은 매우 유사하다. 구체적으로는 다음과 같은 정보를 공통으로 반환한다.

- 성공/실패 여부
- 성공 시 파싱된 값 ParsedGliss / ParsedTrem / ParsedAbsolutePitch / ParsedMicropitch / ParsedTupletPosition
- 성공 시 다음으로 읽을 문자열 인덱스 nextIndex
- 실패 시 오류 정보

이 공통점을 이용해 ParsePartResult<T>라는 하나의 generic 타입으로 묶어서 서로 다른 하위 파서들이 동일 인터페이스를 공유하도록 하였다. 이를 통해 타입 정의의 중복을 줄이고, parser 본체에서 각 하위 파서를 일관된 방식으로 호출할 수 있도록 하였다.


## 15. 최상위 파싱 절차

### 15.1. 종류 판별 규칙

최상위 분기 규칙:

```text
1. rawText 길이가 0이면 invalid
2. "//"로 시작하면 mute
3. 정확히 "/&"이면 pletExtend
4. "/" + number + "(" 패턴으로 시작하면 pletHead
5. 그 외는 note
```

주의:

- 이 분기는 어디까지나 1차 판별이다.
- `pletHead`로 보이더라도 내부 구조가 깨져 있으면 최종 결과는 invalid가 된다.

### 15.2. 최상위 수도코드

```text
parseRawText(rawText):
  if rawText.length == 0
    return invalid("empty_note", 0)

  rawTextType = judgeRawTextType(rawText)

  switch rawTextType
    case "mute":
      return parseMuteText(rawText)
    case "pletExtend":
      return parsePletExtend(rawText)
    case "pletHead":
      return parsePletHead(rawText)
    case "note":
      return parseNoteText(rawText)
```

## 16. parser 출력 이후 런타임 사용 방식

parser 출력은 독립적으로 끝나지 않는다. 런타임에서는 아래처럼 사용한다.

### 16.1. 셀 단위 캐시

```ts
type ParsedCellCacheKey = `${string}|${string}|${number}`;
```

예:

- `${trackId}|${rowId}|${col}`

용도:

- 셀 편집 후 해당 셀만 재파싱
- 동일 셀의 반복 해석 방지

### 16.2. analyzer 입력 단위

analyzer는 단일 `ParsedCell`이 아니라 아래 문맥을 함께 받는다.

```ts
type AnalyzerCellInput = {
  trackId: string;
  rowId: string;
  col: number;
  parsedCell: ParsedCell;
};
```

추가로 analyzer는 다음 문서 전역 정보에 접근할 수 있어야 한다.

- `rowById`
- `cellsByTrackAndCol`
- `globalCellsByKindAndCol`
- `globalCellsInColOrder`
- 현재 track의 이웃 열 정보

즉 parser 출력은 완전한 최종 해석 결과가 아니라 analyzer의 재료이다.

## 17. parser가 하지 말아야 할 것

아래 항목은 parser에서 처리하지 않는다.

- 이전 열의 음과 현재 hold가 실제로 이어지는지 판정
- gliss id의 시작/중간/끝 연결 완성 여부 판정
- `/&` 길이 확정
- 실제 재생 가능한 주파수 계산
- 렌더용 색상 결정
- 오디오 이벤트 duration 확정

이 원칙을 지켜야 parser가 단순하고 안정적으로 유지된다.

## 18. 구현 우선순위

이 문서 기준으로 실제 구현 우선순위는 다음과 같다.

1. `ParseErrorCode` 확정
2. 공통 scan/unescape 유틸 설계
3. `parseMuteText`
4. `parseNoteText`
5. modifier 하위 파서 4종
6. `parsePletHead`
7. `parsePletExtend`
8. 셀 단위 테스트 작성

## 19. 결론

회귀 코드의 parser는 셀 문자열을 안정적으로 해석하는 작은 모듈이어야 한다. 이를 위해 다음 원칙을 고정한다.

- parser 출력은 discriminated union 구조를 사용한다.
- parser는 문자열 자체만 해석한다.
