이해한 현재 상태를 정리하면 다음과 같습니다.

`회귀 코드`는 `regression-code/`가 active 구현이고, `이세계 코드`는 Google Sheets / Apps Script 기반 legacy 참고용입니다. 현재 노래방 모드는 문서상 `game mode`, 사용자 UI상 `practice mode`로 구현되고 있으며, 기존 parser/analyzer/renderer/audio 구조를 바꾸지 않고 `AnalysisResult` 위에 별도 런타임 판정 계층을 얹는 방향입니다.

현재 구현된 핵심 흐름은 이렇습니다.

```text
ScoreFile
 -> parser
 -> analyzer
 -> AnalysisResult
 -> active track NoteEvent 판정 대상
 -> microphone pitch frame
 -> pitch / timing judge
 -> score summary + overlay UI
```

현재 완료된 것으로 보이는 부분은 다음입니다.

- `pitchy` 기반 마이크 pitch detection
- `GamePitchFrame` 생성
- clarity / RMS / frequency range 기반 유효 입력 필터
- score canvas 위 pitch dot 표시
- practice mode 상태 모델: `off`, `preparing`, `countdown`, `ready`, `playing`, `paused`, `finished`, `error`
- practice mode 중 edit / seek / loop / layout / file 관련 조작 잠금
- `ready` 상태에서는 active track 변경 허용, 재생/일시정지/완료 상태에서는 금지
- active track의 `NoteEvent`를 판정 대상으로 사용
- pitch class 기준 판정: 같은 계이름이면 옥타브가 달라도 정답 처리
- `Perfect / Ok / Bad / Miss` 판정
- `Miss`를 평균 정확도 분모에 0%로 포함
- combo, best combo, score, accuracy 표시
- onset 기반 `early / late / timing miss` 판정
- `250ms` 이상 timing miss는 음정이 맞아도 최종 `Miss`로 강등
- `Sync` 보정값: 기본 `+90ms`, `-200ms`~`+200ms`, `10ms` 단위, localStorage 저장
- practice result dialog 표시
- `npm run test:game` 단위 테스트 존재

중요한 현재 정책은 “입력 pitch를 target에 맞게 보정하지 않는다”입니다. 이전에 target 기반 octave correction, 배음 보정, hysteresis lock을 실험했지만 제거했고, 지금은 detector가 반환한 생 pitch를 그대로 표시/판정에 사용합니다. 판정은 raw pitch를 target pitch class와 비교하되, 사용자가 실제로 낸 음을 target 쪽으로 이동시키지는 않습니다.

현재 남은 구현 후보는 크게 이쪽으로 보입니다.

1. `Sync` 보정 UI 보강
   문서에는 Sync 팝업, 반복 beep, 입력 marker, `-10ms / +10ms`, Reset, Apply가 계획되어 있습니다. 현재는 값과 저장/적용 로직은 있으나, calibration UX를 더 구현할 여지가 있습니다.

2. YouTube sync와 practice mode 동시 사용 검증/보강
   스펙상 YouTube follower playback과 공존해야 하고, 설정 변경은 `ready` 상태에서만 허용하는 방향입니다. 실제 브라우저 검증과 예외 처리 보강이 다음 후보입니다.

3. finished/result 화면 정보 보강
   현재 점수 요약은 표시되지만, 스펙상 곡 제목, 아티스트, active track 목록, 판정표 요약까지 결과 화면에 넣을 수 있습니다. 결과는 JSON/localStorage/file로 저장하지 않고 스크린샷용 화면만 제공하는 정책입니다.

4. 권한 거부 / 장치 문제 / 무음 상태 UX 보강
   pitch detection은 안정화됐지만, 마이크 권한 거부, 입력 장치 제거, 장시간 무음, 낮은 RMS/clarity 상태를 사용자에게 더 명확히 안내할 수 있습니다.

5. 효과 판정 연구
   gliss, vibrato, tremolo는 1차 범위 밖입니다. 후속으로 gliss anchor, vibrato 흔들림, tremolo RMS envelope 판정이 후보지만, 지금은 기본 note pitch/timing 판정 안정화가 우선입니다.

검증 기준으로는 `regression-code`에서 최소 `npm run test:game`, `npm run typecheck`, 필요 시 `npm run build`를 사용하면 됩니다. 현재 노래방 모드 관련 주요 파일은 `regression-code/src/app/game/` 아래의 `game_types.ts`, `game_pitch_input.ts`, `game_pitch_math.ts`, `game_pitch_overlay.ts`, `game_judge.ts`, `game_judge_overlay.ts`, `game_ui.ts`, `game_binding.ts`입니다.

좋습니다. 그러면 우선순위는 이렇게 보는 게 맞겠습니다.

1. `Sync` 보정 UI 개선안 설명
2. 결과 화면 보강: 곡/아티스트/active track/타이밍 정확도 추가
3. practice 패널 진단값 UI 흔들림 억제
4. gliss/vibrato/tremolo 효과 판정은 보류

`Sync` 보정 개선은 “자동 보정”보다는 “사용자가 지연을 체감하고 수동으로 맞출 수 있게 하는 도구”로 접근하는 편이 안전합니다. 현재 pitch 판정은 `scoreSeconds - syncOffsetMs / 1000`으로 비교 시간을 이동시키는 구조라, UI만 붙이면 판정 로직 자체를 크게 바꾸지 않아도 됩니다.

가능한 1차 개선안은 다음 정도입니다.

- `Sync +90 ms` 버튼을 누르면 작은 dialog/popup을 연다.
- 팝업 안에서 `-10ms`, `+10ms`, `Reset`, `Apply`를 제공한다.
- 현재 값은 `+90 ms`처럼 크게 표시한다.
- 사용자가 값을 바꾸면 즉시 또는 Apply 시점에 `gameSyncOffsetMs`에 반영한다.
- localStorage 저장은 기존 `saveGameSyncOffsetMsToLocalStorage()` 흐름을 재사용한다.

조금 더 실용적인 2차 개선은 calibration marker입니다.

- 일정 간격으로 짧은 beep를 재생한다.
- 화면 가운데에 기준선을 하나 표시한다.
- 마이크로 들어온 유효 pitch onset 시점을 marker로 찍는다.
- 사용자는 marker가 기준선보다 오른쪽이면 입력이 늦게 들어온다고 보고 `+10ms` 쪽으로, 왼쪽이면 `-10ms` 쪽으로 조정한다.

다만 여기서 중요한 점은 자동 추천값까지 바로 가면 위험합니다. 현재 onset detection은 practice timing judge에도 쓰이지만, 마이크 입력 환경에 따라 튀거나 누락될 수 있습니다. 그래서 첫 단계는 “반복 beep + marker + 수동 조정”까지만 두고, 자동 계산은 나중에 충분히 로그를 본 뒤 넣는 게 좋습니다.

결과 화면 보강은 바로 적용 가능해 보입니다. 현재 `GameScoreSummary`에는 timing 전용 집계가 없으므로, 타이밍 정확도를 넣으려면 summary 타입에 예를 들어 `timingJudgedCount`, `timingOnTimeCount`, `timingEarlyCount`, `timingLateCount`, `timingMissCount`, `timingAccuracyPercent` 같은 값을 추가하는 방향이 자연스럽습니다. `early/late`는 pitch label을 유지하므로 “타이밍 표시 카운트”와 “최종 Miss 강등 카운트”를 분리하면 해석이 명확합니다.

practice 패널의 Hz / Clarity / RMS / Mic 점멸 문제는 CSS와 표시 문자열 고정으로 해결하는 게 좋습니다.

- 숫자 표시 폭을 고정한다: `tabular-nums`, `min-width`
- Hz는 항상 같은 자리수 형식으로 표시한다: 예 `--.-`, `440.0`
- Clarity는 항상 `0.00`
- RMS는 항상 `0.000`
- Mic 텍스트는 긴 reject reason을 그대로 넣지 않고 짧은 고정 label로 매핑한다: `voiced`, `quiet`, `unclear`, `range`, `--`
- 필요하면 `title`에만 상세 reject reason을 둔다

이렇게 하면 진단값은 계속 갱신되더라도 패널 폭이 흔들리지 않고, 시각적으로 점멸하는 느낌이 줄어듭니다. 다음 구현은 `3번 결과 화면 보강 + 4번 진단값 UI 안정화`를 먼저 하고, `1번 Sync calibration popup`은 설계 확정 후 붙이는 순서가 좋아 보입니다.

구현 전에 모호한 부분은 2개입니다.

1. 타이밍 정확도 계산 방식
   제가 보기엔 아래처럼 두는 게 가장 명확합니다.

   `Timing Accuracy = (on-time + early/late * 0.5) / timing judged count`

   기준:
   - `<80ms`: on-time, 100%
   - `80-249ms`: early/late, 50%
   - `>=250ms`: timing miss, 0%
   - timing onset이 매칭되지 않은 sample은 timing judged count에 넣지 않음

   이렇게 할까요, 아니면 early/late도 “타이밍은 어긋났지만 허용”으로 보고 100%에 포함할까요?

2. 결과 화면에 표시할 곡 정보
   결과 화면에는 `Title`, `Artist`, `Tracks`를 추가하면 될 것 같습니다. `Tracks`는 현재 active track 목록을 `Basic, Optional, Extra` 형식으로 표시하겠습니다. 괜찮을까요?

제가 추천하는 기본값은 1번은 `100 / 50 / 0` 방식, 2번은 `Title / Artist / Tracks` 추가입니다.

> 이렇게 보니 80과 249를 동일하게 50% 취급하는 것은 좀 극단적인 감이 있네요. BAD 강등을 추가해서 4단계로 나누면 구조 변경이 큰가요?

구조 변경은 크지 않습니다. 현재 timing은 이미 `none / early / late / miss`로 분리되어 있고, `miss`일 때만 최종 label을 `Miss`로 강등합니다. 여기에 `Bad 강등`을 추가하는 정도는 국소 변경입니다.

바꿀 지점은 대략 이 정도입니다.

- `GameTimingJudgeResult` 타입에 “Bad 강등” 상태 추가
- `judgeTimingForTarget()`의 threshold 분기 추가
- `judgeGameScoringSample()`에서 timing 결과가 bad면 최종 label을 `Bad`로 강등
- `applyGameScoringSample()`은 이미 `Bad`를 처리하므로 큰 변경 없음
- `test_game.ts`에 timing bad 케이스 추가
- Rules dialog / result dialog 문구 갱신

추천 구조는 4단계입니다.

```text
< 80ms       : timing OK, pitch label 유지
80-149ms     : early/late 표시, pitch label 유지
150-249ms    : early/late 표시 + 최종 label Bad로 강등
>= 250ms     : 최종 label Miss로 강등
```

이러면 80ms와 249ms가 같은 취급을 받는 문제를 줄일 수 있습니다. 점수상으로는 timing bad가 `Bad = 20%`가 되므로, “음정은 맞았지만 타이밍이 꽤 어긋남”을 Miss까지는 아니게 표현할 수 있습니다.

한 가지 결정만 필요합니다: `150-249ms`에서 overlay에 `Bad`와 함께 작은 `early/late`를 계속 표시할까요? 저는 표시하는 쪽이 좋다고 봅니다. 사용자는 “왜 Bad가 떴는지”를 바로 알 수 있습니다.

효과별로는 난이도를 이렇게 낮추는 게 좋겠습니다.

**Gliss**
가장 구현 가치가 높습니다.
목표는 정확한 곡선이 아니라 “방향성 있는 연속 이동이 있었나”만 봅니다.

판정 후보:
- 구간 내 voiced frame이 일정 개수 이상
- 시작 근처와 끝 근처의 평균 pitch cent 차이가 충분함
- 이동 방향이 gliss S/E 목표 방향과 같음
- 중간 frame의 큰 역주행 비율이 낮음

이건 덧셈/비교만으로 됩니다.

**Vibrato**
중간 난이도입니다.
정확한 주파수 주기를 분석하지 말고 “작은 pitch 흔들림이 반복됐나”만 봅니다.

판정 후보:
- 구간 내 voiced frame 충분
- pitch cent 표준편차 또는 range가 threshold 이상
- median 또는 이동평균 기준으로 위/아래 crossing이 2~3회 이상
- 전체 pitch가 한 방향으로만 미끄러지는 경우는 제외

Pitchy가 흔들릴 때 false positive가 날 수 있으므로, bonus만 주는 구조와 잘 맞습니다.

**Tremolo**
가장 불안정합니다.
RMS는 마이크 거리, 방 크기, 입력 장치 자동처리 영향을 많이 받습니다.

판정 후보:
- 구간 내 RMS 변화량이 충분함
- high/low envelope crossing이 반복됨
- onset 후보가 여러 번 발생하면 보조 증거로 사용

하지만 trem은 첫 구현에서는 “실험 플래그” 수준이 좋겠습니다. 무거워서가 아니라 신뢰도가 낮을 가능성이 큽니다.

결론적으로 1차 구현 순서는 이렇게 제안합니다.

1. `gliss`만 보너스 판정으로 구현
2. 결과창에 `Effect Bonus`와 `Gliss` count만 표시
3. 같은 ring buffer 구조에 `vib`를 추가
4. `trem`은 RMS 실측 후 유지/폐기 결정

성능상으로는 괜찮습니다. FFT나 별도 pitch detection을 추가하지 않고 기존 frame을 재사용하면, effect target 수와 최근 frame 수만 훑는 정도라 현재 렌더/오디오 비용에 비하면 작습니다. 폐기 판단은 성능보다는 “오탐/미탐이 너무 많아 사용자가 납득하기 어려운가”를 기준으로 잡는 게 맞겠습니다.

> 일단 제가 쓰고 있는 4만원대 마이크 기준으로는 음정 표시 원의 생성으로 판단하건대 트레몰로의 빠른 끊김을 잡아내기는 합니다.
> 우선 gliss를 얘기해봅시다. 복잡하게 할 것 없이 시작점과 끝점 음정이 일치하는가, 그 사이에서 충분히 음이 연속적인가만 보면 충분할 것 같습니다.

좋습니다. gliss 1차 판정은 그 정도로 충분합니다. “곡선 품질”을 보려 하면 바로 무거워지고 억울한 판정이 늘어납니다.

제가 제안하는 gliss 판정은 3조건입니다.

```text
1. 시작점 근처 음정이 start anchor와 맞는가
2. 끝점 근처 음정이 end anchor와 맞는가
3. start -> end 사이에 voiced frame이 충분하고, 큰 단절 없이 이어졌는가
```

점수는 실패 감점 없이, 세 조건이 모두 맞으면 gliss bonus 1회 지급입니다.

구체적으로는 gliss 구간을 이렇게 나눕니다.

```text
start window: glissStartSeconds ± 120ms
end window:   glissEndSeconds ± 120ms
body window:  glissStartSeconds ~ glissEndSeconds
```

판정 기준 초안:

```text
start matched:
  start window 안 voiced frame 중 하나 이상이 start pitch class 기준 Ok 이상

end matched:
  end window 안 voiced frame 중 하나 이상이 end pitch class 기준 Ok 이상

continuous:
  body window 안 voiced frame 수가 최소 3개 이상
  voiced coverage가 구간의 50% 이상
  voiced frame 사이 최대 공백이 180ms 이하
```

여기서 “음이 연속적인가”는 pitch가 매끈하게 이동했는지보다, 마이크가 계속 유효 pitch를 잡았는지로 먼저 보면 됩니다. 이게 가볍고 설명도 쉽습니다.

방향성 체크는 아주 약하게만 추가할 수 있습니다.

```text
if endPitch > startPitch:
  body의 후반 평균 pitch가 전반 평균 pitch보다 높으면 통과
if endPitch < startPitch:
  body의 후반 평균 pitch가 전반 평균 pitch보다 낮으면 통과
```

다만 이건 1차에서는 optional로 두는 게 좋겠습니다. 오타마톤은 글리산도 중간에 pitch가 흔들릴 수 있고, pitch-class 기반 채점과도 조금 충돌할 수 있습니다. 우선은 사용자가 말한 대로 시작/끝/연속성만 보는 편이 안정적입니다.

구현 구조는 가볍게 갈 수 있습니다.

```text
practice 중 pitch frame ring buffer 유지
gliss S/E anchor pair를 effect target으로 변환
gliss endSeconds를 지난 뒤 아직 보너스 미지급이면 판정
성공 시 summary.effectBonus += constant
```

중요한 점은 gliss 판정은 note sample 판정과 별개로 “구간 종료 후 1회”만 해야 합니다. 그래야 긴 gliss가 0.1초 sample마다 중복 보너스를 받지 않습니다.

상수 보너스는 처음엔 작게 잡는 게 좋겠습니다.

```text
gliss success: +trackDifficulty * 0.5
```

표시용으로는 나중에 `Effect Bonus +50`처럼 스케일링해도 됩니다. 내부 점수와 표시 점수 스케일은 분리할 수 있습니다.

> 맞습니다. 다만 표시용은 Effect Bonus로 통일하기보다는 Gliss! Vib! Trem! 식으로 따로 표시될 수 있도록 미리 준비하는 것이 좋겠습니다.

동의합니다. 내부 집계는 공통 `effectBonus`를 두더라도, 표시/확장용 카운트는 효과별로 분리해 두는 게 좋습니다.

gliss 판정을 훨씬 느슨한 interval 방식으로 바꿨습니다.

변경된 방식:
- gliss start~end 구간을 `250ms` 간격으로 나눕니다.
- 각 interval은 한 번만 보너스를 받을 수 있습니다.
- 해당 interval의 scoring tick에서 현재 `GamePitchFrame`이 voiced pitch이면 `Gliss!` 보너스를 지급합니다.
- start/end pitch 일치, 연속성, end 후행 window 조건은 제거했습니다.
- 보너스는 interval당 `trackDifficulty * 0.25`입니다.
- 실패해도 감점은 없습니다.

이제 실제로 gliss 선을 따라 정확히 시작/끝을 맞추지 않아도, 구간 안에서 음정 인식 원이 생성되는 상태라면 `Gliss!`가 잡혀야 합니다.
