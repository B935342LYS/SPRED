**권장 기준**

1. 기준 환경
   - 2020~2021년 보급형 노트북
   - 예: Intel Core i5 10세대/11세대 U 시리즈, Ryzen 5 4000U/5000U 계열
   - RAM 8GB
   - 내장 그래픽
   - Chrome 또는 Edge 최신 버전

2. 하한 환경
   - 2018~2019년 중저가 노트북
   - 예: Intel i5 8세대 U 시리즈, RAM 8GB
   - “느릴 수는 있지만 악보 보기/재생/간단 편집은 가능한 수준” 확인용

3. 개발 기준 환경
   - 현재 개발 PC
   - 정밀 성능 비교용이 아니라 “변경 전후 상대 비교” 기준으로 사용

개인적으로는 **5년 전 보급형 노트북을 실사용 기준으로 잡고, 7~8년 전 노트북을 하한 참고 환경으로 두는 방식**이 좋다고 봅니다. GitHub Pages 웹 앱이고, 대상이 전문 DAW급 앱이 아니라 악보 편집/재생 도구라면 이 정도가 현실적인 선입니다.

성능 목표도 부품명보다 동작 기준으로 정하는 게 안전합니다.

- 1000~3000 column: 편집, 스크롤, 재생이 자연스러워야 함
- 9000+ column: 전체 기능이 즉시 반응하지는 않아도, 스크롤/재생/부분 편집이 멈추지 않아야 함
- drag edit: 입력 중 프레임 드랍은 허용하되, 입력 누락이나 수 초 단위 freeze는 없어야 함
- playback: 스크롤/렌더 부하와 무관하게 오디오 타이밍이 크게 흔들리지 않아야 함
- layout 변경/파일 로드: 순간 대기는 허용 가능하지만, 진행 상태나 명확한 반응이 있어야 함

YouTube `offset < 0` 문제는 다음 구현 이슈로 분리하는 게 좋겠습니다. 원인은 아마 “악보 시간이 0보다 앞서는 구간” 또는 “YouTube seek target이 음수가 되는 구간”을 처리할 때 player가 0초로 clamp되고, 동기화 루프가 계속 다시 seek/play를 걸면서 초반 반복처럼 보이는 흐름일 가능성이 큽니다. 다음 작업으로 잡으면 `scoreTime + offset`이 0 미만일 때 YouTube를 일시 대기시키거나, 0초에 붙여두되 반복 seek를 억제하는 쪽으로 고칠 수 있습니다.

> 네. 동기화 루프의 문제로 보입니다. 어떤 식으로 보정하는 것이 효율적이겠습니까?

가장 효율적인 보정은 **음수 offset 구간을 “YouTube 재생 불가능 구간”으로 명시적으로 상태화**하는 방식입니다. 핵심은 `youtubeTime = scoreTime + offset`이 0보다 작을 때 YouTube player에 반복적으로 `seekTo(0)` / `playVideo()`를 보내지 않는 것입니다.

권장 정책은 이렇습니다.

```ts
youtubeTime = scoreTime + offsetSeconds
```

- `youtubeTime < 0`
  - YouTube는 `0초`에 대기시킴
  - 반복 seek 금지
  - 반복 play 금지
  - score playback은 계속 진행
  - `youtubeTime`이 0 이상이 되는 순간부터 YouTube 재생 시작

- `youtubeTime >= 0`
  - 정상 동기화
  - drift가 허용치보다 클 때만 seek
  - playing 상태면 play
  - paused/stopped 상태면 pause 또는 stop 정책 적용

즉, 음수 offset 구간에서는 YouTube를 “아직 시작 전” 상태로 보고, score만 먼저 흐르게 하는 방식입니다.

구체적으로는 YouTube sync 쪽에 이런 guard가 들어가면 됩니다.

```ts
const targetVideoTime = scoreTimeSeconds + offsetSeconds;

if (targetVideoTime < 0) {
  if (!syncState.isBeforeVideoStart) {
    player.seekTo(0, true);
    player.pauseVideo();
    syncState.isBeforeVideoStart = true;
  }

  return;
}

if (syncState.isBeforeVideoStart) {
  syncState.isBeforeVideoStart = false;
  player.seekTo(targetVideoTime, true);

  if (scorePlaybackState === "playing") {
    player.playVideo();
  }

  return;
}
```

중요한 점은 `targetVideoTime < 0`인 동안 매 tick마다 `seekTo(0)`을 부르면 안 된다는 것입니다. 최초 진입 시 한 번만 0초로 보내고 pause 상태로 둬야 합니다.

추가로 drift 보정도 너무 공격적이면 반복 문제가 생깁니다.

```ts
const drift = Math.abs(player.getCurrentTime() - targetVideoTime);

if (drift > 0.35) {
  player.seekTo(targetVideoTime, true);
}
```

이런 식으로 허용 오차를 두는 게 좋습니다. 매 프레임 정확히 맞추려 하면 오히려 YouTube iframe player 특성상 seek 루프가 생기기 쉽습니다.

정리하면 추천 구현 방향은 다음입니다.

1. `scoreTime + offset < 0` 구간을 별도 상태로 둔다.
2. 그 구간에서는 YouTube를 0초 pause 상태로 한 번만 정렬한다.
3. 0 이상이 되는 경계에서 한 번 seek 후 play한다.
4. 정상 구간에서도 drift threshold 이상일 때만 seek한다.
5. seek 직후에는 짧은 cooldown을 둬서 즉시 재-seek를 막는다.

특히 5번까지 넣으면 안정성이 좋아집니다. 예를 들어 seek 후 300~500ms 동안은 drift 판단을 무시하면 YouTube player의 비동기 반응 때문에 같은 위치로 여러 번 seek하는 문제를 줄일 수 있습니다.

Undo는 말씀하신 대로 `ScoreFile` 전체 snapshot보다 가벼운 방식이 낫겠습니다. 권장 방식은 `inverse cell patch`입니다.

핵심 구조는 이렇게 잡으면 됩니다:

```ts
type CellPatch = {
  selection: ScoreSelection;
  beforeRawText: string | null;
  afterRawText: string | null;
};

type UndoHistoryEntry = {
  id: string;
  label: string;
  patches: CellPatch[];
};
```

의미:
- `beforeRawText: null`이면 원래 cell이 없었다는 뜻입니다.
- `afterRawText: null`이면 편집 후 cell이 삭제되었다는 뜻입니다.
- Undo는 `beforeRawText`를 적용합니다.
- Redo는 `afterRawText`를 적용합니다.
- `null` 적용은 현재 edit 경로에 맞춰 `rawText: ""` delete edit으로 변환하면 됩니다.

이 방식이면 단일 edit, drag batch, range delete, range paste 모두 처리됩니다. 특히 paste가 기존 cell을 덮어쓴 경우도 `beforeRawText`에 원래 값을 저장하므로 되돌릴 수 있습니다. 메모리 사용량은 “전체 악보 크기”가 아니라 “해당 명령으로 실제 변경된 cell 수”에 비례합니다.

metadata 편집은 undo 대상에서 제외하는 쪽으로 명세를 바꾸는 게 맞습니다. 그러면 `Details metadata edit`, `YouTube Reload`는 2.10의 undo 대상에서 빼고, 악보 영역 mutation만 기록하면 됩니다. 다음 작업은 `docs/2.10-undo-redo-edit-history-spec.md`를 snapshot 기반에서 cell patch 기반으로 고치는 것이 좋겠습니다.
