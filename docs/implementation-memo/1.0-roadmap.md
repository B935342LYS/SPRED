# 1.0. 첫 인크리먼트 설계 로드맵
> 상위 경로에 존재하는 md파일 전반은 specification이다.
> 본 경로에 존재하는 md파일들은 세부 구현 일지 및 해설 문서로 그 성격이 명확히 구분된다.
> specification의 넘버링 1.x에서 각 모듈의 기능 정의 / 필요한 클래스 정의 / 모듈 사이의 인터페이스 정의를 완료하였다.
> 이제 실제 ts 파일 구현에 들어가 클래스를 선언하고, 각 모듈의 input에서 output을 도출하는 알고리즘의 구현에 들어간다.
> 본 설계 메모 문서 세트에서는 코드의 구상부터 실제 제작 과정, 이후 제작된 코드의 해설까지 구현 전반에 대해 세부 설명으로 보충하는 내용을 담는다.

## 3. 필요한 JavaScript / TypeScript 개념

## 3.1. HTML, CSS, JavaScript(TypeScript) 파일간 연결

HTML + CSS:
HTML에 CSS를 적용하는 데에는 3가지 방법이 있다.
1. HTML에서 필요한 태그 내부에 직접 CSS 속성 입력
2. HTML의 <head></head> 태그 내부에 <style></style> 만들고 안에 CSS 작성
3. 별도의 css 확장자 파일을 만들고 HTML의 <head></head> 태그 내부에 <link href = "./경로/style.css" rel="stylesheet">로 불러오기

실무적으로는 3번을 주로 사용한다.

HTML + JS:
작성한 JS파일을 HTML 내부로 불러오고자 한다면 2가지 방법을 이용할 수 있다.
1. HTML의 <body></body> 태그 내부에 <script></script> 태그 만들고 안에 코드 작성
2. 별도의 js파일 만들고 HTML의 <body></body> 태그 내부에 <script src="./경로/script.js" defer></script>


현재 구현에 사용하고 있는 ts 확장자 파일은 바로 HTML에 연결할 수 없으며 한번 JS로의 변환을 거쳐야 함에 유의한다.
node.js를 이용하여 편리하게 변환이 가능하다고 한다.

## 3.2. js / ts 파일간 연결
JS(또는 TS) 파일 간의 연결은 import / export 문으로 수행한다. 함수, 객체, 기본 타입의 값을 다른 파일로 넘겨줄 수 있다.

주는 모듈은 다른 모듈에 넘겨주고자 하는 것을 선언할 때 앞에 export를 부착한다.

받는 모듈은 `import (값 | 값 리스트 | *(전체)) from "경로/파일.ts"`의 형식으로 import문을 사용하여 값을 가져온다.

## 3.4. 배럴 파일 (index.ts)

상위 폴더에 index.ts와 같은 이름의 파일을 만들고 내부에 export * "하위 모듈" 문을 여럿 집어넣어 하나의 정리된 목차처럼 사용할 수 있다. 이것을 배럴 파일이라고 부른다.

장점:
개발 편의성의 상승 : 상위 파일에서 각 모듈별로 하나하나 import를 작성하는 수고를 줄이고 가독성을 향상시킬 수 있다.

단점:
대형 프로젝트에서는 하위 폴더에도 하나하나 index.ts 파일을 만들면 연쇄적으로 배럴 파일이 import되어 overhead가 발생하는 문제가 있다고 한다. 본 프로그램은 학부생 수준의 1인 프로젝트니 걱정하지 않아도 될 것 같지만 그래도 한번 검토를 실시한다.

주요 단점은 다음과 같다.
- 성능 오버헤드
    - export로 하위 배럴 파일을 import한다. 배럴 파일은 많은 export 문을 포함하므로 그 중 또 다른 하위 배럴 파일을 포함할 수도 있다. 연쇄적으로 수많은 파일 묶음이 불러와지면서 성능 오버헤드가 발생할 수 있다.

- 모듈간 경계 흐려짐
    - 연쇄적으로 수많은 파일을 불러오게 되면 모든 흐름을 파악하고 통제하기 어렵다. 하나의 모듈에서 필요한 파일만 부르려고 했는데 다른 모듈의 파일이 의도치 않게 불러와질 가능성이 높아지며 따라서 모듈 경계가 흐려져 버린다.

AI 도구의 평가는 다음과 같다.
- 본 프로젝트의 규모는 성능 오버헤드를 걱정할 정도는 아니다.
- 하지만 모듈간 경계 침범은 하나의 index.ts가 수십개의 파일을 포함하게 되거나 계층이 2단계 이상만 늘어나도 신경써야 하는 문제가 된다.

대규모의 export문을 포함한 배럴 파일을 생성하게 될지는 아직 모르겠다. 다만 폴더가 2계층 이상으로 늘어나는 일은 여럿 생길 수 있다. core와 같은 주요 단위의 상위 폴더에서만 index.ts를 가지고 나머지 내부 파일들은 import문만으로 우선 처리하는 방향으로 가기로 한다. 불편성이 커지는 경우에만 배럴 파일의 사용을 결정한다.
