## 2. Implemented App Flow

Examples 진입점은 File menu에 추가했다.

```text
File
  JSON Load
  JSON Download
  Local Save
  Local Load
  Examples
```

사용 흐름:

1. 사용자가 File > Examples를 연다.
2. dialog에 access word를 입력한다.
3. 앱이 Supabase Edge Function `get-example-manifest`에 `POST` 요청을 보낸다.
4. Edge Function이 서버 환경 변수의 `EXAMPLE_ACCESS_WORD`와 비교한다.
5. 성공하면 `example_scores` catalog row를 manifest로 반환한다.
6. 앱은 manifest validator를 통과한 item만 dialog에 표시한다.
7. 사용자가 예제를 선택하고 `Load Example`을 누르면 `scoreUrl`을 fetch한다.
8. Score JSON text는 기존 `loadScoreJsonText()` 경로로 전달한다.

기존 Score JSON load pipeline을 재사용하므로 JSON 크기 제한, JSON parse, ScoreFile validation, runtime rebuild 정책은 파일 로드와 동일하게 적용된다.
로드 실패 시 기존 score는 유지된다.

## 3. Module Boundary

추가/사용된 Examples 모듈:

```text
regression-code/src/app/examples/
  example_types.ts
  example_manifest_validate.ts
  example_config.ts
  example_supabase_provider.ts
  example_dialog.ts
  example_binding.ts
```

역할:

- `example_types.ts`
  - manifest, item, provider, provider error 타입 정의
- `example_manifest_validate.ts`
  - Edge Function 응답을 앱 내부 manifest 타입으로 검증/정규화
  - HTTPS `scoreUrl`, 중복 없는 supported track, 날짜 문자열, 난이도 범위, size/duration 양의 정수 검증
- `example_config.ts`
  - `VITE_EXAMPLE_MANIFEST_FUNCTION_URL`과 선택 `VITE_SUPABASE_PUBLISHABLE_KEY` 읽기
  - 현재 프로젝트 Edge Function URL을 공개 기본값으로 둠
- `example_supabase_provider.ts`
  - Edge Function manifest POST 요청
  - 선택된 `scoreUrl`의 Score JSON text fetch
  - 표준 error response를 앱 오류로 변환
- `example_dialog.ts`
  - access word 입력, manifest list 렌더링, 선택 상태, notice 표시
- `example_binding.ts`
  - File > Examples 버튼과 app state / 기존 score load callback 연결
  - practice mode 중 Examples 진입/로드 차단

## 4. Supabase / Storage Decisions

현재 Supabase 구성:

- Edge Function URL:
  - `https://nflcoszlsiddxjvydjwu.supabase.co/functions/v1/get-example-manifest`
- access word:
  - Edge Function secret `EXAMPLE_ACCESS_WORD`에만 둔다.
  - 프론트엔드 `VITE_` 환경 변수에는 넣지 않는다.
- Storage bucket:
  - `example-scores`
  - 첫 구현은 public bucket으로 운영한다.
- DB table:
  - `example_scores`
  - 앱은 table을 직접 REST select하지 않고 Edge Function만 호출한다.

public bucket을 사용하면 `scoreUrl`을 아는 사람은 access word 없이 Score JSON을 직접 열 수 있다.
따라서 public bucket에는 공개되어도 문제가 없는 예제만 둔다.
저작권 또는 비공개 유지가 필요한 자료는 넣지 않는다.

나중에 파일 자체 접근 제한이 필요하면 다음 구조로 전환한다.

```text
private bucket
  -> DB에는 score_path 저장
  -> 앱은 선택한 example id를 Edge Function에 요청
  -> Edge Function이 access word 확인 후 signed URL 생성 또는 JSON proxy 응답
```

현재 테스트 단계에서는 public URL 방식이 구현과 운영이 가장 단순하다.

## 5. Catalog SQL Generator

처음에는 npm 기반 TypeScript script로 Score JSON metadata를 읽어 upsert SQL과 upload-ready slug 파일명을 생성하는 방향을 검토했다.
그러나 실제 운영에서는 VS Code / terminal 접근성이 낮고, bucket 파일은 이미 수동으로 업로드되어 있으므로 npm script는 제거했다.

대신 브라우저에서 바로 열 수 있는 로컬 HTML 도구를 추가했다.

```text
regression-code/dev/example_catalog_generator.html
```

이 도구의 역할:

- 여러 Score JSON 파일을 브라우저 file picker로 선택한다.
- 각 파일의 `musicData`와 `tracks`를 읽는다.
- Supabase Storage public base URL과 파일명을 조합해 `score_url`을 만든다.
- `insert into public.example_scores ... on conflict (id) do update ...` SQL을 생성한다.
- 생성 SQL을 복사하거나 `.sql` 파일로 다운로드할 수 있다.

도구는 Supabase에 직접 접속하거나 파일을 업로드하지 않는다.
선택한 로컬 JSON 파일명과 bucket에 이미 업로드된 object 파일명이 같아야 한다.

예:

```text
local file:
  ryo-worldismine.json

generated score_url:
  https://nflcoszlsiddxjvydjwu.supabase.co/storage/v1/object/public/example-scores/ryo-worldismine.json
```

파일명 규칙은 다음을 권장한다.

```text
[a-z0-9-]+.json
```

한글 제목과 아티스트명은 Storage object path가 아니라 DB metadata인 `title`, `artist`에 둔다.
