# 1.25. Step5 Layout Editor UI Shell

## 1. Purpose

This memo records the first layout editor UI shell implementation.
It supplements `docs/2.4-layout-edit-ui-spec.md` and does not replace the active layout editing specification.

The implemented scope is intentionally limited to type definitions, dialog structure, basic DOM binding, and visual layout stabilization.
Actual layout draft mutation, score apply, cell deletion, local preset storage, and file preset import/export are not connected yet.

## 2. Implemented Scope

Implemented:

- layout edit type definitions
- `Layout` dialog shell
- `Modify` button -> dialog open
- dialog close/cancel/reset shell behavior
- placeholder notices for Apply/Add/Save/Load actions
- instrument summary display from current `ScoreFile.instData`
- string selector population
- add-row controls for note/gap, height, pitch, insert below/above
- disabled open-string UI placeholders
- CSS layout stabilization for the dialog controls

Not implemented yet:

- real `LayoutDraftBundle` creation from score rows
- editable row list rendering
- existing row height updates
- row insertion/deletion logic
- draft validation
- score apply and full rebuild
- incompatible cell deletion confirmation
- localStorage layout preset persistence
- layout preset JSON file save/load

## 4. UI Decisions

The dialog currently uses one layout preset name as the user-facing name.
There is no separate Instrument Name input.
When the apply path is implemented, this preset name will also be applied to `instData.instName`.

The instrument `family` field is a dropdown rather than a free text input.
Current options are string-instrument-oriented examples:

```text
Otamatone
Haegeum
Violin
Cello
Gayageum
Custom
```

The open-string checkbox and open MIDI dropdowns are visible but disabled.
This keeps the future UI location stable while avoiding active open-string editing for the current Otamatone-centered phase.

The Add Row area uses two direct insertion buttons instead of an insertion-position dropdown:

```text
Insert Below Selected
Insert Above Selected
```

This reduces one intermediate choice and makes the action target explicit.

## 5. Current Dialog Behavior

On `Modify`, the dialog reads current `ScoreFile.instData` and fills:

- preset name
- family
- open-string checkbox state
- string selector
- string summary rows
- add-row pitch dropdown

The row list intentionally shows an empty placeholder.
The next implementation step should replace that placeholder with actual note/gap rows from the selected string.

The `New Preset` button currently adds a temporary option to the preset dropdown and fills the name input.
It is a UI shell behavior only; it does not write localStorage or file data.

## 6. Next Step

The next implementation step should introduce `layout_draft.ts`.

Recommended scope:

- create `LayoutDraftBundle` from current `ScoreFile`
- filter note/gap rows by selected string
- render existing rows into the row list
- allow existing note/gap row height editing in the draft
- keep add/delete/apply/storage as placeholders unless the draft rendering path is stable

This keeps the next step focused on user-editable draft state before score mutation is introduced.
