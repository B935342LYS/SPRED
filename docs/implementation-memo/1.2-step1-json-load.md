# 1.2. 1단계 JSON 로드 구현 메모
> 본 설계 메모 문서 1.2에서는 1.0에서 정한 1단계 구현 중 `score/json_load.ts`에 대한 설명을 기록한다.
> 코드에 대한 기본 설명은 코드 파일 자체의 주석을 중심으로 한다.
> 본 문서는 JSON 로드 함수의 책임 경계와 구현 중 사용한 JavaScript / TypeScript 문법을 보충한다.

## 1. 개요

`json_load.ts`는 외부에서 들어온 JSON 문자열을 JavaScript 객체로 변환하는 첫 번째 관문이다.

현재 단계에서 이 모듈의 책임은 다음으로 한정한다.

- JSON 문자열을 `JSON.parse()`로 파싱한다.
- JSON 문법 오류를 `parse_error`로 변환한다.
- 파싱 결과의 최상위 값이 객체인지 확인한다.
- 정상 로드된 값은 `ScoreFile` 후보 객체로 반환한다.

반대로 다음 책임은 아직 이 모듈에 넣지 않는다.

- `rowId`, `trackId`, `stringId` 중복 검사
- 존재하지 않는 `rowId` 참조 검사
- `gap` 행에 셀이 저장된 경우의 검사
- `col` 범위 검사
- note/global 셀 문자열 문법 검사

위 항목은 JSON 문법 오류가 아니라 악보 파일 구조 검증에 해당한다.
따라서 이후 `score_validate.ts` 또는 별도 검증 계층에서 다루는 편이 모듈 책임이 더 명확하다.

현재 흐름은 아래와 같이 본다.

```text
JSON text
-> json_load.ts
-> ScoreFile 후보 객체
-> score_validate.ts
-> ScoreFile
-> build_score_indexes.ts
-> RuntimeDocument
```

## 2. 결과 타입

`loadScoreJson()`은 성공과 실패를 하나의 union type으로 반환한다.

```ts
export type LoadScoreJsonResult =
  | {
      ok: true;
      value: unknown;
    }
  | {
      ok: false;
      error: JsonLoadError;
    };
```

이 구조는 예외를 그대로 밖으로 던지지 않고, 호출자가 반환값만 보고 성공 여부를 판단할 수 있게 하기 위한 것이다.

```ts
const result = loadScoreJson(jsonText);

if (!result.ok) {
  console.log(result.error.message);
  return;
}

const candidate = result.value;
```

`ok` 필드는 TypeScript의 discriminated union 구분자로 사용된다.
즉 `result.ok === true`인 경우에는 `value`가 존재하고, `result.ok === false`인 경우에는 `error`가 존재한다고 TypeScript가 추론할 수 있다.

## 3. try / catch와 JSON.parse

현재 구현의 핵심은 아래 부분이다.

```ts
try {
  parsed = JSON.parse(jsonText);
} catch (error) {
  return {
    ok: false,
    error: {
      code: "parse_error",
      message: error instanceof Error ? error.message : "Invalid JSON text.",
    },
  };
}
```

`try / catch`는 예외가 발생할 수 있는 코드를 안전하게 감싸는 문법이다.
`try` 블록 안에서 오류가 발생하면 이후 코드를 계속 실행하지 않고 `catch` 블록으로 이동한다.

`JSON.parse(jsonText)`는 문자열이 올바른 JSON 문법을 따르지 않을 때 예외를 던진다.
예를 들어 닫는 중괄호가 없거나, 문자열 따옴표가 잘못된 경우 `SyntaxError`가 발생한다.
`SyntaxError`는 JavaScript가 기본 제공하는 `Error` 계열 객체이다.

## 4. Error 클래스와 error.message

`Error`는 JavaScript가 기본으로 제공하는 내장 클래스이다.
따라서 프로젝트 내부에서 따로 정의하지 않아도 사용할 수 있다.

```ts
error instanceof Error
```

이 코드는 `catch`로 들어온 `error` 값이 `Error` 객체인지 확인한다.
`Error` 객체라면 `message` 속성을 통해 오류 설명 문자열을 얻을 수 있다.

```ts
error.message
```

`JSON.parse()`가 던지는 일반적인 JSON 문법 오류는 `SyntaxError`이며, `SyntaxError`는 `Error`의 하위 클래스이다.
따라서 대부분의 JSON 문법 오류에서는 `error.message`를 사용할 수 있다.

## 5. catch(error)의 타입과 삼항 연산자

TypeScript에서 `catch (error)`의 `error`는 기본적으로 `unknown`으로 취급된다.
이는 JavaScript에서 꼭 `Error` 객체만 던질 수 있는 것이 아니기 때문이다.

JavaScript에서는 아래와 같은 코드도 가능하다.

```ts
throw "error text";
throw 123;
throw { reason: "custom error" };
```

이 경우 `error.message`가 존재한다고 보장할 수 없다.
따라서 바로 `error.message`에 접근하지 않고, 먼저 `Error` 객체인지 확인한다.

```ts
message: error instanceof Error ? error.message : "Invalid JSON text.",
```

위 코드는 삼항 연산자이다.

```text
조건 ? 조건이 참일 때의 값 : 조건이 거짓일 때의 값
```

따라서 의미는 다음과 같다.

```text
error가 Error 객체이면
  error.message를 사용한다.
그렇지 않으면
  "Invalid JSON text."를 사용한다.
```

현재 `JSON.parse()`에서 발생하는 오류는 대부분 `Error` 객체이지만, 타입 안전성과 예외 처리의 안정성을 위해 fallback 문자열을 둔다.

## 6. trailing comma

다음 코드에서 `"Invalid JSON text."` 뒤의 쉼표는 문자열의 일부가 아니다.

```ts
message: error instanceof Error ? error.message : "Invalid JSON text.",
```

이 쉼표는 객체 속성을 구분하기 위한 문법이다.
마지막 속성 뒤에 붙은 쉼표는 trailing comma라고 부르며, JavaScript / TypeScript에서 허용된다.

아래 두 코드는 모두 유효하다.

```ts
const value = {
  message: "Invalid JSON text."
};
```

```ts
const value = {
  message: "Invalid JSON text.",
};
```

trailing comma를 붙이면 이후 객체 속성을 추가할 때 변경 줄이 줄어들고, 코드 포맷터의 출력도 일정하게 유지된다.

## 7. not_object 검사

JSON 문법상 최상위 값은 객체가 아니어도 될 수 있다.
예를 들어 아래 값들도 JSON 문법 자체로는 유효하다.

```json
123
```

```json
"score"
```

```json
[]
```

하지만 회귀 코드의 악보 파일은 항상 최상위 구조가 객체여야 한다.
따라서 `JSON.parse()`가 성공했더라도 다음 검사를 수행한다.

```ts
if (typeof parsed !== "object" || parsed === null || Array.isArray(parsed)) {
  return {
    ok: false,
    error: {
      code: "not_object",
      message: "Score JSON root must be an object.",
    },
  };
}
```

이 조건은 다음 경우를 걸러낸다.

- `typeof parsed !== "object"`
  - 숫자, 문자열, boolean 등 객체가 아닌 값
- `parsed === null`
  - JavaScript에서 `typeof null`은 `"object"`이므로 별도 검사 필요
- `Array.isArray(parsed)`
  - 배열도 객체의 일종이지만 악보 파일의 최상위 구조로는 허용하지 않음

## 8. ScoreFile 검증으로 넘기기

현재 함수의 마지막은 다음과 같다.

```ts
return {
  ok: true,
  value: parsed,
};
```

`json_load.ts`는 여기서 `parsed`를 `ScoreFile`로 단언하지 않는다.
대신 `unknown` 값 그대로 `value`에 담아 반환한다.

이는 JSON 문법 검증과 악보 구조 검증을 분리하기 위한 결정이다.
`JSON.parse()`가 성공하고 최상위 값이 객체라는 사실만으로는 해당 값이 `ScoreFile`의 모든 필드를 만족한다고 볼 수 없다.
따라서 `loadScoreJson()`은 ScoreFile 후보 값을 만들고, `validateScoreFile()`이 실제 ScoreFile 여부를 확인한다.

외부 호출부에서는 편의를 위해 `loadScoreFile()`을 사용할 수 있다.
이 함수는 내부에서 `loadScoreJson()`과 `validateScoreFile()`을 순서대로 호출한다.

```ts
export function loadScoreFile(jsonText: string): LoadScoreFileResult {
  const loadResult = loadScoreJson(jsonText);

  if (!loadResult.ok) {
    return loadResult;
  }

  return validateScoreFile(loadResult.value);
}
```

이 구조를 두면 호출자는 하나의 함수만 사용하면서도, 내부 책임은 JSON 파싱과 ScoreFile 검증으로 분리된다.

## 9. 다음 작업

`json_load.ts` 이후에는 다음 순서로 구현을 진행한다.

1. `minimal-valid-score.json`을 이용해 `loadScoreFile()`이 정상 반환하는지 확인한다.
2. 깨진 JSON 문자열을 입력했을 때 `parse_error`가 반환되는지 확인한다.
3. 최상위 배열이나 숫자를 입력했을 때 `not_object`가 반환되는지 확인한다.
4. 구조가 깨진 ScoreFile 후보를 입력했을 때 `score_validate.ts`의 오류가 반환되는지 확인한다.
5. `build_score_indexes.ts`에서 검증된 `ScoreFile`을 기반으로 런타임 인덱스를 생성한다.
6. `create_runtime_document.ts`에서 원본 `ScoreFile`과 `ScoreIndexes`를 묶어 `RuntimeDocument`를 생성한다.
