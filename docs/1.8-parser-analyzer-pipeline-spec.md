# parser-analyzer 파이프라인 인터페이스 설계

## 1. 문서 목적

- 본 문서는 회귀 코드의 런타임 파이프라인에서 JSON 문서가 어떤 단계를 거쳐 parser와 analyzer로 전달되는지를 인터페이스 중심으로 정리한다.
- 특히 `ScoreFile -> ScoreIndexes -> ParsedScoreDocument -> AnalysisResult`로 이어지는 데이터 흐름을 TypeScript 기준으로 고정하는 것이 목적이다.
- 본 문서는 `1.3-score-json-format.md`, `1.5-note-cell-parser-spec.md`, `1.6-global-cell-parser-spec.md`, `1.7-analyzer-event-list-spec.md`를 실제 구현 단위로 연결하는 중간 설계 문서이다.


## 2. 설계 범위

본 문서는 다음 범위를 다룬다.

- 문서 로드 직후 생성되는 런타임 정규화 인덱스 구조
- parser가 note 셀과 global 셀을 문서 전체 또는 일부 범위 단위로 처리할 때 사용하는 입력 타입
- analyzer가 full analysis 또는 partial analysis를 수행할 때 사용하는 입력 타입
- 각 단계의 반환 타입
- 구현 시 분리할 모듈의 책임 경계

반면 다음 항목은 본 문서의 직접 범위에 포함하지 않는다.

- 실제 문자열 파싱 알고리즘의 세부 절차
- analyzer 내부의 hold / gliss / tuplet 알고리즘 세부 구현
- UI 이벤트 처리나 화면 렌더링 방식
- 파일 입출력과 브라우저 상태 관리


## 3. 난이도 소명

문제 1. 부분 편집 관련 인터페이스

 부분 갱신은 기존 결과물의 일부 범위만을 안정적으로 교체하도록 설계해야 한다.
 부분 갱신 시에 문서의 교체 범위를 어떻게 정해야 하는가? 어떤 종류의 셀을 수정했는가, 주변에 무슨 셀이 존재하느냐에 따라 다르게 정해야 한다. 동시에 이 결정을 수행하는 연산이 병목을 일으켜서도 안된다.

해결 1. 인터페이스 우선 고정, 세부 규칙은 후속 확정 예정
 먼저 수정 범위를 지정하는 객체와 부분 갱신 함수 시그니처를 정의하여 전체 갱신과 부분 갱신이 동일한 구조로 수렴할 토대를 만들었다. 다만 실제 교체 범위 확장 규칙은 구현과 테스트를 통해 구체적으로 확정해야 할 것으로 보인다.


문제 2. 런타임 조회용 인덱스의 내용 선별
 런타임 인덱스 자료를 어디까지 만들 것인가도 문제였다. 필요한 것이 없으면 모듈화가 깨지고 후속 모듈이 JSON 파일까지 거슬러 내려가서 직접 분석에 필요한 사항을 추론해야 하는 일이 발생한다. 반대로 너무 자잘한 자료를 만들면 쓸데없는 생성/유지 비용이 추가된다. 경계를 잘 그어 필요한 만큼만 정규화 자료를 명세해야 했는데, 이러한 경험이 없었던 입장에서 판단이 어려운 문제였다.

해결 2. (개선 가능성 있음) 자주 접근하게 될 데이터를 정리하고 선정 
 parser와 analyzer가 반복적으로 참조하는 최소한의 조회 기준을 확인하여 ScoreIndexes로 정하였다. 구체적으로는 행 정의 조회, 트랙 조회, 열 범위 기반 접근, 전역 행 kind별 접근에 직접 필요한 인덱스를 우선 선정하였다. 불필요한 세분화는 최대한 피하였다고 생각한다.


## 4. 파이프라인 개요

회귀 코드의 런타임 파이프라인은 아래와 같은 흐름으로 본다.

이 파이프라인은 단순한 처리 순서의 나열이 아니라, 저장 구조와 런타임 구조의 층위를 나누기 위한 기준이기도 하다.
회귀 코드에서는 아래 3계층을 구분한다.

1. 저장 구조
   - JSON 파일에 직렬화되는 최소 정보
2. 정규화 구조
   - 로드 직후 한 번 계산하는 문서 전역 인덱스
3. 해석 구조
   - parser, analyzer, renderer, audio generator가 소비하는 런타임 구조

이 구분이 필요한 이유는 다음과 같다.

- 저장 구조는 사람이 읽고 수정하기 쉬워야 한다.
- 런타임 구조는 조회 속도와 부분 갱신 효율이 우선이다.
- parser 산출물, analyzer 결과, 각종 캐시는 문서 원본과 같은 수명을 가지지 않는다.
- 따라서 저장 구조에 런타임 산출물을 그대로 밀어 넣지 않고, 각 단계에서 필요한 구조를 별도로 생성하는 편이 더 안정적이다.

1. JSON 파일 로드
2. validator가 문서 구조 검증
3. 런타임 정규화 인덱스 생성
4. parser가 note 셀과 global 셀의 `rawText`를 각각 `ParsedCell`, `ParsedGlobalCell`로 변환
5. analyzer가 `ParsedScoreDocument`와 `ScoreIndexes`를 바탕으로 `AnalysisResult`를 생성
6. renderer / audio generator가 `AnalysisResult`를 소비

이를 TypeScript 관점에서 단순화하면 아래와 같이 볼 수 있다.

ScoreFile
  -> ScoreIndexes
  -> ParsedScoreDocument
  -> AnalysisResult


이때 중요한 점은 각 단계가 바로 다음 단계가 요구하는 공용 계약만 노출해야 한다는 것이다.
예를 들어 analyzer는 `rawText`를 다시 직접 해석하지 않아야 하며, renderer와 audio generator는 parser 세부 구조가 아니라 `AnalysisResult`만 소비해야 한다.


## 5. 문서 로드와 정규화 인덱스

JSON 파일은 저장 구조에 맞춰 직렬화되어 있으므로, 로드 직후에는 조회와 부분 갱신에 비효율적일 수 있다.
따라서 validator 통과 이후에는 먼저 런타임 정규화 인덱스를 생성하는 것이 바람직하다.

```ts
type RuntimeDocument = {
  score: ScoreFile;
  indexes: ScoreIndexes;
};
```

`RuntimeDocument`는 새로운 진실 원천을 추가하는 타입이 아니라, 저장된 원본 문서와 그 원본에서 파생된 런타임 정규화 인덱스를 한 단위로 묶어 다루기 위한 컨테이너이다.
즉 `indexes`는 독립 데이터가 아니라 특정 `score`에 종속된 파생 구조이며, 둘은 런타임에서 거의 항상 함께 움직인다.
이렇게 묶어 두면 원본 문서와 다른 문서의 인덱스가 섞이는 실수를 줄이고, 이후 parser와 analyzer 단계에서 필요한 기본 문맥을 일관되게 전달하기 쉽다.

```ts
type ScoreIndexes = {
  rowById: Map<string, RowDefinition>;
  rowsInDisplayOrder: RowDefinition[];
  noteRowIdsByStringId: Map<string, string[]>;
  noteRowIdByStringMidi: Map<string, string>;
  trackById: Map<string, Track>;
  cellMapByTrackId: Map<string, Map<string, ScoreCell>>;
  cellsByTrackAndCol: Map<string, Map<number, ScoreCell[]>>;
  globalCellMapByCoord: Map<string, GlobalCell>;
  globalCellsByKindAndCol: Map<GlobalKind, Map<number, GlobalCell>>;
  globalCellsInColOrder: Map<GlobalKind, GlobalCell[]>;
};
```

여기서 `GlobalKind`는 전역 행 종류를 나타내는 타입으로, `"bpm" | "beatsPerBar" | "stepsPerBeat" | "dynamics"`의 네 값을 가진다.

필드 설명:

- `rowById`
  - `rowId -> RowDefinition` 빠른 조회용 맵이다.
  - parser가 `rowId` 문맥에서 note 행인지 global 행인지, 어떤 `stringId`와 `midi`를 가지는지 즉시 판별할 때 사용한다.
  - renderer 역시 행 메타데이터를 반복 참조하므로, 배열 선형 탐색보다 이 맵을 기준으로 접근하는 편이 낫다.

- `rowsInDisplayOrder`
  - 렌더링 순서를 유지한 행 배열이다.
  - `rowById`가 빠른 단건 조회를 위한 구조라면, 이 배열은 실제 화면 배치 순서를 보존하기 위한 구조이다.
  - 즉 같은 행 정보를 두 가지 축으로 동시에 들고 있는 셈이며, 하나는 조회 효율을, 다른 하나는 표시 순서를 담당한다.

- `noteRowIdsByStringId`
  - 같은 현(`stringId`)에 속한 note row들의 순서 정보를 담는다.
  - note 행은 현별로 묶어서 생각해야 하는 경우가 많으므로, `stringId` 기준 그룹 인덱스를 별도로 두는 편이 편리하다.
  - 특정 현에 속한 note row만 빠르게 순회하거나, 현 단위 편집/렌더링 문맥을 만들 때 사용한다.

- `noteRowIdByStringMidi`
  - `stringId + midi` 조합으로 실제 `rowId`를 찾기 위한 맵이다.
  - analyzer의 `@n(midi_num)` 해석에 직접 사용된다.
  - 즉 사람이 보는 음높이 수치와 문서 내부의 행 ID를 직접 이어 주는 역방향 참조 인덱스라고 볼 수 있다.

- `trackById`
  - `trackId -> Track` 조회용 맵이다.
  - 트랙 메타데이터와 해당 트랙의 원본 셀 배열에 접근할 때 기준점이 된다.
  - 이후 partial parse, partial analysis에서도 요청 범위가 `trackId` 기준으로 들어오기 때문에 기본 인덱스로 필요하다.

- `cellMapByTrackId`
  - 좌표 단위 빠른 조회용 맵이다.
  - 정확히는 `trackId -> (rowId|col) -> ScoreCell` 축으로 접근하기 위한 맵이다.
  - 단일 셀 편집 후 해당 좌표의 원본 셀을 즉시 찾거나 교체할 때 유리하다.
  - 문서 전체를 다시 훑지 않고도 특정 셀 존재 여부를 판정할 수 있다는 점에서 편집기 동작과 캐시 무효화의 기반이 된다.

- `cellsByTrackAndCol`
  - track 단위 partial parse / partial analysis의 기본 조회 인덱스이다.
  - `trackId -> col -> ScoreCell[]` 구조를 통해 특정 트랙의 특정 열에 포함된 셀 묶음을 즉시 찾을 수 있다.
  - 회귀 코드는 편집 후 전체 행렬을 다시 스캔하기보다, 영향받는 열 범위를 기준으로 재파싱/재분석하는 방향을 택한다.
  - 따라서 이 인덱스는 부분 갱신의 기본 단위를 제공하는 핵심 구조이다.
  - 재파싱, 재분석, 재렌더링 모두 열 단위 영향 범위를 계산하기 쉽다는 점에서 가장 실용적인 인덱스 중 하나이다.

- `globalCellMapByCoord`
  - 전역 셀 좌표 단위 빠른 조회용 맵이다.
  - note 셀 쪽의 `cellMapByTrackId`와 비슷한 역할을 전역 셀에 대해 수행한다.
  - 전역 셀은 트랙 축이 없으므로 `rowId|col` 좌표만으로 빠르게 접근할 수 있어야 한다.
  - 단일 전역 셀 편집, 좌표 중복 방지, 해당 셀 재파싱 같은 동작에서 직접 사용된다.

- `globalCellsByKindAndCol`
  - `GlobalKind -> col -> GlobalCell` 인덱스이다.
  - 전역 행 parser와 analyzer가 같은 축을 공유하도록 만든다.
  - 같은 kind에서 특정 열의 전역 셀을 즉시 찾을 수 있으므로, 전역 행을 `bpm`, `beatsPerBar`, `stepsPerBeat`, `dynamics`별 독립 스트림으로 다룰 수 있다.
  - parser는 `rowId`를 통해 kind를 유도하고, analyzer는 kind별 시간축 해석을 수행하므로 양쪽이 같은 축을 공유하는 편이 자연스럽다.
  - 전역 행 좌표 중복 검증 결과와도 연결되며, partial rebuild에서 특정 global kind 범위만 교체할 때도 유용하다.

- `globalCellsInColOrder`
  - 같은 kind의 전역 셀을 열 오름차순으로 정렬한 배열이다.
  - timing / dynamics 세그먼트 생성의 직접 입력으로 사용한다.
  - 전역 셀 해석은 단순 단건 조회보다 "앞 셀과 뒤 셀의 관계"가 중요하므로, 정렬된 순회 배열이 별도로 필요하다.
  - analyzer는 BPM, beatsPerBar, stepsPerBeat, dynamics를 각각 독립 스트림처럼 순회하면서 `<`, `>`, `><`의 연결과 세그먼트 경계를 해석한다.
  - 이런 종류의 시간축 생성은 맵 단건 조회보다 열 오름차순 배열을 기준으로 처리하는 편이 단순하고 안정적이다.


## 6. parser 단계 인터페이스

parser는 문자열 단위 해석을 담당하므로, 기본적으로 셀 단위 입력과 문서 범위 입력 두 가지 형태를 지원하는 것이 바람직하다.
이때 note 셀과 global 셀은 문자열 문법과 후속 의미가 다르므로 입력 타입도 분리하는 편이 좋다.
또한 parser는 문자열 해석만 담당하며, 문서 참조 무결성 검증이나 셀 간 관계 판정은 각각 validator와 analyzer에 남겨 두는 편이 모듈 경계상 더 명확하다.

### 6.1. 셀 단위 입력

```ts
type NoteCellParserInput = {
  trackId: string;
  rowId: string;
  col: number;
  rawText: string;
};

type GlobalCellParserInput = {
  rowId: string;
  col: number;
  rawText: string;
};

type GlobalCellParserContext = {
  rowById: Map<string, RowDefinition>;
};
```

이 입력은 가장 작은 parser 처리 단위이다.
UI 편집 중 단일 셀 문자열이 바뀌었을 때는 이 단위로 재파싱할 수 있다.
전역 셀 parser는 `GlobalCellParserInput.rowId`와 `GlobalCellParserContext.rowById`를 이용해 `kind`를 유도한다.

### 6.2. 문서 단위 parser 출력

analyzer는 개별 `ParsedCell`이나 `ParsedGlobalCell`만이 아니라 “문서 안에서 정리된 parsed 결과 묶음”을 필요로 한다.
따라서 parser의 문서 단위 출력은 아래처럼 두는 편이 좋다.

```ts
type ParsedScoreDocument = {
  noteCellsByTrackAndCol: Map<string, Map<number, ParsedCellEntry[]>>;
  globalCellsByKindAndCol: Map<GlobalKind, Map<number, ParsedGlobalCellEntry>>;
};
```

```ts
type ParsedCellEntry = {
  trackId: string;
  rowId: string;
  col: number;
  parsedCell: ParsedCell;
};

type ParsedGlobalCellEntry = {
  rowId: string;
  kind: GlobalKind;
  col: number;
  parsedCell: ParsedGlobalCell;
};
```

이 구조를 두는 이유는 다음과 같다.

- analyzer가 note / global 경로를 분리하면서도 같은 문서 단위 결과를 입력으로 받을 수 있다.
- 부분 갱신 시 특정 track 열 범위 또는 특정 global kind 범위의 parsed 결과만 교체하기 쉽다.
- `cellsByTrackAndCol`, `globalCellsByKindAndCol`와 같은 축을 공유하므로 구조가 일관된다.
- `ParsedScoreDocument`는 문서 단위 parsed 결과의 공식 구조로 취급하며, 배열 목록이나 보조 인덱스는 필요시 파생 캐시로만 둔다.
- 즉 partial rebuild의 진실 원천은 항상 `ParsedScoreDocument` 하나이며, 다른 뷰는 이를 기준으로 다시 만들 수 있어야 한다.

### 6.3. parser 함수 시그니처 초안

```ts
/** 
 * 타입명 : ParseNoteCellFn
 * 단일 노트 셀의 파싱 함수 인터페이스
 * 인수 : 노트 셀 입력 NoteCellParserInput
 * 반환값 : 파싱 결과 ParsedCell
 */
type ParseNoteCellFn = (input: NoteCellParserInput) => ParsedCell;

/** 
 * 타입명 : ParseGlobalCellFn
 * 단일 전역 셀 의 파싱 함수 인터페이스
 * 인수
 *  - 전역 셀 입력 GlobalCellParserInput
 *  - 문맥 GlobalCellParserContext
 * 반환값 : 파싱 결과 ParsedGlobalCell
 */
type ParseGlobalCellFn = (
  input: GlobalCellParserInput,
  context: GlobalCellParserContext,
) => ParsedGlobalCell;

/** 
 * 타입명 : ParseDocumentFn
 * 초기 로딩 또는 새로고침 시의 전체 셀 파싱 함수 인터페이스
 * 인수
 *    - 악보 파일 원본 ScoreFile
 *    - 런타임 조회용 인덱스 ScoreIndexes 
 * 반환값 : 파서의 통합 출력 ParsedScoreDocument
 */
type ParseDocumentFn = (
  document: RuntimeDocument,
) => ParsedScoreDocument;

/** 
 * 타입명 : ParseDocumentRangeFn
 * 부분 갱신시의 파싱 함수 인터페이스
 * 인수
 *    - 기존의 ParsedScoreDocument
 *    - 악보 파일 원본 ScoreFile
 *    - 런타임 조회용 인덱스 ScoreIndexes 
 *    - 상위 모듈이 계산한 부분 갱신 범위 ParseRangeRequest
 * 반환값 : 갱신된 ParsedScoreDocument
 */
type ParseDocumentRangeFn = (
  prev: ParsedScoreDocument,
  document: RuntimeDocument,
  request: ParseRangeRequest,
) => ParsedScoreDocument;
```

```ts
/** 타입명 : ParseRangeRequest
 * 전역/노트 영역의 부분 갱신 범위
 */
type ParseRangeRequest =
  | {
      scope: "track";
      trackId: string;
      colStart: number;
      colEnd: number;
    }
  | {
      scope: "global";
      kind: GlobalKind;
      colStart: number;
      colEnd: number;
    };
```

설계 원칙:

- `ParseDocumentFn`은 full rebuild용이다.
- `ParseDocumentRangeFn`은 partial rebuild용이다.
- parser의 public document-level API는 `RuntimeDocument`를 받아 `ScoreFile`과 `ScoreIndexes`의 짝이 어긋나지 않도록 한다.
- parser 내부 helper는 필요에 따라 `document.score`와 `document.indexes`를 분해해 사용할 수 있다.
- partial rebuild는 해당 범위의 parsed 결과만 다시 계산한 뒤 기존 `ParsedScoreDocument`에 반영하는 방식으로 구현할 수 있다.
- track 셀 수정과 global 셀 수정은 같은 함수 축을 공유하되, request의 `scope`로 처리 경로를 분기하는 편이 좋다.


## 7. analyzer 단계 인터페이스

analyzer는 `ParsedScoreDocument`와 `ScoreIndexes`를 입력으로 받아 `AnalysisResult`를 생성한다.
즉 analyzer의 직접 입력은 더 이상 raw JSON이나 raw 문자열이 아니라, “정규화된 문서 + parser 결과”이다.

```ts
type AnalyzeContext = {
  score: ScoreFile;
  indexes: ScoreIndexes;
  parsed: ParsedScoreDocument;
};
```

### 7.1. full analysis

```ts
/**
 * 타입명 : AnalyzeDocumentFn
 * 전체 분석시 사용하는 함수 인터페이스
 * 인수 : 문맥 분석용 AnalyzeContext
 * 반환값 : 종합 분석 결과 AnalysisResult
 */

type AnalyzeDocumentFn = (
  context: AnalyzeContext,
) => AnalysisResult;
```

이 함수는 아래 상황에서 사용한다.

- 문서를 처음 로드했을 때
- validator 결과가 크게 바뀌었을 때
- partial analysis보다 full rebuild가 더 단순하고 안전한 경우

### 7.2. partial analysis

```ts
/**
 * 타입명 :  AnalyzeDocumentPartialFn
 * 부분 갱신 시 분석에 사용하는 함수 인터페이스
 * 인수
 *    - 기존의 AnalysisResult
 *    - 이전 모듈이 넘긴 출력물의 집합 AnalyzeContext
 *    - 부분 갱신 범위 AnalyzePartialRequest
 * 반환값 : 갱신된 AnalysisResult
 */

type AnalyzeDocumentPartialFn = (
  prev: AnalysisResult,
  context: AnalyzeContext,
  request: AnalyzePartialRequest,
) => AnalysisResult;

// 전역/노트 영역의 부분 갱신 범위
type AnalyzePartialRequest =
  | {
      scope: "track";
      trackId: string;
      colStart: number;
      colEnd: number;
    }
  | {
      scope: "global";
      kind: GlobalKind;
      colStart: number;
      colEnd: number;
    };
```

partial analysis는 `1.7`의 부분 재분석 규칙을 따른다.
즉 입력 request가 단순히 `[colStart, colEnd]`를 받더라도, 실제 analyzer 내부에서는 hold / gliss / tuplet / timing 구조에 따라 범위를 추가로 확장할 수 있다.
global partial request의 `kind`는 "변경이 발생한 전역 행 종류"를 뜻하며, analyzer는 필요시 내부적으로 timing 관련 kind 묶음 전체로 재분석 범위를 확장할 수 있다.

또한 global 수정은 note 수정과 다르게 처리해야 한다.

- BPM, beatsPerBar, stepsPerBeat 수정
  - `timingTimeline` 재분석이 우선이다.
- dynamics 수정
  - `dynamicsTimeline` 재분석이 우선이다.
- global 수정이 track 이벤트 구조까지 바꾸지 않는 경우
  - `trackResults`는 재사용할 수 있다.


## 8. 전역 구조 분석 함수

timing timeline과 dynamics timeline은 track 이벤트와는 성격이 다르므로, 별도 함수로 분리하는 것이 좋다.
이 함수들은 raw JSON이 아니라 parser가 생성한 `ParsedGlobalCell` 결과를 소비해야 한다.

```ts
/**
 * 타입명 : AnalyzeTimingTimelineFn
 * 시간 세그먼트 생성 함수 인터페이스 
 * 인수
 *    - 문맥 분석용 AnalyzeContext
 *    - 선택적 분석 범위 AnalyzeGlobalRange
 * 반환값 : 배열 AnalyzedTimeSegment[]
 */
type AnalyzeTimingTimelineFn = (
  context: AnalyzeContext,
  range?: AnalyzeGlobalRange,
) => AnalyzedTimeSegment[];

/**
 * 타입명 : AnalyzeDynamicsTimelineFn
 * 셈여림 세그먼트 생성 함수 인터페이스 
 * 인수
 *    - 문맥 분석용 AnalyzeContext
 *    - 선택적 분석 범위 AnalyzeGlobalRange
 * 반환값 : 배열 AnalyzedDynamicsSegment[]
 */
type AnalyzeDynamicsTimelineFn = (
  context: AnalyzeContext,
  range?: AnalyzeGlobalRange,
) => AnalyzedDynamicsSegment[];
```

```ts
// 분석 범위 지정 타입
type AnalyzeGlobalRange = {
  kinds: GlobalKind[];
  colStart: number;
  colEnd: number;
};
```

이 구조를 두는 이유는 다음과 같다.

- 전역 행 수정 시 track 이벤트 전체를 다시 계산하지 않고도 timing / dynamics만 재분석할 수 있다.
- `analyze_full.ts`와 `analyze_partial.ts`가 공통으로 호출할 수 있다.
- `globalCellsByKindAndCol`, `globalCellsInColOrder`, `ParsedGlobalCell`을 같은 축에서 소비할 수 있다.
- playback 관련 캐시 무효화 범위를 분리하기 쉽다.


## 9. track 이벤트 분석 함수

track 내부 note / rest / mute / gliss / tupletGroup 이벤트는 track별로 분리해서 계산하는 편이 구현과 테스트 양쪽에 유리하다.

```ts
/**
 * 타입명 : AnalyzeTrackEventsFn
 * 한 트랙 내부의 이벤트를 분석하는 함수 인터페이스  
 * 인수
 *    - 분석 대상 트랙의 trackId
 *    - 문맥 분석용 AnalyzeContext
 *    - 부분 분석 범위 제한용 AnalyzeTrackRange
 * 반환값 : AnalyzedTrackResult
 */
type AnalyzeTrackEventsFn = (
  trackId: string,
  context: AnalyzeContext,
  range?: AnalyzeTrackRange,
) => AnalyzedTrackResult;
```

```ts
// 분석 범위 지정 타입
type AnalyzeTrackRange = {
  colStart: number;
  colEnd: number;
};
```

설계 원칙:

- full analysis에서는 `range` 없이 전체 track을 분석한다.
- partial analysis에서는 확장된 열 범위만 분석한다.
- 반환값은 항상 `AnalyzedTrackResult` 형태로 고정한다.
- 이 함수는 `ParsedCell`과 track 인덱스를 직접 소비하고, 전역 timeline은 `AnalysisResult` 상위 레벨에서 결합하는 편이 책임 경계상 더 명확하다.


## 10. 캐시와 보조 구조

공식 출력 구조는 `AnalysisResult`이지만, 실제 구현에서는 분석 비용과 조회 비용을 줄이기 위해 보조 캐시가 필요할 수 있다.
이 캐시는 저장 구조가 아니라 런타임 파생 구조로 간주한다.

```ts
type AnalysisCache = {
  eventsByStartCol: Map<string, Map<number, AnalyzedEvent[]>>;
  trackResultByTrackId: Map<string, AnalyzedTrackResult>;
  tupletGroupById: Map<string, TupletGroupEvent>;
  glissEventsById: Map<string, GlissEvent[]>;
};
```

설계 원칙:

- 캐시는 `AnalysisResult`로부터 다시 만들 수 있어야 한다.
- 캐시는 진실 원천이 아니다.
- partial analysis 시 필요한 범위만 무효화하는 것이 가능해야 한다.
- global 수정의 경우에도 `AnalysisResult` 전체를 직접 수정하기보다, timeline 재생성 후 파생 캐시를 다시 만드는 편이 더 안전하다.
- 같은 원칙으로 `ParsedScoreDocument` 역시 공식 parsed 결과의 진실 원천이며, parser 단계의 보조 배열/목록/역인덱스는 파생 캐시로만 둔다.


## 11. 구현 모듈 분리 초안

실제 TypeScript 구현에서는 아래와 같은 파일 구조를 우선 후보로 둘 수 있다.

```ts
src/core/parse/parse_note_cell.ts
src/core/parse/parse_global_cell.ts
src/core/parse/parse_document.ts
src/core/parse/parse_document_range.ts

src/core/analyze/analyze_types.ts
src/core/analyze/analyze_full.ts
src/core/analyze/analyze_partial.ts
src/core/analyze/analyze_track.ts
src/core/analyze/analyze_timing.ts
src/core/analyze/analyze_dynamics.ts
src/core/analyze/analyze_cache.ts
```

모듈 책임은 다음과 같이 나누는 편이 좋다.

- `parse_note_cell.ts`
  - 단일 note 셀 문자열을 `ParsedCell`로 해석

- `parse_global_cell.ts`
  - 단일 global 셀 문자열을 `ParsedGlobalCell`로 해석

- `parse_document.ts`
  - 문서 전체를 `ParsedScoreDocument`로 변환

- `parse_document_range.ts`
  - track 범위 또는 global 범위 단위 partial parse

- `analyze_types.ts`
  - analyzer 관련 공용 타입 정의

- `analyze_full.ts`
  - full analysis 오케스트레이션

- `analyze_partial.ts`
  - partial analysis 오케스트레이션

- `analyze_track.ts`
  - track 이벤트 계산

- `analyze_timing.ts`
  - `AnalyzedTimeSegment` 계산

- `analyze_dynamics.ts`
  - `AnalyzedDynamicsSegment` 계산

- `analyze_cache.ts`
  - 파생 캐시 생성 및 무효화


## 12. 권장 진행 순서

이 문서 기준으로 실제 구현에 들어갈 때의 권장 순서는 다음과 같다.

1. `ScoreIndexes` 생성 함수 작성
2. `ParsedScoreDocument` full build 함수 작성
3. `parse_note_cell`, `parse_global_cell` 단일 셀 parser 작성
4. `AnalyzeTimingTimelineFn`, `AnalyzeDynamicsTimelineFn` 작성
5. `AnalyzeTrackEventsFn`의 full analysis 버전 작성
6. `AnalyzeDocumentFn` 작성
7. 이후 partial parse / partial analysis 추가

즉 초기 구현은 full build를 먼저 안정화하고, 그 다음 partial analysis를 붙이는 순서가 바람직하다.


## 13. 결론

회귀 코드의 parser와 analyzer는 개별 알고리즘보다도 먼저, 서로 어떤 입력과 출력을 주고받는지를 안정적으로 고정해야 한다.
이 문서의 목적은 바로 그 인터페이스 경계를 명확히 하는 데 있다.

이 경계가 고정되면 이후 구현은 다음과 같은 장점을 얻는다.

- note 셀과 global 셀을 같은 문서 단위 파이프라인으로 묶을 수 있다.
- parser와 analyzer를 독립적으로 테스트할 수 있다.
- UI / loader / validator / renderer / audio가 공통 계약에 맞춰 연결될 수 있다.
- partial parse와 partial analysis를 같은 축으로 설계할 수 있다.
- 전역 행 수정 시 timing / dynamics 재분석만 수행하는 경로를 확보할 수 있다.
- 모듈 분리 기준이 명확해진다.

따라서 본 문서는 회귀 코드의 실제 TypeScript 구현으로 넘어가기 전, 파이프라인 인터페이스를 고정하는 중간 단계 설계 문서로 사용한다.
