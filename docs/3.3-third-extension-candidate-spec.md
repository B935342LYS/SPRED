## 2. 현재 노래방 / Practice Mode 안정화 상태

현재 practice mode는 초기 목표 기능을 넘어 1차 확장 기능까지 상당 부분 구현된 상태이다.

구현 완료 또는 1차 구현 완료로 보는 항목:

- practice mode 진입 / 종료 UI
- 마이크 권한 요청과 pitch input lifecycle
- `pitchy` 기반 pitch frame 생성
- pitch dot overlay
- active track `NoteEvent` 기반 pitch scoring
- Easy / Standard / Pro judge mode
- timing onset 기반 early / late / Bad / Miss downgrade
- attack credit / score eligibility 분리
- result dialog와 live summary
- practice Sync 보정값과 localStorage 저장
- gliss effect bonus
- vib effect bonus
- explicit trem modifier target과 1tick rapid repeat trem target
- trem attack streak bonus
- practice 진입 시 edit mode 자동 종료
- practice 진입 시 자동 Fit Height
- compact mobile layout 실험 및 배포 반영
- YouTube sync와 practice mode 병행 확인
- YouTube negative offset 시작 대기 보정

남은 안정화 후보:

1. 실제 기기별 practice latency 재검증
   - PC, 저전력 노트북, Android Chrome, Samsung Browser에서 Sync 기본값과 조정 범위를 확인한다.
   - 현재 기본값은 `+90ms`로 유지한다.
   - 사용자 검증 결과 기기별 지연 차이가 더 클 수 있으므로 Sync 조정 범위는 `-500ms..+500ms`로 확장한다.
2. Trem 실기기 검증
   - explicit trem과 1tick rapid repeat trem이 실제 마이크 입력에서 과소/과대 인식되지 않는지 확인한다.
   - 현재 상수 `60ms..450ms`, `streak >= 2`, `trackDifficulty * 0.12`의 체감 비중을 확인한다.
   - 2-division 수준의 빠른 점멸은 마이크/브라우저 입력에서 attack으로 안정 검출되지 않을 수 있으므로, 별도 고정밀 판정 대상이 아니라 한계 사례로 기록한다.
   - 1tick rapid repeat trem 구간과 2~4 division explicit trem 구간은 trem으로 인식된 pitch hit이 timing/attack 조건 때문에 낮게 판정되지 않도록 Perfect 완화를 적용한다.
   - 2~4 division explicit trem은 단일 note event라도 division 위치별 내부 relaxed hit id를 만들어 여러 기본 점수 sample로 집계될 수 있게 한다.
   - 이 Perfect 완화는 pitch accuracy와 timing accuracy 모두 `100%`로 집계한다.
3. 모바일 compact UI 잔여 검증
   - 현재 모바일 compact UI는 충분한 것으로 판단한다.
   - 후속 변경은 새 모바일 회귀가 확인될 때만 진행한다.
4. YouTube offset 저장 정책
   - 혼동을 줄 수 있으므로 score offset과 device offset은 분리하지 않는다.
   - 현재처럼 score의 `musicData.youtube.offsetMs`만 사용한다.
5. 장시간 session 안정성
   - 긴 곡에서 microphone loop, playback RAF, effect target lookup, YouTube follower가 누적 lag를 만들지 않는지 확인한다.
   - 현재 사용자 확인 기준으로 5분 30초 길이 악보까지 정상 동작을 확인했다.
6. 문서 최신화
   - `docs/3.2-karaoke-game-mode-spec.md`의 일부 follow-up 표기를 현재 구현 상태에 맞게 정리한다.
   - 최신 deployment commit과 수동 검증 결과를 implementation memo에 기록한다.

3차 확장 착수 조건:

- `npm run test:game`, `npm run test:youtube`, `npm run build`가 통과한다.
- practice mode의 Easy / Standard / Pro가 최소 한 곡에서 실제 사용자 테스트를 통과한다.
- Trem bonus가 실기기에서 과도한 오탐을 만들지 않는다.
- YouTube negative offset과 일반 positive offset이 모바일에서 동작한다.
- compact mobile layout이 악보 시야를 심각하게 해치지 않는다.

우선순위 초안:

1. Drag note length / pitch edit
2. MIDI import
3. 조바꿈과 전체 remap
4. Track별 dynamics / timbre
5. 오디오 분석 자동 악보 생성
6. 다현 악보 3D 전개

이 순서는 기술 난도만이 아니라 현재 사용 흐름을 얼마나 직접 개선하는지를 함께 본 것이다.

## 4. MIDI 파일 변환 후 ScoreFile Import

### 4.1. 목표

사용자가 `.mid` 또는 `.midi` 파일을 불러오면, MIDI note event를 `ScoreFile`의 track cell 구조로 변환한다.

첫 목표는 완성 악보 자동 생성이 아니라 edit mode에서 보정 가능한 초안 생성이다.

### 4.2. 입력과 출력

입력:

- Standard MIDI File
- tempo events
- time signature events
- note on / note off
- track 또는 channel 정보
- pitch bend는 1차에서 제외하거나 best-effort로 cent 후보만 기록한다.

출력:

- 새 `ScoreFile`
- 또는 현재 score에 import preview 후 선택 track에 삽입

### 4.3. 변환 정책 초안

- MIDI tick을 score column으로 quantize한다.
- note on은 note cell rawText로 변환한다.
- note duration은 `-` hold cell로 변환한다.
- 짧은 note가 1tick 이하로 반복되면 rapid repeat trem 후보가 될 수 있다.
- MIDI track 또는 channel은 `basic` / `optional` / `extra` 중 하나로 mapping한다.
- percussion channel은 첫 구현에서 제외한다.
- polyphony는 가장 가까운 pitch row 하나로 단순화하거나, track 분산 후보로 표시한다.

### 4.4. UI 흐름

1. File menu에 `MIDI Import` 추가
2. 파일 선택
3. import preview dialog 표시
4. quantize resolution, target track, pitch range, transpose 값을 선택
5. ScoreFile 생성 또는 현재 score에 적용

### 4.5. 영향 범위

- storage / import
- parser rawText compose
- layout row mapping
- partial rebuild 또는 full rebuild
- undo history

### 4.6. 포기 기준

- MIDI parser 의존성이 브라우저 번들에 과도하게 크다.
- quantize 결과가 사람이 고치는 것보다 더 번거롭다.
- polyphony 처리 때문에 단일 오타마톤 악보 목적을 흐린다.

## 5. 오디오 분석 후 악보 자동 생성

### 5.1. 목표

사용자가 오디오 파일을 업로드하면 melody 후보를 추출하고, 이를 ScoreFile 초안으로 변환한다.

명확한 비목표:

- YouTube iframe의 오디오를 직접 분석하지 않는다.
- 저작권 있는 음원을 앱이 다운로드, 캐시, 추출, 저장하지 않는다.
- 완성 악보를 자동으로 보장하지 않는다.

### 5.2. YouTube와의 경계

GitHub Pages 웹앱은 YouTube iframe의 audio buffer를 Web Audio API로 직접 읽을 수 없다.
따라서 YouTube URL만으로 자동 채보하는 기능은 첫 구현 범위에서 제외한다.

가능한 입력은 사용자가 직접 제공한 local audio file이다.
사용자는 자신이 분석할 권리를 가진 파일만 사용해야 한다.

### 5.3. 구현 후보

1. Basic Pitch 계열
   - 오디오를 MIDI 후보로 변환한다.
   - pitch bend 후보가 있어 fretless / gliss 후보 분석에 유리할 수 있다.
   - browser TypeScript 경로와 Python/offline 경로를 모두 검토한다.
2. pitch contour 기반 단순 melody extraction
   - `pitchy`, `librosa`, `Essentia` 계열의 pitch frame을 note segmentation으로 후처리한다.
   - monophonic melody에 한정하면 구현 가능성이 높다.
3. 외부 offline 변환 결과 import
   - 앱 내부 분석을 포기하고, 사용자가 별도 도구로 만든 MIDI를 import한다.
   - 3차 확장에서 가장 안전한 fallback이다.

### 5.4. 처리 pipeline 초안

```text
Audio file
  -> decode audio
  -> pitch / onset / beat analysis
  -> melody note candidates
  -> quantize
  -> row mapping
  -> ScoreFile draft
  -> manual correction in edit mode
```

### 5.5. 오픈소스 검토 메모

- Spotify Basic Pitch
  - URL: `https://github.com/spotify/basic-pitch`
  - TypeScript 후보: `https://github.com/spotify/basic-pitch-ts`
  - 사용 방식 후보: 직접 의존 또는 offline 비교 검토
  - 라이선스는 구현 착수 직전 재확인한다.
- Google Magenta Onsets and Frames
  - URL: `https://magenta.withgoogle.com/onsets-frames`
  - 주로 piano transcription에 강하므로 일반 melody / 오타마톤 악보에는 비교 후보로 둔다.
- Essentia / Essentia.js
  - URL: `https://essentia.upf.edu/`
  - melody extraction, feature extraction, onset/beat 보조 분석 후보이다.
- librosa
  - URL: `https://librosa.org/doc/main/generated/librosa.pyin.html`
  - browser 직접 의존보다는 offline / Python 실험 후보이다.

오픈소스 코드를 직접 복사하지 않는다.
직접 의존 또는 알고리즘 참고가 필요하면 license와 bundle size를 별도 기록한다.

### 5.6. 포기 기준

- 결과가 noise와 반주에 너무 취약해 edit 보조로도 가치가 낮다.
- browser bundle size나 실행 시간이 과도하다.
- 저작권 / 음원 입력 UX가 프로젝트 목적을 흐린다.
- MIDI import만으로 충분한 대체 경로가 된다.

## 6. 다현 악보 3D 전개

### 6.1. 목표

다현 악기 또는 여러 string surface를 3D 공간에 펼쳐, string별 note row와 이동을 시각적으로 이해할 수 있게 한다.

### 6.2. 현재 구조와의 관계

현재 `instData.strings[]`는 다현 확장 가능성을 일부 가진다.
그러나 실제 renderer, edit, analyzer target은 단일 row plane 중심이다.

3D 전개는 다음 경계를 건드린다.

- layout preset format
- rowId와 stringId mapping
- hit test
- renderer layer
- edit mode interaction
- track layer visibility

### 6.3. 구현 방향 후보

1. Read-only 3D viewer
   - ScoreFile을 변경하지 않는다.
   - 현재 analyzer 결과를 3D로 보조 표시한다.
   - Three.js 사용 후보.
2. 3D layout editor
   - string별 row를 3D에서 직접 선택하고 편집한다.
   - 난도가 매우 높아 후순위로 둔다.
3. 2.5D unfolded view
   - 실제 3D 대신 string group을 겹치지 않게 기울여 배치한다.
   - 구현 난도와 사용성을 절충한다.

### 6.4. 포기 기준

- 2D 악보 작성/연습 흐름보다 시각적 novelty가 앞선다.
- mobile / low-end device에서 렌더링 성능이 떨어진다.
- 편집 기능과 연결하려면 hit test 복잡도가 지나치게 커진다.

## 7. 조바꿈과 모든 Note Remap

### 7.1. 목표

현재 score의 모든 note를 일정 interval만큼 이동하고, 표시 계이름 또는 default text도 함께 갱신한다.

### 7.2. 변환 대상

- default note text
- absolute pitch modifier
- hold chain의 start anchor
- gliss start/end anchor
- tuplet slot note
- layout rowId remapping
- note label / 계이름 텍스트

### 7.3. 난점

- rawText 안의 사용자 custom text가 실제 계이름인지 일반 문자열인지 구분하기 어렵다.
- row 이동 후 layout 범위를 벗어난 note를 어떻게 처리할지 결정해야 한다.
- cent offset과 absolute pitch가 섞인 경우, 단순 row shift만으로 의미가 보존되지 않을 수 있다.
- gliss와 tuplet은 source cell 하나가 여러 note event를 만들 수 있다.

### 7.4. 정책 후보

1. Structural transpose
   - rawText 의미를 parse한 뒤 note 의미만 바꾼다.
   - 가장 정확하지만 parser compose 기능이 더 필요하다.
2. Row remap only
   - cell rowId를 이동하고 rawText는 그대로 둔다.
   - 빠르지만 계이름 텍스트가 틀릴 수 있다.
3. Assisted transpose
   - 변환 preview에서 위험 cell을 표시하고 사용자가 선택한다.
   - 실용성과 안전성을 절충한다.

### 7.5. 첫 구현 추천

- 새 score copy를 만들어 preview한다.
- layout 범위를 벗어나는 cell은 변환하지 않고 warning으로 표시한다.
- custom text는 자동 변환하지 않는다.
- parser가 확실히 note pitch로 해석한 항목만 변환한다.

## 8. Track별 개별 Dynamics와 음색

### 8.1. 목표

현재 track별 playback gain보다 더 세밀하게 track마다 dynamics curve와 timbre를 적용한다.

### 8.2. 기능 후보

- track별 volume slider
- track별 mute / solo
- track별 wave type
- track별 envelope
- track별 dynamics row 적용 여부
- track별 instrument preset

### 8.3. 저장 정책 후보

1. ScoreFile 저장
   - 곡이 의도한 mix를 보존할 수 있다.
   - 파일 format 확장이 필요하다.
2. local user preference
   - 사용자 playback 환경 조정에 적합하다.
   - score 공유 시 mix가 보존되지 않는다.
3. hybrid
   - score default와 user override를 분리한다.

첫 구현은 local user preference가 안전하다.
ScoreFile format 확장은 실제 필요가 확인된 뒤 검토한다.

### 8.4. 영향 범위

- audio schedule builder
- oscillator backend 또는 future instrument backend
- track control UI
- local preference 저장
- result/practice mode의 playback balance

## 9. Drag Note Length / Pitch Edit

### 9.1. 목표

사용자가 악보 위 note를 직접 드래그해 길이와 높이를 바꿀 수 있게 한다.

현재 edit mode는 cell 단위 입력과 drag install/delete 중심이다.
이 기능은 실제 악보 작성 효율을 크게 높일 수 있다.

### 9.2. 조작 후보

- note right edge drag: hold 길이 변경
- note body vertical drag: row pitch 변경
- modifier handle drag: gliss end pitch 또는 end column 변경
- Shift drag: snap 해제 또는 fine control
- Esc: drag 취소

### 9.3. 구현 전제

- renderer item과 source cell mapping이 안정적이어야 한다.
- drag 중 preview와 실제 rawText commit을 분리해야 한다.
- undo history는 drag 전체를 하나의 transaction으로 묶어야 한다.
- partial rebuild가 drag update 빈도를 감당해야 한다.

### 9.4. 첫 구현 추천

1. single note event의 right edge drag만 구현한다.
2. 변환 결과는 `-` hold cell 추가/삭제로 제한한다.
3. gliss, tuplet, trem, multi-source event는 첫 구현에서 handle drag 대상에서 제외한다.
4. drag end 시 한 번만 commit하고, drag 중에는 DOM/canvas preview만 표시한다.

### 9.5. 포기 기준

- hit test ambiguity 때문에 의도하지 않은 note가 자주 수정된다.
- drag preview와 실제 rawText 결과가 사용자가 이해하기 어렵다.
- partial rebuild 없이 drag commit 후 지연이 심하다.

## 10. 공통 구조 원칙

3차 확장은 다음 원칙을 지킨다.

- ScoreFile 저장 형식 변경은 마지막 수단으로 둔다.
- 변환 기능은 원본을 직접 덮어쓰기 전에 preview와 cancel 경로를 제공한다.
- import / automatic generation은 완성 결과가 아니라 editable draft를 만든다.
- parser / analyzer / renderer / audio boundary를 유지한다.
- runtime-only 기능은 score JSON에 저장하지 않는다.
- browser-only GitHub Pages 배포 범위를 넘는 기능은 별도 phase로 둔다.
- 고비용 기능은 실제 성능 측정과 포기 기준을 먼저 둔다.

## 11. 3차 확장 착수 순서 초안

### Phase 3A. Editing Power Tools

목표:

- drag note length edit
- 제한적 pitch row move
- transpose preview

이유:

- 현재 사용자가 직접 체감하는 악보 작성 효율을 높인다.
- 자동 생성보다 결과 품질을 통제하기 쉽다.

### Phase 3B. Import Tools

목표:

- MIDI import
- MIDI quantize preview
- imported draft correction workflow

이유:

- 자동 오디오 분석보다 기술 위험이 낮다.
- 이후 audio-to-MIDI 결과를 같은 import pipeline에 태울 수 있다.

### Phase 3C. Audio / AI Assisted Draft

목표:

- local audio file analysis experiment
- Basic Pitch 또는 pitch contour 후보 비교
- melody draft import

이유:

- 연구성 기능이므로 MIDI import pipeline이 먼저 있어야 실패 비용이 낮다.

### Phase 3D. Presentation / Instrument Expansion

목표:

- track별 timbre / dynamics
- 다현 2.5D 또는 read-only 3D viewer

이유:

- core editing이 안정된 뒤 시각화와 재생 품질을 넓힌다.

## 12. 검증 후보

공통:

```text
npm run test:score
npm run test:parse
npm run test:analyze
npm run test:edit
npm run test:audio
npm run test:game
npm run build
```

MIDI import:

- 단일 melody MIDI를 ScoreFile로 변환한다.
- tempo change MIDI를 변환해 event seconds가 크게 어긋나지 않는다.
- note off가 없는 비정상 MIDI를 안전하게 거부한다.

Audio draft:

- monophonic test wav에서 melody 후보를 만든다.
- noisy audio에서 낮은 confidence 구간을 draft에서 제외한다.
- analysis failure가 score를 훼손하지 않는다.

Drag edit:

- drag 시작 전/후 rawText patch가 undo 1회로 되돌아간다.
- hold 길이 증가/감소가 analyzer event duration에 반영된다.
- drag preview 중에는 full rebuild가 반복되지 않는다.

Transpose:

- rowId remap이 layout 범위 안에서만 적용된다.
- gliss와 tuplet이 변환 불가 또는 preview warning으로 분류된다.
- custom text는 자동 계이름 변환하지 않는다.

## 13. 현재 결론

3차 확장 후보 중 가장 현실적인 첫 착수 대상은 `drag note length / pitch edit`와 `MIDI import`이다.

오디오 분석 자동 악보 생성은 매력적이지만, GitHub Pages 단독 웹앱에서 YouTube 음원을 직접 분석하는 기능은 범위 밖으로 둔다.
대신 사용자가 권리를 가진 local audio file을 분석해 editable draft를 만드는 연구 기능으로 한정한다.

다현 3D 전개는 시각적으로 강하지만 core editing과 저장 구조에 미치는 영향이 크다.
따라서 read-only viewer 또는 2.5D preview부터 검토한다.

조바꿈, track별 dynamics/timbre, drag edit는 실사용 편의성이 크므로 3차 확장 후보 중 비교적 높은 우선순위를 가진다.
