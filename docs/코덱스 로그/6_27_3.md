문맥 이해는 이렇게 정리됩니다.

`회귀 코드`는 레거시 `이세계 코드`를 그대로 호환하는 포팅이 아니라, `TypeScript + HTML + CSS + GitHub Pages` 기반으로 다시 만든 별도 웹 앱입니다. 현재 주요 레거시 기능 복원, 오디오/렌더러/편집/트랙/레이아웃/YouTube/성능 안정화가 상당히 진행되었고, 게임 모드=노래방 모드는 아직 구현 착수 대상이 아니라 “후순위 확장 기능의 명세 검토 단계”입니다.

게임 모드의 핵심 방향은 명확합니다.

- 사용자 연주를 브라우저 마이크 입력으로 받는다.
- 실시간 pitch detection 결과를 analyzer가 만든 `AnalysisResult`, 특히 active track의 `NoteEvent`와 비교한다.
- score JSON, parser rawText, 저장 포맷은 바꾸지 않는다.
- pitch frame, 판정 결과, 점수, 랭크는 모두 런타임 데이터이며 JSON/localStorage/파일에 저장하지 않는다.
- 첫 구현은 데스크톱 브라우저 기준의 실용적인 연습 보조 모드다.
- 모바일, 완전 저지연, 고정밀 채점, gliss/vib/trem/timing 정밀 평가는 후순위다.

현재 가장 구체적인 기준은 `docs/3.2-karaoke-game-mode-spec.md`입니다. 여기서는 사용자 표시명은 `practice mode`, 내부 구현명은 `game` 또는 `gameMode`로 두는 방향이며, 예상 모듈은 `src/app/game/` 아래에 `game_types`, `game_target_builder`, `game_pitch_input`, `game_pitch_math`, `game_judge`, `game_score`, `game_overlay`, `game_binding`처럼 분리하는 안입니다.

1차 구현 범위는 다음 정도로 보입니다.

1. `practice mode` 버튼과 진입/종료 상태 추가
2. 마이크 권한 요청 및 pitch detector 초기화
3. `GamePitchFrame` 생성
4. 재생 기준선 근처 pitch dot 표시
5. active track `NoteEvent`를 `GameJudgeTarget`으로 변환
6. 현재 score time과 사용자 pitch를 비교해 `Good / Ok / Hmm / Nope` 판정
7. 평균 정확도, 처리한 이벤트 수, 총점 표시
8. edit, seek, loop, score load, layout apply, active track 변경 같은 판정 공정성을 깨는 동작 차단

판정 정책도 꽤 정해져 있습니다.

- 첫 구현은 pitch class 허용: 같은 계이름이면 옥타브가 달라도 정답으로 인정
- 음정 오차 기준 후보:
  - `< 50 cent`: `Good`, 100%
  - `< 100 cent`: `Ok`, 80%
  - `< 200 cent`: `Hmm`, 20%
  - `>= 200 cent`: `Nope`, 0%
- pitch detection 후보는 `pitchy` prototype 우선
- YouTube sync와 공존 가능하지만, video/offset 변경은 `ready` 상태에서만 허용
- loop와 임의 seek는 게임 모드에서 금지
- stop/처음으로 돌아가기는 점수 리셋 후 `ready`
- pause/resume은 점수 유지

검토 포인트도 하나 있습니다. `docs/3.0`에서는 점수 산정에 coverage, 즉 전체 대상 대비 판정 참여율을 반영하는 방향을 제안합니다. 반면 `docs/3.2`에서는 “처리한 이벤트 비율은 1차 공식에 별도 포함하지 않고, 초안의 `middleScore += trackDifficulty * pitchAccuracy`, `totalScore = middleScore * averageAccuracy`를 유지”한다고 더 구체적으로 정리되어 있습니다. 문서 우선순위상 현재 게임 모드 세부 검토에서는 `3.2`의 결정이 더 최신/구체적인 기준으로 보이지만, 추후 공정성 논의에서는 이 차이를 다시 확인하는 게 좋습니다.

결론적으로, 게임 모드는 기존 코어를 흔드는 기능이 아니라 `AnalysisResult` 위에 얹는 별도 런타임 평가 계층입니다. 지금 단계의 명세 검토 초점은 “마이크 입력과 판정이 현재 playback 시간축, active track, YouTube follower 구조와 충돌하지 않게 상태/금지 동작/점수 산식을 고정하는 것”으로 이해했습니다.

> 점수 계산식과 판정 기준은 합리적인지 모르겠습니다.

현재 `3.2`의 점수식과 판정 기준은 “프로토타입용”으로는 충분히 단순하지만, 그대로 확정하기엔 약점이 있습니다.

가장 큰 문제는 점수식입니다.

```text
middleScore += trackDifficulty * pitchAccuracy
totalScore = middleScore * averageAccuracy
```

이 방식은 정확도를 두 번 곱하는 구조라서, 고득점과 저득점의 차이를 크게 벌리는 효과는 있지만 직관성이 떨어집니다. 또 `processed event count`만 쓰고 전체 대상 대비 누락률을 직접 반영하지 않으면, 일부 이벤트를 아예 놓친 경우가 점수에 충분히 불리하게 반영되지 않을 수 있습니다.

더 합리적인 1차 공식은 `3.0`에 있던 coverage 개념을 살리는 쪽입니다.

```text
accuracy = 판정된 이벤트들의 가중 평균 정확도
coverage = hit 또는 miss로 처리된 이벤트 수 / 전체 target 이벤트 수
difficultyScore = trackDifficulty 가중치를 반영한 원점수
finalScore = difficultyScore * accuracy * coverage
```

또는 누락을 더 강하게 벌하려면:

```text
finalScore = difficultyScore * accuracy * coverage^2
```

다만 게임 모드가 “정식 랭킹”이 아니라 “연습 보조”라면 너무 엄격할 필요는 없습니다. 제 판단으로는 1차 구현에서는 아래가 가장 균형이 좋습니다.

```text
eventScore = trackWeight * pitchAccuracy
rawScore = sum(eventScore)
maxScore = sum(trackWeight for all target events)
accuracy = rawScore / maxScore
coverage = judgedEventCount / totalTargetEventCount
finalScore = 100000 * accuracy * coverage
```

이러면 의미가 명확합니다.

- `accuracy`: 맞춘 음정의 품질
- `coverage`: 얼마나 빠뜨리지 않았는지
- `finalScore`: 둘을 합친 최종 점수

판정 기준도 현재 초안은 방향은 괜찮지만 약간 관대합니다.

```text
< 50 cent   Good 100%
< 100 cent  Ok   80%
< 200 cent  Hmm  20%
>= 200 cent Nope 0%
```

오타마톤/마이크/브라우저 pitch detection 불안정성을 고려하면 1차 기준으로는 현실적입니다. 다만 “같은 계이름이면 옥타브가 달라도 정답”까지 켜면 너무 쉬워질 수 있습니다. 연습 보조 목적이라면 허용할 수 있지만, 점수 의미가 “정확한 연주”보다는 “멜로디 계이름 추적”에 가까워집니다.

추천 기준은 이렇게 나누는 것입니다.

```text
Practice 기본 모드:
- pitch class 허용
- < 50 cent: Good
- < 100 cent: Ok
- < 200 cent: Hmm
- 그 이상: Nope

Strict 옵션 후보:
- 옥타브까지 일치해야 함
- < 35 cent: Good
- < 75 cent: Ok
- < 150 cent: Hmm
- 그 이상: Nope
```

지금 명세에 바로 반영한다면, 저는 이렇게 정리하는 편이 좋다고 봅니다.

1. 현재 `middleScore * averageAccuracy` 공식은 폐기하거나 “초기 초안”으로 내린다.
2. 1차 공식은 `accuracy * coverage` 기반으로 바꾼다.
3. pitch class 허용은 기본값으로 두되, “정밀 채점이 아니라 연습 보조용”이라고 명시한다.
4. 옥타브 정확도와 더 엄격한 cent threshold는 후속 옵션으로 둔다.
5. `Good / Ok / Hmm / Nope`는 유지하되, 실제 threshold는 prototype 측정 후 조정 가능하다고 둔다.

즉, 판정 등급 자체는 합리적인 출발점이고, 점수 계산식은 coverage를 포함하는 쪽으로 고치는 것이 더 방어적입니다.

> 같은 열에 트랙별로 이벤트가 겹쳐 존재하는 경우에는 coverage를 강제로 놓칠 수밖에 없는 경우가 생깁니다. 따라서 coverage를 넣는 것이 오히려 불합리할 수도 있습니다.

맞습니다. 현재 `회귀 코드`의 track 구조에서는 coverage를 전체 `NoteEvent` 기준으로 넣으면 불합리해질 수 있습니다.

특히 active track이 여러 개이고 같은 시간 구간에 `basic`, `optional`, `extra` 이벤트가 동시에 있으면, 사용자는 한 번에 하나의 pitch만 연주할 수 있습니다. 그런데 전체 target event 수를 분모로 coverage를 계산하면, 구조적으로 불가능한 나머지 이벤트가 `miss`처럼 작동합니다. 이 경우 coverage는 “성실히 연주했는가”가 아니라 “동시 이벤트가 얼마나 적은 악보인가”를 보상하게 됩니다.

그래서 이 프로젝트에는 단순 coverage보다 `judge slot` 또는 `judge frame group` 개념이 더 맞습니다.

권장 방향은 이렇습니다.

```text
같은 시간 판정 구간에 여러 active track NoteEvent가 겹치면
그 전체를 하나의 judge group으로 본다.

사용자 pitch는 그 group 안의 후보 NoteEvent 중
가장 가까운 target 하나와 비교한다.

그 group에서 하나라도 충분히 맞으면
해당 판정 구간은 처리된 것으로 본다.
```

그러면 coverage를 쓰더라도 분모는 `NoteEvent 개수`가 아니라 `판정 가능한 시간 구간 또는 judge group 개수`가 됩니다.

예시:

```text
col 10:
- basic: C4
- optional: E4
- extra: G4

사용자가 E4를 연주함
=> optional target으로 판정
=> 이 시점 judge group은 hit
=> basic, extra를 놓쳤다고 벌점 주지 않음
```

이 기준이면 `3.2`의 “동일 시점에 여러 active track note가 있으면 사용자 pitch와 가장 가까운 target 하나를 선택한다”는 정책과도 잘 맞습니다.

다만 이 경우에도 전체 coverage를 꼭 넣을 필요는 없습니다. 더 단순하고 합리적인 1차 공식은 다음입니다.

```text
각 판정 샘플 또는 judge group에서
가장 가까운 target 하나만 점수화한다.

finalScore = sum(selectedTarget.trackWeight * pitchAccuracy)
averageAccuracy = 평균 pitchAccuracy
processedEvents = 점수화된 judge group 수
```

그리고 “사용자가 아예 연주하지 않은 구간”만 `Nope` 또는 0점 샘플로 처리하면 됩니다. 즉 coverage를 별도 항으로 넣기보다, 무음/불인식 구간을 해당 judge group의 0점으로 반영하는 편이 자연스럽습니다.

수정 제안은 이렇게 정리할 수 있습니다.

- `coverage = judged NoteEvent / total NoteEvent`는 사용하지 않는다.
- active track의 동시/겹침 이벤트는 런타임 `judge group`으로 묶는다.
- 한 judge group에서는 사용자 pitch와 가장 가까운 target 하나만 판정한다.
- group 내 선택된 target의 `trackDifficulty`만 점수에 반영한다.
- 무음 또는 유효 pitch frame 부족은 해당 group의 `Nope` 또는 0점으로 처리한다.
- 최종 점수는 group 단위 평균 정확도와 누적 난이도 점수로 계산한다.

따라서 사용자 지적대로, 이 프로젝트에서는 coverage를 무비판적으로 넣으면 안 됩니다. `NoteEvent coverage`가 아니라 `judge group 단위 miss 처리`가 더 합리적입니다.

> 악보의 이벤트 겹침 여부를 검사하는 연산이 들어가면 오버헤드가 커집니다. 차라리 아예 자유롭게 풀어두는것이 좋지 않습니까? 놓치면 감점하지 않더라도 놓치지 않을 때보다 점수가 적어 페널티가 여전히 있기도 하고요.

네, 그 방향이 더 실용적일 수 있습니다.

게임 모드를 “공정한 랭킹 채점”이 아니라 “연습 보조 / 노래방 모드”로 본다면, 겹침 이벤트를 정교하게 검사해서 coverage나 judge group을 만들 필요가 크지 않습니다. 오히려 그 구조가 복잡도와 오버헤드를 늘리고, 악보 구조에 따라 예외가 많아질 가능성이 큽니다.

더 단순한 정책은 이렇습니다.

```text
판정 시점마다 현재 score time에 걸쳐 있는 active track NoteEvent 후보를 찾는다.
사용자 pitch와 가장 가까운 후보 하나만 선택한다.
가까우면 점수를 더한다.
너무 멀면 0점 또는 Nope 표시를 한다.
무음/불인식이면 점수를 더하지 않는다.
```

이 경우 놓친 이벤트를 별도로 감점하지 않아도, 점수를 얻을 기회를 놓친 것이므로 최종 점수는 자연스럽게 낮아집니다. 즉 “감점”이 아니라 “가점 기회 상실”로 처리하는 방식입니다. 이 프로젝트의 1차 게임 모드에는 이쪽이 더 맞습니다.

주의할 점은 하나 있습니다. 완전히 frame 단위로만 점수를 더하면 긴 note가 점수를 과도하게 많이 줄 수 있습니다. 하지만 이것도 나쁘다고만 볼 필요는 없습니다. 긴 음을 안정적으로 유지하는 것도 연습 가치가 있기 때문입니다. 다만 점수 폭을 제어하려면 다음 중 하나를 고르면 됩니다.

```text
A안: 샘플 기반
- 80~120ms마다 판정
- 맞은 샘플마다 점수 누적
- 긴 note 유지 능력이 점수에 반영됨

B안: 이벤트 기반
- NoteEvent 하나당 최대 1회 또는 제한된 횟수만 점수화
- 악보 길이와 note 개수 중심의 점수
- 유지력 평가는 약해짐
```

노래방 모드라면 저는 `A안: 샘플 기반`이 더 자연스럽다고 봅니다. 대신 표시값을 이렇게 분리하면 오해가 줄어듭니다.

- `Score`: 맞춘 순간마다 누적되는 총점
- `Accuracy`: 점수가 발생한 판정 샘플의 평균 정확도
- `Events` 또는 `Hits`: 실제 점수화된 판정 횟수

따라서 명세는 이렇게 바꾸는 것이 합리적입니다.

- `coverage`는 1차 점수식에서 제외한다.
- `judge group` 사전 생성도 제외한다.
- 동시 이벤트는 매 판정 시점의 후보 중 가장 가까운 하나를 자유 선택한다.
- 놓친 이벤트는 별도 miss로 감점하지 않는다.
- 다만 연주하지 않으면 가점이 없으므로 최종 점수는 낮아진다.
- 이 방식은 “정식 경쟁 점수”가 아니라 “연습 보조 점수”라고 명시한다.

이러면 구현도 가볍고, 현재 active track 구조와도 충돌이 적습니다.

> 예. 그 외에 명세를 보강하고나 고칠 점이 있다면 말씀해주세요.

보강하면 좋은 지점은 몇 가지 있습니다.

첫째, 점수 모델을 “감점형”이 아니라 “가점형”으로 명확히 쓰는 것이 좋습니다.

```text
게임 모드는 모든 NoteEvent를 빠짐없이 처리했는지 검사하지 않는다.
현재 판정 시점에 연주된 pitch가 active track의 현재 후보 NoteEvent 중 하나와 충분히 가까우면 점수를 더한다.
연주하지 않았거나 pitch가 불안정한 구간은 별도 감점하지 않고, 단지 점수가 추가되지 않는다.
```

이렇게 쓰면 coverage, judge group, missed event 관리가 필요 없고, 동시 이벤트 문제도 자연스럽게 풀립니다.

둘째, “현재 후보 NoteEvent” 조회 범위를 정해야 합니다. 정확히 `[startSeconds, endSeconds)` 안에 있는 이벤트만 볼지, 약간의 앞뒤 허용 시간을 둘지 정해야 합니다. 1차 명세로는 아래가 무난합니다.

```text
currentTime이 NoteEvent의 [startSeconds, endSeconds) 안에 있으면 후보로 본다.
너무 짧은 note는 최소 판정 가능 길이를 보정한다.
시작 타이밍 오차 판정은 1차 범위에서 제외한다.
```

셋째, 샘플 기반 점수의 상한 제어가 필요합니다. 80~120ms마다 점수를 계속 주면 긴 악보나 긴 음에서 점수가 커지는 것은 자연스럽지만, 기기 성능이나 frame rate에 따라 점수가 달라지면 안 됩니다. 따라서 pitch detection frame마다 점수화하지 말고 “고정 scoring interval”을 두는 것이 좋습니다.

```text
pitch detection은 가능한 주기로 실행하되,
점수 계산은 100ms 같은 고정 interval에서만 수행한다.
```

넷째, 난이도 가중치의 의미를 다시 정리하는 것이 좋습니다. 현재처럼 `basic=1.0`, `optional=1.2`, `extra=1.5`는 괜찮지만, 동시 이벤트 중 가장 가까운 target 하나만 선택한다면 “선택된 target의 track weight만 적용”한다고 명시해야 합니다.

다섯째, accuracy 표시 기준을 “전체 평균”이라고 부르면 오해가 생길 수 있습니다. 무음/불인식 구간을 제외한 평균인지, 0점 샘플까지 포함한 평균인지가 다릅니다. 추천은 둘 중 하나입니다.

```text
Pitch Accuracy: 유효하게 판정된 scoring sample의 평균 정확도
Score: 유효 판정 sample에서 누적된 가중 점수
```

무음 구간은 accuracy 평균에는 넣지 않고 score만 증가하지 않게 두는 편이 “연습 보조” 느낌에 맞습니다.

여섯째, 결과 화면의 표시 항목에서 `processed NoteEvent count`보다는 `scored samples` 또는 `hits`가 더 정확합니다. 지금 방식은 이벤트 단위가 아니라 샘플 단위 판정에 가까워지기 때문입니다.

마지막으로 문서의 충돌 지점은 정리하는 편이 좋습니다. `3.0`의 coverage 제안, `3.2`의 `middleScore * averageAccuracy`, 그리고 지금 논의한 자유 가점형 모델이 서로 다릅니다. 최신 결정으로는 다음처럼 고치면 됩니다.

```text
확정 결정:
- coverage 기반 감점/보정은 1차 범위에서 제외한다.
- judge group 사전 생성은 1차 범위에서 제외한다.
- 현재 score time의 active track 후보 중 사용자 pitch와 가장 가까운 NoteEvent 하나만 판정한다.
- 점수는 고정 scoring interval마다 가산한다.
- 무음, 불인식, 허용 오차 초과는 점수 없음으로 처리한다.
- 이 점수는 경쟁용 정밀 채점이 아니라 연습 보조용 누적 점수이다.
```

이 정도를 반영하면 명세가 훨씬 일관됩니다.

> 네. 반영하겠습니다.
> 다만 허용 오차 초과에 한해서 감점 처리합시다. 연주를 했는데 소리가 크게 다른 것이니까요.

좋습니다. 그러면 모델은 “기본 가점형 + 명백한 오답만 약한 감점”으로 정리하면 됩니다.

명세 반영 방향은 이렇게 두는 것이 합리적입니다.

```text
- 무음, 불인식, 신뢰도 부족:
  점수 변화 없음

- 유효 pitch가 있고 현재 후보 NoteEvent와 충분히 가까움:
  pitch accuracy와 track weight에 따라 가점

- 유효 pitch가 있으나 모든 후보 NoteEvent와 허용 오차를 초과함:
  오답 연주로 보고 감점
```

이때 감점은 너무 크지 않게 두는 편이 좋습니다. 마이크 오인식, 배음, 주변 소음 때문에 억울한 감점이 생길 수 있기 때문입니다.

추천 초안:

```text
Good:  +1.00 * trackWeight
Ok:    +0.80 * trackWeight
Hmm:   +0.20 * trackWeight
Nope:  -0.30 * nearestTrackWeight
무음/불인식: 0
```

또는 더 보수적으로:

```text
Nope: -0.20 * nearestTrackWeight
```

`Nope`의 기준은 현재처럼 `200 cent 이상`으로 두면 됩니다. pitch class 허용을 켠 상태라면 200 cent 이상은 같은 계이름 근처도 아니므로 “연주했지만 악보와 명확히 다름”으로 볼 수 있습니다.

문서에는 이렇게 쓰면 깔끔합니다.

```text
1차 게임 모드는 전체 coverage를 계산하지 않는다.
놓친 이벤트는 별도 miss로 감점하지 않는다.
다만 유효한 pitch 입력이 있었고, 현재 score time의 active track 후보 중 어떤 NoteEvent와도 허용 오차 안에 들어오지 않으면 오답 입력으로 보고 소폭 감점한다.
```

그리고 결과 표시명도 조금 조정하면 좋습니다.

- `Score`: 가점과 감점을 합산한 누적 점수
- `Pitch Accuracy`: `Good/Ok/Hmm`처럼 유효하게 맞은 샘플의 평균 정확도
- `Wrong Inputs`: `Nope` 샘플 수
- `Hits`: 점수가 양수로 들어간 샘플 수

이러면 “연주하지 않음”과 “틀리게 연주함”이 구분됩니다. 이 구분은 연습 피드백 측면에서도 유용합니다.

> Wring Inputs는 Miss로 바꾸겠습니다.
> 표시 문자도 전반적으로 Perfect / Ok / Bad / Miss로 바꾸도록 하죠.

네. 그 명칭이 더 직관적입니다. 그러면 판정 라벨과 표시 항목은 이렇게 정리하면 됩니다.

```text
Perfect: 허용 오차가 매우 작음, 최고 가점
Ok:      어느 정도 맞음, 중간 가점
Bad:     크게 벗어났지만 후보 note와 어느 정도 관련 있음, 낮은 가점
Miss:    유효 pitch 입력이 있으나 모든 후보와 허용 오차 초과, 감점
무음/불인식: 점수 변화 없음
```

점수 초안은 다음 정도가 균형이 좋습니다.

```text
Perfect: +1.00 * trackWeight
Ok:      +0.80 * trackWeight
Bad:     +0.20 * trackWeight
Miss:    -0.20 ~ -0.30 * nearestTrackWeight
No input: 0
```

cent 기준은 기존 구조를 이름만 바꿔서 시작해도 됩니다.

```text
pitchError < 50 cent:      Perfect
50 <= pitchError < 100:    Ok
100 <= pitchError < 200:   Bad
200 <= pitchError:         Miss
```

그리고 결과 화면 표시는 다음이 더 맞습니다.

- `Score`: 가점과 감점을 합산한 누적 점수
- `Pitch Accuracy`: `Perfect / Ok / Bad` 판정 샘플의 평균 정확도
- `Hits`: `Perfect / Ok / Bad` 샘플 수
- `Miss`: 유효 pitch가 있었지만 허용 오차를 초과한 샘플 수

명세의 기존 `Good / Ok / Hmm / Nope`, `Wrong Inputs`는 모두 위 이름으로 교체하면 됩니다.

> Miss는 음수로 내려가지 말고 0로 해도 괜찮겠습니다. 이제 지금까지의 논의사항을 반영하여 명세 문서를 고쳐주세요.

짧게 말하면: **Pitchy 자체에 “잡음 제거/보정” 기능이 내장되어 있다고 보기는 어렵습니다.**

Pitchy가 제공하는 것은 주로:

- time-domain sample에서 pitch Hz 추정
- `clarity` 값 반환
- `clarity`는 0~1 범위이며, 낮은 값은 true pitch보다 noise에 가깝다는 신호로 볼 수 있음
- McLeod Pitch Method 기반의 실시간 pitch detection

공식 README도 `findPitch(input, sampleRate)`가 pitch와 `clarity`를 반환하고, 낮은 clarity는 noise 쪽에 가깝다는 식으로 설명합니다. 즉 “잡음을 자동 제거해서 깨끗한 pitch만 준다”가 아니라, **앱이 clarity threshold를 보고 판정에 쓸지 말지 걸러야 하는 구조**입니다. 출처: Pitchy README

따라서 `회귀 코드` 명세의 현재 방향, 즉 아래 필터는 필요합니다.

```text
- clarity >= threshold
- rms >= threshold
- frequencyHz가 유효 범위 안
- 너무 튀는 pitch frame은 smoothing 또는 무시
- 무음/불안정 입력은 Miss가 아니라 no input 처리
```

추가로 브라우저 `getUserMedia`에서 이런 constraints를 줄 수는 있습니다.

audio: {

```ts
function calculateRms(samples: Float32Array): number {
  let sumSquares = 0;

  for (const sample of samples) {
    sumSquares += sample * sample;
  }

  return Math.sqrt(sumSquares / samples.length);
}

function isValidPitchFrame(input: {
  frequencyHz: number;
  clarity: number;
  rms: number;
  minFrequencyHz: number;
  maxFrequencyHz: number;
}): boolean {
  return (
    Number.isFinite(input.frequencyHz) &&
    input.frequencyHz > 0 &&
    input.clarity >= 0.75 &&
    input.rms >= 0.01 &&
    input.frequencyHz >= input.minFrequencyHz &&
    input.frequencyHz <= input.maxFrequencyHz
  );
}
```

선택적으로 pitch 튐을 줄이려면 처음부터 복잡한 보정보다는 최근 3개 정도의 median filter가 적당합니다.

```ts
function median3(a: number, b: number, c: number): number {
  return [a, b, c].sort((x, y) => x - y)[1];
}
```

> - 배음 때문인지 고음부에서 옥타브가 하나 위로 튀네요. 보정이 필요해 보이는데 방법이 있을까요?

네. 고음부에서 한 옥타브 위로 튀는 것은 pitch detection에서 흔한 octave error입니다. 배음이 기본음보다 더 강하게 잡히면 detector가 2배 주파수를 “더 명확한 pitch”로 선택할 수 있습니다.

가장 현실적인 보정 순서는 이렇습니다.

1. **후처리 octave fold**
   감지된 pitch를 그대로 쓰지 않고, `midi`, `midi - 12`, `midi + 12` 후보 중 현재 layout 범위와 가장 잘 맞는 값을 선택합니다.

2. **이전 안정 pitch 기준 smoothing**
   직전 voiced frame이 안정적이면 새 pitch가 정확히 한 옥타브 위/아래로 튀는 경우 보정합니다.

   ```text
   if abs(currentMidi - previousMidi - 12) < smallThreshold:
     currentMidi -= 12
   ```

3. **나중에 scoring 대상이 생기면 target-aware correction**
   현재 재생 중인 `NoteEvent` 후보가 있으면, 감지 pitch의 옥타브 후보들 중 현재 target과 가장 가까운 값을 선택합니다. 이게 가장 정확합니다.

4. **필요하면 BiquadFilter 추가**
   `MediaStreamSource -> BiquadFilter -> AnalyserNode`로 연결해서 너무 강한 고역 배음을 약하게 만들 수 있습니다. 다만 cutoff를 잘못 잡으면 실제 고음도 죽어서, 1차는 후처리 보정이 더 안전합니다.

추천은 우선 **후처리 octave correction**입니다. 현재 dot 표시 단계에서는 scoring target이 아직 없으니:

```text
detectedMidi 후보:
- rawMidi - 24
- rawMidi - 12
- rawMidi
- rawMidi + 12
- rawMidi + 24

그중 현재 layout note row 범위 안에 있고,
이전 voiced pitch와 가장 가까운 후보를 선택한다.
이전 pitch가 없으면 layout 중앙 또는 layout 범위에 가장 가까운 후보를 선택한다.
```

이 방식이면 고음에서 `C5`가 `C6`로 튀는 현상은 꽤 줄일 수 있습니다. 이후 점수 판정 단계에서는 “현재 active target과 가장 가까운 pitch class 후보 선택”으로 더 안정화할 수 있습니다.
