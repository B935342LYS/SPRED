현재 이해한 작업 상태는 이렇습니다.

`회귀 코드`는 이제 `regression-code/`가 주 구현 경로이고, `이세계 코드`는 동작 참고용 레거시로만 보는 방향입니다. 현재 구현은 파서, 분석기, 캔버스 렌더러, 편집 UI, 레이아웃 편집, 트랙 레이어, Web Audio 재생, YouTube 동기화, 긴 악보 viewport bounded rendering, partial rebuild까지 상당 부분 복원 및 개선된 상태입니다. 기준 문맥은 [docs/HARNESS.md](./docs/HARNESS.md)와 [AGENTS.md](./AGENTS.md)입니다.

현재 편집 쪽은 Default `AUTO♯/AUTO♭`, `CUSTOM`, comment, eraser, long/vibrato hold, gliss, tremolo, `absolutePitch`, `microPitch` 입력이 연결되어 있습니다. 저장 rawText는 기존 문법인 `@p(n)` absolute pitch와 `@m(c)` micro pitch를 사용하고, `@p(0)` / `@m(0)`처럼 효과 없는 modifier는 생략하는 정책입니다. 관련 구현은 [edit_default.ts](./regression-code/src/app/edit/edit_default.ts)와 [pitch_label.ts](./regression-code/src/app/pitch_label.ts)에 있습니다.

하모닉스 AbsolutePitch AUTO 검토의 현재 결론은 `@p(h)` 같은 새 저장 토큰을 만들지 않는 것입니다. [1.36-harmonics-auto-pitch-plan.md](./docs/implementation-memo/1.36-harmonics-auto-pitch-plan.md)에 따르면 실험 브랜치의 `@p(h)`, 다중 배음 schedule, 마름모 렌더링 방식은 폐기했고, 다음 방향은 `AUTO◇`를 편의 입력 preset으로 두어 선택된 note row MIDI 기준으로 기존 `@p(n)` / `@m(c)`를 자동 생성하는 것입니다. 초기 기본값은 4th harmonic, 즉 `rowMidi + 24`가 유력합니다.

앞으로 예정된 작업은 크게 다음 흐름으로 보입니다.

1. `AUTO◇` 입력 기능 설계 확정 및 구현
   parser/analyzer/audio/renderer 계약은 넓히지 않고, edit UI/rawText 합성 경계에서만 처리하는 것이 현재 가장 안전합니다. 우선 4th harmonic만 지원하고, 필요 시 2nd/3rd/5th partial 선택을 나중에 확장하는 방향이 맞습니다.

2. `AUTO◇` 검증 추가
   row MIDI 기준 `@p(rowMidi + 24)` 생성, MIDI 범위 초과 차단, 3rd/5th 확장 시 cent 1자리 제한과 `@m` 범위 검증이 필요합니다. 기존 `test:edit` 계열에 케이스를 추가하는 것이 적절합니다.

3. 남은 안정화 작업
   JSON download/load 왕복 확인, active track UI 수동 검증, vibrato/trem/gliss/seek/pause/resume/layout reset 브라우저 검증, YouTube mode 실사용 검증이 남아 있습니다.

4. 재생/성능 후속
   Loop UI를 실제 playback scheduler/controller에 연결하고 YouTube follower playback과 충돌 여부를 확인해야 합니다. 긴 악보 성능은 viewport bounded rendering이 이미 들어갔으므로, 저사양/9000+ column 테스트 후 필요할 때 tile rendering, scroll redraw throttling, runtime index partial update 같은 추가 최적화를 검토하는 단계입니다.

5. 확장 후보
   dynamics automation 세부 조정, tuplet timing 정교화, sampled instrument 또는 Tone.js backend 검토, undo/pending edit grouping, `main.ts` orchestration 추가 분리, production build comment stripping/minify 정책 등이 후순위로 남아 있습니다.

요약하면, 다음 실질 작업은 “하모닉스 입력 편의 기능을 저장 문법 확장이 아니라 UI preset으로 구현할지 확정하고, `AUTO◇ -> 기존 @p/@m rawText` 변환을 edit 경계에 추가하는 것”입니다.

> 그렇습니다. 자동으로 한 옥타브 위의 AbsolutePitch를 입력해주는 AUTO◇를 추가하는 것이 좋겠습니다.
> 다만 실제 연주에서 2/3/5 배음을 하모닉스로 연주하는 빈도가 어느 정도인지에 따라 나머지의 구현 필요성이 달라지니 한번 음악 관련 자료를 찾아 가늠해보는 것이 좋겠습니다.

자료 기준으로 보면, `AUTO◇ = 한 옥타브 위(+12)`는 “하모닉스 입력 편의”로는 조금 애매합니다.

조사 결과의 핵심은 이렇습니다.

- 현악기 artificial harmonic에서 가장 흔하고 표준적인 것은 stopped note 위 완전4도 위치를 가볍게 짚는 방식이고, 실제 소리는 stopped note보다 **두 옥타브 위(+24)** 입니다. Tim Davies도 touch 4를 가장 쉽고 흔한 방식으로 설명하고, 손 모양과 계산이 단순하다는 점을 이유로 듭니다. https://www.timusic.net/debreved/harmonics/
- 첼로 자료에서도 “가장 흔한 artificial harmonic interval은 완전4도”이며, 소리는 lower finger보다 **두 옥타브 위**라고 설명합니다. https://cellofun.eu/home/cello-blog-encyclopaedia/mechanics-technique/left-hand-homepage-a-musician-is-an-athlete-of-the-small-muscles/miscellaneous-left-hand/harmonics/artificial-harmonics/
- ConductIT의 orchestration 자료는 artificial harmonics 후보를 minor third, major third, fourth, fifth touch로 정리하면서, major/minor third 계열은 “limited application”이라고 따로 표시합니다. 완전4도 touch는 +24, 완전5도 touch는 +19입니다. https://conductit.eu/study-room/rehearsal/rehearsal-4/4-6-string-harmonics/
- 완전5도 touch, 즉 3rd partial 계열은 실제로 쓰이긴 하지만 기본값이라기보다는 특정 진행에서 손 이동을 줄이거나 더 잘 울리게 하기 위한 대체 선택에 가깝습니다. Cello Fun은 fifth harmonic을 유용하다고 보면서도 표준 perfect fourth와 구분합니다.
- major third touch, 즉 5th partial 계열은 가능하지만 안정성이 낮고 occasional/advanced 성격입니다. Cello Fun도 “가능하지만 실제로는 꽤 어렵고 reliably speak 하기 어렵다”고 설명합니다.
- 자연 하모닉스에서는 +12도 의미가 있지만, artificial harmonic 입력 preset으로는 가장 흔한 선택이 아닙니다. artificial harmonic에서 +12를 일반 preset으로 두는 것은 실제 현악 연주 관습과는 덜 맞습니다.

따라서 구현 판단은 이렇게 잡는 게 좋아 보입니다.

1. `AUTO◇` 기본값은 `rowMidi + 24`, 즉 4th harmonic / touch-four / 두 옥타브 위로 두는 편이 음악 관습에 가장 맞습니다.
2. `+12` 한 옥타브 위가 정말 필요한 경우는 `AUTO◇`보다 `AUTO+8ve` 같은 별도 편의 preset으로 보는 게 명확합니다.
3. 2/3/5 배음 확장은 지금 당장 전부 구현할 필요는 낮습니다. 우선순위는 `4th partial(+24)` 단독, 그 다음 선택 확장이 필요하면 `3rd partial(+19.0 근사, @p + @m)` 정도입니다.
4. `5th partial(+27.9 근사)`는 UI 복잡도를 늘릴 만큼 사용 빈도가 높아 보이지 않으므로 후순위가 적절합니다.

정리하면, “하모닉스용 AUTO◇”라면 한 옥타브 위가 아니라 **두 옥타브 위 AbsolutePitch 자동 입력**으로 구현하는 쪽을 권합니다. 저장은 기존 방향대로 새 토큰 없이 `@p(rowMidi + 24)`만 생성하면 됩니다.

필요한 함수는 많지 않습니다. 기존 구조를 유지하면 핵심은 3개 정도입니다.

1. `isAutoHarmonicAbsolutePitch(value: string): boolean`
   - 위치 후보: [pitch_label.ts](./regression-code/src/app/pitch_label.ts)
   - `absolutePitchSelect.value`가 특수 옵션인지 판별합니다.
   - 예: value를 `"AUTO_HARMONIC_2OCT"` 또는 `"autoHarmonic2Oct"`로 둡니다.

2. `resolveAutoHarmonicAbsolutePitch(rowMidi: number): string`
   - 위치 후보: [pitch_label.ts](./regression-code/src/app/pitch_label.ts)
   - 선택된 note row MIDI에서 2옥타브 위 MIDI를 계산합니다.
   - 기본 동작: `rowMidi + 24`
   - `0..127` 범위를 넘으면 빈 문자열 또는 blocked 상태로 처리해야 합니다.

```ts
export function resolveAutoHarmonicAbsolutePitch(rowMidi: number): string {
  const midi = rowMidi + 24;
  return midi >= 0 && midi <= 127 ? String(midi) : "";
}
```

3. `resolvePitchModifierInput(state, input, rowId)`
   - 현재의 `resolveAutoDefaultText()`를 확장하거나, 새 함수로 분리하는 쪽이 좋습니다.
   - 역할은 “row 문맥이 필요한 자동 입력”을 한 번에 해결하는 것입니다.
   - `AUTO♯/AUTO♭`의 `autoText` 계산도 하고, `AUTO◇`가 선택되어 있으면 `absolutePitch`를 실제 MIDI 문자열로 치환합니다.

현재 `validateDefaultNoteEditInput()`은 `absolutePitch`가 숫자 문자열인지 검사하므로, `AUTO◇` 같은 특수 값은 rawText 합성 전에 반드시 실제 `"84"` 같은 숫자로 바뀌어야 합니다. 그래서 `composeDefaultNoteRawText()` 안에 특수 분기를 넣기보다, 적용 직전 입력 상태를 resolve하는 함수가 더 깔끔합니다.

추가로 수정될 부분은 다음입니다.

- [index.html](./regression-code/index.html): `absolute-pitch-select`에 `AUTO◇ +2oct` 옵션 추가
- [pitch_label.ts](./regression-code/src/app/pitch_label.ts): 자동 하모닉스 계산 함수 추가

> 좋네요. 모호한 부분이 남아있으면 질문을 통해 정리하겠습니다. 명확해지면 구현에 돌입합시다.

확정이 필요한 부분은 3가지입니다.

1. `AUTO◇` 위치
   저는 `Pitch > absolutePitch` 드롭다운의 첫 특수 옵션으로 넣는 방식을 권합니다. 즉 `(none)`, `AUTO◇ +2oct`, `B9`, `A#9`... 순서입니다. 괜찮을까요?

2. 범위 초과 처리
   낮은/중간 row는 `rowMidi + 24`가 정상 저장되지만, 높은 row에서는 127을 넘을 수 있습니다. 이 경우는 입력을 막고 상태 메시지로 `AUTO◇ pitch exceeds MIDI range.` 같은 blocked message를 띄우는 방식이 맞을까요?

3. 표시 defaultText 처리
   `AUTO◇`는 absolutePitch modifier만 자동 입력하고, `AUTO♯/AUTO♭`의 표시 defaultText 생성 규칙은 그대로 두는 게 좋겠습니다. 예를 들어 C4 row에서 `AUTO♯ + AUTO◇`면 rawText는 `C4@p(84)`가 됩니다. 하모닉스 기호 `◇`를 note text에 자동 삽입하지는 않습니다. 이 방향으로 갈까요?

이 3개가 맞으면 구현은 작게 들어갈 수 있습니다. `pitch_label.ts`에 자동 pitch resolve 함수를 추가하고, 기존 `resolveAutoDefaultText()` 흐름을 `AUTO◇`까지 처리하도록 확장한 뒤 `test:edit` 케이스를 추가하겠습니다.

> 1. 좋습니다.
> 2.  음악적으로 실용적이지 못한 수치라면 입력을 막는 것이 옳겠습니다.
> 3. 하모닉스 기호를 삽입하게 하기는 어렵겠죠? rawText 단계에서 구분이 안되니까요. 괜히 구별을 위해 비용을 늘이기보다는 그대로 두는 것이 좋겠습니다.

기억을 복원해보면, IndexedDB 대신 `localStorage`를 택한 이유는 “저장 대상과 UX가 단순했기 때문”입니다.

6-19 버전의 저장 구조는 이렇게 나뉩니다.

- JSON Download/Load:
  - 파일로 내보내거나 사용자가 고른 JSON 파일을 다시 읽는 기능

  - `Blob`, object URL, 임시 `<a>` 클릭으로 다운로드
  - `File.text()`로 파일 읽기

- Local Save/Load:
  - 같은 브라우저 안에서 빠르게 임시 저장/복구하는 기능
  - [score_local_storage.ts](./regression-code-2026-06-19/regression-code/src/infra/score_local_storage.ts)
  - `ScoreFile` JSON 문자열 하나를 고정 key에 저장

  - [app_file_binding.ts](./regression-code-2026-06-19/regression-code/src/app/app_file_binding.ts)
  - 저장 전 `touchScoreTimestampsForSave()`로 timestamp 갱신
  - load는 결국 기존 score load pipeline 재사용

설명용으로는 이렇게 말하면 됩니다.

주의점도 같이 말할 수 있습니다.

- `localStorage`는 브라우저/도메인 단위라 다른 기기나 다른 브라우저로 이동되지 않습니다.
- 용량 제한이 작고 문자열 저장 중심입니다.

순서는 이렇게 잡았습니다.

1. `core/score`: JSON 로드, 저장 포맷 검증, RuntimeDocument/ScoreIndexes 생성

> 제가 직접 다이어그램을 그려보는 것도 좋을 것 같습니다.
> 다만 함수에 특화된 다이어그램 양식이 있는지 모르겠네요. 제가 알고 있는건 UML의 시퀀스 다이어그램과 커뮤니케이션 다이어그램인데, 객체가 엄청나게 많은 현 구조에서는 둘 다 너무 방대해집니다.

직접 그려보는 건 좋습니다. 다만 지금 구조에는 전통적인 UML 시퀀스 다이어그램을 그대로 쓰면 너무 커집니다. 함수와 모듈 흐름을 설명하려면 더 작은 다이어그램 양식이 맞습니다.

추천은 4가지입니다.

1. **Pipeline Diagram**
가장 적합합니다. “데이터가 어떤 단계로 변환되는가”를 보여줍니다.

```text
Score JSON
-> loadScoreJson()
-> validateScoreFile()
-> createRuntimeDocument()
-> buildParsedDocument()
-> analyzeDocument()
-> buildCanvasRenderInput()
-> renderCanvasScore()
```

면담에서 전체 구조 설명용으로 좋습니다.

2. **Layered Architecture Diagram**
객체가 아니라 계층 책임을 보여줍니다.

```text
UI / app
  -> edit, file, playback binding

renderer
  -> canvas DTO, draw layer

audio
  -> schedule, controller, backend

core/analyze
  -> note/timing/dynamics events

core/parse
  -> rawText parser

core/score
  -> JSON load, validation, indexes
```

“왜 모듈을 이렇게 나눴는가” 질문에 답하기 좋습니다.

3. **Function Call Tree**
특정 진입 함수 하나를 중심으로 그립니다. 예를 들어 `loadRuntimeDocument()`:

```text
loadRuntimeDocument(jsonText)
  -> loadScoreFile(jsonText)
    -> loadScoreJson(jsonText)
      -> JSON.parse()
    -> validateScoreFile(value)
      -> validateRequiredTopLevelFields()
      -> validateBasicShapes()
      -> validateRows()
      -> validateGlobalCells()
      -> validateTrackCells()
      -> validateGlobalStartCells()
  -> createRuntimeDocument(score)
    -> buildScoreIndexes(score)
```

함수 설명에는 이게 제일 실용적입니다.

4. **Data Shape Transformation Diagram**
각 단계에서 타입이 어떻게 바뀌는지 보여줍니다.

```text
string
  -> unknown
  -> ScoreFile
  -> RuntimeDocument
  -> ParsedScoreDocument
  -> AnalysisResult
  -> CanvasAnalyzedRenderInput
```

TypeScript 프로젝트 설명에 특히 좋습니다. 교수님이 “이 함수가 왜 필요한가?”라고 물으면 “타입/데이터 신뢰도가 이 단계에서 바뀝니다”라고 설명할 수 있습니다.

제 추천 조합은 이렇습니다.

- 전체 발표/면담 첫 장: **Layered Architecture Diagram**
- 실제 코드 설명: **Pipeline Diagram**
- 특정 함수 질문 대비: **Function Call Tree**
- 타입 안정성 설명: **Data Shape Transformation Diagram**

UML을 꼭 쓰고 싶다면 시퀀스 다이어그램보다 **활동 다이어그램(Activity Diagram)** 쪽이 낫습니다. 지금 코드는 객체 간 메시지보다 “입력 → 검증 → 변환 → 출력” 흐름이 핵심이라서 활동 다이어그램이 더 작고 읽기 쉽습니다.
