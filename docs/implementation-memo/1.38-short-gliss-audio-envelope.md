# 1.38 Short Gliss Audio Envelope

## Context

Tester report:

- two adjacent 8-cell `6`-tuplet groups
- 240 BPM
- gliss from the last slot of the first tuplet to the first slot of the next tuplet
- source cell examples:
  - `A4@g(a,S)@n(69)`
  - `A#4@g(a,E)@n(70)`
- symptom: crackling / clipped audio artifact

## Cause

For an 8-cell `6`-tuplet at 240 BPM, one slot is very short. The gliss anchor policy places a long tuplet-slot anchor at `slotStart + 0.5 tick`, so the start anchor note can be clipped to about `0.03125s`.

The previous oscillator backend scheduled attack and release independently:

- attack reached velocity at `start + min(attackSeconds, duration / 2)`
- release started at `end - releaseSeconds`

For very short clipped notes, `releaseStart` could be earlier than `attackEnd`. That created non-monotonic gain automation around the event edge and could produce a click/crackle.

## Change

`regression-code/src/audio/oscillator_backend.ts` now uses a shared edge envelope helper for note, standalone gliss, and gliss-chain oscillator events.

The helper clamps attack/release so automation times are monotonic:

- attack is limited to at most half of the event duration
- release is limited to the remaining duration after attack
- release start is never earlier than attack end

This preserves the existing short fade behavior for normal events while avoiding reversed gain automation on fast tuplet boundary gliss cases.
