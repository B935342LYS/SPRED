# 게임 모드 확장 계획

## 1. 문서 목적

- 본 문서는 `회귀 코드`의 후순위 확장 기능인 게임 모드의 목적, 범위, 기술 전제, 판정 기준 초안을 정리한다.
- 현재 단계에서 게임 모드는 구현 착수 대상이 아니라, 이후 설계와 구현에서 고려해야 할 확장 축으로 다룬다.
- 본 문서는 active 구현 기준 문서라기보다 확장 계획 문서이며, 실제 착수 시점에는 별도 세부 명세로 분리하거나 active 문서에 필요한 부분을 흡수할 수 있다.

## 2. 우선순위와 위치

- 게임 모드는 후순위 확장 기능 중에서도 최후순위 기능으로 둔다.
- 1차 구현 목표는 parser / analyzer / renderer / audio generator의 기본 구조를 안정화한 이후에만 검토한다.
- 따라서 현재 설계 단계에서 게임 모드는 “지금 당장 모든 세부 규칙을 고정해야 하는 기능”이 아니라, 코어 구조를 불필요하게 흔들지 않으면서 후속 확장 가능성을 확보해야 하는 대상으로 본다.

## 3. 목표 기능

- 사용자의 실제 연주를 브라우저 오디오 입력으로 받는다.
- 실시간 pitch detection 결과와 악보의 목표 음정/시간축을 비교한다.
- 각 판정 구간마다 리듬게임형 판정 메시지를 표시할 수 있도록 한다.
- 처음 이벤트부터 마지막 이벤트까지의 판정을 누적하여 최종 점수와 랭크를 계산한다.
- reset, jump, seek 같은 시간축 불연속 동작이 발생하면 게임 모드는 자동 중단된다.

## 4. 비목표와 후순위 항목

- 게임 모드는 초기 버전의 필수 기능이 아니다.
- 모든 브라우저, 모든 모바일 환경에서 완전 무버퍼 고정밀 판정을 보장하는 것을 목표로 두지 않는다.
- 1차 구현에서 gliss, vibrato, trem, tuplet의 정교한 연주 품질 평가를 고정하지 않는다.
- score JSON에 게임 모드용 실시간 입력 데이터, 중간 판정 결과, 점수 기록을 저장하는 것을 기본 전제로 두지 않는다.

## 5. 기술 전제

### 5.1. 배포 환경

- 게임 모드는 GitHub Pages 환경에서도 브라우저 오디오 입력 기반으로 구현 가능하다고 본다.
- 다만 안정성은 정적 호스팅 플랫폼 자체보다 사용자 기기 성능, 브라우저 구현, 입력 장치 품질, 주변 소음의 영향을 더 크게 받는다.
- 따라서 1차 구현은 데스크톱 브라우저 중심의 실용적 판정 모드를 목표로 한다.

### 5.2. 오디오 입력과 분석

- 브라우저의 오디오 입력과 실시간 pitch detection을 사용한다.
- 완전 무지연은 불가능하므로, 짧은 분석 윈도우를 전제로 한 실용적 판정 방식을 채택한다.
- 노이즈, 무성 구간, 불안정한 배음 구조를 고려해 최소 음량/신뢰도 필터 같은 보조 규칙이 필요할 수 있다.

### 5.3. 진실 원천

- 게임 모드의 판정 기준 진실 원천은 score JSON이 아니라 analyzer가 생성한 `AnalysisResult`이다.
- 즉 게임 모드는 parser 세부 구조나 rawText를 다시 직접 해석하지 않고, 이미 확정된 이벤트와 시간축을 소비하는 후속 모듈로 설계한다.

## 6. 현재 구조와의 연결점

### 6.1. 직접 사용하는 구조

게임 모드는 현재 설계상 아래 구조를 직접 소비하는 방향이 적절하다.

- `AnalysisResult`
- `AnalyzedTrackResult`
- `NoteEvent`
- `GlissEvent`
- `timingTimeline`

이 중 1차 구현에서 판정의 중심은 `NoteEvent`로 둔다.

이 판단의 이유는 다음과 같다.

- `NoteEvent`는 실제 발음 구간을 나타내는 기본 이벤트이다.
- `NoteEvent.sound`는 목표 MIDI와 cent 보정 정보를 가진다.
- `NoteEvent.time`과 `timingTimeline`을 결합하면 tick 기준 이벤트를 실제 시간축으로 대응시킬 수 있다.
- `GlissEvent`는 관계 이벤트이므로, 1차 구현에서는 정교한 점수 평가보다 후순위 시각/보조 판정 대상으로 두는 편이 현실적이다.

### 6.2. 런타임 데이터 원칙

- pitch detection 결과
- 실시간 판정 메시지
- 현재 목표 이벤트
- 누적 점수/랭크
- 중간 판정 캐시

위 항목들은 저장 구조가 아니라 게임 모드 전용 런타임 데이터로 본다.

## 7. 1차 구현 범위

### 7.1. 판정 대상

- 1차 구현은 `NoteEvent` 중심 판정을 우선한다.
- `GlissEvent`는 직접 점수화 대상이라기보다 후순위 확장 대상으로 둔다.
- `TupletGroupEvent`는 구조 이벤트이므로 직접 점수화보다는, 내부 슬롯에서 생성된 note/rest 이벤트를 통해 간접 처리하는 편이 적절하다.
- `RestEvent`, `MuteEvent`는 1차 구현에서 핵심 점수 산정 대상이 아니다.

### 7.2. 판정 요소

이벤트별 판정은 우선 아래 두 축을 종합한다.

- 이벤트 진입 타이밍
- 목표 음정과의 일치도

세부 점수 공식은 후속 단계에서 조정 가능하나, 1차 구현은 “실용적인 연습용 판정 모드”를 목표로 한다.

### 7.3. 시작과 종료

- 게임 모드는 명시적 시작 동작을 가진다.
- 시작 이후에는 처음 이벤트부터 마지막 이벤트까지의 판정을 누적한다.
- reset, jump, seek, 문서 재로드, 재생 위치 강제 변경 같은 시간축 불연속 동작이 발생하면 자동 중단한다.

## 8. 동시 시점 이벤트와 판정 단위

### 8.1. 문제 인식

복수 track layer를 동시에 활성화한 경우, 동일 시점에 여러 `NoteEvent`가 존재할 수 있다.
게임 모드에서 이들을 모두 개별 독립 과제로 간주할지, 아니면 같은 시점의 대안적 선택지로 묶을지는 별도 판단이 필요하다.

### 8.2. 현재 방향

- 동일 시점에 겹치는 이벤트가 있다면, 1차 구현에서는 그중 하나만 선택하여 연주해도 통과 가능한 방향을 우선 검토한다.
- 다만 이를 위해 지금 단계에서 `AnalysisResult`에 새로운 공식 구조를 추가하지는 않는다.
- 동시 시점 이벤트를 별도 그룹으로 묶는 판단은 필요 시 게임 모드 모듈 내부의 런타임 파생 로직으로 처리한다.

### 8.3. 현재 결론

- 동시 시점 이벤트 묶음은 score JSON이나 `AnalysisResult` 본체에 저장하지 않는다.
- `judge group`은 현재 시점에서 구현 가능성이 아직 모호하므로, core 문서에 선반영하지 않는다.
- 필요 시 게임 모드 모듈이 `AnalysisResult.trackResults`를 바탕으로 후속 단계에서 런타임 파생 구조로 생성할 수 있다.
- `judge group`의 정확한 묶음 규칙과 허용 오차는 후속 세부 명세에서 확정한다.

## 9. 점수 산정 원칙 초안

### 9.1. 기본 원칙

- 최종 점수는 단순히 “맞춘 이벤트만의 평균 정확도”로 계산하지 않는다.
- 일부 이벤트를 의도적으로 생략해 정확도만 높이는 꼼수를 막기 위해, 전체 판정 대상에 대한 커버리지 또는 miss 불이익이 함께 반영되어야 한다.

### 9.2. 현재 방향

- 각 판정 대상은 최소한 `hit / poor / miss` 수준의 구분이 가능해야 한다.
- 최종 점수는 전체 판정 대상 기준으로 산정하는 편이 바람직하다.
- 정확도와 별도로 놓친 이벤트 또는 시도하지 않은 이벤트에 대한 불이익을 반영해야 한다.
- 1차 구현에서는 복잡한 그룹 판정보다, 정확도와 참여율(coverage)을 함께 반영하는 방식으로 “쉬운 이벤트만 골라 정확도를 높이는 방식”을 억제하는 편이 적절하다.
- 즉 최종 점수는 단순 평균 정확도가 아니라, 정확도 항과 coverage 항의 결합으로 계산하는 방향을 우선 검토한다.

예시:

```text
accuracy = 판정이 성립한 이벤트들에서의 평균 품질
coverage = 실제 판정 시도에 들어간 이벤트 수 / 전체 판정 대상 이벤트 수
finalScore = accuracy * coverage
```

또는 놓친 이벤트의 불이익을 더 강하게 주고 싶다면 아래처럼 coverage 가중을 높일 수 있다.

```text
finalScore = accuracy * coverage^2
```

이 방식은 별도 `judge group` 구조를 지금 단계에서 core 문서에 추가하지 않고도,
어려운 이벤트를 의도적으로 생략해 정확도만 높이는 전략을 상당 부분 억제할 수 있다는 장점이 있다.

### 9.3. 아직 미정인 항목

- `perfect / good / bad / miss` 같은 세부 판정 단계 수
- 판정별 가중치
- 음정 오차 허용 범위 (cent 기준)
- 타이밍 오차 허용 범위 (ms 기준)
- sustain 구간 유지율 반영 여부
- accuracy와 coverage의 최종 결합 공식

## 10. 이벤트 식별자와 추적

### 10.1. 필요성

게임 모드는 아래 상황에서 “같은 이벤트를 안정적으로 다시 가리키는 기능”이 필요하다.

- 현재 목표 이벤트 추적
- 이미 판정 완료된 이벤트 추적
- 재생 하이라이트와 판정 상태 동기화
- 부분 갱신 이후 가능한 한 동일 이벤트를 안정적으로 재사용

### 10.2. 현재 방향

- `NoteEvent`, `GlissEvent`에는 런타임 추적용 `eventId`를 우선 둔다.
- 순번 기반 id는 부분 갱신 시 뒤쪽 이벤트 전체 재구성이 필요해지므로 채택하지 않는다.
- 1차 구현에서는 이벤트의 시작 기준 source를 중심으로 짧은 안정 id를 생성하는 편이 적절하다.

예시:

```ts
note:${trackId}:${startRowId}:${startCol}
note:${trackId}:${startRowId}:${startCol}:${slotIndex}
gliss:${trackId}:${glissId}:${startRowId}:${startCol}
```

- `TupletGroupEvent`는 기존 `groupId`를 계속 구조 식별자로 사용한다.
- `RestEvent`, `MuteEvent`의 별도 runtime id는 후속 필요가 생길 때 추가하는 편이 적절하다.

### 10.3. eventId에 포함하지 않는 것

- 전체 `sourceCells` 목록
- score JSON 기반 영구 식별자
- 단순 순번

이유는 다음과 같다.

- 전체 source 정보를 넣으면 길이와 재생성 비용이 과도하다.
- 게임 모드에 필요한 것은 영구 저장용 id가 아니라 같은 분석 결과 내부에서의 안정 추적용 id이다.
- 순번 기반 id는 부분 갱신과 증분 분석에 불리하다.

## 11. 지금 단계에서 추가로 준비할 구조

현재 문서 기준으로, 게임 모드 때문에 즉시 대규모 구조 변경이 필요하다고 보지는 않는다.
다만 아래 준비는 장기적으로 유의미하다.

1. `NoteEvent`와 `GlissEvent`의 runtime 식별자 유지
2. `AnalysisResult`를 진실 원천으로 유지하고, 게임 모드 데이터는 파생 캐시로 분리
3. tick 기반 이벤트를 실제 시간축으로 대응시키는 변환 유틸 또는 캐시 준비

반면 아래 항목은 현 단계에서 core 문서에 강제로 넣지 않는 편이 적절하다.

- 게임 점수 구조
- 실시간 judge 상태 필드
- `judge group`의 공식 저장 구조
- pitch detection 중간 프레임 데이터

## 12. 후속 과제

실제 구현 착수 전에 아래 사항을 별도 명세로 확정하는 것이 바람직하다.

1. pitch detection 후보 알고리즘과 브라우저 실행 방식
2. 오디오 입력 필터링 규칙
3. tick -> ms 변환과 판정 윈도우 계산 절차
4. accuracy / coverage 결합 공식과 랭크 기준
5. 동시 시점 `judge group` 형성 규칙이 실제로 필요한지 여부
6. 데스크톱 우선 지원 브라우저 범위
7. gliss / trem / vibrato / tuplet의 후속 평가 규칙

## 13. 결론

게임 모드는 `회귀 코드`의 코어 구조를 처음부터 뒤집어야만 가능한 기능은 아니다.
현재 설계상 `AnalysisResult`, `NoteEvent`, `timingTimeline`을 중심으로 후속 평가 모듈을 얹는 방향이 가능하다.

다만 이 기능은 입력 장치, 브라우저, 기기 성능의 영향을 크게 받으며, 완전 무버퍼 고정밀 판정보다 “실용적인 연습용 판정 모드”를 1차 목표로 두는 편이 현실적이다.
따라서 현재 단계에서는 코어 구조를 무겁게 만들기보다, 필요한 연결점만 정리해 두고 세부 게임 로직은 후속 문서에서 확정하는 전략이 가장 적절하다.
