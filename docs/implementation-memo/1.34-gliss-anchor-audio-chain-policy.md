# 1.34 Gliss Anchor Audio Chain Policy

## Context

단일 anchor가 쓰인 gliss에서 anchor note attack과 gliss ramp oscillator가 겹쳐 anchor 지점이 두 번 울리는 현상이 확인되었다.

backend의 note 최소 duration을 제거하는 방식은 효과가 작아 취소했다. note duration 보정은 유지하고, audio schedule 단계에서 gliss anchor note 자체의 독립 발음을 줄이는 정책으로 전환했다.

## Implemented Policy

### Start Anchor Timing

단일 source cell로 된 `S` start anchor에서 시작하는 gliss는 audio schedule에서 셀 중심 tick이 아니라 해당 셀의 `startTick`부터 재생한다.

이 변경은 audio schedule에만 적용한다. analyzer의 `GlissEvent.startAnchorTick`은 renderer 연결선 기준점으로도 쓰이므로 기존 중심 tick 기준을 유지한다.

`M` mid anchor는 앞 gliss의 종료점이면서 다음 gliss의 시작점이 될 수 있으므로 audio에서도 기존 중심 tick 기준을 유지한다. 종료 anchor도 기존처럼 anchor 중심 tick을 유지한다.

### Single-cell Anchor

단일 source cell 길이와 같은 gliss boundary anchor note는 audio schedule에서 note event로 내보내지 않는다.

이 경우 gliss event 자체는 유지된다. 따라서 pitch 이동은 gliss fallback 또는 chain event가 담당하고, anchor note의 별도 attack만 제거된다.

### Long-note Extension

gliss 시작 anchor 앞쪽에 hold로 이어진 long note가 있으면 해당 note를 audio schedule에서 제거하고, 같은 pitch를 유지하는 constant segment로 gliss chain 앞에 붙인다.

gliss 종료 anchor 뒤쪽에 hold로 이어진 long note가 있으면 해당 note를 audio schedule에서 제거하고, 같은 pitch를 유지하는 constant segment로 gliss chain 뒤에 붙인다.

단일 gliss segment라도 long-note extension이 붙으면 fallback gliss 대신 `AudioGlissChainScheduleEvent`로 승격한다.

## Effect Handling

chain에 흡수된 long note의 `vibrato`와 `tremolo` effect는 chain effect 배열로 보존한다.

`oscillator_backend.ts`는 gliss fallback과 gliss chain에도 vibrato detune LFO를 적용하도록 확장했다. 따라서 constant pitch segment와 pitch ramp segment 위에서도 vibrato/tremolo 효과가 유지된다.

## Scope

- analyzer의 `NoteEvent`와 `GlissEvent` 구조는 변경하지 않았다.
- renderer 표시 정책은 변경하지 않았다.
- 변경은 audio schedule 생성과 oscillator backend effect 적용에 한정했다.
