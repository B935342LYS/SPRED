# Practice Pitch Detector Backend 비교 및 채택 명세

이 문서는 Practice Mode의 마이크 입력에서 강한 배음을 기본주파수로 오인하는 문제를 해결하기 위한 detector backend 구조, 실험 절차, 채택 기준을 고정한다.

## 1. 문서 목적과 우선순위

- context가 요약되거나 리셋되어도 과거에 rollback한 임의 보정 방식을 다시 제품 경로에 넣지 않게 한다.
- 동일한 원시 오디오로 여러 F0 detector를 비교하고, 측정 결과가 확인된 backend만 제품 후보로 승격한다.
- detector 교체와 Practice 판정 규칙 변경을 분리한다.
- 라이선스 검토 전의 로컬 실험과 실제 배포 채택을 구분한다.

이 문서는 `docs/3.2-karaoke-game-mode-spec.md`의 마이크 입력과 Pitch Detection 절을 구체화한다.
Practice pitch detector backend의 구조, 비교, 채택에 관해서는 이 문서를 우선한다.
점수, timing, effect bonus, UI 상태 등 나머지 Practice Mode 정책은 `3.2`를 따른다.

## 2. 현재 문제와 확인된 사실

현재 제품 기준 backend는 MIT 라이선스의 `pitchy`이며 McLeod Pitch Method를 사용한다.

2026-07-11 실제 마이크에서 E3부터 C#6까지 천천히 올라가는 글리산도를 측정한 `pitchtest.json` 분석 결과는 다음과 같다.

- 저장된 최근 600 frame 중 accepted voiced frame은 516개였다.
- 연속 accepted frame 사이에서 5 semitone 이상 튄 구간은 13개였다.
- 주요 오류는 약 `2x`, `1/2x`, `3x`, `1/3x` 비율로 나타났다.
- 배음 쪽으로 튀는 frame은 주로 clarity `0.75~0.80` 부근이었지만, clarity만으로 모든 오류를 안전하게 분리할 수는 없었다.
- 오류는 A#3 한 음에 한정되지 않고 여러 음역에서 발생했다.

따라서 문제를 무작위 잡음이나 특정 target note의 예외로 취급하지 않는다.
현재 파형의 공통 주기 또는 배음 구조에서 기본주파수 F0를 선택하는 detector 문제로 취급한다.

## 3. 확정 원칙

### 3.1. 제품 pitch를 target에 맞춰 보정하지 않는다

- target note, 현재 score event, pitch class 정답 여부를 detector 입력으로 사용하지 않는다.
- detector가 낸 결과를 `f / 2`, `f / 3`, `f / 4`로 임의 이동시키는 제품 후처리를 추가하지 않는다.
- continuity lock, octave hysteresis, 최근 pitch 기반 강제 보정으로 정상적인 빠른 이동을 막지 않는다.
- pitch dot과 scoring은 선택된 backend가 반환한 F0를 그대로 사용한다.
- scoring의 pitch-class 비교 정책은 detector backend와 독립적으로 유지한다.

### 3.2. detector 자체의 F0 추정을 비교한다

- 시간영역 주기성, 확률적 후보 추적, harmonic model, 학습 기반 F0 추정처럼 공개된 알고리즘 또는 검증된 구현을 우선한다.
- 직접 만든 harmonic score는 검증 backend를 이해하기 위한 진단 실험에만 사용할 수 있다.
- 직접 만든 후보 선택기를 제품 backend로 채택하려면 별도의 명세와 독립적인 비교 자료가 필요하다.

### 3.3. 동일 입력 비교를 우선한다

- 서로 다른 실시간 연주를 backend별로 따로 녹음해 비교하지 않는다.
- 한 번 저장한 원시 mono PCM/WAV를 모든 backend에 재입력한다.
- 실시간 측정은 offline 동일 입력 비교를 통과한 후보에 한해 수행한다.
- `pitchtest.json`처럼 detector 결과만 저장된 자료는 기존 오류 분석에는 사용할 수 있지만 backend 재비교 자료로는 사용하지 않는다.

### 3.4. confidence를 공통 절댓값으로 간주하지 않는다

- `pitchy clarity`, pYIN voiced probability, CREPE confidence 등은 의미와 분포가 다르다.
- 모든 backend에 현재 `MIN_CLARITY = 0.75`를 일괄 적용하지 않는다.
- backend는 자신의 voiced/confident 판단과 원시 confidence를 함께 반환한다.
- RMS와 layout frequency range는 앱 공통 필터로 유지할 수 있다.
- backend별 threshold는 동일 데이터셋의 precision/recall 결과로 정하고 구현 메모에 기록한다.

## 4. 브랜치와 제품 경계

실험은 `experiment/practice-pitch-detection` 브랜치에서 진행한다.

- `main`의 안정 동작과 현재 `pitchy` 경로를 기준선으로 보존한다.
- 알고리즘마다 별도 branch를 만들지 않고 공통 backend 인터페이스로 교체한다.
- 실험용 backend 선택 query, console helper 또는 개발 전용 control을 사용할 수 있다.
- backend 선택 UI는 품질 비교 중에는 일반 사용자 UI에 노출하지 않는다.
- 최종 후보가 정해져도 라이선스와 배포 크기 검토를 통과하기 전에는 `main`에 병합하지 않는다.

## 5. 모듈 경계

기준 흐름은 다음과 같다.

```text
MediaStream / recorded PCM
  -> game_pitch_input.ts
  -> GamePitchDetectorBackend
  -> backend raw F0 + confidence + voiced decision
  -> app RMS / layout range filter
  -> GamePitchFrame
  -> pitch dot / judge / effect runtime
```

권장 파일 역할:

- `game_pitch_detector_backend.ts`: 공통 타입과 현재 `pitchy` 기준 backend
- `game_pitch_backend_*.ts`: 각 후보의 adapter, resampling, model lifecycle
- `game_pitch_input.ts`: 마이크 frame 수집과 공통 앱 필터
- `game_pitch_debug.ts`: backend ID, 원시 결과, accepted 결과 기록
- 별도 실험 runner: 저장된 동일 PCM을 여러 backend에 재입력하고 비교 JSON 생성

공통 계약은 최소한 다음 정보를 표현해야 한다.

```ts
type GamePitchDetection = {
  frequencyHz: number | null;
  confidence: number;
  isConfident: boolean;
  latencyMs?: number;
};

type GamePitchDetectorBackend = {
  readonly id: GamePitchDetectorBackendId;
  readonly frameSize: number;
  detect(samples: Float32Array, sampleRate: number):
    GamePitchDetection | Promise<GamePitchDetection>;
  dispose(): void;
};
```

구현 라이브러리의 추론 API가 비동기이면 공통 계약도 비동기를 허용한다.
느린 추론을 여러 개 겹쳐 실행하지 않고, 한 backend에는 동시에 하나의 detection만 진행한다.
backend가 요구하는 sample rate가 다르면 adapter가 명시적으로 resample하며 원본 sample rate와 처리 sample rate를 debug metadata에 기록한다.

### 5.1. WebAssembly의 역할

WebAssembly는 C/C++ 같은 소스 언어를 TypeScript로 번역하는 도구가 아니다.
브라우저가 실행할 수 있는 저수준 binary instruction format이자 compilation target이다.

일반적인 browser port 흐름은 다음과 같다.

```text
C / C++ source
  -> Emscripten / LLVM compilation
  -> algorithm.wasm
  + JavaScript glue / TypeScript declaration
  -> browser JavaScript adapter
  -> GamePitchDetectorBackend
```

- `.wasm`에는 컴파일된 계산 코드가 들어간다.
- JavaScript glue는 `.wasm`을 fetch/initialize하고 함수 import/export와 memory 접근을 연결한다.
- Embind, WebIDL Binder 또는 library 전용 wrapper가 C/C++ 타입을 JavaScript에서 호출 가능한 API로 노출할 수 있다.
- `Float32Array`를 WASM linear memory로 복사하거나 wrapper vector로 변환하는 비용이 생길 수 있다.
- native code가 file system, thread, OS audio API를 직접 사용했다면 browser 환경에 맞는 대체 경로가 필요하다.
- 따라서 “C/C++ 코드를 WASM으로 컴파일했다”와 “SPRED에서 바로 호출할 수 있다”는 같은 의미가 아니다. browser binding과 lifecycle adapter가 추가로 필요하다.

SPRED에서 WASM backend를 붙이는 개념 코드는 다음과 같다.

```ts
const wasmModule = await loadPitchWasmModule();

const backend: GamePitchDetectorBackend = {
  id: "pyin",
  frameSize: 2048,
  async detect(samples, sampleRate) {
    const result = wasmModule.detectPitch(samples, sampleRate);

    return {
      frequencyHz: result.frequencyHz,
      confidence: result.voicedProbability,
      isConfident: result.voicedProbability >= backendThreshold,
    };
  },
  dispose() {
    wasmModule.dispose();
  },
};
```

위 코드는 특정 library API를 복사한 구현이 아니라 adapter 책임을 설명하는 의사 코드이다.
실제 wrapper가 memory 할당과 해제를 요구하면 `detect()` 또는 runtime 내부에서 누수 없이 처리해야 한다.

### 5.2. ONNX의 역할

ONNX는 Python 코드를 JavaScript로 변환하는 실행 도구가 아니다.
학습된 model의 computation graph, tensor type, operator, weight를 표현하는 공통 intermediate representation이다.

일반적인 browser inference 흐름은 다음과 같다.

```text
PyTorch / TensorFlow / Keras model
  -> framework exporter / converter
  -> model.onnx
  -> ONNX Runtime Web
  -> WebAssembly CPU 또는 WebGPU execution provider
  -> TypeScript preprocessing / inference / postprocessing adapter
  -> GamePitchDetectorBackend
```

- ONNX model은 runtime과 분리된 model format이다.
- 실제 browser 실행은 `onnxruntime-web` 같은 inference runtime이 담당한다.
- CPU fallback은 WebAssembly execution provider를 사용할 수 있다.
- 지원 browser에서는 WebGPU execution provider를 검토할 수 있지만, 첫 비교는 호환성이 넓은 WASM CPU 경로를 기준으로 둔다.
- model이 사용하는 operator와 opset을 browser runtime이 지원해야 한다.
- exporter가 성공해도 원 framework와 ONNX 결과가 수치적으로 같은지 reference input으로 확인해야 한다.

SPRED에서 ONNX backend를 붙이는 개념 코드는 다음과 같다.

```ts
import * as ort from "onnxruntime-web";

const session = await ort.InferenceSession.create(modelUrl, {
  executionProviders: ["wasm"],
});

const inputSamples = resampleToModelRate(samples, sampleRate, 16_000);
const inputTensor = new ort.Tensor("float32", inputSamples, [1, inputSamples.length]);
const outputs = await session.run({ audio: inputTensor });
const result = decodeModelOutputs(outputs);
```

`audio` input name, tensor shape, normalization, frame size, output name과 F0 decoding은 model마다 다르므로 실제 model metadata와 원 구현을 확인해 adapter에서 정의한다.

### 5.3. 모델 변환에 포함되지 않는 것

Python 프로젝트를 ONNX로 export해도 다음 동작이 자동으로 model 안에 포함된다고 가정하지 않는다.

- audio file decoding
- stereo-to-mono 변환
- sample-rate 변환
- framing, overlap, windowing
- STFT 또는 spectrogram 생성 중 graph 밖에서 수행하던 단계
- amplitude normalization
- confidence threshold
- pitch-bin에서 Hz로 바꾸는 계산
- temporal smoothing, HMM, note segmentation

일부 전처리는 model graph에 포함될 수 있지만 이는 model별로 확인해야 한다.
원 Python 구현이 graph 밖에서 수행하던 전처리와 후처리는 TypeScript adapter가 같은 수식과 parameter로 재현하고 reference output으로 검증한다.

### 5.4. TensorFlow.js 대안

Keras 또는 TensorFlow model은 반드시 ONNX를 거칠 필요가 없다.
지원되는 model이면 TensorFlow.js converter를 사용해 `model.json`과 weight shard로 변환한 뒤 `tf.loadGraphModel()` 또는 `tf.loadLayersModel()`로 실행할 수 있다.

```text
Keras / TensorFlow model
  -> TensorFlow.js converter
  -> model.json + weight files
  -> TensorFlow.js browser runtime
```

custom layer, Lambda layer, custom op처럼 TensorFlow.js가 지원하지 않는 요소는 자동 변환되지 않을 수 있다.
CREPE는 공식 또는 재현 가능한 TensorFlow.js 변환본이 있으면 그 경로와 ONNX 경로를 모두 조사하고, model provenance와 실행 결과를 비교한 뒤 하나를 선택한다.

### 5.5. SPRED 적용 판단표

| 원본 형태 | 우선 적용 경로 | SPRED adapter 책임 |
|---|---|---|
| TypeScript/JavaScript library | npm/ES module 직접 호출 | frame과 결과 contract 변환 |
| C/C++ DSP library | 기존 WASM build와 JS binding 우선 | 초기화, memory, sample 변환, 해제 |
| Python wrapper + C/C++ core | core의 공식 WASM 가능성 확인 | Python wrapper가 하던 설정과 후처리 재현 |
| PyTorch model | 공식 ONNX export 또는 배포 ONNX 확인 | preprocessing, tensor, output decoding |
| Keras/TensorFlow model | TensorFlow.js 또는 ONNX 비교 | model load, tensor lifecycle, output decoding |
| Python-only algorithm | 검증된 browser port 탐색 후 독립 구현 검토 | 논문 계약, 테스트, 라이선스 분리 |

Python runtime 자체를 Pyodide 등으로 browser에 싣는 방식은 model/DSP 비교의 첫 선택으로 사용하지 않는다.
초기 다운로드, memory, startup time과 package 호환 부담이 Practice 실시간 입력 목적에 비해 크기 때문이다.
별도 Python server 호출도 GitHub Pages 단독 배포와 microphone audio 비전송 원칙에 맞지 않으므로 현재 범위에서 제외한다.

### 5.6. 배포 파일과 초기화

- `.wasm`, `.onnx`, `model.json`, weight shard는 source code와 별도의 정적 asset으로 취급한다.
- GitHub Pages의 상대 base path에서도 asset URL이 올바른지 production build로 확인한다.
- model과 WASM은 Practice 준비 상태에서 비동기로 load하고, 완료 전에는 pitch loop를 시작하지 않는다.
- load 실패, unsupported operator, WASM compile 실패는 Practice `error` 상태로 전달한다.
- 첫 inference는 warm-up 비용이 클 수 있으므로 초기화 시간과 steady-state 추론 시간을 나누어 측정한다.
- `dispose()`는 tensor, session, wrapper object와 WASM-side allocation 중 library가 요구하는 resource를 해제한다.
- model asset 크기와 cache 이후 재진입 시간을 모두 기록한다.

### 5.7. 보고서 서술 기준과 기술 출처

보고서에서는 다음 표현을 사용한다.

- WebAssembly: 다른 언어 소스의 번역본이라는 표현보다, browser에서 실행 가능한 compilation target과 binary format이라고 설명한다.
- JavaScript glue/binding: WebAssembly module의 초기화, memory와 함수 호출을 웹 앱에 연결하는 계층이라고 설명한다.
- ONNX: 학습 framework와 inference runtime 사이에서 model graph와 weight를 전달하는 공통 model format/IR이라고 설명한다.
- ONNX Runtime Web: ONNX model을 browser의 WASM/WebGPU backend에서 실제로 실행하는 runtime이라고 설명한다.
- TensorFlow.js converter/runtime: Keras/TensorFlow model을 browser용 format으로 변환하고 실행하는 별도 대안이라고 설명한다.

검토한 1차 자료:

- Emscripten documentation, `https://emscripten.org/docs/`, C/C++와 LLVM 기반 언어를 WebAssembly로 compile하고 JavaScript runtime support를 생성하는 공식 설명
- Emscripten C++/JavaScript connection, `https://emscripten.org/docs/porting/connecting_cpp_and_javascript/`, compiled code와 JavaScript binding 경계 설명
- ONNX overview, `https://onnx.ai/onnx/repo-docs/Overview.html`, computation graph를 표현하는 공통 IR 설명
- ONNX Runtime Web, `https://onnxruntime.ai/docs/get-started/with-javascript/web.html`, browser JavaScript API와 WASM/WebGPU execution provider 설명
- TensorFlow.js model conversion, `https://www.tensorflow.org/js/guide/conversion`, Keras/TensorFlow model을 browser format으로 변환하고 실행하는 공식 절차

현재 문서는 위 자료의 개념과 architecture를 참고했으며 source code를 복사하거나 변형하지 않았다.

## 6. 원시 오디오 캡처 명세

다음 backend를 연결하기 전에 동일 입력 비교용 캡처를 구현한다.

- debug mode에서만 활성화한다.
- 마이크의 detector 입력과 같은 mono 신호를 저장한다.
- WAV export 또는 sample rate가 포함된 Float32 PCM export를 제공한다.
- 녹음 시작과 종료는 사용자의 명시적 동작으로 수행한다.
- 기본 최대 길이는 60초로 두고, 초과 시 자동 종료한다.
- 원시 오디오는 메모리에만 두며 score JSON, LocalStorage, Supabase에 저장하지 않는다.
- export 전까지 외부 전송을 하지 않는다.
- `echoCancellation`, `noiseSuppression`, `autoGainControl` 요청값과 실제 track settings를 metadata에 기록한다.
- detector frame 결과에는 backend ID, frame/hop 크기, 처리 sample rate, confidence를 기록한다.

브라우저의 `MediaRecorder` 압축 결과만 저장하면 detector 입력 파형과 달라질 수 있으므로, 비교 기준 자료는 가능하면 detector에 전달되는 PCM에서 WAV를 만든다.

## 7. 비교할 backend와 순서

### 7.1. Pitchy / MPM

- 현재 기준선이다.
- 제품 동작과 비교 데이터 형식을 고정하는 데 사용한다.
- 제거하지 않고 모든 실험 결과에 함께 기록한다.

### 7.2. pYIN

- 여러 YIN 후보와 확률을 만들고 시간 경로를 추적하는 방식이다.
- 짧은 octave error 억제 가능성을 확인한다.
- HMM smoothing이 빠른 글리산도와 실제 octave jump를 늦추는지 반드시 측정한다.
- Essentia.js를 사용할 경우 AGPL 조건 때문에 로컬 실험 대상으로 한정하고 제품 채택 전 별도 검토한다.

### 7.3. CREPE

- 단일음 악기와 음성을 포함한 데이터로 학습된 time-domain neural F0 detector이다.
- 강한 배음과 잡음에서 octave error가 줄어드는지 확인한다.
- model variant별 초기 다운로드 크기, 초기화 시간, main-thread 점유, 추론 지연을 기록한다.
- 원 구현은 MIT이지만 브라우저 port, 변환 model, runtime 각각의 라이선스와 model 배포 권리를 따로 확인한다.

### 7.4. SwiftF0

- spectrogram과 경량 CNN을 사용하는 최근 F0 detector 후보이다.
- ONNX/WASM 브라우저 실행 가능성과 CREPE 대비 지연을 확인한다.
- MIT 후보이지만 공개 기간과 독립 검증이 상대적으로 짧으므로 자체 테스트 결과만으로 우선 채택하지 않는다.

### 7.5. aubio 계열과 harmonic comb

- `yin`, `yinfft`, `mcomb`, `fcomb`, `specacf`를 비교 후보로 둘 수 있다.
- 특히 `mcomb`의 spectral flattening, multi-comb filtering, peak histogramming을 비교한다.
- aubio GPL 구현은 로컬 실험과 알고리즘 비교에 사용할 수 있으나 제품 배포 전 별도 검토한다.
- aubio 코드를 참고해 유사 구현을 직접 재작성하는 방식으로 라이선스 의무를 우회하지 않는다.

### 7.6. 2026-07-11 browser 변환 경로 조사 결과

SwiftF0:

- 공식 저장소는 MIT이며 약 389KB의 `model.onnx`를 직접 포함한다.
- 공식 Python 경로는 mono audio 전체 또는 가변 길이 배열을 16kHz로 resample한 뒤 shape `[1, sampleCount]` Float32 tensor 하나를 model에 전달한다.
- model은 pitch Hz 배열과 confidence 배열을 직접 반환한다.
- 고정 parameter는 16kHz, STFT frame 1024, hop 256, symmetric padding 384, center offset 127.5 samples이다.
- 기본 confidence threshold는 `0.9`, model 물리 범위는 `46.875~2093.75Hz`이다.
- browser demo가 존재하며 ONNX/WASM client-side 실행을 명시한다.
- browser runtime은 MIT의 `onnxruntime-web` WASM backend를 우선 사용한다. 기존 `onnxjs`는 successor인 `onnxruntime-web`으로 대체한다.
- repository history와 공개 기간이 짧으므로 품질 우승을 전제하지 않고 첫 ML 비교 후보로만 채택한다.

Essentia.js / pYIN:

- 공식 Essentia C++ WASM과 JavaScript/TypeScript API가 있고 `PitchYinProbabilistic`이 노출된다.
- pYIN은 여러 후보와 HMM을 이용해 whole-signal pitch/voiced-probability contour를 반환하는 offline 비교에 적합하다.
- AGPL-3.0이므로 로컬 비교 adapter까지만 허용하고 배포 bundle에는 라이선스 결정 전 포함하지 않는다.

CREPE:

- 공식 model과 Python 구현은 MIT이고 16kHz time-domain waveform을 사용한다.
- 공식 repository가 직접 제공하는 browser npm adapter는 확인되지 않았다.
- TensorFlow.js 변환 사례는 있으나 wrapper와 변환 model의 provenance를 별도 검증해야 하므로 SwiftF0 뒤로 둔다.

aubio:

- 공식 C library는 Emscripten 지원과 `yin`, `yinfft`, `mcomb`, `fcomb`, `specacf`를 제공한다.
- browser용 최신 공식 npm adapter와 유지되는 고수준 TypeScript API는 확인이 더 필요하다.
- GPL-3.0이고 build/toolchain 부담이 있어 첫 adapter로 선택하지 않는다.

검토 출처:

- SwiftF0 repository/core/model, `https://github.com/lars76/swift-f0`, MIT, source/model 구조 및 parameter 확인
- SwiftF0 paper, `https://arxiv.org/abs/2508.18440`, algorithm/benchmark 비교
- ONNX Runtime Web, `https://onnxruntime.ai/docs/get-started/with-javascript/web.html`, browser WASM/WebGPU runtime
- `onnxruntime-web` npm, `https://www.npmjs.com/package/onnxruntime-web`, MIT와 TypeScript declaration 확인
- Essentia.js, `https://github.com/MTG/essentia.js`, AGPL-3.0과 WASM API 확인
- aubio, `https://github.com/aubio/aubio`, GPL-3.0과 Emscripten/pitch method 확인
- CREPE, `https://github.com/marl/crepe`, MIT와 16kHz model 경로 확인

### 7.7. 결정 대기: frame backend와 contour backend

현재 `GamePitchDetectorBackend`는 한 frame을 받아 `GamePitchDetection` 하나를 반환한다.
SwiftF0와 pYIN은 자체 hop/timestamp를 가진 전체 pitch contour 배열을 반환하므로 이 계약에 억지로 넣으면 여러 출력 중 하나를 임의 선택해야 한다.

권장 방향은 다음 두 계약을 분리하는 것이다.

```ts
type GamePitchDetectorBackend = {
  // 제품 실시간 경로: 한 입력 window에서 현재 pitch 하나
};

type GamePitchContourBackend = {
  analyze(audio: GamePitchComparisonAudio): Promise<{
    frames: Array<{
      centerTimeSeconds: number;
      frequencyHz: number | null;
      confidence: number;
      isConfident: boolean;
    }>;
  }>;
};
```

- offline comparison은 contour backend의 원래 16kHz/256-hop 결과를 보존한다.
- pitchy baseline은 현재 frame runner 결과를 같은 timestamp DTO로 변환한다.
- backend끼리 frame index를 강제로 같게 하지 않고 timestamp 기준으로 정렬한다.
- offline 결과가 개선된 후보에 한해 별도의 realtime streaming adapter를 설계한다.
- 사용자 확인에 따라 `GamePitchContourBackend`와 contour comparison 결과 계약을 별도로 추가했다.
- SwiftF0는 contour backend로만 먼저 연결하며 제품 실시간 `GamePitchDetectorBackend`에는 아직 연결하지 않는다.

## 8. 라이선스 정책

- MIT 등 permissive 후보도 copyright와 license notice를 확인한다.
- GPL/AGPL 후보도 로컬 실험 브랜치에서 품질 비교를 위해 설치하고 실행할 수 있다.
- GPL/AGPL 후보를 배포 bundle, GitHub Pages, 사용자 테스트 배포판에 포함하는 결정은 보류한다.
- 제품 채택 전 라이브러리, wrapper, model weight, runtime, 변환 산출물의 라이선스를 각각 확인한다.
- 직접 의존, source adaptation, algorithm-only reference, comparison-only review를 보고서에서 구분한다.
- 라이선스가 불명확한 후보는 품질이 좋아도 배포 후보로 승격하지 않는다.

이는 `3.2`의 기존 “GPL 계열은 비교 전용” 정책을 다음처럼 구체화한 것이다.
로컬 실험 통합은 허용하지만, 법적 검토 전 배포 통합은 허용하지 않는다.

## 9. 공통 테스트 코퍼스

최소 코퍼스는 같은 악기, 같은 마이크 위치와 입력 설정에서 다음 자료를 각각 3회 이상 포함한다.

1. A#3 지속음과 A#4 지속음
2. E3에서 C#6까지 느린 상승 글리산도
3. C#6에서 E3까지 느린 하강 글리산도
4. 같은 범위의 빠른 상승·하강 글리산도
5. 실제 1 octave 상행·하행 도약
6. 2, 3, 5, 7 semitone의 정상 도약
7. 낮은 음역, 중간 음역, 높은 음역의 지속음
8. vibrato가 있는 지속음
9. tremolo 또는 반복 attack
10. 무음, 방 소음, 현 접촉음, attack transient

가능하면 합성 신호도 별도 포함한다.

- fundamental보다 2차 또는 3차 배음이 강한 신호
- fundamental이 매우 약한 신호
- harmonic amplitude 비율을 단계적으로 바꾼 신호
- 실제 octave jump와 연속 gliss

합성 신호 통과만으로 backend를 채택하지 않는다.

## 10. 정답 자료와 측정 지표

실제 악기 녹음의 reference pitch는 연주 의도만으로 자동 확정하지 않는다.
가능하면 별도 고품질 offline 분석 결과와 수동 청취·파형 확인을 함께 사용하고, 불확실 구간은 평가에서 제외하거나 별도 표시한다.

필수 지표:

- raw pitch accuracy: reference에서 50 cent 이내인 voiced frame 비율
- octave error rate: reference와 약 1200, 1902, 2400 cent 등 정수배 배음 관계로 크게 어긋난 frame 비율
- gross pitch error rate: 50 cent를 초과한 voiced frame 비율
- voiced recall: 실제 연주 구간 중 voiced로 잡힌 비율
- false voiced rate: 무음과 비음정 noise를 voiced로 잡은 비율
- transition latency: 실제 도약 또는 방향 전환 뒤 새 pitch에 도달하는 시간
- gliss continuity: 진행 방향을 거슬러 크게 튄 frame 수
- jitter: 지속음에서 cent 분산과 중앙 절대 편차
- initialization time, model/download size, 평균·상위 추론 시간
- sustained realtime factor: 입력 1초 동안 들어오는 sample을 1초 안에 계속 처리할 수 있는지
- algorithmic look-ahead: HMM/smoothing이 결과 확정을 위해 요구하는 미래 audio 길이
- end-to-end response latency: audio capture부터 pitch dot과 판정 갱신까지의 실제 지연
- skipped-frame ratio: single-flight runner가 backend 처리 중 버린 입력 frame 비율
- 저사양 desktop-class 환경의 main-thread long task와 체감 UI 정지

평균값만 제시하지 않고 테스트 종류별 결과와 최악 구간을 함께 남긴다.

## 11. 채택 기준

후보 backend는 다음 조건을 모두 만족해야 제품 채택 대상으로 승격한다.

- 실제 마이크 코퍼스의 octave error rate가 `pitchy` 기준선보다 명확히 감소한다.
- A#3 문제만 고치는 것이 아니라 저·중·고 음역에서 개선이 재현된다.
- 정상 octave jump와 2~7 semitone 도약을 잘못 억제하지 않는다.
- 빠른 글리산도의 방향과 도달 pitch를 smoothing으로 훼손하지 않는다.
- voiced recall, false voiced rate, 지속음 jitter 중 어느 하나도 기준선보다 중대한 회귀를 만들지 않는다.
- Practice 화면이 멈추거나 scoring cadence가 깨질 정도의 main-thread 부하를 만들지 않는다.
- target desktop 환경에서 sustained realtime factor가 안정적으로 `1.0`을 초과하고, 일시적 부하에도 backlog가 계속 증가하지 않아야 한다.
- 평균 처리 시간만으로 채택하지 않고 algorithmic look-ahead, skipped-frame ratio, end-to-end response latency가 Practice 판정 허용 범위 안인지 확인한다.

### 11.1. 장치 성능 fallback

- microphone hardware는 음질과 신호 대 잡음비에 영향을 주지만 pitch 연산은 browser의 CPU/GPU/WASM runtime이 수행한다.
- 제품 후보가 정해지면 Practice 시작 전 짧은 warm-up/benchmark로 현재 장치가 해당 backend의 deadline을 지속적으로 만족하는지 확인한다.
- 성능이 부족하면 frame backlog를 쌓아 지연시키지 않고, 이미 구현된 single-flight 정책으로 오래된 frame을 버린다.
- skipped-frame ratio가 허용치를 넘으면 더 가벼운 검증된 backend로 fallback한다.
- 모든 고정밀 후보가 실시간 조건을 만족하지 못하면 `pitchy` raw backend를 유지하고 입력 장치·마이크 위치·noise suppression 설정 등 신호 품질 개선 안내를 제공한다.
- fallback 과정에서도 target pitch, 직전 pitch, octave lowering 등으로 detector 결과를 보정하지 않는다.
- backend 성능 차이에 따라 scoring 기준이 조용히 달라지지 않도록 활성 backend와 fallback 발생 여부를 debug/report metadata에 기록한다.
- desktop-class Chrome/Edge 계열에서 반복 실행이 가능하다.
- 라이선스와 배포 고지 계획이 확정된다.
- `npm run test:game`과 `npm run build`를 통과한다.

수치 threshold는 첫 공통 코퍼스 측정 후 구현 메모에서 잠정값을 정하고, 최소 두 종류의 실제 마이크 또는 입력 환경에서 재검증한 뒤 이 문서에 승격한다.
측정 전 임의 수치로 winner를 정하지 않는다.

## 12. 구현 단계

### Phase A. 비교 기반

1. 공통 backend 인터페이스와 `pitchy` adapter를 만든다.
2. debug JSON에 backend ID를 기록한다.
3. 원시 PCM/WAV 캡처와 metadata export를 만든다.
4. 저장된 동일 입력을 재생하는 offline comparison runner를 만든다.

### Phase B. 전통 DSP 비교

1. pYIN 또는 aubio 계열 중 브라우저 통합 가능한 후보 하나를 연결한다. 완료: Essentia.js pYIN offline adapter
2. `pitchy`와 동일 입력 결과를 비교한다.
3. smoothing latency와 정상 이동 회귀를 먼저 판정한다.

### Phase C. ML 비교

1. CREPE의 가장 작은 실용 model variant를 연결한다.
2. SwiftF0 browser runtime을 연결한다.
3. model load, resampling, 비동기 추론, UI 부하를 함께 측정한다.

### Phase D. 채택 판단

1. backend별 측정 보고서를 작성한다.
2. 품질, 성능, 라이선스 결과를 함께 검토한다.
3. winner가 없으면 `pitchy` 제품 경로를 유지하고 실패 결과를 기록한다.
4. winner가 있으면 일반 사용자 UI 없이 기본 backend 교체부터 검토한다.
5. 필요성이 확인된 경우에만 backend 선택 UI를 별도 명세화한다.

## 13. 금지되는 구현 지름길

- target pitch에 가장 가까운 octave로 입력을 이동한다.
- 직전 frame과 가까운 후보를 무조건 선택한다.
- 실제 octave jump를 일정 시간 차단한다.
- layout 최저음에 맞추기 위해 가능한 가장 낮은 harmonic candidate를 선택한다.
- 단일 FFT의 가장 낮은 energy peak를 fundamental로 간주한다.
- 합성 sine/partial 테스트만 통과한 보정을 실제 마이크 검증 없이 제품에 넣는다.
- 서로 다른 녹음을 사용한 backend 결과를 직접 우열 비교한다.
- backend마다 confidence 의미가 다른데 동일 threshold를 적용한다.
- GPL/AGPL 의존성을 법적 검토 없이 배포 bundle에 포함한다.

## 14. 실패와 rollback 정책

- 후보 backend는 adapter 단위로 제거 가능해야 한다.
- 실험 실패가 `GamePitchFrame`, judge, renderer 계약 변경으로 번지지 않게 한다.
- 후보 제거 후 `pitchy` 기준 backend만으로 build와 game test가 통과해야 한다.
- 실패한 접근도 알고리즘, 버전, 설정, 코퍼스, 결과, 실패 이유를 구현 메모에 남긴다.
- backend 비교 중에는 기존 raw pitch 제품 정책을 유지한다.

## 16. 기존 PCM 식별 가능성 분석과 다음 우선순위

2026-07-12 기존 PCM 분석 결과, 두 실사용 corpus의 약 2배 급상승 전이 16건 중 11건에서 낮은 F0 가설의 홀수차 배음 또는 낮은 주기 자기상관이 확인되었고, 짝수차만 남은 `octave-ambiguous` 판정은 없었다. 대표 오류 구간은 입력 정보가 완전히 사라진 경우 B보다 복원 단서가 남은 경우 A에 가깝다.

따라서 다음 단계는 aubio를 만능 해결책으로 전제하지 않되 독립 DSP 기준 구현으로 검토하고, CREPE 계열 최신 AI 모델도 동일 WAV offline 비교 후보로 격상한다. 두 경로 모두 제품 연결 전에 같은 급상승 전이, voiced coverage, gliss 반응, 처리 지연, 배포 크기와 라이선스 gate를 통과해야 한다.
