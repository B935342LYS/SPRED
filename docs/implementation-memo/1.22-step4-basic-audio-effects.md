# 1.22. 4단계 basic audio effect 구현 메모
> 본 구현 메모 문서 1.22에서는 기본 note event의 vibrato / tremolo audio 적용을 기록한다.
> 본 문서는 active specification을 대체하지 않는다.
> 함수별 세부 설명은 코드 파일의 `/** ... */` 주석을 중심으로 하고, 본 문서는 구현 판단과 참고 출처를 보충한다.

## 1. 개요

이번 구현에서는 analyzer가 만든 `NoteEvent.effects`를 audio backend에서 실제 Web Audio 효과로 해석하기 시작했다.
추가로 `GlissEvent`를 audio schedule 단계에서 fallback bridge 발음으로 변환하는 첫 경로를 연결했다.

대상은 기본 note event에 포함된 두 effect이다.

```text
NoteEvent.effects
-> AudioScheduleEffect[]
-> OscillatorBackend
-> Web Audio node / AudioParam automation
```

첫 구현 범위는 아래로 제한했다.

- vibrato: oscillator detune에 LFO를 연결한다.
- tremolo: oscillator와 note envelope 사이에 gain gate를 둔다.
- gliss: 별도 ramp oscillator를 짧은 fade in/out envelope와 함께 예약한다.
- dynamics, note 병합 voice span은 다음 단계에서 별도로 다룬다.

- `regression-code/src/audio/oscillator_backend.ts`
  - vibrato LFO와 tremolo gain gate를 생성하고 예약한다.

## 3. Vibrato 구현

Vibrato는 Tone.js의 `Vibrato`처럼 주파수 LFO와 depth 값을 분리한 모델을 참고했다.

현재 Web Audio 구현은 아래 구조를 사용한다.

```text
LFO OscillatorNode
-> depth GainNode
-> note oscillator.detune
```

초기 기본값은 legacy 구현과 맞춰 아래 값으로 두었다.

```text
rate: 5 Hz
depth: 35 cents
```

effect segment가 시작되면 depth gain을 35 cents로 만들고, 종료 시 0으로 되돌린다.

## 4. Tremolo 구현

Tremolo는 재타격 oscillator를 여러 개 만드는 방식이 아니라 gain gating 방식으로 구현했다.

현재 Web Audio node 흐름은 아래와 같다.

```text
note oscillator
-> tremolo GainNode
-> envelope GainNode
-> master GainNode
```

`division`은 tick당 tremolo 분할 수로 해석한다.
schedule 단계에서 `durationTicks`를 함께 내려보내기 때문에, backend는 긴 segment에서도 대략 `durationTicks * division` 개수의 pulse를 만들 수 있다.

각 pulse는 완전한 0 대신 작은 gain 값까지 내려간다.
클릭 노이즈를 줄이기 위해 on/off 경계에 짧은 ramp를 둔다.

## 5. 구현 판단

Tremolo를 gain gating으로 둔 이유는 다음과 같다.

- legacy 문서에서 “실제로 끄지 않고 볼륨을 조절해서 꺼진 것 같은 효과”를 허용한다.
- legacy gliss+trem 구현도 oscillator 하나에 gate gain을 거는 방식에 가깝다.
- oscillator를 pulse마다 새로 만들지 않으므로 scheduling 부담이 작다.
- 이후 gliss pitch ramp와 동시에 적용할 때 pitch는 oscillator frequency, tremolo는 gain으로 분리할 수 있다.

Vibrato와 tremolo가 같은 note에 이어서 나타나는 경우는 analyzer 단계에서 이미 `vib > trem` 규칙을 effect segment로 반영한다.
backend는 전달된 segment를 그대로 해석한다.

## 6. Gliss fallback 구현

`GlissEvent`는 active spec에서 기본적으로 "두 anchor 사이의 관계 이벤트"로 정의되어 있고, analyzer 의미상 독립 발음 이벤트는 아니다.
이번 구현은 이 의미를 바꾸지 않고, audio schedule 단계에서만 fallback 발음 이벤트를 파생한다.

현재 Web Audio 구현은 아래 구조를 사용한다.

```text
gliss fallback oscillator
-> tremolo GainNode (start anchor에 trem이 있을 때)
-> fallback gain envelope
-> master GainNode
```

frequency는 `startSound`에서 시작해 `endSound`까지 `linearRampToValueAtTime()`으로 이동한다.
gain은 `crossfadeSeconds` 동안 fade in하고, 종료 직전 같은 길이로 fade out한다.

gliss 시작 anchor에 tremolo가 붙어 있으면 해당 division을 gliss fallback 전체 구간에 적용한다.
renderer가 trem+gliss를 outgoing segment의 점선으로 표시하는 정책과 맞추기 위해, anchor cell 내부에서 실제 trem effect가 겹치는 절반 구간만이 아니라 gliss segment 전체에 gain gate를 건다.
이렇게 해야 `@g(id,S) @t(n)` 조합이 "글리산도 구간이 tremolo로 끊기는 효과"로 들린다.

이 방식은 완전한 voice-span 병합이 아니다.
기존 note event를 하나의 oscillator chain으로 합치거나 앞뒤 note gain을 ducking하지 않기 때문에, 긴 롱노트-글리산도-롱노트 전체를 하나의 음원으로 통합하지 않는다.
대신 `GlissEvent` segment 단위로만 예약하므로 부분 갱신과 스케줄 캐시 경계가 단순하다.

초기 기본값은 다음이다.

```text
crossfadeSeconds: 0.02
curve: linear
```

후속 단계에서 실제 청감이 과하게 겹치면 note envelope 쪽에 ducking automation을 추가하거나, 특정 attach 조합에 한해 voice-span 병합을 도입할 수 있다.
