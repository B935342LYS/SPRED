# 1.37. 면담용 코드 리뷰 준비 계획

## 1. 배경

교수님 면담 준비를 위해 최신 구현 작업은 잠시 중단한다.

최신 `regression-code/`에는 발표 이후 기능이 많이 추가되어 제한된 면담 시간 안에 전체를 설명하기 어렵다. 따라서 이번 면담에서는 복원한 과거 스냅샷을 기준으로 설명한다.

- 스냅샷 폴더: `regression-code-2026-06-19/`
- 실제 구현 경로: `regression-code-2026-06-19/regression-code/`

이 폴더는 면담 설명과 코드 읽기 연습을 위한 참조 스냅샷이다. 현재 작업용 구현 대상은 아니다.

## 2. 중단 전 최신 작업 기록

개발을 잠시 멈추기 직전, 작업용 `regression-code/`에는 다음 UX 보강을 추가했다.

1. 하모닉스 입력 보조 기능
   - `absolutePitch` UI에 `AUTO◇ +2oct` 옵션을 추가했다.
   - 다른 note row를 클릭해도 자동 preset 선택 상태를 유지한다.
   - rawText 합성 시 기존 absolute pitch token인 `@p(rowMidi + 24)`를 만든다.
   - `AUTO♯` / `AUTO♭` 모드에서는 defaultText도 2옥타브 위 pitch 이름으로 표시한다.
   - parser / analyzer / audio / renderer에 새 token은 추가하지 않았다.

2. 전역 행 시작 셀 삭제 방지
   - global row의 0번 column 값은 수정할 수 있지만 삭제할 수 없게 했다.
   - 보호 로직은 `edit_apply.ts`에 두어 click, drag, batch edit 경로가 같은 방어를 공유한다.
   - 필수 global start cell이 사라져 Local Save 후 다시 불러올 때 검증에 실패하는 문제를 막기 위한 조치이다.

배포 staging repository인 `regression-code-test-publish/`도 다음 commit까지 동기화해 push했다.

- `c3f487d Sync harmonics and global edit safeguards`

## 3. 면담 설명 범위

면담 설명은 최신 main 구현이 아니라 2026-06-19 스냅샷을 기준으로 한다.

스냅샷 검증:

- 명령어: `npm run build`
- 경로: `regression-code-2026-06-19/regression-code/`
- 결과: 통과

주요 설명 주제:

- renderer 계층
- audio 계층
- UI 계층
- JSON file save/load와 browser local save/load

필요할 때만 짧게 언급할 보조 문맥:

- renderer와 audio 앞에는 parser/analyzer 산출물이 있다.
- score JSON과 RuntimeDocument가 여러 계층의 공통 입력이 된다.
- active spec 문서는 `docs/`에 남아 있지만, 면담 코드 설명은 스냅샷 파일을 기준으로 진행한다.

## 6.1. File Save / Local Save 읽기 대상

먼저 볼 파일:

- `regression-code-2026-06-19/regression-code/src/app/app_file_binding.ts`
- `regression-code-2026-06-19/regression-code/src/infra/score_file_io.ts`
- `regression-code-2026-06-19/regression-code/src/infra/score_local_storage.ts`
- `regression-code-2026-06-19/regression-code/src/app/score_timestamp.ts`

설명 목표:

- JSON file save/load와 browser local quick save/load를 구분한다.
- JSON Download는 `Blob`, object URL, 임시 anchor click을 사용한다고 설명한다.
- JSON Load는 사용자가 선택한 `File`을 text로 읽은 뒤 일반 score load pipeline을 재사용한다고 설명한다.
- Local Save는 직렬화한 `ScoreFile` JSON 문자열을 하나의 고정 `localStorage` key에 저장한다고 설명한다.
- Local Load도 저장된 문자열을 꺼낸 뒤 일반 score load pipeline을 재사용한다고 설명한다.
- export/local save 직전 `touchScoreTimestampsForSave()`로 저장 timestamp를 갱신한다고 설명한다.

IndexedDB를 쓰지 않은 이유:

- 이 단계의 MVP는 같은 브라우저 안에서 빠르게 복구할 수 있는 단일 score JSON 문자열 저장만 필요했다.
- `localStorage`는 구조가 단순하고, 별도 schema나 transaction wrapper가 필요 없다.
- 개발 중 브라우저 devtools에서 값을 확인하기 쉽다.
- 당시 UX는 single-slot local save에 가까웠으므로 IndexedDB는 과한 선택이었다.

IndexedDB가 더 적절해지는 경우:

- 여러 악보를 목록으로 관리해야 하는 경우
- 큰 악보나 binary asset을 함께 저장해야 하는 경우
- 검색, indexing, versioned record가 필요한 경우
- 큰 데이터를 UI blocking 없이 비동기로 쓰고 읽어야 하는 경우

## 7. 준비 방법

각 계층마다 다음 질문에 답할 수 있게 준비한다.

1. 이 계층은 어떤 입력 데이터를 받는가?
2. 어떤 출력을 만드는가?
3. 진입점 모듈은 어디인가?
4. 어떤 함수가 순수 변환 로직이고, 어떤 함수가 browser API를 직접 만지는가?
5. 첫 구현에서 어떤 tradeoff를 선택했는가?
6. 무엇을 후속 과제로 남겼는가?

이번 리뷰는 새 기능 구현보다 설명 가능성을 높이는 데 우선순위를 둔다.

## 7.1. 바닥부터 올라가는 주석 보강 순서

설명용 스냅샷의 주석 보강은 프로그램 동작 순서와 데이터 의존성 순서에 맞춰 진행한다.

권장 순서:

1. `core/score`
   - JSON 문자열을 ScoreFile 후보로 읽는다.
   - 저장 포맷을 검증한다.
   - RuntimeDocument와 ScoreIndexes를 만든다.
2. `core/parse`
   - global cell과 note cell rawText를 구조화된 parser result로 바꾼다.
3. `core/analyze`
   - parser result를 note/timing/dynamics/gliss/tuplet event로 바꾼다.
4. `renderer`
   - score/analyzer 결과를 canvas DTO로 바꾸고 layer별로 그린다.
5. `audio`
   - analyzer event를 seconds 기반 schedule로 바꾸고 Web Audio에 예약한다.
6. `app`
   - DOM event, AppState, file/edit/playback/render orchestration을 연결한다.

1차 core 주석 보강 대상:

- `regression-code-2026-06-19/regression-code/src/core/score/json_load.ts`
- `regression-code-2026-06-19/regression-code/src/core/score/score_validate.ts`
- `regression-code-2026-06-19/regression-code/src/core/score/create_runtime_document.ts`

보강 관점:

- 외부 입력을 어디서 신뢰 가능한 값으로 바꾸는지
- 왜 shape 검사와 참조 무결성 검사를 나누는지
- 왜 검증된 ScoreFile에 대해서만 ScoreIndexes를 만드는지
- 후속 parser/analyzer/renderer/audio 계층을 어떤 오류로부터 보호하는지

## 8. 설명용 주석 정책

작업용 구현과 설명용 스냅샷은 의도적으로 분리한다.

- `regression-code/`는 계속 작업용 구현으로 유지한다.
- `regression-code-2026-06-19/`에는 교수님 코드 읽기용으로 더 촘촘한 주석을 추가할 수 있다.
- 설명용 주석은 active `regression-code/`에 자동 역반영하지 않는다.
- 설명용 스냅샷에서도 모든 표현식을 다시 말하는 주석보다, 동작 블록 앞 주석을 우선한다.
- `types.ts` 파일은 예외적으로 필드별 설명 주석을 적극적으로 추가한다. 데이터 구조를 먼저 이해하면 함수 흐름 설명이 쉬워지기 때문이다.
- type 필드 주석은 `score/types.ts`처럼 "무엇을 저장하는가", "어느 계층에서 쓰는가", "좌표/시간/표시 단위가 무엇인가"를 중심으로 작성한다.

첫 주석 보강 적용 파일:

- `regression-code-2026-06-19/regression-code/src/app/main.ts`
  - boot sequence, shared app session, state/render/playback reset flow
- `regression-code-2026-06-19/regression-code/src/renderer/canvas_score_renderer.ts`
  - renderer DTO extraction, canvas layer resize, draw order by layer
- `regression-code-2026-06-19/regression-code/src/audio/playback_controller.ts`

- `regression-code-2026-06-19/regression-code/src/core/score/json_load.ts`
  - JSON 입력 경계, unknown 값 처리, ScoreFile 검증 위임
- `regression-code-2026-06-19/regression-code/src/core/score/score_validate.ts`
  - shape 검사와 참조 무결성 검사 순서, 후속 계층 보호 목적
- `regression-code-2026-06-19/regression-code/src/core/score/create_runtime_document.ts`
  - 검증된 ScoreFile과 ScoreIndexes를 묶는 이유
- `regression-code-2026-06-19/regression-code/src/renderer/canvas_types.ts`
  - renderer DTO 필드의 좌표계, layer, 표시 단위
- `regression-code-2026-06-19/regression-code/src/audio/audio_types.ts`
  - audio schedule 필드의 score time / audio time 구분, backend 계약
- `regression-code-2026-06-19/regression-code/src/core/parse/types.ts`
  - rawText parser 결과, modifier, tuplet slot, global parser 결과의 의미
- `regression-code-2026-06-19/regression-code/src/core/analyze/types.ts`
  - display와 sound 분리, sourceCells, TimeFraction, timeline segment 의미
- `regression-code-2026-06-19/regression-code/src/app/app_types.ts`
  - AppState의 parser/analyzer/render 산출물 보관 구조와 DOM reference 역할
