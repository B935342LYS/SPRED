# 1.9. 1단계 PletHead Parser 구현 메모
> 본 설계 메모 문서 1.9에서는 `parse_note_cell.ts`의 tuplet head 파싱 구현 범위를 기록한다.
> 본 문서는 active specification을 대체하지 않는다.
> 함수별 세부 설명은 코드 파일의 `/** ... */` 주석을 중심으로 하고, 본 문서는 흐름과 구현 판단을 보충한다.

## 1. 개요

일반 note와 modifier 파싱 이후, `/n(...)` 형태의 tuplet head 파싱을 추가했다.
이번 구현은 parser 책임에 해당하는 문자열 구조화까지만 수행한다.

현재 tuplet head 파싱 흐름은 아래와 같다.

```text
parseNoteCell()
-> parsePletHead()
   -> findMatchingParen()
   -> splitTupletSlots()
   -> parsePletSlot()
      -> parsePletSlotNote()
         -> scanDefaultText()
         -> parseModifiers(..., stopBeforeTupletPosition = true)
         -> parseTupletPosition()
```

## 2. 구현 범위

현재 구현한 문법은 다음과 같다.

```text
/n(slot_1|slot_2|...|slot_n)
```

구현된 제약은 다음과 같다.

- `n`은 2 이상의 정수여야 한다.
- 실제 slot 개수는 `n`과 정확히 일치해야 한다.
- 빈 slot은 rest slot으로 기록한다.
- 내용이 있는 slot은 반드시 `@n(midi_num)` 위치 토큰을 가져야 한다.
- slot 내부 nested tuplet은 금지한다.
- slot 내부 modifier는 일반 note와 같은 `@g -> @t -> @p -> @m` 순서를 따른다.
- `~` hold와 `@t(...)` tremolo 충돌은 slot 내부에서도 동일하게 `vib_and_trem`으로 처리한다.

## 3. 구현 판단

### 3.1. parser 책임 제한

`parsePletHead()`는 slot 문자열을 구조화하고, slot별 위치 토큰의 MIDI 번호를 읽는 데까지만 책임진다.
다음 내용은 analyzer 책임으로 남겼다.

- `pletHead`와 `/&` extend 결합
- tuplet 전체 길이 확정
- slot별 tick 범위 계산
- `@n(midi_num)`을 실제 `rowId`와 연결
- slot note/rest와 hold/gliss/trem의 시간축 의미 확정

이 경계를 지키면 tuplet parser가 기존 `이세계 코드`처럼 분석, 렌더링, 편집 책임까지 끌어안지 않는다.

### 3.2. modifier parser 재사용

slot note는 일반 note와 거의 같은 문법을 사용하지만 마지막에 `@n(...)`이 필수이다.
따라서 `parseModifiers()`에 `stopBeforeTupletPosition` 인수를 추가했다.

일반 note에서는 modifier 파싱 뒤 남은 문자열이 있으면 오류가 된다.
tuplet slot에서는 남은 문자열이 `@n(`이면 정상적으로 멈추고, 이후 `parseTupletPosition()`이 위치 토큰을 읽는다.

### 3.3. slot 분리

`splitTupletSlots()`는 단순 문자열 split 대신 최상위 `|`만 분리한다.
modifier 인수 괄호 내부의 문자가 slot 구분자로 오해되지 않도록 괄호 깊이를 추적한다.

현재 명세상 modifier 인수 안에 `|`를 넣는 사용은 권장되지 않지만, 이 구현은 파서가 쉽게 깨지지 않도록 최소한의 top-level 분리를 유지한다.

### 3.4. 오류 위치 보정

slot 내부 parser들은 slot 문자열 기준의 상대 인덱스를 반환한다.
사용자가 보는 오류 위치는 전체 rawText 기준이어야 하므로 `offsetCharIndex()`로 slot 시작 위치를 더한다.

## 4. 이번 구현에서 사용한 주요 TypeScript 문법

### 4.1. 선택 인수 기본값

`parseModifiers()`는 일반 note와 tuplet slot에서 함께 사용된다.

```ts
function parseModifiers(
  rawText: string,
  startIndex: number,
  hold: HoldKind | null,
  stopBeforeTupletPosition = false,
): ParseModifiersResult
```

마지막 인수는 기본값이 `false`이므로 기존 일반 note 호출부를 유지하면서 tuplet slot만 다른 동작을 요청할 수 있다.

### 4.2. 구조적 타입 호환

`ParsedModifiers`와 `ParsedPletSlotModifiers`는 필드 구조가 같다.
따라서 slot note의 `modifiers` 필드에는 `parseModifiers()`가 만든 값을 그대로 넣을 수 있다.

### 4.3. 제네릭 결과 타입 재사용

slot parser도 기존 `ParsePartResult<T>` 형태를 재사용한다.

```ts
type ParsePletSlotResult = ParsePartResult<ParsedPletSlot>;
type ParsePletSlotNoteResult = ParsePartResult<ParsedPletSlotNote>;
type ParseTupletPositionResult = ParsePartResult<ParsedTupletPosition>;
```

성공 값 타입만 바꾸고 성공/실패 형태는 동일하게 유지한다.

### 4.4. 리터럴 상수와 `typeof`

tuplet slot 위치 토큰은 문자열 상수로 둔다.

```ts
const TUPLET_POSITION_TOKEN = "@n(" as const;
type TupletPositionToken = typeof TUPLET_POSITION_TOKEN;
```

`as const`를 붙이면 `TUPLET_POSITION_TOKEN`의 타입이 넓은 `string`이 아니라 정확한 `"@n("` 리터럴 타입으로 유지된다.
`typeof TUPLET_POSITION_TOKEN`은 값에서 타입을 뽑아 오는 문법이다.
이 타입을 `ModifierToken | TupletPositionToken`처럼 union에 포함하면, `readModifierArgs()`가 일반 modifier 토큰과 tuplet 위치 토큰을 같은 방식으로 처리할 수 있다.

### 4.5. 정규식 literal

tuplet head 후보와 division 숫자 검사는 정규식 literal을 사용한다.

```ts
/^\/[0-9]+\(/.test(rawText)
/^[0-9]+$/.test(divText)
```

`^`는 문자열 시작, `$`는 문자열 끝을 뜻한다.
`[0-9]+`는 숫자가 1개 이상 이어지는 형태를 뜻한다.
`/`와 `(`는 정규식에서 특별한 의미가 있으므로 `\/`, `\(`처럼 escape한다.

### 4.6. nullish coalescing

slot 파싱에서는 배열에서 값을 꺼낼 때 `??`를 사용한다.

```ts
const slotText = splitResult.value.slots[slotIndex] ?? "";
const slotStartIndex = splitResult.value.startIndexes[slotIndex] ?? bodyStartIndex;
```

`??`는 왼쪽 값이 `null` 또는 `undefined`일 때만 오른쪽 fallback을 사용한다.
빈 문자열 `""`이나 숫자 `0`은 유효한 값으로 보존된다.
slot 문자열에서는 빈 slot이 rest를 뜻하므로, `||`보다 `??`가 의도에 맞다.

### 4.7. `startsWith()`와 위치 인수

`startsWith()`는 두 번째 인수로 검사 시작 위치를 받을 수 있다.

```ts
rawText.startsWith(TUPLET_POSITION_TOKEN, index)
```

이 문법을 사용하면 문자열을 매번 slice하지 않고, 현재 parser index에서 특정 토큰이 시작되는지만 확인할 수 있다.
parser는 index를 계속 이동시키므로 이 방식이 흐름을 읽기 쉽다.

### 4.8. while loop와 괄호 깊이 추적

`splitTupletSlots()`는 `body.split("|")`를 사용하지 않고 `while` loop로 한 글자씩 읽는다.

```text
depth += 1 when "("
depth -= 1 when ")"
depth === 0인 "|"만 slot 구분자로 사용
```

이 방식은 modifier 인수 괄호 안의 문자를 최상위 slot 구분자로 오해하지 않게 한다.
parser에서 단순 문자열 분리가 위험한 경우, 이렇게 작은 상태값을 두고 직접 scan하는 편이 안정적이다.

## 5. 테스트 확인

`dev/test_parse.ts`에 pletHead 직접 샘플을 추가했다.

valid 샘플은 다음 경로를 확인한다.

- 일반 slot 3개
- modifier가 있는 slot
- 빈 rest slot
- hold와 absolute pitch가 있는 slot

invalid 샘플은 다음 오류 코드가 유지되는지 확인한다.

- `invalid_tuplet_division`
- `tuplet_slot_count_mismatch`
- `tuplet_position_required`
- `tuplet_nested_forbidden`
- `vib_and_trem`

현재 성공 출력은 다음 항목을 포함한다.

```text
Tuplet head sample parse completed.
valid pletHead samples: 2
unexpected valid-pletHead samples: 0
Invalid tuplet head sample parse completed.
invalid pletHead samples: 5
unexpected invalid-pletHead results: 0
```
