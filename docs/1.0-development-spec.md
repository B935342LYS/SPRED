# 개발 전 알아둘 사항

## 1. 개요
 작업공간 내의 "이세계 코드"는 구글 스프레드시트와 앱스스크립트를 기반으로 동작하는 코드이다.
이를 타입스크립트, html, css를 기반으로 동작하고 깃허브 페이지로 배포될 웹 앱 "회귀 코드"로 리팩토링하고 성능 개선과 확장을 진행코자 한다.
이는 1년간 졸업 프로젝트로서 진행되는 사안이며, 따라서 이후 작성되는 모든 문서는 명세이자 설계문서인 동시에 교수님을 대상으로 한 보고서의 성격도 겸한다.

교수님의 요구사항에 따라, 1.7 이후의 구현 보고서에는 난이도 소명, challenging한 부분, 그 부분이 challenging한 이유와 구체적인 난이도 계산, 그 해결 과정이 명확히 기재되어 있어야 한다.


1. 이세계 코드와 회귀 코드는 별개이다. 회귀 코드는 이세계 코드와의 호환을 목표로 하지 않는다.
2. 이세계 코드는 알고리즘과 기능 흐름을 참고하기 위한 구버전 구현체로만 취급한다.
3. 회귀 코드는 `TypeScript + HTML + CSS + GitHub Pages` 기반 웹 앱으로 재구성한다.
4. 설계 판단이 충돌할 경우, 이세계 코드나 이세계 코드 명세보다 회귀 코드 명세를 우선한다.
5. 구현에 들어가기 전에는 설계 문서를 우선 정리하고, 현재 단계와 다음 단계를 `1.0` 문서에 기록한다.
6. 내용 추가/수정시 파일을 바로 수정하지 않고 변경 이유와 방향을 먼저 설명한다. 이후 개발자가 직접 내용 수정을 진행하며, 파일 직접 수정은 요청시에만 실시한다.
7. 한 단락의 작업이 끝나면 HARNESS와 본 문서를 갱신한다.

- 표, 이미지, 세부 서식은 Markdown 변환 과정에서 일부 단순화될 수 있다.

## 3. 설계 로드맵

현재는 구현 단계가 아니라 설계 단계이다. 아래 순서를 기준으로 진행한다.

1. 요구사항과 범위 고정
   - 회귀 코드가 지원할 목표 기능과 비목표를 정리한다.
   - 이세계 코드와의 비호환 원칙, 단일 현 우선 원칙, 확장 방향을 고정한다.

2. 아키텍처 방향 정리
   - Presentation / Application / Core / Infrastructure 계층을 정리한다.
   - 브라우저 실행 구조와 정적 배포 구조를 정리한다.

3. 악보 JSON 데이터 구조 확정
   - 문서 포맷, 메타데이터, 악기 정보, 전역 행, 레이아웃, 트랙 구조를 고정한다.
   - 참조 무결성과 검증 규칙을 정리한다.

4. 노트 문자열 문법 확정
   - note 셀과 global 셀의 토큰 규칙을 정리한다.
   - 문법 오류와 의미 오류의 경계를 구분한다.

5. 파서 중간 산출물 구조 확정
   - `ParsedCell` 및 하위 구조를 고정한다.
   - parser가 책임질 범위와 analyzer가 책임질 범위를 분리한다.

6. 파싱 알고리즘 수도 코드 정리
   - `parseMuteText`, `parseNoteText`, `parsePletHead`, `parsePletExtend` 절차를 작성한다.
   - escape 규칙, 토큰 순서, 충돌 규칙을 parser 단계에서 어디까지 검증할지 명시한다.

7. analyzer / renderer / audio 입력 인터페이스 정리
   - parser 출력이 후속 모듈에서 어떤 형태로 소비될지 정의한다.
   - 필요한 추가 산출물과 이벤트 리스트 구조를 정리한다.

8. 그 이후에 최소 구현 범위 확정
   - 설계가 충분히 고정된 뒤에야 TypeScript 코드 골격 구축으로 넘어간다.


## 4. 현재 진척도

### 4.1. 완료

- `1.3` 악보 JSON 데이터 구조 초안 작성
- `1.4` 노트 문자열 문법 초안 작성
- `1.1`, `1.2` DOCX 문서의 Markdown 참조본 정리
- 회귀 코드가 이세계 코드와 별개라는 원칙 확인
- 파서를 `lexer -> parser` 분리 구조 대신 hand-written scanner/parser 결합형으로 두는 방향 정리
- `1.5` 파서 중간 산출물 데이터 구조 설계 정리
  - `ParsedCell`, `ParsedInvalidCell`, `ParseError`, `ParseErrorCode`
  - `ParsedGliss`, `ParsedTrem`, `ParsedAbsolutePitch`, `ParsedMicroPitch`, `ParsedTupletPosition`
  - parser 산출물 discriminant를 `kind` 기준으로 정리
  - `glissKind`, `charIndex` 등 주요 필드명 정리
- parser 단계와 validator / analyzer 단계의 책임 경계 정리
- `ParsePartResult<T>` 기반 공통 하위 파서 인터페이스 정리
- `parseNoteText`, `parseMuteText`, `parsePletHead`, `parsePletExtend`, `parseRawText` 수도 코드 정리
- 종류별 파싱 규칙과 수도코드를 통합한 문서 구조로 `1.5` 재편
- 전역 행 정보 누락을 확인하고 저장 구조와 parser 문서를 보강
  - `1.3`에서 전역 행 저장 구조를 `globalLines.cells[{ rowId, col, rawText }]` 기준으로 보완
  - `1.6`에서 전역 행 문자열 문법, `ParsedGlobalCell` 구조, `parseGlobalCell` 수도 코드 정리
  - `rowId`를 기준으로 전역 행 kind를 유도하는 parser 문맥 구조 정리
- `1.7` analyzer 이벤트 리스트 데이터 구조 설계 정리
  - `AnalysisResult`, `AnalyzedTrackResult`, `AnalyzedEvent`, `TimeFraction`, `TimeRange`
  - `NoteEvent`, `RestEvent`, `MuteEvent`, `GlissEvent`, `TupletGroupEvent`
  - `AnalyzedTimeSegment`, `AnalyzedDynamicsSegment`, `AnalyzedIssue`
  - hold / gliss / tuplet / mute / 일반 note의 analyzer 변환 규칙 정리
  - timing timeline / dynamics timeline 분리
  - 부분 갱신 규칙, 런타임 파생 캐시, analyzer 모듈 분리 방침 정리
  - `AnalysisResult`를 공식 분석 결과, `AnalysisCache`를 파생 캐시로 분리하는 방향 정리
  - `sourceCells`의 구현 우선순위를 핵심 이벤트 / 세그먼트 중심으로 정리
- `1.8` parser-analyzer 파이프라인 인터페이스 문서 정리
  - `ScoreFile -> ScoreIndexes -> ParsedScoreDocument -> AnalysisResult` 흐름 고정
  - `GlobalCellParserInput`은 `rowId`만 받고 `GlobalCellParserContext.rowById` 문맥에서 전역 행 `kind`를 유도하도록 정리
  - `ParsedScoreDocument`를 parser 단계의 공식 결과, 보조 배열/인덱스를 파생 캐시로 두는 방향 정리
- `1.6` 보완안 문서의 핵심 내용을 active 문서로 흡수
  - 저장 구조 원칙은 `1.3`에 반영
  - analyzer 입력 경계는 `1.7`에 반영
  - 저장 구조 / 정규화 구조 / 해석 구조 구분과 `RuntimeDocument` 설명은 `1.8`에 반영
  - `1.6`은 구현 기준 문서에서 reference 문서로 역할 조정
- `1.0~1.8` 문서 간 용어 및 타입 정합성 보정
  - `ParsedTupletPosition` 명칭 통일
  - trem / vibrato 충돌 수도코드 수정
  - global partial request를 단수 `kind` 기준으로 정리
- 제출용 보고서 본문 작성 완료
  - JSON 저장 구조 설계
  - 노트 문자열 문법 설계
  - parser 중간 산출물 구조 설계
  - 종류별 파싱 절차 및 최상위 파싱 절차 정리

### 4.2. 진행 중

- `ScoreIndexes`, `ParsedScoreDocument`, `AnalysisResult`를 실제 TypeScript 타입 정의로 옮기기 위한 초안 작성
- full parse / full analysis 기준 최소 구현 범위 정리
- analyzer 모듈 경계와 파생 캐시 책임 범위를 실제 파일 구조 단위로 구체화
- `sourceCells` 1차 구현 우선 대상과 후순위 대상의 구현 순서 정리
- 설계 문서를 기준으로 구현 착수 가능한 최소 골격 범위 재점검

## 5. 작업 원칙

- 구현보다 설계를 우선한다.
- 문법 오류는 parser 단계에서 최대한 판정한다.
- 셀 단독으로 판단할 수 없는 사항만 validator / analyzer 단계로 넘긴다.
- UI 입력 경로와 외부 import 경로를 분리해서 생각한다.
- JSON에는 `rawText`를 저장하고, `parsedCell`은 런타임 산출물로 유지한다.

## 6. 메모

- `1.1-project-plan.md`와 `1.2-master-spec.md`는 반복 대화용 참조본이다.
- DOCX 기반 원본 내용은 유지하되, 이후 실작업 참조는 가능하면 Markdown 사본을 우선 사용한다.
- parser 단계 설계와 제출용 보고서 본문은 현재 기준으로 일단락된 상태로 본다.
- analyzer 이벤트 리스트 구조는 `1.7-analyzer-event-list-spec.md`에 정리하였다.
- `1.7`에는 이벤트 구조뿐 아니라 전역 timing/dynamics 세그먼트, 진단 구조, 부분 갱신 규칙까지 반영되었다.
- 전역 행은 저장 구조 누락을 확인한 뒤 `rawText` 기반 셀 구조와 parser 문서를 보강하였다.
- `1.8` 문서까지 포함해 parser-analyzer 파이프라인 인터페이스 초안은 일단락된 상태로 본다.
- 다음 단계는 `1.0~1.8`에서 고정한 설계를 실제 TypeScript 타입 / 함수 시그니처 / 최소 구현 골격으로 옮기는 일이다.
