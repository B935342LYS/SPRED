# 1.5. 1단계 RuntimeDocument 생성 메모
> 본 설계 메모 문서 1.5에서는 1단계 구현 중 `create_runtime_document.ts`의 역할과 구현 판단을 기록한다.
> 본 문서는 active specification을 대체하지 않는다.
> 함수별 세부 설명은 코드 파일의 `/** ... */` 주석을 중심으로 하고, 본 문서는 흐름과 책임 경계를 보충한다.

## 1. 개요

`RuntimeDocument`는 저장 구조인 `ScoreFile`과 런타임 조회 구조인 `ScoreIndexes`를 한 단위로 묶는 타입이다.
`ScoreIndexes`는 독립된 원본 데이터가 아니라 특정 `ScoreFile`에서 파생된 구조이므로, 두 값을 함께 전달하는 컨테이너가 필요하다.

현재 score 모듈의 로드 흐름은 아래와 같다.

```text
JSON text
-> loadScoreJson()
-> validateScoreFile()
-> ScoreFile
-> buildScoreIndexes()
-> RuntimeDocument
```

`create_runtime_document.ts`는 이 흐름의 마지막 score 모듈 진입점이다.
parser/analyzer는 이후 `RuntimeDocument.score`와 `RuntimeDocument.indexes`를 함께 받아 동작하는 방향을 기본으로 본다.

## 2. create_runtime_document.ts의 현재 책임

현재 파일은 두 함수를 제공한다.

- `createRuntimeDocument(score)`
  - 이미 검증된 `ScoreFile`을 받아 `buildScoreIndexes()`를 호출한다.
  - `{ score, indexes }` 형태의 `RuntimeDocument`를 반환한다.
- `loadRuntimeDocument(jsonText)`
  - JSON 문자열에서 `loadScoreFile()`을 먼저 호출한다.
  - 성공 시 `createRuntimeDocument()`로 런타임 문서를 만든다.
  - 실패 시 `JsonLoadError` 또는 `ScoreValidationError`를 그대로 반환한다.

책임 경계는 다음과 같다.

```text
json_load.ts
  - JSON parsing
  - ScoreFile validation delegation

score_validate.ts
  - minimum ScoreFile structure/reference validation

build_score_indexes.ts
  - runtime lookup index generation

create_runtime_document.ts
  - ScoreFile and ScoreIndexes bundling
  - convenience full load entry point
```

## 3. 구현 판단

### 3.1. createRuntimeDocument()와 loadRuntimeDocument()를 분리한 이유

`createRuntimeDocument()`는 이미 검증된 `ScoreFile`을 받는 낮은 단계 함수이다.
테스트나 내부 로직에서 `ScoreFile`을 직접 가지고 있을 때 JSON 파싱을 다시 거치지 않고 런타임 문서를 만들 수 있다.

`loadRuntimeDocument()`는 브라우저 파일 입력이나 import 경로에서 사용할 수 있는 높은 단계 편의 함수이다.
JSON 문자열 하나를 받아 파싱, 검증, 인덱스 생성까지 수행한다.

이 두 함수를 분리하면 다음 장점이 있다.

- 로드 경로와 인덱스 생성 경로를 독립적으로 테스트할 수 있다.
- 후속 UI에서는 `loadRuntimeDocument()`만 호출해도 전체 score 로드가 끝난다.
- parser/analyzer 테스트에서는 직접 만든 `ScoreFile`로 `createRuntimeDocument()`만 호출할 수 있다.

### 3.2. 실패 타입을 그대로 전달한 이유

`loadRuntimeDocument()`는 자체 오류 타입을 새로 만들지 않고, 이전 단계의 오류를 그대로 반환한다.

```ts
error: JsonLoadError | ScoreValidationError
```

런타임 문서 생성 단계 자체는 현재 실패 조건을 따로 가지지 않는다.
실패는 JSON 파싱 또는 ScoreFile 검증 단계에서 발생한다.
따라서 새 오류 코드를 만들기보다 기존 오류 타입을 합쳐 전달하는 편이 호출부에서 원인을 파악하기 쉽다.

### 3.3. ScoreIndexes를 RuntimeDocument 내부에 묶는 이유

`ScoreIndexes`는 특정 `ScoreFile`에 종속된 파생 구조이다.
서로 다른 악보의 `score`와 `indexes`가 섞이면 parser/analyzer 결과가 틀어질 수 있다.

`RuntimeDocument`로 묶어두면 후속 모듈 호출 시 다음 형태를 유지할 수 있다.

```ts
{
  score,
  indexes
}
```

즉 `RuntimeDocument`는 새로운 진실 원천이 아니라, 원본과 파생 조회 구조가 같은 문서에 속한다는 점을 표현하는 컨테이너이다.

## 4. 이번 구현에서 사용한 주요 TypeScript 문법

### 4.1. union result type

`LoadRuntimeDocumentResult`는 성공과 실패를 union으로 표현한다.

```ts
export type LoadRuntimeDocumentResult =
  | {
      ok: true;
      document: RuntimeDocument;
    }
  | {
      ok: false;
      error: JsonLoadError | ScoreValidationError;
    };
```

`ok` 필드를 기준으로 성공/실패 분기를 나누면 TypeScript가 각 분기에서 사용할 수 있는 필드를 좁혀준다.

### 4.2. import type

런타임 값으로 필요한 함수는 일반 import를 사용하고, 타입만 필요한 항목은 `import type`을 사용한다.

```ts
import { buildScoreIndexes } from "./build_score_indexes";
import type { RuntimeDocument, ScoreFile } from "./types";
```

이 구분은 파일이 실제 실행 시 필요한 값과 타입 검사에만 필요한 이름을 분리해 준다.

### 4.3. 오류 전파

`loadRuntimeDocument()`는 `loadScoreFile()` 실패 결과를 그대로 반환한다.

```ts
if (!loadResult.ok) {
  return loadResult;
}
```

이 분기에서 TypeScript는 `loadResult`를 실패 케이스로 좁힌다.
반환 타입도 `{ ok: false; error: JsonLoadError | ScoreValidationError }`와 호환되므로 별도 변환이 필요 없다.

## 5. 테스트 확인

`dev/test_score.ts`는 이제 `loadRuntimeDocument()`를 호출한다.
따라서 테스트 경로는 다음 전체 흐름을 포함한다.

```text
JSON file
-> loadRuntimeDocument()
-> loadScoreFile()
-> validateScoreFile()
-> buildScoreIndexes()
-> RuntimeDocument
```

현재 fixture 기준 성공 출력은 다음과 같다.

```text
Runtime document load succeeded.
title: testScore
rows: 72
tracks: 1
global cells: 4
indexed rows: 72
indexed tracks: 1
indexed note rows for s1: 34
```
