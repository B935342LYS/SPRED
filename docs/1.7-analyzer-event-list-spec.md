# analyzer 이벤트 리스트 데이터 구조 설계

## 1. 문서 목적

- 본 문서는 analyzer가 `ParsedCell`과 `ParsedGlobalCell`, 문서 전역 인덱스를 입력으로 받아 생성하는 이벤트 리스트 구조를 정의한다.
- parser가 확정한 셀 단위 문법 해석 결과를, renderer와 audio generator가 공통으로 소비할 수 있는 시간축 이벤트로 변환하는 것이 목적이다.
- 특히 hold, gliss, tuplet, mute, 일반 note가 어떤 analyzer 이벤트로 귀결되는지 명확히 정리한다.
- 본 문서는 `1.5-note-cell-parser-spec.md`, `1.6-global-cell-parser-spec.md`, `1.8-parser-analyzer-pipeline-spec.md`를 직접 이어받는다.


## 2. 난이도 소명

 문제 1. 분기된 해석 결과를 다시 하나의 분석 결과로 통합해야 한다.

전역 셀과 노트 셀은 parser 단계부터 서로 다른 경로로 해석되며, 그 내부에서도 다시 행 종류별, 셀 종류별, modifier 종류별 분기가 계속 발생한다. 그런데 analyzer 단계에서는 이 분기된 결과를 단순히 나열하는 것이 아니라 renderer와 audio generator가 함께 사용할 수 있는 하나의 이벤트 구조로 재조직해야 했다. 즉 서로 다른 분기 결과를 어떤 기준으로 다시 묶어 공통 산출물로 만들 것인가가 어려운 지점이었다.

해결 1. 이벤트 의미 기준으로 재조직
 설계 과정에서 먼저 전체 분기 구조를 도식으로 정리하여 어떤 경로가 어디에서 갈라지고 다시 합쳐지는지를 시각적으로 명확히 하였다. 그 후 analyzer 산출물을 의미에 따라 발음, 관계, 구조, 전역 타임라인이라는 기준으로 다시 묶었다. 노트 셀과 전역 셀의 해석 경로는 분리하되, 최종적으로는 trackResults, timingTimeline, dynamicsTimeline, analysisIssues를 묶은 공통 구조로 수렴하도록 정리하였다.


문제 2. 구현 이전 시점에서 후속 모듈의 동작 예측이 필요
 이벤트리스트는 이후 renderer와 audio generator가 직접 소비하는 핵심 입력이다. 그러나 현 단계에서 renderer와 audio generator의 세부 알고리즘은 작성되지 않았다. 따라서 어떤 속성이 실제 렌더링과 재생에 필요할지, 그리고 두 모듈이 동시에 사용하기에 적절한 구조가 무엇인지를 실제 테스트 없이 예측해야 했다.

해결 2. 프로토타입의 외부 동작과 예상 Use Case를 기준으로 역추론


문제 3. 노트 셀 이벤트들의 교집합
 5.3 항목에서 본 바와 같이 노트 셀을 기반으로 생성되는 이벤트는 상호 독립적이지 않고 그물처럼 엮이는 관계를 가지고 있다. 하나의 셀을 하나의 이벤트와만 대응하는 구조는 성립할 수 없으며, 다양한 중첩 상황을 고려해야만 했다. 특히 잇단음표는 전체 길이를 측정하기 위한 구조와 그걸 나눈 실체를 가진 구조가 동시에 필요함이 자명한 상황이었다.

해결 3-1. 의미와 용도를 명확히 구분하고 명세
 의미와 용도를 최대한 명확하게 구분하였다. 음을 내는 것은 NoteEvent, 구조 정의는 TupletGroupEvent, 구간 내 음정 변화 관계는 GlissEvent가 담당하도록 구분지었다. 이 이벤트들이 시간축 위에서 겹쳐 존재할 수 있음을 전제로 설계를 진행하였다. 또한 주요 이벤트에 sourceCells를 두어 하나의 셀이 여러 결과 구조에 참여하더라도 원인을 추적할 수 있게 하였다.

해결 3-2. (부분적) 잇단음표를 소리-구조의 두 층으로 분리
 잇단음표는 그룹 구조를 나타내는 TupletGroupEvent와 실제 발음 단위를 나타내는 슬롯 note/rest 이벤트를 분리하여 설계하였다. 그룹과 슬롯은 groupId로 연결되며, 부분 갱신 시에도 슬롯 하나만 수정하지 않고 관련 그룹 전체를 다시 계산하는 방향으로 정리하여 구조적 일관성을 확보하였다. 다만, 이 부분에 대해서는 조금 더 고민해볼 예정이다. 현재의 셀 기반 편집 구조에서 잇단음표를 편집하는 최선의 방법이라고는 생각한다. 하지만 UX 측면에서는 하나의 셋잇단을 편집하기 위해 슬롯과 TupletGroup에 모두 관여해야 하는 상황이라 편의성을 챙기기 위해 신중한 UI 설계가 필요하다.


문제 4. 편집 중 자연스러운 미완성 상태와 오류의 구분
 편집은 하나의 셀 단위로 이루어지므로 악보 완성 전에는 다양한 미완성/미완결 상태가 발생할 수 밖에 없다. 이를 오류로 판정할 것인가? 

해결 4. (다소 보류) UX 편의를 위한 방치 결정
 편집 과정에서 계속 오류 알림이나 메세지가 발생하면 UX를 크게 해칠 수 밖에 없다. 따라서 자연스럽게 발생하는 gliss anchor의 연결 대상 부재, pletHead 단독 존재와 같은 상태는 analyzer의 정식 오류 또는 경고로 기본 보고하지 않기로 결정하였다. 아예 전체 구조의 생성이 불가능한 경우만 analyzer가 오류라 보고한다. 자연스러운 미완성 상태는 필요하면 막지는 않고 UI로 알려주기만 하는 방향으로 정책을 결정한다.



## 3. analyzer의 위치와 책임

회귀 코드의 데이터 흐름은 아래와 같이 본다.

1. JSON 파일 로드
2. validator가 문서 구조 검증
3. parser가 각 셀의 `ParsedCell`, `ParsedGlobalCell` 생성
4. analyzer가 셀 간 관계와 시간축 의미를 확정
5. renderer / audio generator가 analyzer 이벤트를 소비

analyzer는 아래 항목을 책임진다.

- hold 연결 성립 여부 판정
- gliss 연결 상대 탐색
- tuplet head와 extend 결합
- tuplet 내부 slot의 절대 시간 위치 계산
- `@n(midi_num)`를 실제 `rowId`로 해석 (헤드 셀로부터 상속받은 stringId 내부에서 찾는다)
- `@p(midi_num)`의 음정 유효성 검증

- 전역 셀의 열 순서 관계 해석
- BPM/박자/step/dynamics 세그먼트 생성
- 전역 행 issue 생성

- renderer / audio generator 공통 입력인 이벤트 리스트 생성

반대로 analyzer는 문자열을 다시 해석하지 않는다. `rawText`의 문법 해석은 parser에서 끝나야 한다.


## 4. Analyzer의 입력과 출력

analyzer는 단일 셀 하나만을 입력으로 받아 동작하는 모듈이 아니다.
실제로는 parser가 생성한 ParsedCell들과, 이를 해석하기 위한 문서 전역 인덱스를 함께 참조하여 시간축 의미를 확정한다.

### 4.1. 입력

analyzer의 입력은 크게 다음 세 층으로 나뉜다.

1. 문서 원본 정보
    - ScoreFile

2. 런타임 정규화 인덱스
    - rowById
    - rowsInDisplayOrder
    - noteRowIdsByStringId
    - noteRowIdByStringMidi
    - cellsByTrackAndCol
    - globalCellsByKindAndCol
    - globalCellsInColOrder

3. parser 산출물
    - 문서 단위로 파싱 결과를 묶은 ParsedScoreDocument
      - 각 노트 셀에 대해 계산된 ParsedCell
      - 각 전역 셀에 대해 계산된 ParsedGlobalCell


여기서 중요한 점은, JSON 파일이 저장 단계에서 이미 정렬되어 있더라도 analyzer가 곧바로 raw JSON을 직접 소비하지는 않는다는 것이다.
저장 구조의 정렬은 파일 안정성과 diff 품질을 위한 규칙이며, analyzer 입력 구조와는 목적이 다르다.

ParsedScoreDocument는 정렬을 위한 자료구조가 아니라, parser가 문자열 해석을 완료한 뒤 analyzer에 전달하는 런타임 문서 상태이다.
즉 analyzer는 원본 rawText를 다시 해석하지 않고, parser가 생성한 ParsedCell, ParsedGlobalCell 및 이를 문서 단위로 묶은 ParsedScoreDocument를 입력으로 사용한다.

이 구분은 책임 경계를 분명히 하기 위해 중요하다.
저장 구조인 JSON은 사람이 읽고 수정하기 쉬운 최소 정보만 담고, analyzer는 그 위에 직접 올라가지 않는다.
analyzer는 반드시 `ScoreFile`에서 파생된 `ScoreIndexes`와 parser가 만든 `ParsedScoreDocument`를 기준으로 동작하며, 저장 구조의 배열 정렬 규칙이나 원본 문자열 자체를 다시 해석하지 않는다.

노트 셀과 전역 셀은 analyzer 내부에서 서로 다른 경로로 처리한다. 두 입력은 모두 셀 좌표를 기준으로 다루지만, 필요한 문맥과 후속 처리 목표가 다르므로 별도 타입으로 분리하는 편이 적절하다.

``` ts
type AnalyzerNoteCellInput = {
  trackId: string;
  rowId: string;
  col: number;
  parsedCell: ParsedCell;
};

type AnalyzerGlobalCellInput = {
  rowId: string;
  kind: GlobalKind;
  col: number;
  parsedCell: ParsedGlobalCell;
};
```

Analyzer는 노트 셀 입력과 전역 셀 입력을 서로 다른 경로로 처리한다.
다만 실제 analyzer는 위와 같은 개별 셀 입력만으로 동작하지 않고, 반드시 같은 시점의 문서 인덱스와 주변 셀 문맥을 함께 참조해야 한다.
예시: 노트 행의 hold 연결, gliss anchor 탐색, tuplet extend 결합, @n(midi_num)의 실제 행 해석은 모두 단일 셀 정보만으로는 확정할 수 없다.
예시: 전역 행의 선형 변화 구간은 선형 변화 토큰을 가진 셀들을 짝지어야 확정할 수 있다.

### 4.2. 출력

analyzer의 출력은 AnalysisResult이다.
AnalysisResult는 renderer와 audio generator가 공통으로 사용할 수 있도록, 문서 전체의 시간축 해석 결과와 이벤트 리스트, 그리고 분석 단계 진단 정보를 함께 포함한다.

즉 analyzer 출력은 최소한 아래 성격의 정보를 포함해야 한다.

 - analyzer가 생성한 최종 이벤트 리스트
 - 전역 셀을 해석한 timing/dynamics 세그먼트 정보
 - 분석 중 발생한 오류 및 경고 진단 정보

 이때 중요한 점은 renderer와 audio generator가 ParsedCell, ParsedGlobalCell의 세부 규칙을 다시 해석하지 않도록 하는 것이다.
후속 단계는 가능한 한 AnalysisResult만을 소비해야 하며, analyzer 이전 단계의 세부 문법 구조에는 직접 의존하지 않는 것을 원칙으로 한다.

즉 역할을 짧게 정리하면 아래와 같다.

- JSON 저장 구조는 원본 문자열과 참조 관계를 보존한다.
- parser는 셀 문자열의 1차 해석을 담당한다.
- analyzer는 셀 간 관계와 시간축 의미를 확정한다.
- renderer와 audio generator는 되도록 analyzer 결과만 소비한다.


## 5. Analyzer가 확정해야 하는 정보

### 5.1. 요구사항 명세

parser가 단일 셀의 해석을 다루었다면, analyzer는 주로 “관계”를 다룬다. 즉 analyzer는 문서 전역의 행/열 인덱스, 현재 셀과 주변 셀의 ParsedCell/ParsedGlobalCell을 함께 참조하여 renderer와 audio generator가 공통으로 사용할 수 있는 이벤트를 생성해야 한다.

Analyzer는 모든 일반 셀 또는 잇단음표 슬롯에 대해 우선 다음 두 값을 확정해야 한다.

1. FinalSoundPitch
    - 실제 발음에 사용될 최종 음정이다.
    - modifier가 없는 일반 셀의 기본 음정은 해당 rowId가 나타내는 음정이다.
    - @p(midi_num)가 있으면 그 값을 우선 적용한다.
    - @m(cent_num)가 있으면 `-100~100` 범위의 유한 실수 cent 단위 미세 음정 보정을 추가한다.

2. FinalDisplayPosition
    - 렌더링에 사용될 최종 표시 위치 정보이다.
    - 이는 픽셀 좌표 자체를 의미하지 않으며, 렌더러가 실제 y좌표로 변환할 수 있는 의미적 위치 정보로 본다.
    - 기본적으로는 배치된 rowId를 기준으로 한다.
    - @n(midi_num)가 있는 잇단음표 슬롯은 해당 음정에 대응하는 표시 행을 기준으로 한다.
    - @m(cent_num)가 있으면 표시 위치에도 `-100~100` 범위의 유한 실수 cent 단위 보정 정보가 추가된다.


이후 hold, gliss, trem, tuplet 관련 연결 판정은 모두 동일 trackId 내부에서만 수행한다.
서로 다른 track에 존재하는 셀들은 절대로 하나의 이벤트로 결합하지 않는다.
또한 gliss를 제외한 연결은 시간축상 서로 인접한 셀 또는 슬롯 사이에서만 성립한다고 본다.

1. hold의 연결
        - holdKind가 null이 아닌 셀/슬롯은 hold 연결의 후보이다.
        - 연결 가능 여부는 최종 발음 음정과 최종 표시 위치가 모두 일치하는지를 기준으로 판단한다.
        - rowId 및 @m의 cent 보정까지 일치해야 연결됨을 명시한다.
        - 연결 가능한 시작점이 존재하면 현재 셀 또는 슬롯은 기존 음의 연장 구간으로 편입된다.
        - 연결 가능한 시작점이 존재하지 않으면 현재 셀 또는 슬롯이 스스로 새로운 음의 시작점이 된다.

        - hold == "~"인 경우 연결 규칙 자체는 "-"와 동일하되, 추가적으로 Vib 속성을 함께 부여한다.
        - 어떤 시작 셀/슬롯의 바로 오른쪽에 “~”이 연결되는 경우 시작 셀에도 Vib 속성이 적용된다.

    2. gliss의 연결
    - gliss 수식어가 존재하는 셀/슬롯은 gliss anchor의 후보이다.
    - gliss 연결은 동일 trackId, glissId여야 성립한다.
    - gliss 연결은 빈칸을 건너뛴 탐색을 허용한다.

    - glissKind === “start”인 anchor는 gliss의 시작점이 된다.
    - glissKind === “mid” 또는 glissKind === “end”인 셀/슬롯은 왼쪽으로 탐색하여 가장 가까운 유효한 “start” 또는 “mid”와 연결된다.
    - 이때 연결은 인접 anchor 쌍마다 하나의 gliss segment를 생성한다. 전체 [start, mid, …, end] anchor를 통합하여 하나의 긴 체인을 구성하지 않는다. 이는 추후 증분 분석을 쓸데없이 복잡하게 만들 것으로 예상되므로 채택하지 않는다.
        예시: [start, mid, end]는 (start → mid), (mid → end)의 음형 구간으로 분리된다. 

    - “end” anchor는 글리산도의 종착점이다. 다음 gliss 구간의 시작점으로 재사용되지 않는다.

    - gliss anchor는 기본적으로 음의 시작점으로 동작한다. 부드러운 음의 연결 효과는 페이드 인/아웃으로 소리를 겹쳐서 구현한다.
    - 다만 start anchor에 hold가 결합된 경우에는 새로운 발음을 다시 시작하지 않고, 기존 음의 연장 위에서 gliss가 시작될 수 있다.
        예시: ["~", "~", "@g(a, start)"], ["-", "-", "@g(a, start)"]는 @g에서 음이 재시작되지 않고 부드럽게 이어진다.

    - gliss segment의 실제 음형은 양 끝 anchor의 finalSoundPitch를 기준으로 선형 보간되는 것으로 본다. renderer는 이를 선형 경사선으로 표현할 수 있고, audio generator는 이를 시간에 따른 연속 pitch 변화로 해석할 수 있다.

    - 여러 anchor가 같은 열에 존재하여 하나의 목표 anchor에 대해 1:n 연결이 생기는 경우도 문법상 금지하지 않는다. 이 경우 각 시작점에서 독립적인 gliss segment가 생성될 수 있다
        예시: ["A3@g(a, start)", "C4@g(a, start)"]가 같은 열에 두개 있고, 가장 가까운 다음 anchor가 [E4@g(a, end)]라면, 두 start에서 모두 음이 시작해서 선형적으로 E4로 수렴한다.

    - 시간축상 일렬로 놓인 anchor들에 대해서는 항상 가장 가까운 이전 유효 anchor만 연결 대상으로 인정한다.
        예시: [start, start, mid, end] : 우측의 start~end 세트만 연결된다.
                [start, end, end] : 좌측의 start~end만 연결된다.


3. 잇단음표의 해석
    - pletHead와 오른쪽으로 연속된 pletExtend를 결합하여 하나의 tupletGroup을 형성한다.
    - pletHead를 포함한 연속 열 구간의 길이를 전체 길이로 본다.
    - 이 전체 길이를 divNum으로 나눈 값이 각 슬롯의 길이이다.

    - 각 슬롯의 시작 시각과 길이는 유리수 tick으로 기록한다.
    - 슬롯의 표시 위치는 @n(midi_num)이 지정하는 행을 우선 기준으로 한다.

    - @n(midi_num)가 실제 행으로 해석되지 않는 경우, 해당 슬롯은 시간 위치만 유지한 rest slot으로 대체하고 analyzer 진단을 생성한다. 이 경우 잘못된 slot 하나 때문에 전체 tupletGroup을 폐기하지는 않는다.
    - 슬롯 내부의 "-"와 "~"는 우선 직전 슬롯의 음을 참조한다. 직전 슬롯에 참조 가능한 음이 없거나, 연결 기준을 만족시키지 않으면 해당 슬롯은 스스로 새로운 시작점이 된다.

    - tuplet의 첫 슬롯은 필요할 경우 바로 왼쪽 일반 셀 또는 다른 tuplet의 마지막 슬롯과도 연결될 수 있다.

    - 반대로 tuplet의 마지막 슬롯 역시 바로 오른쪽 일반 셀 또는 다른 tuplet의 첫 슬롯과 연결될 수 있다.

4. trem의 연결
    - trem 수식어는 셀 또는 슬롯에 주기적으로 끊기는 발음 효과를 부여한다.
    - trem은 어떤 시작 지점에 부착되어 시작할 수 있으며, 이후 오른쪽으로 연속되는 hold("-") 구간 전체에 적용될 수 있다.
    - 이때 trem의 지속 여부 역시 hold와 동일한 연결 판정 결과를 따른다.
    - 왼쪽에서 이미 진행 중이던 hold 구간에 trem이 중간부터 추가된 경우, trem은 그 지점부터 새로 시작된 것으로 해석한다.
    - "~"가 시작되는 순간 기존 trem은 종료된다.
    - 따라서 trem과 vibrato는 같은 연속 구간에서 동시에 지속되는 효과로 보지 않는다.

- gliss와 trem이 함께 존재하는 경우, 선형 음정 이동과 주기적 분할 발음 효과가 동시에 적용될 수 있다. 즉 gliss는 음정 변화 규칙이고, trem은 발음 패턴 규칙이므로 서로 다른 층위의 속성으로 취급한다.

5. 실제 음정 토큰 @p의 해석
    - modifier가 없는 일반 셀의 기본 음정은 해당 셀이 배치된 "rowId"의 음정이다.

    - @p(midi_num)가 존재하는 경우, 발음 음정은 배치 행의 음정이 아니라 midi_num으로 대체된다.
    - @m(cent_num)가 존재하는 경우, 최종 발음 음정은 기준 음정에 `-100~100` 범위의 유한 실수 `cent_num`만큼의 미세 음정 보정을 더한 값으로 계산한다.

    - hold, gliss, trem의 연결 판정은 모두 이 최종 발음 음정을 기준으로 수행한다.

6. 미분음 토큰 @m의 해석
    - @m(cent_num)가 존재하는 경우 analyzer는 발음 음정과 표시 위치 양쪽에 `-100~100` 범위의 유한 실수 미세 보정 정보를 반영해야 한다.
    - 발음 음정 측면에서는 기준 음정에 cent 보정을 더한다.
    - 표시 위치 측면에서는 기준 행 위치에 대응하는 미세한 시각적 오프셋 정보를 함께 계산한다.

    - 단, analyzer는 실제 픽셀 좌표를 계산하지 않고, 렌더러가 후속 단계에서 사용할 수 있는 의미적 보정값만 산출한다.

7. 시간 단위
    - analyzer의 기본 시간 단위는 tick이며, 기본 열 1칸을 1 tick으로 둔다.
    - tuplet slot은 유리수 tick offset을 가질 수 있다.
    - note, gliss, trem, tuplet 이벤트 자체는 우선 tick 기반으로 생성한다.

    - 전역 행의 BPM, 박자 관련 셀 관계는 별도의 시간 맵 구조로 해석해야 한다.
    - 즉 analyzer는 단순한 음표 이벤트 리스트만 만드는 것이 아니라, tick -> 실제 시간(ms) 변환에 필요한 시간 세그먼트 정보도 함께 산출해야 한다.
    - renderer와 audio generator는 analyzer 결과에 포함된 이벤트 리스트와 시간 맵을 함께 소비한다.

    - 이 원칙을 통해 후속 단계가 전역 행 원문이나 parser 세부 구조를 직접 다시 해석하지 않도록 한다.

### 5.2. AnalysisResult 최상위 구조

analyzer가 확정한 결과는 최종적으로 `AnalysisResult` 구조에 담는다.

```ts
type AnalysisResult = {
  timingTimeline: AnalyzedTimeSegment[];
  dynamicsTimeline: AnalyzedDynamicsSegment[];
  trackResults: AnalyzedTrackResult[];
  analysisIssues: AnalyzedIssue[];
};

type AnalyzedTrackResult = {
  trackId: string;
  events: AnalyzedEvent[];
};

type AnalyzedIssue = {
  level: "warning" | "error";
  code: AnalyzedIssueCode;
  message: string;
  trackId?: string | null;
  sourceCells: SourceCellRef[];
};

type AnalyzedEvent =
  | NoteEvent
  | RestEvent
  | MuteEvent
  | GlissEvent
  | TupletGroupEvent;
```

필드 설명:
- timingTimeline
  - BPM, 박자, 스텝의 전역 변화를 시간 세그먼트로 정규화한 결과이다.
  - 세부 구조는 이후 절에서 정의한다.
  - 이는 analyzer가 tick -> 실제 시간(ms) 변환과 전역 상태 해석을 위해 생성하는 문서 전체 시간 맵이다.
  - AnalyzedTimeSegment는 하나의 timing 해석 규칙이 유지되는 시간 구간의 배열이다.

- dynamicsTimeline
  - Dynamics 전역 행의 변화를 세그먼트로 정규화한 결과이다.
  - 세부 구조는 이후 절에서 정의한다.
  - 즉시 변하는 상수값 또는 구간내 선형 변화를 구현한다.
  - 사용자가 UI의 볼륨바를 통해 조절하는 볼륨과는 별도의 개념으로, 최종 소리 크기는 두 결과를 합쳐 계산해 반영한다. Dynamic로 조절되는 볼륨은 상대 볼륨, 볼륨바를 통해 조절되는 볼륨은 절대 볼륨으로 정의한다.

- trackResults
  - 트랙별 analyzer 이벤트 리스트이다.
  - 명세상 track 간 결합을 허용하지 않으므로, 이벤트 역시 문서 전체 단일 배열보다 track별 결과 묶음으로 보관하는 편이 해석과 부분 갱신에 유리하다.
  - 음표, 글리산도, 잇단음표 그룹 등 트랙 내부에서 형성되는 이벤트는 이 구조 안에 저장한다.
  - trackResults[].events[]는 시간순으로 startTick 오름차순 정렬을 보장한다.
  - 같은 시작 시각의 이벤트가 여러 개일 경우, 기본 점유 이벤트(note, rest, mute)를 먼저 두고, 관계 이벤트(gliss), 구조 이벤트(tupletGroup) 순으로 정렬한다. 같은 종류 안에서는 endTick과 sourceCells 좌표를 기준으로 안정 정렬한다.

- analysisIssues
  - 셀 간 관계를 분석하는 과정에서 발생한 의미 오류나 경고를 담는다.
  - parser 단계에서 발견되지 않는 문맥 의존 오류는 여기서 보고한다.

즉 AnalysisResult는 전역 시간 정보, 트랙별 이벤트 정보, 분석 진단 정보를 하나의 공통 출력 구조로 묶은 결과이다.
AnalysisResult는 renderer, audio generator, application layer가 공통으로 참조하는 공식 분석 결과이며, 조회 최적화용 인덱스나 캐시는 포함하지 않는다.
조회 최적화용 구조는 별도 AnalysisCache 또는 파생 캐시로만 두고, 항상 AnalysisResult로부터 재생성 가능해야 한다.
AnalyzedEvent의 종류에 따른 필드 구조는 6절에서 별도로 정의한다.


## 6. 이벤트리스트에 담겨야 하는 정보

analyzer 이벤트 리스트는 renderer와 audio generator가 공통으로 소비하는 최종 의미 단위여야 한다.
따라서 각 이벤트는 단순히 “어느 셀에서 왔는가”만이 아니라, 실제 시간축에서 어떤 의미를 가지는지와 후속 모듈이 바로 사용할 수 있는 최소한의 해석 결과를 함께 포함해야 한다.

크게 다음과 같이 정리할 수 있다.

 - 소속 trackId
 - 이벤트의 종류 (Note / Rest / Mute / Gliss / TupletGroup)
 - 표시 위치
 - 음의 시작 시점과 지속 길이
 - 실제 발음 음정 정보
 - 렌더링과 오디오에 반영될 효과 정보
 - 원본 셀 정보
 - (오류 발생시) 진단 정보

파서 때와 동일하게 discriminated union 타입을 채택하여 구조 설계를 진행하고자 한다.

서로 다른 종류의 analyzer 이벤트는 시간 범위가 겹칠 수 있다.
특히 NoteEvent와 GlissEvent, NoteEvent와 TupletGroupEvent의 시간 범위 중첩은 정상적인 경우로 본다.

다만 독립적인 발음 구간을 형성하는 이벤트는 NoteEvent뿐이다.
GlissEvent는 별도의 발음원을 생성하지 않고 기존 NoteEvent 구간에 적용되는 연속 음정 이동 관계만 표현하며,
TupletGroupEvent 역시 슬롯 note/rest 이벤트를 묶는 구조 이벤트일 뿐 독자적인 발음을 생성하지 않는다.


## 7. 노트 셀 이벤트리스트

### 7.1. 공통 베이스 정보

모든 Analyzer의 생성 이벤트는 아래와 같은 공통 필드를 가지도록 한다.

```ts
 type AnalyzedEventBase = {
  eventKind: AnalyzedEventKind; 
  trackId: string;
  time: TimeRange;
  sourceCells: SourceCellRef[];
};

type AnalyzedEventKind =
  | "note"
  | "rest"
  | "mute"
  | "gliss"
  | "tupletGroup";

type SourceCellRef = {
  rowId: string;
  col: number;
  slotIndex?: number;
};
```

필드 설명:
- eventKind
    - 이벤트의 종류를 나타내는 구분자이다.
    - 후속 모듈은 이 값을 기준으로 타입 분기를 수행한다.

- trackId
    - 해당 이벤트가 속한 track을 나타낸다.
    - 서로 다른 track의 이벤트는 연결되거나 병합되지 않는다.

- time
    - 이벤트의 시작 시각과 종료 시각을 나타낸다.
    - 일반 셀은 정수 tick 범위를 가지며, tuplet slot은 유리수 tick 범위를 가질 수 있다.

- sourceCells
    - 이 이벤트가 어떤 원본 셀 또는 슬롯들로부터 생성되었는지를 추적하기 위한 정보이다.
    - 디버깅, 선택 동기화, 오류 보고, 부분 갱신에 필요하다.
    - slotIndex는 일반 셀에서는 사용하지 않으며, pletHead 내부의 특정 슬롯을 가리킬 때만 사용한다. 즉 같은 rowId, col을 공유하더라도 서로 다른 tuplet 슬롯은 slotIndex로 구분한다.

### 7.2. 시간 정보 구조

핵심 정보이므로 별도로 분리해 관리한다.

```ts
type TimeFraction = {
  numerator: number;
  denominator: number;
};

type TimeRange = {
  startTick: TimeFraction;
  endTick: TimeFraction;
};
```

설계 원칙:
analyzer 내부 시간 단위는 tick을 기준으로 한다.
일반 셀은 보통 [col, (col + 1)] 범위를 갖는다.
tuplet slot은 분수 tick 범위를 가질 수 있으므로 분수(유리수) 표현을 허용한다.
endTick은 배타적 끝점으로 두는 것이 범위 병합과 겹침 판정에 유리하다.


### 7.3. 최종 발음 위치와 표시 위치 정보

```ts
type FinalDisplayPosition = {
  rowId: string;
  centOffset: number;
};

type FinalSoundPitch = {
  midi: number;
  centOffset: number;
};
```

설명:
- FinalDisplayPosition
    - 렌더러가 어떤 행에 음표를 그릴지 결정할 수 있다.
    - @m에 따른 시각적 보정 정보도 함께 표현할 수 있다.

- FinalSoundPitch
    - audio generator가 실제 발음 주파수를 계산할 수 있다.
    - @p와 @m이 결합된 경우에도 일관되게 표현할 수 있다.


### 7.4. NoteEvent

 실제 발음 구간을 가지는 가장 기본적인 이벤트이다.
 일반 셀, hold로 연결된 셀 묶음. tuplet slot에서 시작된 발음 구간 등이 최종적으로 이 구조로 귀결된다.
 trem과 vibrato 같은 발음 패턴 효과는 NoteEvent의 effect 구간에만 저장하며, GlissEvent는 연속 음정 이동 관계만 표현한다.

 ```ts
type NoteEvent = AnalyzedEventBase & {
  eventKind: "note";
  display: FinalDisplayPosition;
  sound: FinalSoundPitch;
  effects: NoteEffectSegment[];
  glissRole?: GlissAnchorRole | null;
  tuplet?: {
    groupId: string;
    slotIndex: number;
    divNum: number;
  } | null;
};

type NoteEffectSegment = {
  time: TimeRange;
  vib: boolean;
  trem?: TremInfo | null;
};

type TremInfo = {
  division: number;
};

type GlissAnchorRole = {
  glissId: string;
  role: "start" | "mid" | "end";
};
```
NoteEvent가 담아야 하는 핵심 정보는 다음과 같다.

 - 최종 표시 위치
 - 최종 발음 음정
 - 길게 연결된 결과 시간 범위
 - 시간 구간별 vib / trem 효과
 - gliss anchor 역할 여부

특히 중요한 점은 NoteEvent가 단일 셀 하나와 1:1로 대응될 필요는 없다는 점이다. hold가 연결된 여러 셀 또는 슬롯은 analyzer 단계에서 하나의 연속된 NoteEvent로 병합될 수 있다.
또한 vibrato와 trem은 이벤트 전체에 단순 boolean으로 붙는 속성이 아니라, 긴 note 내부의 일부 시간 구간에만 적용될 수 있는 효과이므로 effects 배열 형태로 두는 것이 적절하다.
예를 들어 ["A3", "-", "-", "~", "~", "-"]와 같은 입력은 하나의 NoteEvent로 유지하되, 그 중간 일부 구간만 vib: true인 효과 세그먼트로 표현할 수 있다.


### 7.5. RestEvent

RestEvent는 시간축상 자리를 차지하지만 발음되지 않는 구간을 명시적으로 나타낼 필요가 있을 때 사용한다.
특히 tuplet slot 내부의 빈 슬롯이나, 분석 과정에서 발음 가능한 음으로 확정되지 못했지만 시간 위치는 유지해야 하는 경우를 표현할 수 있다.

```ts
type RestEvent = AnalyzedEventBase & {
  eventKind: "rest";
  display?: FinalDisplayPosition | null;
};
```

RestEvent가 필요한 이유는 다음과 같다.
 - tuplet 내부 빈 슬롯의 시간 점유를 유지할 수 있다.
 - 렌더러가 필요시 쉼표 또는 빈 슬롯성 표시를 할 수 있다.
 - 후속 단계가 “이 구간에는 발음 이벤트가 없다”는 사실을 명시적으로 다룰 수 있다.
 - 단, 일반 빈 칸 전체를 모두 RestEvent로 만들 필요는 없다. 이벤트 리스트는 “의미 있는 셀 또는 슬롯으로부터 유도된 시간 구간”만을 다루는 것이 바람직하다.
 - RestEvent.display는 표시 행을 해석할 수 있는 경우에만 채워진다.
 - 1차 구현에서는 sourceCells 유지 우선순위를 낮게 둘 수 있으며, 필요 시 후속 단계에서 보강한다.


### 7.6. MuteEvent

 MuteEvent는 표시상 텍스트나 시각 요소는 유지하되, 발음 이벤트로는 취급하지 않는 셀을 나타낸다.

 ```ts
type MuteEvent = AnalyzedEventBase & {
  eventKind: "mute";
  display: FinalDisplayPosition;
  text: string;
};
```

이 구조를 두는 이유는 다음과 같다.
 - mute 셀은 화면에 표시될 수 있다.
 - 그러나 오디오 생성기 입장에서는 발음 이벤트가 아니다.
 - 따라서 note와는 구분되는 독립 이벤트 종류로 두는 편이 명확하다.
 - 즉 MuteEvent는 “그려질 수는 있으나 소리 나지는 않는 이벤트”를 표현한다.
 - 1차 구현에서는 sourceCells 유지 우선순위를 낮게 둘 수 있으며, 필요 시 후속 단계에서 보강한다.


### 7.7. GlissEvent

GlissEvent는 두 gliss anchor 사이의 연속 음정 이동 구간을 나타낸다. 이는 NoteEvent를 대체하는 이벤트가 아니라, 기존 발음 구간 사이 또는 발음 구간 내부에 적용되는 연속 음정 이동 관계를 나타내는 별도 이벤트이다.

```ts
type GlissEvent = AnalyzedEventBase & {
  eventKind: "gliss";
  startDisplay: FinalDisplayPosition;
  endDisplay: FinalDisplayPosition;
  startSound: FinalSoundPitch;
  endSound: FinalSoundPitch;
  glissId: string;
  fromKind: "start" | "mid";
  toKind: "mid" | "end";
  startAttach: "attack" | "legato";
  endAttach: "release" | "holdContinue";
};
```
GlissEvent에 포함되어야 하는 핵심 정보는 다음과 같다.
 - 시작점과 끝점의 표시 위치
 - 시작점과 끝점의 실제 발음 음정
 - 시작점과 끝점 anchor의 속성
 - glissId
 - 시간 범위
 - anchor 앞뒤의 연결 방식

 여기서 startAttach와 endAttach는 gliss가 독립된 새 음으로 시작되는지, 아니면 앞뒤 hold 구간과 부드럽게 이어지는지를 표현하기 위한 값이다. 예를 들면 다음과 같다.
    ex1: ["-", "-", "-", "@g(a, start)", ...] -> gliss 시작점 앞에 이미 이어지던 음이 있으므로 startAttach = "legato"가 될 수 있다.
    ex2: ["@g(a, end)", "-", "-", "-"] -> gliss 종착점 뒤에 같은 음이 계속 이어지므로 endAttach = "holdContinue"가 될 수 있다.

 * GlissEvent.sourceCells는 해당 segment를 형성한 양 끝 anchor의 source 참조를 담는다.
 * fromKind와 toKind는 본질적으로 대응되는 NoteEvent.glissRole.role로부터 추론 가능한 값이다. 다만 구현/디버그 편의성을 위해 배치한다.
 * 같은 시간 범위를 공유하는 GlissEvent가 여러 개 존재할 수 있다

 * 한편 GlissEvent는 연속 음정 이동 관계만 표현하며, trem과 vibrato 같은 발음 패턴 효과는 NoteEvent의 effect 구간에만 저장한다.


## 7.8. TupletGroupEvent

tuplet은 슬롯 단위로 분해되지만, 악보의 편집을 위해서는 동시에 하나의 그룹으로서도 관리할 필요가 있다.
예를 들어 renderer는 슬롯 각각의 note뿐 아니라, 그 슬롯들이 하나의 pletHead와 pletExtend로부터 생성된 그룹임을 알아야 한다.

```ts
type TupletGroupEvent = AnalyzedEventBase & {
  eventKind: "tupletGroup";
  groupId: string;
  divNum: number;
  headCell: SourceCellRef;
  extendCells: SourceCellRef[];
  slots: TupletSlotInfo[];
};

type TupletSlotInfo = {
  slotIndex: number;
  parsedKind: "note" | "rest" | "invalid";
};
```

TupletGroupEvent가 필요한 이유는 다음과 같다.
 - 렌더러가 tuplet 묶음 표시를 할 수 있다.
 - analyzer가 슬롯과 그룹 정보를 동시에 유지할 수 있다.
 - 부분 갱신 시 어떤 슬롯들이 하나의 그룹에 속하는지 추적할 수 있다.
즉 tuplet은 "slot 단위의 note/rest 이벤트"와 "그룹 단위의 구조 이벤트"를 동시에 가질 수 있다.

필드 설명:
AnalyzedEventBase.time : pletHead셀과 pletExtend셀의 합으로, 이것이 TupletGroupEvent의 전체 길이가 된다.

groupid는 헤드 셀의 정보를 기반으로 다음과 같이 정의된다.

```ts
groupId = `${trackId}:${headRowId}:${headCol}`
```
각 tuplet 슬롯이 실제로 생성한 note/rest 이벤트는 NoteEvent.tuplet.groupId, NoteEvent.tuplet.slotIndex 또는 대응되는 rest 이벤트의 sourceCells.slotIndex를 통해 역으로 연결할 수 있다.
즉 TupletGroupEvent는 그룹 구조를 나타내고, 실제 발음 및 시간 정보는 슬롯 note/rest 이벤트가 담당한다.


## 8. 전역 셀 분석 결과 (AnalyzedTimeSegment, AnalyzedDynamicsSegment)
 analyzer는 note 이벤트만 생성하는 것이 아니라, 전역 행의 BPM, 박자, step, dynamics 변화를 시간축 세그먼트로 정규화해야 한다.
이 결과는 renderer와 audio generator가 공통으로 사용할 수 있는 문서 전체 시간 맵 역할을 한다.

입력은 다음과 같이 주어진다.
  - ParsedGlobalCell
  - globalCellsByKindAndCol
  - globalCellsInColOrder
  - rowById
  - columnCount

analyzer는 이를 직접 후속 모듈에 노출하지 않고, 일정한 전역 상태가 유지되는 구간 단위로 재구성하여 AnalyzedTimeSegment[], AnalyzedDynamicsSegment[]를 생성한다.
두 생성물은 다음 용도로 사용된다.
  - audio generator에서는 노트 이벤트의 명확한 시간 길이와 상대 볼륨 계산에 사용된다.
  - renderer에서는 BPM 변화 세로 경계선, Dynamics 볼륨 가이드 가로선 렌더링에 사용된다.


### 8.1. 처리 절차

1. 입력 정규화
  - analyzer는 `globalCellsInColOrder`를 기준으로 같은 kind의 전역 셀을 열 오름차순으로 순회한다.
  - `parsedCell.kind === "invalid"`인 전역 셀은 세그먼트를 생성하지 않는다.
  - invalid 전역 셀은 경계 열 계산에서 제외한다.
  - 이후 절차는 유효한 `ParsedGlobalCell`만을 대상으로 수행한다.

2. 초기 상태 결정
  - 전역 세그먼트 분석은 숨겨진 런타임 상수값이 아니라 문서에 저장된 전역 셀을 기준으로 시작한다.
  - 각 전역 kind는 `col = 0`에 유효한 시작 셀을 하나 가져야 하며, analyzer는 이를 해당 kind의 초기 상태로 사용한다.
  - 따라서 첫 번째 세그먼트는 항상 `col = 0`에서 시작한다. validator가 이 조건을 보장한다.
  - 이 규칙을 통해 전역 초기 상태가 파일 안에 명시적으로 드러나도록 하고, analyzer가 별도의 암묵 기본값에 의존하지 않도록 한다.

3. 경계 열 집합 생성
  - timing timeline은 `bpm`, `beatsPerBar`, `stepsPerBeat` 스트림의 경계 열 합집합을 기준으로 분할한다.
  - dynamics timeline은 `dynamics` 스트림의 경계 열만을 기준으로 분할한다.
  - 각 세그먼트의 시간 범위는 `현재 경계 열 이상 다음 경계 열 미만`으로 정의한다.
  - 마지막 세그먼트의 끝은 항상 `columnCount`이다.

4. instant 전역값 해석
  - `beatsPerBar`, `stepsPerBeat`는 항상 instant 값으로 해석한다.
  - `bpm`, `dynamics`에서도 ramp가 성립하지 않는 셀은 instant 값으로 해석한다.
  - instant 세그먼트는 시작 열부터 다음 경계 열 직전까지 일정한 상수값을 유지한다.

5. ramp 해석 규칙
  - `none`은 해당 열부터 시작하는 상수 구간을 의미한다.
  - `start`는 오른쪽의 가장 가까운 `end` 또는 `endStart`와 연결되는 선형 변화 시작점 후보이다.
  - `end`는 왼쪽의 가장 가까운 열린 `start` 또는 `endStart`를 닫는 종료점 후보이다.
  - `endStart`는 같은 열에서 이전 선형 변화를 끝내고, 동시에 새로운 선형 변화를 다시 시작하는 경계점 후보이다.
  - `bpm`, `dynamics`에서 선형 구간이 성립하면 시작 셀 값과 종료 셀 값을 사용하여 `linear` 세그먼트를 생성한다.
  - 짝이 지어지지 않은 `start`, `end`, `endStart`는 미완성 편집 상태로 간주하며, 오류나 경고를 보고하지 않고 instant 값으로 강등한다.

6. sourceCells와 값 결정
  - dynamics 세그먼트의 `sourceCells`는 dynamics 스트림만을 기준으로 계산한다.
  - dynamics가 instant이면 `sourceCells`에는 해당 값을 시작시킨 dynamics 셀 1개만 포함한다.
  - dynamics가 linear이면 `sourceCells`에는 시작 dynamics 셀과 종료 dynamics 셀을 함께 포함한다.

  - timing 세그먼트의 `startBpm`, `endBpm`은 BPM 스트림 해석 결과를 따른다.
  - BPM이 instant이면 timing 세그먼트에는 현재 BPM 값을 시작시킨 셀 1개만 포함한다.
  - BPM이 linear이면 timing 세그먼트에는 BPM 시작 셀과 BPM 종료 셀을 함께 포함한다.
  
  - `beatsPerBar`, `stepsPerBeat`는 timing 세그먼트 내부에서 항상 상수값이며, 각 세그먼트 시작 시점에 유효한 셀 1개씩을 source로 가진다.
  - 따라서 timing 세그먼트의 `sourceCells`는 `bpm`, `beatsPerBar`, `stepsPerBeat` 세 스트림 중 해당 구간의 상태 결정에 실제로 기여한 셀들을 합쳐 구성한다.
  - `sourceCells` 배열은 `bpm`, `beatsPerBar`, `stepsPerBeat` 순으로 정렬하고, 같은 kind 안에서는 `col` 오름차순으로 정렬한다.
  - instant 세그먼트에서는 시작값과 끝값이 동일하다.

7. 복구와 진행 원칙
  - 일부 전역 셀이 invalid이거나 ramp 연결에 실패하더라도, 같은 kind의 나머지 유효 셀 분석은 계속 진행한다.
  - 열린 ramp가 끝까지 닫히지 않으면 선형 세그먼트를 만들지 않고 instant 세그먼트로 처리한다.
  - analyzer는 가능한 한 많은 유효 세그먼트를 복구하여 생성하되, 실제로 시간축을 확정할 수 없는 경우에만 issue를 생성한다.

### 8.2. 세그먼트 생성

```ts
type AnalyzedTimeSegment = {
  time: TimeRange;
  startBpm: number;
  endBpm: number;
  bpmCurve: "instant" | "linear";
  beatsPerBar: number;
  stepsPerBeat: number;
  sourceCells: SourceCellRef[];
};

type AnalyzedDynamicsSegment = {
  time: TimeRange;
  startValue: number;
  endValue: number;
  curve: "instant" | "linear";
  sourceCells: SourceCellRef[];
};
```

AnalyzedTimeSegment 필드 설명:
- time
  - 하나의 timing 해석 규칙이 유지되는 시간 구간이다.
  - startTick 이상 endTick 미만 범위에서 아래 전역 값들이 유효함을 의미한다.
- startBpm
  - 시작 시점의 BPM 값이다.
- endBpm
  - 종료 시점의 BPM값이다.
- bpmCurve
  - 변화 양상이 즉시(instant)인지, 선형(linear) 보간인지를 나타낸다.
  - instant이면 startBpm === endBpm이다.
  - BPM만 변화하는 값이며, "beatsPerBar", "stepsPerBeat"는 구간 내 상수값이다.
- beatsPerBar
  - 이 구간에서의 박자 수이다.
- stepsPerBeat
  - 이 구간에서 한 박을 몇 개의 step으로 나누는지 나타낸다.
- sourceCells
  - 이 세그먼트를 형성하는 데 관여한 전역 행 원본 좌표를 추적하기 위한 정보이다.
  - 전역 행 역시 rowId와 col을 가지므로, 일반 셀과 동일한 SourceCellRef 구조로 추적할 수 있다.


AnalyzedDynamicsSegment 필드 설명:
- time
  - 해당 전역 상태가 유지되는 시간 구간이다.
- startValue
  - 시작 시점의 상대 볼륨이다.
- endValue
  - 종료 시점의 상대 볼륨이다.
- curve
  - 변화 양상이 즉시(instant)인지, 선형(linear) 보간인지를 나타낸다.
  - instant이면 startValue === endValue이다.
- sourceCells
  - 이 세그먼트를 형성하는 데 관여한 전역 행 원본 좌표를 추적하기 위한 정보이다.
  - 전역 행 역시 rowId와 col을 가지므로, 일반 셀과 동일한 SourceCellRef 구조로 추적할 수 있다.

설계 원칙:
- ParsedGlobalCell은 특정 열에서 전역 상태가 어떻게 바뀌는지를 나타낸다.
- analyzer는 이를 순회하여 상태가 일정한 구간들로 재구성한다.
- renderer와 audio generator는 원문이나 ParsedGlobalCell이 아니라 AnalyzedTimeSegment, AnalyzedDynamicsSegment만 참조하는 것을 원칙으로 한다.


## 9. 진단 내용 정리

analyzer는 가능한 한 많은 유효 이벤트를 생성하되, 문맥 의존 분석 과정에서 발견된 문제는 별도의 진단 구조로 함께 보고해야 한다.
parser가 문자열 자체의 문법 오류를 다룬다면, analyzer는 셀 간 관계와 시간축 해석 과정에서 드러나는 의미 오류와 경고를 다룬다.

```ts
type AnalyzedIssue = {
  level: "warning" | "error";
  code: AnalyzedIssueCode;
  message: string;
  trackId?: string | null;
  sourceCells: SourceCellRef[];
};

type AnalyzedIssueCode =
  | "tuplet_slot_row_not_found"
  | "tuplet_slot_invalid_pitch"
  | "global_event_conflict"
  | "timing_segment_invalid"
  | "dynamics_segment_invalid";
```

필드 설명:
- level
  - 오류의 심각성에 따라 "warning"과 "error"로 분리해 관리한다.
  - "error"는 이벤트 전체를 되살릴 수 없는 경우로, 해당 이벤트를 삭제한다.
  - "warning"은 이벤트에서 오류가 생긴 부분을 보정하거나 대체해서 사용할 수 있다.
  - gliss anchor의 연결 대상 부재, pletHead 단독 존재와 같은 상태는 셀 단위 편집 과정에서 자연스럽게 발생할 수 있는 미완성 상태이므로, analyzer의 정식 오류 또는 경고로 기본 보고하지 않는다. 이러한 상태는 필요할 경우 editor 전용 표시 상태로 별도 관리한다.

- code
  - 문제의 종류를 식별하는 안정된 진단 코드이다.
  - UI 표시, 테스트, 로그 분석에서 문자열 메시지보다 우선적으로 사용한다.

- message
  - 사람이 읽을 수 있는 설명 문구이다.
  - 디버깅과 UI 경고 표시를 돕기 위한 필드이다.

- trackId
  - 문제가 특정 track 내부에서 발생한 경우 그 track을 가리킨다.
  - 전역 시간/셈여림 세그먼트 해석에서 발생한 문제는 null 또는 생략을 허용한다.

- sourceCells
  - 해당 문제를 유발한 원본 셀 또는 전역 행 좌표를 나타낸다.
  - 선택 동기화, 경고 표시, 디버그 로그, 테스트 검증에 필요하다.


warning과 error의 구분 기준:
analyzer 진단은 단순히 오류를 나열하는 것이 아니라, 해당 문제를 만났을 때 분석을 어느 수준까지 계속할 수 있는지도 함께 표현해야 한다.

구분 기준은 다음과 같다.

* warning
  - 일부 정보가 유실되거나 대체되더라도, 나머지 이벤트를 계속 생성할 수 있는 경우
  - 예: @n(midi_num)가 실제 행으로 해석되지 않아 해당 슬롯을 rest로 대체한 경우

* error
  - 해당 해석 결과를 정상적으로 확정할 수 없어서, 관련 이벤트 생성이 중단되거나 무효 처리되는 경우
  - 예: timing segment가 모순되어 시간 맵 자체를 확정할 수 없는 경우
  - 예: 전역 셀 해석 결과가 서로 충돌하여 구간 상태를 만들 수 없는 경우

즉 analyzer는 가능한 한 부분 복구를 우선하되, 복구가 가능한 문제는 warning, 복구가 불가능한 문제는 error로 분리한다.


## 10. 부분 갱신 및 재분석 범위 규칙

회귀 코드는 긴 악보에서도 편집 응답성을 유지해야 하므로, 셀 하나의 수정이 있을 때마다 문서 전체를 다시 분석하는 방식은 가능한 한 피하는 것이 좋다.
따라서 analyzer는 `AnalysisResult` 전체를 항상 처음부터 다시 만드는 구조가 아니라, 영향받은 범위를 중심으로 부분 재분석할 수 있도록 설계하는 것이 바람직하다.

다만 부분 갱신은 “모든 경우를 최소 범위로 완벽히 자르기”보다,  
정확성을 우선하면서도 충분히 작게 제한된 범위를 다시 분석하는 전략을 채택한다.

### 10.1. 기본 원칙

- 부분 갱신의 기본 단위는 열 범위와 track 범위이다.
- 일반적인 note, mute, rest 수정은 우선 “수정된 셀의 열”을 기본 재분석 범위로 삼는다.
- 이후 hold, gliss, tuplet, 전역 시간 구조처럼 좌우 문맥을 참조하는 규칙이 있을 경우, analyzer가 재분석 범위를 양옆으로 확장한다.
- 부분 갱신 결과는 기존 `AnalysisResult`를 직접 수선하는 방식보다, 영향을 받은 범위의 이벤트를 제거하고 해당 범위를 다시 분석하여 재삽입하는 방식을 원칙으로 한다.
- 정확한 최소 범위를 계산하기 어려운 경우에는, 과도하게 좁은 범위보다 다소 넓더라도 안전한 범위를 택한다.

즉 회귀 코드의 부분 갱신은 “정확한 최소 범위”보다 “충분히 작은 안전 범위”를 추구한다.

### 10.2. 기본 재분석 범위

기본적으로 셀 하나가 수정되면 아래 범위를 우선 재분석 후보로 잡는다.

- 수정된 `trackId`
- 수정된 `col`
- 해당 열의 직접 인접 열

즉 기본 시작점은 아래와 같이 잡을 수 있다.

```ts
type AnalyzePartialRequest = {
  trackId: string;
  colStart: number;
  colEnd: number;
};

type AnalyzeGlobalPartialRequest = {
  colStart: number;
  colEnd: number;
};
```

단일 셀 수정의 기본 범위는 아래처럼 두는 것이 안전하다.

- `colStart = editedCol - 1`
- `colEnd = editedCol + 1`

이 정도의 좌우 1칸 패딩은 hold의 시작/종료, 단순한 gliss anchor 연결 변화, 음의 시작점 변화에 대응하기 위한 최소 범위로 볼 수 있다.

이는 이세계 코드에서 사용하던 “직접 수정 열만 다시 보면 부족하고, 좌우 문맥까지 함께 다시 봐야 한다”는 경험칙을 계승한 것이다.
다만 회귀 코드는 `cellsByTrackAndCol`, `groupId`, `trackResults`를 갖는 정규화 구조를 사용하므로, 전체 행렬 스캔이 아니라 명시적인 열 범위 재분석으로 정리하는 편이 더 적절하다.

### 10.3. hold에 의한 범위 확장

hold는 좌우 인접 음과 연결되므로, 수정된 열만 다시 보면 기존 note event의 시작점 또는 종료점이 잘못 유지될 수 있다.
따라서 수정된 범위 안에 hold 후보가 있거나, 수정된 결과로 hold 연결 성립 여부가 바뀔 가능성이 있는 경우 재분석 범위를 좌우로 확장해야 한다.

확장 원칙은 다음과 같다.

- 왼쪽으로는 hold 연결이 끊어질 때까지 확장한다.
- 오른쪽으로는 hold 연결이 끊어질 때까지 확장한다.
- 연결 판정 기준은 본 문서 5.1에서 정의한 `FinalSoundPitch`, `FinalDisplayPosition`, `holdKind` 규칙을 따른다.
- `"-"`와 `"~"`는 동일한 hold 연결 규칙을 따르되, effect 구간 여부만 달라진다.

즉 hold 영향 범위는 “같은 note event로 병합될 수 있는 연속 구간” 전체라고 볼 수 있다.

### 10.4. gliss에 의한 범위 확장

gliss는 빈칸을 건너뛴 탐색을 허용하므로, 기본 좌우 1칸 재분석만으로는 충분하지 않다.
수정된 셀 또는 그 인접 셀에 gliss anchor가 존재하거나, gliss 연결 상대가 바뀔 가능성이 있는 경우 아래 규칙으로 확장한다.

- 같은 `trackId` 내에서 같은 `glissId`를 가진 anchor들을 탐색한다.
- 수정된 anchor가 기존 segment의 시작점 또는 끝점이었다면, 그 segment 전체 범위를 재분석한다.
- 수정된 anchor가 `"start"`, `"mid"`, `"end"` 중 어느 위치에 있든, 인접 anchor 쌍으로 생성되는 모든 `GlissEvent`를 다시 계산한다.
- gliss는 전체 체인을 하나로 관리하지 않고 인접 anchor 쌍 단위로 관리하므로, 재분석 범위 역시 “해당 anchor가 관여하는 인접 segment들”만 다시 계산하면 된다.

이 방식은 이세계 코드에서 사용하던 “왼쪽 anchor 탐색 기반의 연결 규칙”의 실용성을 계승하되,  
회귀 코드에서는 이를 명시적 `glissId`와 `GlissEvent` segment 단위로 제한함으로써 증분 분석 비용을 줄이는 방향이다.

### 10.5. tuplet에 의한 범위 확장

tuplet은 `pletHead`와 연속된 `pletExtend`의 길이에 의해 슬롯 시간 구조 전체가 결정되므로,  
head 또는 extend 중 하나라도 수정되면 해당 그룹 전체를 다시 계산해야 한다.

재분석 규칙은 다음과 같다.

- 수정된 셀이 `pletHead` 또는 `pletExtend`이면, 같은 `trackId`, 같은 head row에서 형성되는 `TupletGroupEvent` 전체를 다시 찾는다.
- 해당 group의 `AnalyzedEventBase.time` 범위를 재분석 범위로 사용한다.
- 이 그룹에 속한 슬롯 note/rest 이벤트는 개별 수정하지 않고, group과 함께 일괄 제거 후 재생성한다.
- tuplet의 첫 슬롯과 마지막 슬롯은 바깥 일반 셀 또는 다른 tuplet 슬롯과 hold 연결될 수 있으므로, group 범위 양옆 1칸을 추가 패딩하는 것이 안전하다.

즉 tuplet 부분 갱신의 핵심은 “슬롯 하나를 수선”하는 것이 아니라,  
관련 `TupletGroupEvent`와 그에 종속된 슬롯 이벤트들을 묶어서 다시 생성하는 것이다.

### 10.6. 전역 구조 수정 시 재분석

전역 셀 수정은 track 내부 note event 자체보다 timing/dynamics 해석에 먼저 영향을 준다.
따라서 전역 셀 변경은 일반 note 수정과는 다른 규칙으로 다루는 것이 좋다.

구분 원칙은 다음과 같다.

- BPM, beatsPerBar, stepsPerBeat 수정
  - `timingTimeline`을 다시 계산한다.
  - note/gliss/tuplet 이벤트의 구조 자체는 바뀌지 않을 수 있으므로, 필요시 이벤트 리스트는 재사용 가능하다.
  - 다만 playback 캐시, tick -> ms 변환 캐시는 반드시 무효화한다.

- dynamics 수정
  - `dynamicsTimeline`을 다시 계산한다.
  - analyzer 이벤트 구조 자체는 바뀌지 않을 수 있다.
  - 오디오 관련 후속 캐시는 무효화한다.

즉 전역 행 수정은 항상 모든 track 이벤트를 다시 분석해야 하는 것은 아니며,  
시간 맵 또는 제어값 타임라인만 재생성하고 트랙 이벤트는 유지하는 전략이 가능하다.

### 10.7. 이벤트 병합 방식

부분 재분석 결과를 기존 `AnalysisResult`에 반영할 때는 아래 원칙을 따른다.

- 영향 범위와 겹치는 기존 이벤트를 제거한다.
- 같은 범위에 대해 새로 분석한 이벤트를 삽입한다.
- 삽입 후에는 5.2에서 정의한 정렬 규칙을 다시 만족하도록 안정 정렬한다.
- `TupletGroupEvent`, `GlissEvent`, `NoteEvent`처럼 서로 시간이 겹칠 수 있는 이벤트는 종류별로 분리 삭제하지 말고, “범위 중첩 여부”를 기준으로 함께 교체하는 편이 안전하다.

즉 부분 갱신은 “기존 이벤트 객체를 부분 수정”하기보다 “영향 범위 교체” 방식으로 구현하는 것이 적절하다.

### 10.8. 런타임 파생 캐시

부분 갱신 비용을 더 줄이기 위해, 공식 출력 구조인 `AnalysisResult` 외에 아래와 같은 파생 캐시를 둘 수 있다.

- `eventsByStartCol`
- `trackResultByTrackId`
- `tupletGroupById`
- `glissEventsById`

이 캐시들은 저장 구조가 아니라 런타임 파생 구조로 간주한다.
즉 진실 원천은 여전히 `AnalysisResult`이며, 캐시는 조회 성능과 부분 갱신 범위 계산을 돕는 보조 구조일 뿐이다.

이 원칙을 지키면 캐시 무효화가 단순해지고, 공식 데이터 구조와 조회 최적화 구조가 뒤섞이지 않는다.

### 10.8.1. sourceCells 구현 우선순위

문서 규범 차원에서는 주요 이벤트와 세그먼트가 `sourceCells`를 유지하는 방향을 유지한다. 다만 실제 구현은 단계적으로 진행하는 편이 비용상 합리적이다.

1차 구현에서 우선 유지할 대상을 아래처럼 둔다.

- `NoteEvent`
- `GlissEvent`
- `TupletGroupEvent`
- `AnalyzedTimeSegment`
- `AnalyzedDynamicsSegment`
- `AnalyzedIssue`

반면 아래 항목은 1차 구현에서 선택 대상으로 둘 수 있다.

- `RestEvent`
- `MuteEvent`

즉 문서의 최종 목표는 유지하되, 실제 구현 로드맵에서는 디버깅과 부분 갱신에 더 직접적으로 기여하는 구조부터 우선 적용하는 편이 적절하다.

### 10.9. 구현 모듈 분리 방침

부분 갱신 로직은 범위 계산, 이벤트 제거/삽입, 캐시 무효화, 전역 구조 재생성 여부 판단까지 포함하므로 구현 규모가 커질 가능성이 높다.
따라서 처음부터 별도 모듈로 분리하는 것이 바람직하다.

예를 들면 다음과 같은 구성이 가능하다.

- `analyze_full.ts`
- `analyze_partial.ts`
- `analyze_timing.ts`
- `analyze_dynamics.ts`
- `analyze_cache.ts`

이때 문서도 반드시 별도로 분리해야 하는 것은 아니다.
현재 `1.7`은 analyzer 출력 구조와 부분 갱신 원칙을 정의하는 상위 설계 문서로 유지하고,
구현 규모가 실제로 커질 경우에만 후속 문서로 `1.8. 부분 갱신 규칙` 또는 `1.8. analyzer 증분 갱신 설계` 같은 별도 문서를 분리하는 편이 적절하다.

즉 현 단계에서는 `1.7` 안에 원칙을 유지하고, 세부 구현이 복잡해지는 시점에만 하위 문서를 분리하는 전략이 가장 실용적이다.


## 11. 설계 해설

본 절은 `1.7`의 규범적 명세를 다시 반복하기보다,  
설계 과정에서 예상할 수 있는 의문점에 대해 “왜 이러한 판단을 했는가”를 설명하기 위한 해설 절이다.
즉 보고서 관점에서의 설계 근거와 판단 이유를 보충하는 역할을 한다.

### 11.1. 왜 gliss를 `~`처럼 NoteEvent 내부 속성으로 두지 않았는가?

`~`는 기본적으로 하나의 연속된 발음 구간 내부에서 vibrato 효과를 부여하는 속성이다.
즉 “같은 note event 내부의 구간 효과”로 이해하는 것이 자연스럽다.

반면 gliss는 두 지점 사이의 연속 음정 이동 관계를 나타낸다.
이는 단일 note 내부 속성이라기보다, 두 anchor 사이의 관계를 표현하는 별도 구조로 보는 것이 더 적절하다.

gliss를 단순히 `NoteEvent` 내부 속성으로만 두지 않은 이유는 다음과 같다.

- gliss는 시작점과 끝점이 필요하다.
- hold와 결합될 경우, 새 음의 시작인지 기존 음의 연장 위의 이동인지 구분이 필요하다.
- renderer는 이를 선형 경사선 또는 연결선으로 그려야 한다.
- audio generator는 이를 시간에 따른 pitch ramp로 해석해야 한다.

즉 gliss는 하나의 note가 “가지고 있는 성질”이라기보다,  
둘 이상의 note 또는 anchor 사이에 성립하는 관계이므로 별도의 `GlissEvent`로 분리하는 것이 더 명확하다.

### 11.2. 왜 발음 이벤트와 구조 이벤트를 분리했는가?

회귀 코드의 analyzer 출력에는 실제 소리가 나는 이벤트와, 소리는 나지 않지만 구조를 설명하는 이벤트가 함께 존재한다.

예를 들면 다음과 같다.

- `NoteEvent`
  - 실제 발음 구간
- `RestEvent`
  - 발음되지 않지만 시간 구간을 차지하는 슬롯
- `GlissEvent`
  - 음정 이동 관계
- `TupletGroupEvent`
  - 잇단음표 컨테이너 구조

이들을 모두 하나의 단일 이벤트 의미로 섞으면, renderer와 audio generator가 각자 필요한 정보를 다시 분리 해석해야 한다.
따라서 회귀 코드에서는 다음 원칙을 채택하였다.

- 발음의 본체는 `NoteEvent`
- 관계는 `GlissEvent`
- 구조는 `TupletGroupEvent`

이렇게 구분하면 각 이벤트의 책임이 분명해지고, 부분 갱신 시 어떤 이벤트를 다시 계산해야 하는지도 더 명확해진다.

### 11.3. 왜 TupletGroupEvent와 슬롯 note/rest 이벤트를 동시에 두는가?

잇단음표는 두 층위에서 동시에 해석해야 한다.

첫째, 실제로는 각 슬롯이 시간축 위의 발음 또는 쉼 구간을 형성한다.
둘째, 동시에 이 슬롯들은 하나의 `pletHead`와 `pletExtend`에서 비롯된 그룹 구조에 속한다.

만약 슬롯 이벤트만 남기면, renderer는 tuplet 묶음 자체를 표시하기 어렵다. 에딧 모드 UI상 사용자가 이 묶음을 삭제 + 연장하는 방식으로 셋잇단의 수정을 실시해야 하기에 이는 치명적이다.

반대로 그룹만 남기면 audio generator와 renderer가 슬롯별 발음 정보를 다시 분해해야 한다.

따라서 회귀 코드에서는 다음과 같이 역할을 나누었다.

- 실제 시간과 발음은 슬롯 note/rest 이벤트가 담당
- 그룹의 길이와 소속 구조는 `TupletGroupEvent`가 담당

즉 tuplet은 “발음 단위”와 “구조 단위”를 동시에 가지므로, 둘을 분리하는 것이 오히려 중복이 아니라 책임 분리라고 볼 수 있다.

### 11.4. 왜 timing timeline과 dynamics timeline을 분리했는가?

처음에는 BPM, 박자, step, dynamics를 하나의 전역 시간 세그먼트 구조에 함께 넣는 방식도 가능해 보였다.
그러나 실제로는 두 정보의 성격이 달라 곧 수정하였다.

- BPM, 박자, step
  - 시간 해석 규칙
  - `tick -> ms` 변환에 직접 관여
- dynamics
  - 시간축 위에 적용되는 연속 제어값
  - 최종 음량 계산에 사용

특히 dynamics는 향후 선형 변화, 램프, 오디오 볼륨 자동화와 연결될 가능성이 크다.
따라서 이를 timing과 같은 구조에 묶기보다 별도 `dynamicsTimeline`으로 분리하는 편이 확장성과 책임 분리 측면에서 더 적절하다고 판단하였다.

### 11.5. 왜 편집 중 미완성 상태를 AnalyzedIssue로 보지 않았는가?

회귀 코드의 UI는 셀 단위 편집을 전제로 하므로, 사용자가 하나의 기능을 완성하기 전까지는 중간 상태가 자연스럽게 존재할 수 있다.

예를 들면 다음과 같다.

- gliss start anchor만 먼저 입력된 상태
- `pletHead`만 입력되고 아직 extend가 추가되지 않은 상태

이러한 상태를 analyzer의 공식 `warning` 또는 `error`로 처리하면,
실제로는 정상 편집 과정임에도 불필요한 오류 출력이 계속 발생하게 된다.
이는 UX와 테스트 모두를 불안정하게 만든다.

따라서 회귀 코드에서는 다음과 같이 구분하였다.

- 실제 의미 해석 실패
  - `AnalyzedIssue`
- 편집 중 미완성 상태
  - editor 전용 상태

이 구분을 통해 analyzer는 최종 의미 구조에 집중하고, UI는 편집 과정의 불완전성을 별도로 다룰 수 있게 된다.

### 11.6. 왜 부분 갱신을 처음부터 고려했는가?

회귀 코드는 긴 악보와 빈번한 편집 상황을 모두 고려해야 한다.
따라서 설계 단계에서부터 full rebuild만 상정하면, 이후 partial analysis를 붙이는 과정에서 자료구조 전체를 다시 뜯어고쳐야 할 가능성이 높다.

특히 아래 구조는 부분 갱신을 처음부터 염두에 두고 정한 것이다.

- `cellsByTrackAndCol`
- `trackResults`
- `groupId`
- `glissId`
- `timingTimeline`, `dynamicsTimeline`

즉 부분 갱신은 “나중에 최적화로 추가하는 부가 기능”이 아니라,
처음부터 analyzer 출력 구조와 모듈 경계를 정할 때 같이 고려해야 하는 핵심 요구사항으로 보았다.

### 11.7. 왜 `AnalysisResult`를 단일 이벤트 배열이 아니라 track 단위로 나누었는가?

문서 전체 이벤트를 하나의 배열에 모두 넣는 방식은 직관적으로 단순해 보일 수 있다.
그러나 회귀 코드에서는 track은 레이어의 개념으로 서로 다른 track 간의 결합을 허용하지 않으므로, 이벤트 역시 track 경계를 유지하는 것이 더 자연스럽다.

`trackResults` 구조를 채택한 이유는 다음과 같다.

- 분석 책임의 경계가 명확하다.
- 부분 갱신 범위를 track 단위로 제한하기 쉽다.
- renderer와 audio generator가 활성 track만 선택해 소비하기 쉽다.
- 이후 캐시 구조(`trackResultByTrackId`)와 자연스럽게 연결된다.

즉 단일 배열보다 `trackResults`가 회귀 코드의 실제 의미 구조와 더 잘 맞는다.

### 11.8. 왜 전역 초기 상태를 숨은 기본값이 아니라 `col = 0` 시작 셀로 고정했는가?

초기에는 BPM, beatsPerBar, stepsPerBeat, dynamics의 시작값을 런타임 상수로 두는 방식도 생각할 수 있다.
그러나 이 방식은 파일만 보아서는 실제 초기 상태를 알 수 없고, import/export나 디버깅 시에도 문서 바깥의 암묵 지식이 필요해진다.

따라서 회귀 코드에서는 각 전역 kind가 `col = 0`에 유효한 시작 셀을 반드시 가지도록 정리하였다.

이 판단의 장점은 다음과 같다.

- 파일 자체가 초기 상태를 직접 드러낸다.
- analyzer가 숨은 기본값에 의존하지 않는다.
- import/export 시 규칙이 단순해진다.
- 특정 세그먼트의 원인을 `sourceCells`로 추적하기 쉬워진다.

즉 전역 초기 상태는 런타임 내부 상수보다 저장 구조 안에 명시적으로 존재하는 편이 더 안정적이라고 판단하였다.

### 11.9. 왜 JSON이 정렬되어 있어도 `ParsedScoreDocument`를 별도로 두는가?

저장 단계에서 JSON을 정렬하는 것은 diff 안정성과 사람이 읽기 쉬운 형태를 유지하기 위한 규칙이다.
그러나 analyzer가 실제로 필요로 하는 것은 “정렬된 원본 문자열”이 아니라 “파싱이 완료된 문서 상태”이다.

따라서 회귀 코드에서는 `ParsedScoreDocument`를 단순 정렬 자료구조가 아니라,
parser가 `rawText`를 `ParsedCell`, `ParsedGlobalCell`로 변환한 뒤 analyzer에 넘기는 런타임 문서 상태로 본다.

이 구조를 두면 다음과 같은 장점이 있다.

- analyzer가 rawText를 다시 해석하지 않아도 된다.
- invalid 셀을 포함한 파싱 결과를 그대로 보존할 수 있다.
- 부분 재파싱 후 해당 범위의 결과만 교체할 수 있다.
- parser와 analyzer의 입력/출력 경계를 테스트하기 쉬워진다.

즉 JSON 정렬 규칙과 `ParsedScoreDocument`의 존재 이유는 서로 다르며, 둘은 대체 관계가 아니라 역할 분담 관계라고 보는 편이 적절하다.

### 11.10. 왜 note 셀 분석 입력과 전역 셀 분석 입력을 분리했는가?

겉으로 보기에는 note 셀과 전역 셀 모두 `rowId`, `col`, parsed 결과를 가지므로 하나의 공통 입력 타입으로 합칠 수도 있어 보인다.
그러나 실제 analyzer가 참조하는 문맥과 생성하는 산출물은 크게 다르다.

- note 셀 분석은 `trackId`, 이웃 note 셀, hold/gliss/tuplet 문맥을 사용한다.
- 전역 셀 분석은 kind별 정렬 스트림, ramp 연결, timeline 경계를 사용한다.

따라서 회귀 코드에서는 상위 개념 차원에서는 모두 “analyzer가 소비하는 셀 입력”이라고 볼 수 있더라도,
실제 설계와 구현에서는 `AnalyzerNoteCellInput`, `AnalyzerGlobalCellInput`으로 분리하는 편이 더 안전하다고 판단하였다.

이렇게 하면 다음과 같은 장점이 있다.

- 함수 책임이 명확해진다.
- optional 필드 남용을 피할 수 있다.
- note/global 분석 경로가 뒤섞이지 않는다.
- 부분 갱신 범위 계산도 각각의 성격에 맞게 분리할 수 있다.

### 11.11. 왜 `sourceCells`와 `AnalyzedIssue`를 끝까지 유지하는가?

analyzer의 역할은 단순히 “정답 이벤트를 계산하는 것”에서 끝나지 않는다.
실제 편집기와 디버깅 환경에서는 “왜 이런 결과가 나왔는가”를 역으로 추적할 수 있어야 한다.

이를 위해 회귀 코드에서는 주요 이벤트와 세그먼트에 `sourceCells`를 유지하고,
문맥 의존 오류는 `AnalyzedIssue` 구조로 별도 보존하도록 하였다.

이 판단의 이유는 다음과 같다.

- 편집기에서 원인 셀과 결과 이벤트를 선택 동기화할 수 있다.
- 부분 갱신 시 어떤 범위를 무효화해야 하는지 계산하기 쉬워진다.
- 테스트에서 단순 값 비교를 넘어서 원인 좌표까지 검증할 수 있다.
- 디버깅 시 “결과가 틀렸다”에서 멈추지 않고 “어느 셀 해석이 원인이었는가”까지 추적할 수 있다.

즉 `sourceCells`와 `AnalyzedIssue`는 부가 정보가 아니라,
복잡한 analyzer를 실제로 운영 가능한 구조로 만들기 위한 핵심 설계 장치라고 볼 수 있다.
