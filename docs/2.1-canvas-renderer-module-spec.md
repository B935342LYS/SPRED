# Canvas renderer module 명세

## 1. 문서 목적

본 문서는 `회귀 코드`의 canvas renderer 구현을 위해 필요한 파일 구조, 타입 경계, 모듈 책임을 정의한다.

`1.9-mvp-analyzer-renderer-ui-spec.md`는 analyzer 결과까지 연결된 MVP renderer의 장기 입력 범위를 정의하고, `2.0-ui-mvp-spec.md`는 UI 화면과 canvas layer 구조를 정의한다.
본 문서는 그중 실제 TypeScript renderer 모듈을 어떤 파일과 타입으로 나눌지 고정한다.

첫 구현 목표는 다음이다.

- app/controller가 `RuntimeDocument`에서 추출한 `CanvasRenderInput`을 읽어 실제 악보 grid를 canvas에 표시한다.
- layout label 영역과 score canvas 영역이 동일한 row/column 좌표계를 공유하게 한다.
- 이후 `AnalysisResult` 기반 note event, marker, playback scroll, virtualization을 추가할 수 있는 입력 경계를 만든다.

## 2. 문서 우선순위

본 문서는 `2.0-ui-mvp-spec.md`의 후속 구현 명세이다.

우선순위는 다음과 같이 본다.

1. 저장 형식과 `RuntimeDocument` 타입은 `1.3`, `1.8`, 현재 `core/score` 구현을 따른다.
2. analyzer 결과 기반 renderer의 장기 입력 방향은 `1.9`를 따른다.
3. UI 영역과 canvas layer 구분은 `2.0`을 따른다.
4. 본 문서는 renderer 내부 파일 구조와 단계별 입력 경계를 구체화한다.

본 문서가 상위 활성 명세와 충돌하면 상위 활성 명세를 우선하고, 본 문서는 구현 세부안으로만 해석한다.

## 3. Renderer 입력 단계

renderer 입력은 단계별로 확장한다.

1차 입력은 `CanvasRenderInput`이다.

```text
RuntimeDocument
  -> app/controller adapter
  -> CanvasRenderInput
  -> CanvasScoreLayout
  -> layout/base canvas draw
```

이 단계에서는 UI/app controller가 `RuntimeDocument`를 renderer 전용 입력 계약인 `CanvasRenderInput`으로 바꾼다.
renderer는 `RuntimeDocument`, `ScoreFile`, `ScoreIndexes`를 직접 받지 않고, 이미 renderer에 필요한 형태로 좁혀진 행, 열, row label, row height, column width만 canvas 좌표로 변환한다.
이 1차 구현은 `1.9`의 전체 renderer MVP 중 layout/base grid를 먼저 안정화하는 선행 단계이다.
따라서 이 단계에서 note rectangle을 아직 표시하지 않더라도 `1.9`와 충돌하지 않으며, note rectangle MVP는 아래 2차 입력에서 `AnalysisResult` 변환을 추가해 연결한다.

2차 입력은 analyzer 결과에서 변환된 renderer item이다.

```text
RuntimeDocument + AnalysisResult
  -> app/controller adapter: CanvasRenderInput
  -> renderer item builder: CanvasNoteLayoutItem[] / CanvasMarkerItem[]
  -> CanvasScoreLayout + canvas render items
  -> layout/base/global-marker/note-marker/note canvas draw
```

이 단계에서는 renderer 내부 item builder가 analyzer가 확정한 `NoteEvent`를 `CanvasNoteRenderItem` 또는 `CanvasNoteLayoutItem`으로 변환하고, layer renderer는 note rectangle으로 표시한다.
renderer는 parser의 rawText를 다시 해석하지 않는다.
다만 `AnalysisResult` 직접 의존은 renderer 전체에 퍼뜨리지 않고 item builder 파일에 격리한다.

3차 입력은 viewport 상태이다.

```text
CanvasRenderInput + CanvasNoteRenderItem[] + CanvasViewport
  -> visible render items
  -> virtualized canvas draw
```

이 단계에서는 스크롤 위치와 zoom을 기준으로 보이는 영역만 다시 그린다.

## 4. 파일 구조

초기 renderer 파일 구조는 base layer 구현에 필요한 통합 구조로 시작한다.

```text
regression-code/src/app/
  main.ts
  canvas_renderer_adapter.ts

regression-code/src/renderer/
  canvas_types.ts
  canvas_coordinate.ts
  canvas_grid_renderer.ts
  canvas_score_renderer.ts
```

`app/main.ts`
- 브라우저 DOM 진입점이다.
- sample score 로드, canvas element 조회, status 표시, scroll sync 같은 app/controller 책임을 담당한다.
- core loader와 renderer adapter, renderer 진입점을 연결한다.
- renderer 내부 모듈이 DOM 조회나 파일 로드를 직접 하지 않도록 한다.

`app/canvas_renderer_adapter.ts`
- renderer 바깥에서 core 산출물을 canvas renderer DTO로 변환한다.
- `RuntimeDocument`에서 `CanvasRenderInput`을 만든다.
- layout/base layer 구현에 필요한 score layout DTO 변환만 담당한다.
- 이 파일은 score core 타입을 import할 수 있지만, analyzer 결과를 canvas item으로 변환하지 않는다.
- global row label은 이 adapter에서 사용자 표시 문자열로 변환한다.
  - `bpm` -> `BPM`
  - `beatsPerBar` -> `BeatsPerBar`
  - `stepsPerBeat` -> `StepsPerBeat`
  - `dynamics` -> `Dynamics`
  - note row -> `displayLabel`
  - gap row -> 빈 문자열

`canvas_types.ts`
- renderer 계층의 입력 계약, 좌표 타입, target 타입, options 타입을 한곳에 정의한다.
- `CanvasRenderInput`, `CanvasSourceRow`, `CanvasRenderOptions`, `CanvasRenderTarget`, `CanvasScoreLayout`을 포함한다.
- 후속 note/marker 구현에 필요한 `CanvasNoteRenderItem`, `CanvasNoteLayoutItem`, `CanvasMarkerItem`도 이 파일에 먼저 둔다.
- `RuntimeDocument`, `ScoreFile`, `ScoreIndexes`, `AnalysisResult` 타입은 import하지 않는다.

`canvas_coordinate.ts`
- `CanvasRenderInput`을 `CanvasScoreLayout`으로 변환한다.
- `CanvasRenderInput.rows`, `baseColumnWidthPx`, `columnCount`를 좌표 모델로 바꾼다.
- row label 표시 문자열과 row y 좌표를 확정한다.
- layout-side fixed padding cell과 playback boundary 좌표를 포함한 stage metric을 계산한다.
- CSS pixel 기준 stage 크기와 device pixel ratio를 canvas bitmap 크기에 반영한다.
- `columnToX()`와 canvas layer resize helper를 함께 둔다.
- 이 파일은 좌표계와 stage metric만 만들며, 실제 grid나 label을 그리지 않는다.

`canvas_item_builder.ts`
- 후속 note/marker layer 구현에서 analyzer 결과를 canvas용 item으로 변환한다.
- `AnalysisResult`와 `CanvasScoreLayout`을 받아 `CanvasNoteLayoutItem`, `CanvasMarkerItem` 등을 만든다.
- renderer 내부에서 analyzer 타입을 직접 import할 수 있는 유일한 파일로 둔다.
- parser rawText를 다시 해석하지 않고 analyzer가 확정한 event/timeline만 소비한다.

`canvas_grid_renderer.ts`
- layout label canvas와 base score canvas의 grid/background를 그린다.
- 전역 행과 gap 행은 같은 진한 배경으로 처리하고, note 행만 연한 배경으로 덮는다.
- playback 기준선 표시가 layout 영역에 속하는 경우 layout label renderer에서 처리한다.

`canvas_score_renderer.ts`
- renderer 계층의 진입점이다.
- `CanvasRenderInput`과 canvas target을 받아 변환 모듈과 layer renderer를 순서대로 호출한다.
- 초기에는 note/marker layer를 clear만 할 수 있다.
- DOM 조회, 파일 입력, 앱 상태 변경은 담당하지 않는다.

후속 기능이 붙어 파일이 커지면 아래처럼 분리할 수 있다.

```text
regression-code/src/renderer/
  canvas_note_renderer.ts
  canvas_marker_renderer.ts
  canvas_viewport.ts
  canvas_visible_range.ts
```

분리 기준은 다음이다.

- note rectangle/text draw가 base grid와 독립적으로 커질 때 `canvas_note_renderer.ts`를 만든다.
- beat/bar/gliss/loop marker draw가 독립적으로 커질 때 `canvas_marker_renderer.ts`를 만든다.
- scroll 기반 visible range 계산이 들어갈 때 viewport 관련 파일을 만든다.

## 5. 주요 타입

초기 context 타입은 renderer 입력 계약을 담당한다.
이 타입은 core 저장 구조를 노출하지 않는 renderer-owned DTO이다.

```ts
export type CanvasRenderInput = {
  rows: CanvasSourceRow[];
  columnCount: number;
  baseColumnWidthPx: number;
};
```

초기 grid 구현에서는 note item이 없어도 된다.
analyzer 연결 이후에도 layer draw 함수는 `AnalysisResult`를 직접 받지 않고 변환된 note/marker item만 받는다.
`AnalysisResult` 해석은 renderer 내부 `canvas_item_builder.ts`에 격리한다.

```ts
export type CanvasAnalyzedRenderInput = CanvasRenderInput & {
  noteItems: CanvasNoteRenderItem[];
  muteItems: CanvasMuteRenderItem[];
  globalTextItems: CanvasGlobalTextRenderItem[];
  globalMarkerItems: CanvasMarkerItem[];
  noteMarkerItems: CanvasMarkerItem[];
};
```

`CanvasRenderInput`은 `RuntimeDocument`와 모양이 비슷해서는 안 된다.
`RuntimeDocument`는 score 로드 계층의 산출물이고, `CanvasRenderInput`은 renderer 계층의 닫힌 입력 계약이다.
renderer 모듈은 `RuntimeDocument`, `ScoreFile`, `ScoreIndexes` 타입을 import하지 않는다.
`AnalysisResult` 타입 import는 후속 `canvas_item_builder.ts`에만 허용한다.

renderer source row 타입은 app/controller adapter가 core row definition에서 필요한 값만 복사해 만든다.

```ts
export type CanvasSourceRowKind = "global" | "note" | "gap";

export type CanvasSourceRow = {
  rowId: string;
  kind: CanvasSourceRowKind;
  label: string;
  height: number;
};
```

renderer 옵션은 좌표와 stage 크기 계산에 필요한 사용자/UI 설정을 담당한다.
값이 없으면 `CanvasRenderInput`의 기본 column width와 renderer 기본값을 사용한다.

```ts
export type CanvasRenderOptions = {
  zoom: number;
  devicePixelRatio: number;
  columnWidth?: number;
};
```

`zoom`은 CSS pixel 좌표계에 곱해지는 표시 배율이다.
`devicePixelRatio`는 canvas bitmap 해상도 보정에만 사용하고, score의 논리 좌표를 바꾸는 값으로 쓰지 않는다.
layout label 영역의 기준 좌우 여백은 각각 21px이고, 기준 중앙 label 영역은 100px이다.
실제 layout label canvas 폭은 `(21px + 100px + 21px) * zoom`이다.
label 텍스트 크기도 기준 12px에 `zoom`을 곱해 계산한다.
이 좌우 여백 cell은 라벨 문자열이 들어가는 중앙 영역과 세로 윤곽선으로 구분한다.
좌표 계산 단계는 옵션 값을 아래처럼 방어적으로 정규화한다.

- `zoom <= 0` 또는 유한하지 않은 값이면 `1`을 사용한다.
- `devicePixelRatio <= 0` 또는 유한하지 않은 값이면 `1`을 사용한다.
- `columnWidth`가 없거나 `<= 0` 또는 유한하지 않은 값이면 `CanvasRenderInput.baseColumnWidthPx`를 사용한다.

canvas target 타입은 renderer 진입점의 DOM 의존성을 낮추기 위한 전달 구조이다.

```ts
export type CanvasLayerTarget = {
  canvas: HTMLCanvasElement;
  context: CanvasRenderingContext2D;
};

export type CanvasRenderTarget = {
  layout: CanvasLayerTarget;
  base: CanvasLayerTarget;
  marker: CanvasLayerTarget;
  noteMarker: CanvasLayerTarget;
  note: CanvasLayerTarget;
};
```

canvas 좌표 타입은 renderer 내부에서 아래 책임을 갖는다.

```ts
export type CanvasRowKind = "global" | "note" | "gap";

export type CanvasLayoutRow = {
  rowId: string;
  kind: CanvasRowKind;
  label: string;
  y: number;
  height: number;
};

export type CanvasScoreLayout = {
  rows: CanvasLayoutRow[];
  columnCount: number;
  columnWidth: number;
  scoreContentWidth: number;
  stageWidth: number;
  stageHeight: number;
  layoutWidth: number;
  layoutLabelWidth: number;
  layoutLeftPaddingWidth: number;
  layoutRightPaddingWidth: number;
  layoutPlaybackBoundaryX: number;
  layoutFontSize: number;
};
```

`CanvasLayoutRow.y`는 해당 행의 상단 y 좌표이다.
`stageHeight`는 모든 row height의 합이다.
`scoreContentWidth`는 `columnCount * columnWidth`이다.
`stageWidth`는 score canvas가 그릴 실제 note/global column 영역의 너비이며, 기본값은 `scoreContentWidth`이다.
layout-side padding cell은 score canvas의 x 좌표에 섞지 않는다.
`layoutPlaybackBoundaryX`는 layout label canvas 안에서 강조선을 그릴 x 좌표이며, MVP에서는 layout 영역의 우측 경계에 둔다.
즉 `layoutPlaybackBoundaryX = layoutWidth`이다.
layout 우측 padding 강조 overlay는 `layoutRightPaddingWidth > 0`일 때 `layoutWidth - layoutRightPaddingWidth / 2`에서 시작해 `layoutRightPaddingWidth / 2` 너비로 그린다.
재생 표시선은 별도 윤곽선으로 그리지 않고 우측 고정 여백 cell의 반투명 배경으로만 표시한다.
score 영역의 재생 기준 x 좌표는 base renderer의 책임이 아니라 playback/scroll controller의 책임으로 둔다.
따라서 초기 `CanvasScoreLayout`에는 score-side playback boundary를 포함하지 않는다.

후속 확장에서 아래 타입을 추가한다.

```ts
export type CanvasViewport = {
  scrollLeft: number;
  scrollTop: number;
  width: number;
  height: number;
  zoom: number;
};

export type CanvasNoteRenderItem = {
  sourceEventId: string;
  rowId: string;
  startTick: number;
  endTick: number;
  text: string;
  trackId?: string;
};

export type CanvasNoteLayoutItem = CanvasNoteRenderItem & {
  x: number;
  y: number;
  width: number;
  height: number;
};

export type CanvasMarkerItem =
  | {
      kind: "beat" | "bar";
      tick: number;
    }
  | {
      kind: "gliss";
      startX: number;
      startY: number;
      endX: number;
      endY: number;
    };
```

초기 grid renderer는 `CanvasViewport` 없이 전체 stage를 그릴 수 있다.
가상화 도입 시 `CanvasViewport`를 받아 visible row/column만 그리는 경로를 추가한다.
초기 base layer 구현에서는 `CanvasNoteLayoutItem`과 `CanvasMarkerItem` 생성 함수를 작성하지 않아도 된다.
다만 note, gliss, beat/bar line 확장 시 marker renderer가 note renderer 내부 상태를 직접 읽지 않도록, 화면 좌표가 확정된 layout item을 별도 생성 단계로 둔다.

## 6. 진입점 책임

`canvas_score_renderer.ts`는 renderer 내부의 orchestration 모듈이다.

담당한다.

- `CanvasRenderInput`을 renderer 입력으로 받는다.
- `CanvasRenderOptions`를 받아 zoom과 DPR을 layout 변환과 canvas resize에 전달한다.
- `buildCanvasScoreLayout()`을 호출한다.
- layout/base/global marker/note marker/note layer를 정해진 순서로 호출한다.
- canvas 크기 또는 stage 크기 같은 렌더 결과 metadata를 반환한다.

담당하지 않는다.

- DOM에서 canvas element를 찾는 일
- JSON 파일을 읽는 일
- `RuntimeDocument`를 생성하거나 검증하는 일
- `RuntimeDocument`, `ScoreFile`, `ScoreIndexes`에서 renderer 입력을 추출하는 일
- parser/analyzer를 실행하는 일
- UI 상태를 저장하거나 변경하는 일

예상 호출 흐름은 다음이다.

```text
ui/app_controller
  -> RuntimeDocument 준비
  -> createCanvasRenderInput(runtimeDocument)        // renderer 바깥 adapter
  -> canvas element/context 준비
  -> CanvasRenderOptions 준비
  -> renderCanvasScore(target, renderInput, options)

canvas_score_renderer
  -> buildCanvasScoreLayout(renderInput, options)
  -> resizeCanvasLayers(target, layout, options)
  -> drawLayoutGrid(layoutContext, layout)
  -> drawScoreGrid(baseContext, layout)
  -> drawScoreMarkers(globalMarkerContext, layout, globalMarkerItems)
  -> drawScoreMarkers(noteMarkerContext, layout, noteMarkerItems)
  -> drawScoreNotes(noteContext, layout, noteItems, muteItems, globalTextItems)
  -> drawScoreOverlayMarkers(noteContext, layout, noteMarkerItems)

note/marker 후속 구현 시
  -> renderCanvasScore(target, renderInput, analysisResult, options)
  -> buildCanvasScoreLayout(renderInput, options)
  -> buildCanvasNoteLayoutItems(analysisResult, layout)
  -> buildCanvasMarkerItems(analysisResult, layout, noteLayoutItems)
  -> drawScoreNotes(noteContext, layout, noteLayoutItems)
  -> drawScoreMarkers(markerContext, layout, markerItems)
```

## 7. Layer별 입력 원칙

하위 layer renderer에는 가능한 한 필요한 최소 입력만 넘긴다.

권장 입력은 다음이다.

```text
drawLayoutGrid(context, layout)
drawScoreGrid(context, layout)
drawScoreNotes(context, layout, noteItems)
drawScoreMarkers(context, layout, markerItems)
drawScoreMarkersInRange(context, layout, markerItems, dirtyTickRange)
```

하위 renderer가 `RuntimeDocument`, `CanvasRenderInput`, `ScoreFile` 전체를 직접 참조하면, 저장 구조 변경이 곧 draw 함수 변경으로 번진다.
따라서 원본 문서 해석은 convert 모듈이나 진입점에서 끝내고, layer renderer는 canvas용 타입만 소비한다.
다만 `CanvasRenderInput` 자체도 이미 renderer용으로 좁힌 DTO여야 하며, core 저장 구조를 감싼 얇은 wrapper가 되어서는 안 된다.

## 8. 좌표 계산 원칙

x 좌표는 column/tick 기준이다.

```text
x = column * columnWidth
width = (endTick - startTick) * columnWidth
```

score content의 실제 첫 column은 score canvas의 x=0에서 시작한다.
layout label 영역 안의 좌우 padding cell은 기준 21px에 zoom을 곱한 별도 좌표로 처리한다.
marker, note, playback scroll 계산은 모두 같은 `columnToX()` helper를 공유해야 한다.
이 helper는 초기에는 `canvas_coordinate.ts`에 두고, 후속 좌표 유틸이 커지면 별도 모듈로 분리할 수 있다.
후속 playback/scroll controller는 현재 tick을 별도 재생 기준선에 맞추기 위해 아래 형태의 계산을 사용할 수 있다.

```text
scrollLeft = columnToX(currentTick) - playbackBoundaryX
```

`playbackBoundaryX`는 renderer base layout 타입이 아니라 playback/scroll controller 쪽 상태 또는 옵션으로 둔다.

y 좌표는 `CanvasLayoutRow.y`를 기준으로 한다.

```text
y = row.y
height = row.height
```

note rectangle은 셀 테두리에 정확히 맞아야 한다.
따라서 둥근 사각형이 아니라 grid cell 경계와 같은 사각형을 기본으로 한다.

1px stroke를 선명하게 표시해야 할 때는 `0.5` 보정을 사용할 수 있다.
단, fill 영역 자체는 cell의 정수 좌표를 기준으로 유지한다.

canvas bitmap 크기는 CSS pixel 크기에 `devicePixelRatio`를 곱해 잡고, context transform은 CSS pixel 좌표계로 맞춘다.
renderer 내부 좌표와 layout 계산은 항상 CSS pixel 기준으로 유지한다.
이렇게 하면 DPR 보정이 note 위치, scroll 위치, hit test 좌표와 섞이지 않는다.

## 9. 색상과 배경 원칙

초기 canvas grid는 legacy `이세계 코드`의 어두운 회색 계통을 따른다.

기본 렌더링 순서는 다음이다.

1. 전체 score 영역을 진한 회색 배경으로 칠한다.
2. note 행만 연한 회색 사각형으로 덮는다.
3. column grid와 row grid를 그린다.
4. global marker layer에 beat/bar/BPM/dynamics marker를 그린다.
5. note marker layer에 gliss/orphan marker를 그린다.
6. note layer에 note rectangle, display text, mute text, global rawText text, note overlay marker를 그린다.

전역 행과 gap 행은 같은 진한 회색으로 처리한다.
전역 행을 별도로 다시 칠하지 않는다.

layout label 영역도 같은 row layout을 사용한다.
note 행의 라벨 텍스트는 좌우 여백 cell을 제외한 중앙 라벨 영역에서 가운데 정렬한다.
layout 영역의 playback boundary 강조가 필요한 경우 score 영역이 아니라 layout 우측 고정 여백 cell 안에 반투명 배경으로 표시한다.

## 10. 가상화 대비

초기 구현은 전체 canvas stage를 그려도 된다.
다만 다음 조건을 지켜 가상화 도입을 쉽게 한다.

- row y 좌표와 stage height를 변환 모듈에서 한 번 계산한다.
- draw 함수 내부에 `RuntimeDocument`나 `ScoreFile` 해석 로직을 두지 않는다.
- renderer 모듈 내부에서 `RuntimeDocument`, `ScoreFile`, `ScoreIndexes` 타입을 import하지 않는다.
- `AnalysisResult` 타입 import와 해석은 후속 `canvas_item_builder.ts` 하나로 제한한다.
- column/tick에서 x 좌표로 바꾸는 계산을 한곳에 둔다.
- note render item은 `rowId`, `startTick`, `endTick` 기준으로 유지한다.
- viewport 기반 visible range 계산은 별도 모듈로 추가할 수 있게 한다.
- 전체 stage를 다시 그리는 경로와 visible range만 그리는 경로가 같은 `CanvasScoreLayout`을 소비하게 한다.

후속 가상화 모듈 후보는 다음이다.

```text
canvas_item_builder.ts
canvas_viewport.ts
canvas_visible_range.ts
```

## 11. 구현 순서

권장 구현 순서는 다음이다.

1. `src/renderer/canvas_types.ts` 생성
2. `src/renderer/canvas_coordinate.ts` 생성
3. `src/renderer/canvas_grid_renderer.ts` 생성
4. `src/renderer/canvas_score_renderer.ts` 생성
5. `src/app/canvas_renderer_adapter.ts` 생성
6. app/controller adapter에서 sample `RuntimeDocument`를 `CanvasRenderInput`으로 바꾼다.
7. `index.html`의 임시 inline draw 로직을 renderer 모듈 호출로 대체한다.
8. 실제 layout 기반 grid 표시를 확인한다.

초기에는 note layer가 임시 note item을 그리거나 비어 있어도 된다.
핵심은 app/controller adapter가 실제 score layout에서 추출한 row/column grid가 표시되는 것이다.

## 12. 검증 기준

초기 renderer는 최소 다음을 만족해야 한다.

- app/controller adapter가 `RuntimeDocument.score.layout.rowDefinitions` 순서를 보존해 `CanvasRenderInput.rows`를 만든다.
- `src/renderer/` 하위 base renderer 파일은 `RuntimeDocument`, `ScoreFile`, `ScoreIndexes`, `AnalysisResult` 타입을 직접 import하지 않는다.
- 후속 note/marker 구현에서 `AnalysisResult` 타입 import는 `canvas_item_builder.ts` 하나로 제한한다.
- 각 row의 `height`가 canvas y 좌표에 반영된다.
- `CanvasRenderInput.columnCount`와 `baseColumnWidthPx`가 stage width에 반영된다.
- score canvas의 content x 좌표와 layout label 영역의 fixed padding/playback 강조 좌표가 구분된다.
- layout label canvas와 score base canvas의 row 경계가 일치한다.
- global 행과 gap 행은 같은 진한 회색으로 표시된다.
- note 행만 연한 회색으로 표시된다.
- layout 영역의 좌우 여백 cell은 기준 21px에 zoom을 곱한 폭으로 라벨 영역과 세로 윤곽선으로 구분된다.
- 재생 표시선은 별도 윤곽선 없이 layout 우측 여백 cell 내부의 반투명 배경으로 표시된다.
- note/marker/playback scroll 계산이 같은 column-to-x helper를 사용한다.
- DPR이 1보다 큰 화면에서도 canvas가 흐릿하거나 좌표가 두 배로 밀리지 않는다.
- renderer 모듈이 core score/analyze 타입을 직접 import하지 않는다.
- TypeScript typecheck를 통과한다.

브라우저 수동 확인 항목은 다음이다.

- sample score 로드 후 grid가 보인다.
- label과 row line이 세로 스크롤 시 어긋나지 않는다.
- note 행과 gap/global 행의 색상 구분이 legacy와 유사하게 보인다.
- zoom 또는 viewport 크기 변경 이후에도 canvas가 빈 화면이 되지 않는다.
