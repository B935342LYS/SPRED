# Audio playback module 명세

## 1. 문서 목적

본 문서는 `회귀 코드`의 audio generator와 playback controller 구현을 위한 모듈 구조, 입력 계약, 핵심 알고리즘, 단계별 구현 범위를 정의한다.

`1.9-mvp-analyzer-renderer-ui-spec.md`는 MVP에서 audio / playback이 필요하다는 상위 요구를 정의하고, `1.7-analyzer-event-list-spec.md`는 renderer와 audio generator가 공통으로 소비할 `AnalysisResult` 구조를 정의한다.
본 문서는 그중 실제 TypeScript audio 모듈을 어떤 파일과 타입으로 나눌지 고정한다.

첫 구현 목표는 다음이다.

- `AnalysisResult`를 초 단위 audio schedule event로 변환한다.
- Web Audio API의 clock을 기준으로 note start / stop을 예약한다.
- play / pause / stop / seek 상태를 playback controller에서 관리한다.
- loop 구간을 playback controller와 scheduler가 해석한다.
- score scroll과 audible playback을 분리하되 같은 score time을 공유하게 한다.
- gliss, vibrato, tremolo, dynamics automation을 후속 확장할 수 있는 경계를 만든다.

본 문서는 구현 명세이며, parser / analyzer의 기존 계약을 변경하지 않는다.

## 2. 문서 우선순위

본 문서는 `1.9-mvp-analyzer-renderer-ui-spec.md`의 audio / playback 항목을 구체화하는 후속 구현 명세이다.

우선순위는 다음과 같이 본다.

1. 저장 형식과 런타임 문서 구조는 `1.3`, `1.8`, 현재 `core/score` 구현을 따른다.
2. analyzer event 의미와 `AnalysisResult` 구조는 `1.7`, `1.8`, 현재 `core/analyze` 구현을 따른다.
3. MVP audio / playback 범위는 `1.9`를 따른다.
4. UI 상태와 busy 처리 방식은 `2.0`, `2.2`를 따른다.
5. 본 문서는 audio module 내부 파일 구조, schedule DTO, playback state, Web Audio 연결 방식을 구체화한다.

본 문서가 상위 활성 명세와 충돌하면 상위 활성 명세를 우선하고, 본 문서는 구현 세부안으로 해석한다.

## 3. 기본 원칙

audio module의 가장 중요한 원칙은 다음이다.

```text
ScoreFile rawText
  -> parser
  -> analyzer
  -> AnalysisResult
  -> audio schedule DTO
  -> Web Audio API
```

audio module은 `rawText`를 다시 해석하지 않는다.
`@p`, `@m`, `@g`, `@t`, `-`, `~`, tuplet slot 같은 문법의 의미는 parser와 analyzer 단계에서 확정되어야 한다.
audio module은 analyzer가 확정한 event와 timeline을 초 단위 schedule로 바꾸고, 실제 브라우저 오디오 clock에 예약하는 역할만 맡는다.

renderer와 audio generator는 같은 `AnalysisResult`를 소비하지만, 서로의 DTO를 공유하지 않는다.
renderer는 canvas 좌표와 표시 item을 만들고, audio module은 초 단위 발음 event를 만든다.

## 4. 오픈소스 조사 반영

`docs/implementation-memo/1.19-step4-audio-open-source-survey.md`의 조사 결과를 다음처럼 반영한다.

- Web Audio API / MDN
  - 직접 구현 기준으로 사용한다.
  - `AudioContext.currentTime`, `AudioScheduledSourceNode.start()`, `stop()`, `AudioParam` automation을 기준으로 삼는다.
- IRCAM Web Audio scheduling tutorial
  - lookahead scheduler 개념을 참고한다.
  - JavaScript timer는 예약 범위를 갱신하고, 실제 발음은 Web Audio clock에 맡긴다.
- Tone.js
  - 직접 의존 후보로 유지한다.
  - 첫 구현에서는 analyzer 중심 계약을 보존하기 위해 backend interface 뒤의 대체 구현 후보로 둔다.
- smplr
  - sample 기반 backend 후보로 유지한다.
  - sample asset 라이선스 검토 전에는 repository에 sample을 포함하지 않는다.

따라서 1차 구현은 native Web Audio API + oscillator backend로 진행한다.
이 선택은 외부 코드 검토를 반영하되, 배포와 라이선스 위험을 줄이고, gliss / vibrato / tremolo의 AudioParam automation을 직접 검증하기 위한 것이다.

## 5. 입력과 출력

## 5.1. 입력

audio module의 공식 입력은 `AnalysisResult`이다.

```ts
type AudioBuildInput = {
  analysis: AnalysisResult;
  activeTrackIds: TrackId[];
};
```

초기 구현에서는 `activeTrackIds`에 `"basic"`만 넣을 수 있다.
후속 구현에서 basic / optional / extra track의 mix, mute, solo, volume을 확장한다.

audio module은 다음 analyzer 구조를 직접 참조한다.

- `AnalysisResult.timingTimeline`
- `AnalysisResult.dynamicsTimeline`
- `AnalysisResult.trackResults`
- `NoteEvent`
- `GlissEvent`
- `NoteEffectSegment`

`MuteEvent`, `TupletGroupEvent`, `TupletExtendGroupEvent`는 초기 audio schedule 대상이 아니다.
tuplet의 실제 발음은 group event가 아니라 group에 속한 `NoteEvent`와 `RestEvent`로부터 나온다.

## 5.2. 출력

audio module은 `AnalysisResult`를 바로 Web Audio node로 바꾸지 않고, 먼저 audio schedule DTO를 만든다.

```ts
type AudioSchedule = {
  durationSeconds: number;
  events: AudioScheduleEvent[];
};

type AudioScheduleEvent = {
  eventId: string;
  trackId: TrackId;
  startTick: TimeFraction;
  endTick: TimeFraction;
  startSeconds: number;
  endSeconds: number;
  midi: number;
  centOffset: number;
  velocity: number;
  effects: AudioScheduleEffect[];
  automation: AudioAutomationEvent[];
  sourceEventKind: "note";
};
```

이 DTO를 두는 이유는 다음이다.

- analyzer event는 의미 구조이고, audio backend는 초 단위 scheduling 구조를 필요로 한다.
- tick to seconds 변환을 backend 밖에서 테스트할 수 있다.
- oscillator backend, Tone.js backend, smplr backend가 같은 schedule DTO를 소비할 수 있다.
- partial analysis가 붙은 뒤에도 audio cache invalidation 단위를 분리하기 쉽다.

## 6. 시간 문법과 변환 규칙

## 6.1. tick

`tick`은 analyzer의 기본 시간 단위이다.
기본 note cell 1칸은 `[col, col + 1)`의 1 tick 구간이다.
tuplet slot은 유리수 tick을 가질 수 있으므로 `TimeFraction`을 그대로 유지한다.

```ts
type TimeFraction = {
  numerator: number;
  denominator: number;
};
```

audio module은 모든 tick을 number로 단순히 저장하지 않고, 경계에서는 `TimeFraction`을 유지한다.
다만 초 단위 계산 시에는 `numerator / denominator`로 변환한다.

## 6.2. tempo segment

`AnalyzedTimeSegment`는 특정 tick 범위에서 BPM, 박자, step 값이 어떻게 유지되는지를 나타낸다.

```ts
type AnalyzedTimeSegment = {
  time: TimeRange;
  startBpm: number;
  endBpm: number;
  bpmCurve: "instant" | "linear";
  beatsPerBar: number;
  stepsPerBeat: number;
  sourceCells: SourceCellRef[];
};
```

초기 구현에서는 `startBpm === endBpm`인 constant tempo segment를 우선 지원한다.

tick 하나의 길이는 다음처럼 계산한다.

```text
secondsPerBeat = 60 / bpm
secondsPerTick = secondsPerBeat / stepsPerBeat
```

예를 들어 BPM 120, `stepsPerBeat = 4`이면 1 beat는 0.5초이고 1 tick은 0.125초이다.

## 6.3. tick to seconds

`tickToSeconds()`는 score 시작점을 0초로 보고 tick 위치를 초로 변환한다.

```ts
type TickTimeMapper = {
  tickToSeconds(tick: TimeFraction): number;
  secondsToTick(seconds: number): TimeFraction;
  getDurationSeconds(): number;
};
```

초기 구현에서는 각 segment 내부가 constant tempo인 경우를 먼저 지원한다.
단 구현 구조는 timing segment가 여러 개 존재할 수 있음을 전제로 한다.
즉 mapper 생성 시 segment별 `startTick`, `endTick`, `startSeconds`, `secondsPerTick`을 누적 계산해 두고, `tickToSeconds()`와 `secondsToTick()`은 해당 값이 속한 segment를 찾아 변환한다.
후속 구현에서 `bpmCurve: "linear"`인 tempo ramp를 지원할 때도 같은 segment lookup 구조를 유지하되, segment 내부 변환 수식만 확장한다.

`secondsToTick()`은 seek bar와 scroll 위치에서 score time을 역산할 때 필요하다.
초기 구현에서 seek UI가 아직 연결되지 않더라도 타입 경계는 미리 둔다.

## 7. 음정 문법과 변환 규칙

## 7.1. MIDI note number

analyzer의 `FinalSoundPitch.midi`는 발음할 MIDI note number이다.
기본 frequency는 A4 = MIDI 69 = 440Hz 기준으로 계산한다.

```text
frequency = 440 * 2 ^ ((midi - 69 + centOffset / 100) / 12)
```

TypeScript 함수 후보는 다음이다.

```ts
function midiToFrequency(midi: number, centOffset: number): number;
```

## 7.2. cent offset

`centOffset`은 반음의 1/100 단위이다.
현재 프로젝트 결정상 `cent_num`은 `-100` 이상 `100` 이하의 유한 실수이고, 소수점 이하 1자리까지 허용한다.

audio module은 이 값을 다시 검증하지 않는다.
parser와 analyzer가 이미 유효한 값을 보장한다고 본다.
다만 defensive programming 차원에서 `midiToFrequency()` 내부에서 finite number 여부를 확인하고, 잘못된 값은 schedule build 단계의 error로 보고할 수 있다.

## 7.3. detune

Web Audio API의 `OscillatorNode.detune`은 cent 단위 `AudioParam`이다.
따라서 `centOffset`은 두 방식 중 하나로 반영할 수 있다.

1. `frequency`에 cent를 포함해 계산한다.
2. `frequency`는 MIDI 정수 기준으로 두고 `detune`에 cent를 넣는다.

초기 구현은 단순성을 위해 `frequency`에 cent를 포함해 계산한다.
후속 gliss / vibrato 구현에서는 `detune` automation 또는 `frequency` automation 중 더 테스트하기 쉬운 쪽을 선택한다.

## 8. 효과 문법과 audio 해석

## 8.1. hold

`-` hold는 audio module에 별도 토큰으로 전달되지 않는다.
analyzer가 여러 셀을 하나의 `NoteEvent.time`으로 병합했으므로, audio module은 긴 note 하나를 예약하면 된다.

즉 다음 입력은 audio 입장에서 동일하다.

```text
C4 - - -
```

```text
NoteEvent.time = [0, 4)
```

audio backend는 `[startSeconds, endSeconds)` 동안 하나의 oscillator를 울린다.

## 8.2. vibrato

`~`는 analyzer에서 `NoteEffectSegment.vib = true`로 전달된다.

초기 구현에서는 vibrato를 소리에 반영하지 않고, effect segment를 schedule DTO에 보존한다.
후속 구현에서는 아래 방식 중 하나를 사용한다.

- LFO OscillatorNode를 만들어 `frequency` 또는 `detune` AudioParam에 연결한다.
- `setValueCurveAtTime()`으로 pitch curve를 직접 만든다.

vibrato는 note 전체가 아니라 effect segment 단위로 적용되어야 한다.

## 8.3. tremolo

`@t(n)`은 analyzer에서 `NoteEffectSegment.trem.division = n`으로 전달된다.

초기 구현에서는 tremolo를 소리에 반영하지 않고 schedule DTO에 보존한다.
후속 구현에서는 gain automation으로 구현한다.

기본 해석은 다음이다.

- note 구간을 division 기준으로 더 작은 subsegment로 나눈다.
- subsegment의 일부만 gain을 열거나, 반복적인 gain envelope을 만든다.
- tremolo와 vibrato가 같은 segment에서 동시에 지속되지 않는다는 analyzer 정책을 따른다.

## 8.4. gliss

gliss는 `NoteEvent` 내부 효과가 아니라 `GlissEvent` 관계 이벤트이다.
초기 구현에서는 gliss를 소리에 반영하지 않고, 일반 note pitch만 재생한다.
후속 구현에서는 `GlissEvent.startSound`, `endSound`, `startAnchorTick`, `endAnchorTick`을 audio schedule automation으로 변환한다.

gliss audio 해석 원칙은 다음이다.

- start anchor와 end anchor 사이에서 pitch를 선형 보간한다.
- Web Audio에서는 `frequency.linearRampToValueAtTime()` 또는 `detune.linearRampToValueAtTime()`을 사용한다.
- `startAttach`, `endAttach` 값은 note 재시작 여부와 fade 처리 정책에 사용한다.

## 8.5. dynamics

`AnalyzedDynamicsSegment`는 시간축 위의 음량 제어값이다.
초기 구현에서는 전체 velocity 기본값을 `1.0`으로 둔다.
후속 구현에서는 dynamics timeline을 읽어 각 note의 gain envelope에 반영한다.

초기 DTO의 `velocity` 필드는 후속 dynamics 연결을 위해 미리 둔다.

## 9. Playback 상태

playback controller는 Web Audio node 자체가 아니라 재생 상태와 scheduler lifecycle을 관리한다.

```ts
type PlaybackState =
  | { kind: "stopped" }
  | {
      kind: "playing";
      audioStartedAt: number;
      scoreStartedAt: number;
    }
  | {
      kind: "paused";
      pausedAtScoreSeconds: number;
    };
```

필드 의미:

- `audioStartedAt`
  - `AudioContext.currentTime` 기준 재생 시작 시각이다.
- `scoreStartedAt`
  - 재생을 시작한 score time이다. 처음 play는 0, seek 후 play는 seek 위치가 된다.
- `pausedAtScoreSeconds`
  - pause 시점의 score time이다.

현재 score time은 다음처럼 계산한다.

```text
currentScoreSeconds =
  scoreStartedAt + (audioContext.currentTime - audioStartedAt)
```

이 값은 audio scheduler와 UI scroll 동기화가 함께 사용한다.

## 10. Scheduler 구조

초기 scheduler는 lookahead 방식을 사용한다.

```text
setInterval
  -> currentScoreSeconds 계산
  -> [current, current + lookahead] 범위 조회
  -> 아직 예약하지 않은 AudioScheduleEvent 예약
```

기본 후보값은 다음이다.

```ts
type AudioSchedulerOptions = {
  lookaheadSeconds: number;      // 초기값 0.15
  schedulerIntervalMs: number;   // 초기값 25
  noteReleaseSeconds: number;    // 초기값 0.03
};
```

JavaScript timer는 정확한 발음 시점을 책임지지 않는다.
timer는 가까운 미래 event를 찾아 Web Audio clock에 넘기는 역할만 한다.

## 11. Backend 구조

## 11.1. AudioBackend interface

backend는 실제 sound source를 생성하고 예약한다.

```ts
type AudioBackend = {
  ensureStarted(): Promise<void>;
  scheduleEvent(event: AudioScheduleEvent, offsetSeconds: number): void;
  stopAll(): void;
  dispose(): void;
};
```

`offsetSeconds`는 schedule event의 score time이 아니라, 현재 audio context 기준으로 몇 초 뒤에 시작할지를 의미한다.
실제 구현에서는 다음처럼 변환한다.

```text
audioStartTime = audioContext.currentTime + offsetSeconds
duration = event.endSeconds - event.startSeconds
```

## 11.2. OscillatorBackend

1차 backend는 oscillator + gain node를 사용한다.

```text
OscillatorNode
  -> GainNode
  -> master GainNode
  -> AudioContext.destination
```

note 시작 시 gain을 급격히 켜면 click noise가 생길 수 있으므로 짧은 attack / release envelope을 둔다.

초기 envelope 후보:

- attack: 0.005초
- release: 0.03초
- sustain gain: `velocity`

## 11.3. ToneBackend 후보

Tone.js를 직접 의존으로 도입할 경우에도 `AnalysisResult`를 Tone.js가 직접 해석하게 하지 않는다.
먼저 `AudioScheduleEvent`를 만든 뒤, Tone backend가 이를 Tone.js synth 또는 transport에 전달한다.

Tone.js는 다음 목적에 적합하다.

- poly synth 관리
- envelope / gain / effect boilerplate 감소
- 보고서에서 명확한 오픈소스 활용 사례 제시

다만 첫 구현에서는 native backend를 우선 작성해 회귀 코드의 시간 변환과 event queue를 검증한다.

## 11.4. SmplrBackend 후보

smplr는 MIDI note number 기반 sampled instrument backend 후보이다.
코드 라이선스는 MIT이나 sample asset 출처는 별도 확인이 필요하다.

초기 구현에서는 포함하지 않고, 후속 음색 개선 단계에서 optional dependency로 검토한다.

## 12. UI 연결

초기 UI 연결 범위는 다음이다.

- play
- pause
- stop
- seek bar 또는 score click 기반 seek
- loop on/off
- loop start / end
- volume
- score horizontal scroll 동기화

단 현재 UI에서 해당 버튼이 placeholder인 경우, audio module 먼저 작성한 뒤 app controller에 단계적으로 연결한다.

UI 연결 원칙:

- play는 user gesture 안에서 `AudioContext.resume()` 또는 backend `ensureStarted()`를 호출해야 한다.
- stop은 scheduler interval을 멈추고 예약/활성 node를 정리한다.
- pause는 현재 score time을 저장하고 stop과 같은 방식으로 활성 node를 정리한다.
- loop는 audio buffer 자체의 loop 기능이 아니라 score time 구간 반복으로 처리한다.
- score scroll은 `requestAnimationFrame`에서 `currentScoreSeconds`를 읽어 갱신한다.
- score scroll 기준선은 renderer가 가진 playback boundary 좌표 정책을 따른다.

## 12.1. Loop 정책

회귀 코드의 loop는 `AudioBufferSourceNode.loop`를 직접 사용하는 기능이 아니다.
Web Audio API의 buffer loop는 이미 준비된 하나의 audio buffer에서 `loopStart`, `loopEnd` 사이를 반복하는 기능에 가깝다.
반면 회귀 코드는 oscillator 또는 sampled instrument를 여러 note event 단위로 예약하므로, loop는 playback controller와 scheduler가 score time을 되감는 방식으로 구현한다.

```ts
type PlaybackLoopState =
  | { enabled: false }
  | {
      enabled: true;
      startSeconds: number;
      endSeconds: number;
    };
```

loop 상태 규칙:

- `startSeconds < endSeconds`일 때만 loop를 활성화한다.
- 현재 score time이 `endSeconds` 이상이 되면 scheduler는 다음 조회 기준을 `startSeconds`로 되돌린다.
- 이미 Web Audio clock에 예약된 loop end 이후의 node는 stop 또는 fade out 처리한다.
- lookahead window가 loop end를 가로지르면 `[current, end)`와 `[start, wrappedEnd)` 두 구간으로 나누어 event를 조회한다.
- 같은 event가 같은 loop cycle에서 중복 예약되지 않도록 `scheduledEventKey`에 cycle index를 포함한다.

loop는 오디오 샘플 버퍼를 미리 렌더링해 반복하는 방식이 아니므로, 이벤트 수가 수백 개여도 전체를 하나의 거대한 audio buffer로 만들 필요가 없다.
다만 lookahead window 안에 동시에 예약되는 event 수가 지나치게 많으면 source node 생성 비용이 증가할 수 있으므로, scheduler는 가까운 미래 구간만 예약한다.

## 13. 파일 구조

초기 파일 구조 후보는 다음이다.

```text
regression-code/src/audio/
  audio_types.ts
  tick_time_mapper.ts
  audio_schedule_builder.ts
  audio_event_queue.ts
  oscillator_backend.ts
  playback_controller.ts
```

`audio_types.ts`
- audio module 공용 타입을 정의한다.
- `AudioSchedule`, `AudioScheduleEvent`, `AudioBackend`, `PlaybackState`를 포함한다.

`tick_time_mapper.ts`
- `AnalyzedTimeSegment[]`에서 tick to seconds 변환기를 만든다.
- constant BPM segment부터 구현한다.

`audio_schedule_builder.ts`
- `AnalysisResult`와 active track 목록을 `AudioSchedule`로 변환한다.
- `NoteEvent`만 초기 schedule event로 만든다.
- gliss / vib / trem / dynamics는 DTO에 보존하거나 후속 automation 후보로 변환한다.

`audio_event_queue.ts`
- `AudioScheduleEvent[]`를 start time 기준으로 정렬하고 lookahead 범위 조회를 제공한다.
- playback seek 시 cursor를 재설정한다.

`oscillator_backend.ts`
- Web Audio API oscillator 기반 발음을 담당한다.
- `midiToFrequency()`와 gain envelope을 포함한다.

`playback_controller.ts`
- play / pause / stop / seek 상태와 scheduler interval을 관리한다.
- UI가 사용할 public method를 제공한다.

## 14. 핵심 문법 설명

이 절은 구현 전에 알아야 할 TypeScript / Web Audio 핵심 문법을 정리한다.

## 14.1. discriminated union

playback state와 schedule result는 `kind` 필드로 구분한다.

```ts
type PlaybackState =
  | { kind: "stopped" }
  | { kind: "playing"; audioStartedAt: number; scoreStartedAt: number }
  | { kind: "paused"; pausedAtScoreSeconds: number };
```

이 방식은 `state.kind === "playing"` 분기 안에서 TypeScript가 `audioStartedAt` 필드 존재를 자동으로 좁혀 주기 때문에, class 상속보다 현재 프로젝트 구조에 잘 맞는다.

## 14.2. AudioContext

`AudioContext`는 브라우저의 오디오 처리 엔진이다.

```ts
const context = new AudioContext();
await context.resume();
```

브라우저는 자동 재생을 막기 때문에, `resume()`은 play button click 같은 사용자 gesture 안에서 호출해야 한다.

## 14.3. currentTime

`context.currentTime`은 초 단위 audio clock이다.

```ts
const now = context.currentTime;
oscillator.start(now + 0.1);
oscillator.stop(now + 0.5);
```

`setTimeout()`이 발음 시각을 직접 책임지는 것이 아니라, `currentTime` 기준 예약을 Web Audio에 넘기는 것이 핵심이다.

## 14.4. AudioBufferSourceNode loop와 score loop의 차이

`AudioBufferSourceNode.loop`는 하나의 decoded audio buffer 안에서 일정 초 구간을 반복한다.

```ts
source.loop = true;
source.loopStart = 1.0;
source.loopEnd = 3.0;
```

이 방식은 배경음악 파일이나 녹음된 sample loop에는 적합하지만, 회귀 코드의 note event 재생에는 맞지 않는다.
회귀 코드의 loop는 악보 event list의 특정 score time 구간을 반복해야 하므로, scheduler가 반복 구간의 `AudioScheduleEvent`를 다시 예약해야 한다.

## 14.5. OscillatorNode

`OscillatorNode`는 간단한 파형을 생성한다.

```ts
const oscillator = context.createOscillator();
oscillator.type = "sine";
oscillator.frequency.setValueAtTime(440, context.currentTime);
```

초기 구현은 `sine` 또는 `triangle` 파형을 사용한다.
오타마톤에 가까운 음색은 후속 단계에서 waveform / filter / sampled backend로 조정한다.

## 14.6. GainNode

`GainNode`는 음량을 조절한다.

```ts
const gain = context.createGain();
gain.gain.setValueAtTime(0, startTime);
gain.gain.linearRampToValueAtTime(velocity, startTime + attack);
gain.gain.setValueAtTime(velocity, endTime - release);
gain.gain.linearRampToValueAtTime(0, endTime);
```

짧은 attack / release를 두면 note 시작과 끝의 click noise를 줄일 수 있다.

## 14.7. AudioParam automation

`frequency`, `detune`, `gain`은 `AudioParam`이다.
AudioParam은 시간에 따른 값 변화를 예약할 수 있다.

```ts
oscillator.frequency.setValueAtTime(startFrequency, startTime);
oscillator.frequency.linearRampToValueAtTime(endFrequency, endTime);
```

gliss는 frequency ramp, dynamics는 gain ramp, vibrato는 LFO 또는 value curve로 확장할 수 있다.

## 14.8. setInterval과 lookahead

lookahead scheduler는 일정 간격으로 가까운 미래 event를 예약한다.

```ts
const timerId = window.setInterval(() => {
  scheduleLookaheadWindow();
}, 25);
```

이 timer는 정확한 음 시작 시점을 만드는 도구가 아니다.
정확한 시작 시각은 `oscillator.start(audioStartTime)`에 전달되는 Web Audio clock 값이 담당한다.

## 16. 검증 기준

타입/단위 검증:

- `npm run typecheck`가 통과해야 한다.
- `tickToSeconds()`가 BPM 120, `stepsPerBeat = 4`에서 1 tick = 0.125초를 반환해야 한다.
- tuplet 유리수 tick이 초 단위로 안정적으로 변환되어야 한다.
- `midiToFrequency(69, 0)`은 440Hz를 반환해야 한다.
- `midiToFrequency(69, 100)`은 MIDI 70과 같은 frequency를 반환해야 한다.
- `NoteEvent.time`이 긴 hold 병합 구간이면 하나의 `AudioScheduleEvent`가 생성되어야 한다.

수동 브라우저 검증:

- play 버튼으로 sample score note가 들린다.
- stop 버튼으로 재생 중인 소리가 멈춘다.
- pause 후 다시 play하면 pause 위치에서 이어진다.
- loop on 상태에서 loop end에 도달하면 loop start로 되돌아가 재생된다.
- loop 구간을 가로지르는 lookahead window에서 note가 중복 예약되지 않는다.
- hold 병합 note가 여러 짧은 음이 아니라 하나의 긴 음으로 들린다.
- 재생 중 score horizontal scroll이 현재 score time에 맞춰 움직인다.
