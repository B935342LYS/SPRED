# 1.17. 2단계 gliss / mute marker rendering 구현 메모
> 본 설계 메모 문서 1.17에서는 gliss analyzer/render 연결, orphan gliss 표시, mute 표시, gliss+trem 시각 구분 구현 흐름을 기록한다.
> 본 문서는 active specification을 대체하지 않는다.
> 함수별 세부 설명은 코드 파일의 `/** ... */` 주석을 중심으로 하고, 본 문서는 흐름과 사용 문법, 설계 판단을 보충한다.

## 1. 개요

이번 구현 구간의 목표는 parser가 이미 구조화하던 gliss와 mute token을 analyzer event와 renderer DTO로 연결하는 것이었다.

기존 흐름은 아래 token들을 note event 안에서 대부분 처리했다.

```text
- hold
~ vibrato hold
@t tremolo
@p absolutePitch
@m microPitch
```

이번 구현 후 note 계열 시각화 흐름은 다음으로 확장되었다.

```text
ParsedNoteCell @g(...)
-> NoteEvent + GlissEvent
-> CanvasMarkerItem(kind: "gliss")
-> marker layer gliss line

ParsedMuteCell //text
-> MuteEvent
-> CanvasMuteRenderItem
-> note layer white text
```

또한 gliss 편집 UX를 위해 연결되지 않은 gliss anchor를 짧은 보조 선으로 표시한다.

## 3. gliss analyzer 연결

`@g(id,S|M|E)`는 parser 단계에서 `ParsedGliss`로 구조화되어 있었다.
이번 구현에서는 `analyze_track.ts`가 일반 note event를 생성하면서 gliss modifier를 별도 anchor 목록에 누적한다.

각 gliss anchor는 아래 정보를 가진다.

- `glissId`
- `role`
  - `S` -> `start`
  - `M` -> `mid`
  - `E` -> `end`
- 원본 `source`
- 원본 셀의 `time`
- 해당 note event의 display/sound 정보

분석이 끝나면 anchor 목록을 시간 순서로 훑어 `GlissEvent`를 만든다.

연결 규칙은 active analyzer spec의 방향을 따른다.

- 같은 track 안에서만 연결한다.
- 같은 gliss id 안에서만 연결한다.
- `start` 또는 `mid` 뒤에 오는 `mid` / `end`를 인접 segment로 연결한다.
- 전체 chain을 하나의 긴 이벤트로 합치지 않고, 인접 anchor 쌍마다 하나의 `GlissEvent`를 만든다.
- `end`는 다음 segment의 시작점으로 재사용하지 않는다.

즉 `S -> M -> E`는 두 개의 `GlissEvent`가 된다.

```text
S -> M
M -> E
```

## 4. 동일 열 중복 gliss anchor 정책

동일한 `trackId + glissId + role + col`에 gliss anchor가 여러 개 있으면 수직 연결선이 생길 수 있다.
이는 실제 연주/편집 관점에서 유효한 gliss 흐름으로 보기 어렵다.

현재 구현은 같은 열의 동일 id/role anchor 중 **layout 표시 순서상 위쪽 anchor 하나만 인식**한다.

이 기준을 선택한 이유는 다음과 같다.

- score JSON에는 설치 시각이 없으므로 "나중 설치" 또는 "처음 설치" 기준은 안정적으로 판정할 수 없다.
- rowId 문자열 정렬은 실제 화면의 위/아래와 다를 수 있다.
- `ScoreIndexes.rowsInDisplayOrder`는 renderer와 같은 행 순서를 제공하므로 analyzer에서도 같은 기준을 사용할 수 있다.
- 구현 비용이 낮고 결과가 결정적이다.

초기 구현에서는 `mid`만 이 정책을 적용했으나, `start`는 같은 열에 여러 개 있을 때 뒤쪽 anchor가 연결 후보 map을 덮어쓰는 문제가 있었다.
그 결과 다른 role은 위쪽 anchor가 유지되는데 `start`만 아래쪽 anchor가 인식되어 통일성이 깨졌다.
현재는 `start`, `mid`, `end` 모두 같은 중복 제거 정책을 사용한다.

중복으로 제외된 anchor는 gliss segment 생성에는 쓰지 않는다.
다만 사용자가 해당 token을 설치했다는 사실은 편집 화면에서 확인할 수 있어야 하므로, 연결 후보에서 제외된 anchor도 orphan marker로는 표시한다.
즉 "하나만 인식"은 연결 관계 판정에만 적용하고, 시각적 편집 보조 표시는 남긴다.

단, tuplet slot은 같은 head cell의 `rowId + col`을 공유하므로 중복 판정 key에 `slotIndex`를 포함한다.
따라서 일반 cell의 동일 열 중복 anchor는 제한하되, 같은 tuplet head 내부의 서로 다른 slot anchor는 독립적으로 인식한다.

## 5. gliss renderer DTO

`canvas_item_builder.ts`는 `GlissEvent`를 renderer-owned DTO로 변환한다.

```text
GlissEvent
-> CanvasMarkerItem(kind: "gliss")
```

gliss marker item은 양 끝 anchor의 표시 row와 tick을 가진다.

중요한 점은 `GlissEvent.time.endTick`을 선 끝점으로 직접 쓰지 않는 것이다.
analyzer의 `endTick`은 끝 anchor 셀의 배타적 끝점이므로, renderer 좌표로 쓰면 선 끝이 한 칸 오른쪽으로 밀린다.

초기에는 marker item의 시각 anchor tick을 `sourceCells[0].col`, `sourceCells[1].col`로 넘기고 renderer가 여기에 `columnWidth / 2`를 더했다.
이후 tuplet slot gliss와 hold 결합을 처리하면서 anchor 중심 tick을 analyzer가 직접 제공하도록 확장했다.
일반 cell은 `col + 0.5`, tuplet slot은 slot 시간 범위의 중심 tick을 사용한다.
이때 gliss 연결 후보에 저장하는 anchor time은 반드시 clone해야 한다.
그렇지 않으면 gliss end가 붙은 note가 뒤따르는 `-`로 병합될 때 anchor time이 병합된 note event 전체 시간으로 늘어나, gliss 선이 긴 직사각형의 중앙을 향해 그려진다.
현재 구현은 anchor가 설치된 첫 cell/slot의 time을 복사해 보존하므로, hold로 연장되어도 선의 시작/끝은 원래 anchor 위치에 고정된다.

## 6. gliss marker layer

gliss line은 `canvas_marker_renderer.ts`가 그린다.

현재 canvas layer 순서는 다음이다.

```text
base layer
marker layer
note layer
```

처음에는 marker layer가 note layer 위에 있었지만, gliss 선이 note rectangle보다 위에 올라와 시각적으로 과하게 보였다.
현재는 marker를 note 뒤에 두어 gliss 선이 note rectangle 뒤로 지나가게 했다.

일반 gliss는 실선으로 그린다.

```text
start anchor center -> end anchor center
```

## 7. orphan gliss anchor 표시

편집 중에는 gliss anchor가 아직 연결되지 않은 상태가 자연스럽게 발생한다.
이 상태를 아무 표시 없이 두면 사용자가 현재 gliss 토큰의 방향과 미완성 상태를 확인하기 어렵다.

이를 위해 `NoteEvent.glissAnchors`에 원본 anchor 목록을 남기고, renderer DTO 변환 단계에서 연결된 `GlissEvent` source cell에 포함되지 않은 anchor를 찾는다.

```text
NoteEvent.glissAnchors
-> connected GlissEvent sourceCells와 비교
-> 연결되지 않은 anchor
-> CanvasMarkerItem(kind: "glissOrphanAnchor")
```

orphan 표시 규칙은 다음이다.

- `start`
  - 오른쪽으로 짧은 선
- `end`
  - 왼쪽으로 짧은 선
- `mid`
  - 좌우 양쪽 짧은 선

marker layer가 note 뒤에 있으므로, 선이 note rectangle에 완전히 가려지지 않게 셀 중앙이 아니라 note rectangle 바깥쪽 반 칸 구간에 그린다.

예를 들어 `start`는 anchor cell 중심에서 오른쪽 반 칸 이후부터 다음 반 칸까지 표시한다.

## 8. mute analyzer / renderer 연결

parser는 `//text`를 `ParsedMuteCell`로 만든다.
이번 구현에서는 이를 `MuteEvent`로 승격했다.

`MuteEvent`는 발음 이벤트가 아니라 표시 전용 이벤트이다.
따라서 다음 정보만 renderer에 필요하다.

- track
- one-cell time range
- display row
- text

renderer DTO는 `CanvasMuteRenderItem`이다.

```text
MuteEvent
-> CanvasMuteRenderItem
-> note layer text draw
```

현재 시각 표현은 다음과 같다.

- 사각형 없음
- 텍스트 색상: 흰색
- font: `21px Arial`
- 위치: 셀 중앙

font size는 우선 고정값으로 두었다.
향후 zoom과의 관계를 더 정교하게 다룰 수 있지만, 이번 구현에서는 사용자가 요청한 21px 기준을 우선 적용했다.

## 9. gliss + trem 시각 표시

음악적으로 gliss와 trem은 동시에 적용될 수 있다.
분석 구조에서도 두 효과는 역할이 다르다.

- gliss
  - 두 anchor 사이의 연속 pitch 이동 관계
  - `GlissEvent`
- trem
  - note event 내부의 발음 분할 효과
  - `NoteEvent.effects[].trem`

따라서 analyzer 타입을 합치지 않고, renderer DTO 변환 단계에서 두 구조의 시간 범위 겹침을 확인한다.

```text
GlissEvent time range
vs
same track NoteEvent.effects trem ranges
```

초기 구현은 gliss segment 시간 범위에 겹치는 trem 구간이 하나라도 있으면 `CanvasMarkerItem(kind: "gliss")`에 `hasTrem: true`를 기록했다.
이 방식은 mid 또는 end anchor 쪽 trem도 왼쪽으로 들어오는 gliss segment를 점선으로 만들 수 있었다.

현재 구현은 trem+gliss 점선을 **시작 anchor의 오른쪽 outgoing segment**에만 적용한다.
즉 `GlissEvent.sourceCells[0]`이 가리키는 note/slot의 anchor 시간에 trem effect가 있을 때만 `hasTrem: true`가 된다.
mid anchor에 trem이 있으면 incoming segment는 실선이고, 해당 mid에서 오른쪽으로 나가는 다음 segment만 점선이다.

시각 표현은 고정 밀도 점선이다.

```ts
context.setLineDash([6, 4])
```

처음에는 trem division에 따라 점선 밀도를 달리하는 방안도 검토했다.
그러나 연주자 관점에서 `@t(2)`와 `@t(4)`가 gliss 선의 밀도 차이로까지 구분되어야 할 필요는 낮다고 판단했다.
따라서 renderer는 "gliss에 trem 성격이 섞였다"는 사실만 보여주고, division 차이는 analyzer/audio 데이터에 남긴다.

## 10. app runtime 연결

`app_runtime.ts`의 full rebuild 경로는 이제 renderer 입력에 세 가지 item 목록을 붙인다.

```text
AnalysisResult
-> buildCanvasNoteRenderItems()
-> buildCanvasMuteRenderItems()
-> buildCanvasMarkerItems()
-> CanvasAnalyzedRenderInput
```

즉 edit mode에서 score cell rawText를 변경하면 아래 전체 경로가 다시 수행된다.

```text

-> renderCanvasScore()
```

## 11. 테스트 확인

`dev/test_analyze.ts`에는 다음 검증이 추가되었다.

- `S -> M -> E`가 두 개의 `GlissEvent`를 만든다.
- gliss marker item의 시각 tick은 source cell col 기준이다.
- 동일 열 중복 mid는 위쪽 하나만 연결 대상으로 쓴다.
- standalone start/mid/end는 orphan marker item을 만든다.
- 연결된 mid와 같은 col/id의 중복 mid는 orphan marker로 만들지 않는다.
- `//memo`는 `MuteEvent`와 `CanvasMuteRenderItem`으로 변환된다.
- `@g(...)@t(3)`처럼 gliss와 trem이 겹치면 gliss marker item의 `hasTrem`이 true가 된다.
- trem이 겹치지 않는 gliss segment는 `hasTrem`이 false이다.

확인한 명령은 다음이다.

```bash
npm run typecheck
npm run test:analyze
npm run test:parse
npm run test:edit
npm run test:score
npm run build
```

## 12. 남은 작업

다음 구현 단위는 tuplet analyzer / renderer 연결이다.

예상 작업은 다음과 같다.

1. `ParsedPletHeadCell`과 오른쪽 `ParsedPletExtendCell`을 하나의 group으로 결합한다.
2. `TupletGroupEvent`를 생성한다.
3. slot note/rest를 분수 tick 기반 `NoteEvent` / `RestEvent`로 변환한다.
4. `@n(midi)`를 실제 note rowId로 매핑한다.
5. tuplet bracket 또는 slot group marker를 renderer DTO로 만든다.
6. 기존 hold/gliss/trem/mute 처리와 tuplet slot이 충돌하지 않도록 테스트를 확장한다.
