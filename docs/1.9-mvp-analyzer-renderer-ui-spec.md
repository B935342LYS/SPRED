# MVP analyzer / renderer / UI 구현 명세

## 1. 문서 목적

본 문서는 `회귀 코드`의 첫 번째 사용자 화면 연결을 위해 필요한 최소 구현 범위를 고정한다.

앞선 문서들이 저장 구조, parser, analyzer 출력 구조, parser-analyzer 파이프라인을 정의했다면, 본 문서는 그중 실제 웹 앱 MVP에서 우선 구현할 analyzer, renderer, audio, UI 범위를 정한다.

이번 MVP의 목표는 다음과 같다.

- `ScoreFile -> RuntimeDocument -> ParsedScoreDocument -> AnalysisResult -> 화면 표시 / 기본 재생` 흐름을 실제 브라우저 UI에서 확인한다.
- analyzer는 일반 note의 `default_text`와 hold 토큰 `"-"`만 우선 구현한다.
- renderer와 audio generator는 legacy `이세계 코드`에서 참고 가능한 흐름을 가져올 수 있지만, 입력 계약은 `회귀 코드`의 `AnalysisResult`를 따른다.
- UI는 legacy 앱의 기본 사용 흐름을 참고하되, `회귀 코드`에서 추가된 기능을 수용할 수 있는 확장 지점을 함께 둔다.

## 2. 문서 위치와 우선순위

본 문서는 `1.8-parser-analyzer-pipeline-spec.md` 다음 단계의 MVP 구현 명세이다.

우선순위는 다음과 같이 본다.

1. 저장 형식, parser 결과, analyzer 출력 타입은 기존 활성 명세를 따른다.
2. 본 문서는 그 타입들 중 MVP에서 실제 구현할 부분집합을 제한한다.
3. 본 문서와 기존 활성 명세가 충돌하면 기존 활성 명세를 우선하고, 본 문서는 MVP 제한 사항으로만 해석한다.

즉 본 문서는 장기 명세를 대체하지 않고, 구현 가능한 첫 단계를 명확히 자르는 역할을 한다.

## 3. MVP 전체 흐름

MVP의 런타임 흐름은 아래와 같다.

```text
JSON fixture / imported ScoreFile
  -> loadRuntimeDocument()
  -> buildParsedDocument()
  -> analyzeDocument()
  -> renderScore()
  -> prepareAudioEvents()
  -> playbackController.play()
```

이번 단계에서 반드시 확인해야 하는 것은 analyzer가 만든 `AnalysisResult`만으로 renderer와 audio generator가 동작할 수 있는가이다.

따라서 renderer와 audio generator는 parser 내부 구조나 rawText를 직접 다시 해석하지 않는다.

## 4. analyzer MVP 범위

### 4.1. 입력

analyzer는 기존 `AnalyzeContext`를 입력으로 받는다.

```ts
type AnalyzeContext = {
  score: ScoreFile;
  indexes: ScoreIndexes;
  parsed: ParsedScoreDocument;
};
```

MVP analyzer는 `ParsedScoreDocument.noteCellsByTrackAndCol`을 기준으로 track별 이벤트를 생성한다.

### 4.2. 지원하는 note 셀

MVP에서 지원하는 note 셀은 다음으로 제한한다.

- `ParsedNoteCell`
- `displayText`가 있는 일반 note 셀
- hold 값이 `null` 또는 `"-"`인 note 셀
- modifier가 없는 note 셀

즉 다음 원본 문자열이 1차 대상이다.

```text
default_text
-
```

`default_text`는 해당 셀이 위치한 `rowId`의 음정과 표시 위치를 가진 note 시작점으로 해석한다.

`"-"`는 같은 track 안에서 바로 왼쪽에 연결 가능한 note가 있으면 해당 note의 duration을 연장한다.

### 4.3. hold 연결 규칙

`"-"`의 연결은 아래 조건을 모두 만족할 때만 성립한다.

- 같은 `trackId` 안에 있어야 한다.
- 시간축상 바로 이전 tick에 연결 대상 note가 있어야 한다.
- 현재 셀의 `rowId`와 이전 note의 표시 `rowId`가 같아야 한다.
- 현재 셀의 최종 발음 음정과 이전 note의 최종 발음 음정이 같아야 한다.
- MVP에서는 modifier를 지원하지 않으므로 cent 보정은 항상 `0`으로 본다.

연결이 성립하면 기존 `NoteEvent.time.endTick`을 현재 셀의 다음 tick까지 늘린다.

연결이 성립하지 않는 `"-"`는 MVP에서 독립 note 시작점으로 처리한다. 이때 표시 텍스트는 parser의 `displayText`를 따른다.

### 4.4. NoteEvent 생성

일반 note 셀은 다음 구조의 `NoteEvent`가 된다.

- `eventKind`: `"note"`
- `time.startTick`: `col`
- `time.endTick`: `col + 1`
- `display.rowId`: 셀의 `rowId`
- `display.centOffset`: `0`
- `sound.midi`: note 행의 `midi`
- `sound.centOffset`: `0`
- `effects`: vib와 trem이 없는 기본 effect segment
- `sourceCells`: 해당 note를 만든 셀 좌표

hold로 병합된 note는 `sourceCells`에 병합된 셀들을 순서대로 보존한다.

### 4.5. MVP에서 제외하는 analyzer 기능

다음 기능은 이번 MVP analyzer에서 구현하지 않는다.

- `"~"` vibrato hold
- `@g(...)` gliss
- `@t(...)` trem
- `@p(...)` absolute pitch
- `@m(...)` micro pitch
- `pletHead`, `pletExtend`, tuplet slot
- mute event
- partial analysis
- analyzer cache

parser가 이미 위 구조를 만들 수 있더라도, MVP analyzer는 해당 기능을 소비하지 않는다.
테스트 fixture에는 우선 MVP 지원 범위 밖의 셀을 넣지 않는다.

## 5. timing / dynamics MVP 범위

renderer가 tick 기반 위치를 표시하고 audio generator가 기본 재생을 수행하려면 최소 timing 정보가 필요하다.

MVP에서는 전역 timeline을 아래처럼 제한한다.

- 기본 tick 단위는 기존 명세대로 1 column = 1 tick으로 둔다.
- `bpm`, `beatsPerBar`, `stepsPerBeat`, `dynamics`는 우선 `col = 0`의 시작값만 사용한다.
- 선형 변화 토큰, 중간 tempo 변경, 중간 dynamics 변경은 후속 단계로 둔다.
- timing/dynamics parser 결과가 유효하지 않은 fixture는 MVP 테스트 대상에서 제외한다.

MVP audio가 이번 구현 단계에 포함되므로 `AnalysisResult.timingTimeline`과 `AnalysisResult.dynamicsTimeline`은 빈 배열이 아니라 최소 기본 세그먼트를 반환해야 한다.

## 6. renderer MVP 범위

renderer는 `AnalysisResult`와 `RuntimeDocument.indexes`를 입력으로 받아 화면 표시를 수행한다.

### 6.1. legacy 참고 가능 범위

legacy `이세계 코드`에서 다음 흐름은 참고할 수 있다.

- base grid 렌더링
- row label 표시
- note rectangle 표시
- scroll-position 기반 재생 기준선 강조 표시
- zoom 또는 scroll 처리의 기본 UI 흐름

다만 legacy 코드는 Google Sheets / Apps Script / canvas 분할 구조에 묶여 있으므로 그대로 이식하지 않는다.
`회귀 코드` renderer는 TypeScript 모듈로 작성하고, 입력은 `AnalysisResult` 중심으로 둔다.

### 6.2. MVP 표시 대상

MVP renderer는 다음을 표시한다.

- note row grid
- column grid
- row label
- `NoteEvent` rectangle
- note의 `displayText`
- hold로 병합된 note의 긴 rectangle

MVP renderer는 다음을 표시하지 않는다.

- gliss line
- vibrato wave
- tremolo pattern
- tuplet bracket / slot
- dynamics curve
- edit selection overlay

### 6.3. 좌표 계산

MVP의 x축은 column/tick 기준으로 계산한다.

- `startTick = col`
- `endTick = col + duration`
- rectangle x 위치는 `startTick`
- rectangle width는 `endTick - startTick`

y축은 `rowId`가 가리키는 note row의 display order를 기준으로 계산한다.

행 높이와 열 너비는 초기에는 고정값으로 둘 수 있다.
다만 이후 사용자별 layout customization을 수용할 수 있도록, 좌표 계산 로직은 renderer 내부 상수에 직접 흩어 두지 않는다.

legacy `이세계 코드`와 동일하게 layout label area 쪽에는 여백용 column 영역을 둔다.
이 여백은 실제 note event column과 구분되는 표시/스크롤 보정 영역이며, renderer layout config에서 `leftPaddingColumns`, `rightPaddingColumns` 같은 명시적 값으로 관리한다.
재생 기준은 layout label area와 score canvas area의 경계이다.
다만 여백용 column 중 오른쪽 column의 우측 절반을 반투명 red overlay로 칠해, 이 경계가 현재 재생 기준선임을 시각적으로 강조한다.
playback controller는 현재 tick이 이 경계에 오도록 horizontal scroll을 계산한다.

## 7. audio / playback MVP 범위

audio generator와 playback controller는 이번 MVP의 필수 구현 대상이다.

audio generator는 `AnalysisResult.trackResults[].events` 중 `NoteEvent`만 소비한다.
playback controller는 UI의 play / pause / stop 조작과 score scroll 기반 재생 위치를 관리하고, 내부적으로 audio generator가 준비한 재생 이벤트를 사용한다.

### 7.1. legacy 참고 가능 범위

legacy `이세계 코드`에서 다음 흐름은 참고할 수 있다.

- Web Audio API oscillator 사용
- note start / stop scheduling
- play / stop / seek 제어 흐름
- loop UI 흐름

단 legacy의 정규식 기반 오디오 해석은 가져오지 않는다.
`회귀 코드`에서는 parser/analyzer가 이미 이벤트를 확정하므로, audio generator는 `NoteEvent`를 다시 문자열로 해석하지 않는다.

### 7.2. 모듈 책임

이번 단계에서는 audio generator와 playback controller를 구분한다.

- audio generator
  - `NoteEvent`를 재생 가능한 오디오 스케줄 입력으로 변환한다.
  - MIDI pitch를 frequency로 변환한다.
  - tick duration을 tempo 기준 시간으로 변환한다.
  - Web Audio API의 oscillator / gain node 생성을 담당한다.
- playback controller
  - play / pause / stop 버튼 동작을 담당한다.
  - 현재 재생 위치와 score horizontal scroll 상태를 관리한다.
  - 재생 중 현재 tick이 layout label area와 score canvas area의 경계에 오도록 scroll 위치를 갱신한다.
  - 이후 loop 확장을 받을 수 있는 경계를 제공한다.

### 7.3. MVP 재생 대상

MVP audio / playback은 다음만 처리한다.

- `NoteEvent`
- MIDI pitch를 frequency로 변환
- tick duration을 tempo 기준 시간으로 변환
- play
- pause
- stop
- seek bar와 score scroll 동기화

MVP audio / playback은 다음을 처리하지 않는다.

- loop
- vibrato
- tremolo
- gliss pitch ramp
- dynamics automation
- accompaniment
- YouTube sync
- game mode input

## 8. UI MVP 범위

UI는 legacy 앱의 기본 작업 흐름을 참고하되, `회귀 코드`의 추가 기능을 위한 확장 지점을 가진다.

legacy `이세계 코드`에서 참고할 UI 구조는 다음이다.

- 상단 transport 영역: play, pause, stop
- visual menu: fullscreen, fit height, reverse, theme, zoom, speed, text scale
- track menu: track 표시/숨김
- playback/audio controls: beat/bar line 표시, volume, waveform
- edit mode toggle과 edit panel
- labels canvas와 roll canvas를 분리한 viewer 구조
- base, marker, note, fx 같은 역할별 overlay layer

다만 MVP에서는 위 구조를 모두 동작시키지 않는다.
초기 화면은 legacy의 배치를 참고하되, 실제 구현은 `회귀 코드`의 상태 모델과 TypeScript 모듈 경계에 맞춰 새로 작성한다.

### 8.1. 기본 화면 구성

MVP UI는 다음 영역을 가진다.

- 상단 toolbar
- score viewport
- diagnostics/status 영역
- 필요 시 inspector 영역

상단 toolbar에는 최소한 다음 조작을 둔다.

- initial sample auto load
- JSON file load
- JSON file download placeholder
- analyze/render
- play
- pause
- stop
- seek bar
- volume
- waveform
- zoom control
- active track 표시 또는 선택

loop는 legacy 구조상 중요하지만 이번 MVP에서는 비활성 확장 지점으로 둔다.

### 8.2. 추가 기능을 위한 UI 확장 지점

`회귀 코드`는 legacy보다 명시적인 기능 확장을 전제로 하므로, MVP UI에도 후속 기능을 배치할 자리를 둔다.

다음 기능은 버튼이나 패널 위치만 먼저 고려하고, 실제 동작은 비활성 상태로 둘 수 있다.

- edit mode
- loop
- global row editor
- tuplet tools
- gliss tools
- dynamics view
- layout preset
- import/export
- audio settings

비활성 기능은 사용자가 구현된 기능으로 오해하지 않도록 disabled 상태 또는 준비 중 상태로 표시한다.

### 8.3. 모바일 범위

기존 프로젝트 결정에 따라 초기 모바일 범위는 view/load/play로 제한한다.

MVP UI는 모바일에서도 악보를 볼 수 있는 반응형 구조를 목표로 하되, 모바일 edit mode는 구현하지 않는다.

## 9. 구현 순서

본 문서 기준의 권장 구현 순서는 다음과 같다.

1. analyzer test fixture 작성
2. `analyze_track.ts`에서 default note와 `"-"` hold 병합 구현
3. `analyze_full.ts`에서 trackResults, timingTimeline, dynamicsTimeline 조립
4. `test_analyze.ts` 작성
5. 최소 renderer 작성
6. app entry에서 load/parse/analyze/render 연결
7. audio generator 작성
8. playback controller 작성
9. UI toolbar와 status 영역 정리

초기 구현에서는 full analysis만 구현한다.
partial analysis는 analyzer와 renderer의 기본 계약이 안정화된 뒤 추가한다.

## 10. 검증 기준

MVP analyzer는 최소 다음 테스트를 통과해야 한다.

- 단일 `default_text` 셀은 길이 1 tick의 `NoteEvent`가 된다.
- 같은 row에서 `default_text`, `"-"`, `"-"`가 이어지면 길이 3 tick의 `NoteEvent` 하나가 된다.
- 다른 row의 `"-"`는 이전 note와 병합되지 않는다.
- 서로 다른 track의 note는 서로 병합되지 않는다.
- analyzer는 rawText를 직접 다시 해석하지 않는다.

MVP renderer는 최소 다음 상태를 수동 확인한다.

- fixture 로드 후 grid가 보인다.
- note rectangle이 row와 column에 맞게 보인다.
- hold 병합 note가 긴 rectangle로 보인다.
- layout label area 쪽 여백용 column 영역이 note column과 구분되어 보인다.
- 오른쪽 여백 column의 우측 절반이 반투명 red overlay로 강조되어 경계 기준선이 보인다.
- zoom 또는 viewport 크기 변경 후에도 note 위치가 크게 어긋나지 않는다.

MVP audio / playback은 최소 다음 상태를 확인한다.

- play 버튼으로 note가 재생된다.
- stop 버튼으로 재생이 중지된다.
- hold 병합 note가 여러 짧은 음이 아니라 하나의 긴 음으로 재생된다.
- 재생 중 score horizontal scroll이 시간 흐름에 따라 이동하며, layout label area와 score canvas area의 경계가 현재 재생 기준선으로 동작한다.

## 11. 난이도 소명

이번 단계의 난점은 parser 구현 자체보다 후속 모듈의 입력 경계를 지키는 데 있다.

legacy `이세계 코드`는 Google Sheets와 Apps Script 환경에서 렌더링, 재생, 편집 흐름이 강하게 결합되어 있었다.
반면 `회귀 코드`는 parser, analyzer, renderer, audio generator를 분리하고, renderer와 audio generator가 같은 `AnalysisResult`를 소비하도록 해야 한다.

이 구조를 지키지 않으면 초기 구현은 빨라질 수 있지만, 이후 gliss, tuplet, dynamics, partial update가 붙을 때 renderer와 audio generator가 각자 parser 구조를 다시 해석하는 문제가 생긴다.

따라서 MVP에서는 기능 수를 줄이는 대신, 아래 원칙을 검증하는 데 집중한다.

- parser는 문자열을 구조화한다.
- analyzer는 셀 간 관계와 duration을 확정한다.
- renderer는 analyzer 결과를 표시한다.
- audio generator는 analyzer 결과를 재생한다.

즉 이번 MVP는 기능 완성도보다 경계 검증이 핵심이다.

## 12. 후속 단계

MVP 이후에는 다음 순서로 확장한다.

1. `"~"` vibrato hold
2. mute event
3. global timing/dynamics 중간 변경
4. audio dynamics 반영
5. gliss
6. tuplet
7. partial analysis
8. edit mode
9. accompaniment
10. game mode

이 순서는 구현 중 발견되는 병목이나 UI 우선순위에 따라 조정할 수 있으나, parser/analyzer/renderer/audio의 책임 경계는 유지한다.
