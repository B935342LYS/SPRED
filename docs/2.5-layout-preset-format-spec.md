# 2.5. 레이아웃 프리셋 포맷 명세

## 1. 목적

본 문서는 `회귀 코드`의 사용자 레이아웃 프리셋 저장 포맷을 정의한다.

레이아웃 프리셋은 현재 악보 전체를 저장하는 파일이 아니다.
사용자가 자주 쓰는 악기 튜닝, 현 범위, note/gap 행 구성, 행 높이를 재사용하기 위한 별도 데이터이다.

같은 포맷을 다음 두 저장 경로에서 공유한다.

- 브라우저 로컬 저장
- JSON 파일 저장 및 불러오기

따라서 Local Save/Load와 File Save/Load는 저장 위치만 다르고, 저장 데이터 구조는 동일해야 한다.

## 2. 핵심 결정

- 프리셋은 score file과 별도로 저장한다.
- 프리셋은 `ScoreFile.instData`와 전역 행을 제외한 note/gap `rowDefinitions`를 함께 저장한다.
- 프리셋은 score의 track cell, global cell, metadata, zoom, scroll position을 저장하지 않는다.
- 프리셋 파일 포맷과 localStorage 저장값은 동일한 `UserLayoutPresetData` 구조를 사용한다.
- 로컬 프리셋은 `instrumentPresetId`별 고정 3슬롯으로 분리한다.
- 첫 구현에서는 `layoutPresetId`를 앱이 생성하는 내부 ID로 사용한다.
- `layoutPresetDisplayName`은 사용자에게 표시되는 이름이며, 최대 30글자이다.
- 프리셋 JSON은 UTF-8 기준 256 KiB를 넘을 수 없다.
- Local Save/Load는 개인 작업 환경용이고, File Save/Load는 백업/공유/기기 이동용이다.

## 3. 저장 대상과 비저장 대상

### 3.1. 저장 대상

프리셋은 다음 값을 저장한다.

- 프리셋 포맷 버전
- 프리셋 식별자
- 프리셋 표시명
- 대상 악기 프리셋 ID
- 생성/수정 시각
- 악기 layout 정보
- note/gap rowDefinitions

### 3.2. 저장하지 않는 값

프리셋은 다음 값을 저장하지 않는다.

- `ScoreFile.musicData`
- `ScoreFile.globalLines`
- `ScoreFile.tracks`
- global rowDefinitions
- zoom level
- scroll position
- 현재 선택 track
- edit mode 상태
- playback 위치

프리셋 적용으로 삭제된 track cell은 프리셋만으로 복구할 수 없다.
삭제될 cell이 있는 경우 적용 전에 사용자 확인을 받아야 한다.

## 4. 타입 정의

### 4.1. UserLayoutPresetData

```ts
type UserLayoutPresetData = {
  formatVersion: "1";
  layoutPresetId: string;
  layoutPresetDisplayName: string;
  instrumentPresetId: string;
  createdAt: string;
  updatedAt: string;
  instData: LayoutPresetInstrumentData;
  rowDefinitions: LayoutPresetRowDefinition[];
};
```

필드 의미:

| 필드 | 의미 |
| --- | --- |
| `formatVersion` | 레이아웃 프리셋 포맷 버전. 첫 버전은 `"1"`이다. |
| `layoutPresetId` | 앱이 생성하는 프리셋 내부 ID이다. |
| `layoutPresetDisplayName` | UI에 표시되는 프리셋 이름이다. |
| `instrumentPresetId` | 이 프리셋이 기준으로 삼은 `ScoreFile.instData.presetId`이다. |
| `createdAt` | ISO 8601 문자열 생성 시각이다. |
| `updatedAt` | ISO 8601 문자열 마지막 수정 시각이다. |
| `instData` | 프리셋 적용 시 사용할 악기 layout 정보이다. |
| `rowDefinitions` | 전역 행을 제외한 note/gap rowDefinitions이다. |

### 4.2. LayoutPresetInstrumentData

첫 구현에서는 기존 `InstrumentData`와 같은 구조를 저장한다.

```ts
type LayoutPresetInstrumentData = {
  presetId: string;
  family: string;
  instName: string;
  supportsOpen: boolean;
  strings: LayoutPresetInstrumentString[];
};

type LayoutPresetInstrumentString = {
  stringId: string;
  stringName: string;
  openMidi?: number;
  minMidi: number;
  maxMidi: number;
};
```

정책:

- `presetId`는 원본 악기 프리셋 계열을 식별하기 위해 유지한다.
- `instName`은 프리셋 적용 시 `layoutPresetDisplayName`과 동기화할 수 있다.
- `supportsOpen === false`이면 `openMidi`는 없어도 된다.
- 첫 구현에서 string 추가/삭제는 지원하지 않더라도, 포맷은 여러 string을 담을 수 있다.

### 4.3. LayoutPresetRowDefinition

프리셋에는 note/gap row만 저장한다.

```ts
type LayoutPresetRowDefinition =
  | LayoutPresetNoteRowDefinition
  | LayoutPresetGapRowDefinition;

type LayoutPresetNoteRowDefinition = {
  rowId: string;
  type: "note";
  stringId: string;
  midi: number;
  height: number;
  displayLabel: string;
};

type LayoutPresetGapRowDefinition = {
  rowId: string;
  type: "gap";
  stringId: string;
  fromMidi: number;
  toMidi: number;
  height: number;
};
```

global row는 파일에 포함하지 않는 것이 원칙이다.
외부 파일에 global row가 들어 있어도 첫 구현에서는 무시하거나 오류로 막을 수 있다.
권장 동작은 오류로 막는 것이다.

## 6. row 저장 규칙

### 6.1. note row

- `type`은 `"note"`이다.
- `rowId`는 앱이 생성한다.
- 권장 형식은 `{stringId}-note-{midi}`이다.
- `midi`는 정수 `0..127`이다.
- 같은 `stringId + midi` 조합은 중복될 수 없다.
- `height`는 양의 정수이다.
- 첫 구현에서 note row 높이는 공통값으로 편집하더라도, 저장은 각 note row의 `height` 필드에 반영한다.
- `displayLabel`은 `midi`에서 자동 생성한 계이름이다.

### 6.2. gap row

- `type`은 `"gap"`이다.
- `rowId`는 앱이 생성한다.
- 권장 형식은 `{stringId}-gap-{fromMidi}-{toMidi}`이다.
- `fromMidi`와 `toMidi`는 내부 정렬과 식별을 위한 값이다.
- `fromMidi`와 `toMidi`는 사용자에게 직접 노출하지 않아도 된다.
- `fromMidi < toMidi`를 만족해야 한다.
- 첫 구현의 gap boundary MIDI 허용 범위는 `-1..128`이다.
- `0` 아래쪽 gap이나 `127` 위쪽 gap을 표현하기 위해 `-1`과 `128`을 허용한다.
- 같은 string 안에서 gap row가 연속으로 두 개 올 수 없다.
- 같은 `stringId + fromMidi + toMidi` 조합은 중복될 수 없다.
- `height`는 양의 정수이다.

## 7. 검증 규칙

### 7.1. 포맷 검증

프리셋 JSON을 불러올 때 다음 조건을 확인한다.

- JSON parse가 가능해야 한다.
- 최상위 값은 object여야 한다.
- `formatVersion === "1"`이어야 한다.
- 필수 필드가 모두 존재해야 한다.
- `layoutPresetId`, `layoutPresetDisplayName`, `instrumentPresetId`는 빈 문자열이 아니어야 한다.
- `layoutPresetDisplayName`은 30글자 이하여야 한다.
- `createdAt`, `updatedAt`은 문자열이어야 한다.
- `instData`는 object여야 한다.
- `rowDefinitions`는 배열이어야 한다.
- 입력 JSON 문자열은 UTF-8 기준 256 KiB 이하여야 한다.

### 7.2. 악기 정보 검증

- `instData.presetId`는 빈 문자열이 아니어야 한다.
- `instData.family`는 빈 문자열이 아니어야 한다.
- `instData.instName`은 빈 문자열이 아니어야 한다.
- `instData.instName`은 30글자 이하여야 한다.
- `instData.supportsOpen`은 boolean이어야 한다.
- `instData.strings`는 하나 이상의 항목을 가져야 한다.
- `stringId`는 프리셋 안에서 중복될 수 없다.
- `minMidi`와 `maxMidi`는 정수 `0..127`이어야 한다.
- `minMidi <= maxMidi`를 만족해야 한다.
- `supportsOpen === true`이면 필요한 string에 `openMidi`가 있어야 한다.
- `openMidi`가 있으면 정수 `0..127`이어야 한다.

### 7.3. row 검증

- 모든 row의 `rowId`는 프리셋 안에서 유일해야 한다.
- 모든 row의 `stringId`는 `instData.strings[].stringId` 중 하나여야 한다.
- 모든 row의 `height`는 양의 정수여야 한다.
- note row의 `midi`는 정수 `0..127`이어야 한다.
- note row의 `midi`는 해당 string의 `minMidi..maxMidi` 범위 안에 있어야 한다.
- 같은 string 안에서 note row MIDI는 중복될 수 없다.
- gap row의 `fromMidi/toMidi`는 정수여야 한다.
- gap row의 `fromMidi/toMidi`는 `-1..128` 범위여야 한다.
- gap row는 `fromMidi < toMidi`를 만족해야 한다.
- 같은 string 안에서 gap row가 연속될 수 없다.

## 8. Local Save/Load 저장 정책

### 8.1. localStorage key

로컬 저장은 악기 프리셋 기준으로 분리하고, 각 악기 프리셋마다 Slot 1..3만 제공한다.

```text
layout-slot:{instrumentPresetId}:1
layout-slot:{instrumentPresetId}:2
layout-slot:{instrumentPresetId}:3
```

설명:

- `instrumentPresetId`는 현재 score의 `ScoreFile.instData.presetId`이다.
- `layoutPresetId`는 local 저장에서는 `slot-1`, `slot-2`, `slot-3` 중 하나로 고정한다.
- 첫 구현에서는 `scoreWriter`를 key에 포함하지 않는다.
- Local Save는 사용자가 선택한 슬롯 하나를 덮어쓴다.
- 슬롯에 이미 프리셋이 있으면 덮어쓰기 전에 확인한다.
- 빈 슬롯은 `Slot N Empty`로 표시한다.
- File Save/Load는 이 3슬롯 제한을 받지 않는다.

`scoreWriter`를 key에 포함하지 않는 이유:

- 레이아웃 프리셋은 작성자보다 악기/튜닝 선호에 가깝다.
- 같은 사용자가 여러 악보 작성자 이름을 사용할 수 있다.
- 개인 브라우저 저장소 안에서는 악기 기준 필터링이 더 단순하다.

후속으로 사용자 계정이나 공유 프리셋 기능이 생기면 namespace 정책을 다시 정의할 수 있다.

### 8.2. slot 값

각 slot key에는 `UserLayoutPresetData` 전체를 저장한다.
빈 슬롯은 key가 존재하지 않는 상태로 표현한다.

Local Load 시에는 사용자가 선택한 슬롯 key를 읽고, 해당 데이터를 layout draft로 복원한다.
이 시점에는 현재 score에 적용하지 않는다.
실제 score 반영은 사용자가 `Apply`를 누를 때 수행한다.

### 8.3. 슬롯 수 제한

로컬 프리셋은 `instrumentPresetId`별 3개로 제한한다.

제한 이유:

- 브라우저 localStorage 용량은 score 본편 저장과 공유된다.
- 로컬 프리셋은 자주 쓰는 개인 설정을 빠르게 불러오기 위한 기능이지, 대량 프리셋 라이브러리가 아니다.
- 슬롯 UI는 저장 위치를 사용자에게 명확하게 보여준다.

3개를 초과해 보관해야 하는 프리셋은 File Save로 JSON 파일을 내려받아 관리한다.
로컬 저장은 자동으로 오래된 슬롯을 삭제하지 않는다.
사용자가 명시적으로 슬롯을 선택해 덮어써야 한다.

### 8.4. 용량 제한

Local Save와 File Save/Load는 같은 `UserLayoutPresetData` JSON 문자열을 사용하며,
UTF-8 기준 256 KiB를 상한으로 둔다.

산정 기준:

- 4개 string
- 각 string의 note row가 MIDI `0..127` 전체를 포함
- 각 string의 gap row가 하단 boundary, 모든 인접 note 사이, 상단 boundary를 포함
- 총 note row 512개, gap row 516개, 전체 row 1,028개
- 현재 pretty JSON 직렬화 기준 약 154 KiB

따라서 256 KiB는 위 최대급 draft에 약 1.6배 여유를 둔 값이다.
이 제한을 넘는 프리셋은 비정상적으로 큰 레이아웃 또는 손상된 입력으로 보고 저장/불러오기를 막는다.

## 9. File Save/Load 저장 정책

### 9.1. File Save

File Save는 현재 layout draft를 `UserLayoutPresetData` JSON 파일로 다운로드한다.

권장 파일명:

```text
layout-{layoutPresetDisplayName}.json
```

파일명에 사용할 수 없는 문자는 `_`로 치환한다.
파일 저장 시 `updatedAt`은 저장 시각으로 갱신한다.

### 9.2. File Load

File Load는 사용자가 선택한 JSON 파일을 읽어 `UserLayoutPresetData`로 검증한 뒤 현재 layout draft에 반영한다.

File Load 후에도 현재 score는 바뀌지 않는다.
사용자가 `Apply`를 눌러야 ScoreFile에 반영된다.

File Load 시 `instrumentPresetId`가 현재 score의 `instData.presetId`와 다르면 경고한다.
첫 구현의 권장 동작은 사용자 확인 후 draft load를 허용하는 것이다.

## 10. Apply와의 관계

프리셋 Load는 draft만 바꾼다.
ScoreFile 변경은 항상 Apply에서만 수행한다.

Apply 흐름:

```text
preset load 또는 dialog 편집
-> layout draft 갱신
-> Apply
-> 삭제될 track cell 계산
-> 삭제될 cell이 있으면 사용자 확인
-> 구조 공유 방식으로 next ScoreFile 생성
-> instData와 rowDefinitions 교체
-> 필요 시 유효하지 않은 track cell 삭제
-> validateScoreFile(nextScore)
-> runtime rebuild
-> playback runtime reset
```

프리셋 저장 포맷은 삭제될 cell 목록을 저장하지 않는다.
삭제 판정은 현재 score와 draft를 비교해 Apply 시점에 계산한다.

## 11. 호환성 정책

### 11.1. formatVersion

첫 버전은 `"1"`이다.
향후 breaking change가 생기면 `formatVersion`을 올린다.

예상 가능한 변경:

- string 추가/삭제 지원
- per-string note height 정책
- display label 표기 체계 저장
- 프리셋 작성자 또는 설명 필드 추가
- 여러 악기 family별 전용 옵션 추가

### 11.2. 알 수 없는 필드

첫 구현에서는 알 수 없는 최상위 필드를 무시해도 된다.
단, 필수 필드의 타입 오류는 무시하지 않는다.

권장:

- top-level unknown field: ignore
- row unknown field: ignore
- required field missing: error
- required field invalid type: error
- unsupported `formatVersion`: error

### 11.3. 표시명 중복

로컬 저장은 3슬롯 고정 방식이므로 표시명 중복을 저장 차단 조건으로 삼지 않는다.
사용자가 같은 이름을 여러 슬롯에 저장할 수는 있지만, UI는 슬롯 번호를 함께 표시해 구분한다.

File Load의 경우 표시명 중복은 저장소 충돌이 아니므로 오류로 막지 않는다.

## 12. 구현 모듈 권장 구조

권장 구현 파일:

```text
regression-code/src/app/layout/
  layout_preset.ts

regression-code/src/infra/
  layout_preset_storage.ts
```

`layout_preset.ts` 책임:

- `UserLayoutPresetData` 생성
- `layoutPresetId` 생성
- 파일명 생성
- 프리셋 JSON parse 및 검증
- draft와 preset data 간 변환

`layout_preset_storage.ts` 책임:

- localStorage slot key 생성
- Slot 1..3 저장 상태 조회
- 선택 slot 저장/조회
- localStorage 오류를 사용자 메시지로 변환

기존 `layout_apply.ts`는 preset 저장 포맷을 직접 알 필요가 없다.
`layout_apply.ts`는 오직 draft를 ScoreFile에 적용하는 책임만 가진다.

## 13. 현재 결정 요약

- Local Save/Load와 File Save/Load를 모두 유지한다.
- 두 기능은 같은 `UserLayoutPresetData` 포맷을 사용한다.
- Local은 악기 프리셋별 Slot 1..3 고정 저장을 사용하는 개인 프리셋 재사용용이다.
- File은 백업, 공유, 환경 이동용이다.
- 프리셋 Load는 draft만 변경한다.
- ScoreFile 반영은 Apply에서만 수행한다.
- 삭제될 track cell 확인은 preset load가 아니라 Apply 단계에서 수행한다.
