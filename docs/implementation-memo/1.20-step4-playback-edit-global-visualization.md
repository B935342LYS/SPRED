# 1.20. 4단계 playback / edit UX / global visualization 구현 메모
> 본 구현 메모 문서 1.20에서는 audio playback 1차 연결, 테스트 편의용 edit UX 개선, 전역 행 rawText 시각화 구현 흐름을 기록한다.
> 본 문서는 active specification을 대체하지 않는다.
> 함수별 세부 설명은 코드 파일의 `/** ... */` 주석을 중심으로 하고, 본 문서는 보고서와 발표에 필요한 흐름, 구현 판단, 핵심 문법을 보충한다.

## 1. 개요

이번 구현 구간의 목표는 기존 parser / analyzer / renderer 흐름 위에 실제 사용자가 검증할 수 있는 실행 루프를 더하는 것이었다.

이전 단계까지는 아래 흐름이 중심이었다.

```text
ScoreFile JSON
-> RuntimeDocument
-> buildParsedDocument()
-> analyzeDocument()
-> Canvas item 변환
-> canvas score rendering
```

이번 구현 후에는 아래 흐름이 추가되었다.

```text
AnalysisResult
-> AudioSchedule
-> AudioEventQueue
-> LookaheadScheduler
-> OscillatorBackend
-> PlaybackController
-> score scroll sync
```

또한 악보 테스트 입력을 빠르게 만들 수 있도록 edit mode의 셀 입력 방식이 확장되었다.

```text
pointer edit
-> hitTestScoreCell()
-> rawText compose / click cycle / number compose
-> ScoreFile batch mutation
-> full rebuild
-> canvas redraw
```

전역 행 입력은 내부 JSON에는 저장되고 있었지만 화면에 보이지 않았으므로, 전역 셀 rawText를 renderer item으로 변환해 흰색 텍스트로 표시하도록 연결했다.

## 3. Audio playback 1차 구현

### 3.1. 채택한 방향

오디오 모듈은 Tone.js를 바로 의존성으로 추가하지 않고, 브라우저 표준 Web Audio API 위에서 최소 backend를 직접 구현하는 방향으로 시작했다.

이유는 다음과 같다.

- 현재 analyzer 출력은 이미 `NoteEvent`, `GlissEvent`, `TimeRange`로 구조화되어 있다.
- 초기 검증에는 기본 note start / stop scheduling이 가장 중요하다.
- Web Audio API는 `AudioContext.currentTime` 기준으로 정확한 예약 재생을 제공한다.
- Tone.js는 후속 가상악기, 샘플러, 더 풍부한 synth backend 후보로 남길 수 있다.
- scheduler / backend 경계를 분리하면 Tone.js 도입 시에도 analyzer와 renderer 구조를 다시 만들 필요가 없다.

### 3.2. 전체 흐름

현재 playback의 주요 흐름은 다음과 같다.

```text
buildAudioSchedule({
  analysis,
  activeTrackIds
})
-> createAudioEventQueue(schedule)
-> createOscillatorBackend({ waveType, masterVolume })
-> createAudioLookaheadScheduler({ queue, backend, lookaheadSeconds })
-> createPlaybackController({ schedule, scheduler, backend })
```

UI에서 `Play`를 누르면 `PlaybackController.playFromStart()`가 실행된다.
controller는 scheduler를 시작하고, scheduler는 일정 주기마다 Web Audio clock 기준 lookahead 범위 안에 들어온 note event를 backend로 넘긴다.

### 3.3. lookahead scheduler

lookahead scheduler는 긴 곡 전체를 한 번에 `setTimeout()`으로 예약하지 않는다.

대신 짧은 주기로 다음 구간을 확인한다.

```text
현재 audio time
-> 현재 time + lookaheadSeconds 사이의 event 조회
-> backend.scheduleNote() 호출
-> 다음 timer tick에서 반복
```

이 구조의 장점은 다음과 같다.

- 긴 악보에서도 한 번에 모든 note를 예약하지 않는다.
- UI thread timer jitter를 Web Audio 예약 clock으로 완화한다.
- 이후 pause / seek / loop 구현 시 queue를 다시 만드는 방식으로 확장할 수 있다.

### 3.4. score scroll sync

재생 중 악보 스크롤은 score canvas의 왼쪽 경계를 재생 기준선으로 둔다.

```text
playback.getCurrentScoreSeconds()
-> TickTimeMapper.secondsToTick()
-> columnToX(currentTick)
-> scoreArea.scrollLeft
```

즉, 빨간 기준선이나 별도 cursor를 악보 중앙에 두는 방식이 아니라, layout과 score canvas의 경계가 현재 재생 위치가 되도록 score 영역을 스크롤한다.

### 3.5. 현재 한계

현재 audio backend는 기본 note event 중심이다.

아직 직접 반영하지 않은 항목은 다음과 같다.

- gliss pitch ramp
- vibrato LFO / AudioParam automation
- tremolo gain chopping
- dynamics timeline의 실제 volume automation
- 복수 timing segment에 따른 tempo 변화
- loop range playback
- sampled instrument 또는 Tone.js backend

따라서 현재 audio 구현은 "기본 note event가 analyzer에서 오디오 예약으로 연결되는지" 검증하는 1차 단계이다.

## 4. Edit UX 개선

### 4.1. 구현 목적

테스트 악보를 만들려면 같은 열 또는 근처 열에 많은 rawText를 반복 입력해야 한다.
기존 단일 클릭 입력만으로는 10000열 이상의 긴 악보나 여러 note row를 빠르게 채우기 어렵다.

이번 구현에서는 legacy `이세계 코드`의 입력 감각을 참고해 아래 기능을 추가했다.

- 동일 셀 반복 클릭 loop
- drag 입력
- 우클릭 drag 삭제
- note row 근접 hit 보정
- 전역 행 number 입력
- note/global batch edit 적용

### 4.2. 동일 셀 반복 클릭 loop

기본 note 입력에서 같은 셀을 반복 클릭하면 아래 순서로 rawText가 바뀐다.

```text
currentText -> - -> ~ -> currentText -> ...
```

여기서 `currentText`는 기존 셀 값이 아니라 현재 Default UI에서 합성한 rawText이다.

예를 들어 현재 Default UI가 `C4`를 만들고 있다면 같은 note cell 반복 클릭은 다음처럼 동작한다.

```text
1회: C4
2회: -
3회: ~
4회: C4
```

`-`와 `~` 단계는 수식자가 붙지 않는 순수 rawText이다.

### 4.3. drag 입력과 drag loop

drag 입력은 pointer down / move / up 흐름으로 처리한다.

```text
pointerdown
-> drag state 생성
-> pointermove 중 hit cell 수집
-> 누락 column 보정
-> pointerup
-> batch edit 적용
```

drag 중에는 각 셀의 기존 rawText를 읽고 셀별 loop를 적용한다.

```text
빈 셀       -> currentText
일반 기존 값 -> -
기존 -      -> ~
기존 ~      -> currentText
```

이 방식은 단순히 같은 rawText를 모든 셀에 찍는 방식보다 테스트 악보 수정에 유리하다.
이미 입력된 노트 위를 다시 드래그하면 hold / vibrato hold로 빠르게 바꿀 수 있기 때문이다.

### 4.4. 우클릭 drag 삭제

편집 모드의 우클릭은 브라우저 context menu 대신 삭제 입력으로 해석한다.

단일 우클릭은 해당 셀 삭제이고, 우클릭 drag는 지나간 셀을 batch로 삭제한다.

```text
right pointerdown
-> drag state
-> hit cell 수집
-> rawText ""
-> batch edit
```

### 4.5. note row 근접 hit 보정

현재 악보의 note row는 7px 수준으로 얇다.
정확히 note row를 누르기 어려우므로 edit용 hit-test에는 근접 note row 보정 옵션을 추가했다.

기본 hit-test는 원래 동작을 유지한다.
편집 pointer 처리에서만 아래 옵션을 사용한다.

```ts
hitTestScoreCell(event, dom.scoreStage, state.layout, {
  nearestNoteSlopPx: NOTE_ROW_HIT_SLOP_PX,
});
```

이 옵션이 켜져 있으면 gap row 또는 비어 있는 y 좌표에서도 가까운 note row가 허용 거리 안에 있으면 note row로 보정한다.

또한 pointerdown 지점이 gap 또는 무효 row여도 drag state를 만들 수 있다.
이후 pointermove 중 note/global hit가 잡히면 그 지점부터 입력된다.

### 4.6. legacy 참고와 현재 채택 범위

legacy `이세계 코드`에서 확인한 주요 보정은 다음과 같다.

- pointer 좌표를 editable cell로 변환하는 `pickCellFromEvent_()`
- 현재 셀 rawText를 기준으로 hold/vibrato를 순환하는 `cycleValue_()`
- drag 중 셀 사이를 선분으로 보간하는 `applyDragEditBetweenCells_()`
- 가까운 note row를 선택하는 일부 보정 흐름
- pending edit와 undo action grouping

현재 버전에 직접 반영한 것은 다음이다.

- 기존 rawText 기준 click / drag loop
- 같은 행에서 누락된 column 보정
- note row 근접 hit 보정
- batch edit 적용

보류한 것은 다음이다.

- 대각선 Bresenham 보간
- pending edit queue와 undo stack
- tuplet slot drag 수정

대각선 보간은 note row가 매우 얇은 현재 UI에서 의도하지 않은 row까지 수정할 수 있어 보류했다.

## 5. Number UI와 전역 행 입력

### 5.1. 기본 규칙

Number UI는 별도 편집 모드가 아니라 상시 입력 패널로 둔다.

편집 모드에서 전역 행을 클릭하면 Default note 입력 대신 Number UI 값을 사용한다.

```text
Number input value + ramp token
-> globalLines.cells rawText
```

예시는 다음과 같다.

```text
120
120<
120>
120><
```

### 5.2. 전역 행별 허용 규칙

전역 행 종류에 따라 허용되는 rawText가 다르다.

| 전역 행 | 예시 | ramp 허용 |
|---|---|---|
| `bpm` | `120`, `120<` | 허용 |
| `dynamics` | `100`, `100><` | 허용 |
| `beatsPerBar` | `4` | 불허 |
| `stepsPerBeat` | `4` | 불허 |

`beatsPerBar`, `stepsPerBeat`에서 ramp token을 선택한 상태로 입력하면 UI에서 차단 메시지를 표시한다.

### 5.3. rawText 저장 위치

전역 행 입력은 track cell이 아니라 `ScoreFile.globalLines.cells`에 저장된다.

```text
ScoreFile.globalLines.cells[]
  rowId
  col
  rawText
```

따라서 note cell 적용 함수와 전역 cell 적용 함수를 같은 batch edit 경계에서 다루되, 실제 mutation 대상은 분리한다.

## 6. Batch edit 적용 경계

기존 편집은 셀 하나를 수정할 때마다 full rebuild를 수행했다.

```text
single edit
-> ScoreFile clone
-> createRuntimeDocument()
-> buildParsedDocument()
-> analyzeDocument()
-> render
```

drag 입력에서는 이 방식이 너무 무겁다.
따라서 여러 셀 편집을 하나로 모은 뒤 한 번만 full rebuild하는 batch 경계를 추가했다.

```text
ScoreTextEdit[]
-> applyScoreCellRawTextBatch()
-> createRuntimeDocument()
-> buildRuntimeArtifacts()
-> render
```

이때 `ScoreTextEdit`는 아래 구조이다.

```ts
export type ScoreTextEdit = {
  selection: ScoreEditSelection;
  rawText: string;
};
```

`selection.rowKind`에 따라 note cell과 global cell의 실제 저장 위치가 달라진다.

```text
rowKind note   -> track.cells
rowKind global -> globalLines.cells
```

## 7. 전역 행 rawText 시각화

### 7.1. 문제

전역 행 입력은 내부적으로 `globalLines.cells`에 저장되고 parser에서도 읽을 수 있었다.
하지만 renderer에 전달되는 canvas item이 없었기 때문에 화면에는 보이지 않았다.

### 7.2. 구현 흐름

이번 구현에서 전역 셀 rawText 표시 흐름은 다음과 같다.

```text
ScoreFile.globalLines.cells
-> buildCanvasGlobalTextRenderItems(score)
-> CanvasGlobalTextRenderItem[]
-> renderCanvasScore()
-> drawScoreNotes(..., globalTextItems)
-> drawGlobalText()
```

전역 텍스트 item은 다음 구조이다.

```ts
export type CanvasGlobalTextRenderItem = {
  rowId: string;
  col: number;
  text: string;
};
```

renderer는 `rowId`로 global row layout을 찾고, `col + 0.5` 위치를 x 중심으로 사용한다.

```text
x = columnToX(col + 0.5)
y = row.y + row.height / 2
```

색상은 `CANVAS_COLORS.globalText = "#ffffff"`이다.

### 7.3. 표시 단계 판단

전역 텍스트는 base grid가 아니라 note overlay layer에 그린다.

이유는 다음과 같다.

- base grid는 배경과 격자 책임만 유지한다.
- 전역 rawText는 입력 결과 overlay에 가깝다.
- note/mute text와 같은 canvas layer에서 지우고 다시 그리기 쉽다.

## 8. 핵심 TypeScript 문법 설명

### 8.1. union type과 type narrowing

편집 적용 결과는 성공/실패 union으로 표현한다.

```ts
export type ScoreEditApplyResult =
  | {
      ok: true;
      score: ScoreFile;
      isDelete: boolean;
      updated: number;
    }
  | {
      ok: false;
      level: "warning" | "error";
      message: string;
    };
```

사용할 때는 `if (!applyResult.ok)`처럼 판별하면 TypeScript가 실패 타입으로 좁힌다.

```ts
if (!applyResult.ok) {
  return {
    level: applyResult.level,
    text: applyResult.message,
  };
}
```

이 방식은 예외를 남발하지 않고, 사용자에게 보여줄 warning/error를 함수 반환값으로 유지할 수 있게 한다.

### 8.2. `as const`

일부 edit rawText 합성에서는 문자열 literal 타입을 유지하기 위해 `as const`를 사용한다.

```ts
{
  kind: "apply" as const,
  rawText: "/&",
}
```

`as const`가 없으면 TypeScript가 `"apply"`를 넓은 `string`으로 추론할 수 있다.
그러면 `kind` 기반 union 판단이 약해진다.

### 8.3. optional chaining

전역/트랙 cell 조회처럼 없을 수 있는 값을 다룰 때 optional chaining을 사용한다.

```ts
const cell = trackCellMap?.get(`${selection.rowId}|${selection.col}`);
return cell?.rawText ?? "";
```

`?.`는 왼쪽 값이 `null` 또는 `undefined`이면 뒤 호출을 실행하지 않고 `undefined`를 반환한다.
`??`는 왼쪽 값이 `null` 또는 `undefined`일 때만 기본값을 사용한다.

### 8.4. `Map`

drag edit에서는 같은 셀을 여러 번 지나갈 수 있다.
중복 적용을 피하기 위해 key 기반 `Map`을 사용한다.

```ts
dragState.edits.set(getScoreTextEditKey(edit), edit);
```

같은 key가 다시 들어오면 이전 edit이 덮어써진다.
따라서 최종 pointerup 시에는 셀 하나당 edit 하나만 남는다.

### 8.5. default parameter

hit-test의 근접 보정 옵션은 기본값을 빈 객체로 둔다.

```ts
export function hitTestScoreCell(
  event: MouseEvent,
  stage: HTMLElement,
  layout: CanvasScoreLayout,
  options: ScoreHitTestOptions = {},
): ScoreHit | null
```

기존 호출부는 네 번째 인수를 넘기지 않아도 그대로 동작한다.
편집 pointer 처리만 선택적으로 `nearestNoteSlopPx`를 넘긴다.

## 9. 검증

이번 구현 구간에서 사용한 주요 검증 명령은 다음이다.

```bash
npm run typecheck
npm run test:edit
npm run test:audio
npm run test:analyze
npm run build
```

각 명령의 목적은 다음과 같다.

- `typecheck`

## 10. 보고서 / 발표용 요약

이번 단계의 핵심 성과는 다음과 같다.

1. analyzer 결과가 실제 Web Audio scheduler로 연결되었다.
2. 악보 왼쪽 경계를 기준선으로 삼는 재생 스크롤 흐름이 구현되었다.
3. 긴 악보 테스트를 위해 click cycle, drag edit, right drag delete, note row hit 보정을 추가했다.
4. note/global cell을 batch로 수정하고 한 번만 full rebuild하는 편집 경계를 만들었다.
5. 전역 행 rawText가 화면에 표시되어 tempo / meter / dynamics 입력 검증이 가능해졌다.

발표에서 강조할 수 있는 구조적 포인트는 다음이다.

- parser는 rawText를 의미 구조로 바꾼다.
- analyzer는 note / mute / gliss / tuplet / timing event를 만든다.
- renderer는 analyzer 또는 ScoreFile에서 온 DTO만 그린다.
- audio는 analyzer 결과를 schedule로 바꾸고 Web Audio clock에 예약한다.
- edit mode는 ScoreFile을 직접 수정한 뒤 전체 pipeline을 다시 통과시킨다.

## 11. 남은 작업

이번 구현 이후 남은 주요 작업은 다음이다.

- 전역 행 tempo 변화의 복수 segment 분석과 audio scroll mapping 반영
- loop playback range 지정과 반복 재생
- gliss pitch ramp audio 구현
- vibrato / tremolo Web Audio automation 구현
- dynamics timeline을 실제 gain 값에 반영
- Tone.js 또는 sample 기반 backend 후보 검토와 adapter 설계
- drag edit undo / pending edit queue 설계
- tuplet slot drag 수정과 복잡한 tuplet hit-test 보정
- 긴 악보에서 full rebuild 비용을 줄이는 partial rebuild 또는 virtualization 검토
