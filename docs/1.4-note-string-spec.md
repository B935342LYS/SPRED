# 노트 문자열 문법 설계

## 1. 용어와 원칙

### 1.1. 용어 정의

- `셀`
  - 악보의 한 좌표에 해당하는 입력 단위이다.
  - 악보의 시각적 형태는 스프레드시트와 유사한 테이블이며, 셀은 그 한 칸에 해당한다.
  - 셀에 저장되는 문자열 원본을 `rawText`로 정의한다.
  - 파서가 셀 문자열을 해석하여 생성하는 중간 산출물을 `ParsedCell`로 정의한다.

- `노트 문자열`
  - `note` 행 셀에 입력되는 문자열이다.
  - `default_text`와 각종 토큰의 조합으로 구성된다.
  - 본 문서에서는 이를 구성하는 토큰들을 `노트 행 토큰`으로 칭한다.

- `노트`
  - 노트 문자열에서 유추되는 소리 패턴이다.

- `이벤트`
  - 하나 이상의 노트의 집합으로 형성되는 음악적으로 의미 있는 최소 단위이다.

- `전역 문자열`
  - `global` 행 셀에 입력되는 문자열이다.
  - 숫자와 선형 변화 토큰으로 구성된다.

- `토큰`
  - 문자열 안에서 특별한 의미를 가지는 기호 또는 표기이다.

- `문법 오류`
  - 문자열의 구조가 형식과 불일치하는 경우이다.
  - 예: 닫히지 않은 괄호, 인수 개수 불일치

- `의미 오류`
  - 형식은 맞지만 semantic 규칙을 위반한 경우이다.
  - 예: 음악적으로 연주 불가능한 경우, 의미 중복으로 금지된 경우

### 1.2. 허용되지 않는 입력의 처리 방침

- 입력 거부
- 경고 출력 후 무효화
- 자동 정규화

## 2. 토큰 목록

세부사항은 종합명세서에 정리되어 있으므로, 본 문서에서는 문법 설계에 필요한 수준으로만 요약한다.

### 2.1. 노트 행 토큰

| 토큰 | 이름 | 의미 |
|---|---|---|
| `default_text` | 기본 텍스트 | 노트 위에 출력되는 기본 텍스트. 노트 표시 텍스트 또는 사용자 지정 텍스트 |
| `//` | 무음 토큰 | 뒤에 오는 문자열을 토큰으로 해석하지 않고 주석처럼 처리한다 |
| `-` | 타이 토큰 | 바로 앞에 음이 존재하며 동일 피치일 경우 길게 잇는다 |
| `~` | 비브라토 타이 토큰 | 타이 효과와 비브라토 효과를 함께 부여한다 |
| `@g(id, kind)` | 글리산도 토큰 | 동일 `id`의 셀을 `kind: start, mid, end` 순으로 부드럽게 잇는다 |
| `@t(div_num)` | 트레몰로 토큰 | 한 셀의 음을 `div_num` 토막으로 끊어 떨리듯 연주한다 |
| `/n(a_1|...|a_n)` | 잇단음표 헤드 토큰 | 잇단음표 생성에 사용한다. `n`개의 분할음에 `a_1 ~ a_n`의 노트 문자열을 대입한다 |
| `/&` | 잇단음표 연장 토큰 | 잇단음표 헤드와 연결되어 전체 길이 계산에 이용된다 |
| `@n(midi_num)` | 위치 지정 토큰 | 셋잇단 분할음의 렌더링 위치를 MIDI 기준으로 명시한다 |
| `@p(midi_num)` | 실제 음정 토큰 | 렌더링 위치와 실제 음정이 분리될 때 실제 음정을 명시한다 |
| `@m(cent_num)` | 미분음 토큰 | cent 단위의 미세 피치 조정을 지정한다 |

### 2.2. 전역 행 토큰

| 토큰 | 이름 | 요약 |
|---|---|---|
| `number` | 숫자 | BPM, beatsPerBar, stepsPerBeat, dynamics 값을 지정한다. BPM 행은 실수값을 허용하고, 나머지는 정수만 허용한다 |
| `<`, `>`, `><` | 선형 변화 토큰 | 숫자 뒤에 붙어 두 지점 간 선형 변화를 지시한다. 각각 시작, 끝, 끝내고 새로 시작을 의미한다 |

## 3. 노트 문자열의 EBNF 문법 초안

- 아래는 노트 표기법의 문법 초안이다.
- 의미적 제약조건은 다음 장에서 별도로 다룬다.
- 약식 표현을 위해 일부는 엄밀한 EBNF보다 읽기 쉬운 표기법을 우선한다.
- 목표는 노트 또는 분할음 문자열 길이가 L일때 복잡도를 O(L)로 하는 것이다.

```ebnf
<note>          ::= <mute_text>
                |   <note_text>
                |   <plet_head>
                |   <plet_extend>
                ;

<mute_text>     ::= "//" <custom_text>
                ;

<note_text>  ::= [<hold>] [<default_text>] [<gliss>] [<trem>] [<absolute_pitch>] [<micro_pitch>]
              ;


<default_text>  ::= <pitch_text>
                |   <custom_text>
                ;

<pitch_text>    ::= <note_name> [<accidental>] <octave> ;

<note_name>     ::= "A" | "B" | "C" | "D" | "E" | "F" | "G" ;
<accidental>    ::= "＃" | "♭" ;
<octave>        ::= "1" | "2" | "3" | "4" | "5" | "6" | "7" | "8" | "9" ;

<custom_text>       ::= { <custom_char> } ;
<custom_char>       ::= <plain_char> | <escaped_char> ;
<escaped_char>      ::= "\" <escapable_char> ;
<escapable_char>    ::= "\" | "/" | "@" | "|" | "(" | ")" | "-" | "~" ;
<plain_char>        ::=  any char except "\" , "/" , "@", "|" , "(" , ")" , "-" , "~" ;


<plet_head>   ::= "/" <plet_num> "(" <plet_slot> { "|" <plet_slot> } ")" ;
<plet_extend> ::= "/&" ;

<plet_num>    ::= 2 | 3 | 4 | ... | plet_max ;
<plet_slot>  ::= [<hold>] [<default_text>] [<gliss>] [<trem>] [<absolute_pitch>] [<micro_pitch>] <position> ;
<position>    ::= "@n(" <midi_num> ")" ;


<hold>  ::= "-" | "~" ;


<modifier>  ::= <gliss> | <trem> | <absolute_pitch> | <micro_pitch> ;

<trem>    ::= "@t(" <div_num> ")" ;
<div_num> ::= 2 | 3 | 4 ;

<gliss>         ::= "@g(" <gliss_id> "," <gliss_kind> ")" ;
<gliss_id>      ::= "a" | ... | "z" ;
<gliss_kind>    ::= "start" | "mid" | "end" ;

<absolute_pitch> ::= "@p(" <midi_num> ")" ;
<micro_pitch>    ::= "@m(" <cent_num> ")" ;


<midi_num>  ::= 0 | ... | 127 ;
<cent_num>  ::= finite decimal number in [-100, 100] with at most 1 fractional digit ;

```

## 4. 문법 해설 및 셀 단위 제약조건 초안

### 4.1. 기본 원칙

- 하나의 셀은 `mute_text`, `note_text`, `plet_head`, `plet_extend` 중 하나로 해석된다.
- 빈 문자열은 빈 셀로 간주하며, 문법 분석 대상에서 제외한다.
- note 셀에서 공문자열은 허용하지 않는다.
- `note_text`는 최소한 `hold`, `default_text`, 수식어 중 하나를 반드시 포함해야 한다.
- 수식어만 있는 note 셀도 허용한다. 예: `@t(3)`, `@g(a,start)`, `@p(64)`, `@m(10)`
- 수식어만 있는 note 셀의 렌더링 위치와 기본 음정은 해당 셀의 `rowId`를 기준으로 해석한다.
- 토큰끼리는 붙여 쓴다.
- UI를 통한 일반 입력에서는 각 토큰을 개별 상태로 관리한 뒤, 저장 시 정해진 순서로 조합하여 `rawText`를 생성한다.
- 따라서 정상적인 UI 입력 경로에서는 별도의 문자열 재정렬 정규화가 필요하지 않다.
- 다만 외부 파일 import, JSON 직접 수정처럼 UI를 거치지 않은 입력은 문법 검사를 별도로 수행해야 한다.
- 문법에 맞지 않거나 토큰 순서 규칙을 위반한 문자열은 문법 오류로 처리한다.

### 4.2. 주석 셀 규칙

- `//`로 시작하는 문자열은 주석 셀로 해석한다.
- 주석 셀은 음악적 발음을 가지지 않는다.
- `//` 뒤에 오는 텍스트는 표시용 문자열로 취급한다.
- 주석 셀에서 노트 토큰의 허용 범위와 무시 규칙은 후속 확정이 필요하다.

### 4.3. 수식어 통합 규칙

- 하나의 문자열에서 동일한 수식어의 중복은 허용하지 않는다. (최대 1회)
- 일반적인 UI 입력에서는 수식어를 개별 상태로 관리하므로, 저장되는 문자열의 수식어 순서는 항상 고정된다.

### 4.3.1. 타이와 비브라토 규칙

- `-`와 `~`는 동시에 독립 의미로 공존할 수 없다.
- `-~`, `~-`와 같은 입력은 문법 오류로 처리한다.
- `@t(div_num)`과 `~`는 동시에 허용하지 않는다.

### 4.3.2. 글리산도 규칙

- `@g(id, kind)`의 `kind`는 `start`, `mid`, `end` 중 하나여야 한다.
- 하나의 셀에 동일한 (id, kind)를 중복해서 두는 것을 금지한다.
- 연결 상대의 존재 여부는 분석 단계에서 판단한다.

### 4.3.3. 트레몰로 규칙

- `@t(div_num)`의 `div_num`은 `2`, `3`, `4`만 허용한다.
- 허용 범위를 벗어난 값은 의미 오류이다.
- `@t(div_num)`과 `~`는 동시 사용 불가이다.

### 4.3.4. 실제 음정 및 미분음 규칙

- `@p(midi_num)`은 0~127 범위의 정수만 허용한다.
- `@m(cent_num)`은 -100~100 범위의 유한 실수만 허용하며, 소수점 이하 자릿수는 최대 1자리까지 허용한다. 0은 변화가 없다는 뜻이므로 분석 단계에서 제거해도 무방하다.

### 4.6. 잇단음표 규칙

- `/n(a_1|...|a_n)`에서 실제 slot 개수는 반드시 `n`과 일치해야 한다.
- 빈 slot은 허용하며 쉼표 취급한다(해당 분할음의 위치는 비우는 것으로 한다).
- 내용이 있는 slot은 반드시 실제 위치 토큰 @n을 포함한다.
- nested tuplet은 1차 구현에서 금지한다.
- `/&`는 단독으로 음악적 의미를 가지지 않는다.
- 앞선 헤드와 연결되지 않은 `/&`는 무효 입력 또는 무음 처리 후보이다.

### 4.7. 위치 지정 토큰 규칙

- `@n(midi_num)`은 잇단음표 내부 분할음에서 렌더링 위치를 지정한다.
- `@n()`은 `rowId`가 아니라 `midi_num`만 기록한다.
- 해석 시 현재 헤드 셀의 `rowId`가 속한 `stringId`를 상속받아, 같은 `stringId` 내부에서 `midi_num`과 일치하는 `note` 행을 찾는다.
- 해당 `stringId` 내부에 대응하는 `note` 행이 없으면 의미 오류이다.

### 4.8. 기본 텍스트 규칙

- 기본 텍스트는 토큰과 충돌하지 않는 일반 문자열이어야 한다.
- UI 입력 단계에서는 기본 텍스트를 `pitch_text`와 사용자 지정 `custom_text`로 구분할 수 있다.
- 파싱 단계에서 `pitch_text`는 의미 단위로 분해할 필요가 없다. 자세한 사항은 파서 산출물 설계 문서를 참고.
- 실제 음정은 셀의 행 정보, tuplet 내부의 `@n(midi_num)`, 또는 `@p(midi_num)`에 의해 결정한다.
- 길이 제한은 종합명세서 기준을 따른다.

### 4.9. 커스텀 텍스트 규칙

- `custom_text`는 기본적으로 일반 문자들의 연속이다.
- 토큰으로 해석될 수 있는 문자 `/`, `@`, `|`, `(`, `)`, `-`, `~`, `\` 를 일반 문자로 쓰고 싶을 때는 앞에 역슬래시 `\`를 붙여 escape한다.
- 예: `\@`, `\/`, `\|`, `\(`, `\)`, `\-`, `\~`, `\\`
- 파서가 `custom_text`를 구성할 때 escape 시퀀스는 실제 문자 1개로 복원한다.
- 정의되지 않은 escape 시퀀스는 문법 오류로 처리한다.

- escape 시퀀스의 복원은 `custom_text`를 읽는 과정 내부에서만 수행한다.
- escape로 복원된 문자는 `custom_text`의 값으로 확정되며, 이후 토큰으로 다시 해석하지 않는다.
- 따라서 `\@`, `\/`, `\|` 등은 토큰이 아니라 일반 문자 `@`, `/`, `|`로 저장된다.

## 5. 검토 메모

- `@n(midi_num)`은 현재 셀의 소속 `stringId`를 상속받는 문맥 의존 토큰으로 본다.
- 따라서 순수 파서 단계에서는 `midi_num`만 기록하고, 실제 `rowId` 해석은 후속 resolver 또는 analyzer에서 수행하는 편이 자연스럽다.
- 전역 행 문자열의 문법은 별도 장 또는 별도 문서에서 다루는 편이 적절하다.
