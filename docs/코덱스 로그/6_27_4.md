제가 이해한 현재 방향은 다음과 같습니다.

노래방 모드, 즉 내부 구현명 `gameMode` / UI 명칭 `practice mode`는 기존 악보 데이터나 parser/analyzer 의미를 바꾸지 않는 별도 런타임 계층입니다. 진실 원천은 `ScoreFile.rawText`가 아니라 이미 분석된 `AnalysisResult`이며, 마이크 입력, pitch frame, 점수, 판정 결과는 score JSON에 저장하지 않습니다.

현재 구현 상태는 2차 단계까지 와 있습니다.

- `practice mode` UI와 상태 전환이 있음
- 마이크 권한 요청 후 `pitchy`로 실시간 pitch detection 수행
- `GamePitchFrame` 생성
- clarity / RMS / layout 기반 frequency range 필터 적용
- 유효 pitch만 초록색 pitch dot으로 표시
- 현재 active track의 재생 기준선 근처 `NoteEvent`를 target 후보로 사용
- target 후보를 기준으로 입력 음정을 “옥타브 관계일 때만” 보정해서 dot 위치를 표시

입력 음정 보정의 현재 의미는 “감지된 pitch를 무조건 악보 target에 끌어붙이는 것”이 아니라, 같은 pitch class로 볼 수 있는 경우에만 target 옥타브 주변으로 접는 처리입니다. 예를 들어 target이 C4이고 사용자가 C5를 내면 C4 근처로 접어서 표시하지만, target이 C4인데 입력이 D5나 C#5처럼 다른 계이름이면 보정하지 않고 원 입력 pitch를 유지합니다. 이 정책은 [game_pitch_math.ts](./regression-code/src/app/game/game_pitch_math.ts)의 `resolveClosestPitchClassCandidateMidi()`에 들어 있습니다.

현재 보정 흐름은 대략 이렇습니다.

```text
마이크 입력
-> pitchy frequency / clarity
-> RMS + clarity + frequency range 필터
-> GamePitchFrame
-> 현재 화면 playback 기준 tick 계산
-> tempo map 기준 +100ms lookahead
-> active track NoteEvent 후보 수집
-> pitch class 오차가 100 cent 미만이면 target 주변 옥타브로 보정
-> renderer layout row y 좌표로 변환
-> 초록색 pitch dot 표시
```

중요한 점은 이 보정이 아직 점수 판정 로직이 아니라 시각화 안정화에 먼저 쓰이고 있다는 점입니다. 문서상 다음 단계는 `GameJudgeTarget` 생성, 현재 score time target lookup, `Perfect / Ok / Bad / Miss` 판정, 평균 정확도와 점수 누적입니다. 이때도 같은 철학을 유지하면, 입력 음정 보정은 판정에서도 “옥타브 허용 모드”를 구현하는 핵심 함수가 됩니다.

제가 보기엔 앞으로 보강할 수 있는 “입력 음정 보정”의 핵심 후보는 이 범위입니다.

- target 후보가 여러 개일 때 가장 가까운 pitch class target 선택을 점수 판정과 공유
- pitch dot용 보정과 scoring용 보정을 같은 helper로 정리
- 최근 pitch frame smoothing / median filter 추가
- 불안정한 frame이 target 주변으로 튀어 붙지 않도록 hysteresis 또는 이전 안정 pitch 기준 추가
- 현재 `100 cent` 후보 한계를 수동 테스트 후 조정 가능하게 상수화
- 무음/불인식은 `Miss`가 아니라 “판정 없음”으로 유지

즉 현재 구현은 “악보 음정을 기준으로 마이크 입력을 보정해서 더 읽기 쉬운 노래방 표시를 만든 단계”이고, 다음 작업은 이 보정 정책을 판정/점수 산정으로 확장하는 단계로 이해했습니다.

**Hysteresis 설명**
현재 보정은 매 frame마다 “현재 target 후보 중 pitch class가 가까운 것”을 즉시 고릅니다. 그래서 입력 pitch가 흔들리거나 detector가 순간적으로 한 옥타브 위/아래를 잡으면 dot이 바로 튈 수 있습니다.

hysteresis는 “한번 선택한 보정 기준을 쉽게 바꾸지 않는 장치”입니다. 예를 들면:

- C4 target에 대해 C5 입력이 안정적으로 들어오면 `C4 주변으로 접기` 상태를 잠금
- 다음 frame에서 detector가 잠깐 C6나 B4 쪽으로 튀어도 바로 바꾸지 않음
- 일정 시간, 예를 들어 150-250ms 이상 계속 벗어나거나 더 확실한 target이 몇 frame 연속 나오면 그때만 잠금 변경
- target 변경도 “새 후보가 기존 후보보다 50cent 이상 더 낫고 2-3 frame 연속 유지될 때”처럼 조건을 둠

적용 형태는 세 단계로 나눌 수 있습니다.

1. target lock hysteresis
현재 target note/event를 잠그는 방식입니다. 재생 기준선 근처 target이 C4라면 잠깐 다른 후보가 보여도 C4 기준을 유지합니다. 게임 판정에도 가장 직접적으로 유용합니다.

2. octave fold hysteresis
“C5를 C4로 접는다”는 보정 상태 자체를 유지합니다. 같은 pitch class 입력이 계속 들어오는 동안에는 detector의 옥타브 튐을 무시합니다.

3. visual smoothing
최종 표시 y 좌표만 부드럽게 이동시킵니다. dot 흔들림은 줄지만 판정 로직 안정화에는 덜 직접적입니다.

제 추천은 `1 + 2`를 먼저 적용하는 것입니다. `3`은 보기 좋게 만드는 보조 처리라서, 판정 안정화 목적이면 우선순위가 낮습니다.

**오픈소스 참고**
현재 바로 참고할 만한 후보는 있습니다. 다만 “우리 웹 앱에 그대로 복사해 넣을 수 있는 pitch correction 보정 알고리즘”은 많지 않고, 대부분은 음성 자체를 보정해서 출력하는 Auto-Tune 계열입니다. 우리는 음성을 변조하지 않고 “입력 pitch를 악보 target과 비교하기 좋게 보정”하는 쪽이라 개념 참고가 적절합니다.

- `pitchy`: 현재 이미 쓰는 JS pitch detection 라이브러리입니다. McLeod Pitch Method 기반이고 실시간 tuner 용도를 목표로 합니다. 지금 구조에는 계속 적합합니다. 출처: https://github.com/ianprime0509/pitchy
- `TarsosDSP`: YIN, McLeod Pitch Method, Dynamic Wavelet 등 여러 pitch detection 알고리즘이 있는 Java DSP 라이브러리입니다. 다만 GPL 계열이라 직접 코드 차용은 피하고 알고리즘 비교 참고용이 안전합니다. 출처: https://github.com/JorenSix/TarsosDSP
- `Autotone`: 웹 기반 vocal pitch correction 앱입니다. CREPE pitch detection + Fourier 기반 pitch shifting 흐름이라 “웹에서 pitch 보정 파이프라인을 어떻게 나누는지” 참고할 수 있습니다. 우리 기능과 달리 실제 오디오를 보정하는 쪽입니다. 출처: https://github.com/alexcrist/autotone
- `Autotalent`: 실시간 pitch correction plugin 계열입니다. 허용 note에 pitch를 끌어붙이는 구조가 개념적으로 유사하지만, GPL 계열 포트가 많아 직접 사용보다는 정책 참고용입니다. 출처: https://tombaran.info/autotalent.html

결론적으로, 지금 작업에는 새 라이브러리를 추가하기보다 현재 `pitchy` 출력 위에 `target lock + octave fold hysteresis`를 직접 구현하는 편이 가장 맞아 보입니다. 오픈소스는 pitch detection 품질 비교와 “target note snap 정책” 참고 자료로 쓰는 게 안전합니다.

1. `clarity >= 0.75`가 높을 수 있습니다.

> 마이크의 입력 지연은 어떻게 고려하면 됩니까?

마이크 입력 지연은 “사용자가 낸 소리가 앱의 `GamePitchFrame`으로 들어오는 시각이 실제 재생 시각보다 늦다”는 문제로 보면 됩니다. 지금 구조에서는 이 지연을 아직 보정하지 않고, `scoreSeconds` 현재값과 최신 pitch frame을 바로 비교하고 있습니다. 그래서 실제로 맞게 연주해도 판정 시점에는 조금 늦은 입력으로 보일 수 있습니다.

권장 방식은 `inputLatencyMs`를 game runtime 설정으로 두고, 판정할 때만 시간을 보정하는 것입니다.

```text
판정용 입력 시각 = 현재 scoreSeconds - inputLatencyMs
```

또는 구현 관점에서는 pitch frame에 `capturedAtMs`가 있으므로 더 정확히는:

```text
frameScoreSeconds = playbackScoreSeconds - estimatedMicLatencySeconds
```

첫 구현은 단순하게 고정값으로 시작하면 됩니다.

- 기본값: `80ms` 또는 `100ms`
- UI에서 조정 가능 범위: `0ms ~ 250ms`
- step: `10ms`
- 표시명: `Mic delay`
- ready 상태에서만 조정 허용

주의할 점은 “YouTube offset”과 “마이크 입력 지연”은 다른 값이라는 점입니다.

- YouTube offset: 영상/반주가 score playback과 맞는지 조정
- Mic delay: 사용자의 실제 연주 입력이 pitch detector에 늦게 도착하는 것을 보정

현재 scoring sample이 100ms 간격이므로, 마이크 지연이 50-120ms만 있어도 판정이 흔들릴 수 있습니다. 특히 짧은 note에서는 더 크게 체감됩니다.

제가 추천하는 다음 구현은 이 순서입니다.

1. `GamePitchFrame`에 `scoreSeconds` 또는 판정용 timestamp를 추가
2. 최근 pitch frame history를 500ms 정도 보관
3. scoring 시점에 `scoreSeconds - micDelayMs`에 가장 가까운 voiced frame을 선택
4. practice panel에 `Mic delay` slider 추가
5. 기본값은 `100ms`

이 방식이 최신 frame만 보는 현재 구조보다 훨씬 안정적입니다. 단순히 현재 score time을 뒤로 미는 것보다, frame history에서 해당 시점의 입력을 찾아 쓰는 편이 정확합니다.

현재 방식:
- 같은 시간에 울리는 event 수를 `N`이라고 하면, 각 event에 `gainScale = 1 / N`을 곱합니다.
- 기본 velocity/dynamics가 모두 1이라고 보면 전체 선형 gain 합은:

```text
N * (1 / N) = 1
```

즉 1개가 울리든 3개가 울리든 8개가 울리든, 이론상 총합은 1 근처로 눌립니다.

같은 pitch끼리만 보정하는 방식:
- 같은 시점의 발음을 pitch별 그룹으로 나눕니다.
- 어떤 pitch 그룹에 `k`개가 겹치면 그 그룹 안의 각 event만 `1 / k`로 낮춥니다.
- 서로 다른 pitch는 `k = 1`이라 gainScale이 `1` 그대로입니다.

따라서 pitch 그룹 수를 `M`이라고 하면 전체 선형 gain 합은:

```text
Σ 각 pitch 그룹의 합 = M
```

예시:

| 동시 발음 | pitch 그룹 | 각 event scale | 전체 gain 합 |
|---|---:|---|---:|
| C 하나 | 1 | C: 1 | 1 |
| C + C | 1 | C들: 1/2 | 1 |
| C + E | 2 | C: 1, E: 1 | 2 |
| C + E + G | 3 | 전부 1 | 3 |
| C + C + E | 2 | C들: 1/2, E: 1 | 2 |
| C + C + E + G + G | 3 | C들: 1/2, E: 1, G들: 1/2 | 3 |

기본 volume이 현재처럼 master에서 `volume * volume`이고 기본값이 `0.7`이면 master gain은 `0.49`입니다. 그래서 기본 조건에서는 대략:

```text
현재 방식: 0.49 * 1 = 0.49
같은 pitch만 보정: 0.49 * pitch 그룹 수 M
```

즉 서로 다른 3화음이면 현재보다 이론상 약 3배 큰 선형 합이 됩니다. 다만 실제 청감/피크는 주파수와 위상 때문에 항상 선형 합 그대로 들리지는 않습니다. 그래도 클리핑 위험과 체감 볼륨 증가는 분명히 커집니다.

정리하면, “같은 pitch 중복만 보정”은 전체 볼륨 정규화가 아니라 **동일음 중복 방지**에 가깝습니다. 화음은 의도적으로 더 커지게 됩니다. 구현한다면 추가로 전체 상한을 `sqrt(M)` 또는 `1 / sqrt(M)` 계열로 살짝 누를지 여부를 같이 정하는 편이 좋습니다.
