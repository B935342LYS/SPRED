# 1.48 Practice Attack Credit Spec

## 1. Scope

이번 메모는 practice mode timing anti-cheat의 attack credit / score eligibility 설계와 1차 구현 결과를 기록한다.

핵심 결정:

- no-onset을 직접 `Bad` 또는 `Miss`로 강등하지 않는다.
- pitch label과 score eligibility를 분리한다.
- 새 attack이 필요한 note에서 attack이 확인되지 않으면 label은 유지하되 score와 combo 지급을 막는다.

## 2. Problem

같은 pitch가 반복되는 구조에서는 한 번만 누르고 있어도 뒤 note들이 pitch 기준 `Perfect`로 표시될 수 있다.

하지만 no-onset을 곧바로 timing 실패로 처리하면 다음 정상 구조가 과도하게 실패한다.

- gliss end anchor
- gliss end 직후 같은 pitch long/vib note
- hold sustain
- vibrato hold
- 같은 pitch legato
- 짧은 tuplet 또는 같은 pitch 연속 note

따라서 `onset 없음 = 틀림`으로 판단하는 방식은 현재 pitch detector와 analyzer 문맥에서는 너무 공격적이다.

## 3. Direction

현재 구현은 `GameScoringSampleResult.label`을 강등하기보다 `scoreEligible`을 별도로 계산한다.

예상 효과:

- pitch가 맞으면 `Perfect` / `Ok` 피드백은 유지할 수 있다.
- attack이 필요한 note에서 attack이 없으면 scoreContribution은 `0`이 된다.
- combo는 scoreEligible이고 `Miss`가 아닌 sample에서만 증가한다.
- 정상 legato를 `Bad` / `Miss`로 잘못 표시할 위험을 줄인다.

## 4. Candidate Types

```ts
type GameJudgeTarget = {
  eventId: string;
  trackId: TrackId;
  startSeconds: number;
  endSeconds: number;
  targetMidi: number;
  targetCentOffset: number;
  attackRequired: boolean;
};

type GameAttackJudgeResult =
  | { kind: "notRequired" }
  | { kind: "satisfied"; onsetId: number }
  | { kind: "missing" };

type GameScoringSampleResult = {
  targetEventId: string;
  trackId: TrackId;
  scoreSeconds: number;
  targetMidi: number;
  targetCentOffset: number;
  pitchAccuracy: number;
  label: "Perfect" | "Ok" | "Bad" | "Miss";
  status: "hit" | "miss";
  scoreEligible: boolean;
  scoreBlockedReason: "attackRequired" | null;
  scoreContribution: number;
  timing: GameTimingJudgeResult;
  timingOnsetId: number | null;
  timingJudgedEventId: string | null;
};
```

## 5. Initial Policy

초기 `attackRequired` 구현:

| 구조 | attackRequired |
| --- | --- |
| 일반 새 `NoteEvent` | true |
| 같은 `NoteEvent` 내부 hold sustain | true로 시작하되, 한 번 attack credit을 받은 뒤 같은 event의 후속 sample은 scoreEligible 유지 |
| gliss destination note | false |
| gliss destination 직후 같은 pitch long/vib neighbor | false 후보 |
| 같은 pitch의 반복 별도 note event | true |
| tuplet 내부 짧은 same-pitch note | 보류 |

초기 scoring 후보:

```text
if label is Perfect/Ok/Bad
and attackRequired
and attack is missing:
  scoreEligible = false
  scoreContribution = 0
  combo does not increase
else:
  scoreEligible = true
  use existing pitch/timing score
```

## 6. UI Policy

첫 구현에서는 결과 화면 항목을 늘리지 않는다.

재생 중 overlay:

- `scoreEligible = true`: 기존 `Perfect` / `Ok` / `Bad` / `Miss` 표시와 Combo 표시를 유지한다.
- `scoreEligible = false`: judge overlay를 표시하지 않고 score summary에도 누적하지 않는다.
- 필요하면 후속으로 `no attack` 또는 `hold` 보조 텍스트를 작게 추가한다.

Rules dialog:

- 첫 구현 이후 `Score Formula`에 attack credit 설명을 추가한다.
- `Pitch Judge`와 `Timing Judge` 표의 accuracy 단계는 현재 구현 기준 `100% / 66.7% / 33.3% / 0%`로 통일한다.

## 7. Implementation Notes

예상 구현 위치:

- `src/app/game/game_types.ts`
- `src/app/game/game_judge.ts`
- `src/app/playback/app_playback_binding.ts`
- `src/app/game/game_judge_overlay.ts`
- `dev/test_game.ts`

1차 구현 반영:

- `GameJudgeTarget.attackRequired`를 추가했다.
- `GameScoringSampleResult.scoreEligible`과 `scoreBlockedReason`을 추가했다.
- 일반 새 `NoteEvent`는 attack이 필요하다.
- `glissRole.role === "end"`인 note target은 no-onset만으로 score를 차단하지 않는다.
- playback runtime은 onset이 note에 배정되면 해당 `eventId`를 attack credit 획득 상태로 저장한다.
- attack이 필요한 note가 attack credit 없이 pitch만 맞으면 label은 유지하되 `scoreContribution = 0`으로 두고 combo를 증가시키지 않는다.
- scoreEligible이 false인 sample은 overlay를 표시하지 않고 live/result pitch count에도 더하지 않는다.
- 결과 화면에는 별도 attack count를 추가하지 않았다.

주의:

- score JSON에는 attack judge 상태를 저장하지 않는다.
- attack state는 practice session runtime 데이터이다.
- `Bad` / `Miss` 강등은 matched onset이 있는 timing 판정에서만 유지한다.
- no-onset 자체를 final label 강등 근거로 쓰지 않는다.

## 8. Verification Plan

필수 테스트:

- 일반 반복 same-pitch note를 한 번만 누르면 뒤 note sample은 scoreEligible false가 된다.
- scoreEligible false인 `Perfect` sample은 scoreContribution이 0이다.
- scoreEligible false인 `Perfect` sample은 combo를 증가시키지 않는다.
- gliss destination note는 no-onset만으로 scoreEligible false가 되지 않는다.
- gliss destination 직후 같은 pitch long/vib neighbor는 첫 후보 구현에서 scoreEligible true를 유지한다.
- matched onset이 150ms 이상 어긋나면 기존처럼 `Bad`로 강등된다.
- matched onset이 250ms 이상 어긋나면 기존처럼 `Miss`로 강등된다.

브라우저 수동 확인:

- 같은 pitch 반복을 한 번만 눌러도 visual pitch feedback은 과도하게 실패하지 않지만 score/combo가 부풀지 않는다.
- gliss 끝 / long note 이웃 구조가 정상 legato 연주에서 `Bad` / `Miss`를 남발하지 않는다.
