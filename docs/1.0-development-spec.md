# 개발 전 알아둘 사항

## 1. 개요
 작업공간 내의 "이세계 코드"는 구글 스프레드시트와 앱스스크립트를 기반으로 동작하는 코드이다.
이를 타입스크립트, html, css를 기반으로 동작하고 깃허브 페이지로 배포될 웹 앱 "회귀 코드"로 리팩토링하고 성능 개선과 확장을 진행코자 한다.
이는 1년간 졸업 프로젝트로서 진행되는 사안이며, 따라서 이후 작성되는 모든 문서는 명세이자 설계문서인 동시에 교수님을 대상으로 한 보고서의 성격도 겸한다.

교수님의 요구사항에 따라, 1.7 이후의 구현 보고서에는 난이도 소명, challenging한 부분, 그 부분이 challenging한 이유와 구체적인 난이도 계산, 그 해결 과정이 명확히 기재되어 있어야 한다.


1. 이세계 코드와 회귀 코드는 별개이다. 회귀 코드는 이세계 코드와의 호환을 목표로 하지 않는다.
2. 이세계 코드는 알고리즘과 기능 흐름을 참고하기 위한 구버전 구현체로만 취급한다.
3. 회귀 코드는 `TypeScript + HTML + CSS + GitHub Pages` 기반 웹 앱으로 재구성한다.
4. 설계 판단이 충돌할 경우, 이세계 코드나 이세계 코드 명세보다 회귀 코드 명세를 우선한다.
5. 구현에 들어가기 전에는 설계 문서를 우선 정리하고, 현재 단계와 다음 단계를 `1.0` 문서에 기록한다.
6. 내용 추가/수정시 파일을 바로 수정하지 않고 변경 이유와 방향을 먼저 설명한다. 이후 개발자가 직접 내용 수정을 진행하며, 파일 직접 수정은 요청시에만 실시한다.
7. 한 단락의 작업이 끝나면 HARNESS와 본 문서를 갱신한다.

- 표, 이미지, 세부 서식은 Markdown 변환 과정에서 일부 단순화될 수 있다.

## 3. 구현 로드맵

현재는 설계 고정 단계를 지나 실제 구현에 들어간 단계이다. 아래 순서를 기준으로 진행한다.

1. 요구사항과 범위 고정
   - 회귀 코드가 지원할 목표 기능과 비목표를 정리한다.
   - 이세계 코드와의 비호환 원칙, 단일 현 우선 원칙, 확장 방향을 고정한다.

2. 아키텍처 방향 정리
   - Presentation / Application / Core / Infrastructure 계층을 정리한다.
   - 브라우저 실행 구조와 정적 배포 구조를 정리한다.

3. 악보 JSON 데이터 구조 확정
   - 문서 포맷, 메타데이터, 악기 정보, 전역 행, 레이아웃, 트랙 구조를 고정한다.
   - 참조 무결성과 검증 규칙을 정리한다.

4. 노트 문자열 문법 확정
   - note 셀과 global 셀의 토큰 규칙을 정리한다.
   - 문법 오류와 의미 오류의 경계를 구분한다.

5. 파서 중간 산출물 구조 확정
   - `ParsedCell` 및 하위 구조를 고정한다.
   - parser가 책임질 범위와 analyzer가 책임질 범위를 분리한다.

6. 파싱 알고리즘 수도 코드 정리
   - `parseMuteText`, `parseNoteText`, `parsePletHead`, `parsePletExtend` 절차를 작성한다.
   - escape 규칙, 토큰 순서, 충돌 규칙을 parser 단계에서 어디까지 검증할지 명시한다.

7. analyzer / renderer / audio 입력 인터페이스 정리
   - parser 출력이 후속 모듈에서 어떤 형태로 소비될지 정의한다.
   - 필요한 추가 산출물과 이벤트 리스트 구조를 정리한다.

8. parser / analyzer 최소 구현
   - full parse 기준 구현을 먼저 안정화한다.
   - analyzer는 full analysis 최소 골격부터 순차적으로 붙인다.

9. 증분 갱신과 UI/오디오 확장
   - partial parse / partial analysis를 후순위로 붙인다.
   - renderer / audio generator / 앱 계층을 단계적으로 연결한다.

## 5. 작업 원칙

- 구현 단계에서도 활성 문서를 우선 확인한 뒤 착수한다.
- 모호한 부분은 질문을 통해 먼저 확정하고, 답변으로 기준이 고정된 뒤 구현에 들어간다.
- 문법 오류는 parser 단계에서 최대한 판정한다.
- 셀 단독으로 판단할 수 없는 사항만 validator / analyzer 단계로 넘긴다.
- UI 입력 경로와 외부 import 경로를 분리해서 생각한다.
- JSON에는 `rawText`를 저장하고, `parsedCell`은 런타임 산출물로 유지한다.

## 6. 메모

- `1.1-project-plan.md`와 `1.2-master-spec.md`는 반복 대화용 참조본이다.
- DOCX 기반 원본 내용은 유지하되, 이후 실작업 참조는 가능하면 Markdown 사본을 우선 사용한다.
- parser 단계 설계 문서는 현재 기준으로 일단락되었고, 실제 구현 반영 단계에 들어갔다.
- analyzer 이벤트 리스트 구조는 `1.7-analyzer-event-list-spec.md`에 정리하였다.
- `1.7`에는 이벤트 구조뿐 아니라 전역 timing/dynamics 세그먼트, 진단 구조, 부분 갱신 규칙까지 반영되었다.
- 전역 행은 저장 구조 누락을 확인한 뒤 `rawText` 기반 셀 구조와 parser 문서를 보강하였다.
- `1.8` 문서까지 포함해 parser-analyzer 파이프라인 인터페이스 초안은 일단락된 상태로 본다.
- 현재는 `1.0~1.8`에서 고정한 설계를 실제 TypeScript 구현으로 옮기고 정합성을 맞추는 단계이다.

## 7. 확장 메모

### 7.1. 게임 모드

- 게임 모드는 후순위 확장 기능 중에서도 최후순위 기능으로 둔다.
- GitHub Pages 배포 환경에서도 브라우저 오디오 입력 기반으로 구현 가능하다고 보지만, 안정성은 사용자 기기, 브라우저, 입력 장치 품질에 영향을 받는다.
- 따라서 1차 구현 목표는 “실용적인 연습용 판정 모드”이며, 모든 브라우저/모바일 환경에서의 완전 무버퍼 고정밀 판정을 전제하지 않는다.
- 게임 모드는 실제 사용자의 연주를 실시간 오디오 입력으로 받아 pitch detection 결과와 악보 이벤트를 비교하는 별도 평가 모듈로 설계한다.
- 판정 기준의 진실 원천은 score JSON이 아니라 analyzer가 생성한 `AnalysisResult`로 둔다.
- 1차 구현에서는 `NoteEvent` 중심 판정을 우선하고, gliss/vibrato/trem/tuplet의 정교한 연주 품질 평가는 후순위로 둔다.
- 게임 모드의 pitch detection 결과와 중간 판정 데이터는 런타임 데이터로만 취급하며 score JSON에는 저장하지 않는다.
- 처음 이벤트부터 마지막 이벤트까지의 판정을 누적하여 최종 점수와 랭크를 계산한다.
- reset, jump, seek 등 시간축을 비연속적으로 변경하는 동작이 발생하면 게임 모드는 자동 중단된다.
- 이벤트별 진입 타이밍과 음정 일치율을 종합하여 리듬게임형 판정 메시지를 표시할 수 있도록 한다.
