# 1.23. 4단계 audio 구현 보고서 / 발표 준비 초안
> 본 문서는 보고서와 발표 자료 작성을 위한 초안이다.
> 제출용 최종본이 아니라, 최근 audio 구현 내용을 설명 순서와 발표 논리로 재구성한 작업 기록이다.
> 본 문서는 active specification을 대체하지 않는다.

## 1. 작성 방향

이번 보고서와 발표는 "회귀 코드가 analyzer 결과를 실제 재생 가능한 Web Audio schedule로 변환하기 시작했다"는 점을 중심으로 설명한다.

핵심 메시지는 다음이다.

1. `rawText`를 audio module에서 다시 해석하지 않고, analyzer가 만든 `AnalysisResult`를 소비한다.
2. score tick을 실제 초 단위 score time으로 변환한 뒤 lookahead scheduler가 Web Audio clock에 예약한다.
3. 기본 note playback 위에 vibrato, tremolo, gliss fallback을 단계적으로 얹었다.
4. gliss는 전체 voice chain 병합 대신 fallback bridge로 시작해 부분 갱신과 성능 위험을 낮췄다.

## 2. 구현 범위 요약

이번 단계에서 연결된 기능은 다음이다.

- 기본 note event 재생
- Web Audio oscillator backend
- pause / resume / seek playback state
- score scroll sync
- edit mode pitch preview
- vibrato: oscillator detune LFO
- tremolo: gain gate
- gliss: separate ramp oscillator fallback bridge

아직 남은 범위는 다음이다.

- dynamics timeline 기반 gain automation
- sampled instrument backend
- gliss 전체 voice-span 병합
- 브라우저 수동 청감 테스트 확대
- loop range UI

## 3. 핵심 처리 흐름

현재 audio 흐름은 아래와 같다.

```text
ScoreFile rawText
-> parser
-> analyzer
-> AnalysisResult
-> AudioSchedule
-> AudioEventQueue
-> LookaheadScheduler
-> OscillatorBackend
-> Web Audio API
```

이 구조에서 parser와 analyzer는 음악적 의미를 확정하고, audio module은 이미 확정된 이벤트를 시간과 발음 예약 구조로 바꾼다.
따라서 audio module은 `@t`, `@g`, `~` 같은 문자열을 직접 다시 읽지 않는다.

## 4. Challenging Point 1: score tick과 실제 초 단위 시간

악보의 한 칸은 단순한 밀리초가 아니라 BPM, stepsPerBeat, linear tempo 변화에 의해 실제 길이가 달라진다.

해결 방식은 `TickTimeMapper`를 두는 것이다.

```text
AnalyzedTimeSegment[]
-> segment별 tick range / seconds range 사전 계산
-> tickToSeconds()
-> secondsToTick()
```

이 방식의 장점은 다음이다.

- renderer와 audio가 같은 analyzer timing 결과를 공유한다.
- seek 위치를 score seconds로 관리할 수 있다.
- BPM linear 변화가 있어도 segment 내부 수식만 바꾸면 전체 scheduler 구조를 유지할 수 있다.

## 5. Challenging Point 2: 정확한 발음 시각

JavaScript의 `setInterval()`은 정확한 발음 시점을 보장하지 못한다.
따라서 scheduler timer는 "가까운 미래 event를 찾는 역할"만 수행하고, 실제 발음 시각은 Web Audio clock에 맡긴다.

```text
currentScoreSeconds
-> [current, current + lookahead] 범위 조회
-> backend.scheduleEvent(event, offsetSeconds)
-> oscillator.start(audioContext.currentTime + offsetSeconds)
```

이 구조는 공개 Web Audio scheduling 자료와 Tone.js 같은 음악 라이브러리의 시간 예약 구조를 비교한 뒤 채택했다.
외부 코드를 복사하지 않고, scheduling 개념만 프로젝트 구조에 맞게 적용했다.

## 6. Challenging Point 3: vibrato와 tremolo

vibrato와 tremolo는 둘 다 "시간에 따라 값이 흔들리는 효과"지만 제어 대상이 다르다.

- vibrato: pitch 변화
- tremolo: amplitude 변화

따라서 구현도 분리했다.

```text
vibrato:
LFO OscillatorNode
-> depth GainNode
-> note oscillator.detune

tremolo:
note oscillator
-> tremolo GainNode
-> envelope GainNode
-> master GainNode
```

vibrato는 Tone.js의 LFO + depth 모델을 개념적으로 참고했다.
tremolo는 oscillator를 반복 생성하지 않고 gain gate로 처리해 scheduling 비용을 낮췄다.

## 7. Challenging Point 4: gliss 연결 방식

롱노트-글리산도-롱노트 전체를 하나의 oscillator chain으로 합치면 청감은 자연스럽다.
하지만 부분 갱신에서는 하나의 gliss 변경이 앞뒤 긴 note 전체를 다시 계산하게 만들 수 있다.

따라서 첫 구현은 fallback bridge 방식을 채택했다.

```text
existing note event
+ gliss fallback oscillator with pitch ramp
+ following note event
```

gliss fallback event는 `GlissEvent.startSound`, `endSound`, `startAnchorTick`, `endAnchorTick`을 사용해 별도 oscillator를 예약한다.
frequency는 선형 ramp로 이동하고, gain은 짧은 fade in/out으로 겹침을 완화한다.

이 선택의 장점은 다음이다.

- `GlissEvent` segment 단위로 스케줄 캐시를 나눌 수 있다.
- 넓은 gliss나 긴 앞뒤 note가 있어도 전체 voice chain 병합을 강제하지 않는다.
- 추후 청감 문제가 확인되면 note ducking이나 선택적 voice-span 병합으로 확장할 수 있다.

## 8. 검증 현황

현재 통과한 자동 검증은 다음이다.

```text
npm run typecheck
npm run test:audio
npm run test:score
npm run test:parse
npm run test:analyze
npm run test:edit
npm run build
```

`test:audio`에서는 다음을 확인한다.

- BPM / stepsPerBeat 기반 tick-to-seconds 변환
- linear BPM segment 변환과 역변환
- MIDI note number to frequency 변환
- `NoteEvent.effects`에서 vibrato / tremolo schedule 보존
- `GlissEvent`에서 gliss fallback schedule 생성
- scheduler lookahead와 loop wrap 예약
- playback controller의 play / pause / resume / seek / stop 상태 전환

## 9. 남은 테스트와 리스크

자동 테스트만으로는 audio quality를 충분히 판단하기 어렵다.
다음 항목은 브라우저 수동 테스트와 실제 악보 예제로 확인해야 한다.

- tremolo division별 청감과 click noise
- vibrato depth가 과하거나 부족하지 않은지
- gliss fallback이 앞뒤 note와 과하게 겹치지 않는지
- 매우 짧은 gliss에서 crossfade가 음을 흐리지 않는지
- 긴 롱노트-글리산도-롱노트에서 fallback 방식이 충분히 자연스러운지

## 11. 보고서 문장 초안

이번 단계에서는 회귀 코드의 audio module을 단순한 음 발생 기능이 아니라, analyzer 결과를 기반으로 한 이벤트 스케줄러로 구현하였다.
기존 이세계 코드에서는 Google Sheets와 Apps Script 구조 안에서 문자열 해석과 재생 로직이 결합되어 있었지만, 회귀 코드에서는 parser와 analyzer가 음악적 의미를 확정하고 audio module은 이를 초 단위 재생 이벤트로 변환하도록 책임을 분리하였다.

특히 Web Audio API의 `AudioContext.currentTime`을 기준으로 lookahead scheduling을 수행하여 JavaScript timer의 부정확성을 줄였다.
또한 vibrato는 oscillator detune에 LFO를 연결하고, tremolo는 gain gate로 구현하여 pitch 변화와 amplitude 변화를 서로 다른 AudioParam 경로로 분리하였다.

gliss는 전체 note chain을 하나로 병합하는 방식이 가장 자연스러울 수 있으나, 부분 갱신과 긴 note 범위 재계산 비용이 커질 위험이 있다.
따라서 첫 구현에서는 `GlissEvent`를 별도 ramp oscillator fallback으로 변환하여 start pitch에서 end pitch까지 선형 이동시키고, 짧은 fade in/out을 적용하는 절충안을 채택하였다.
이 방식은 완전한 음원 병합보다 청감상 한계가 있을 수 있지만, analyzer의 segment 단위 구조와 잘 맞으며 추후 note ducking 또는 선택적 voice-span 병합으로 확장할 수 있다.
