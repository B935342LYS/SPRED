# 1.2. 회귀 코드 종합명세서

> DOCX 원본을 바탕으로 재정리한 Markdown 참조본이다.
> 본 문서는 초기 요구사항, 아키텍처, UI 구상까지 넓게 포괄하는 상위 명세서이다.
> 다만 현재 구현 기준의 세부 문법과 타입 정의는 `1.3`, `1.5`, `1.6`, `1.7`, `1.8`을 우선한다.
> 따라서 본 문서에 남아 있는 초기 구상은 후속 active 문서와 충돌할 경우 최신 active 문서를 우선한다.

## 개요

- 초기 계획에서는 3~4월을 사전 학습에만 쓰려 했으나, 구현 범위를 고려하면 위험하다고 판단했다.
- 따라서 3~4월은 설계, 학습, 일부 쉬운 구현을 병행하는 방향으로 로드맵을 수정했다.
- 설계 순서는 대체로 다음과 같다.

1. 요구사항 명세
2. 아키텍처 디자인
3. 데이터 구조 디자인
4. 인터페이스 디자인
5. 구성 요소와 클래스 디자인

- JSON 구조가 손으로 작성하기 어려우므로, 이후 테스트를 위해 기초적인 에딧 모드도 필요하다고 본다.

## 1. 요구사항 명세

### 1.1. 악보 파일과 레이아웃 변경

#### 악보 파일에 포함할 정보

- 음악과 악보 정보
  - 곡 이름, 작곡가, 장르
  - 악보 작성자
  - 악보 난이도 리스트
  - 생성일, 수정일
  - 작성자 코멘트
  - 코멘트는 초기 구상에서 100자 제한을 염두에 두고 있었다.
- 악기 정보
  - 대응 프리셋
  - 분류, 악기명
  - 현 개수
  - 개방현 지원 여부
  - 현 튜닝
  - 음역대
  - 개방현은 선택 필드 `openMidi` 성격으로 구상하였다.
  - 지판을 눌러 연주하는 최저음은 `baseMidi` 성격으로 다루는 방안이 검토되었다.
- 초기값
  - BPM
  - `stepsPerBeat`
  - `beatsPerBar`
  - `youtube: { link, offset }`
- 레이아웃 정보
  - 전체 열 수
  - 전체 행 정보
  - 각 노트 행의 음정
  - 행 높이(px)
- 그리드 기반 셀 정보
  - 페이지
  - 트랙 id, 이름
  - 트랙 종류
  - 셀 정보

#### 레이아웃 제약과 무결성

- 하나의 현 내부에 동일한 MIDI 번호를 갖는 노트 행이 여러 개 존재해서는 안 된다.
- 레이아웃 정보는 단순 표시 정보가 아니라, 파서와 analyzer가 참조하는 음정 해석의 기준이 된다.
- view 관련 정보는 문서 저장 포맷보다는 사용자 설정에 두는 방향이 초기 구상에 포함되어 있었다.

#### 저장 책임 분리

- 악보 해석에 영향을 주는 레이아웃 정보는 악보 파일에도 저장한다.
- 사용자 편의 설정은 별도의 로컬 설정으로 저장한다.
- 사용자 설정 중 줌 배율과 마지막 스크롤 위치는 저장하지 않는다.
- 글자 크기는 사용자 설정으로 분리하되, 앱 전체의 기본 초기값은 제공한다.
- 사용자 레이아웃 커스텀 설정은 악기 프리셋별로 하나씩 저장하는 것을 1차 기준으로 한다.
- 즉 "악보 기본 레이아웃"과 "프리셋별 사용자 로컬 레이아웃"이 동시에 존재할 수 있다.

#### 레이아웃 변경 기능

- 악기 설정 변경시 해당 악기의 기본 레이아웃으로 바뀌어야 한다.
- zoom은 악보 로드시마다 1배율로 초기화한다.
- 글씨 크기는 새 악보 로드 후에도 사용자 설정을 유지한다.
- 음역대를 자유롭게 조정할 수 있어야 한다.
- 노트 행을 위아래로 추가할 수 있어야 한다.
- 노트 행 사이 갭 행 높이를 개별 조정할 수 있어야 한다.
- 사용자 변경 설정은 저장 후 다시 불러올 수 있어야 한다.
- 셀 가로 길이를 일정 범위 내에서 조정할 수 있어야 한다.
- 갭 행은 편집 불가, 나머지 행은 편집 가능으로 고정한다.
- 사용자 설정 저장 위치로는 초기 구상에서 `localStorage` 또는 쿠키가 후보였으며, 현재는 브라우저 로컬 저장소 계열을 우선 가정한다.

#### 레이아웃 충돌 처리

- 악보 파일의 기본 레이아웃과 사용자 로컬 커스텀 레이아웃이 충돌하면, 먼저 커스텀 설정을 적용할지 사용자에게 확인한다.
- 사용자가 적용을 선택하면 현재 악보를 커스텀 레이아웃에 맞게 변형한다.
- 단순한 행 높이, 셀 너비, 열 너비 차이는 렌더링 조정으로 처리한다.
- `rowId` 종류 불일치, `rowId` 개수 불일치, 필수 전역 행 누락, 삭제된 `rowId`에 셀 데이터가 존재하는 경우는 구조 충돌로 본다.
- 1차 구현에서는 이런 구조 충돌이 발생하면 import를 실패로 처리한다.
- 후속 단계에서는 보정 후 이식 알고리즘을 구현한다.
- 후속 보정 단계에서는 삭제 또는 invalid 처리 대상에 대한 경고 로그를 만들고, 사용자 확인 후 적용한다.

### 1.2. 노트 행 토큰

- 노트 셀에 들어갈 토큰 규칙을 정의한다.
- 명세서에서 간과한 조합은 모순 여부와 실제 연주 가능 여부를 기준으로 판단한다.
- 초기 DOCX에는 토큰별 설명이 UI 구상과 강하게 결합되어 있었으므로, 여기서는 그 내용을 분리하여 서술한다.
- 현재 구현 기준의 실제 표기법은 후속 active 문서의 `@g`, `@t`, `@p`, `@m`, `@n` 체계를 우선한다.

#### `//`

- 무음 또는 주석 토큰이다.
- 뒤에 오는 문자열은 음악 토큰으로 해석하지 않고 메모로 취급한다.
- 아무 음도 출력하지 않는다.
- DOCX 초기안에서는 위치 지정 토큰만 예외적으로 인식하는 방안도 검토되었으나, 최신 active 문서에서는 주석 뒤를 토큰으로 재해석하지 않는 쪽이 더 명확하다.
- 포맷 규칙 차원에서 과도한 길이를 제한한다.
- 노트 텍스트와 별도로 폰트 크기 조절 기능을 부여하는 방안도 포함되어 있었다.

#### `default_text`

- 노트 위에 기본적으로 표시되는 문자열이다.
- 계이름 + 옥타브 + 샵, 계이름 + 옥타브 + 플랫, 사용자 지정 텍스트를 지원하는 방향을 고려했다.
- 손가락 번호 등 자주 쓰는 특수 문자의 단축 입력 버튼도 고려되었다.
- 토큰과 패턴이 겹치는 문자열은 UI 레벨에서 입력을 제한하는 방향을 검토했다.

#### `-`

- 타이 또는 붙임줄 역할의 토큰이다.
- 왼쪽에 동일 음정의 셀이 존재하면 길게 잇는다.
- 왼쪽에 이어질 음이 없으면 그 자체가 시작점이 될 수 있다.
- 트레몰로와는 개념이 반대이므로 직접 결합을 허용하지 않는 초기 구상이 있었다.
- 셋잇단 내부 분할음에서도 앞뒤 조건이 맞으면 길게 잇는 것으로 구상되었다.

#### `~`

- 비브라토 타이 토큰이다.
- 기본적으로는 타이처럼 왼쪽 음과 부드럽게 이어지며, 동시에 바이브레이션 효과를 가진다.
- `-`와 동시 입력되는 경우 어떤 쪽을 우선할지에 대한 초기 구상은 있었으나, 최신 active 문서에서는 두 토큰의 동시 사용 자체를 문법적으로 금지하는 쪽이 더 일관된다.
- 트레몰로와는 동시에 허용하지 않는 방향이 초기부터 명확했다.

#### `@g(id, kind)`

- 글리산도 토큰이다.
- `id`는 같은 글리산도 체인을 식별하기 위한 식별자이다.
- `kind`는 `start`, `mid`, `end`이다.
- 동시점에 여러 글리산도가 겹칠 가능성을 고려해 식별자를 두었다.
- 식별자 범위는 알파벳 정도면 충분하다고 보았다.
- 하나의 열에 동일 식별자가 여러 개 들어오는 입력은 제한하는 방향을 검토했다.
- 트레몰로와 조합 가능, 시작점 글리산도와 타이의 조합 가능 등 복합 규칙도 초기부터 고려되었다.

#### `@t(div_num)`

- 트레몰로 토큰이다.
- 셀 길이를 `div_num`개로 나누어 끊어 연주하는 효과를 나타낸다.
- `div_num`은 2, 3, 4 정도로 제한하는 방향을 구상했다.
- 뒤에 타이가 이어지면 효과도 지속 연장하는 방향을 생각했다.
- 구현 방식은 실제 오실레이터 on/off 대신 볼륨 조절로 연출해도 충분하다고 보았다.

#### `/n(a_1|...|a_n)`

- 잇단음표 헤드 토큰이다.
- `n`분할을 의미하며, 내부의 각 `a_i`는 하나의 완성된 노트 문자열로 본다.
- 괄호 안의 분할음 개수는 반드시 `n`과 일치해야 한다.
- 빈 문자열 분할음은 쉼표로 취급하는 방향이 포함되어 있었다.
- nested tuplet은 1차 구현에서 금지하는 것이 적절하다고 판단했다.
- 헤드 토큰 자체는 음악적 의미를 가지지 않고, 내부 분할음이 실제 음악 의미를 가진다.

#### `/&`

- 잇단음표 연장 토큰이다.
- 헤드와 연결되어 전체 길이 계산에 사용된다.
- 단독으로는 음악적 의미를 갖지 않는다.
- 헤드와 연결되지 않은 경우 무음 또는 무효 입력으로 본다.

#### `@n(midi_num)`

- 위치 지정 토큰이다.
- 초기 DOCX에서는 분할음의 기본 음정과 렌더링 행을 함께 결정하는 역할로 설명되었다.
- 최신 active 문서에서는 잇단음표 슬롯의 렌더링 위치를 결정하는 문맥 의존 토큰으로 더 정교화되었다.
- 레이아웃에 존재하지 않는 `midi_num`은 허용하지 않는 방향이었다.
- 외부 악보 파일의 레이아웃과 현재 사용자 레이아웃이 충돌할 경우, 덮어쓰기, 자동 확장, import 취소 같은 선택지를 제안하는 UX도 검토되었다.

#### `@p(midi_num)`

- 실제 음정 토큰이다.
- 하모닉스나 배음처럼 표시 위치와 실제 발음 음정이 달라질 때를 위한 토큰이다.
- 존재할 경우 배치 행이나 `@n`보다 우선한다.
- `midi_num`은 0~127 범위의 정수이다.
- 하나의 노트 문자열에 하나만 존재할 수 있도록 제한하는 방향을 생각했다.

#### `@m(cent_num)`

- 미분음 토큰이다.
- 센트 단위 미세 피치 조정을 나타낸다.
- 현재 기준으로 `cent_num`은 `-100` 이상 `100` 이하의 유한 실수이다.
- 소수점 이하 자릿수는 최대 2자리까지 허용한다.
- 행 할당 음정 또는 실제 음정을 기준으로 `cent_num`만큼 피치를 보정한다.
- 렌더링 위치도 미세하게 이동한다.
- 초기 DOCX에는 `-50~50` 정수 범위가 적혀 있었으나, 이는 이후 설계에서 `-100~100` 실수 범위로 확장되었다.

#### 문자열 길이 제한

- DoS 방지와 비정상 입력 방어를 위해 포맷 규칙 차원에서 문자열 길이 제한을 둔다.
- `score comment`의 길이 제한은 100자로 둔다.
- 셀 `rawText`의 길이 제한은 100자로 둔다.
- 이 값은 이론적으로 가능한 모든 극단 조합을 허용하기 위한 값이 아니라, 연습용 악보라는 본 프로젝트의 성격을 기준으로 현실적인 사용 범위를 보장하면서 과도한 입력을 차단하기 위한 제한이다.
- 즉 매우 긴 실험적 복합 토큰 조합은 포맷 차원에서 비지원 범위로 둔다.

#### JSON 파일 크기 제한

- JSON 파일 전체 크기 상한은 8MB로 둔다.
- 회귀 코드는 서버가 아닌 브라우저 클라이언트가 악보 JSON 파일을 직접 읽고, JSON 파싱, 문서 인덱스 생성, 셀 문자열 파싱, 분석 전처리를 수행한다.
- 따라서 입력 파일 크기 제한은 단순한 저장소 제약이 아니라, 클라이언트 메모리 사용량과 초기 응답 지연을 통제하기 위한 안전장치이다.
- 초기 로드 비용은 알고리즘이 충분히 효율적이라고 가정할 때, 파일 크기 `F`, 셀 수 `C`, 셀 문자열 길이 합 `ΣL`에 대해 대략 `O(F + C + ΣL)`로 볼 수 있다.
- analyzer가 생성한 이벤트 수를 `E`라고 두면, 초기 전처리의 전체 비용은 더 넓게는 `O(F + C + ΣL + E)`로 보는 편이 적절하다.
- 이를 단계별로 나누면 파일 읽기 `O(F)`, JSON 파싱 `O(F)`, 인덱스 생성 `O(C + Rows + Tracks)`, 문자열 파싱 `O(ΣL)`, analyzer 전처리 `O(C)`와 그 결과 이벤트 구조 정리 `O(E)` 정도로 정리할 수 있다.
- 구체적인 지연시간은 기기 사양에 따라 달라지므로 단정적인 시간을 보장할 수는 없으나, 일반적인 데스크톱 환경에서 수백 ms에서 1초 내외의 초기 응답 범위를 목표로 하는 것이 바람직하다.
- 입력 JSON은 메모리에서 원본 텍스트만 존재하는 것이 아니라, 파싱된 객체, 문자열 필드, `Map` 기반 인덱스, parser 결과, analyzer 결과 및 캐시로 재구성되므로 실제 런타임 메모리 사용량은 원본 파일 크기의 수 배까지 증가할 수 있다.
- 이후 단계의 비용은 다시 셀 수가 아니라 이벤트 수를 기준으로 설명하는 편이 타당하다.
- renderer는 셀 원본이 아니라 analyzer가 만든 이벤트 리스트를 소비하므로, 부분 렌더링 전략을 전제로 할 때 그 비용은 전체 셀 수보다 현재 화면에 보이는 이벤트 수 `E_visible`과 격자/마디선/행 등 배경 요소 수 `G_visible`에 더 직접적으로 비례하며, 대략 `O(E_visible + G_visible)`로 볼 수 있다.
- audio generator 역시 이벤트 리스트를 입력으로 받아 실제 발음에 관계되는 이벤트를 오디오 스케줄 데이터로 변환하므로, 그 비용은 대략 `O(E_playable)`로 볼 수 있다.
- 이를 세분하면 `NoteEvent`, gliss 구간, 전역 제어값 세그먼트 수에 따라 `O(N_note + N_gliss + N_control)` 정도로 설명할 수 있다.
- 즉 후속 단계의 복잡도는 단순히 "셀 수를 평균 이벤트당 셀 개수로 나눈 값"으로 환산하기보다, analyzer가 얼마나 많은 이벤트를 생성했고 현재 화면/재생 구간에서 그 중 얼마를 소비하는가로 설명하는 편이 더 정확하다.
- 본 프로젝트는 10분 이상 길이의 악보도 수용해야 하므로 셀 수 자체에는 상한을 두지 않는다.
- 대신 셀 단위에서는 `rawText` 길이를 제한하고, 파일 전체에 대해서는 JSON 크기 상한을 두는 이중 방어 방식을 채택한다.
- 이 방식은 긴 곡의 악보를 과도하게 배제하지 않으면서도, 비정상적으로 큰 입력으로 인한 과도한 초기 지연, 메모리 사용량 증가, 브라우저 정지 가능성을 줄이기 위한 절충안이다.

### 1.3. 전역 행 토큰

- BPM, `beatsPerBar`, `stepsPerBeat`, Dynamics의 4개 전역 행에서만 사용하는 토큰을 둔다.

#### `number`

- 숫자 토큰이다.
- BPM은 실수값 입력을 허용한다.
- `beatsPerBar`, `stepsPerBeat`는 1 이상의 정수로 제한한다.
- Dynamics는 0~150 범위의 정수로 제한하는 초기 구상이 있었다.

#### `<`, `>`, `><`

- 선형 변화 토큰이다.
- 숫자 뒤에 붙어 점진적 변화를 표현한다.
- `start`, `end`, `endStart`에 대응하는 개념이다.
- accel., rit., cresc. 같은 전역 변화를 상정한다.

### 1.4. 시각화(렌더링)

- 주석 문자를 제외한 노트는 기본적으로 셀 규격의 사각형으로 표현한다.
- 음정에 따라 다른 색의 사각형을 사용한다.
- 트랙 레이어에 따라 사각형 및 텍스트 색상 세트를 구분한다.
- 기본 목표는 사용자가 효과의 종류를 한눈에 파악할 수 있는 시각 표현이다.

#### 렌더링 세부 구상

- 배경 격자, 노트, 스키마 표시 영역을 서로 다른 레이어로 분리한다.
- 변경이 있는 레이어만 다시 그려 불필요한 연산을 줄인다.
- 부분 렌더링과 전체 사전 렌더링을 모두 후보로 두고, 실제 성능 측정으로 최종 전략을 정한다.
- 주석 텍스트는 사각형 없이 별도 스타일로 렌더링하는 방향을 구상했다.
- 글리산도는 anchor 간 선 또는 지그재그 선으로 표현하는 방향을 생각했다.
- 트레몰로는 줄무늬, 비브라토는 사인파, 잇단음표는 외곽 컨테이너 + 내부 분할음 시각화로 표현하는 구상이 포함되어 있었다.
- 미분음은 엄밀한 음악 이론 기호 대신 화살표 표기처럼 단순한 시각 표기로 타협하는 방안도 검토되었다.

### 1.5. 오디오

- 음을 시작하는 `start` 판정과 앞 음을 잇는 `hold` 판정이 존재해야 한다.
- 빈 셀은 무음으로 취급한다.
- 주석 셀도 무음이다.
- Web Audio API 사용을 전제로 한다.
- 고품질 가상악기보다 연습용으로 충분한 기본 오실레이터를 우선한다.
- 여러 음이 겹칠 때 최대 볼륨 상한선도 필요하다.
- 글리산도, 트레몰로, 비브라토, 셈여림, 빠르기 변화는 가능한 범위에서 재생 반영 대상이다.
- 초기 오디오 품질 목표는 "연습용 음정 확인이 가능한 수준의 단순 합성음 재생"이다.
- 고음질 음원이나 복잡한 가상악기는 초기 목표 범위에 포함하지 않는다.

#### 반주 재생 확장

- 반주 재생과 유튜브 연동은 같은 확장 축으로 본다.
- 이 기능은 후순위 확장 기능이다.
- 1단계는 로컬 오디오 파일 또는 일반 URL 기반 반주 동기화이다.
- 2단계는 유튜브 연동이다.
- 이세계 코드에 이미 관련 구현이 존재하므로 기능 흐름과 UX는 참고할 수 있으나, 현재 구조에 맞춘 재설계가 필요하다.

#### 게임 모드 확장

- 게임 모드는 후순위 확장 기능 중에서도 최후순위로 둔다.
- 사용자의 실제 연주를 브라우저 오디오 입력으로 받아 pitch detection으로 악보와의 일치도를 판정하는 방향을 고려한다.
- 1차 구현은 GitHub Pages 환경에서 동작하는 데스크톱 브라우저 중심의 실용적 판정 모드를 목표로 한다.
- 완전 무버퍼, 모든 브라우저/모바일 환경에서의 고정밀 판정을 초기 목표로 두지 않는다.
- 판정 기준은 analyzer가 생성한 `AnalysisResult`의 이벤트와 시간축을 사용하고, score JSON에는 게임 모드용 실시간 데이터를 저장하지 않는다.
- 1차 구현은 `NoteEvent` 중심 판정을 우선하며, gliss/vibrato/trem/tuplet의 정교한 평가와 모바일 안정화는 후순위로 둔다.
- 처음 이벤트부터 마지막 이벤트까지의 판정을 누적하여 점수와 랭크를 계산한다.
- reset, jump, seek 같은 시간축 불연속 동작이 발생하면 게임 모드는 자동 중단된다.

### 1.6. 악보 작성(에딧 모드)

- 좌클릭 루프 기능을 지원한다.
  - `default text -> - -> ~ -> default text`
- 셋잇단 모드에서도 별도의 좌클릭 루프를 지원한다.
  - `헤드 토큰 -> 연장 토큰 -> 헤드 토큰`
- 종류별 입력 토큰을 UI에서 선택한 뒤 셀 클릭으로 통합 입력한다.
- Undo/Redo를 제공한다.
- 임시저장, 파일 저장 버튼도 고려되었다.
- 편집 후에는 전체 갱신이 아니라 영향 범위와 인접 범위만 부분 갱신하는 방향이 핵심이다.

#### 에딧 모드 패널 초기 구상

- `number`
  - 전역 행용 입력 패널
  - 숫자 입력창과 변화 토큰 버튼으로 구성
- `Default`
  - 기본 텍스트와 주석 입력을 담당
  - `AUTO#`, `AUTO♭`, `CUSTOM`, `comment` 같은 모드를 상정
- `Long`
  - 타이/비브라토 선택 패널
- `Slide`
  - 글리산도 식별자와 kind를 선택하는 패널
- `Trem`
  - `T2`, `T3`, `T4` 같은 트레몰로 선택 패널
- `Tuplet Mode`
  - 분할 수 선택, 행 선택, finalize 버튼, 동적 슬롯 입력창을 갖는 패널
- `Custom Pitch`
  - 실제 음정 토큰과 미분음 토큰을 함께 조정하는 패널
- `Control`
  - Pending 표시, Undo, Redo, Clear, 임시저장, 파일 저장 버튼 등

초기 UI 구상은 비교적 구체적이었지만, 실제 구현에서는 현재 데이터 구조와 문법 체계에 맞춰 재설계될 수 있다.

#### 모바일 범위

- 모바일은 초기 버전의 공식 편집 대상이 아니다.
- 모바일에서의 목표 범위는 파일 불러오기, 재생, 일시정지, 정지, 스크롤 기반 열람이다.
- 셀 편집, 드래그 편집, 정교한 토큰 입력은 모바일 비목표로 둔다.

## 5. 추가 설계 메모

### 노트 문자열 구상

- 셀 내부 문자열을 규칙 기반 토큰 문법으로 해석한다.
- 디폴트 텍스트, 주석, 레가토, 바이브, 글리산도, 트레몰로, 잇단음, 위치 지정 기능이 필요하다.
- 직접 문자열을 입력하기보다는 에딧 모드 UI를 통해 쉽게 조합하도록 하는 것을 목표로 한다.
- 토큰은 사용자 지정 문자와 겹치지 않도록 특수문자 위주로 설계하려는 초기 발상이 있었다.

### 해석 방향

- 셀 문자열과 인접 셀 관계를 분석해 중간 산출물을 만든다.
- 이 산출물은 이후 렌더링과 오디오의 공통 입력이 된다.
- 잇단음은 내부에 다시 하위 문자열을 가지므로 별도의 추가 처리가 필요하다.
- 초기 DOCX는 이 중간 산출물을 CSV 유사 리스트나 이벤트 리스트 같은 형식으로 상정했다.

### 시각화 방향

- Canvas 2D API 사용을 전제로 한다.
- 배경, 노트, 스키마 표시 영역을 레이어로 분리한다.
- 부분 렌더링과 전체 사전 렌더링 중 어떤 전략이 유리한지 성능 측정을 통해 결정한다.

### 오디오 방향

- Web Audio API 사용을 전제로 한다.
- `start`와 `hold` 판정이 필요하다.
- 주석 셀과 빈 셀은 무음이다.
- 기본 오실레이터 수준의 단순한 음원부터 시작한다.
- 초기 목표는 이세계 코드 수준의 재생 품질을 대체로 재현하는 것이다.

### 전역 변화 지원

- BPM, 박자, 스텝, 셈여림을 상단 전역 행으로 지원한다.
- 숫자 뒤의 토큰을 이용한 점진적 변화도 고려한다.

### 전역 수치 범위

- BPM은 양의 실수로 두고, 현재 단계에서는 상한을 고정하지 않는다.
- `beatsPerBar`는 1 이상의 정수이다.
- `stepsPerBeat`는 1 이상의 정수이다.
- Dynamics는 0~150 범위의 정수로 둔다.
- BPM 상한 제약은 후속 단계에서 필요시 추가한다.

### 에딧 모드 방향

- 선택/드래그 기반 편집 UI를 제공한다.
- 버튼으로 토큰을 조합해 셀에 삽입한다.
- Undo/Redo를 제공한다.
- 변경 부분과 그 주변 범위만 부분 갱신한다.

## 7. 유사 프로그램과의 비교

### 상용 제품 비교 관점

- `Logic Pro`, `FL Studio`와 공통점
  - 음악 프로그램
  - 피아노롤 형식 악보의 재생, 작성, 저장 기능 지원
  - 클릭, 드래그 기반 조작
- 차이점
  - 본 프로젝트는 무료 웹앱을 목표로 한다.
  - 목표는 정교한 작곡이 아니라 프렛리스 현악기 초심자 연습 보조이다.
  - 음정 간격을 피아노 키처럼 균일하게 두지 않고, 현악기의 물리적 간격을 반영한다.
  - 다양한 고품질 가상악기 대신 기초적인 오실레이터를 우선한다.

### 오픈소스 및 무료 도구 비교 관점

- `Chrome Music Lab`
  - 직관적 웹 기반 음악 체험이라는 점에서 유사한 방향
  - 하지만 매우 짧은 길이와 제한적 기능을 전제로 한다.
- `MuseScore`
  - 무료 도구라는 점은 유사
  - 그러나 오선보 중심의 기보 시스템이라 직관적 연습 보조용 피아노롤과는 목표가 다르다.

이 비교를 통해 본 프로젝트는 `작곡 도구`보다 `연습 보조용 시각화 피아노롤 편집기`에 더 가깝다는 점을 분명히 한다.
