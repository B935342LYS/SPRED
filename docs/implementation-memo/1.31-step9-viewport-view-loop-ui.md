# 1.31. Step9 viewport, View, Loop UI 메모

## 1. 목적

이 메모는 장악보 렌더링 1차 개선과 그 이후 추가된 View/Loop UI 정리 작업을 기록한다.

기준 문서:

- `docs/2.0-ui-mvp-spec.md`
- `docs/2.1-canvas-renderer-module-spec.md`
- `docs/2.8-edit-invalidation-and-partial-rebuild-spec.md`

이 문서는 구현 해설용 메모이다.
저장 포맷, parser/analyzer 공식 계약, audio timing 계약은 active spec 문서를 따른다.

## 2. 장악보 viewport bounded rendering

긴 악보 렌더링은 tile cache가 아니라 viewport bounded rendering으로 1차 구현했다.

현재 방향:

- `ScoreFile`, `RuntimeDocument`, `AnalysisResult`, `CanvasAnalyzedRenderInput`은 전체 악보 기준으로 유지한다.
- `CanvasScoreLayout`도 전체 row/column 좌표계를 유지한다.
- 실제 dynamic layer canvas bitmap과 draw 대상은 현재 viewport plus overscan 범위로 제한한다.
- `.score-canvas-stage`는 전체 scroll extent를 유지해 사용자가 전체 악보를 스크롤할 수 있게 한다.
- dynamic layer canvas는 viewport 시작 x 좌표에 배치하고, renderer context transform으로 기존 전체 좌표계 draw 함수를 계속 사용할 수 있게 한다.

레이어 정책:

- static row background는 회색 roll background, note row background, row horizontal line을 담당한다.
- static row background는 가로 스크롤 때 다시 그리지 않는다.
- dynamic grid는 현재 viewport 범위의 박자선/마디선만 다시 그린다.
- global marker, note marker, note layer는 visible tick range와 겹치는 item만 그린다.

이 구조는 “회색 줄무늬 배경은 고정되어 있고, 박자선/마디선과 note/marker만 움직이며 다시 그려지는” 시각 효과를 만든다.

## 3. visible range filtering

`src/renderer/canvas_visible_range.ts`는 renderer item만 소비하는 순수 helper로 둔다.
analyzer나 score 저장 타입은 import하지 않는다.

현재 filtering은 다음 group을 대상으로 한다.

- note item
- global text item
- global marker item
- note marker item
- runtime loop boundary marker

초기 병목 측정에서 `app.renderDynamicViewportLayers`, note filtering, note draw가 가장 큰 비용으로 확인되었다.
후속 개선에서 note/global text/marker group에 tick-range index를 두어 매 scroll마다 전체 item 배열을 훑는 비용을 줄였다.
테스트는 `dev/test_partial_rebuild.ts`에 visible range와 loop marker filtering 검증을 추가했다.

성능 측정용 개발 함수는 병목 확인 뒤 제거했다.
따라서 현재 배포 코드에는 임시 profiling API를 남기지 않는다.

## 4. scroll redraw 정책

scroll event는 즉시 draw하지 않고 `requestAnimationFrame`으로 병합한다.
한 frame 안에 여러 scroll event가 들어오면 마지막 viewport 상태 기준으로 한 번만 dynamic layer를 다시 그린다.

기본 정책은 frame skip 없는 animation frame당 최대 1회 redraw이다.
저사양 장비에서 추가 지연이 확인되면 후속 후보는 다음이다.

- dynamic layer tile/chunk cache
- time-budget 기반 scroll redraw throttle
- note marker와 특수 연결선의 추가 visibility index

현재까지 측정된 병목은 1차 최적화 뒤 평균 수 ms 이하로 줄었지만, 저사양 노트북의 실사용 확인을 계속 받는다.

## 5. zoom과 Speed 위치 보존

zoom 변경과 Speed 변경은 현재 viewport 왼쪽 edge tick을 기준으로 위치를 보존한다.
기존 `scrollLeft`가 가리키던 tick을 새 layout 계산 뒤 다시 viewport 왼쪽 edge에 오도록 `scrollLeft`를 재설정한다.

Speed는 재생 속도가 아니다.
`tickToSeconds()`, BPM, seek seconds, audio schedule, YouTube offset에는 영향을 주지 않는다.
renderer column width에 배율만 적용해 화면상 note rectangle 이동 속도처럼 보이는 값만 바꾼다.

현재 Speed 범위는 `1.0x`부터 `4.0x`이다.
Speed 값은 runtime UI preference이며 score JSON에 저장하지 않는다.

## 6. View 메뉴 정리

View 메뉴의 Text Scale은 제거했다.
텍스트 크기는 zoom과 renderer font metric을 따른다.

Text off는 다음 text만 숨긴다.

- note의 `displayText`
- mute cell의 `MuteText`

Text off가 켜져도 다음 요소는 계속 표시한다.

- note rectangle
- hold, gliss, trem, tuplet, marker
- global row rawText

Text off는 parser/analyzer 결과와 저장 데이터에 영향을 주지 않는 renderer option이다.

## 7. Expand right 제한

한 번의 Expand right 입력값에 별도 상한을 두는 정책은 제거했다.
현재 제한은 전체 score column 수의 상한인 `MAX_SCORE_COLUMN_COUNT`를 따른다.

즉 사용자가 한 번에 큰 column 수를 입력해도, 결과 column count가 전체 상한을 넘지 않으면 허용한다.
상한을 넘는 경우에는 앱 상태를 바꾸지 않고 status message로 오류를 표시한다.

## 8. Loop UI first pass

Loop UI는 playback 연결 전 runtime view state와 marker 표시까지 먼저 구현했다.

현재 구현:

- Loop on/off toggle
- Start/End select의 First/Last 기본값
- `Select Column` pick mode
- pick mode에서 악보 column 반복 클릭 시 start/end boundary 갱신
- start는 선택 column 왼쪽 경계, end는 선택 column 오른쪽 경계로 저장
- start/end 값은 작은 값이 start, 큰 값이 end가 되도록 정규화
- 보라색 loop boundary marker를 marker layer에 표시
- marker label은 전역 행 텍스트와 겹치지 않도록 canvas 하단에 반투명으로 표시
- edit mode 진입 시 loop는 off, pickMode는 null로 전환
- edit mode 중 loop toggle과 start/end select는 disabled

Loop state는 `AppState.loop`에만 둔다.
score JSON, parser result, analyzer event에는 저장하지 않는다.

reset 정책:

- score load
- Clear All
- layout apply
- column expand
- column trim

위 동작 뒤에는 loop state를 First/Last 기본값으로 되돌린다.
active track 변경은 loop range를 유지한다.

## 9. Loop playback 미연결 범위

이번 단계는 Loop UI와 marker overlay first pass이다.
playback controller/scheduler에 UI range를 전달해 실제 반복 재생하는 연결은 아직 후속 작업이다.

후속 연결 시 확인할 점:

- loop start/end tick을 current time mapper로 seconds에 변환한다.
- Loop off 또는 edit mode 진입 시 controller loop state를 off로 만든다.
- seek, pause/resume, stop, playback scroll이 loop boundary와 일관되게 동작해야 한다.
- YouTube mode에서는 iframe player가 audio controller의 follower로 동작하므로, 반복 seek가 YouTube player에도 적용되는지 확인해야 한다.

## 10. 배포 staging 반영

이번 변경은 main implementation인 `regression-code/`와 별도 배포 staging copy인 `regression-code-test-publish/`에 함께 반영했다.

반영 대상:

- TypeScript source
- HTML entry
- CSS
- Vite build 결과 생성 경로

배포 staging copy는 원본 저장소에 포함하지 않는 local publish workspace로 유지한다.

## 11. 검증

주요 확인 명령:

```text
npm run typecheck
npx tsc --noEmit --noUnusedLocals --noUnusedParameters
npm run test:view
npm run test:partial
npm run test:track
npm run build
```

배포 staging copy에서는 변경 반영 뒤 `npm run typecheck`와 `npm run build`를 확인했다.

## 12. 남은 작업 후보

- 저사양 노트북에서 9000+ column score의 scroll/render 체감 성능을 다시 확인한다.
- 필요할 경우 dynamic layer tile/chunk rendering 또는 time-budget throttle을 검토한다.
- Loop UI range를 playback controller/scheduler에 연결한다.
- Loop playback과 YouTube follower playback의 seek/repeat 상호작용을 수동 테스트한다.
- Loop panel의 CSS 폭과 select/capsule 정렬은 실제 브라우저 렌더링 기준으로 계속 미세 조정한다.
