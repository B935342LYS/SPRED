# 1.11. 2단계 Static UI Shell 구현 메모
> 본 설계 메모 문서 1.11에서는 `regression-code/index.html`과 `regression-code/styles/app.css`에 작성된 정적 MVP UI shell의 첫 구현 범위를 기록한다.
> 본 문서는 active specification을 대체하지 않는다.
> UI 동작 계약은 `docs/2.0-ui-mvp-spec.md`를 우선 기준으로 하며, 본 문서는 구현 과정에서의 판단과 다음 작업 전제를 보충한다.

## 1. 개요

1단계 parser 조립 이후, analyzer / renderer / audio 연결 전에 사용자가 볼 수 있는 정적 UI 골격을 먼저 작성했다.
이 UI shell은 아직 TypeScript 상태 모델이나 parser/analyzer 결과와 연결되지 않은 HTML/CSS 기반 화면이다.

현재 흐름은 아래와 같다.

```text
index.html
-> app-shell
   -> menu-area
   -> edit-panel
   -> score-viewer
   -> status-area
styles/app.css
-> viewport 고정 앱 레이아웃
-> hover/focus 메뉴
-> edit panel 표시/숨김
-> 임시 score grid 표시
```

## 2. 구현 범위

현재 구현한 UI 영역은 다음이다.

- 상단 메뉴 영역
  - File / View / Loop / Help hover menu
  - Edit Mode toggle
  - Track / Expand / Reload 자리
- 중앙 플레이어 영역
  - artist / title 표시 자리
  - play / stop 버튼 자리
  - seek bar, volume, waveform 자리
- YouTube placeholder 영역
  - mode toggle, offset, re-sync, 연결 상태 표시 자리
- Edit Panel
  - Number
  - Default
  - Long
  - Gliss.
  - Trem.
  - Pitch
  - Tuplet
  - Edit Actions
- Score Viewer
  - 좌측 layout label 영역
  - layout preset toolbar 자리
  - canvas layer 전환 전까지의 악보 표시 자리
- Info dialog
  - 비영리, 음원 미제공, 저작권 존중 안내
- Status Area
  - sample load / analysis / renderer 상태 표시 자리

MVP에서 아직 연결하지 않은 항목은 대부분 disabled 또는 placeholder 상태로 표시했다.

## 3. 구현 판단

### 3.1. 상단 메뉴는 DOM으로 유지

상단 메뉴, 입력창, select, dialog는 canvas가 아니라 HTML DOM으로 구현했다.
이 영역은 접근성, focus 처리, disabled 처리, keyboard 동작, 반응형 레이아웃 면에서 DOM이 유리하다.

canvas는 다음 단계에서 악보 본문과 layout label 영역에 우선 적용한다.

### 3.2. 현재 단계에서는 `index.html`에 UI 태그를 직접 둔다

UI shell 분리 방안으로 다음을 검토했다.

- TypeScript template string
- `index.html` 내부 `<template>`
- 별도 HTML fragment import
- 별도 HTML fragment runtime fetch
- 컴포넌트 프레임워크

현재 UI는 약 300줄 규모의 정적 골격이며, 아직 상태 동기화나 반복 컴포넌트가 많지 않다.
따라서 HTML fragment import나 framework 도입은 현재 단계에서 구조 비용이 더 크다고 판단했다.

결론적으로, 현 단계에서는 `index.html`에 UI 태그를 직접 두고 CSS만 `styles/app.css`로 분리한다.
이 방식은 파일을 직접 열어 확인하기 쉽고, GitHub Pages 정적 배포 전 단계에서도 가장 단순하다.

### 3.3. `index.html`은 배럴 파일처럼 쓰지 않는다

`index.html`은 TypeScript 모듈 export를 모으는 배럴 파일과 성격이 다르다.
현재처럼 정적 앱의 첫 화면 구조를 담는 entry document로 사용하는 것은 과한 구조가 아니다.

다만 다음 상황이 오면 다시 분리한다.

- UI HTML이 700~1000줄 이상으로 커짐
- 같은 UI 조각이 반복되어 복제가 늘어남
- 상태 변경에 따라 DOM 갱신 코드가 여러 영역에 퍼짐
- edit panel / modal / layout preset UI가 독립 컴포넌트처럼 커짐
- 빌드 도구 도입이 확정되어 HTML fragment 또는 framework 사용 이득이 분명해짐

### 3.4. CSS는 먼저 하나의 파일로 유지

현재 `styles/app.css`는 UI shell 전체 레이아웃을 담고 있다.
CSS 역시 아직 파일을 나눌 단계는 아니다.

다만 canvas 전환 후 다음처럼 자연스럽게 분리할 수 있다.

```text
styles/app.css
styles/shell.css
styles/score_viewer.css
styles/dialog.css
```

분리는 파일 크기보다 책임 경계가 명확해졌을 때 진행한다.

## 4. 현재 구조의 제약

### 4.1. TypeScript UI 상태와 미연결

현재 edit mode open/close는 hidden checkbox와 CSS sibling selector로 처리한다.
이는 정적 UI 검증에는 단순하지만, 이후 playback 중 edit mode 금지 같은 guard를 적용하려면 TypeScript 상태 제어로 바뀌어야 한다.

다음 단계에서 UI 이벤트 연결을 시작할 때는 아래 상태가 필요하다.

```text
uiMode
playbackState
activeTrackIds
zoom
volume
waveform
seekTick
```

### 4.2. 악보 영역의 canvas layer 전환

초기 `.score-grid`와 `.note-rect`는 화면 밀도와 스크롤 영역을 확인하기 위한 임시 DOM이었다.
이 임시 DOM은 canvas layer 전환 단계에서 제거하고, `score-area` 내부에 canvas stage를 두는 구조로 바꾼다.

전환 후 기본 구조는 아래와 같다.

```text
score-viewer
  layout-area
    layout-canvas-stage
      label-layer
  score-area
    base canvas
    note canvas
    marker canvas
```

선택/호버/드래그 하이라이트를 당장 표시하지 않기로 했으므로 interaction layer는 첫 전환 범위에서 제외한다.
실제 edit mode overlay가 필요해지는 시점에 후속 확장으로 추가한다.

### 4.3. 직접 파일 열기와 빌드 도구

현재 상태는 `index.html`을 직접 열어 UI shell을 확인할 수 있다.
이는 초기 UI 확인에는 유리하다.

반면 TypeScript 기반 이벤트 연결, sample auto load, renderer 초기화가 들어오면 브라우저 실행 경로와 빌드 경로를 다시 정해야 한다.
그때까지는 `index.html`과 CSS만으로 확인 가능한 정적 shell을 유지한다.

## 5. 다음 작업 기준

다음 구현은 UI shell을 더 분리하는 것이 아니라, temporary score grid를 canvas layer slot으로 교체하는 작업이 적절하다.

권장 순서는 다음이다.

1. `score-area` 내부의 임시 `.score-grid`를 canvas layer host로 교체한다.
2. layout label 영역과 score canvas 영역의 scroll 동기화 전제를 잡는다.
3. canvas layer의 DOM 구조와 CSS stacking 순서를 정한다.
4. renderer 좌표계에서 사용할 column width, row height, padding column, playback boundary 값을 임시 상수로 둔다.
5. 실제 analyzer 결과 연결 전까지는 static sample draw 함수로 base grid와 note rectangle만 그린다.

이 순서로 진행하면 기존 정적 UI shell을 크게 흔들지 않고 renderer 구현으로 넘어갈 수 있다.

## 6. 이번 구현에서 사용한 주요 HTML / CSS 문법

### 6.1. 시맨틱 HTML 요소

UI shell은 `div`만 사용하지 않고 역할에 맞는 요소를 섞어 사용했다.

```html
<main>
<header>
<nav>
<section>
<aside>
<footer>
<dialog>
```

이 요소들은 화면 구조를 브라우저와 보조 기술에 더 명확히 전달한다.
예를 들어 상단 메뉴는 `header`, 좌측 도구 묶음은 `nav`, 악보 좌측 라벨은 보조 영역이므로 `aside`로 표현했다.

### 6.2. `aria-label`

아이콘 버튼이나 시각적 구조만으로 의미가 부족한 영역에는 `aria-label`을 붙였다.

```html
<main class="app-shell" aria-label="SPRED score app">
<nav class="left-tool-group" aria-label="Tool menus">
<section class="player-group" aria-label="Playback controls">
```

`aria-label`은 화면에 보이지 않는 접근성 이름이다.
이름이 없는 container나 버튼을 screen reader가 설명할 수 있도록 돕는다.

### 6.3. form control 속성

정적 UI에서도 나중에 실제 동작으로 연결할 입력 제한을 HTML 속성에 먼저 둔다.

```html
<input type="range" min="50" max="200" value="100">
<input type="number" min="-100" max="100" step="0.1">
<button type="button" disabled>
```

`type`, `min`, `max`, `step`, `disabled`는 브라우저가 기본 입력 동작과 비활성 상태를 처리하게 한다.
특히 `button`은 form 내부가 아니더라도 기본 submit 동작을 피하기 위해 `type="button"`을 명시했다.

### 6.4. `<dialog>`와 `method="dialog"`

Info popup은 HTML `<dialog>` 요소로 만들었다.

```html
<dialog id="legal-info" class="info-dialog">
  <form method="dialog">
    <button type="submit">Close</button>
  </form>
</dialog>
```

`dialog.showModal()`을 호출하면 modal dialog로 열린다.
내부 form에 `method="dialog"`를 쓰면 submit 버튼으로 dialog를 닫을 수 있다.
현재는 Help menu의 inline handler로 호출하지만, 이후 TypeScript 이벤트 바인딩으로 옮길 수 있다.

### 6.5. CSS custom properties

반복되는 색상과 치수는 `:root`의 CSS 변수로 모았다.

```css
:root {
  --bg: #f5f7f8;
  --label-width: clamp(150px, 18vw, 208px);
  --status-height: 28px;
}
```

다른 규칙에서는 `var(--label-width)`처럼 참조한다.
이 방식은 layout label 폭, status 높이, 색상 계열을 한 곳에서 조정할 수 있게 한다.

### 6.6. CSS Grid와 `minmax(0, 1fr)`

앱 전체와 상단 메뉴, score viewer는 CSS Grid로 배치했다.

```css
.app-shell {
  display: grid;
  grid-template-rows: auto auto minmax(0, 1fr) var(--status-height);
}

.score-viewer {
  display: grid;
  grid-template-columns: var(--label-width) minmax(0, 1fr);
}
```

`minmax(0, 1fr)`은 grid item이 내부 콘텐츠 때문에 최소 크기를 과도하게 키우지 않도록 한다.
악보 영역처럼 scroll container가 들어가는 레이아웃에서는 이 설정이 중요하다.

### 6.7. hover / focus-within 드롭다운

메뉴 panel은 기본적으로 숨기고, 부모가 hover 또는 focus-within 상태일 때 표시한다.

```css
.menu-panel {
  display: none;
}

.menu-dropdown:hover > .menu-panel,
.menu-dropdown:focus-within > .menu-panel {
  display: grid;
}
```

`:focus-within`은 내부 버튼이나 입력 요소가 focus를 가진 경우에도 부모를 활성 상태로 만든다.
마우스 hover뿐 아니라 keyboard tab 이동에도 메뉴가 유지된다.

### 6.8. hidden checkbox와 sibling selector

Edit Panel은 임시로 hidden checkbox와 sibling selector를 사용해 표시한다.

```css
#edit-mode-toggle:checked ~ .edit-panel {
  display: block;
}
```

`~`는 같은 부모 아래에서 뒤에 오는 형제 요소를 선택하는 general sibling selector이다.
checkbox가 checked 상태일 때 뒤쪽의 `.edit-panel`을 표시한다.
이 구조는 JavaScript 없이 정적 UI를 확인하기 쉽지만, 이후 playback guard가 필요해지면 TypeScript 상태 제어로 바꿔야 한다.

### 6.9. `position: absolute`와 stacking

드롭다운 메뉴, 플레이어 상태 pill, layout toolbar는 부모 기준 절대 위치로 배치했다.

```css
.menu-dropdown {
  position: relative;
}

.menu-panel {
  position: absolute;
  z-index: 200;
}
```

자식의 `position: absolute`는 가까운 positioned ancestor를 기준으로 삼는다.
따라서 `.menu-dropdown`에 `position: relative`를 주어 menu panel이 해당 버튼 아래에 뜨도록 했다.
겹침 순서는 `z-index`로 조정한다.

### 6.10. `clamp()`와 responsive constraint

버튼 padding, label width, seek bar 폭에는 `clamp()`를 사용했다.

```css
--label-width: clamp(150px, 18vw, 208px);
width: clamp(170px, 38vw, 420px);
```

`clamp(min, preferred, max)`는 preferred 값을 쓰되 최소/최대 범위를 넘지 않게 한다.
viewport에 따라 조금 움직이지만 지나치게 좁아지거나 넓어지지 않아야 하는 UI에 적합하다.

### 6.11. media query

폭이 좁은 화면에서는 상단 메뉴를 한 열로 바꾼다.

```css
@media (max-width: 900px) {
  .menu-area {
    grid-template-columns: minmax(0, 1fr);
  }
}
```

`@media`는 조건부 CSS 규칙이다.
모바일에서는 편집 기능이 초기 범위 밖이지만, view/load/play 화면이 깨지지 않도록 최소 반응형 배치를 둔다.

## 7. 테스트 확인

본 작업은 HTML/CSS 정적 UI shell 작성이므로 parser/score 로직 테스트와 직접 연결되지 않는다.

현재 코드 기준으로는 다음 검증 명령이 유지되어야 한다.

```text
npm run typecheck
npm run test:score
npm run test:parse
```

UI shell 변경 후에는 별도 빌드 없이 `regression-code/index.html`을 브라우저로 열어 정적 화면을 확인한다.

## 8. 보류한 결정

다음 결정은 아직 확정하지 않았다.

- Vite 또는 다른 web build 도구 도입 시점
- Vue / React / Svelte 등 컴포넌트 프레임워크 도입 여부
- HTML fragment 분리 방식
- CSS 파일 분리 시점
- canvas renderer 모듈의 정확한 파일명과 공개 함수명

현재 판단은 프레임워크를 도입하지 않고, core TypeScript 모듈과 canvas renderer 구현을 먼저 진행하는 것이다.
