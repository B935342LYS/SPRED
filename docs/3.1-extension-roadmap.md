# 회귀 코드 확장 계획 정리

## 1. 문서 목적

본 문서는 `회귀 코드`의 MVP 이후 확장 후보를 한곳에 모아 관리하기 위한 확장 계획 문서이다.

이 문서는 active 구현 명세가 아니다.
실제 구현 착수 시에는 관련 기능을 별도 세부 명세로 분리하거나, 기존 active 문서에 필요한 내용을 흡수한다.

참고한 자료:

- `1. 개발문서/주간보고서/6월 2주차 최종보고서.pdf`
- `docs/HARNESS.md`
- `docs/1.0-development-spec.md`
- `docs/3.0-extendplan-game-mode.md`

## 2. 확장 우선순위 원칙

확장 기능은 다음 순서로 판단한다.

1. 현재 MVP 사용 흐름을 직접 안정화하는 기능
2. 긴 악보, 많은 셀, 반복 편집에서 병목을 줄이는 기능
3. 실제 악보 작성 편의성을 높이는 기능
4. 재생 품질과 반주 동기화를 높이는 기능
5. 데이터 공유, 보안, 장기 운영 기능
6. 게임 모드처럼 별도 연구 성격이 강한 후순위 기능

현재 구조의 중심 원칙은 유지한다.

- score JSON은 저장 원천이다.
- parser / analyzer / renderer / audio generator 경계는 유지한다.
- renderer와 audio는 `AnalysisResult`를 소비하고 rawText를 직접 재해석하지 않는다.
- 고비용 확장은 실제 병목을 확인한 뒤 단계적으로 붙인다.

## 3. 확장 후보 요약

| 확장 후보 | 우선도 | 예상 난도 | 현재 판단 |
| --- | --- | --- | --- |
| 레이아웃 변경 UI | 높음 | 중~하 | 1차 안정화 완료. 보고서/구현 메모 정리 단계 |
| 트랙 레이어 기능 | 높음 | 중 | 1차 구현 완료. empty active 상태, inactive alpha, active track edit/audio filtering의 브라우저 사용성 검증 필요 |
| 기본 오디오 검증 | 높음 | 중 | YouTube 모드 전에 청감/동기화 기준선 확인 필요 |
| YouTube 반주 싱크 | 높음 | 중상 | musicData의 youtube 필드는 준비됨. 기본 오디오 검증 후 착수 |
| 부분 갱신 | 높음 | 상 | 긴 악보 편집 성능 개선 핵심 후보 |
| viewport 가상화 | 높음 | 상 | 전체 canvas 렌더링 병목 확인 후 검토 |
| 재생 속도 변경 | 중간 | 상 | 가상화/스크롤 구조 안정화 이후 검토 |
| 마이크 인식 + 음정 평가 | 낮음 | 매우 높음 | 게임 모드 문서에서 별도 관리 |
| 다중 현 / 다중 악보 전개 | 중간 | 매우 높음 | score/layout/analyzer/renderer 전반 영향 큼 |
| 보안 검토 | 높음 | 중 | 외부 JSON import 안정성 강화 필요 |
| DB / 공유 저장소 | 중간 | 중 | GitHub Pages 단독 범위를 넘는 장기 후보 |

## 4. 레이아웃 편집 후속 과제

현재 레이아웃 편집은 다음 범위까지 연결되어 있다.

- layout dialog draft 생성
- note/gap row 추가, 삭제, 높이 수정
- draft apply 시 구조 공유 방식으로 next ScoreFile 생성
- runtime rebuild와 playback runtime reset
- 삭제될 track cell 확인
- Local Slot 1..3 저장/불러오기
- layout preset JSON 파일 저장/불러오기
- toolbar의 Default Layout / Local Slot 적용
- score-load-time layout snapshot 기반 Default Layout 복원
- layout preset 이름 30자 제한과 JSON 256 KiB 제한

후속 후보:

- `instData.presetId`의 의미 재정의 또는 v2 포맷에서 명칭 변경 검토
- layout apply의 구조 공유 방식 부분 복사 유지 및 병목 측정
- layout 변경 undo stack 검토
- 외부 score JSON import 시 structural reconciliation 도입
- local slot 삭제 또는 clear UI 추가 여부 검토

현재 판단:

- layout apply는 전체 `ScoreFile` deep clone 대신 바뀌는 `instData`, `layout.rowDefinitions`, 필요 시 `tracks[].cells`만 교체하는 부분 복사 방식을 사용한다.
- 긴 악보에서 실제 병목이 확인되면 runtime rebuild, renderer, audio schedule 생성 비용까지 함께 측정한다.

## 5. 트랙 레이어 확장

목표:

- `basic`, `optional`, `extra` 레이어를 사용자가 선택적으로 편집하고 재생할 수 있게 한다.
- 활성화된 레이어를 편집/표시/재생 대상으로 함께 사용하고, 동시 재생 정책을 제공한다.

1차 구현 명세는 `docs/2.6-track-layer-ui-spec.md`를 따른다.

검토 항목:

- activeTrackIds 선택 UI 추가
- renderer에서 inactive track 반투명 표시 처리
- audio schedule 생성 시 active track 필터링
- 동일 시점 다중 track note가 있을 때 audio 믹싱과 clipping 방지
- 상단 tracks 패널에서 현재 active track 목록을 명확히 표시
- 같은 위치의 note가 겹칠 때 `extra -> optional -> basic` 순서로 그려 `basic`이 가장 위에 보이도록 하는 정책
- 재생 중에는 active track toggle을 비활성화하고, 정지/일시정지 상태와 edit mode에서는 조작 허용
- `src/track/track_control.ts`에 track 순서, inactive alpha, gain 같은 공용 순수 정책 배치

주의:

- score JSON의 track 구조는 이미 3트랙 고정 전제를 가진다.
- 첫 구현은 track 구조 변경보다 UI/재생 필터링을 먼저 연결하는 편이 안전하다.

## 6. 오디오와 반주 확장

### 6.1. 기본 오디오 품질

후속 후보:

- 현재 native Web Audio 출력의 청감 테스트 체크리스트 작성
- layout 변경, seek, pause/resume 이후 playback reset과 재생 시작 동작 확인
- dynamics automation 연결
- gliss fallback의 청감 품질 개선
- tremolo/vibrato 파라미터 조정
- Tone.js 또는 sampled instrument backend 검토
- loop playback range 선택

현재 판단:

- Web Audio native path는 기본 schedule 검증용으로 유지한다.
- YouTube 모드 전에 local playback 기준선과 known issue를 먼저 정리한다.
- Tone.js나 sample backend는 기본 event path가 안정화된 뒤 교체 가능한 backend 후보로 검토한다.

### 6.2. YouTube 반주 싱크

목표:

- score metadata에 저장된 YouTube video id와 offset을 사용해 반주 영상을 악보 재생과 동기화한다.

검토 항목:

- YouTube iframe API 도입
- playback controller와 YouTube player 상태 동기화
- offsetMs 기준 보정
- seek/pause/resume 시 양쪽 상태 일치
- 네트워크 지연과 브라우저 autoplay 제약 대응

현재 판단:

- 반주 기능은 두 단계로 나눈다.
- 1단계는 local/general audio synchronization, 2단계는 YouTube integration이다.

## 7. 성능 확장

### 7.1. 부분 갱신

목표:

- edit mode에서 한 셀 또는 제한된 범위만 수정했을 때 전체 parse/analyze/render를 반복하지 않는다.

후속 후보:

- parser cache를 cell 단위로 갱신
- analyzer 영향 범위 계산
- gliss, tuplet, hold처럼 인접/연결 이벤트가 있는 구조의 invalidate range 정의
- renderer item cache와 dirty rectangle 또는 layer 단위 갱신
- audio schedule 부분 재생성

주의:

- 단순 note 셀보다 gliss, tuplet, timing segment가 영향 범위를 크게 만든다.
- 먼저 병목을 측정하고, 가장 큰 비용부터 부분 갱신으로 대체한다.

### 7.2. Viewport 가상화

목표:

- 전체 악보를 한 번에 렌더링하지 않고 화면에 보이는 범위 중심으로 그린다.

검토 항목:

- horizontal viewport range 계산
- 현재 playback 기준선 주변 pre-render buffer
- scroll 중 redraw 빈도 제한
- hit test와 virtualization 좌표계 일치
- export나 전체 보기 모드와의 분리

주의:

- 가상화는 renderer, hit test, playback scroll을 함께 건드리므로 난도가 높다.
- 긴 악보 성능 측정 이후 착수한다.

## 8. 재생 속도 변경

목표:

- 사용자가 연습 속도를 조절할 수 있게 한다.

가능한 접근:

- tick -> seconds mapping에 playback rate를 반영
- 또는 column width / scroll 속도를 바꿔 체감 속도를 조정

현재 판단:

- 오디오, 스크롤, renderer 기준선이 함께 움직이므로 단순 UI 옵션보다 영향 범위가 크다.
- viewport 가상화와 playback controller가 더 안정화된 뒤 검토한다.

## 9. 다중 현 확장

목표:

- 하나의 악보 안에서 여러 현 또는 여러 악기 줄을 동시에 표시하고 편집한다.

검토 항목:

- `instData.strings[]`의 string별 row group 전개
- string별 min/max/open MIDI 정책
- string별 layout preset 저장
- 같은 MIDI가 여러 string에 존재할 때 row 선택과 hit test
- analyzer 이벤트가 stringId를 충분히 보존하는지 확인
- renderer label, 색상, row grouping 시각화

현재 판단:

- 현 구조는 저장 타입에 일부 준비되어 있지만, 실제 UI/분석/렌더링은 단일 현 중심이다.
- 다중 현은 layout, edit, renderer, audio 전반을 건드리므로 큰 확장으로 분류한다.

## 10. 게임 모드 / 마이크 음정 평가

게임 모드는 후순위 확장 기능 중에서도 가장 연구 성격이 강한 기능이다.

핵심 방향:

- 브라우저 오디오 입력을 받는다.
- pitch detection 결과를 analyzer의 `NoteEvent`와 비교한다.
- 점수와 랭크는 런타임 데이터로 계산하고 score JSON에는 저장하지 않는다.
- reset, seek, jump 같은 시간축 불연속 동작이 발생하면 게임 모드를 중단한다.

세부 계획은 `docs/3.0-extendplan-game-mode.md`를 따른다.

## 11. 보안과 import 안정성

후속 후보:

- JSON file size 제한 적용과 오류 메시지 정리
- 비정상적으로 큰 columnCount, row count, cell count에 대한 방어
- 매우 긴 rawText, comment, metadata 문자열 제한 재검증
- 악의적이거나 깨진 layout preset localStorage 처리
- 외부 score JSON import의 structural conflict 보고
- 렌더링 성능을 악용할 수 있는 입력 패턴 검토

현재 판단:

- 현재도 `validateScoreFile()`이 기본 구조와 참조 무결성을 확인한다.
- 다만 외부 입력은 신뢰할 수 없으므로, 실패 원인 표시와 방어적 제한을 더 세분화할 필요가 있다.

## 12. DB / 공유 기능

장기 후보:

- 공개 악보 목록
- 악보 다운로드용 DB
- 사용자 계정 또는 개인 저장소
- 공유 layout preset
- 버전별 score migration

현재 판단:

- GitHub Pages 정적 배포 범위를 넘는 기능이다.
- 코어 구현과 파일 기반 저장이 안정화된 뒤 별도 배포/운영 계획과 함께 검토한다.

## 13. 보고서와 검증 확장

후속 보고서에서는 기능 구현뿐 아니라 아래 항목을 함께 정리한다.

- 각 확장 기능의 난도와 위험 요인
- 현재 구조가 확장을 어떻게 가능하게 하는지
- 실제 병목 측정 결과
- open-source reference 검토 여부와 사용 방식
- 구현한 기능과 개념 비교만 한 기능의 구분

검증 후보:

- 긴 악보 성능 fixture
- 복잡한 gliss/tuplet 조합 fixture
- layout apply와 file/local preset round trip
- audio playback 청감 테스트 체크리스트
- 악의적 JSON import 실패 케이스

## 14. 현재 결론

가장 가까운 작업 순서는 track layer 사용성 검증, 기본 audio 검증, YouTube 반주 싱크이다.

부분 갱신과 viewport 가상화는 장기적으로 중요하지만, 실제 병목 측정 뒤 착수하는 편이 안전하다.

게임 모드, 다중 현, DB 공유 기능은 현재 코어 구조를 흔들지 않도록 별도 확장 단계로 분리해 관리한다.
