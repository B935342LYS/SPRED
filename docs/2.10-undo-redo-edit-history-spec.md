# 2.10. Undo / Redo Edit History 명세

## 1. 문서 목적

본 문서는 `회귀 코드`의 edit mode에서 잘못된 복사/붙여넣기, 삭제, 일반 cell 입력을 되돌리기 위한 Undo / Redo 기능을 정의한다.

목표는 다음이다.

- 사용자가 한 번의 의도로 수행한 score mutation을 하나의 undo group으로 되돌린다.
- 기존 score text edit batch / partial rebuild 경로와 충돌하지 않는다.
- UI에 이미 있는 `Undo` 버튼을 활성화하고, keyboard shortcut을 함께 지원한다.
- 저장 JSON 포맷은 바꾸지 않고 app runtime state 안에서만 history를 관리한다.

본 문서는 `docs/2.2-ui-state-edit-mode-spec.md`, `docs/2.4-layout-edit-ui-spec.md`, `docs/2.8-edit-invalidation-and-partial-rebuild-spec.md`, `docs/2.9-range-selection-edit-spec.md`의 후속 구현 명세이다.

## 2. 기본 원칙

Undo history는 score JSON에 저장하지 않는다.

```text
user command
  -> before score snapshot or inverse edit group
  -> score mutation
  -> rebuild / partial rebuild
  -> undo stack push
```

첫 구현은 안정성을 우선한다.
메모리 비용이 조금 크더라도 `ScoreFile` snapshot 기반 history를 먼저 사용한다.
이후 성능과 메모리 부담이 확인되면 inverse edit diff 방식으로 줄일 수 있다.

Undo / Redo는 parser, analyzer, renderer 저장 계약을 바꾸지 않는다.
되돌리기 또는 다시 실행할 때는 기존 score load/edit rebuild 경로를 재사용한다.

## 3. 범위

첫 구현의 undo 대상:

- 단일 note/global cell edit
- drag edit의 frame batch 또는 pointerup flush batch
- range delete
- range paste
- YouTube metadata reload
- Details metadata edit

첫 구현에서 제외하거나 후속으로 둔다:

- score file load
- Clear All
- Expand Right / Trim Right
- layout apply / layout preset apply
- local save / file download
- track active toggle
- view option 변경
- playback seek / play / pause
- range selection 자체

제외 이유:

- file load / Clear All / layout apply는 구조 변경 규모가 크고 사용자 확인 정책과 얽힌다.
- track/view/playback은 score JSON을 바꾸지 않는 runtime UI 상태이다.
- range selection은 score mutation이 아니므로 undo 대상이 아니다.

후속 확장에서 layout apply까지 undo 대상으로 넣고 싶다면 별도 `structure` history group으로 확장한다.

## 4. 상태 모델

App runtime state에 history를 추가한다.

```ts
type UndoHistoryState = {
  undoStack: UndoHistoryEntry[];
  redoStack: UndoHistoryEntry[];
  maxEntries: number;
};

type UndoHistoryEntry = {
  id: string;
  label: string;
  beforeScore: ScoreFile;
  afterScore: ScoreFile;
  restoreSelection: ScoreSelection | null;
  restoreRangeSelection: ScoreRangeSelection | null;
};
```

첫 구현 권장값:

```text
maxEntries = 50
```

규칙:

- undoStack은 가장 최근 command가 끝난 뒤 push한다.
- 새 mutation이 성공하면 redoStack은 비운다.
- undo를 실행하면 현재 entry를 redoStack으로 옮긴다.
- redo를 실행하면 entry를 다시 undoStack으로 옮긴다.
- history entry는 app session 안에서만 유지한다.
- score load 성공 시 history를 비운다.

## 5. Snapshot 정책

첫 구현은 `beforeScore`와 `afterScore`를 모두 저장한다.

이 방식의 장점:

- undo / redo 적용 시 score 전체를 확실히 복원할 수 있다.
- note/global/metadata처럼 서로 다른 mutation 종류를 같은 방식으로 다룰 수 있다.
- range delete/paste처럼 많은 cell이 바뀌는 command도 별도 inverse 계산이 필요 없다.

주의:

- 큰 score에서 snapshot 비용이 생길 수 있다.
- history entry 수를 제한해야 한다.
- snapshot은 `structuredClone()` 또는 JSON clone 중 프로젝트 환경에서 안정적인 쪽을 사용한다.

권장 구현:

```text
cloneScoreFile(score)
  -> structuredClone 가능하면 structuredClone
  -> fallback JSON.parse(JSON.stringify(score))
```

## 6. Command Grouping

사용자 의도 1개를 undo group 1개로 본다.

group 예:

- cell click edit 1회 -> 1 group
- left drag edit 1회 -> pointerdown~pointerup 동안 flush된 전체 edit을 1 group으로 묶는 것이 이상적
- 현재 구현의 RAF drag batch -> 첫 구현에서는 각 applied batch가 1 group이어도 허용
- range delete 1회 -> 1 group
- range paste 1회 -> 1 group
- Details apply 1회 -> 1 group
- YouTube Reload 1회 -> 1 group

drag edit은 UX상 pointerdown~pointerup 전체가 1 group인 편이 가장 좋다.
다만 현재 drag edit은 RAF 단위로 `applyScoreTextEdits()`를 호출한다.
첫 구현에서는 두 단계 중 하나를 선택한다.

권장 1차:

- range delete/paste와 단일 click edit은 정확히 1 group으로 묶는다.
- drag edit은 현재 batch 단위 history로 시작한다.
- 후속으로 pointer drag transaction을 도입해 drag 전체를 1 group으로 묶는다.

## 7. 적용 흐름

### 7.1. mutation 기록

score mutation을 수행하는 app 계층에서 mutation 전후 score를 history에 기록한다.

```text
previousScore = clone(current score)
apply mutation
nextScore = state.document.score
pushUndoHistory(previousScore, nextScore, label)
clear redoStack
```

mutation 실패 시 history를 기록하지 않는다.

동일한 score로 끝난 no-op mutation도 기록하지 않는다.
첫 구현에서는 `beforeScore === afterScore` 비교가 아니라 command가 실제 edit을 만든 경우만 기록해도 된다.

### 7.2. undo

Undo 실행:

```text
entry = undoStack.pop()
if entry exists:
  restore entry.beforeScore
  redoStack.push(entry)
  rebuild runtime artifacts
  reset playback preserving/stopping policy
```

권장 playback 정책:

- 재생 중이면 pause 또는 stop 후 undo를 적용한다.
- 첫 구현은 undo 실행 시 playback을 reset/stop한다.
- YouTube player는 score time follower이므로 playback reset 정책을 따른다.

### 7.3. redo

Redo 실행:

```text
entry = redoStack.pop()
if entry exists:
  restore entry.afterScore
  undoStack.push(entry)
  rebuild runtime artifacts
  reset playback preserving/stopping policy
```

## 8. UI / Shortcut

현재 HTML의 edit action 영역에는 `Undo`, `Save`, `Reset` placeholder button이 있다.
첫 구현에서는 `Undo` 버튼만 연결한다.

권장 UI:

- Undo button
  - undoStack이 비어 있으면 disabled
  - title에 마지막 undo label 표시
- Redo button
  - 현재 HTML에 없으므로 첫 구현에서는 keyboard shortcut만 지원하거나 버튼 추가를 후속으로 둔다

Keyboard:

- `Ctrl+Z` / `Cmd+Z` -> undo
- `Ctrl+Y` -> redo
- `Ctrl+Shift+Z` / `Cmd+Shift+Z` -> redo

단축키 예외:

- input, textarea, select, contenteditable 안에서는 브라우저 기본 편집 undo/redo를 방해하지 않는다.
- busy 상태에서는 undo/redo를 실행하지 않는다.

## 9. Selection 복원 정책

Undo / Redo 후 selection은 다음 정책을 사용한다.

- 첫 구현은 `selection`과 `rangeSelection`을 `null`로 해제한다.
- status message에 undo/redo label을 표시한다.

후속 개선:

- entry에 저장된 `restoreSelection` / `restoreRangeSelection`을 복원할 수 있다.
- range paste undo 후 원래 붙여넣기 영역을 다시 표시할 수 있다.

첫 구현에서 selection을 해제하는 이유:

- undo 결과와 이전 선택 영역이 어긋나는 혼란을 피한다.
- layout/row 변경까지 history가 확장되기 전에는 복원 정책을 단순하게 유지한다.

## 10. 기존 기능과의 관계

### 10.1. Range selection

Range selection 자체는 undo 대상이 아니다.
Range delete / paste가 score mutation을 만들 때만 history entry를 만든다.

삭제/붙여넣기 이후 range selection은 `2.9` 정책에 따라 해제된다.
undo 후에도 range selection은 해제 상태를 유지한다.

### 10.2. Partial rebuild

Undo / Redo는 score 전체 snapshot을 복원하지만, 첫 구현에서는 안전성을 위해 full runtime artifact rebuild를 허용한다.

후속 최적화:

- history entry에 원래 edits를 저장한다.
- undo/redo도 note/global partial rebuild 경로를 사용할 수 있다.

첫 구현은 구현 안정성과 버그 회피를 우선한다.

### 10.3. Save / Local Save

Undo history는 저장되지 않는다.
사용자가 file download 또는 local save를 수행하면 현재 score 상태만 저장한다.

저장 직후 history를 비울 필요는 없다.
저장 후 undo를 허용하면 사용자가 저장 직전 상태로 돌아갈 수 있다.
단, status message는 현재 score가 저장본과 달라질 수 있음을 명시할 필요가 있으면 후속 dirty-state 기능에서 다룬다.

## 13. 검증 기준

단위 검증:

- mutation 성공 시 undoStack에 entry가 추가된다.
- 새 mutation 성공 시 redoStack이 비워진다.
- undo는 beforeScore를 복원하고 entry를 redoStack으로 이동한다.
- redo는 afterScore를 복원하고 entry를 undoStack으로 이동한다.
- maxEntries를 넘으면 가장 오래된 entry가 제거된다.
- score load 성공 시 history가 비워진다.

브라우저 수동 검증:

- 단일 cell edit 후 Undo button이 활성화된다.
- Undo button 또는 Ctrl+Z로 edit 전 상태가 복원된다.
- range delete 후 Ctrl+Z로 삭제 전 cell들이 복원된다.
- range paste 후 Ctrl+Z로 paste 전 상태가 복원된다.
- undo 후 Ctrl+Y 또는 Ctrl+Shift+Z로 redo된다.
- input에 focus가 있을 때 Ctrl+Z는 input 편집 undo로 동작하고 score undo를 실행하지 않는다.
- busy 상태에서는 undo/redo가 실행되지 않는다.
- score load 후 이전 score의 undo history가 남지 않는다.
