**프로젝트 중심**

`regression-code/`가 현재 활성 구현본입니다. 프렛리스 현악기용 악보 프로그램 `회귀 코드`이며, 기존 Google Sheets + Apps Script 기반 `이세계 코드`와는 별도 TypeScript/HTML/CSS 웹앱으로 재구현하는 방향입니다. GitHub Pages 배포를 염두에 둔 Vite 프로젝트이고, 현재 의존성은 `pitchy`를 포함합니다.

`regression-code-report-commentary/`는 보고서/해설용 복사본입니다. 여기는 런타임 동작 변경용이 아니라, 2차 구현 설명을 위해 세부 주석을 촘촘히 다는 용도입니다. 활성 제품 수정은 여전히 `regression-code/`에서 해야 합니다.

**문서 기준**

- canvas renderer

**현재 작업 모드**

현재는 추가 기능 구현보다 보고서 준비와 해설용 코드 정리에 무게가 옮겨간 상태입니다. HARNESS에는 “구현 변경은 일시 정지, 실사용 테스트 이슈가 나오면 수정”이라는 흐름이 명시되어 있고, `regression-code-report-commentary/`에서 해설용 주석 작업이 진행 중인 것으로 되어 있습니다.

현재 `pitchy`는 McLeod Pitch Method 기반입니다. 이미 autocorrelation 계열의 꽤 좋은 실시간 검출기라서, AMDF/ASDF/YIN으로 단순 교체해도 “한 옥타브 위 배음 선택” 문제가 근본적으로 사라질 가능성은 낮습니다. 같은 시간영역 주기성 계열 안에서 이동하는 셈입니다.

추천 순서는 이렇습니다.

1. **마이크 입력 제약부터 조정**
   - `getUserMedia`에서 가능하면 `echoCancellation: false`, `noiseSuppression: false`, `autoGainControl: false`를 요청해 보는 게 1순위입니다.
   - 브라우저/마이크가 실제로 지켜줄지는 장치별로 다르지만, MDN 기준으로 이런 제약은 지원 여부 확인과 요청이 가능합니다. 특히 noise suppression이나 auto gain은 음악 음정에는 오히려 파형을 망가뜨릴 수 있습니다.
   - 참고: MDN `echoCancellation`, `noiseSuppression`, `autoGainControl`.

2. **`pitchy`는 유지하되, FFT 기반 harmonic 검증을 붙이기**
   - `pitchy`가 낸 후보 `f`에 대해 `f / 2`, `f`, `2f` 주변의 스펙트럼 에너지를 비교합니다.
   - 단순히 “결과를 반으로 보정”하는 게 아니라, 후보 선택 시점에 “이 주파수가 기본음인가, 2배음인가”를 검사하는 방식입니다.
   - 예: `f`가 강하지만 `f/2`의 2배, 3배, 4배 위치에도 일관된 harmonic pattern이 있으면 `f/2` 후보를 더 신뢰합니다.
   - 구현 난이도는 CREPE보다 훨씬 낮고, 현재 이미 `fft.js`가 `pitchy` 의존성으로 들어와 있어 방향이 잘 맞습니다.

3. **비교용/실험용으로 Essentia.js 검토**
   - Essentia.js는 WebAssembly + JS/TypeScript API를 제공하고, `PitchYin`, `PitchYinFFT`, `PitchMelodia`, `MultiPitchKlapuri` 같은 후보가 있습니다.
   - 특히 `PitchMelodia` 계열은 harmonic salience와 contour 기반이라 배음 문제를 다루는 방향과 맞습니다.
   - 단, npm의 `essentia.js`는 AGPL-3.0으로 표시됩니다. 이 프로젝트 정책상 AGPL 계열은 직접 의존성으로 넣기 전에 배포 계획을 따로 판단해야 합니다. 그래서 저는 “직접 채택”보다 “비교 실험/알고리즘 참고”를 권합니다.

4. **CREPE는 후순위**
   - CREPE는 논문상 강력한 딥러닝 pitch estimator이고, TensorFlow.js로 브라우저 실행 자체는 가능합니다.
   - 하지만 GitHub Pages 앱에 넣기에는 모델 크기, 초기 로딩, 모바일/저사양 성능, TypeScript 통합, 테스트 복잡도가 확 커집니다.
   - “현재 DSP 구조로 해결이 안 된다”는 근거가 생긴 뒤 옵션으로 두는 게 좋아 보입니다.

제가 추천하는 안전장치는 이렇습니다.

1. **`f/2` 위치의 실제 에너지 요구**
   - `f/2` 자체 peak가 너무 약하면 `f/2` 후보를 버립니다.
   - “missing fundamental”까지 허용하면 다시 위험해지므로, 처음 구현은 실제 저주파 성분이 있는 경우만 인정하는 게 안전합니다.

2. **홀수 배음 확인**
   - 실제 `f/2`가 기음이면 `f/2`, `3f/2`, `5f/2` 같은 홀수 배음 흔적이 나올 수 있습니다.
   - 반대로 실제 `f`를 연주한 경우에는 `f`, `2f`, `3f` 중심으로 강하고, `3f/2`, `5f/2`는 약할 가능성이 큽니다.
   - 따라서 `f/2` 후보는 `f`, `2f`만으로 인정하지 말고 홀수 배음 증거를 요구하는 편이 좋습니다.

3. **저역 통과 검증**
   - 후보가 `f`인데 `f/2`도 의심된다면, 원본 프레임을 `0.75f` 정도 아래로 low-pass한 뒤 다시 pitch를 봅니다.
   - 진짜 `f/2`가 있으면 저역만 남겨도 `f/2` 주기성이 남습니다.
   - 실제 `f` 연주라면 `f` 성분이 잘려서 안정적인 `f/2`가 나오기 어렵습니다.
   - 이건 “사후 보정”이라기보다 후보 검증 단계라서 부작용을 비교적 통제하기 좋습니다.

4. **점수표의 기대 음정과 너무 멀리 벗어난 후보는 보수적으로 처리**
   - Practice mode는 현재 타겟 음정을 알고 있으니, 검출기가 `target`보다 정확히 한 옥타브 위를 자주 내는 상황만 좁게 다룰 수 있습니다.
   - 다만 이걸 과하게 쓰면 사용자가 실제로 옥타브 위를 연주했을 때도 맞았다고 처리할 수 있으니, `f/2`의 스펙트럼 증거가 있을 때만 보조 조건으로 쓰는 게 좋습니다.

즉 구현 정책은:

```text
pitchy 후보 f가 나옴
→ f/2 후보를 검사
→ f/2 실제 peak + 홀수 배음 + 저역 검증이 충분할 때만 f/2 채택
→ 증거가 애매하면 원래 f 유지 또는 unvoiced/low confidence 처리
```

이렇게 가야 합니다.

제일 피해야 할 방식은:

```text
타겟과 한 옥타브 차이 나면 그냥 /2
```

이건 이전 hysteresis류 보정처럼 새 부작용을 만들 가능성이 큽니다.

그래서 결론은, **f/2 오인식 가능성은 분명히 있고**, 그걸 막으려면 harmonic 검증을 “옥타브 내림 장치”가 아니라 **기음 후보 간 confidence 비교기**로 설계해야 합니다. 첫 구현은 보수적으로, `f/2` 증거가 아주 강할 때만 바꾸는 쪽이 좋겠습니다.

네. **켑스트럼 분석은 실험해볼 가치가 충분히 있습니다.** 다만 지금 문제에 대한 “완전한 해답”이라기보다는, `pitchy`와 다른 관점의 **비교용 pitch backend**로 붙여보는 게 좋아 보입니다.

켑스트럼이 기대되는 이유는 이렇습니다.

- 일반 spectrum은 `A#4` 같은 강한 배음 peak를 그대로 크게 봅니다.
- 켑스트럼은 spectrum peak 자체보다 **harmonic spacing**, 즉 반복 간격을 봅니다.
- 그래서 기본음 `A#3` 자체가 약해도, `A#4`, `F5`, `A#5...` 같은 배음 간격이 `A#3` 주기를 지지하면 `A#3` 쪽 quefrency peak가 나올 수 있습니다.
- 이론적으로는 “배음 중 가장 큰 peak”에 끌려가는 문제를 줄일 수 있습니다.

하지만 한계도 분명합니다.

- 마이크 입력에 `A#4` 단일 성분만 거의 남으면 켑스트럼도 `A#3`를 복구하기 어렵습니다.
- 짧은 window에서는 저음 주기 해상도가 떨어집니다. A#3는 약 4.29ms 주기라 아주 불리하진 않지만, 안정성을 위해 window/overlap 조정이 필요합니다.
- 프렛리스 악기의 gliss/vib처럼 pitch가 움직이는 신호에서는 window가 길면 반응이 둔해집니다.
- 켑스트럼 peak picking도 octave/subharmonic 오류가 생길 수 있어서 범위 제한과 smoothing은 여전히 필요합니다.
