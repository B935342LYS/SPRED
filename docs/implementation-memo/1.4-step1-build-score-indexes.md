# 1.4. 1단계 Score 인덱스 생성 메모
> 본 설계 메모 문서 1.4에서는 1단계 구현 중 `build_score_indexes.ts`의 역할과 구현 판단을 기록한다.
> 본 문서는 active specification을 대체하지 않는다.
> 함수별 세부 설명은 코드 파일의 `/** ... */` 주석을 중심으로 하고, 본 문서는 흐름과 사용 문법, 설계 판단을 보충한다.

## 1. 개요

`score_validate.ts`를 통과한 `ScoreFile`은 저장 구조로서는 유효하지만, 후속 parser/analyzer/renderer가 반복 조회하기에는 배열 중심 구조이다.
따라서 로드 직후 한 번 `ScoreIndexes`를 생성하여 자주 쓰는 조회 축을 `Map` 기반으로 정규화한다.

현재 score 모듈의 흐름은 아래와 같다.

```text
JSON text
-> loadScoreJson()
-> validateScoreFile()
-> ScoreFile
-> buildScoreIndexes()
-> ScoreIndexes
```

`ScoreIndexes`는 새로운 저장 데이터가 아니라, 특정 `ScoreFile`에서 파생된 런타임 조회 구조이다.
따라서 JSON export 대상이 아니며, 악보 원본이 바뀌면 다시 생성해야 한다.

## 2. build_score_indexes.ts의 현재 책임

`build_score_indexes.ts`는 validator 통과 이후의 `ScoreFile`을 입력으로 받아 `ScoreIndexes`를 생성한다.
이 단계에서는 문자열 파싱, analyzer 이벤트 생성, 렌더링 계산을 수행하지 않는다.

현재 생성하는 인덱스는 다음과 같다.

- `rowById`
  - `rowId -> RowDefinition`
  - 행 종류와 note/global/gap 세부 정보를 빠르게 조회한다.
- `rowsInDisplayOrder`
  - `layout.rowDefinitions`의 표시 순서를 보존한 배열
  - renderer가 행 순서를 그대로 사용할 수 있게 한다.
- `noteRowIdsByStringId`
  - `stringId -> RowId[]`
  - 특정 현에 속한 note row 목록을 찾는다.
- `noteRowIdByStringMidi`
  - `${stringId}|${midi} -> RowId`
  - analyzer의 `@n(midi_num)` 같은 음정 기반 참조를 실제 행으로 연결하기 위한 기반이다.
- `trackById`
  - `trackId -> Track`
  - 트랙 단위 parser/analyzer 진입점에서 사용한다.
- `cellMapByTrackId`
  - `trackId -> (rowId|col -> ScoreCell)`
  - 단일 note cell 좌표 조회에 사용한다.
- `cellsByTrackAndCol`
  - `trackId -> (col -> ScoreCell[])`
  - 열 범위 기반 partial parse/analysis의 기본 축이다.
- `globalCellMapByCoord`
  - `rowId|col -> GlobalCell`
  - 단일 global cell 좌표 조회에 사용한다.
- `globalCellsByKindAndCol`
  - `GlobalKind -> (col -> GlobalCell)`
  - 전역 종류별 단건 조회와 parser 입력 구성에 사용한다.
- `globalCellsInColOrder`
  - `GlobalKind -> GlobalCell[]`
  - timing/dynamics timeline 생성처럼 열 순서 순회가 필요한 단계에서 사용한다.

## 3. 구현 판단

### 3.1. validator 이후 함수로 둔 이유

`buildScoreIndexes()`는 입력을 다시 세부 검증하지 않고, `score_validate.ts`를 통과한 `ScoreFile`이라는 전제에서 동작한다.
인덱스 생성 함수가 검증까지 반복하면 모듈 책임이 겹치고, 오류 정책도 분산된다.

현재 책임 분리는 다음과 같다.

```text
json_load.ts          : JSON 문법과 최상위 object 확인
score_validate.ts     : ScoreFile 최소 구조와 참조 무결성 확인
build_score_indexes.ts: 검증된 ScoreFile의 런타임 조회 구조 생성
```

다만 `buildGlobalCellsByKindAndCol()`과 `buildGlobalCellsInColOrder()`에서는 `row?.type !== "global"` 검사를 한 번 더 수행한다.
이는 validator를 대체하려는 검사가 아니라, TypeScript의 discriminated union 좁히기를 위해 필요한 방어적 타입 확인이다.

### 3.2. rowsInDisplayOrder를 복사한 이유

`rowsInDisplayOrder`는 현재 `score.layout.rowDefinitions`와 같은 순서를 가진다.
구현에서는 원본 배열을 그대로 참조하지 않고 얕은 복사 배열을 만든다.

```ts
const rowsInDisplayOrder = [...score.layout.rowDefinitions];
```

이 배열은 표시 순서용 파생 view라는 의미를 가진다.
현재 단계에서 row 객체 자체를 복제하지는 않는다.
행 정의 객체는 저장 구조의 원본이며, 인덱스는 그것을 빠르게 참조하기 위한 구조이기 때문이다.

### 3.3. 좌표 key 생성 helper를 둔 이유

cell 좌표 key는 `${rowId}|${col}` 형식을 사용한다.
이 형식을 여러 위치에서 직접 만들면 구분자나 순서가 섞일 위험이 있다.

따라서 다음 helper로 좌표 key 생성을 고정하였다.

- `createCellCoordKey()`
- `createGlobalCellCoordKey()`
- `createStringMidiKey()`

현재 helper는 파일 내부 함수이지만, 이후 여러 모듈에서 같은 key 생성 규칙이 필요해지면 별도 공용 모듈로 이동할 수 있다.

### 3.4. 정렬은 globalCellsInColOrder에만 적용

`globalCellsInColOrder`는 analyzer가 시간축을 순회할 때 사용할 배열이므로 `col` 오름차순 정렬을 보장한다.
같은 `col` 안에서는 안정적인 순서를 위해 `rowId`를 보조 기준으로 사용한다.

반면 `cellsByTrackAndCol`은 특정 열에 속한 셀 묶음을 찾는 구조이므로, 현재 단계에서는 내부 배열 정렬을 강제하지 않는다.
track cell의 열 내부 순서가 필요한 시점이 오면 parser/analyzer 쪽 요구에 맞춰 정렬 기준을 추가한다.

## 4. 이번 구현에서 사용한 주요 TypeScript 문법

### 4.1. Map 중첩 구조

이번 구현에서는 여러 축의 조회를 표현하기 위해 중첩 `Map`을 사용한다.

```ts
Map<TrackId, Map<CellCoordKey, ScoreCell>>
Map<TrackId, Map<number, ScoreCell[]>>
Map<GlobalKind, Map<number, GlobalCell>>
```

첫 번째 key는 큰 분류축이고, 두 번째 key는 좌표나 열이다.
이렇게 나누면 특정 트랙 또는 특정 전역 종류를 먼저 좁힌 뒤, 그 내부에서 빠르게 조회할 수 있다.

### 4.2. nullish coalescing

Map에서 기존 배열 또는 내부 Map을 가져오고, 없으면 새 값을 만드는 패턴에 `??`를 사용한다.

```ts
const rowIds = noteRowIdsByStringId.get(row.stringId) ?? [];
const cellsByCol = globalCellsByKindAndCol.get(row.kind) ?? new Map();
```

`??`는 왼쪽 값이 `null` 또는 `undefined`일 때만 오른쪽 값을 사용한다.
Map의 `get()`은 key가 없으면 `undefined`를 반환하므로 이 패턴이 적합하다.

### 4.3. Template literal type

좌표 key와 현/음정 key는 template literal type으로 표현한다.

```ts
type CellCoordKey = `${RowId}|${number}`;
type GlobalCellCoordKey = `${RowId}|${number}`;
```

구현 함수의 반환 타입도 이 형식에 맞춘다.

```ts
function createStringMidiKey(
  stringId: StringId,
  midi: number,
): `${StringId}|${number}`
```

문자열 key가 단순한 `string`이 아니라 어떤 조합 규칙을 가지는지 타입과 코드 양쪽에 남길 수 있다.

### 4.4. Discriminated union 좁히기

`RowDefinition`은 `type` 필드로 구분되는 union이다.
따라서 `row.type !== "note"` 또는 `row?.type !== "global"` 같은 조건을 사용하면 TypeScript가 해당 분기에서 row 타입을 좁힌다.

```ts
if (row.type !== "note") {
  continue;
}

noteRowIdByStringMidi.set(
  createStringMidiKey(row.stringId, row.midi),
  row.rowId,
);
```

위 코드에서 조건문 이후에는 `row`가 `NoteRowDefinition`으로 좁혀지므로 `stringId`, `midi`에 접근할 수 있다.

### 4.5. Array.prototype.sort()

`globalCellsInColOrder`는 `Array.prototype.sort()`와 비교 함수를 사용해 정렬한다.

```ts
cells.sort(compareCellsByColThenRowId);
```

비교 함수는 먼저 `col`을 비교하고, 같은 열이면 `rowId` 문자열을 비교한다.

```ts
return left.rowId.localeCompare(right.rowId);
```

이 정렬은 전역 timeline 생성 단계가 셀 배열을 열 순서대로 순회할 수 있게 하기 위한 준비 작업이다.

## 5. 테스트 확인

`dev/test_score.ts`는 현재 fixture를 `loadScoreFile()`로 로드한 뒤 `buildScoreIndexes()`를 호출한다.
그리고 다음 기본 대응 관계를 확인한다.

- `indexes.rowById.size === score.layout.rowDefinitions.length`
- `indexes.trackById.size === score.tracks.length`
- `indexes.globalCellMapByCoord.size === score.globalLines.cells.length`

현재 fixture 기준 출력은 다음과 같다.

```text
Score load succeeded.
title: testScore
rows: 72
tracks: 1
global cells: 4
indexed rows: 72
indexed tracks: 1
indexed note rows for s1: 34
```

## 6. 다음 작업

다음 구현 대상은 `create_runtime_document.ts`이다.

```text
loadScoreFile()
```
