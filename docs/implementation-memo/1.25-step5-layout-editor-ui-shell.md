# 1.25. Step5 레이아웃 편집기 UI Shell 구현 메모

## 1. 목적

이 메모는 첫 layout editor UI shell 구현을 기록한다.

이 문서는 `docs/2.4-layout-edit-ui-spec.md`를 보완하며, active layout editing specification을 대체하지 않는다.

구현 범위는 type definition, dialog structure, 기본 DOM binding, visual layout 안정화로 제한했다.
실제 layout draft mutation, score apply, cell deletion, local preset storage, file preset import/export는 아직 연결하지 않은 상태였다.

갱신 메모:

- 이 문서는 초기 UI shell 상태만 기록한다.
- 이후 연결된 draft/apply/preset 구현은 `docs/implementation-memo/1.26-step5-layout-draft-apply-preset.md`에 기록되어 있다.

## 2. 구현된 범위

구현된 범위는 다음과 같다.

- layout edit type definition
- `Layout` dialog shell
- `Modify` button에서 dialog open
- dialog close/cancel/reset shell behavior
- Apply/Add/Save/Load action placeholder notice
- 현재 `ScoreFile.instData` 기반 instrument summary 표시
- string selector 채우기
- note/gap, height, pitch, insert below/above add-row control
- disabled open-string UI placeholder
- dialog control의 CSS layout 안정화

아직 구현하지 않은 범위는 다음과 같았다.

- score row에서 실제 `LayoutDraftBundle` 생성
- 편집 가능한 row list 렌더링
- 기존 row height update
- row insertion/deletion logic
- draft validation
- score apply와 runtime rebuild
- incompatible cell deletion confirmation
- localStorage layout preset persistence
- layout preset JSON file save/load

## 3. 관련 파일

- `regression-code/src/app/layout/layout_types.ts`
  - draft, preset, deletion summary, apply result type을 정의한다.
- `regression-code/index.html`
  - `#layout-editor` dialog와 layout toolbar의 `Modify` button class를 추가한다.
- `regression-code/styles/app.css`
  - layout dialog, preset, instrument, row list, add-row control style을 추가한다.
- `regression-code/src/app/app_types.ts`
  - `AppDom`에 layout dialog DOM reference를 추가한다.
- `regression-code/src/app/app_dom.ts`
  - 정적 layout dialog DOM element를 수집한다.
- `regression-code/src/app/app_view_binding.ts`
  - layout dialog를 열고, 현재 instrument/string shell value를 채우며, placeholder action을 연결한다.
- `regression-code/src/app/app_ui_sync.ts`
  - app이 busy 상태일 때 layout modify button을 비활성화한다.
- `regression-code/src/app/pitch_label.ts`
  - 전체 `0..127` layout pitch dropdown을 채우는 `populateLayoutPitchOptions()`를 추가한다.
- `regression-code/src/app/main.ts`
  - boot 중 layout pitch option을 채운다.
- `docs/2.4-layout-edit-ui-spec.md`
  - layout editor의 UI와 storage decision을 기록한다.

## 4. UI 판단

Dialog는 현재 사용자에게 보이는 이름으로 layout preset name 하나만 사용한다.
별도 Instrument Name input은 두지 않는다.
Apply path가 구현되면 이 preset name을 `instData.instName`에도 적용할 계획이었다.

Instrument `family` field는 free text input이 아니라 dropdown으로 둔다.
현재 option은 string instrument 중심 예시이다.

```text
Otamatone
Haegeum
Violin
Cello
Gayageum
Custom
```

Open-string checkbox와 open MIDI dropdown은 보이지만 disabled 상태로 둔다.
이는 future UI 위치를 안정적으로 유지하면서, 현재 Otamatone 중심 단계에서는 active open-string editing을 피하기 위한 선택이다.

Add Row 영역은 insertion-position dropdown 대신 직접 insertion button 두 개를 사용한다.

```text
Insert Below Selected
Insert Above Selected
```

이 방식은 중간 선택 단계를 줄이고 action target을 더 분명하게 보여준다.

## 5. 당시 Dialog 동작

`Modify`를 누르면 dialog가 현재 `ScoreFile.instData`를 읽어 다음 값을 채웠다.

- preset name
- family
- open-string checkbox state
- string selector
- string summary rows
- add-row pitch dropdown

Row list는 의도적으로 빈 placeholder를 표시했다.
다음 구현 단계에서는 이 placeholder를 선택 string의 실제 note/gap row로 교체해야 했다.

`New Preset` button은 preset dropdown에 임시 option을 추가하고 name input을 채웠다.
이는 UI shell 동작일 뿐이며, localStorage나 file data를 쓰지 않았다.

## 6. 다음 단계로 계획했던 작업

다음 구현 단계에서는 `layout_draft.ts`를 도입하는 것이 적절하다고 보았다.

권장 범위는 다음과 같았다.

- 현재 `ScoreFile`에서 `LayoutDraftBundle` 생성
- 선택 string 기준 note/gap row filtering
- 기존 row를 row list에 렌더링
- 기존 note/gap row height를 draft 안에서 편집
- draft rendering path가 안정될 때까지 add/delete/apply/storage는 placeholder로 유지

이렇게 하면 score mutation을 도입하기 전에 사용자 편집 가능한 draft state에 집중할 수 있다.

## 7. 검증

UI shell 변경은 다음 명령으로 검증했다.

```text
npm run typecheck
npm run build
```

최종 UI alignment adjustment 이후 두 명령이 모두 통과했다.
