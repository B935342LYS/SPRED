# 1.26. Step5 Layout Draft Apply Preset

## 1. Purpose

This memo records the current layout editor implementation after the draft, apply, local slot preset, file preset, and toolbar preset flows were connected.

It supplements:

- `docs/2.4-layout-edit-ui-spec.md`
- `docs/2.5-layout-preset-format-spec.md`

The memo is implementation commentary only. Storage format and active behavioral rules remain defined by the active specs.

## 2. Implemented Scope

Implemented:

- `LayoutDraftBundle` creation from the current `ScoreFile`
- selected string and selected row state for the layout dialog
- editable note/gap row list rendering
- common note row height editing through one `Note Height` input
- gap row height editing through per-gap inputs and preview drag handles
- note row add/delete draft mutations
- gap row add/delete draft mutations
- draft preview with actual row heights
- draft -> `ScoreFile` apply
- deletion summary for track cells attached to removed note rows
- user confirmation before layout apply deletes track cells
- full rebuild after layout apply
- local layout preset Slot 1..3 save/load
- layout preset JSON file save/load
- toolbar dropdown integration for Default Layout and local slots

## 3. Key Files

- `regression-code/src/app/layout/layout_draft.ts`
  - Creates and mutates layout draft state.
  - Handles row selection, common note height, note/gap insertion, note/gap deletion, and gap boundary calculation.
- `regression-code/src/app/layout/layout_apply.ts`
  - Applies a draft to a cloned `ScoreFile`.
  - Preserves global rows.
  - Replaces note/gap rows.
  - Removes invalid track cells only after the caller allows deletion.
  - Calls `validateScoreFile()` as the final ScoreFile guard.
- `regression-code/src/app/layout/layout_preset.ts`
  - Creates `UserLayoutPresetData`.
  - Parses and validates preset JSON.
  - Restores a draft from preset data.
  - Creates preset JSON file names.
- `regression-code/src/infra/layout_preset_storage.ts`
  - Stores local layout presets in fixed Slot 1..3 keys per `instrumentPresetId`.
- `regression-code/src/app/app_view_binding.ts`
  - Currently owns layout dialog binding, preview drag handling, apply confirmation, local/file preset actions, and toolbar preset actions.
- `regression-code/src/app/app_runtime.ts`
  - Adds `applyLayoutDraftEditToState()` for draft apply + full rebuild.
- `regression-code/src/infra/score_file_io.ts`
  - Adds generic JSON/text download helpers reused by layout preset file save.
- `regression-code/dev/test_layout_apply.ts`
  - Covers layout apply, deletion blocking/allowing, preset round-trip, and local slot storage behavior.

## 4. Draft Editing Decisions

Note row height is handled as a common value.
This matches the idea that note row height represents a finger contact point, not a string-specific physical distance.
Each note row still stores its own `height` field in the final row definition, but the UI edits all note rows together.

Note row MIDI selection is automatic.
When the user inserts a note row above or below the selected row, the new MIDI is derived from the selected row and insertion direction.
The UI no longer asks the user to choose pitch manually for each new row.

Gap rows are no longer restricted to positions strictly between two note rows.
Top and bottom gap rows are allowed.
The boundary MIDI range is `-1..128` so gaps below MIDI 0 or above MIDI 127 can be represented if needed.

Gap rows cannot be adjacent.
This keeps the layout structure lighter and avoids suffix-based rowId generation for redundant gap rows.

Note row deletion is restricted to the top or bottom note row for the selected string.
Middle note row deletion is blocked to avoid destructive layout edits that are easy to trigger accidentally.
Gap rows can be deleted directly.

## 5. Apply Decisions

The apply path is a full replacement model:

```text
current ScoreFile
-> clone
-> keep existing global rows
-> replace note/gap rows from draft
-> delete track cells whose rowId no longer points to a note row
-> validateScoreFile(nextScore)
-> createRuntimeDocument()
-> buildParsedDocument()
-> analyzeDocument()
-> createCanvasRenderInput()
-> render
```

Cell deletion is not treated as a core validation error.
It is a normal but destructive app-level consequence of applying a layout that removed note rows.
Therefore deletion is summarized in `layout_apply.ts`, confirmed in the UI boundary, and then applied only when explicitly allowed.

`ScoreFile.musicData.createdAt` and `ScoreFile.musicData.updatedAt` are not copied from layout preset metadata.
Preset timestamps describe the preset itself, not the score.
Score timestamp updates remain part of score JSON save behavior.

## 6. Preset Decisions

The same `UserLayoutPresetData` structure is used for Local Save/Load and File Save/Load.

Local presets use fixed slots:

```text
layout-slot:{instrumentPresetId}:1
layout-slot:{instrumentPresetId}:2
layout-slot:{instrumentPresetId}:3
```

This replaces the earlier index-based local preset idea.
The fixed slot model was chosen because:

- localStorage space should be protected for the main score JSON
- the feature is for frequently used personal presets, not a large preset library
- slot UI makes overwrite behavior clear
- no separate delete flow is required for the MVP

File Save/Load has no slot limit.
It is used for backup, sharing, and moving presets across browsers or devices.

## 7. Toolbar Behavior

The layout toolbar dropdown now shows:

```text
Default Layout
Slot 1: ...
Slot 2: ...
Slot 3: ...
```

Selecting `Default Layout` is a no-op.
Selecting a filled local slot applies that preset immediately through the same deletion confirmation and full rebuild path as the dialog Apply button.
Selecting an empty slot falls back to Default and shows a warning message.

The toolbar was visually narrowed after Save/Reset buttons were removed from the outer toolbar.
Those actions remain inside the layout dialog where the draft context is clear.

## 9. Remaining Refactor Candidate

`app_view_binding.ts` currently contains a large amount of layout dialog and toolbar binding logic.
Functionally this is acceptable for the current report milestone, but the next cleanup should extract layout UI binding into a dedicated module such as:

```text
regression-code/src/app/layout/layout_dialog_binding.ts
```

That module can own:

- dialog rendering
- preview rendering and drag resize
- local/file preset buttons
- toolbar preset selection
- apply confirmation UI

`app_view_binding.ts` can then return to general view controls.
