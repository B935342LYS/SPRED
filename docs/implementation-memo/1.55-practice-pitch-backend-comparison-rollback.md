# 1.55 Practice Pitch Backend Comparison and Rollback

## 1. 목적

실사용 마이크에서 pitchy가 기본음 대신 2배음 또는 3배음을 선택하는 문제를 해결하기 위해, 동일 PCM을 pitchy, Essentia.js pYIN, SwiftF0 ONNX에 입력하고 제품 채택 가능성을 검증했다.

실험은 experiment/practice-pitch-detection 브랜치에서만 진행했고 target pitch 또는 직전 frame에 맞춘 보정은 사용하지 않았다.

## 2. 유지한 공통 비교 기반

다음 기반은 이후 aubio 비교에도 재사용할 수 있으므로 유지한다.

- GamePitchDetectorBackend와 pitchy adapter
- backend-owned confidence와 optional processing time
- 동기/Promise detector를 직렬화하는 single-flight runner
- debug frame JSON
- AudioWorklet mono PCM 캡처
- PCM16 WAV와 입력 metadata export
- mono PCM16 WAV decoder
- 저장 WAV의 pitchy baseline comparison
- frame center timestamp, F0, confidence, RMS, 처리 시간 기록

## 3. 실제 입력 corpus

첫 기준 corpus:

- spred-pitch-capture-2026-07-11T11-55-28-143Z.wav
- 48kHz, 1,309,952 samples, 27.291초
- pitchy confidence+RMS 통과 1,565 / 2,551 frames
- pitchy 평균 처리 약 0.583ms/frame
- 5 semitone 이상 jump 18회
- 반복 오류: 2x/0.5x octave burst, 약 3x/1/3x third-harmonic burst

두 번째 실사용 corpus:

- spred-pitch-capture-2026-07-12T01-29-25-370Z.wav
- 35.979초
- A#3/A#4 반복, 저음·전체 gliss, vibrato, 빠른 gliss 포함

## 4. 세 backend offline 비교

첫 corpus의 유효 시간 coverage:

- pitchy 61.2%
- pYIN 61.5%
- SwiftF0 threshold 0.9 46.1%

인접 유효 frame의 5 semitone 이상 jump:

- pitchy 18회
- pYIN 0회
- SwiftF0 threshold 0.9 0회

pYIN과 SwiftF0가 합의하고 pitchy만 약 2배로 올라간 구간이 확인됐다. 그러나 SwiftF0는 pitchy와 pYIN이 약 205Hz에 합의할 때 617Hz를, 약 249Hz에 합의할 때 1250Hz를 선택하는 독자적인 고배음 오류도 만들었다.

두 번째 corpus에서 pYIN timestamp를 약 -35ms 정렬한 뒤 세 backend가 모두 유효한 878개 시점을 분류한 결과:

- 세 backend 근접 552
- pitchy+SwiftF0 합의, pYIN 이탈 261
- pYIN+SwiftF0 합의, pitchy 이탈 38
- pitchy+pYIN 합의, SwiftF0 이탈 27

pitchy와 SwiftF0가 5 semitone 이상 충돌한 65 frames 중 약 58.5%는 pitchy 오류 가능성이 높았고 약 41.5%는 SwiftF0 오류 가능성이 높았다. 어느 한쪽을 충돌 시 무조건 채택할 근거가 없었다.

## 5. pYIN 실험과 실패 판단

구현:

- Essentia.js 0.1.3
- 공식 whole-signal PitchYinProbabilistic
- 4096 frame / 256 hop / low RMS 0.01
- HMM voiced/unvoiced 출력을 backend confidence 판정으로 사용
- whole-audio contour와 고유 timestamp 보존

실제 WAV에서는 pitchy 배음 오류 일부를 안정적으로 낮은 F0로 추적했다. 그러나 다음 문제가 확인됐다.

- 27.291초 입력 처리에 약 38.763초가 걸림
- whole-signal HMM이라 미래 문맥을 사용하며 실시간 frame 계약으로 직접 변환할 수 없음
- 0.5~2초 합성 gliss/step에서 약 1,000~2,200cent의 서브하모닉 오류 발생
- 배음을 포함한 합성 현악기형 입력에서도 octave-down 문제가 유지됨
- 새 실제 corpus에서 pitchy와 SwiftF0가 합의할 때 pYIN만 이탈한 frame이 261개
- Essentia.js는 AGPL-3.0이라 별도 라이선스 결정 없이 제품 의존성 채택 불가

결론:

- 일부 실제 오류를 설명하는 offline 참고 자료로는 유용했다.
- 정확도, 실시간 구조, 처리 시간, 라이선스 조건을 함께 통과하지 못했다.
- 제품 backend와 배포 dependency에서 제거한다.

## 6. SwiftF0 offline·합성 실험

구현:

- 공식 SwiftF0 ONNX model
- ONNX Runtime Web WASM
- 16kHz / 1024 frame / 256 hop
- browser OfflineAudioContext resampling
- backend-native contour timestamp

합성 현악기형 입력:

- 2초 200→400Hz gliss: median 5.74cent, p90 13.59cent
- 500ms 200→400Hz gliss: median 7.09cent, p90 16.61cent
- 250→500Hz step: median 10.5cent, 응답 약 12ms
- 처리 시간은 입력 길이의 약 0.12배

첫 corpus threshold sweep:

- 0.8: coverage 51.0%, large jump 2
- 0.85: coverage 48.5%, large jump 2
- 0.9: coverage 46.1%, large jump 0

두 번째 corpus:

- 0.85: coverage 42.8%, large jump 0
- 0.9: coverage 39.3%, large jump 0

threshold 0.9는 인접 jump를 제거했지만 실제 pitch dot에 필요한 voiced coverage가 부족했다. threshold를 낮추면 2배·3배·5배 고배음 오류가 다시 증가했다.

## 7. SwiftF0 sliding-window 실험

첫 corpus, 250ms update:

- 0.5초 window: coverage 61.1%, 평균 처리 72.33ms, p90 74.9ms, 중앙 전달 지연 약 84ms
- 1초 window: coverage 67.9%, 평균 처리 136.05ms, 중앙 전달 지연 약 152ms
- 2초 window: coverage 77.5%, 평균 처리 268.69ms, update interval 대비 처리 비율 1.075

0.5초 window만 실시간 후보로 남았다.

두 번째 실제 corpus, 0.5초 window:

- 100ms update: 평균 71.69ms, p90 74ms, 최대 189.4ms
- 150ms update: 평균 69.91ms, p90 72.3ms, 최대 95.5ms
- 250ms update: 평균 73.85ms, p90 86.3ms, 최대 124.6ms

150ms update가 가장 안정적이었지만 freshness 150ms 적용 coverage는 약 47.7%였다. update 주기를 줄여도 fresh coverage가 개선되지 않아 confidence/모델 출력이 병목임을 확인했다.

## 8. 실시간 conflict gate 실험

설정:

- window 500ms
- update 150ms
- confidence 0.9
- maximum lookback 150ms
- pitchy와 5 semitone 이상 다르면 해당 pitchy frame을 backend conflict로 거부
- SwiftF0가 없거나 stale이면 pitchy 유지

실사용 debug:

- spred-pitch-live-debug-2026-07-12T02-44-23-689Z.json
- 2,290 frames / 약 83.77초
- pitchy base voiced 963
- final voiced 937
- fresh verifier available 524 / 963, 54.4%
- conflict 26 frames, base voiced의 2.7%
- verifier age median 115.6ms, p90 141.0ms
- pitchy raw large jump 7회
- gate 이후 accepted large jump 1회

수치상 jump 6회를 막았지만 실사용에서는 배음 문제가 끊김으로 바뀌었고 지연과 gliss 반응이 악화됐다.

원인:

- gate는 틀린 음을 고치지 않고 pitch dot을 제거함
- verifier가 fresh한 비율이 54.4%라 나머지 pitchy 오류는 그대로 통과함
- verifier가 stale인 복귀 구간에서 511.5→257.2Hz, 510.9→256.2Hz 점프가 통과함
- 약 7.7초에는 pitchy 약 236Hz가 정상인 동안 SwiftF0가 약 1661Hz를 반환해 정상 frame을 거부함
- verifier age가 대부분 100~150ms라 빠른 gliss에서 서로 다른 시점의 결과가 충돌함

## 9. 최종 실패 판단

pYIN과 SwiftF0 모두 일부 pitchy 배음 오류를 피했지만 전체 실사용 안정성 기준을 통과하지 못했다.

- pYIN: short-window 정확도 실패, whole-signal HMM 구조, 실시간 처리 실패, AGPL 배포 gate
- SwiftF0 standalone: fresh voiced coverage 부족, confidence를 낮추면 고배음 오류 증가
- SwiftF0 replacement: 충돌 frame의 약 41.5%에서 SwiftF0가 틀려 무조건 교체 불가
- SwiftF0 conflict gate: false rejection, stale 비교, pitch dot 끊김, gliss 반응 저하
- 두 접근 모두 현재 pitchy보다 전체 실사용 안정성이 나빠짐

pYIN/SwiftF0 adapters, contour 전용 계약, 합성/threshold/window/conflict 도구, ONNX model, Essentia/ONNX 의존성을 작업 트리에서 제거한다.
