네, 구조 자체는 맞습니다.

- parser: `@g(id,S/M/E)`를 note modifier로 파싱
- analyzer: 같은 track/id의 S-M/E를 연결해 `GlissEvent` 생성
- game: `collectGameEffectBonusTargets()`가 active track의 `GlissEvent`를 gliss bonus target으로 변환
- playback loop: `judgeGlissIntervalBonus()`가 target 구간 안의 voiced frame을 보고 `Gliss!` 보너스 반영

카운트가 안 오른 실제 원인은 구조 부재가 아니라 lifecycle 버그였습니다. countdown 끝에서 `effectBonusTargets`를 수집한 직후, 실제 playback 시작 후 RAF를 정리하는 `stopPlaybackAnimation()` 호출이 practice 판정 상태까지 리셋하면서 `effectBonusTargets`를 비워버리고 있었습니다. 그래서 게임 모듈이 Gliss를 인식할 수 있는 구조는 있었지만, 재생 시작 순간 target 목록이 사라졌습니다.
