# Practice PCM 배음 식별 가능성 분석

## 1. 목적

Pitchy, pYIN, SwiftF0가 실제 마이크 입력에서 서로 다른 실패를 보인 뒤 detector 교체만 반복하지 않고, 저장된 PCM에 낮은 F0를 식별할 정보가 남아 있는지 먼저 확인한다.

핵심 구분은 다음과 같다.

- 경우 A: 실제 F0 성분은 약해도 3F, 5F 같은 홀수차 배음 또는 낮은 주기의 반복성이 남아 있어 낮은 F0를 복원할 단서가 있다.
- 경우 B: 2F, 4F, 6F 같은 짝수차 성분만 사실상 남아 단일 frame만으로 F0와 2F를 구별할 수 없다.

이 분석은 제품 pitch를 보정하지 않는다. 기존 PCM을 조사하는 debug 전용 오프라인 도구다.

## 2. 구현

추가 파일:

- `src/app/game/game_pitch_identifiability.ts`
  - 기존 Pitchy comparison frame의 검출 주파수를 `2F`로 가정한다.
  - 같은 PCM window에서 낮은 가설의 1F~6F 정현파 투영 진폭을 계산한다.
  - 1F·3F·5F와 2F·4F·6F의 RMS 비율을 dB로 기록한다.
  - 검출 주기와 낮은 가설 주기의 normalized autocorrelation을 비교한다.
  - 인접 frame의 약 2배 급상승을 별도 `octaveUpTransitions` 목록으로 기록한다.
- `dev/analyze_pitch_identifiability.ts`
  - WAV와 같은 basename의 `-pitchy-comparison.json`을 읽는다.
  - 기본적으로 최종 143Z corpus와 두 번째 실사용 corpus를 분석한다.
  - `-identifiability.json` 결과를 `피치테스트/`에 저장한다.
- `npm run analyze:pitch-identifiability`
  - 기본 corpus 두 개를 다시 분석한다.

판정 threshold는 해결 알고리즘이 아니라 탐색용 보수적 분류다.

- 낮은 홀수차/짝수차 비와 1F 또는 3F의 2F 대비 비가 모두 `-15dB` 이상이면 `lower-f0-supported`
- 두 값이 모두 `-24dB` 이하이면 `octave-ambiguous`
- 그 사이는 `insufficient-signal`

## 3. 합성 검증

`npm run test:game`에 다음 두 신호를 추가했다.

- 200, 400, 600, 800Hz: 낮은 200Hz 가설의 홀수차 배음이 있으므로 `lower-f0-supported`
- 400, 800, 1200Hz: 200Hz 가설에서는 짝수차만 있으므로 `octave-ambiguous`

두 판정이 모두 통과했다.

## 4. 기존 WAV 결과

### 143Z 최종 공통 corpus

- 분석 frame: 1,565
- 전체 frame 중 낮은 F0 지지: 54, 3.45%
- 전체 frame 중 옥타브 모호: 710, 45.37%
- 약 2배 급상승 전이: 7
- 전이 중 `lower-f0-supported`: 5
- 전이 중 `insufficient-signal`: 2
- 전이 중 `octave-ambiguous`: 0

대표 전이 시점은 6.539, 8.213, 8.715, 9.611, 10.613, 13.760, 16.213초다. 낮은 가설의 1F는 일부 구간에서 2F보다 약 30~44dB 약했지만, 3F는 다수 전이에서 2F보다 약 6~14dB 약한 수준으로 남았다. 낮은 주기의 autocorrelation도 대부분 약 `+0.2` 우세했다.

### 두 번째 실사용 corpus

- 분석 frame: 2,118
- 전체 frame 중 낮은 F0 지지: 112, 5.29%
- 전체 frame 중 옥타브 모호: 758, 35.79%
- 약 2배 급상승 전이: 9
- 전이 중 `lower-f0-supported`: 6
- 전이 중 `insufficient-signal`: 3
- 전이 중 `octave-ambiguous`: 0

## 5. 해석

전체 frame 비율은 대부분 정상 검출 frame도 `검출값/2` 가설로 다시 검사하므로 오류율이나 해결 가능 비율로 해석하면 안 된다. 판단에 직접 사용하는 자료는 실제 약 2배 급상승 전이다.

두 corpus의 16개 급상승 전이에서 `octave-ambiguous`가 한 건도 없었고, 11건은 낮은 F0를 지지했다. 따라서 대표 octave-up 오류는 현재 자료상 경우 B보다 경우 A에 가깝다. 특히 실제 F0 자체보다 3F와 낮은 주기 자기상관이 중요한 단서다.

이 결과는 전통 DSP가 반드시 제품 수준으로 해결한다는 뜻은 아니다. 빠른 글리산도, attack/decay 변화, frame 경계, 후보 점수 정책 때문에 단서가 있어도 detector가 안정적으로 선택하지 못할 수 있다. 다만 “PCM에 정보가 완전히 없다”는 가설만으로 CREPE 도입을 정당화할 수는 없게 되었다.

## 6. 다음 판단

다음 실험은 두 갈래를 병행할 가치가 있다.

1. 기존 알고리즘 경로
   - 급상승 전이 주변의 3F 단서와 autocorrelation 후보 점수를 이용해 aubio 같은 독립 기준 구현이 낮은 F0를 선택하는지 확인한다.
   - 2F notch는 제품 전처리가 아니라 원인 확인용 진단으로만 사용한다.
2. 최신 AI 모델 경로
   - CREPE 계열 또는 최근 악기 F0 모델을 동일 WAV에 offline 적용한다.
   - 정확도뿐 아니라 voiced coverage, gliss 반응, 처리 지연, model 크기, 브라우저 runtime, 라이선스를 함께 측정한다.
   - AI 결과도 악기·악보 prior에 의한 추정일 수 있으므로 raw PCM의 객관적 정답과 동일시하지 않는다.

기존 자료로 부족한 항목은 stereo 원채널별 위상과 PCM16 이전의 약한 성분이다. 현재 결과가 경계선인 구간만 이후 Float32·채널별 캡처로 재검증한다.
