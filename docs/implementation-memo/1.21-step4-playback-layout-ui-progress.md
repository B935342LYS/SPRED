# 1.21. 4단계 playback / layout / UI progress 구현 메모
> 본 구현 메모 문서 1.21에서는 pause/seek playback 상태, player metadata/details 연결, score view helper, edit preview sound, app playback 폴더 분리, layout compatibility 계획을 기록한다.
> 본 문서는 active specification을 대체하지 않는다.
> 함수별 세부 설명은 코드 파일의 `/** ... */` 주석을 중심으로 하고, 본 문서는 보고서와 발표에 필요한 흐름, 구현 판단, 핵심 문법을 보충한다.

## 1. 개요

이번 구현 구간은 기본 playback 연결 이후 실제 편집/검증 흐름에서 필요한 조작을 보강하는 단계였다.

이전 단계의 playback은 주로 아래 흐름을 검증했다.

```text
AnalysisResult
-> AudioSchedule
-> LookaheadScheduler
-> OscillatorBackend
-> play / stop
```

이번 단계에서는 playback이 단순 실행 버튼이 아니라 사용자 조작 상태를 가진 기능으로 확장되었다.

```text
stopped / playing / paused
-> pause / resume / seek
-> scheduler clear and restart
-> score time display
-> current BPM display
```

또한 metadata 표시, Details 편집, Fit Height / Fullscreen, edit preview sound, layout test fixtures, layout replacement policy까지 함께 정리하였다.

## 3. Playback state와 pause/seek

### 3.1. 상태 분리

Playback은 단순 boolean보다 명시적인 상태가 필요해졌다.
`playing` 안에서도 현재 score time과 audio clock origin이 필요하고, `paused`는 현재 score second를 보존해야 하기 때문이다.

현재 상태 구조는 다음 의미를 가진다.

```text
stopped
  - 재생 중인 backend node와 scheduler가 없다.
  - seek 위치는 0 또는 마지막 사용자가 선택한 score second이다.

playing
  - scheduler가 동작 중이다.
  - score second는 audio clock origin과 elapsed time에서 계산한다.

paused
  - scheduler와 backend node는 정리되어 있다.
  - 다시 resume할 score second를 보존한다.
```

### 3.2. seek 처리

seek는 schedule 자체를 다시 만들지 않는다.
현재 `AudioSchedule`은 score 전체에 대한 정렬된 event 목록이므로, seek 시에는 scheduler의 소비 상태와 backend node만 정리하고 같은 schedule 위에서 새 시작 score second를 지정한다.

```text
seekToSeconds(scoreSeconds)
-> stop active backend nodes
-> clear scheduler scheduled keys
-> reset controller origin
-> if playing, restart scheduler from scoreSeconds
```

이 방식은 구현 비용이 낮고, score 내용이 바뀌지 않는 seek 조작과 잘 맞는다.
score mutation이 발생한 경우에만 schedule을 새로 만든다.

## 4. Seek UI와 현재 BPM 표시

Seek slider는 score seconds를 직접 나타낸다.
표시는 소수점 seconds 대신 `mm:ss`로 단순화했다.

```text
00:00 / 01:24
```

기존 `stepMs` 표시 영역은 현재 BPM 표시로 바꾸었다.
현재 BPM은 다음 흐름으로 계산한다.

```text
current score seconds
-> timeMapper.secondsToTick()
-> analysis.timingTimeline segment lookup
-> constant BPM or linear BPM interpolation
```

이 때문에 BPM ramp가 있는 경우에도 현재 재생 위치의 보간 BPM을 표시할 수 있다.
단, BPM 표시는 playback 위치 표시용이며 beat/bar marker의 기준을 바꾸지 않는다.

## 5. Metadata와 Details

중앙 player 영역은 `ScoreFile.musicData.musicTitle`, `ScoreFile.musicData.musicArtist`를 표시한다.
Details dialog에서는 생성일과 수정일을 제외한 `musicData` 주요 필드를 편집할 수 있다.

현재 편집 대상은 다음이다.

- title
- artist
- genre
- score writer
- comment
- basic / optional / extra difficulty
- YouTube video id
- YouTube offset ms

difficulty 입력은 최대 두 자리 정도의 짧은 값을 전제해 한 줄에 배치했다.
YouTube video id와 offset은 약 8:2 비율로 배치해 긴 링크성 입력과 짧은 offset 입력의 성격을 나누었다.

## 6. Score view helper

Fit Height는 score canvas 시작 높이부터 창 하단까지 악보 row 전체가 들어오도록 zoom을 계산한다.
Fullscreen은 전체 app shell에 적용한다.

수동 zoom은 100% 아래로 내려가지 않게 제한했다.
다만 Fit Height는 악보 전체 높이를 맞추기 위해 내부적으로 100% 미만 zoom도 사용할 수 있다.

Edit mode를 켤 때는 한 번 Fit Height를 자동 실행한다.
작은 화면에서 edit panel이 score 영역을 줄이기 때문에, 편집 진입 직후 악보 전체를 다시 맞추기 위한 처리이다.

## 7. Edit preview sound

Edit mode에서 pointer down 또는 drag hit가 note row에 닿으면 해당 row의 pitch를 짧게 재생한다.

드래그 중 같은 셀 또는 가까운 row에서 과도하게 oscillator가 겹치면 지직거리는 소리가 발생할 수 있다.
이를 줄이기 위해 현재 구현은 다음 제한을 둔다.

- edit mode에서만 preview sound를 재생한다.
- drag preview는 같은 rowId에 대해 반복 재생하지 않는다.
- preview 사이에 최소 시간 간격을 둔다.
- 기존 preview를 즉시 강제 종료하지 않고 짧은 envelope로 자연스럽게 사라지게 한다.

이 기능은 최종 playback 품질이 아니라 edit feedback이 목적이므로, 짧고 낮은 volume으로 유지한다.

## 8. App playback 폴더 분리

앱 계층의 playback 관련 모듈이 늘어나면서 `src/app/playback/` 폴더를 만들었다.

```text
src/app/playback/
  app_playback.ts
  app_note_preview.ts
```

`main.ts`는 여전히 많은 UI orchestration을 담당하지만, playback controller 생성과 edit preview backend는 별도 파일로 이동했다.
이 분리는 edit 폴더 분리와 같은 방향이며, 후속 loop playback, accompaniment, YouTube sync가 들어올 때 app root가 더 비대해지는 것을 줄이기 위한 준비이다.

## 9. Layout test fixtures와 compatibility 정책

레이아웃 변경 기능을 검토하기 위해 내용물이 비어 있는 테스트 score file을 루트에 추가했다.

- `layout-test-2oct-gap21.json`
  - 2옥타브 음역대
  - gap row 높이 21px 고정
- `layout-test-3oct-no-gap.json`
  - 3옥타브 음역대
  - gap row 없음

또한 layout replacement에서 기존 셀이 새 layout과 호환되지 않는 경우의 정책을 `docs/1.3-score-json-format.md`에 추가했다.

핵심은 import 검증과 사용자 조작을 분리하는 것이다.

- 외부 JSON import
  - 존재하지 않는 `rowId` 참조는 유효하지 않은 파일이므로 실패한다.
- 이미 로드된 score에 layout을 교체하는 UI 조작
  - 호환되지 않는 셀을 먼저 요약한다.
  - 사용자가 `No`를 선택하면 아무 것도 바꾸지 않는다.
  - 사용자가 `Yes`를 선택하면 호환되지 않는 셀을 삭제하고 layout을 적용한다.

MIDI가 같은 note row로 자동 이동하는 기능은 첫 구현에 포함하지 않는다.
이는 삭제보다 더 큰 의미 변경이므로 별도 reconciliation 기능으로 분리하고, 사용자 확인과 warning log를 거쳐야 한다.

## 10. 검증

이번 구간에서 확인한 주요 명령은 다음이다.

```text
npm run typecheck
npm run build
npm run test:audio
```

layout test fixture는 `loadRuntimeDocument()`로 로드 가능함을 별도로 확인했다.
루트에서 `npx tsx`를 실행했기 때문에 package 기준 실행 위치와 달라 npm 경고가 출력되었지만, JSON 구조 검증 자체는 통과했다.
