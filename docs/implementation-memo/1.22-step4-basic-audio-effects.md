# 1.22. 4단계 basic audio effect 구현 메모
> 본 구현 메모 문서 1.22에서는 기본 note event의 vibrato / tremolo audio 적용을 기록한다.
> 본 문서는 active specification을 대체하지 않는다.
> 함수별 세부 설명은 코드 파일의 `/** ... */` 주석을 중심으로 하고, 본 문서는 구현 판단과 참고 출처를 보충한다.

## 1. 개요

이번 구현에서는 analyzer가 만든 `NoteEvent.effects`를 audio backend에서 실제 Web Audio 효과로 해석하기 시작했다.
추가로 `GlissEvent`를 audio schedule 단계에서 fallback bridge 발음으로 변환하는 첫 경로를 연결했다.

대상은 기본 note event에 포함된 두 effect이다.

```text
NoteEvent.effects
-> AudioScheduleEffect[]
-> OscillatorBackend
-> Web Audio node / AudioParam automation
```

첫 구현 범위는 아래로 제한했다.

- vibrato: oscillator detune에 LFO를 연결한다.
- tremolo: oscillator와 note envelope 사이에 gain gate를 둔다.
- gliss: 별도 ramp oscillator를 짧은 fade in/out envelope와 함께 예약한다.
- connected gliss: 같은 gliss id의 인접 세그먼트를 하나의 oscillator chain으로 예약한다.
- dynamics, note 병합 voice span은 다음 단계에서 별도로 다룬다.

## 3. Vibrato 구현

Vibrato는 Tone.js의 `Vibrato`처럼 주파수 LFO와 depth 값을 분리한 모델을 참고했다.

현재 Web Audio 구현은 아래 구조를 사용한다.

```text
LFO OscillatorNode
-> depth GainNode
-> note oscillator.detune
```

초기 기본값은 legacy 구현과 맞춰 아래 값으로 두었다.

```text
rate: 5 Hz
depth: 35 cents
```

effect segment가 시작되면 depth gain을 35 cents로 만들고, 종료 시 0으로 되돌린다.

## 4. Tremolo 구현

Tremolo는 재타격 oscillator를 여러 개 만드는 방식이 아니라 gain gating 방식으로 구현했다.

현재 Web Audio node 흐름은 아래와 같다.

```text
note oscillator
-> tremolo GainNode
-> envelope GainNode
-> master GainNode
```

`division`은 tick당 tremolo 분할 수로 해석한다.
schedule 단계에서 `durationTicks`를 함께 내려보내기 때문에, backend는 긴 segment에서도 대략 `durationTicks * division` 개수의 pulse를 만들 수 있다.

각 pulse는 완전한 0 대신 작은 gain 값까지 내려간다.
클릭 노이즈를 줄이기 위해 on/off 경계에 짧은 ramp를 둔다.

## 5. 구현 판단

Tremolo를 gain gating으로 둔 이유는 다음과 같다.

- legacy 문서에서 “실제로 끄지 않고 볼륨을 조절해서 꺼진 것 같은 효과”를 허용한다.
- legacy gliss+trem 구현도 oscillator 하나에 gate gain을 거는 방식에 가깝다.
- oscillator를 pulse마다 새로 만들지 않으므로 scheduling 부담이 작다.
- 이후 gliss pitch ramp와 동시에 적용할 때 pitch는 oscillator frequency, tremolo는 gain으로 분리할 수 있다.

Vibrato와 tremolo가 같은 note에 이어서 나타나는 경우는 analyzer 단계에서 이미 `vib > trem` 규칙을 effect segment로 반영한다.
backend는 전달된 segment를 그대로 해석한다.

## 6. Gliss fallback 구현

`GlissEvent`는 active spec에서 기본적으로 "두 anchor 사이의 관계 이벤트"로 정의되어 있고, analyzer 의미상 독립 발음 이벤트는 아니다.
이번 구현은 이 의미를 바꾸지 않고, audio schedule 단계에서만 fallback 발음 이벤트를 파생한다.

현재 Web Audio 구현은 아래 구조를 사용한다.

```text
gliss fallback oscillator
-> tremolo GainNode (start anchor에 trem이 있을 때)
-> fallback gain envelope
-> master GainNode
```

단독 gliss fallback의 frequency는 `startSound`에서 시작해 `endSound`까지 `linearRampToValueAtTime()`으로 이동한다.
gain은 `crossfadeSeconds` 동안 fade in하고, 종료 직전 같은 길이로 fade out한다.

S-M-E처럼 같은 `glissId`의 세그먼트가 같은 내부 anchor tick/pitch에서 이어지면, schedule builder는 개별 fallback을 만들지 않고 `AudioGlissChainScheduleEvent` 하나로 묶는다.
chain event 내부에는 각 analyzer `GlissEvent`에 대응하는 segment DTO를 보존한다.
audio backend는 chain 전체를 하나의 oscillator로 시작하고, segment마다 같은 oscillator의 `frequency`에 `linearRampToValueAtTime()`을 이어 붙인다.
따라서 analyzer/rendering의 세그먼트 구조는 유지하면서도 내부 anchor에서 oscillator stop/start가 발생하지 않는다.

이전 시행착오로는 연결된 segment를 개별 oscillator로 유지하고 내부 anchor 주변에 짧은 overlap/crossfade를 둔 방식이 있었다.
`0.02s` overlap은 청감상 변화가 작았고, `0.1s` 이상은 연결부 볼륨이 튀기 쉬웠다.
`0.05s`와 constant-sum에 가까운 envelope는 볼륨 튐은 줄였지만, gain 정규화와 맞물리면 연결부가 치직거리거나 끊기는 문제가 남았다.
결론적으로 gliss 연결부는 "두 oscillator를 자연스럽게 잇기"보다 "하나의 voice를 유지하고 pitch automation만 잇기"가 더 안정적이었다.

gliss 시작 anchor에 tremolo가 붙어 있으면 해당 division을 gliss fallback 전체 구간에 적용한다.
renderer가 trem+gliss를 outgoing segment의 점선으로 표시하는 정책과 맞추기 위해, anchor cell 내부에서 실제 trem effect가 겹치는 절반 구간만이 아니라 gliss segment 전체에 gain gate를 건다.
이렇게 해야 `@g(id,S) @t(n)` 조합이 "글리산도 구간이 tremolo로 끊기는 효과"로 들린다.

또한 gliss fallback이 시작되는 note event는 audio schedule 단계에서만 종료 tick을 gliss 시작 tick으로 앞당긴다.
특히 짧은 tuplet slot gliss는 renderer 표시를 위해 anchor를 slot 중심에 둘 수 있는데, note event를 원래 slot 끝까지 울리면 첫 slot 음과 fallback gliss가 과하게 겹쳐 롱노트처럼 들린다.
renderer의 anchor 위치와 analyzer event는 유지하되, audio note schedule만 clip해서 이 겹침을 줄인다.

이 방식은 전체 note voice-span 병합은 아니다.
기존 note event를 하나의 oscillator chain으로 합치거나 앞뒤 note gain을 ducking하지 않기 때문에, 긴 롱노트-글리산도-롱노트 전체를 하나의 음원으로 통합하지 않는다.
대신 연결된 `GlissEvent` segment 묶음만 chain으로 예약하므로 부분 갱신과 스케줄 캐시 경계가 비교적 단순하다.

초기 기본값은 다음이다.

```text
crossfadeSeconds/fadeSeconds: 0.02
curve: linear
```

후속 단계에서 실제 청감이 과하게 겹치면 note envelope 쪽에 ducking automation을 추가하거나, 특정 attach 조합에 한해 voice-span 병합을 도입할 수 있다.

## 7. 동시 발음 gain 정규화

트랙 레이어가 둘 이상 활성화되면 같은 시간대에 여러 `AudioScheduleEvent`가 동시에 울릴 수 있다.
이번 조정에서는 active track 수가 아니라 실제 schedule event의 초 단위 overlap 구간을 기준으로 gain을 정규화한다.

알고리즘은 다음 순서로 동작한다.

1. 전체 schedule event의 `startSeconds`, `endSeconds`를 각각 sweep point로 만든다.
2. sweep point를 시간순으로 정렬하고, 같은 시각의 start/end delta를 한 번에 반영한다.
3. 인접한 sweep point 사이에서 active event 수가 `n`이면 그 span의 `gainScale`을 `1 / n`으로 둔다.
4. 인접 span의 scale 값이 같으면 하나의 span으로 병합한다.
5. 각 event는 자기 시간 범위와 겹치는 `GainScaleSpan`만 `gainScale` automation으로 변환한다.

`gainScale`은 기존 dynamics와 분리된 automation kind이다.
backend에서는 `dynamicsGain -> scaleGain -> envelopeGain` 순서로 gain node를 연결한다.
이렇게 해야 dynamics row에 의한 표현 gain과 동시 발음 수 정규화 gain이 서로 덮어쓰지 않고 곱해진다.

이 방식은 compressor나 limiter가 아니라 schedule 생성 시점의 결정적 정규화이다.
따라서 같은 score와 같은 active track 상태에서는 항상 같은 gain automation이 만들어진다.
초기 구현처럼 각 event/span마다 전체 event 목록을 다시 세지 않기 때문에, 동시 발음 수 span 생성은 정렬 비용을 제외하면 sweep point 수에 선형으로 처리된다.

## 8. 중간 재개 event clipping

pause/resume 또는 seek 후 play가 긴 event 중간에서 시작되면, 기존 scheduler는 `startSeconds`가 이미 지난 event를 예약하지 않았다.
이를 보완하기 위해 scheduler가 lookahead 시작점에서 `getEventsOverlappingRange(t, t)`를 조회한다.
조회된 event가 아직 원본 event로 예약되지 않았다면 `:resume` event를 만들어 현재 score time부터 즉시 예약한다.

note event는 다음처럼 처리한다.

- `startSeconds`를 현재 재생 위치로 바꾼다.
- `endSeconds`, pitch, velocity는 원본 값을 유지한다.
- effect와 automation은 현재 위치 이후에 유효한 것만 남긴다.
- linear gain automation이 현재 위치를 가로지르면 시작값을 현재 위치의 보간값으로 바꾼다.

gliss fallback event는 다음처럼 처리한다.

- `startSeconds`를 현재 재생 위치로 바꾼다.
- 현재 위치가 원래 gliss ramp에서 차지하는 비율을 계산한다.
- `startMidi + startCentOffset / 100`과 `endMidi + endCentOffset / 100` 사이를 보간해 새 시작 pitch를 만든다.

gliss chain event는 다음처럼 처리한다.

- `startSeconds`를 현재 재생 위치로 바꾼다.
- 이미 끝난 segment는 제거한다.
- 현재 위치가 segment 중간이면 해당 segment의 시작 pitch를 현재 위치의 보간 pitch로 바꾼다.
- 뒤쪽 segment는 원래 pitch ramp 정보를 유지한다.

backend는 잘린 event를 일반 event처럼 받는다.
따라서 기존 note/gliss envelope의 짧은 fade-in이 중간 진입 시에도 적용되어, 재개 순간 볼륨이 튀는 문제를 줄인다.

## 9. 2026-06-23 gliss chain 전환 후 추가 진단

연결된 gliss segment를 overlap/crossfade 방식에서 `AudioGlissChainScheduleEvent` 방식으로 교체했다.
이 변경으로 내부 anchor에서 oscillator stop/start는 사라졌고, 하나의 oscillator에 segment별 `frequency.linearRampToValueAtTime()`을 이어 붙이게 되었다.
typecheck, audio test, track test, production build는 통과했다.

다만 실제 청감에서는 아직 연결부가 끊기듯 들리는 문제가 남았다.
처음에는 gliss chain 자체보다 chain에 함께 적용되는 tremolo gate를 가장 의심했다.

현재 gliss chain은 시작 anchor에 tremolo가 있으면 첫 gliss segment 전체에 tremolo effect를 붙인다.
`scheduleTremoloGate()`는 각 pulse 끝에서 gain을 `TREMOLO_GATE_MIN_GAIN`까지 내리고, effect가 끝난 뒤 `TREMOLO_GATE_RAMP_SECONDS`만큼 1로 복귀한다.
따라서 tremolo effect의 `endSeconds`가 gliss chain의 내부 anchor와 정확히 겹치면, oscillator는 계속 살아 있어도 연결 시점의 tremolo gain이 거의 0에 가까워진다.
이 경우 사용자는 pitch ramp가 이어지는 중에도 "연결부에서 소리가 끊긴다"처럼 들을 수 있다.
하지만 사용자가 tremolo를 적용하지 않은 상태에서도 gliss anchor 부분이 끊기듯 들린다고 확인했으므로, tremolo gate는 주원인에서 제외한다.

그 다음 후보는 gliss chain의 내부 anchor마다 같은 시각에 `frequency.setValueAtTime()`과 `linearRampToValueAtTime()`이 이어지는 구조였다.
연결된 segment는 앞 segment의 끝 pitch와 뒤 segment의 시작 pitch가 같다는 전제로 묶이므로, 내부 anchor에 새 set event를 넣을 필요가 없다.
따라서 backend의 `scheduleGlissChainFrequency()`를 수정해 chain 시작점에만 `setValueAtTime()`을 두고, 내부 segment는 이전 ramp endpoint에서 다음 `linearRampToValueAtTime()`으로 바로 이어지게 했다.

현재 의심 순위는 다음과 같다.

1. segment 경계에 `frequency.setValueAtTime()`과 `linearRampToValueAtTime()`을 같은 시각에 연달아 배치하면서 생기는 미세 automation 경계
2. dynamics/gainScale automation이 gliss chain 내부 경계 근처에서 값 경계를 만들 때 생기는 gain 변화
3. gliss 시작 anchor note clipping으로 앞 note와 chain 사이가 비는 경우
4. gliss chain만으로는 부족하고 앞 note, gliss, 뒤 note까지 하나의 full voice span으로 묶어야 하는 경우

1번은 코드상 즉시 제거 가능한 중복 schedule event였으므로 먼저 수정했다.
보고서에는 이 지점을 "연결된 oscillator를 만들었더라도 AudioParam automation event 경계가 청감상 단절처럼 들릴 수 있어 automation timeline을 더 단순화한 사례"로 남길 수 있다.

다음 진단 순서는 다음이 적절하다.

1. frequency boundary 단순화 후 같은 score를 다시 청감 확인한다.
2. 그래도 남으면 dynamics/gainScale automation을 임시 비활성화한 비교 fixture를 만든다.
3. 다음으로 gliss 시작 anchor note clipping을 끄거나 clip 위치를 조정한 비교 fixture를 만든다.
4. 그래도 남으면 full voice-span 병합이 필요한 사례로 분류하고, 앞 note + gliss chain + 뒤 note를 하나의 oscillator voice로 묶는 설계를 검토한다.
