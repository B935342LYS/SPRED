# 참고할 만한 오픈소스 탐색

## 1. 문서 목적

본 문서는 회귀 코드의 설계 및 구현 과정에서 비교 검토할 만한 오픈소스 사례를 탐색하고, 그 참고 가능성과 한계를 정리하기 위한 부록 초안이다.

이 문서의 목적은 특정 오픈소스를 그대로 이식하는 데 있지 않다. 오히려 다음과 같은 판단 근거를 확보하는 데 있다.

1. 회귀 코드와 구조적으로 유사한 선행 사례가 무엇인지 파악한다.
2. 각 사례에서 어떤 설계 선택을 참고할 수 있는지 정리한다.
3. 반대로 회귀 코드의 요구사항과 맞지 않아 채택하지 않을 부분을 분리한다.
4. 최종적으로 회귀 코드의 자료구조, parser-analyzer 분리, 이벤트 기반 처리, 대형 그리드 렌더링 설계가 자의적인 선택이 아니라 비교 검토를 거친 결과임을 보조적으로 설명한다.


## 2. 탐색 관점

회귀 코드는 단순한 악보 뷰어가 아니라, 다음 성격을 동시에 가지는 웹 기반 프로그램을 목표로 한다.

### 2.1. 음악 표현의 논리적 분해

1. 재현이 어려운 뉘앙스와 환경 의존 요소를 최대한 배제하고, 디지털적으로 다룰 수 있는 음악 정보를 구조화된 입력으로 환원한다.
2. 음악을 셀, 문자열, 전역 행, 이벤트, 시간 세그먼트 등으로 분해하고 다시 조립하는 절차를 갖는다.
3. 사용자는 전통적인 테이블 편집기와 유사한 방식으로 음악 정보를 편집하지만, 내부적으로는 parser와 analyzer를 거친 의미 구조가 형성된다.

### 2.2. 대형 테이블과 대규모 런타임 데이터

1. 120BPM, 4/4박자, 1박 = 4step, 10분 길이의 곡을 가정하면 약 4800열이 필요하다.
2. 음역대를 3옥타브 정도로 두면 전역 행, 노트 행, 갭 행을 합쳐 수십 행 규모가 된다.
3. 단일 트랙 기준으로도 수십만 셀 규모가 될 수 있으며, 복수 트랙을 고려하면 처리 대상은 더욱 증가한다.
4. 최초 로드 시에는 셀 기반 데이터를 파싱하고 분석해야 하지만, 이후 렌더링과 오디오 처리는 가능한 한 이벤트 리스트와 세그먼트 구조를 중심으로 다루는 편이 유리하다.

즉 회귀 코드는 “음악 DSL”, “대형 테이블”, “부분 갱신”, “이벤트 기반 재생”이 동시에 결합된 구조로 볼 수 있다.


## 3. 탐색 기준

오픈소스 탐색은 아래 기준을 중심으로 수행한다.

### 3.1. 입력 표현과 문법

- 음악을 텍스트, 패턴, 열 기반 구조로 기술하는가
- 사용자 입력과 내부 해석 구조가 분리되어 있는가
- 작은 DSL(특정 영역에 특화된 언어) 또는 규칙 기반 표기법을 사용하는가

### 3.2. 내부 자료구조와 변환 흐름

- 원본 입력을 중간 구조로 변환하는 parser 계층이 있는가
- 시간축 이벤트나 시퀀스 리스트를 별도로 생성하는가
- 렌더링과 오디오가 같은 분석 결과를 공유하는가

### 3.3. 대형 UI 및 렌더링 처리

- Canvas 또는 유사한 방식으로 대형 격자를 처리하는가
- 전체 재렌더 대신 부분 갱신 또는 가상화 전략이 보이는가
- 많은 셀 또는 이벤트를 다루기 위한 캐시 또는 인덱스 구조가 있는가

### 3.4. 회귀 코드와의 거리

- 스프레드시트/트래커/시퀀서 전통과 얼마나 가까운가
- 회귀 코드의 핵심 요구사항과 직접 맞닿는가
- 단지 장르가 비슷한 수준인지, 구조적 참고가 가능한 수준인지 구분할 수 있는가


## 4. 사례 정리 방식

각 오픈소스는 아래 항목으로 정리하는 것을 원칙으로 한다.

1. 개요
   - 무엇을 하는 프로젝트인지 간단히 설명한다.
2. 구조적으로 참고할 점
   - 회귀 코드에 참고 가능한 구조, 알고리즘, 인터페이스 철학을 적는다.
3. 회귀 코드와의 차이
   - 왜 그대로 가져올 수 없는지, 어떤 요구사항이 다른지 적는다.
4. 채택 가능성
   - 직접 이식, 부분 참고, 개념만 참고, 비채택 중 어느 쪽에 가까운지 정리한다.


## 5. 1차 검토 사례

## 5.1. Google Sheets MIDI Sequencer

- 링크: https://github.com/bepitulaz/google-sheets-midi-sequencer

### 개요

구글 시트를 MIDI 시퀀서처럼 사용하는 실험적 프로젝트이다. 앱스스크립트 기반 환경에서 입력된 연주 정보를 시트 형태의 테이블로 출력한다.

회귀 코드의 직접적 전신은 아니지만, “스프레드시트형 테이블을 음악 입력 인터페이스로 사용한다”는 점에서 비교 가치가 있다. 특히 회귀 코드가 스프레드시트 기반 프로토타입으로부터 출발했다는 점을 고려하면, 개념적 출발점의 유사성이 있다.

### 구조적으로 참고할 점

1. 스프레드시트형 인터페이스가 음악 입력 장치로 실제 사용 가능하다는 점을 보여준다.
2. 음악 데이터를 행/열 테이블로 대응시키는 방식이 가능함을 확인시켜준다.
3. “음정 정보”, “세기 정보” 등 음악 정보를 분리된 스키마로 다루는 발상은 회귀 코드의 저장 구조와 비교 대상으로 삼을 수 있다.

### 회귀 코드와의 차이

1. 회귀 코드는 단순 MIDI 시퀀싱보다 더 강한 문법 기반 문자열 입력을 전제한다.
2. 회귀 코드는 Google Sheets/App Script가 아니라 TypeScript + HTML + CSS + GitHub Pages 기반 웹 앱이다.
3. 회귀 코드는 parser, analyzer, renderer, audio generator를 분리한 런타임 파이프라인을 목표로 하므로, 단순 시트 출력 구조만으로는 충분하지 않다.

### 채택 가능성

- 직접 이식 가능성: 낮음
- 부분 참고 가능성: 중간
- 판단: 스프레드시트형 음악 입력 인터페이스의 가능성을 보여주는 사례로는 유의미하나, 회귀 코드의 실제 구현 구조를 직접 제공하는 사례로 보기는 어렵다.


## 6. 추가 검토 사례

## 6.1. textbeat

- 링크: https://github.com/flipcoder/textbeat

### 개요

textbeat는 plaintext 기반 음악 시퀀서이자 인터랙티브 셸이다. README에 따르면 음악 흐름은 세로 방향이며, 공백 또는 고정 폭으로 구분된 열들이 각각의 트랙을 이룬다.

회귀 코드와 가장 가까운 지점은 “문자열 기반 입력을 열/행 문맥 안에서 음악으로 해석한다”는 점이다. 전통적인 피아노롤과 달리, 입력 그 자체가 작은 문법 체계를 가지며, 같은 칸 안에 노트와 효과를 함께 기술한다는 점이 특히 유사하다.

### 구조적으로 참고할 점

1. 열 기반 입력을 곧바로 트랙 개념과 연결하는 방식은 회귀 코드의 `trackId + col + rowId` 축과 비교하기 좋다.
2. 한 셀 또는 한 줄의 입력이 단순 음표 하나가 아니라 “note + effect” 조합을 가질 수 있다는 점은 회귀 코드의 note 문자열 설계와 가깝다.
3. 박자와 subdivision을 문법 차원에서 함께 다루는 점은 회귀 코드의 전역 행과 비교 가능한 관찰 지점이다.

### 인용 메모

README에는 다음과 같은 설명이 있다.

> "vertical and column-based"

> "Each cell row in a track can contain both note data and associated effects."

첫 번째 문장은 입력 형식이 단순한 선형 문자열이 아니라, 세로 방향과 열 분리를 전제한 구조라는 점을 보여준다.  
두 번째 문장은 한 셀 단위에 note와 effect가 함께 존재할 수 있음을 드러낸다. 이는 회귀 코드에서 note 문자열이 display text, hold, gliss, trem, micro pitch 등을 함께 가질 수 있도록 설계한 방향과 구조적으로 비교할 가치가 있다.

또 README 예시에는 다음과 같은 전역성 명령이 등장한다.

> `%t120 x2`

이는 tempo와 grid subdivision을 문자열 명령으로 제어한다는 뜻이며, 회귀 코드에서 BPM / stepsPerBeat를 전역 행으로 분리한 설계와 비교 가능한 사례로 볼 수 있다. 다만 회귀 코드는 이를 하나의 텍스트 파일 헤더가 아니라 문서 좌표 위의 global row 셀로 재구성한다는 차이가 있다.

### 회귀 코드와의 차이

1. textbeat는 순수 텍스트 중심 도구이며, 회귀 코드처럼 대형 Canvas 테이블과 시각적 행 레이아웃을 전제로 하지 않는다.
2. textbeat는 트래커/텍스트 시퀀서 전통에 더 가깝고, 회귀 코드의 전역 행, gap 행, rowDefinition 구조와는 차이가 있다.
3. 회귀 코드는 parser와 analyzer를 문서 차원에서 명시적으로 분리하지만, textbeat는 문법형 입력 자체에 더 무게가 실려 있다.

### 채택 가능성

- 직접 이식 가능성: 낮음
- 부분 참고 가능성: 높음
- 판단: 문자열 DSL, 열 기반 트랙 모델, note와 effect의 결합 방식은 유의미한 참고처이다. 다만 UI와 런타임 파이프라인은 회귀 코드가 독자적으로 설계해야 한다.


## 6.2. txtseq

- 링크: https://github.com/samblenny/txtseq

### 개요

txtseq는 CircuitPython 환경에서 동작하는 작은 plaintext MIDI sequencer이다. 규모는 작지만, README에 parser 구조와 이벤트 정렬 방식이 비교적 직접적으로 설명되어 있어 구현 참고 가치가 높다.

회귀 코드와의 직접적 유사성은 “작은 문법을 파싱해서 시간순 이벤트 리스트로 바꾼다”는 점에 있다. 특히 제한된 환경에서 동작하도록 메모리와 할당 비용을 강하게 의식했다는 점은, 웹 환경에서 대형 문서를 다루려는 회귀 코드에도 시사점이 있다.

### 구조적으로 참고할 점

1. 파서가 원문을 읽고 최종 재생 이벤트 리스트를 만든 뒤, 정렬된 시간순 리스트를 재생에 사용한다는 점이 회귀 코드의 analyzer/event list 구조와 닿아 있다.
2. README에 따르면 각 voice의 이벤트를 먼저 만든 뒤 마지막에 병합 정렬하는 방식을 사용한다. 이는 회귀 코드에서 track 단위 분석 후 문서 단위 결과를 결합하는 사고와 비교할 수 있다.
3. 바이트 단위 상태 기계형 파서와 generator 기반 재생기는 “작고 예측 가능한 런타임 구조”라는 관점에서 참고 가치가 있다.

### 인용 메모

README에는 다음과 같은 설명이 있다.

> "packed as uint32 values"

> "Playback uses a generator"

> "state machine loops that examine one byte at a time"

첫 번째 문장은 parser 결과를 시간순 정렬 가능한 compact event representation으로 보관한다는 뜻이다.  
이는 회귀 코드가 analyzer 이후 `NoteEvent`, `GlissEvent`, `TupletGroupEvent` 등 이벤트 리스트를 공식 산출물로 두려는 방향과 맞닿아 있다.

두 번째 문장은 playback state를 generator 루프에 실어, 이벤트 반복과 외부 루프 결합을 단순하게 만든다는 뜻이다.  
회귀 코드가 같은 방식을 그대로 채택할 필요는 없지만, “재생기 상태를 거대한 class graph보다 작고 예측 가능한 실행 단위에 묶는다”는 발상은 참고 가치가 있다.

세 번째 문장은 parser가 토큰 배열을 한 번에 만드는 구조보다, 문자 단위 상태 기계에 가까운 방식으로 구현되었음을 보여준다.  
이는 회귀 코드가 `lexer -> parser` 분리보다 hand-written scanner/parser를 선택한 판단과 매우 가까운 방향이다.

### 회귀 코드와의 차이

1. txtseq는 마이크로컨트롤러 환경을 전제로 하므로, 브라우저 UI와 대형 그리드 렌더링 문제를 직접 다루지 않는다.
2. 회귀 코드는 시각화, 편집기, 전역 행, 부분 재분석 같은 상위 요구사항이 더 크다.
3. txtseq의 문법은 staff 중심 plaintext이고, 회귀 코드는 좌표 기반 셀 문자열 및 전역 문자열 구조를 가진다.

### 채택 가능성

- 직접 이식 가능성: 낮음
- 부분 참고 가능성: 높음
- 판단: parser의 상태 기계 스타일, voice별 이벤트 생성 후 병합 정렬, generator 기반 재생 루프는 회귀 코드의 core 계층에 개념적으로 참고할 만하다.


## 6.3. MusicToy

- 링크: https://github.com/maximecb/MusicToy

### 개요

MusicToy는 웹 브라우저에서 동작하는 minimalist grid sequencer이다. README에 따르면 HTML5 Web Audio와 Canvas API를 사용한다.

회귀 코드와의 가장 큰 접점은 “웹 기반 격자 UI”와 “브라우저 내 오디오 처리”이다. 입력 문법 자체는 다르지만, 브라우저 환경에서 grid, audio, state를 묶어 다룬다는 점에서 비교 가치가 높다.

### 구조적으로 참고할 점

1. 저장소 파일 목록에 `canvas.js`, `main.js`, `mixer.js`, `music.js`, `musictoy.js`, `piece.js`가 분리되어 있다. 이는 UI, 오디오, 곡 데이터, 앱 진입점을 나누는 고전적 구조를 보여준다.
2. 회귀 코드가 목표로 하는 `renderer / audio / document model` 분리와 직접 같지는 않지만, 웹 기반 음악 도구에서 모듈을 어떻게 나눌지 생각할 때 비교 기준이 된다.
3. 샘플 디렉터리와 mixer 관련 파일이 별도 존재한다는 점은, 재생기와 편집기 구조를 완전히 한 파일에 섞지 않는 편이 낫다는 사례로 볼 수 있다.

### 인용 메모

README에는 다음과 같은 설명이 있다.

> "uses the HTML5 web audio and canvas APIs"

또 저장소 파일 목록에는 `canvas.js`, `main.js`, `mixer.js`, `music.js`, `musictoy.js`, `piece.js` 등이 분리되어 있다.

첫 문장은 기술 스택의 직접적 유사성을 보여준다. 즉 MusicToy는 브라우저 내부에서 Canvas와 Web Audio를 함께 다루는 사례이며, 이 점만으로도 회귀 코드의 presentation/core 경계 설계에 비교 가치가 있다.

두 번째 사실은 구현이 단일 스크립트에 몰려 있지 않고, 적어도 canvas, mixer, piece 같은 책임 단위로 나뉘어 있음을 보여준다.  
이는 회귀 코드에서도 `parse`, `analyze`, `renderer`, `audio`, `document model`을 모듈 경계로 분리하는 편이 장기 유지보수 비용을 줄일 가능성이 높다는 점을 뒷받침한다.

### 회귀 코드와의 차이

1. MusicToy는 grid sequencer이지만, 회귀 코드처럼 셀 문자열 DSL을 parser/analyzer로 분리하는 구조는 README 기준으로 드러나지 않는다.
2. 회귀 코드는 전역 행, note row, gap row, 트랙 레이어 등 더 복잡한 문서 모델을 전제로 한다.
3. MusicToy는 웹 기반 렌더링/오디오 참고처로는 좋지만, 데이터 구조와 문법 설계의 직접적 선행 사례로 보기는 어렵다.

### 채택 가능성

- 직접 이식 가능성: 낮음
- 부분 참고 가능성: 중간
- 판단: 웹 기반 grid 렌더링, 모듈 분리, audio/UI 공존 구조를 살피는 데 의미가 있다. 다만 회귀 코드의 parser-analyzer 코어 설계에는 간접 참고 수준으로 보는 편이 적절하다.


## 7. 후속 검토 후보

아래 프로젝트들은 아직 상세 분석을 완료하지 않았으나, 회귀 코드와의 구조적 유사성 또는 비교 가치가 있어 후속 검토 후보로 보관한다.

## 7.1. 1bitr

- 링크: https://github.com/zserge/1bitr

예상 참고 지점:

- 매우 작은 규칙 집합으로 음악을 조직하는 철학
- 텍스트 기반 tracker류 인터페이스의 단순화 방식

예상 한계:

- 음원과 목적이 매우 제한적일 수 있다.
- 회귀 코드의 다층적 데이터 구조나 악보 편집 요구와는 거리감이 있을 수 있다.


## 7.2. Efflux Tracker 계열 웹 트래커 사례

- 링크: https://github.com/topics/music-tracker

예상 참고 지점:

- 웹 환경에서 tracker식 편집과 재생을 어떻게 구성하는지
- 패턴/열/행 기반 인터페이스를 브라우저에서 어떻게 구현하는지
- 대형 편집 UI와 재생 상태 동기화 방식

예상 한계:

- topic 페이지는 집합 링크이므로, 실제 참고를 위해서는 개별 프로젝트를 다시 선별해야 한다.
- tracker 전통은 회귀 코드와 가깝지만, 회귀 코드의 문자열 DSL과 전역 행 구조를 직접 제공하지는 않는다.


## 7.3. Noteahead

- 링크: https://github.com/juzzlin/Noteahead

예상 참고 지점:

- MIDI tracker/시퀀서 계열의 이벤트 중심 구조
- 패턴 기반 입력을 음정/이벤트 처리 구조로 바꾸는 방식

예상 한계:

- 데스크톱 중심 구현일 가능성이 높아, 브라우저 기반 렌더링/오디오 구조는 직접 참고하기 어려울 수 있다.


## 7.4. ZyTrax

- 링크: https://github.com/reduz/zytrax

예상 참고 지점:

- 전통적인 tracker UI를 현대적으로 재구성하는 방식
- 패턴 단위 편집과 이벤트 단위 재생 구조의 관계

예상 한계:

- 회귀 코드의 스프레드시트형 좌표 구조와는 입력 철학이 다를 수 있다.


## 8. 현 단계 판단

현재까지의 판단은 다음과 같다.

1. 회귀 코드와 가장 직접적으로 맞닿는 비교 축은 “스프레드시트형 음악 입력”과 “텍스트 기반 트래커/시퀀서”이다.
2. 그러나 회귀 코드의 핵심 난제는 단순 UI 모사보다, 문자열 기반 셀 문법을 parser와 analyzer를 통해 일관된 이벤트 구조로 변환하는 데 있다.
3. 따라서 오픈소스 탐색은 구현의 직접 이식보다, 설계 선택의 비교 근거를 확보하는 방향으로 활용하는 것이 적절하다.
4. 웹 기반 grid sequencer나 tracker 사례는 렌더링, 입력 처리, 재생 동기화 측면에서 부분 참고 가치가 있다.
5. 반면 회귀 코드의 parser-analyzer 분리, 전역 행 처리, 부분 재분석 구조는 자체 설계 비중이 크므로, 최종 구조는 독자적으로 확정해야 할 가능성이 높다.
