# 1.14. 2단계 analyzer MVP NoteEvent 구현 메모
> 본 설계 메모 문서 1.14에서는 2단계 구현 중 `defaultText`와 `"-"` hold를 `NoteEvent`로 변환하는 analyzer MVP 구현 판단을 기록한다.
> 본 문서는 active specification을 대체하지 않는다.
> 함수별 세부 설명은 코드 파일의 `/** ... */` 주석을 중심으로 하고, 본 문서는 흐름과 사용 문법, 설계 판단을 보충한다.

## 1. 개요

이번 구현은 parser가 만든 `ParsedScoreDocument`를 analyzer가 소비하여 첫 `NoteEvent`를 생성하는 경로를 만든 것이다.

현재 analyzer MVP 흐름은 아래와 같다.

```text
RuntimeDocument
-> buildParsedDocument()
-> analyzeDocument()
-> analyzeTrackEvents()
-> NoteEvent[]
```

이번 범위는 `1.9-mvp-analyzer-renderer-ui-spec.md`에 맞춰 아래 두 note 셀만 처리한다.

- modifier 없는 일반 note 셀
- `"-"` hold 셀

다음 기능은 parser가 이미 구조를 만들 수 있더라도 analyzer MVP에서는 아직 소비하지 않는다.

- `"~"` vibrato hold
- gliss
- trem
- `@p`, `@m`
- mute
- tuplet
- partial analysis

## 2. 관련 파일

이번 구현에서 추가된 주요 파일은 다음과 같다.

- `regression-code/src/core/analyze/analyze_full.ts`
  - analyzer 전체 진입점이다.
  - timing, dynamics, track 분석 결과를 `AnalysisResult`로 조립한다.
- `regression-code/src/core/analyze/analyze_track.ts`
  - track 단위 note event를 생성한다.
  - 현재는 default note와 `"-"` hold 병합만 처리한다.
- `regression-code/src/core/analyze/analyze_timing.ts`
  - col 0의 bpm, beatsPerBar, stepsPerBeat 값을 읽어 단일 timing segment를 만든다.
- `regression-code/src/core/analyze/analyze_dynamics.ts`
  - col 0의 dynamics 값을 읽어 단일 dynamics segment를 만든다.
- `regression-code/dev/test_analyze.ts`
  - fixture 기반 analyzer 결과를 확인한다.

## 3. analyzer 입력 경계

analyzer는 raw JSON이나 rawText를 다시 해석하지 않는다.
입력은 `AnalyzeContext` 하나로 묶는다.

```ts
type AnalyzeContext = {
  score: ScoreFile;
  indexes: ScoreIndexes;
  parsed: ParsedScoreDocument;
};
```

이 구조의 의미는 다음과 같다.

- `score`
  - 문서 길이, track 목록 같은 원본 구조를 참조한다.
- `indexes`
  - rowId에서 note row의 midi, display row를 찾는다.
- `parsed`
  - parser가 확정한 셀 문법 결과를 읽는다.

즉 analyzer는 셀 문자열이 아니라, parser 결과와 문서 인덱스의 관계를 해석한다.

## 4. analyze_full.ts의 책임

`analyzeDocument()`는 전체 analyzer 오케스트레이션 함수이다.

현재는 아래 네 가지를 조립한다.

```text
timingTimeline
dynamicsTimeline
trackResults
analysisIssues
```

`trackResults`는 `score.tracks` 순서대로 생성한다.
따라서 validator가 보장하는 `basic`, `optional`, `extra` track이 모두 결과에 포함된다.
비어 있는 optional/extra track은 빈 event 목록을 가진다.

## 5. analyze_track.ts의 책임

`analyzeTrackEvents()`는 한 track의 parsed note cell들을 시간순으로 읽고 `NoteEvent`를 만든다.

처리 순서는 다음과 같다.

```text
1. trackId에 해당하는 parsed cell map을 가져온다.
2. col, rowId 순서로 entry를 정렬한다.
3. ParsedNoteCell이 아니면 제외한다.
4. MVP 지원 범위 밖이면 제외한다.
5. rowId를 NoteRowDefinition으로 확인한다.
6. "-" hold이면 바로 이전 연결 가능 NoteEvent를 찾는다.
7. 연결 가능하면 기존 NoteEvent의 endTick과 sourceCells를 확장한다.
8. 연결 불가하면 새 NoteEvent를 만든다.
```

## 6. MVP note 셀 판정

현재 `isMvpNoteCell()`은 아래 조건을 만족하는 셀만 true로 본다.

- `parsedCell.kind === "note"`
- `hold`가 `null` 또는 `"-"`
- 모든 modifier가 `null`

즉 `~`, gliss, trem, absolutePitch, microPitch가 포함된 셀은 이번 analyzer 단계에서 아직 event를 만들지 않는다.
이는 기능 누락이 아니라 `1.9`에서 정한 MVP 제한이다.

## 7. "-" hold 병합 규칙

`"-"` 셀은 먼저 기존 note에 연결 가능한지 확인한다.

현재 연결 조건은 다음이다.

- 같은 track 안에서 이미 생성된 note event여야 한다.
- 기존 event의 `endTick`이 현재 셀의 `col`과 같아야 한다.
- 기존 event의 display rowId가 현재 note rowId와 같아야 한다.
- 기존 event의 sound midi가 현재 note row의 midi와 같아야 한다.
- centOffset은 둘 다 0이어야 한다.

연결되면 기존 event를 아래처럼 갱신한다.

- `time.endTick = currentCol + 1`
- `sourceCells`에 현재 셀 좌표 추가
- effect segment를 확장된 전체 시간 범위로 다시 만든다.

연결되지 않은 `"-"`는 `1.9`의 MVP 규칙에 따라 독립 note 시작점으로 처리한다.
이 경우에도 note row의 midi와 rowId를 기준으로 `NoteEvent`를 만든다.

## 8. NoteEvent 생성 규칙

modifier 없는 일반 note는 아래 값으로 `NoteEvent`가 된다.

- `eventKind: "note"`
- `time.startTick = col`
- `time.endTick = col + 1`
- `display.rowId = 셀이 놓인 note rowId`
- `display.centOffset = 0`
- `sound.midi = note row의 midi`
- `sound.centOffset = 0`
- `effects = vib false, trem null`
- `sourceCells = 이벤트를 만든 셀 좌표 목록`

`eventId`는 MVP에서 아래 형식으로 만든다.

```text
{trackId}:note:{rowId}:{col}
```

이 id는 첫 source cell 기준의 안정적인 디버깅용 id이다.
후속 partial analysis에서 더 엄격한 id 정책이 필요하면 별도 규칙을 둘 수 있다.

## 9. timing / dynamics MVP

`1.9`는 audio/playback MVP를 위해 빈 timeline이 아니라 최소 기본 세그먼트를 요구한다.
따라서 이번 구현은 col 0의 global cell 값을 읽어 문서 전체에 적용되는 단일 segment를 만든다.

timing segment는 아래 값을 사용한다.

- `bpm`
- `beatsPerBar`
- `stepsPerBeat`

dynamics segment는 아래 값을 사용한다.

- `dynamics`

현재 fixture에서는 각각 다음 값이다.

```text
bpm = 120
beatsPerBar = 4
stepsPerBeat = 4
dynamics = 100
```

세그먼트 시간 범위는 `0..score.globalLines.columnCount`이다.

## 10. 테스트 확인

`dev/test_analyze.ts`는 현재 fixture를 기준으로 아래를 확인한다.

- `trackResults`가 3개 생성된다.
- optional/extra track은 빈 event 목록을 가진다.
- basic track은 4개의 `NoteEvent`를 만든다.
- `E3`, `-`, `-`, `-`는 하나의 `0..4` note로 병합된다.
- `F3`, `-`는 `4..6` note로 병합된다.
- 다음 `F3`, `-`는 별도 `6..8` note로 병합된다.
- `F#3`는 `8..9` note가 된다.
- timing segment는 BPM 120, 4/4, stepsPerBeat 4를 가진다.
- dynamics segment는 100을 가진다.

확인 명령은 다음이다.

```text
npm run test:analyze
```

이번 구현 후 함께 확인한 명령은 다음이다.

```text
npm run typecheck
npm run test:score
npm run test:parse
npm run test:analyze
npm run build
```
