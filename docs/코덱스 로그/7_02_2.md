현재 이해한 진행 상황은 다음과 같습니다.

`회귀 코드`는 Google Sheets + Apps Script 기반의 `이세계 코드`를 대체하는 TypeScript / HTML / CSS 웹 앱이며, 실제 안정 구현 루트는 `regression-code/`입니다. `이세계 코드`와 `1. 개발문서`는 필요할 때만 참고하는 legacy/archive 자료이고, 새 작업은 기본적으로 `regression-code/`와 `docs/` 기준으로 진행해야 합니다.

현재 폴더 역할은 이렇게 분리되어 있습니다.

- `regression-code/`: 안정판 SPRED / 회귀 코드 본 구현
- `regression-code-extend/`: 3차 확장 실험용, 특히 MIDI import 같은 고위험 확장 작업
- `regression-code-test-publish/`: GitHub Pages 테스트 배포 staging
- `regression-code-2026-06-19/`: advisor meeting 설명용 스냅샷
- `docs/`: 활성 사양과 구현 메모
- `0. 이세계 코드 (legacy)/`, `1. 개발문서/`: legacy/reference only

기능 진행 상태는 `HARNESS.md` 기준으로, 기본 악보 파서/분석기/렌더러/오디오/편집 기능은 이미 상당히 진행되어 있고, 이후 트랙 레이어, YouTube sync, 부분 rebuild, range selection, undo/redo, Examples, Practice Mode까지 확장된 상태입니다.

노래방/Practice Mode 2차 구현은 완료되어 장기 테스트 단계에 들어간 것으로 이해했습니다. 현재 구현에는 마이크 pitch detection, pitch/timing 판정, combo, result dialog, gliss/vib 효과 보너스, FULL COMBO 표시, practice 중 편집/파일 교체 잠금 등이 포함되어 있습니다. trem 판정, Pro Mode, timing anti-cheat 세부 튜닝 등은 후속 후보로 남아 있습니다.

3차 확장인 MIDI 변환/import 실험은 안정판에서 분리되어 `regression-code-extend/`로 이동했습니다. 따라서 MIDI 관련 작업은 기본 SPRED 안정판에 직접 섞지 않고 별도 실험 라인에서 다뤄야 합니다.

악보 DB 기능은 2차 구현 안정판 쪽, 즉 `regression-code/`에 Examples 기능으로 들어간 상태입니다. 구조는 Supabase Edge Function 기반으로, `File > Examples`에서 access word를 입력하고 manifest를 받아온 뒤, 선택한 공개 Storage Score JSON을 기존 Score JSON load/validation pipeline으로 읽는 방식입니다. 로컬 fallback 예제 JSON을 번들에 넣는 구조는 아니며, 실패 시 현재 악보를 유지하는 정책입니다.

매뉴얼은 이제 작성 단계입니다. 관련 기준 문서는 `docs/2.11-example-score-and-manual-support-spec.md`와 `docs/2.12-manual-content-spec.md`이고, 현재 앱에는 1차 정적 매뉴얼 파일 `regression-code/public/manual/index.html`이 이미 존재합니다. 다만 지금 파일은 짧은 초안 수준이고, `2.12` 기준으로는 한국어 오타마톤 매뉴얼을 먼저 확장해야 합니다. 주요 작성 범위는 시작하기, 화면 구성, Examples/악보 DB, 재생, 저장/불러오기, 편집, Track, Layout, YouTube sync, Practice Mode, 문제 해결, 테스트 피드백, 용어집입니다.

다음 작업의 중심은 코드 기능 추가보다는 사용자 테스트용 Manual 보강입니다. 특히 실제 UI 메뉴명과 버튼명을 기준으로, 처음 쓰는 사용자가 예제 악보를 불러오고 재생하고 Practice Mode까지 시도할 수 있게 안내하는 문서가 우선입니다.

구현은 충분히 가능합니다. 다음 단계로는 “비공개”를 임시 테스터 제한으로 볼지, 실제 접근 제어로 볼지만 결정하면 됩니다. 저는 현재 단계에서는 `access_group = public/tester` 기반의 약한 비공개부터 권장합니다.

> 네. 권장하신 안이 적절해 보입니다.

좋습니다. 그러면 다음 구현 방향은 이렇게 잡으면 됩니다.

- `accessWord` 없이 Examples를 열면 공개 악보만 조회
- 올바른 `accessWord`를 입력하면 공개 악보 + 테스터용 비공개 악보 조회
- 잘못된 `accessWord`를 입력했을 때는 정책상 둘 중 하나를 선택
  - 권장: 오류를 표시하고 목록은 갱신하지 않음
  - 대안: 공개 악보만 보여주고 “비공개 해제 실패” 안내

DB 쪽은 기존 `access_group`을 그대로 활용하면 됩니다.

```text
public  # 비밀번호 없이 노출
tester  # access word 필요
```

구현 범위는 다음 정도입니다.

- Edge Function query 조건 변경
- 프론트의 “빈 access word 차단” 제거
- dialog 기본 문구 변경
- 필요하면 manifest item에 `accessGroup` 또는 `visibility` 표시 추가
- 문서 `2.11`에 공개/비공개 조회 정책 갱신

제가 보기에는 `visibility` 표시까지 넣는 편이 좋습니다. 목록에서 사용자가 “이 악보가 공개 예제인지, 테스터용 예제인지” 바로 알 수 있어야 DB 보강과 테스트 피드백을 관리하기 쉽습니다.

> 이미 비밀번호 입력만으로도 그건 구분이 되지 않을까요?

사용자 입장에서는 어느 정도 구분됩니다. 비밀번호 없이 보이는 악보는 공개, 비밀번호를 넣고 추가로 보이는 악보는 비공개라고 이해할 수 있으니까요.

다만 운영/테스트 관점에서는 표시가 있으면 더 명확합니다.

예를 들어 access word를 입력한 뒤에는 목록에 **공개 악보와 비공개 악보가 함께 섞여** 보입니다. 이때 badge가 없으면 사용자는 어떤 항목이 원래 공개였고 어떤 항목이 비공개로 추가된 것인지 바로 알기 어렵습니다. 특히 피드백을 받을 때 “이 악보는 공개 목록에서도 보이나요?” 같은 확인이 생길 수 있습니다.

그래도 첫 구현에서는 badge 없이 가도 됩니다. 구현을 더 단순하게 하려면:

- access word 없음: 공개 악보만 표시
- access word 있음: 공개 + 비공개 악보 표시
- 별도 `Public / Private` 표시는 없음

이 방식이 충분히 직관적이고, UI도 덜 복잡합니다. 나중에 목록이 많아지거나 테스트 피드백에서 혼동이 생기면 badge를 추가하면 됩니다. 지금은 badge 없이 시작해도 무방합니다.
