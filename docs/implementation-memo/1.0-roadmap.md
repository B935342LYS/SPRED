# 1.0. 첫 단계 설계 로드맵
> 상위 경로에 존재하는 md파일 전반은 specification이다.
> 본 경로에 존재하는 md파일들은 세부 구현 일지 및 해설 문서로 그 성격이 명확히 구분된다.
> specification의 넘버링 1.x에서 각 모듈의 기능 정의 / 필요한 클래스 정의 / 모듈 사이의 인터페이스 정의를 완료하였다.
> 이제 실제 ts 파일 구현에 들어가 클래스를 선언하고, 각 모듈의 input에서 output을 도출하는 알고리즘의 구현에 들어간다.
> 본 설계 메모 문서 세트에서는 코드의 구상부터 실제 제작 과정, 이후 제작된 코드의 해설까지 구현 전반에 대해 세부 설명으로 보충하는 내용을 담는다. 

## 1. 전체 목표 정리

- 1단계 목표  (~5월 초)
    - 목표 : 기반 다지기
    - 범위 : 모든 토큰
    - 데이터 구조와 런타임 인덱스 확정
    - 모든 토큰의 파싱 알고리즘을 구현한다.
    - ParsedDocument 구조까지 확정시킨다.
    - 부분 갱신은 구현하지 않는다.

- 2단계 목표 (~5월 말)
    - 목표 : JSON파일 테스트 케이스 제작 자동화
    - 범위 : 노트 셀의 defaultText와 hold 토큰 "-" 한정
    - 이후 analyze 단계부터 국소적 MVP 기능 구현을 적용한다.
    - 1차 분석 범위에 관여하는 MVP render, audio 생성 구조 완성
    - 기본 UI, 화면 표시, 기본 편집, 기본 재생과 이에 필요한 기본 초기화 에셋을 구현한다.
    - 부분 갱신은 구현하지 않는다.

- 3단계 목표 (~6월)
    - 목표 : 살 붙이기
    - 범위 : 나머지 note 문자열 토큰들
    - gliss, trem으로 analyzer 소비 확장
    - tuplet analyzer 소비 확장
    - 모든 토큰의 render, audio 확장

## 3.1. HTML, CSS, JavaScript(TypeScript) 파일 간 연결

CSS의 연결:
HTML 또는 CSS에 CSS를 적용하는 데에는 3가지 방법이 있다. 실무적으로는 3번을 주로 사용한다.
1. HTML에서 필요한 태그 내부에 직접 CSS 속성 입력
2. HTML의 <head></head> 태그 내부에 <style></style> 만들고 안에 CSS 작성
3. 별도의 css 확장자 파일을 만들고 HTML의 <head></head> 태그 내부에 <link href = "./경로/style.css" rel="stylesheet">로 불러오기

HTML + JS:
작성한 JS파일을 HTML 내부로 불러오고자 한다면 2가지 방법을 이용할 수 있다. 실무적으로는 2번을 주로 사용한다.
1. HTML의 <body></body> 태그 내부에 <script></script> 태그 만들고 안에 코드 작성
2. 별도의 js파일dmf 만들고 HTML의 <body></body> 태그 내부에 <script src="./경로/script.js" defer></script>

현재 구현에 사용하고 있는 ts 확장자 파일은 바로 HTML에 연결할 수 없으며 한번 JS로의 변환을 거쳐야 함에 유의한다. node.js를 이용하여 편리하게 변환이 가능하다고 한다.

참고: https://tango1202.github.io/typescript/typescript-config/


## 3.2. JS / TS 파일간 연결

JS(또는 TS) 파일 간의 연결은 import / export 문으로 수행한다. 함수, 객체, 기본 타입의 값을 다른 파일로 넘겨줄 수 있다.

주는 모듈은 다른 모듈에 넘겨주고자 하는 것을 선언할 때 앞에 export를 부착한다.

받는 모듈은 `import (값 | 값 리스트 | *(전체)) from "경로/파일.ts"`의 형식으로 import문을 사용하여 값을 가져온다.


## 3.3. 일반적인 스크립트 파일 구조와 관습

참고: https://gamza1013.tistory.com/114

상단 링크에서 대중적인 파일 구조를 포함해 여러 유형을 확인할 수 있었다. 용도가 일치하는 폴더는 전통적인 이름을 따오기로 한다. 
한편 전체적인 파일 구조는 협업을 위한 일관된 관습을 찾지 못하였다. 구분만 명확하다면 임의로 나누어도 문제가 없는 것으로 보인다. 내 관점에서 이해가 쉬운 방식으로 구성하도록 한다.


## 3.4. 배럴 파일 (index.ts)

상위 폴더에 index.ts와 같은 이름의 파일을 만들고 내부에 export * "하위 모듈" 문을 여럿 집어넣어 하나의 정리된 목차처럼 사용할 수 있다. 이것을 배럴 파일이라고 부른다.

장점:
개발 편의성의 상승 : 상위 파일에서 각 모듈별로 하나하나 import를 작성하는 수고를 줄이고 가독성을 향상시킬 수 있다.

단점:
대형 프로젝트에서는 하위 폴더에도 하나하나 index.ts 파일을 만들면 연쇄적으로 배럴 파일이 import되어 overhead가 발생하는 문제가 있다고 한다. 본 프로그램은 학부생 수준의 1인 프로젝트니 걱정하지 않아도 될 것 같지만 그래도 한번 검토를 실시한다.

주요 단점은 다음과 같다.
- 성능 오버헤드
    - export로 하위 배럴 파일을 import한다. 배럴 파일은 많은 export 문을 포함하므로 그 중 또 다른 하위 배럴 파일을 포함할 수도 있다. 연쇄적으로 수많은 파일 묶음이 불러와지면서 성능 오버헤드가 발생할 수 있다.

- 모듈간 경계 흐려짐
    - 연쇄적으로 수많은 파일을 불러오게 되면 모든 흐름을 파악하고 통제하기 어렵다. 하나의 모듈에서 필요한 파일만 부르려고 했는데 다른 모듈의 파일이 의도치 않게 불러와질 가능성이 높아지며 따라서 모듈 경계가 흐려져 버린다.

AI 도구의 평가는 다음과 같다.
- 본 프로젝트의 규모는 성능 오버헤드를 걱정할 정도는 아니다.
- 하지만 모듈간 경계 침범은 하나의 index.ts가 수십개의 파일을 포함하게 되거나 계층이 2단계 이상만 늘어나도 신경써야 하는 문제가 된다.

대규모의 export문을 포함한 배럴 파일을 생성하게 될지는 아직 모르겠다. 다만 폴더가 2계층 이상으로 늘어나는 일은 여럿 생길 수 있다. core와 같은 주요 단위의 상위 폴더에서만 index.ts를 가지고 나머지 내부 파일들은 import문만으로 우선 처리하는 방향으로 가기로 한다. 불편성이 커지는 경우에만 배럴 파일의 사용을 결정한다.

## 5. 1단계 구현 순서

1. 전 타입 파일의 구현부터 우선 진행한다.
    - 이후 구현될 모든 모듈은 타입 인터페이스를 공유해야 동작할 수 있다. 따라서 1순위. 이미 명세서에 구체적 설계 있으므로 바로 진입 가능하다.
    - score/types.ts : 초안 작성 완료
    - parse/types.ts : 초안 작성 완료
    - analyze/types.ts : 초안 작성 완료
    - common.types.ts : 위의 세 파일에서 공통적으로 쓰이는 타입을 추려서 옮긴다. 현재는 보류 중이다.

2. score 폴더 내부 모듈 중 다음 4개의 구현을 순서대로 진행한다.
    - json_load.ts
    - score_validate.ts
    - build_score_indexes.ts
    - create_runtime_document.ts
    - parser와 analyzer의 연산은 원본 JSON을 쓰지 않고 score 모듈이 정리한 데이터 구조와 런타임 인덱스 기반이다. 따라서 파서보다 먼저 구현되어야 한다.
    - 초기에는 score_validate.ts를 뒤로 미루려 했으나, 후속 인덱스 생성 전에 ScoreFile의 최소 구조와 참조 무결성을 보장하는 편이 안정적이라고 판단하여 1단계 최소 구현 범위에 포함하였다.

3. parse 폴더 내부 모듈 중 다음 3개의 구현을 순서대로 진행한다.
    - parse_note_cell.ts
    - parse_global_cell.ts
    - build_parsed_document.ts
    - 부분 갱신은 복잡하므로 현 단계에서는 build_parsed_document_partial.ts을 생략한다.

4. 테스트 코드를 제작한다.
    - JSON과 rawText의 테스트케이스를 제작한다.
    - 초기 테스트 환경은 Jest 도입 전 단계로 `typescript`, `tsx`, `@types/node` 기반의 타입 검사와 fixture 실행 스크립트로 구성한다.
    - 테스트 코드를 ./dev 경로에 작성한다.
    - 충분히 안정된 것 같다면 2단계로 넘어간다.
