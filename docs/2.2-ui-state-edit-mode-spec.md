# UI state / edit mode 상태 구조 명세

## 1. 문서 목적

본 문서는 `회귀 코드`의 UI 상태 구조와 edit mode 입력 흐름을 정의한다.

`2.0-ui-mvp-spec.md`가 화면 구조와 event-action table을 정의하고, `2.1-canvas-renderer-module-spec.md`가 canvas renderer 모듈 경계를 정의했다면, 본 문서는 다음 구현 단계에서 필요한 상태 모델을 고정한다.

특히 같은 사용자 동작이 현재 상태에 따라 서로 다른 의미를 가져야 하는 문제를 다룬다.

```text
score canvas click
  -> view mode에서는 위치 선택 또는 seek
  -> custom text edit mode에서는 cell rawText 입력
  -> eraser mode에서는 cell 삭제
  -> tuplet slot select mode에서는 slot text 저장
  -> layout edit mode에서는 layout row 선택
```

따라서 edit mode 구현은 단순 DOM 이벤트 핸들러 묶음이 아니라, 작고 명시적인 상태 구조를 기준으로 처리한다.

## 2. 적용 패턴

본 프로젝트에서는 GoF의 클래스 기반 State 패턴을 그대로 사용하지 않는다.

대신 TypeScript의 discriminated union과 action handler 함수를 사용한다.

```text
DOM event
  -> event decode / hit test
  -> AppAction
  -> current state + action 해석
  -> score mutation 또는 UI state update
  -> parse / analyze / render
```

이 방식을 선택하는 이유는 다음과 같다.

- 앱 규모상 class 계층을 만들기에는 아직 무겁다.
- `mode.kind`, `tool.kind` 분기가 TypeScript에서 타입 좁히기와 잘 맞는다.
- 같은 click이라도 mode/tool별 동작이 다르다는 점을 코드 구조에 직접 드러낼 수 있다.
- 후속 `tupletSlotSelect`, `layoutEdit`, `eraser`, `drag edit` 확장 시 분기 위치가 명확하다.

## 3. 상태 구조

### 3.1. AppState

1차 구현의 앱 상태는 아래 구조를 기준으로 한다.

```ts
type AppState = {
  document: RuntimeDocument;
  parsed: ParsedScoreDocument;
  analysis: AnalysisResult;
  renderInput: CanvasAnalyzedRenderInput;
  activeTrackId: TrackId;
  mode: AppMode;
  busy: AppBusyState;
  statusMessage: UiStatusMessage;
  selection: ScoreSelection | null;
};
```

`document.score`는 현재 런타임 score JSON 상태의 진실 원천이다.
`parsed`, `analysis`, `renderInput`은 `document.score`에서 재생성 가능한 런타임 산출물이다.

초기 edit 구현에서는 mutation 이후 full rebuild를 허용한다.

```text
RuntimeDocument score mutation
  -> buildParsedDocument()
  -> analyzeDocument()
  -> buildCanvasNoteRenderItems()
  -> renderCanvasScore()
```

partial parse / partial analysis는 후속 단계로 둔다.

### 3.2. UiStatusMessage

UI에는 footer의 개발/파이프라인 상태 요약과 별도로, 왼쪽 메뉴 영역 아래에 작은 user-facing status line을 둔다.
이 영역에는 사용자가 방금 수행한 조작의 결과, busy 메시지, 오류 메시지를 표시한다.

```ts
type UiStatusMessage = {
  level: "info" | "warning" | "error";
  text: string;
};
```

상태 메시지 규칙은 다음이다.

- `busy.kind !== "idle"`이면 busy message가 status line에 우선 표시된다.
- score load, edit apply, rebuild 실패는 `level: "error"`로 표시한다.
- 비활성 기능 클릭, note row가 아닌 곳 클릭 등은 `level: "warning"`으로 표시할 수 있다.
- 정상 edit 적용, score load 성공 등은 `level: "info"`로 표시한다.
- 긴 메시지는 UI에서 한 줄 말줄임으로 처리하고 full text는 `title`에 둘 수 있다.
- 메시지는 score JSON에 저장하지 않는 UI runtime state이다.

### 3.3. AppBusyState

`AppMode`는 사용자가 어떤 의미로 화면을 조작하는지 나타낸다.
반면 `AppBusyState`는 앱이 현재 입력을 받을 수 있는지를 나타내는 전역 상태이다.

```ts
type AppBusyState =
  | { kind: "idle" }
  | { kind: "loadingScore"; message: string }
  | { kind: "applyingEdit"; message: string }
  | { kind: "rebuilding"; message: string };
```

busy 상태는 mode와 별도 축으로 둔다.
예를 들어 사용자가 edit mode에 있어도, score mutation 후 full rebuild 중에는 `mode.kind === "edit"`을 유지한 채 `busy.kind === "rebuilding"`으로 입력을 잠근다.
rebuild가 끝나면 다시 `busy.kind === "idle"`로 돌아가며 edit mode는 유지된다.

busy 상태에서 잠글 대상은 다음이다.

- score canvas pointer 입력
- layout label pointer 입력
- edit panel 입력과 버튼
- file load / reload 버튼
- track 선택 버튼
- zoom slider
- seek bar와 playback button
- JSON download button

busy 상태에서도 허용할 수 있는 대상은 다음이다.

- loading 상태 표시
- 오류 메시지 확인용 닫기 버튼

1차 구현에서는 단순히 주요 button/input/select를 `disabled` 처리하고, score canvas click handler 초입에서 busy 상태면 return한다.

### 3.4. AppMode

```ts
type AppMode =
  | { kind: "view" }
  | { kind: "edit"; tool: EditTool }
  | { kind: "layoutEdit" };
```

MVP 1차 구현에서는 `"view"`와 `"edit"`만 사용해도 된다.
`"layoutEdit"`은 후속 layout save/reset 구현과 충돌하지 않도록 상태 공간에 미리 둔다.

### 3.5. EditTool

```ts
type EditTool =
  | { kind: "customText"; text: string }
  | { kind: "eraser" }
  | { kind: "tupletSlotSelect"; slotIndex: number; draft: TupletDraft };
```

1차 구현에서는 `customText`만 실제 동작한다.

`eraser`는 Default 드롭다운의 `ERASER`와 연결할 수 있다.
`tupletSlotSelect`는 tuplet UI가 활성화된 뒤 같은 canvas click을 “셀 입력”이 아니라 “slot 문자열 채우기”로 해석하기 위해 둔다.

### 3.6. ScoreSelection

```ts
type ScoreSelection = {
  trackId: TrackId;
  rowId: RowId;
  col: number;
  rowKind: "global" | "note" | "gap";
};
```

`ScoreSelection`은 UI 선택 상태이다.
저장 JSON에 직접 들어가지 않는다.

1차 edit 기능은 `rowKind === "note"`인 hit에만 rawText mutation을 적용한다.
global row edit은 후속 단계로 둔다.

## 4. Action 구조

DOM 이벤트는 직접 score를 수정하지 않고 `AppAction`으로 변환한다.

```ts
type AppAction =
  | { type: "toggleEditMode"; enabled: boolean }
  | { type: "setDefaultMode"; mode: DefaultEditMode }
  | { type: "setCustomText"; text: string }
  | { type: "scorePointerClick"; hit: ScoreHit }
  | { type: "setActiveTrack"; trackId: TrackId }
  | { type: "setBusy"; busy: AppBusyState }
  | { type: "setStatusMessage"; message: UiStatusMessage }
  | { type: "clearStatusMessage" }
  | { type: "reloadScore" };
```

`DefaultEditMode`는 UI의 Default 드롭다운 값을 좁힌 상태이다.

```ts
type DefaultEditMode =
  | "autoSharp"
  | "autoFlat"
  | "custom"
  | "comment"
  | "eraser";
```

1차 구현에서는 `"custom"`만 실제 score mutation을 수행한다.
다른 값은 UI 표시 또는 disabled 상태로 둘 수 있다.

`reloadScore`와 파일 로드 action은 `busy = loadingScore`를 거쳐야 한다.
로드 중에는 기존 score에 대한 edit action, scroll/seek action, track 변경 action을 모두 무시한다.
로드 성공 시 새 `RuntimeDocument`를 기준으로 parsed/analysis/renderInput을 재생성하고, 실패 시 기존 정상 state를 유지한다.

## 5. Click 처리 흐름

### 5.1. Hit test

canvas click은 먼저 renderer 좌표계로 변환한다.

```ts
type ScoreHit = {
  rowId: RowId;
  rowKind: "global" | "note" | "gap";
  col: number;
};
```

변환 규칙은 다음이다.

- `x`는 score canvas content 좌표 기준으로 계산한다.
- `col = floor(x / layout.columnWidth)`로 둔다.
- `y`는 score canvas content 좌표 기준으로 계산한다.
- `layout.rows`를 기준으로 `row.y <= y < row.y + row.height`인 row를 찾는다.
- hit가 `columnCount` 밖이거나 row 밖이면 click은 무시한다.

layout label canvas click은 1차 edit mutation 대상으로 보지 않는다.

### 5.2. Mode dispatch

hit test 이후 동작은 현재 mode가 결정한다.
단, mode dispatch 전에 `busy.kind !== "idle"`이면 입력 action은 무시한다.

```ts
function handleScoreClick(state: AppState, hit: ScoreHit): AppState {
  if (state.busy.kind !== "idle") {
    return state;
  }

  switch (state.mode.kind) {
    case "view":
      return handleViewScoreClick(state, hit);
    case "edit":
      return handleEditScoreClick(state, hit);
    case "layoutEdit":
      return handleLayoutScoreClick(state, hit);
  }
}
```

`handleViewScoreClick`은 1차 구현에서 선택 상태만 갱신하거나 아무 동작도 하지 않을 수 있다.

`handleEditScoreClick`은 다시 `tool.kind`로 분기한다.

```ts
function handleEditScoreClick(state: AppState, hit: ScoreHit): AppState {
  switch (state.mode.tool.kind) {
    case "customText":
      return applyCustomTextToScore(state, hit, state.mode.tool.text);
    case "eraser":
      return eraseScoreCell(state, hit);
    case "tupletSlotSelect":
      return setTupletSlotTextFromHit(state, hit);
  }
}
```

## 6. currentText 조합 정책

legacy `이세계 코드`의 `buildCurrentNoteText_()` 개념은 유지한다.
다만 회귀 코드에서는 이를 `composeCurrentEditText()` 같은 순수 함수로 분리한다.

```text
Default token
  + Long token
  + Gliss token
  + Trem token
  + Pitch modifier
  + Tuplet wrapper
  -> current edit rawText
```

1차 구현에서는 다음으로 제한한다.

```text
Default CUSTOM text input
  -> current edit rawText
```

즉 처음에는 `currentText = customText`이다.
이후 Long, Gliss, Trem, Pitch, Tuplet UI가 활성화되면 같은 composer 함수에 토큰 조합 규칙을 추가한다.

legacy의 click cycle 규칙도 후속 단계에서 추가할 수 있다.

```text
empty -> currentText
normal note -> -
- -> ~
~ -> currentText
```

다만 1차 구현에서는 테스트 편의성을 우선하므로, 클릭 시 항상 현재 custom text로 덮어쓴다.

## 7. Score mutation 규칙

### 7.1. note cell upsert

`customText` edit의 1차 대상은 active track의 note cell이다.

```text
trackId = state.activeTrackId
rowId = hit.rowId
col = hit.col
rawText = state.mode.tool.text
```

적용 규칙은 다음이다.

- `hit.rowKind !== "note"`이면 mutation하지 않는다.
- `rawText.trim() === ""`이면 해당 track의 같은 `rowId + col` cell을 제거한다.
- 기존 cell이 있으면 `rawText`를 교체한다.
- 기존 cell이 없으면 새 cell을 추가한다.
- mutation 후 track cells는 `col` 오름차순, `rowId` 오름차순으로 정렬한다.

저장 JSON에는 parser/analyzer 결과를 저장하지 않는다.

### 7.2. full rebuild

mutation 후에는 다음 순서로 전체 런타임 산출물을 다시 만든다.

```text
busy = applyingEdit
score mutation
busy = rebuilding
createRuntimeDocument(score)
buildParsedDocument(runtimeDocument)
analyzeDocument(context)
createCanvasRenderInput(runtimeDocument)
buildCanvasNoteRenderItems(analysis)
renderCanvasScore(...)
busy = idle
```

이 경로는 처음에는 비용이 커도 허용한다.
partial update는 기능 검증이 끝난 뒤 병목이 확인되는 구간부터 도입한다.

rebuild 중 예외가 발생하면 `busy`는 반드시 해제되어야 한다.
구현에서는 `try/finally` 또는 동등한 오류 처리 흐름을 사용한다.
오류가 발생한 경우 마지막 정상 score 상태를 유지하고 status 영역에 실패 메시지를 표시한다.

### 7.3. import / download와의 관계

edit mode mutation은 현재 브라우저 런타임의 score JSON 상태를 바꾼다.

JSON download 기능이 붙기 전까지는 파일 저장을 보장하지 않는다.
추후 download가 활성화되면 현재 `state.document.score`를 직렬화 대상으로 삼는다.

## 8. Tuplet mode 확장 기준

tuplet mode에서는 같은 score click이 곧바로 score cell mutation이 아닐 수 있다.

예를 들어 `tupletSlotSelect` tool은 다음처럼 동작한다.

```text
score canvas click
  -> hit row의 note name 또는 current composer 결과 계산
  -> selected tuplet slot draft에 문자열 저장
  -> score JSON은 아직 변경하지 않음
```

tuplet head 최종 적용은 별도 action으로 처리한다.

```text
finalizeTuplet
  -> TupletDraft를 /m(slot1|slot2|...) rawText로 변환
  -> selected head cell에 score mutation
  -> parse/analyze/render 재실행
```

이 구조를 두는 이유는 다음이다.

- tuplet slot은 같은 `col` 안에서 분수 tick 위치를 갖는다.
- slot 내부 `-`, `~`는 외부 note cell과 연결될 수 있다.
- slot 입력 중간 상태는 유효한 score rawText가 아닐 수 있다.
- 따라서 slot 편집 상태와 score JSON 확정 상태를 분리하는 편이 안전하다.

## 9. 구현 순서

1. app state 타입과 현재 score rebuild 함수 작성
2. score canvas hit test 작성
3. `AppBusyState`와 UI disabled 동기화 함수 작성
4. 왼쪽 메뉴 영역 아래 user-facing status line 추가
5. `UiStatusMessage`를 status line에 반영하는 함수 작성
6. Default UI의 `CUSTOM` 드롭다운과 text input 활성화
7. active track 기본값을 `basic`으로 연결
8. edit mode on 상태에서 score click 시 custom text upsert
9. 빈 custom text로 cell 삭제
10. mutation 후 busy 상태로 full parse/analyze/render 재실행
11. 선택 cell과 edit 결과를 status line 또는 edit panel preview에 표시
12. 이후 `ERASER`, Long `-`/`~`, legacy click cycle, tuplet slot select 순서로 확장

## 10. edit 모듈 경계

edit 기능이 modifier UI와 tuplet 입력까지 확장되면 `main.ts`에 입력 합성, score mutation, DOM event dispatch가 함께 쌓일 수 있다.
따라서 edit 관련 순수 로직은 `src/app/edit/` 아래에 둔다.

초기 모듈 경계는 다음을 기준으로 한다.

```text
src/app/edit/
  edit_core.ts     // 현재 edit 상태를 최종 rawText 적용/삭제 명령으로 합성하는 진입점
  edit_default.ts  // 일반 note 셀의 defaultText, hold, gliss, trem, pitch modifier 조합
  edit_tuplet.ts   // tuplet draft, slot 입력, finalize 조합
  edit_apply.ts    // ScoreFile에 note cell upsert/delete 적용
```

### 10.1. edit_core.ts

`edit_core.ts`는 edit 입력 합성의 상위 진입점이다.
DOM 상태나 ScoreFile을 직접 다루지 않고, 현재 edit 입력 상태를 받아 다음 중 하나로 좁힌다.

```ts
type EditRawTextResult =
  | { kind: "apply"; rawText: string }
  | { kind: "delete" }
  | { kind: "blocked"; message: string };
```

1차 구현에서는 default CUSTOM 입력만 처리한다.
CUSTOM 입력이 비어 있거나 공백뿐이면 `delete`를 반환한다.
그 외에는 parser가 읽을 수 있는 escaped rawText를 `apply`로 반환한다.

### 10.2. edit_default.ts

`edit_default.ts`는 일반 note cell rawText 생성을 담당한다.

1차 구현 책임은 다음이다.

- CUSTOM 입력창에 보이는 문자열을 defaultText로 취급한다.
- parser 예약문자 `\`, `/`, `@`, `|`, `(`, `)`, `-`, `~`는 저장 rawText에서 backslash escape한다.
- UI 입력창에는 escape 문자를 노출하지 않는다.

후속 구현에서는 같은 모듈에 아래 조합을 추가한다.

```text
[hold] [escaped defaultText] [gliss] [trem] [absolutePitch] [microPitch]
```

### 10.3. edit_tuplet.ts

`edit_tuplet.ts`는 tuplet draft와 slot 입력을 담당한다.

tuplet은 일반 note cell과 달리 slot 입력 중간 상태가 곧바로 저장 가능한 rawText가 아닐 수 있다.
따라서 다음 상태를 분리한다.

- slot 선택과 slot별 draft text
- `/n(...)` rawText로 finalize 가능한 상태
- finalize 전 blocked 상태

1차 구현에서는 타입과 placeholder 함수를 먼저 두고 실제 finalize는 후속 단계에서 구현한다.

### 10.4. edit_apply.ts

`edit_apply.ts`는 ScoreFile 변경만 담당한다.

적용 규칙은 기존 7.1과 같다.
단, parse/analyze/render full rebuild는 이 모듈에서 수행하지 않는다.
이 모듈은 변경된 `ScoreFile`과 적용 결과만 반환하고, `main.ts` 또는 상위 app controller가 `createRuntimeDocument -> parse -> analyze -> render` 흐름을 실행한다.

## 11. 검증 기준

1차 edit mode는 최소 다음을 만족해야 한다.

- `CUSTOM` text input에 `C#4🙂` 같은 문자열을 넣고 note cell을 클릭하면 악보에 그대로 표시된다.
- 같은 cell을 다른 문자열로 다시 클릭하면 기존 cell이 교체된다.
- 빈 문자열 상태로 클릭하면 해당 cell이 제거된다.
- click 후 parser/analyzer/render가 재실행되어 화면이 즉시 갱신된다.
- rebuild 중에는 score click, edit input, zoom, reload, track 선택이 비활성화된다.
- rebuild가 끝나면 이전 edit mode와 custom text 상태가 유지된다.
- load/edit/rebuild 성공과 실패가 왼쪽 status line에 짧게 표시된다.
- renderer는 parser rawText를 직접 해석하지 않고 `AnalysisResult` 기반 item만 소비한다.
- edit mode off 상태에서는 score click이 score JSON을 변경하지 않는다.
