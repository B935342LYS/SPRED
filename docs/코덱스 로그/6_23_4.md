**현재 작업 상태**

`회귀 코드`는 `이세계 코드`의 Google Sheets + Apps Script 구조를 그대로 이식하는 프로젝트가 아니라, `regression-code/` 아래에서 TypeScript + HTML + CSS + Vite 기반 웹 앱으로 새로 구현 중인 상태입니다. 기준 문서는 `docs/HARNESS.md`이며, 구현 판단은 `1.8 -> 1.9 -> 2.x -> 1.5/1.6/1.7 -> 1.3` 순서의 active spec을 우선합니다.

구현은 이미 단순 MVP 단계를 넘어섰습니다. 현재 들어가 있는 주요 축은 다음입니다.

- Score JSON 로드/검증/RuntimeDocument/ScoreIndexes 구성
- note/global cell parser
- analyzer 이벤트 생성
- canvas renderer 기반 악보 렌더링
- edit mode와 partial rebuild
- layout editor와 layout preset 저장/불러오기
- track layer UI와 active/inactive track 정책
- Web Audio 기반 basic playback, pause/seek/resume, vibrato/tremolo/gliss 일부 처리
- YouTube iframe sync 1차 구현
- local/file score 저장, JSON load/download
- GitHub Pages 테스트 배포용 별도 `regression-code-test-publish/` 준비

실제 코드 구조도 이 흐름과 맞습니다. `src/core/`는 score/parse/analyze, `src/renderer/`는 canvas 렌더러, `src/audio/`는 재생, `src/app/`는 UI 바인딩과 orchestration, `src/orchestration/partial_rebuild/`는 부분 재빌드, `src/track/`는 트랙 정책을 맡고 있습니다.

**앞으로 예정된 작업**

가장 가까운 중심 과제는 성능 개선입니다. 특히 긴 악보에서 현재 임시 전체 캔버스 정책이 한계가 있으므로, 다음 단계는 viewport/tile rendering 쪽으로 보입니다.

우선순위가 높은 예정 작업은 다음으로 이해했습니다.

1. 긴 악보 렌더링 개선
   현재 ordinary note/global rawText edit은 partial rebuild가 들어갔지만, 렌더링은 여전히 전체 score 폭의 부담이 큽니다. visible columns + overscan 기반 viewport rendering 또는 column tile/dirty range rendering이 다음 핵심입니다.

2. partial rebuild 추가 고도화
   이미 parsed/analyzer/render item 일부 재사용은 구현됨. 남은 것은 RuntimeDocument/ScoreIndexes 부분 갱신, audio schedule 부분 갱신, analyzer range partial 쪽입니다.

3. 브라우저 수동 검증
   active track UI, inactive alpha, multi-track edit, paused-state resume, playback reset, JSON round trip, YouTube 실영상 동기화 등을 실제 브라우저에서 확인해야 합니다.

4. playback 기능 확장
   dynamics automation, refined tuplet timing, loop playback range selection, scheduler/controller loop 지원이 다음 오디오 작업 후보입니다.

5. UI/UX 안정화
   tuplet container hit-test, complex token anchor 편집 UX, undo 또는 pending-edit grouping, layout preset clear/delete UI 검토가 남아 있습니다.

6. 배포 경로 정리
   GitHub Pages용 production build에서 comment strip/minify 등 배포 최적화 방향을 정해야 합니다.

다음에 코드를 만진다면, 현재 문맥상 가장 자연스러운 시작점은 `docs/2.8-edit-invalidation-and-partial-rebuild-spec.md`, `docs/2.1-canvas-renderer-module-spec.md`를 기준으로 viewport/tile rendering 설계를 좁히고, `src/renderer/`와 `src/app/*view*`, `src/orchestration/partial_rebuild/` 경계를 확인하는 쪽입니다.

명세 기준으로는 **긴 악보 개선의 1차 목표를 “viewport bounded rendering”으로 잡는 게 맞습니다.** tile renderer까지 바로 가지 않고, 먼저 “현재 화면에 보이는 열 + overscan만 그리는 경로”를 안정화하는 순서가 명세와 현재 구현 상태에 가장 잘 맞습니다.

**핵심 판단**

`docs/2.1`은 renderer 3차 입력으로 viewport 상태를 명시합니다.

```text
CanvasRenderInput + CanvasNoteRenderItem[] + CanvasViewport
  -> visible render items
  -> virtualized canvas draw
```

`docs/HARNESS.md`도 장악보 성능 로드맵의 첫 목표를 이렇게 정리합니다.

```text
visible score columns plus a small overscan range
```

따라서 사용할 기본 로직은:

1. 전체 `ScoreFile`, `ParsedScoreDocument`, `AnalysisResult`, `CanvasAnalyzedRenderInput`은 계속 전체 악보 기준으로 유지
2. `CanvasScoreLayout`도 전체 column/row 좌표계를 유지
3. 실제 canvas bitmap과 draw 대상만 현재 `scoreArea.scrollLeft`, `clientWidth`, `overscanPx` 기준으로 제한
4. 스크롤 시 note/noteMarker/globalMarker/base 중 필요한 layer만 viewport 범위로 다시 그림
5. 편집 시에는 기존 `2.8` partial rebuild 결과를 유지하되, redraw 물리 범위를 viewport 또는 dirty tick range와 결합

**적용할 렌더링 모델**

1차 구현은 “전체 stage DOM width는 유지하되, 큰 bitmap canvas를 만들지 않는 방식”이 적절합니다.

- `.score-canvas-stage`
  - 전체 scroll width 유지
  - 사용자가 전체 악보 폭을 가로 스크롤할 수 있어야 함
- canvas layer
  - 실제 bitmap/css width는 viewport + overscan 정도로 제한
  - `transform: translateX(startX)`로 전체 stage 안의 해당 위치에 배치
  - context transform에는 `-startX`를 반영해서 기존 draw 함수가 전체 좌표계 기준으로 그릴 수 있게 함

현재 `canvas_coordinate.ts`에 이미 `resizeCanvasLayerToDynamicViewport()`가 있어 이 방향의 초안이 들어가 있습니다. 이걸 note layer 일부에만 쓰는 실험 단계에서, base/grid/marker까지 더 일관된 viewport 경로로 확장하는 것이 자연스럽습니다.

**layer별 정책**

명세 기준 layer 정책은 이렇게 잡습니다.

- `layout` label layer
  - 세로 전체 행 라벨이 필요하므로 폭은 작고 문제 없음
  - 기존처럼 전체 높이 기준 유지 가능
- `base` layer
  - 현재는 전체 악보 폭 grid를 그리는 부담이 큼
  - viewport bounded draw 대상으로 전환
  - visible columns + overscan 범위의 row background, column grid, row grid만 그림
- `globalMarker` layer
  - beat/bar/BPM/dynamics marker
  - viewport tick range에 걸치는 marker만 그림
- `noteMarker` layer
  - gliss/orphan/tuplet marker
  - gliss는 화면 밖 anchor와 연결될 수 있으므로 visible range 판정에 padding 또는 item overlap 판정 필요
- `note` layer
  - note/mute/global text를 viewport range 기준으로 그림
  - text overflow 때문에 x overscan 또는 clear padding 필요

**visible range 계산**

새 helper를 두는 게 명세와 맞습니다.

후보 파일:

```text
src/renderer/canvas_viewport.ts
src/renderer/canvas_visible_range.ts
```

필요 타입:

```ts
type CanvasViewport = {
  scrollLeft: number;
  width: number;
  overscanPx: number;
};

type CanvasVisibleTickRange = {
  startTick: number;
  endTick: number;
  startX: number;
  endX: number;
};
```

계산은 기존 좌표 helper를 재사용합니다.

```text
startX = max(0, scrollLeft - overscanPx)
endX = min(stageWidth, scrollLeft + viewportWidth + overscanPx)
startTick = floor(xToColumn(startX))
endTick = ceil(xToColumn(endX))
```

**편집 partial rebuild와의 결합**

`docs/2.8` 기준으로 현재 partial rebuild는 이미 다음을 합니다.

- note/global invalidation 분리
- renderer item group 분리
- note/global partial render
- dirty tick range 기반 note redraw
- drag input batching

긴 악보 렌더링 개선에서는 이 구조를 바꾸지 말고, 마지막 “물리 canvas redraw”만 제한합니다.

즉:

```text
edit
  -> 기존 partial rebuild로 renderer item group 갱신
  -> dirty tick range 계산
  -> viewport range와 dirty range를 합침
  -> 화면에 필요한 범위만 canvas redraw
```

dirty range가 현재 viewport와 겹치지 않으면 renderer item state만 갱신하고, 즉시 redraw는 생략할 수 있습니다. 이후 사용자가 해당 위치로 스크롤하면 viewport render가 최신 item으로 그립니다.

**우선순위**

1. `base`, `marker`, `noteMarker`, `note` layer의 viewport resize/draw 경로 정리
2. scroll 이벤트에서 모든 동적 layer가 viewport 기준으로 갱신되도록 통일
3. full render 후에도 거대한 bitmap을 만들지 않도록 base layer까지 bounded 처리
4. partial edit redraw에서 viewport와 dirty tick range를 교차 적용
5. 이후 필요하면 tile cache 도입

tile rendering은 2차 목표로 미루는 게 맞습니다. 지금은 canvas bitmap 한계와 전체 redraw 비용이 1차 병목이므로, viewport bounded rendering만으로도 큰 개선을 기대할 수 있습니다.

1. 긴 악보 1차 개선은 `tile cache`가 아니라 `viewport bounded rendering`으로 고정했습니다.
2. 전체 score/analyze/render input은 유지하고, 실제 canvas bitmap과 draw 대상만 `visible range + overscan`으로 제한합니다.
3. `.score-canvas-stage`는 전체 scroll extent를 유지하고, 각 layer canvas만 viewport 폭으로 줄여 `translateX(startX)`와 context `-startX` transform을 사용합니다.
4. `canvas_viewport.ts`, `canvas_visible_range.ts`는 renderer 계층에 둡니다.
5. `orchestration/partial_rebuild`에는 viewport 자체가 아니라, edit dirty range와 current viewport가 겹치는지 판단하는 redraw plan 정도만 둘 수 있다고 명시했습니다.

질문하신 `orchestration/partial_rebuild` 배치에 대해서는 절반만 맞다고 봅니다.

- 어울리는 것: “편집으로 생긴 dirty tick range가 현재 viewport와 겹치는가”, “겹치면 note/global/structure 중 어떤 redraw scope를 호출할 것인가” 같은 render plan 조율
- 어울리지 않는 것: visible range 계산, canvas resize, item visibility filter, draw 함수

이유는 viewport rendering이 edit에만 쓰이지 않기 때문입니다. initial render, scroll, resize, zoom, fullscreen, playback scroll에서도 같은 viewport renderer가 필요합니다. 그래서 viewport 좌표/그리기 로직은 renderer에 두고, partial rebuild는 edit invalidation과 viewport 교차 판단까지만 맡기는 쪽이 경계가 깔끔합니다.

> tile cache는 타일 패턴을 저장한 후 화면에 들어오기 직전 청크 단위로 까는 것을 의미하나요?

대체로 맞습니다. 여기서 말한 `tile cache`는 악보 전체를 한 장의 거대한 canvas로 그리지 않고, 가로 방향을 일정 폭의 타일/청크로 나눈 뒤 각 타일을 따로 그려 저장해 두는 방식입니다.

예를 들면:

```text
columns 0-255     -> tile 0
columns 256-511   -> tile 1
columns 512-767   -> tile 2
...
```

스크롤 위치가 바뀌면 현재 화면에 필요한 타일과 양옆 여유 타일만 canvas에 붙입니다. 이미 그려진 타일은 재사용하고, 아직 없거나 dirty 처리된 타일만 다시 그립니다.

다만 “타일 패턴”을 반복 배경처럼 저장한다기보다는, 이 앱에서는 보통 다음에 가깝습니다.

- base grid tile: row 배경, column grid, row grid
- global marker tile: beat/bar/BPM/dynamics
- note tile: 해당 column 범위의 note, text, trem/vib, mute
- note marker tile: gliss, tuplet, orphan anchor 등

장점:
- 5000열 이상에서도 한 장짜리 초대형 canvas를 만들지 않음
- 스크롤 시 이미 그린 타일을 빠르게 재사용 가능
- 편집 시 해당 타일만 dirty 처리 가능

단점:
- gliss처럼 타일 경계를 가로지르는 선 처리 필요
- zoom/layout/DPR 변경 시 캐시 폐기 필요
- note text overflow, tuplet, hold 연결처럼 주변 타일 영향을 관리해야 함
- 메모리 관리와 tile eviction 정책이 필요함

그래서 지금은 먼저 `viewport bounded rendering`, 즉 “현재 보이는 범위만 즉석에서 그리기”를 1차로 두는 게 안전합니다. 그다음에도 스크롤 중 redraw 비용이 크면 tile cache로 확장하는 순서가 맞습니다.

> 레이아웃이 중간에 변하는 일이 없으므로, 회색 줄무늬 패턴의 배경은 새로 그리는 대상에서 제외하면 어떤가요? 실제로는 박자선/마디선과 노트/마커 레이어만 새로 그리면서 이동하는 듯한 시각 효과를 내는 것입니다.

좋은 방향입니다. 지금 장악보 병목을 줄이는 데 꽤 효과적일 가능성이 큽니다.

정리하면 `base grid`를 두 부분으로 쪼개는 방식입니다.

1. **정적 배경**
   - 회색 roll background
   - note/gap/global row 줄무늬
   - row horizontal line
   - layout 변경, zoom 변경, reverse 변경 때만 갱신
   - scroll, note edit, global edit 때는 다시 그리지 않음

2. **동적 전경**
   - column grid 또는 필요 시 column tick line
   - beat/bar marker
   - BPM/dynamics marker
   - note rectangle/text
   - gliss/tuplet/vib/trem/mute marker
   - viewport 범위만 redraw

특히 “회색 줄무늬 패턴”은 x축으로 반복되는 성격이라, 매번 전체 폭을 그릴 필요가 없습니다. 구현 후보는 두 가지입니다.

- `score-canvas-stage`에 CSS/DOM 배경으로 깔기
  - row별 높이가 가변이라 단순 CSS repeating-gradient만으로는 부족할 수 있음
  - 하지만 renderer가 계산한 row y/height로 작은 vertical strip 또는 background layer를 만들 수 있음

- `base` canvas를 “전체 폭 canvas”가 아니라 “viewport 높이 기준 정적 row background layer”로 유지
  - x 방향은 1px 또는 viewport 폭만 칠해도 같은 시각 효과
  - scrollLeft 변화와 무관하므로 redraw 불필요
  - layout/zoom/row order 변경 때만 다시 그림

제가 보기엔 1차 구현에서는 **base layer를 row background 전용 static layer로 분리하고, column grid는 dynamic viewport layer로 옮기는 방식**이 가장 안전합니다.

```text
static row background
  - roll background
  - note row background
  - row horizontal line

dynamic viewport layers
  - column grid
  - beat/bar/BPM/dynamics
  - notes
  - note markers
```

주의할 점은 column grid입니다. 세로 column line은 scrollLeft에 따라 화면에서 움직여야 하므로, 회색 줄무늬 배경과 달리 완전히 정적 배경으로 빼면 안 됩니다. 다만 이것도 가벼운 선 반복이라 CSS background나 viewport draw로 처리하면 됩니다.

이 접근의 장점은 명확합니다.

- scroll 시 row background redraw 제거
- edit 시 base redraw 제거
- 긴 악보에서도 full-width base canvas 불필요
- layout이 안정적인 동안 캐시 무효화 조건이 단순함

무효화 조건은 이렇게 잡으면 됩니다.

- static row background 재생성:
  - score load
  - layout apply/preset
  - zoom
  - reverse row
  - DPR 변경
  - fullscreen/viewport height 변화 중 canvas bitmap 재조정 필요 시

- 동적 layer redraw:
  - scroll
  - note/global edit
  - playback scroll
  - active track toggle
  - timing/global marker 변경

따라서 제안하신 “박자선/마디선과 노트/마커 레이어만 새로 그리면서 이동하는 듯한 시각 효과”는 타당합니다. 명세에는 `base layer`를 `static row background`와 `viewport column/grid foreground`로 나누는 방향으로 추가하는 게 좋겠습니다.

> 현재의 전체 악보 draw보다 가벼운 것은 확실할까요?

확실히 가벼워질 가능성이 높습니다. 다만 “항상 모든 상황에서”는 아닙니다. 병목이 canvas bitmap 크기와 full-width draw에 있다면 거의 확실히 개선됩니다.

현재 전체 악보 draw의 비용은 대략 악보 전체 폭에 비례합니다.

```text
현재:
draw cost ~= 전체 column 수 * row 수 * layer 수
canvas bitmap ~= 전체 score width * score height
```

viewport bounded rendering은 화면에 보이는 폭 + overscan만 그립니다.

```text
변경 후:
draw cost ~= (viewport width + overscan) * row 수 * dynamic layer 수
canvas bitmap ~= (viewport width + overscan) * score height
```

예를 들어 5000 columns, columnWidth 21px이면 전체 score width는 약 105,000px입니다. 화면 viewport가 1,700px이고 overscan을 양쪽 합쳐 850px 정도 둔다면 실제 draw 폭은 약 2,550px입니다.

```text
105,000px -> 2,550px
약 41배 폭 감소
```

게다가 회색 row background를 static으로 빼면 scroll 때마다 다시 그리는 대상은 더 줄어듭니다.

**가벼워지는 부분**

- full-width canvas bitmap 생성 방지
- full-width `clearRect`, row fill, column grid draw 방지
- 보이지 않는 note/text/marker draw 생략
- scroll 중 정적 row background redraw 생략
- browser canvas size limit 회피

**가벼워지지 않는 부분**

- 전체 `AnalysisResult` 보관 비용
- 전체 note item 배열 보관 비용
- score load나 analyzer full rebuild 비용
- visible item filtering을 매번 선형 탐색하면 item 수가 많을 때 일부 비용 발생
- gliss처럼 viewport 밖 연결을 고려해야 하는 marker 판정 비용

그래서 구현 품질이 중요합니다. visible item filtering을 매 scroll마다 전체 item 배열 선형 탐색으로만 하면, draw는 줄어도 JS filtering 비용이 남을 수 있습니다. 1차는 그래도 개선될 가능성이 크지만, 더 안정적으로 하려면 item을 tick 기준으로 정렬해두고 visible tick range 근처만 찾는 방식이 좋습니다.

결론은:

- **canvas draw/bitmap 병목에는 확실히 유리합니다.**
- **전체 item scan이 새 병목이 되지 않도록 visible range filtering을 조심해야 합니다.**
- 현재 문제인 “긴 악보 full canvas/render 한계”에는 viewport bounded rendering이 맞는 1차 처방입니다.
