네, **분석 단계는 BPM과 거의 같은 방식**으로 보면 됩니다. 이미 `analyze_global_segments.ts`가 `bpm`과 `dynamics`를 같은 “linear 가능 전역 stream”으로 처리하고 있고, `analyze_dynamics.ts`도 `resolveLinearGlobalRange()`를 써서 `startValue`, `endValue`, `curve`가 있는 segment를 만들고 있습니다.

다만 **audio 연결 방식은 BPM과 다릅니다.**

BPM은 시간축 자체를 바꿉니다.

```text
tick -> seconds 변환
```

Dynamics는 시간축을 바꾸지 않고, 이미 변환된 note/gliss schedule 위에 음량 값을 얹습니다.

```text
tick -> seconds 변환은 timingTimeline 사용
해당 tick/seconds 구간의 dynamicsTimeline 조회
note/gliss gain 또는 velocity에 반영
```

구현 방향은 이렇게 잡는 게 좋습니다.

1. `dynamicsTimeline`을 조회하는 mapper 추가
   - `tick_time_mapper.ts`처럼 별도 `dynamics_mapper.ts`를 두거나, `audio_schedule_builder.ts` 안의 private helper로 시작할 수 있습니다.
   - 입력: `AnalyzedDynamicsSegment[]`
   - 출력: 특정 note/gliss 구간에 겹치는 gain automation 목록.

2. dynamics 값 스케일 결정
   - 현재 기본 dynamics는 `100`입니다.
   - 가장 단순한 1차 구현은 `100 -> gain 1.0`, `50 -> gain 0.5`, `0 -> gain 0` 같은 선형 매핑입니다.
   - 즉 `gain = clamp(value / 100, 0, 1)`.

   - 타입에는 이미 `AudioAutomationEvent`와 `gainRamp` 후보가 있습니다.

4. `oscillator_backend.ts`에서 gain envelope에 dynamics automation 반영
   - 현재 note gain은 `attack -> sustain -> release`만 합니다.

```text
oscillator
-> tremoloGain?
-> dynamicsGain
-> envelopeGain
-> masterGain
```

그러면 envelope은 시작/끝 클릭 방지, dynamicsGain은 악보상 셈여림 변화만 담당합니다.

**전역 행 시각화 명세 초안**

대상 입력:
- `AnalysisResult.timingTimeline`
- `AnalysisResult.dynamicsTimeline`
- renderer는 `rawText`를 다시 해석하지 않고 analyzer timeline만 소비한다.

BPM marker:
- `timingTimeline`의 BPM 변화 정보를 `CanvasMarkerItem`으로 변환한다.
- BPM이 즉시 변경되는 지점은 **반투명 하늘색 세로선**으로 표시한다.
- BPM이 linear ramp인 구간은 ramp 시작 지점에 세로선을 표시한다.
  - `startBpm < endBpm`: 증가, **빨간색 세로선**
  - `startBpm > endBpm`: 감소, **초록색 세로선**
- BPM marker는 score 영역 전체 높이에 걸쳐 그린다.
- beat/bar marker와 같은 marker layer에 그리되, BPM marker가 beat/bar보다 시각적으로 더 잘 보이도록 beat/bar 이후에 그린다.
- `beatsPerBar`, `stepsPerBeat` 변화는 BPM marker가 아니다. 기존 beat/bar grid 재계산에만 영향을 준다.

Dynamics guide:
- `dynamicsTimeline`을 기반으로 dynamics 행 위에 **연회색 가로선**을 그린다.
- dynamics 값 범위는 현재 parser 기준 `0..150`이다.
- 선 두께는 dynamics row height에 정비례한다.

```text
lineThickness = rowHeight * (value / 150)
```

- `value = 150`이면 dynamics row height 전체를 채운다.
- `value = 0`이면 보이지 않는다.
- 가로선은 dynamics row의 세로 중앙을 기준으로 위아래로 두께가 퍼진다.
- instant segment는 일정 두께의 직사각형 또는 굵은 선으로 표시한다.
- linear segment는 시작 두께와 종료 두께가 다르므로, 단순 `lineWidth` 하나로 그리지 않고 trapezoid 형태 또는 짧은 구간 분할로 두께 변화를 표현한다.
- dynamics rawText는 기존처럼 note/global text layer에서 흰색으로 계속 표시한다. dynamics guide는 그 아래쪽 marker/base 성격으로 둔다.

색상 후보:
```text
BPM instant: rgba(96, 200, 255, 0.45)
BPM accel:   rgba(255, 80, 80, 0.65)
BPM rit:     rgba(80, 220, 130, 0.65)
Dynamics:    rgba(220, 220, 220, 0.45)
```

남은 확인점은 하나입니다.

BPM의 `col = 0` 초기값도 세로선으로 표시할까요?
제 추천은 **초기값은 rawText로 이미 보이므로 세로선은 제외**하고, 실제 변화 지점만 표시하는 것입니다. 이 기준이면 화면이 덜 복잡합니다.

네, 현재 방식이 보통은 레거시보다 연산이 덜 들어갑니다.

레거시는 dynamics/expression을 column 단위로 순회하면서 각 column마다 `lineWidth`를 계산하고, `beginPath -> moveTo -> lineTo -> stroke`를 반복합니다. 즉 긴 구간이면 열 개수만큼 선분을 계속 그립니다.

현재 `회귀 코드`는 dynamics timeline segment 하나마다 사다리꼴 path 하나를 만들고 `fill()` 한 번으로 그립니다. constant dynamics면 직사각형, linear dynamics면 위아래 폭이 달라지는 사다리꼴입니다. 그래서 구간 길이가 길수록 `N개 column 선분`이 아니라 `1개 도형`에 가까워져서 draw call 수가 줄어드는 구조입니다.

다만 tradeoff는 있습니다. 현재 방식은 analyzer가 만든 dynamics segment 기준이라 “각 column마다 끊기는 계단형 표현”이 아니라 “구간 단위의 연속 면”입니다. 지금 목표인 `instant/linear` 변화 시각화에는 오히려 더 잘 맞고, 성능상으로도 더 유리한 편입니다.

> 하나의 도형이 긴 악보의 화면 끝까지 지속되는 경우에도 선보다 렌더링이 유리한 것이죠?

대체로 유리합니다. Canvas 기준으로는 긴 선 하나나 긴 사각형/사다리꼴 하나나 픽셀을 칠하는 면적은 비슷하지만, 현재 방식은 `column마다 stroke()`를 반복하지 않고 segment당 `fill()` 한 번만 호출하므로 JS 루프와 Canvas draw call 비용이 훨씬 적습니다.

정리하면:

- 레거시 방식: 긴 구간이 1,000칸이면 대략 1,000번 선분 계산 + 1,000번 stroke

- 화면 끝까지 이어지는 긴 도형이어도 draw call 수는 여전히 1번
- 실제 칠해야 하는 픽셀 수는 늘지만, 보통 병목은 수많은 draw call과 per-column 루프 쪽이 더 큽니다

> 큰 흐름은 이렇겠네요.
> 0. UI와 기능 연결
> 1. 타입 정의
> 2. 사용자 수정 내용 객체 생성
> 3. 유효성 검증 및 경고
> 4. 적용
> 5. 프리셋 형태로 저장

네, 그 흐름이 맞습니다. 다만 구현 순서로는 0번 UI 연결을 맨 앞에 두기보다, 내부 기능을 먼저 만든 뒤 마지막에 UI에 붙이는 편이 안정적입니다.

정리하면 이렇게 잡는 게 좋겠습니다.

1. **타입 정의**
   - `LayoutDraftBundle`
   - `UserLayoutPresetData`
   - apply 결과 타입

2. **사용자 수정 내용 객체 생성**
   - 현재 `ScoreFile`에서 draft 생성
   - note/gap row 추가, 삭제
   - rowId, displayLabel, gap `fromMidi/toMidi` 자동 생성

3. **유효성 검증 및 경고**
   - draft 단계 경고
   - `validateScoreFile()` 최종 검증 확장
   - 삭제될 cell 계산

4. **적용**
   - score clone 생성
   - `instData`와 rowDefinitions 덮어쓰기
   - 삭제 cell 반영
   - full rebuild

5. **프리셋 저장**
   - localStorage 저장/조회
   - JSON 파일 저장/불러오기
   - preset 표시명 중복 검증

6. **UI와 기능 연결**
   - Modify 버튼
   - dialog
   - Add/Delete/Apply/Reset
   - Local/File Save/Load

개념 흐름으로는 말씀하신 0→5가 맞고, 실제 구현 순서는 1→5→0이 더 적절합니다. UI를 먼저 만들면 아직 검증/적용 함수가 없어 임시 코드가 늘어날 가능성이 큽니다.

> 바로 전부 만들기보단 순차적으로 진행하는 것이 좋아보입니다. 우선 타입과 팝업 UI부터 만들면 어떻겠습니까?

좋습니다. 첫 단계로 **타입 + 팝업 UI 골격**만 만드는 것이 적절합니다.

범위는 이렇게 자르면 됩니다.

1. `layout_types.ts`
   - `LayoutDraftBundle`
   - `LayoutDraftRow`
   - `UserLayoutPresetData`
   - `UserLayoutPresetIndexItem`
   - `LayoutApplyResult` 정도만 정의

2. 팝업 UI 골격
   - `Modify` 버튼 클릭 시 dialog 열기
   - layout preset 이름 입력
   - 악기 정보 영역
   - string 선택 드롭다운
   - row 목록 영역
   - row 추가 입력 영역
   - `Add Row`, `Delete`, `Apply`, `Cancel`, `Reset`, `Local Save/Load`, `File Save/Load` 버튼 배치

3. 아직 하지 않을 것
   - 실제 row 추가/삭제 로직
   - score 적용
   - cell 삭제
   - localStorage 저장
   - 파일 저장/불러오기

즉 첫 PR/첫 작업 단위는 “기능이 동작하는 UI”가 아니라 **이후 기능을 붙일 수 있는 타입과 화면 구조**입니다. 이러면 적용 로직을 섞지 않고 DOM 구조와 상태 흐름을 먼저 안정화할 수 있습니다.
