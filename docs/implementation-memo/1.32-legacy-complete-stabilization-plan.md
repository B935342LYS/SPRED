# 1.32. Legacy complete stabilization plan

## 1. 목적

이 메모는 레거시 `이세계 코드` 시절의 주요 기능이 `회귀 코드`에 1차 구현된 뒤, 다음 단계에서 반복 테스트, 성능 최적화, 확장 개발을 어떤 순서로 진행할지 정리한다.

이 문서는 구현 메모이며 active spec을 대체하지 않는다.
저장 형식, parser/analyzer/audio/renderer 계약은 기존 active spec을 따른다.

## 2. 현재 단계 판단

현재 구현은 레거시 기능 재현의 1차 완료 상태로 본다.

완료된 축:

- Score JSON load/download, Local Save/Load
- parser/analyzer 기반 note/global 해석
- canvas 악보 렌더링
- edit mode note/global 입력
- default, long hold, vibrato hold, gliss, tremolo, tuplet 입력/렌더
- layout editor, layout preset local/file save/load
- active track filter, inactive track render, active track playback
- Web Audio playback, pause/resume/seek/stop
- YouTube sync first pass
- long-score viewport bounded rendering
- Loop UI, loop marker, loop playback, loop select UX
- GitHub Pages test publish staging

따라서 다음 단계의 기본 목표는 새 기능 추가보다 다음 세 가지를 우선한다.

1. 기능 회귀를 잡기 위한 반복 테스트
2. 긴 악보와 반복 편집에서의 성능 병목 제거
3. 확장 기능을 넣기 전에 module boundary를 안정화

## 3. 안정화 테스트 범위

### 3.1. 기본 입출력

확인할 항목:

- 기본 template score 자동 로드
- JSON Load로 저장 파일 로드
- JSON Download 후 다시 JSON Load
- Local Save 후 Local Load
- 8 MiB JSON load 제한
- 3 MiB local save 제한
- 100자 rawText 제한
- invalid rowId, invalid trackId, 구조 불일치 import 실패

완료 기준:

- 저장 전후 score metadata, global cell, track cell, layout row가 유지된다.
- 제한 초과는 앱이 멈추지 않고 status message로 실패한다.
- import 실패 후 기존 score 상태가 유지된다.

### 3.2. 편집 입력

확인할 항목:

- Default AUTO sharp/flat
- CUSTOM text escaping
- comment cell
- eraser
- long hold `-`
- vibrato hold `~`
- gliss S/M/E와 ID
- tremolo division
- absolutePitch, microPitch
- number UI global row edit
- tuplet slot filling, finalize, extend
- click cycle `current -> - -> ~ -> current`
- left drag input
- right drag delete
- active track 다중 입력
- active track 0개 상태에서 note edit block

완료 기준:

- rawText가 의도한 parser token으로 저장된다.
- renderer와 analyzer output이 편집 직후 일치한다.
- drag edit 중 UI가 과도하게 멈추지 않는다.

### 3.3. 렌더링

확인할 항목:

- 기본 note rectangle과 text
- hold merge
- mute marker
- cent offset vertical shift
- tremolo chop line
- vibrato sine line
- gliss connected chain
- orphan gliss marker
- tuplet container와 slot note/rest
- global row rawText
- beat/bar marker
- active/inactive track alpha
- reverse row view
- text off
- speed scale
- loop marker

완료 기준:

- viewport scroll 후 표시 누락이나 잔상이 없다.
- zoom/speed 변경 후 현재 위치가 유지된다.
- long score에서 dynamic layer가 현재 viewport 기준으로 정상 redraw된다.

### 3.4. 재생

확인할 항목:

- play from start
- pause/resume
- seek bar
- score scroll seek
- stop
- active track playback filter
- mid-event resume
- note overlap gainScale
- dynamics gain automation
- vibrato LFO
- tremolo gain gate
- standalone gliss fallback
- connected gliss chain
- loop start/end
- loop end hard cut
- loop 범위 안 seek 후 play
- loop 범위 밖 seek 후 start 보정

완료 기준:

- loop end column의 이벤트는 재생되고 `end + 1` 경계에서 끊긴다.
- start를 걸치는 이벤트는 start 지점부터 중간 진입 재생된다.
- loop boundary에서 긴 note/gliss가 다음 cycle로 이어지지 않는다.
- pause/resume/seek 후 scroll과 audible playback이 같은 score time을 가리킨다.

### 3.5. YouTube sync

확인할 항목:

- video ID 또는 URL 입력
- offset 입력
- Reload
- mode on/off
- play/pause/stop/seek follower 동작
- loop boundary에서 YouTube seek 보정
- empty video input
- embedding blocked video
- score load 후 input sync

완료 기준:

- YouTube가 audio controller의 score time을 follower로 따라간다.
- offset 조정 후 reload하면 현재 score time 기준으로 재동기화된다.
- YouTube 오류가 score/playback 조작을 막지 않는다.

### 3.6. Layout

확인할 항목:

- Modify dialog open/close
- note/gap row add/delete
- common note height 변경
- Apply
- 삭제 확인 boundary
- Default Layout 복귀
- Local Slot 1..3 save/load
- File preset save/load
- layout apply 후 playback reset
- layout apply 후 loop reset

완료 기준:

- structural conflict가 명시적으로 처리된다.
- layout 변경 뒤 renderer row label, hit-test, playback scroll이 같은 layout을 기준으로 동작한다.

## 4. 성능 테스트 범위

### 4.1. 긴 악보 기준

기준 악보 후보:

- 1000 column 기본 score
- 3000 column 중간 score
- 9000+ column long score
- note/gliss/tuplet이 조밀한 stress score

확인할 항목:

- 최초 load 시간
- full render 시간
- horizontal scroll 체감 FPS
- edit 1회 latency
- drag edit latency
- loop marker 변경 redraw 시간
- active track toggle 후 redraw 시간

우선 개선 후보:

1. dynamic layer tile/chunk rendering
2. time-budget 기반 scroll redraw throttle
3. note marker visibility index 추가
4. runtime index partial update
5. analyzer dirty-column partial rebuild
6. audio schedule partial rebuild

현재 정책:

- 실제 병목이 확인되기 전에는 tile cache 같은 큰 구조 변경을 미루고, viewport bounded rendering + partial rebuild의 효과를 먼저 실측한다.

### 4.2. 메모리와 canvas bitmap

확인할 항목:

- 고 DPI 화면에서 canvas bitmap allocation
- zoom 200%, speed 4.0x, 9000+ column 조합
- fullscreen / Fit Height 반복
- layout 변경 후 이전 canvas state 정리

완료 기준:

- 브라우저가 canvas allocation 실패 없이 동작한다.
- dynamic layer가 full-score width bitmap으로 되돌아가지 않는다.

## 5. 자동 검증 명령

기본 검증 명령:

```text
npm run typecheck
npm run test:score
npm run test:parse
npm run test:edit
npm run test:track
npm run test:view
npm run test:analyze
npm run test:audio
npm run test:layout
npm run test:youtube
npm run test:partial
npx tsc --noEmit --noUnusedLocals --noUnusedParameters
npm run build
```

publish staging 검증:

```text
npm run typecheck
npm run build
```

자동 테스트에 없는 범위:

- 실제 Web Audio 청감
- YouTube iframe 동작
- fullscreen browser behavior
- pointer drag 체감 성능
- 긴 악보 scroll FPS

위 항목은 수동 테스트 체크리스트로 유지한다.

## 6. 배포 전 확인

배포 전 확인 순서:

1. main implementation `regression-code/`에서 검증 명령 실행
2. 브라우저에서 핵심 수동 시나리오 확인
3. publish staging copy에 `src/`, `styles/`, `index.html` 반영
4. publish staging copy에서 `npm run typecheck`, `npm run build`
5. publish repo commit/push
6. GitHub Pages에서 로드, 재생, Loop, YouTube mode smoke test

주의:

- `regression-code-test-publish/`는 main SPRED repository의 소스가 아니라 배포 staging copy이다.
- staging copy에는 dev test 파일이 없으므로 테스트는 main implementation에서 우선 실행한다.

## 7. 확장 진입 기준

새 확장 기능은 다음 조건을 만족한 뒤 시작한다.

- 레거시 기능 완료 체크리스트에서 치명적 항목이 없다.
- 9000+ column score에서 scroll/edit이 실사용 가능한 수준이다.
- JSON save/load round trip에 known regression이 없다.
- playback loop, seek, pause/resume이 안정적이다.
- YouTube mode가 오류 상태에서도 앱 전체를 막지 않는다.

우선 확장 후보:

1. sampled instrument 또는 Tone.js backend 검토
2. dynamics automation 세부 청감 조정
3. tuplet timing refinement
4. undo / pending edit grouping
5. local layout slot clear/delete UI
6. production build comment stripping/minification policy
7. game mode 설계 착수

game mode는 별도 큰 단계로 분리한다.
초기 game mode는 score JSON에 결과를 저장하지 않고, browser audio input과 runtime pitch detection을 사용한다는 기존 결정을 유지한다.

## 8. 회귀 발견 시 처리

회귀가 발견되면 다음 정보를 남긴다.

- 사용한 score JSON 또는 fixture
- 브라우저와 OS
- 재현 단계
- 기대 동작
- 실제 동작
- 관련 module boundary
- 자동 테스트로 옮길 수 있는지 여부

수정 후에는 가능한 한 해당 회귀를 `dev/test_*.ts` fixture나 순수 helper test로 추가한다.
브라우저 API 의존이 강한 경우에는 수동 테스트 항목으로 유지한다.
