# 1.18. 3단계 tuplet analyzer first pass 구현 메모
> 본 설계 메모 문서 1.18에서는 tuplet token을 analyzer event로 변환하는 첫 구현 단위를 기록한다.
> 본 문서는 active specification을 대체하지 않는다.
> renderer bracket 표시는 아직 다음 단계로 남겨 두었고, 이번 문서는 group/slot event 생성과 테스트 범위를 중심으로 한다.

## 1. 개요

이번 구현 구간의 목표는 parser가 이미 구조화하던 tuplet token을 analyzer 출력으로 승격하는 것이다.

대상 token 흐름은 다음과 같다.

```text
ParsedPletHeadCell /n(...)
+ 오른쪽으로 연속된 ParsedPletExtendCell /&
-> TupletGroupEvent
-> slot NoteEvent / RestEvent
```

renderer bracket은 아직 만들지 않았다.
다만 slot note는 기존 `NoteEvent` 구조를 사용하므로, 현재 note renderer는 tuplet slot의 표시 텍스트를 유리수 tick 위치에 그릴 수 있다.

## 2. 관련 파일

- `regression-code/src/core/analyze/analyze_track.ts`
  - `ParsedPletHeadCell`을 `TupletGroupEvent`와 slot event 목록으로 변환한다.
  - `/&`가 head 오른쪽 같은 row에 연속되어 있으면 group 길이에 포함한다.
  - slot note의 `@n(midi)`를 실제 note rowId로 매핑한다.
  - slot rest를 `RestEvent`로 만든다.
- `regression-code/dev/test_analyze.ts`
  - head + extend 2 tick 구간에 3분할 tuplet을 배치해 group, note slot, rest slot, 유리수 tick을 검증한다.

## 3. group 길이 계산

tuplet head는 해당 cell 하나만으로 group 전체 길이를 확정하지 않는다.
head 오른쪽 같은 row에 `/&`가 연속되어 있으면 그 cell들을 group 길이에 포함한다.

```text
/3(...) /&
```

위 예시는 정수 tick 기준 2칸 길이의 group이다.

현재 구현은 같은 row에서 오른쪽으로 연속된 `/&`만 포함한다.
중간에 다른 cell이 있거나 `/&`가 끊기면 group 탐색을 종료한다.

## 4. slot 시간 계산

slot 시간은 `TimeFraction`으로 표현한다.
예를 들어 col 42에서 시작하고, group 길이가 2칸이며, divNum이 3이면 slot 범위는 다음과 같다.

```text
slot 0: 42 .. 42 + 2/3
slot 1: 42 + 2/3 .. 43 + 1/3
slot 2: 43 + 1/3 .. 44
```

구현에서는 분수를 약분하지 않는다.
renderer와 hold 연결 비교는 `numerator / denominator` 값으로 비교한다.

## 5. slot note 매핑

tuplet slot note는 parser 단계에서 반드시 `@n(midi)` 위치 token을 가진다.
analyzer는 head가 놓인 note row의 `stringId`와 slot의 `midiNum`을 묶어 `ScoreIndexes.noteRowIdByStringMidi`에서 실제 표시 rowId를 찾는다.

```text
`${headRow.stringId}|${slotNote.position.midiNum}`
```

row 매핑에 실패하면 현재 first pass에서는 해당 slot을 invalid 상태로 기록하고, 시간 점유용 `RestEvent`를 만든다.
진단 생성은 이후 `analysisIssues` 연결을 정리할 때 확장할 수 있다.

## 6. 기존 note 처리와 공유한 부분

slot note는 일반 note와 같은 `NoteEvent`를 사용한다.
다만 `sourceCells[0].slotIndex`와 `tuplet` membership이 추가로 붙는다.

공유한 처리 범위는 다음이다.

- display position
- sound pitch
- display text anchor
- hold merge
- vib `~`
- trem `@t`
- gliss anchor `@g`

hold merge는 정수 col이 아니라 유리수 시작 tick으로 이어짐을 판정한다.
따라서 일반 note용 `findConnectablePreviousEvent`와 별도로 `findConnectablePreviousEventAtTick`을 두었다.

## 7. 테스트 확인

`dev/test_analyze.ts`에 다음 fixture를 추가했다.

```text
/3(C@n(60)||E@t(3)@n(64)) /&
```

검증한 항목은 다음이다.

- `TupletGroupEvent`가 1개 생성된다.
- group id는 `trackId:tuplet:rowId:col` 형식으로 안정적이다.
- group time은 head와 extend를 합쳐 42..44이다.
- slots 상태는 `note,rest,note`이다.
- slot note 2개와 rest 1개가 생성된다.
- slot source에는 `slotIndex`가 보존된다.
- 각 slot의 유리수 tick 범위가 기대값과 일치한다.
- slot note의 `@t(3)`가 `NoteEvent.effects`에 보존된다.

확인한 명령은 다음이다.

```bash
npm run typecheck
npm run test:analyze
npm run test:parse
npm run test:edit
npm run test:score
npm run build
```

## 8. renderer first pass

tuplet group은 `CanvasMarkerItem(kind: "tupletContainer")`로 변환한다.

현재 DTO는 다음 정보만 가진다.

- `rowId`
  - `TupletGroupEvent.headCell.rowId`
- `startTick`
  - group 시작 tick
- `endTick`
  - group 끝 tick
- `divNum`
  - 중앙에 표시할 작은 분할 숫자
- `trackId`

시각 표현은 레거시의 컨테이너 방식을 따르되 bracket은 보류했다.

```text
head row 중심의 21px 기준 점선 사각형
+ 사각형 중앙의 작은 divNum 라벨
```

점선 컨테이너는 일반 note rectangle 위에 올라와야 한다.
반면 gliss line은 note rectangle 뒤에 있어야 한다.
물리 canvas layer를 추가하지 않고, draw pass를 분리해 처리했다.

```text
base layer
marker canvas: gliss / orphan gliss underlay
note canvas: note / mute
note canvas 후처리: tuplet container overlay
```

이 구조는 marker item의 의미를 유지하면서도 canvas DOM layer를 늘리지 않는 절충안이다.
향후 beat/bar/loop처럼 재생 보조선이 늘어나면 marker item의 `underlay` / `overlay` 성격을 명시 타입으로 분리할 수 있다.

레거시처럼 bracket을 함께 그리는 방식은 이번 단계에서 제외했다.
grid 기반 편집 화면에서는 점선 사각형 자체가 group 길이와 편집 좌표를 보여주므로 bracket 정보가 중복될 수 있기 때문이다.
대신 `divNum` 라벨만 작게 두어 `/3`, `/4` 같은 분할 수 정보가 사라지지 않게 했다.

## 9. extend-only container

head cell이 삭제되고 `/&`만 남으면 원래 tuplet head 위치를 알기 어렵다.
따라서 head에 소비되지 않은 pletExtend 연속 구간도 `TupletExtendGroupEvent`로 만든다.

이 이벤트는 정상 `TupletGroupEvent`와 구분된다.

- `eventKind`
  - `tupletExtendGroup`
- `rowId`
  - extend cell이 놓인 row
- `time`
  - 같은 row에서 연속된 `/&` run의 시작과 끝
- `extendCells`
  - 삭제 또는 편집 대상으로 삼을 원본 extend cell 목록

renderer에서는 이 이벤트도 `CanvasMarkerItem(kind: "tupletContainer")`로 변환한다.
다만 head가 없으므로 `divNum`은 `null`이고, 중앙 숫자 라벨은 그리지 않는다.

이 로직은 hold 연결 함수를 직접 재사용하지 않았다.
hold는 display/sound/pitch가 같은 note event를 연결하는 로직이지만, extend-only 표시는 같은 row에서 col이 연속된 `/&` run만 묶으면 되기 때문이다.
따라서 별도의 run-length scan으로 구현했다.

## 10. tuplet slot gliss anchor

tuplet slot 내부 gliss는 일반 cell과 다르게 같은 head cell source를 공유한다.
따라서 renderer가 `source.col`만 사용하면 모든 slot anchor가 같은 x 좌표에 놓이고, S/M/E가 수직선처럼 보인다.

이를 해결하기 위해 `GlissEvent`에 anchor별 시각 중심 tick을 추가했다.

```text
startAnchorTick = center(startAnchor.time)
endAnchorTick = center(endAnchor.time)
```

renderer DTO 변환은 이제 `source.col` 대신 이 중심 tick을 사용한다.
일반 note cell에서는 중심 tick이 `col + 0.5`가 되고, tuplet slot에서는 slot 시간 범위의 중심이 된다.

예를 들어 `/3(...)`가 col 56의 1칸 group이면 slot 중심은 다음과 같다.

```text
slot 0: 56 + 1/6
slot 1: 56 + 1/2
slot 2: 56 + 5/6
```

또한 같은 head cell 안의 여러 slot은 같은 `rowId + col`을 공유하므로, gliss event id와 mid 중복 판정 key에 `slotIndex`를 포함했다.
이로써 `M, M, M`처럼 같은 tuplet head 내부에 여러 mid anchor가 있을 때도 서로 다른 slot이면 독립 anchor로 인식된다.
반대로 일반 cell에서 동일 col/id의 mid가 여러 row에 중복되는 경우에는 기존처럼 하나만 유지한다.

## 11. first-slot based placement

tuplet head는 사용자가 클릭한 row가 아니라 첫 slot의 `@n(midi)` 위치 row에 설치되도록 보정한다.
이 처리가 없으면 같은 열의 여러 row에 동일한 tuplet을 중복 설치할 수 있고, 점선 컨테이너 위치와 실제 저장 head 위치가 어긋난다.

적용 흐름은 다음이다.

```text
tuplet draft
-> composeEditRawText()
-> parseNoteCell(rawText)
-> first slot @n(midi) 추출
-> clicked row의 stringId + first slot midi로 rowId 조회
-> applyRawTextToScore() 대상 hit.rowId 교체
```

즉 col은 사용자가 클릭한 열을 유지하지만, row는 첫 slot 위치로 고정한다.

또한 analyzer는 저장된 head cell row와 별도로 `TupletGroupEvent.containerRowId`를 계산한다.
이 값도 첫 slot `@n(midi)` 기준이다.
따라서 기존 JSON이나 수동 입력으로 head cell이 다른 row에 저장되어 있어도 renderer의 점선 컨테이너는 첫 slot row에 표시된다.

## 12. 테스트 확인

renderer first pass 이후 `dev/test_analyze.ts`에는 다음 검증도 추가되었다.

- `TupletGroupEvent`가 `CanvasMarkerItem(kind: "tupletContainer")`로 변환된다.
- container marker는 head cell row를 사용한다.
- container marker는 group span과 `divNum`을 보존한다.
- head 없이 연속된 `/&`가 `TupletExtendGroupEvent`와 라벨 없는 container marker로 변환된다.
- tuplet slot 내부 S/M/E gliss가 slot 중심 tick 기준 marker로 변환된다.
- head cell 저장 row와 첫 slot row가 다를 때도 container marker가 첫 slot row에 표시된다.
- tuplet 적용 hit가 클릭 row가 아니라 첫 slot row로 보정된다.

확인한 명령은 다음이다.

```bash
npm run typecheck
npm run test:analyze
npm run test:parse
npm run test:edit
npm run test:score
npm run build
```

## 13. 남은 작업

다음 tuplet 작업은 화면 확인 이후 bracket/label 위치나 hit-test 흐름을 조정하는 것이다.

예상 구현 단위는 다음과 같다.

1. tuplet container의 실제 화면 시인성을 확인한다.
2. divNum 라벨이 note text와 과하게 겹치면 위치 또는 배경 처리를 조정한다.
3. tuplet head cell hit-test / slot edit UX와 container marker를 연결한다.
4. bracket은 필요성이 확인될 때 별도 overlay marker로 추가한다.
