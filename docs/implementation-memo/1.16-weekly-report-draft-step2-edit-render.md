# 1.16. 주간 보고서 초안 - edit/render 연결과 유사 오픈소스 비교
> 본 문서는 주간 보고서 작성을 위한 초안이다.
> 제출용 보고서가 아니라, 최근 구현 내용을 보고서 순서로 재구성하고 유사 오픈소스 검토 내용을 함께 정리하기 위한 작업 기록이다.
> 아래 오픈소스/공개 문서 검토는 직접 코드 이식이 아니라 비교 검토와 설계 타당성 확인 용도이다.

## 1. 보고서 작성 방향

이번 보고서는 canvas grid renderer 이후 구현된 기능을 다음 순서로 설명한다.

1. `defaultText`와 `"-"` 시각화
2. EditMode 입력 활성화
3. 악보 영역 좌표 인식 기능
4. JSON I/O 기능 추가
5. 편집 시 full rebuild 기능 추가
6. `"~"`와 modifier 시각화

각 절은 다음 구조로 작성한다.

- 구현 목표
- 핵심 로직
- 유사 오픈소스 또는 공개 구현 검토
- 본 프로젝트에서의 적용 방식

교수님이 강조한 오픈소스 활용 항목은 단순히 라이브러리명을 나열하는 방식으로 쓰지 않는다.
대신 공개 구현의 자료구조, 연결 판정 방식, 렌더링 계층 분리 방식을 확인하고, 본 프로젝트 구조에 맞게 어떤 판단을 했는지 설명한다.

이번 구현은 음악 프로그램이지만, 실제 구현 문제는 아래 분야의 오픈소스와도 유사하다.

- grid editor: cell을 span 단위로 병합하거나 표시 범위로 다루는 문제
- timeline editor: start/end를 가진 event를 화면에 배치하는 문제
- block/flow editor: 연결 가능한 후보를 type/check/handle로 판정하는 문제
- canvas/rendering library: layer 분리, render order, redraw 범위를 관리하는 문제

따라서 보고서에서는 음악 기보 라이브러리뿐 아니라 grid/timeline/block/rendering 계열 오픈소스도 함께 비교 대상으로 둔다.

## 2. `defaultText`와 `"-"` 시각화

### 2.1 구현 목표

첫 번째 목표는 parser가 만든 기본 note 셀을 analyzer가 `NoteEvent`로 바꾸고, renderer가 이를 canvas note rectangle으로 표시하는 것이었다.

처음 시각화 대상은 아래 두 종류였다.

- 일반 `defaultText` note
- `"-"` hold 셀

### 2.2 핵심 로직

기본 흐름은 다음과 같다.

```text
RuntimeDocument
-> buildParsedDocument()
-> analyzeDocument()
-> analyzeTrackEvents()
-> buildCanvasNoteRenderItems()
-> drawScoreNotes()
```

`analyzeTrackEvents()`는 track 내부 parsed note cell을 col, rowId 순으로 순회한다.
일반 note를 만나면 새 `NoteEvent`를 만들고, `"-"` hold를 만나면 바로 왼쪽 tick에 끝난 이전 note와 연결 가능한지 확인한다.

연결 가능 여부는 단순히 rawText가 `"-"`인지로만 판단하지 않는다.
현재 note의 최종 표시 위치와 최종 발음 음정을 기준으로 connection key를 만든다.

```text
display.rowId
+ display.centOffset
+ sound.midi
+ sound.centOffset
```

이 key가 같고, 이전 event의 `endTick`이 현재 cell의 `col`과 같으면 하나의 긴 `NoteEvent`로 병합한다.

병합된 event는 전체 시간 범위는 하나로 유지하되, `displayTextAnchors`에 각 cell의 표시 문자열과 위치를 보존한다.
따라서 긴 note 사각형 위에서도 원래 cell별 텍스트 위치를 확인할 수 있다.

### 2.3 유사 오픈소스 검토

이 부분은 음악 기보 라이브러리의 tie 처리와 비교할 수 있다.

VexFlow의 `StaveTie`는 `first_note`, `last_note`, `first_indices`, `last_indices`를 받아 두 note endpoint 사이에 tie를 그리는 구조이다.
즉 연결할 두 note가 이미 정해져 있고, renderer는 그 관계를 시각화한다.

- 참고: VexFlow `StaveTie`
- URL: https://app.unpkg.com/vexflow@4.2.3/files/src/stavetie.ts
- 활용 범위: tie endpoint를 명시적으로 받아 렌더링하는 구조 검토
- 코드 이식 여부: 없음

MuseScore의 tie 개념도 같은 pitch의 note들을 하나의 연주 단위처럼 이어 주는 방식이다.
이는 본 프로젝트에서 같은 최종 pitch/display 조건의 hold cell만 병합해야 한다는 판단과 연결된다.

- 참고: MuseScore ties handbook
- URL: https://musescore.org/en/handbook/3/ties
- 활용 범위: 같은 pitch note 연결 개념 검토
- 코드 이식 여부: 없음

다만 본 프로젝트는 VexFlow처럼 tie endpoint가 미리 주어지지 않는다.
grid cell의 rawText를 analyzer가 순회하면서 연결 가능한 후보를 찾아야 한다.
따라서 본 프로젝트의 방식은 전통 기보 라이브러리보다 grid/timeline editor의 adjacent block merge와도 가깝다.

React Flow의 edge 모델도 비교 대상으로 삼을 수 있다.
React Flow에서는 edge가 source node와 target node를 가지며, 여러 handle이 있을 때 `sourceHandle`, `targetHandle`로 연결 지점을 지정한다.
이는 본 프로젝트에서 hold 연결 후보를 `display/sound` 기반 connection key로 식별하는 방식과 비교할 수 있다.

- 참고: React Flow Edge type
- URL: https://reactflow.dev/api-reference/types/edge
- 활용 범위: node/handle 기반 연결 관계 표현 방식 비교
- 코드 이식 여부: 없음

### 2.4 본 프로젝트에서의 적용 방식

보고서에서는 이 부분을 다음처럼 설명할 수 있다.

```text
VexFlow와 MuseScore의 tie 처리에서는 연결 대상 note가 이미 정해져 있거나 같은 pitch note를 연결하는 개념이 중심이다.
React Flow의 edge 모델은 연결 대상 node와 handle을 명시적으로 기록한다.
반면 본 프로젝트는 grid cell의 rawText에서 hold token을 분석하여 연결 대상을 찾아야 한다.
따라서 analyzer는 최종 표시 위치와 발음 음정을 connection key로 만들고, 시간적으로 인접한 cell만 하나의 NoteEvent로 병합하는 방식을 사용하였다.
```

## 3. EditMode 입력 활성화

### 3.1 구현 목표

두 번째 목표는 사용자가 UI에서 rawText를 직접 만들고, 선택한 score cell에 입력할 수 있게 하는 것이었다.
이를 통해 테스트케이스 제작을 위한 기본 입력 체계를 마련했다.

현재 EditMode는 다음 입력을 지원한다.

- CUSTOM defaultText
- long hold `-`
- vibrato hold `~`
- gliss modifier `@g(id,kind)`
- trem modifier `@t(div)`
- absolutePitch `@p(midi)`
- microPitch `@m(cent)`
- comment
- eraser

### 3.2 핵심 로직

EditMode 입력은 UI 값을 직접 score에 쓰지 않는다.
먼저 edit composer가 rawText command를 만든다.

```text
UI control state
-> edit command
-> rawText composition
-> applyNoteCellRawText()
-> ScoreFile cell upsert/delete
```

CUSTOM 입력은 사용자가 입력한 문자열을 그대로 UI에 보여 주지만, parser 예약문자는 rawText 저장 시 escape 처리한다.

absolutePitch는 MIDI 번호 직접 입력 대신 계이름 dropdown을 사용한다.
AUTO sharp/flat 표시도 활성화했고, absolutePitch가 있는 경우에는 AUTO 표시 문자열도 absolutePitch를 우선한다.

microPitch 입력은 parser 규칙에 맞춰 `-100..100`, 소수점 이하 최대 1자리로 제한한다.
`@p(0)`과 `@m(0)`은 효과가 없으므로 rawText에 생성하지 않는다.

### 3.3 유사 오픈소스 검토

EditMode의 입력 연결은 block editor의 typed connection 모델과 비교할 수 있다.
Blockly는 block connection에 check를 두어 어떤 block이 어떤 block 뒤에 올 수 있는지 제한한다.
특히 previous/next connection check를 통해 block의 순서와 연결 가능성을 제어한다.

- 참고: Blockly connection check playbook
- URL: https://docs.blockly.com/guides/create-custom-blocks/inputs/connection-check-playbook/
- 활용 범위: 입력 단위의 연결 가능성을 type/check로 제한하는 구조 검토
- 코드 이식 여부: 없음

또한 spreadsheet/grid editor의 merge cell 구조도 참고할 수 있다.
Handsontable의 merge cell 기능은 `{ row, col, rowspan, colspan }` 형태로 병합 범위를 표현한다.
이는 여러 cell을 하나의 표시 단위로 다룬다는 점에서 본 프로젝트의 긴 note span과 유사하다.

- 참고: Handsontable merge cells documentation
- URL: https://handsontable.com/docs/9.0/merge-cells/
- 활용 범위: grid cell 범위를 하나의 표시 단위로 다루는 방식 검토
- 코드 이식 여부: 없음

React Flow의 Handle 문서도 EditMode 입력과 연결 판정 설명에 참고할 수 있다.
React Flow는 edge가 붙는 지점을 handle로 정의하고, 필요하면 `isValidConnection` 같은 유효성 판정 로직을 둔다.
이는 본 프로젝트에서 score cell click, tuplet slot 선택, SELECT ROW 입력처럼 "어떤 UI 위치가 어떤 의미 입력으로 연결되는가"를 분리해 처리하는 방식과 비교할 수 있다.

- 참고: React Flow Handles
- URL: https://reactflow.dev/learn/customization/handles
- 활용 범위: 시각적 연결 지점과 연결 유효성 판정 분리 방식 비교
- 코드 이식 여부: 없음

### 3.4 본 프로젝트에서의 적용 방식

본 프로젝트는 Blockly처럼 사용자가 block을 끌어다 붙이는 구조는 아니다.
그러나 입력 가능한 token 조합을 제한하고, 의미적으로 올바른 rawText command를 만든다는 점에서 typed input model과 유사하다.

보고서에서는 다음처럼 설명할 수 있다.

```text
EditMode 입력은 단순 문자열 입력창이 아니라, 각 UI control의 상태를 parser가 이해할 수 있는 token 조합으로 변환하는 구조이다.
Blockly의 connection check 모델처럼 입력 가능한 조합과 순서를 제한하고, 최종적으로는 parser canonical order에 맞춘 rawText를 생성하도록 구현하였다.
또한 React Flow의 handle/edge 모델처럼, score click이나 slot click 같은 UI 지점을 의미 있는 입력 연결점으로 해석하도록 분리하였다.
```

## 4. 악보 영역 좌표 인식 기능

### 4.1 구현 목표

세 번째 목표는 사용자가 canvas 악보 영역을 클릭했을 때, 해당 화면 좌표를 score file의 행렬 좌표로 변환하는 것이었다.

EditMode 입력은 결국 특정 track, rowId, col에 rawText를 적용해야 한다.
따라서 화면에서 발생한 click 좌표를 아래 score selection 구조로 바꿔야 한다.

```text
MouseEvent
-> ScoreHit(rowId, rowKind, col)
-> ScoreSelection(rowId, rowKind, col, activeTrackId)
```

### 4.2 문법 해설

canvas 내부 click 좌표를 구하는 가장 단순한 방식은 아래와 같다.

```ts
const x = event.clientX - ctx.canvas.offsetLeft;
const y = event.clientY - ctx.canvas.offsetTop;
```

여기서 `event.clientX`, `event.clientY`는 browser viewport 기준의 pointer 좌표이다.
`ctx.canvas.offsetLeft`, `ctx.canvas.offsetTop`은 canvas element의 offset parent 기준 위치이다.
따라서 두 값을 빼면 canvas 내부에서 click한 대략적인 x/y 좌표를 얻을 수 있다.

이렇게 얻은 좌표는 score layout 정보와 비교할 수 있다.

```text
x / columnWidth
-> col

y가 포함되는 layout row
-> rowId, rowKind
```

즉 canvas click 좌표와 score file에서 얻은 layout row/column 정보를 비교하면, 사용자가 악보의 어느 행과 열을 클릭했는지 계산할 수 있다.

### 4.3 실제 구현

실제 구현은 `score_hit_test.ts`의 `hitTestScoreCell()`에 분리했다.

```text
MouseEvent.clientX/clientY
-> score stage getBoundingClientRect()
-> stage 내부 x/y 좌표
-> floor(x / layout.columnWidth)
-> layout.rows에서 y가 포함되는 row 탐색
-> ScoreHit(rowId, rowKind, col)
-> activeTrackId를 붙여 ScoreSelection 생성
```

코드는 `offsetLeft`, `offsetTop` 대신 `getBoundingClientRect()`를 사용한다.

```ts
const rect = stage.getBoundingClientRect();
const x = event.clientX - rect.left;
const y = event.clientY - rect.top;
```

`getBoundingClientRect()`를 사용한 이유는 `event.clientX/clientY`가 viewport 기준 좌표이기 때문이다.
`getBoundingClientRect()`도 viewport 기준의 element 위치와 크기를 반환하므로, 두 값을 빼면 같은 좌표계 안에서 stage 내부 좌표를 계산할 수 있다.

반면 `offsetLeft/offsetTop`은 offset parent 기준 값이다.
부모 element, scroll container, CSS layout, border, transform 등에 따라 viewport 좌표와 바로 대응되지 않을 수 있다.
현재 악보는 scroll container 안에 있고, layout label 영역과 score canvas stage가 분리되어 있으며, zoom도 적용된다.
따라서 viewport 기준 좌표끼리 계산하는 `getBoundingClientRect()` 방식이 더 안정적이다.

### 4.4 layout 정보와 행렬 좌표 변환

이때 renderer가 마지막으로 계산한 `CanvasScoreLayout`을 사용한다.
따라서 zoom, row height, layout row order가 바뀌어도 hit test는 화면에 실제로 그려진 좌표계와 같은 기준을 사용한다.

column은 아래 방식으로 계산한다.

```ts
const col = Math.floor(x / layout.columnWidth);
```

row는 layout rows 중 y 좌표가 포함되는 row를 찾는다.

```ts
const row = layout.rows.find(
  (layoutRow) => layoutRow.y <= y && y < layoutRow.y + layoutRow.height,
);
```

범위를 벗어난 click은 `null`로 처리해 score mutation 대상에서 제외한다.
note row가 아닌 global/gap row click도 `rowKind`를 통해 후속 edit command에서 차단하거나 다른 의미로 해석할 수 있다.

### 4.5 유사 오픈소스 검토

좌표 인식은 canvas/grid editor의 hit testing 문제와 유사하다.
Konva는 stage에서 pointer position을 얻는 API를 제공하고, 각 shape/layer가 stage 좌표계 안에서 상호작용한다.
본 프로젝트는 Konva를 사용하지 않지만, 비슷하게 DOM pointer 좌표를 stage 내부 좌표로 바꾼 뒤 renderer layout 좌표계에서 row/col을 판정한다.

- 참고: Konva Pointer Position
- URL: https://konvajs.org/docs/sandbox/Relative_Pointer_Position.html
- 활용 범위: pointer 좌표를 canvas/stage 좌표계로 변환하는 hit test 개념 비교
- 코드 이식 여부: 없음

Handsontable 같은 grid editor도 pointer interaction을 결국 row/column 좌표로 해석한다.
본 프로젝트는 별도 grid library를 사용하지 않고, score file의 layout definition과 renderer가 만든 `CanvasScoreLayout`을 기준으로 자체 hit test를 구현했다.

### 4.6 본 프로젝트에서의 적용 방식

보고서에서는 이 부분을 다음처럼 설명할 수 있다.

```text
canvas click 좌표는 browser viewport 기준의 MouseEvent 좌표에서 시작한다.
초기 학습 단계에서는 clientX/clientY에서 canvas.offsetLeft/offsetTop을 빼는 방식으로 canvas 내부 좌표를 얻을 수 있음을 확인하였다.
실제 구현에서는 scroll container와 CSS layout에 더 안정적으로 대응하기 위해 getBoundingClientRect()를 사용하였다.
이후 renderer가 계산한 CanvasScoreLayout의 columnWidth와 row y/height 정보를 비교하여 click 위치를 score의 rowId와 col로 변환하였다.
```

## 5. JSON I/O 기능 추가

### 5.1 구현 목표

네 번째 목표는 편집한 score JSON을 파일로 저장하고 다시 불러오는 기능을 연결하는 것이었다.

File panel에는 다음 기능을 추가했다.

- JSON Download
- JSON Load
- Local Save
- Local Load

### 5.2 핵심 로직

JSON I/O는 `src/infra/` 계층으로 분리했다.

```text
score_file_io.ts
- serializeScoreFile()
- downloadScoreJson()
- readTextFile()

score_local_storage.ts
- saveScoreToLocalStorage()
- loadScoreFromLocalStorage()
```

JSON Download는 현재 `state.document.score`를 직렬화해 `Blob`으로 만들고, object URL을 통해 다운로드한다.

JSON Load는 사용자가 선택한 파일의 text를 읽고 `loadRuntimeDocument()`를 호출한다.
로드에 실패하면 현재 score state는 유지한다.

Local Save/Load는 localStorage를 사용한다.
저장 key는 `regression-code:score-json`이다.

### 5.3 공개 API 문서 검토

이 기능은 특정 오픈소스 코드보다 Web API 공식 문서와 더 직접적으로 관련된다.

MDN Blob 문서는 browser에서 파일-like object를 만들고, `Blob.text()`나 `FileReader` 등으로 내용을 읽는 방식을 설명한다.
본 프로젝트는 JSON 다운로드 시 `Blob`을 사용하고, 파일 load에서는 `File.text()`를 사용한다.

- 참고: MDN Blob
- URL: https://developer.mozilla.org/en-US/docs/Web/API/Blob/
- 활용 범위: JSON 다운로드용 Blob 사용 방식 확인
- 코드 이식 여부: 없음

MDN Web Storage API 문서는 localStorage가 key/value 문자열 저장소이며, `setItem()`과 `getItem()` 사용을 권장한다고 설명한다.
본 프로젝트의 Local Save/Load는 이 방식을 사용한다.

- 참고: MDN Using the Web Storage API
- URL: https://github.com/mdn/content/blob/main/files/en-us/web/api/web_storage_api/using_the_web_storage_api/index.md
- 활용 범위: localStorage 저장/불러오기 방식 확인
- 코드 이식 여부: 없음

### 5.4 본 프로젝트에서의 적용 방식

보고서에서는 JSON I/O를 다음처럼 설명할 수 있다.

```text
JSON I/O는 app controller가 직접 브라우저 저장 API를 다루지 않도록 infra 계층으로 분리하였다.
파일 다운로드는 Blob 기반 object URL을 사용하고, localStorage 저장은 Web Storage API의 setItem/getItem 구조를 사용하였다.
로드 시에는 단순 JSON.parse에 그치지 않고 loadRuntimeDocument()를 통해 구조 검증과 index 생성을 다시 수행하도록 하였다.
```

## 6. 편집 시 full rebuild 기능 추가

### 6.1 구현 목표

다섯 번째 목표는 EditMode에서 score cell이 바뀔 때 화면이 즉시 최신 분석 결과를 반영하도록 만드는 것이었다.

초기 구현에서는 partial update보다 정확성과 단순성을 우선했다.
따라서 편집이 발생하면 현재 score 전체를 다시 runtime document, parser, analyzer, renderer 입력으로 재생성한다.

### 6.2 핵심 로직

편집 후 rebuild 흐름은 다음과 같다.

```text
applyRawTextToScore()
-> applyNoteCellRawText()
-> createRuntimeDocument()
-> buildParsedDocument()
-> analyzeDocument()
-> buildCanvasNoteRenderItems()
-> renderCanvasScore()
```

이 방식은 대형 score에서는 비용이 커질 수 있다.
하지만 현재 단계에서는 parser/analyzer/render 경계가 아직 확장 중이므로, full rebuild가 디버깅과 검증에 유리하다.

특히 테스트케이스 제작 단계에서는 "입력한 rawText가 실제로 score JSON에 저장되었는가", "parser가 이를 올바르게 읽는가", "analyzer가 event를 만드는가", "renderer가 표시하는가"를 한 번에 확인할 수 있다.

### 6.3 유사 오픈소스 검토

full rebuild 전략은 많은 UI framework의 상태 기반 렌더링과 유사하다.
상태가 바뀌면 view를 다시 계산하고, 실제 DOM 또는 canvas 반영은 renderer가 담당한다.

또한 calendar/timeline 계열 오픈소스는 이벤트를 `start`, `end` 같은 시간 범위 객체로 다루고, 이를 화면에 다시 배치한다.
FullCalendar의 Event Object도 event의 start/end를 핵심 속성으로 사용한다.
이는 본 프로젝트의 `NoteEvent.time.startTick/endTick` 구조와 비교할 수 있다.

- 참고: FullCalendar Event Object
- URL: https://fullcalendar.io/docs/event_data/Event_Object
- 활용 범위: 시간 범위 event를 view item으로 다루는 구조 비교
- 코드 이식 여부: 없음

렌더링 계층 분리 측면에서는 Konva의 layer 관리 문서를 참고할 수 있다.
Konva는 stage 안에 여러 layer를 두며, DOM을 보면 layer마다 canvas element가 존재한다고 설명한다.
또한 성능 관점에서 layer 수를 관리해야 한다고 설명한다.
본 프로젝트도 layout/base/note/marker canvas layer를 분리하되, 각 layer의 책임을 제한하는 방향으로 구현했다.

- 참고: Konva Layer Management
- URL: https://konvajs.org/docs/performance/Layer_Management.html
- 활용 범위: canvas layer 분리와 성능 고려 방식 비교
- 코드 이식 여부: 없음

PixiJS의 Container 문서도 render order와 scene hierarchy를 설명하는 참고 자료로 사용할 수 있다.
PixiJS는 Container가 scene graph의 기반이며 `zIndex`, `sortableChildren` 등으로 render order를 제어할 수 있다고 설명한다.
본 프로젝트는 PixiJS를 사용하지는 않지만, canvas layer와 draw 순서를 나누는 판단에 이와 유사한 관점을 적용했다.

- 참고: PixiJS Container
- URL: https://pixijs.com/8.x/guides/components/scene-objects/container
- 활용 범위: render order와 layer/object hierarchy 비교
- 코드 이식 여부: 없음

### 6.4 본 프로젝트에서의 적용 방식

보고서에서는 full rebuild를 다음처럼 설명할 수 있다.

```text
초기 단계에서는 부분 갱신보다 전체 재생성을 선택하였다.
사용자가 cell을 편집하면 score JSON을 수정한 뒤 runtime document, parsed document, analysis result, renderer item을 모두 다시 만든다.
이 방식은 성능 최적화 측면에서는 후속 개선 여지가 있지만, 현재처럼 parser/analyzer/renderer 경계를 검증하는 단계에서는 가장 명확하고 오류 추적이 쉬운 구조이다.
renderer는 Konva나 PixiJS처럼 별도 rendering framework를 도입하지 않았지만, 공개 구현에서 확인할 수 있는 layer/order 분리 원칙을 단순한 HTML canvas 다층 구조로 구현하였다.
```

## 7. `"~"`와 modifier 시각화

### 7.1 구현 목표

여섯 번째 목표는 default note와 `"-"` hold 이후, gliss와 tuplet을 제외한 주요 token을 analyzer-renderer 경로에 연결하는 것이었다.

현재 연결된 token은 다음과 같다.

- `"~"` vibrato hold
- `@t(div)` tremolo
- `@p(midi)` absolutePitch
- `@m(cent)` microPitch

### 7.2 핵심 로직

`@p`는 최종 발음 MIDI를 바꾼다.
renderer의 note fill 색상도 `event.sound.midi`를 기준으로 결정한다.
따라서 사용자는 실제 연주될 음을 색으로 확인할 수 있다.

`@m`은 최종 발음 cent offset과 표시 cent offset에 모두 반영한다.
renderer는 display cent offset을 y 좌표로 보간한다.
`+100cent`는 위쪽 note row 중앙, `-100cent`는 아래쪽 note row 중앙과 일치한다.

`@t`는 `NoteEffectSegment`의 trem 정보로 기록한다.
renderer는 이를 chop line으로 표시한다.
chop line 색은 note row background와 맞춰, 사각형이 끊기는 느낌을 주도록 했다.

`"~"`는 hold와 같은 연결 규칙을 사용하되, effect segment에는 vibrato로 기록한다.
renderer는 연속된 vib segment를 하나의 draw range로 병합하고, 칸당 1주기 기준의 sine wave를 하나의 path로 그린다.

### 7.3 유사 오픈소스 검토

modifier 시각화는 음악 기보 라이브러리보다 timeline editor, block editor, grid merge, connected tile rendering과 더 비슷한 측면이 있다.

`@p`, `@m`이 포함된 hold 연결은 단순히 같은 row의 block을 이어 붙이는 것이 아니다.
최종 표시 위치와 최종 발음 음정이 같아야 연결된다.
이는 Blockly connection check처럼 연결 가능한 후보에 type/check 조건을 두는 방식과 비교할 수 있다.

- 참고: Blockly connection check playbook
- URL: https://docs.blockly.com/guides/create-custom-blocks/inputs/connection-check-playbook/
- 활용 범위: 연결 가능성을 type/check로 제한하는 모델 비교
- 코드 이식 여부: 없음

연속된 `"~"` segment를 하나의 sine path로 병합하는 방식은 connected tile 또는 autotiling 계열과도 비교할 수 있다.
인접한 동일 속성 block을 하나의 시각적 연속체처럼 표시한다는 점에서 유사하다.
다만 본 프로젝트는 tile sprite를 교체하는 것이 아니라, 연속 tick range를 하나의 path로 그린다.

- 참고: Tiled map editor
- URL: https://www.mapeditor.org/
- 활용 범위: grid 기반 object/tile editing과 인접 요소 시각화 개념 비교
- 코드 이식 여부: 없음

렌더링 표현 자체는 canvas layer와 draw order 관리 문제이기도 하다.
Konva는 stage/layer 구조를 통해 서로 다른 그리기 대상을 별도 canvas layer로 분리한다.
본 프로젝트도 grid는 base layer, note rectangle과 effect는 note layer, 박자선/마디선/재생 marker는 marker layer로 분리하는 방향을 유지한다.
이는 trem/vib/gliss 같은 note 의미 효과가 marker layer가 아니라 note layer에 속해야 한다는 판단과도 연결된다.

- 참고: Konva Framework Overview
- URL: https://konvajs.org/docs/overview.html
- 활용 범위: stage/layer 기반 canvas 구조 비교
- 코드 이식 여부: 없음

### 7.4 본 프로젝트에서의 적용 방식

보고서에서는 이 부분을 다음처럼 설명할 수 있다.

```text
modifier 시각화는 단순히 문자열을 화면에 표시하는 작업이 아니라, token을 최종 발음 정보와 표시 위치, effect segment로 해석하는 작업이다.
특히 @p와 @m은 hold 연결 조건에도 영향을 주므로, analyzer는 최종 sound/display 정보를 기준으로 연결 key를 만든다.
이는 block editor의 connection check처럼 의미적으로 연결 가능한 후보만 이어 붙이는 방식이다.
```

```text
vibrato는 각 cell마다 wave를 따로 그리지 않고, 연속된 effect segment를 하나의 draw range로 병합해 하나의 path로 그린다.
이는 grid 기반 editor에서 인접한 동일 속성 요소를 하나의 연속된 시각 단위로 다루는 방식과 유사하다.
또한 canvas layer 기반 렌더링 오픈소스에서 확인할 수 있는 것처럼, 효과별 draw order를 명확히 나누어 rectangle, effect, text가 서로 덮이는 순서를 제어하였다.
```

## 8. 구현 난이도 정리

현재까지의 구현 난이도는 특정 고난도 수학 알고리즘보다는, 여러 계층의 경계를 맞추는 데 있었다.

주요 난점은 다음과 같다.

- Google Sheets 기반 legacy 구조를 TypeScript web app 구조로 분리하는 것
- rawText 입력, score JSON 저장, parser, analyzer, renderer를 하나의 검증 루프로 연결하는 것
- hold token을 단순 문자열이 아니라 의미적으로 같은 note span으로 병합하는 것
- `@p`, `@m`, `@t`, `"~"`를 각각 sound/display/effect 구조로 해석하는 것
- canvas 확대율, row 높이, note rectangle, microPitch 보간, text 크기, effect line을 서로 어긋나지 않게 조정하는 것
- renderer가 analyzer 결과를 직접 해석하지 않도록 renderer-owned DTO 경계를 유지하는 것
- layout/base/note/marker layer의 책임과 draw order를 분리하는 것
- 음악 기보 구현뿐 아니라 grid/timeline/block/rendering 오픈소스 구조와 비교하여 현재 구현을 설명 가능한 형태로 정리하는 것

보고서에서는 "난이도가 높은 기능"을 과장하기보다, 복잡한 상태 흐름과 계층 분리를 안정화한 점을 강조하는 것이 적절하다.

## 9. 현재 한계와 다음 작업

현재 한계는 다음과 같다.

- tuplet 문자열 입력은 가능하지만 analyzer/render 연결은 아직 없다.
- gliss 입력은 가능하지만 analyzer/render 연결은 아직 없다.
- central player group은 실제 score metadata와 아직 완전히 연결되지 않았다.
- Details dialog의 정보 수정 기능은 아직 활성화되지 않았다.
- audio generator와 playback controller는 아직 본격 구현 전이다.
- edit mutation은 full rebuild 방식이므로, 대형 score에서는 추후 partial update가 필요하다.

다음 작업은 아래 순서가 적절하다.

1. 현재 edit/analyze/render 경로를 JSON download/load round trip으로 확인한다.
2. tuplet analyzer와 renderer 연결을 구현한다.
3. gliss analyzer와 renderer 연결을 구현한다.
4. 중앙 player group을 실제 score 정보와 연결한다.
5. Details dialog에서 score metadata를 수정할 수 있게 한다.
6. audio generator/playback controller 경계를 설계한다.

## 11. 보고서용 요약 문장

이번 구현에서는 `회귀 코드`의 입력-분석-렌더링 검증 루프를 구축하였다.
사용자가 EditMode에서 입력한 rawText는 score JSON에 저장되고, parser와 analyzer를 거쳐 renderer 전용 DTO로 변환된 뒤 canvas에 표시된다.
초기에는 `defaultText`와 `-` hold를 하나의 긴 `NoteEvent`로 병합하는 경로를 만들었고, 이후 JSON I/O와 full rebuild를 연결해 편집 결과를 즉시 확인할 수 있게 했다.
또한 `"~"`, `@t`, `@p`, `@m` modifier를 analyzer-renderer 경로에 연결하여 vibrato, tremolo, absolute pitch, micro pitch를 시각적으로 확인할 수 있게 했다.
구현 과정에서는 VexFlow, MuseScore, Blockly, React Flow, Handsontable, FullCalendar, Tiled, Konva, PixiJS 등 유사 오픈소스와 공개 문서를 검토하여 note 연결, block connection, grid merge, timeline event span, canvas layer/render order 처리 방식을 비교했고, 이를 본 프로젝트의 grid 기반 악보 구조와 HTML canvas 다층 렌더링 구조에 맞게 재구성하였다.
