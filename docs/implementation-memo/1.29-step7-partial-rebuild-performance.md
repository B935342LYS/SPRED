# 1.29. Step7 partial rebuild 성능 개선 메모

## 1. 목적

이 메모는 YouTube 모드 안정화 이후 긴 악보 편집 지연을 줄이기 위해 진행한 partial rebuild와 drag input 성능 개선 내용을 기록한다.

기준 문서:

- `docs/2.8-edit-invalidation-and-partial-rebuild-spec.md`
- `docs/2.1-canvas-renderer-module-spec.md`

이 문서는 구현 해설용 메모이다.
저장 포맷, parser 단일 셀 문법, analyzer 공식 이벤트 계약은 active spec 문서를 따른다.

## 2. 계측 결과

초기 계측은 `applyScoreTextEdits()` 경계에서 다음 구간을 나눠 확인했다.

```text
busy ui
apply raw text + rebuild artifacts
partial plan
render input patch
render + ui sync
playback reset
```

긴 악보에서 최대 병목은 처음에 `apply raw text + rebuild artifacts`로 관찰되었다.
후속 내부 계측에서는 해당 구간을 다시 나눴다.

```text
apply score cell rawText batch
create runtime document + indexes
create render base input
partial artifacts
full artifacts fallback
```

최적화 후 artifact 내부 구간은 0.1ms에서 0.7ms 수준으로 내려갔다.
따라서 이후 체감 지연은 artifact 생성 자체보다 drag 중 pointermove마다 rebuild/render/playback reset이 누적되는 구조가 더 큰 후보로 판단했다.

## 3. ScoreFile clone 최적화

기존 `applyScoreCellRawTextBatch()`는 cell 하나를 바꿔도 `JSON.stringify()` / `JSON.parse()`로 전체 `ScoreFile`을 deep clone했다.

현재 구현은 score text edit에 필요한 부분만 복제한다.

- note edit: edited track의 `cells` 배열만 복제
- global edit: `globalLines.cells` 배열만 복제
- mixed edit: global cell 배열과 edited track cell 배열을 함께 복제

이 방식은 `ScoreFile` 최상위 객체는 새로 만들지만, 변경되지 않은 metadata, layout, instrument data, unedited track 객체는 구조적으로 공유한다.
편집 함수는 복제된 `cells` 배열만 교체하므로 이전 `AppState`의 score 원본은 변경하지 않는다.

## 4. Parsed document 재사용

기존 partial artifact 경로는 analyzer를 edited track 중심으로 줄였더라도 `buildParsedDocument()`로 전체 track/global parser 결과를 다시 만들었다.

현재 구현은 `buildParsedTrackCells()`와 `buildParsedGlobalCells()`를 partial artifact에서 재사용할 수 있게 열고, 이전 `ParsedScoreDocument`를 group 단위로 재사용한다.

note edit:

```text
previous ParsedScoreDocument
-> noteCellsByTrackAndCol Map shallow copy
-> edited track parsed map만 교체
-> globalCellsByKindAndCol 재사용
```

global edit:

```text
previous ParsedScoreDocument
-> noteCellsByTrackAndCol 재사용
-> globalCellsByKindAndCol만 재생성
```

이 단계는 analyzer event 구조나 parser 결과 타입을 바꾸지 않는다.

## 5. Renderer partial 안정화

Renderer input patch는 source event id를 기준으로 note/mute item을 교체한다.
다만 `noteMarkerItems`는 전체 group으로 교체한다.

이유:

- orphan gliss marker는 특정 `GlissEvent` 하나가 아니라 주변 note anchor의 연결 상태에서 파생된다.
- source event id 단위 patch를 적용하면 변경 이벤트에 직접 속하지 않는 orphan marker가 남거나 누락될 수 있다.

물리 redraw에서는 note row만 dirty tick range를 사용한다.
`noteMarker` canvas는 gliss 연결선의 기울기 변경과 orphan marker 잔상을 피하기 위해 note edit 시 전체 redraw한다.

note layer dirty redraw는 다음 보정을 둔다.

- row height가 note rectangle보다 작아도 note rectangle 전체를 지우도록 y 범위 확장
- display text가 사각형 밖으로 넘쳐도 잘리지 않도록 x clear padding과 redraw 후보 tick 범위를 함께 확장

## 6. Drag input batching

Drag edit은 pointermove마다 즉시 `applyScoreTextEdits()`를 호출하면 작은 편집이 프레임 안에 여러 번 누적될 수 있다.

현재 구현은 drag 중 생성된 `ScoreTextEdit[]`를 `requestAnimationFrame` 단위로 모아 적용한다.

```text
pointermove
-> addDragEditForHit()
-> active track edit 확장
-> pendingDragEdits Map에 key별 저장
-> requestAnimationFrame에서 applyScoreTextEdits()
```

중복 key는 마지막 edit으로 덮어쓴다.
pointerup 또는 pointercancel에서는 pending edit을 즉시 flush해 마지막 셀이 누락되지 않게 한다.

단일 클릭 편집은 기존처럼 즉시 적용한다.

## 7. Dynamic viewport canvas

추가 계측에서 edit 처리 자체는 2ms에서 10ms 안쪽이었지만, 4000열 악보에서는 다음 paint까지 70ms에서 300ms 수준의 지연이 발생했다.
`requestAnimationFrame` 기준으로 보면 sync work 이후 첫 paint 전 대기 시간이 커졌고, `Long Task`는 관찰되지 않았다.
동적 canvas draw를 건너뛰는 실험에서 frame 지연이 크게 줄었으므로 병목은 JavaScript 계산보다 browser가 큰 bitmap canvas를 raster/commit/composite하는 비용으로 판단했다.

현재 구현은 partial render 경로에서 `note`와 `noteMarker` 동적 layer를 viewport 폭 bitmap으로 줄인다.

- `CanvasRenderOptions.dynamicViewport`는 현재 scrollLeft, viewport width, overscan px를 전달한다.
- `resizeCanvasLayerToDynamicViewport()`는 canvas CSS 위치를 실제 score x 좌표로 옮기고, context transform에 viewport 시작 x를 반영한다.
- note edit partial render는 `note`와 `noteMarker`를 viewport bitmap으로 다시 그리고, dynamic viewport 경로에서는 dirty range 대신 visible group 전체를 다시 그린다.
- global edit partial render는 global marker를 갱신하고, global rawText가 note layer에 함께 그려지는 구조 때문에 visible note layer를 다시 합성한다.
- horizontal/vertical scroll은 `requestAnimationFrame`에서 dynamic viewport layer를 다시 그려 viewport 밖 item이 잘못 보이는 문제를 막는다.

이 변경 뒤 1000열과 4000열 악보의 drag input 체감 차이는 사라졌다.
다만 base/layout grid와 global marker layer는 아직 full stage canvas이므로, 더 큰 악보나 저성능 환경에서는 tile 또는 chunk cache가 다음 후보로 남는다.

## 8. 남은 병목 후보

현재 ordinary rawText edit에서 남은 후보는 다음이다.

- `createRuntimeDocument()`가 여전히 `ScoreIndexes` 전체를 재생성한다.
- audio schedule과 playback runtime은 아직 full rebuild/reset이다.
- base/layout grid와 global marker canvas는 여전히 전체 stage width를 유지한다.
- viewport dynamic layer 이후에도 8000열 이상 악보에서는 tile rendering 또는 chunk cache가 필요할 수 있다.

후속 순서:

1. runtime index partial update 가능성 검토
2. base/layout/global marker tile 또는 chunk cache 도입
3. dynamic viewport overscan 및 scroll redraw 정책 조정
4. audio schedule partial update 검토
5. analyzer range partial은 hold/gliss/tuplet 범위 확장 규칙이 더 안정된 뒤 진행
