## 1. 문제 상황

실제 프렛리스 현악기 연주에서 기본음보다 배음이 강하게 입력되면서 `pitchy`가 잘못된 주기성을 선택하는 문제가 확인됐습니다.

대표 사례는 다음과 같습니다.

- 실제 연주: A#3
- 검출 결과: A#4
- 즉, 2배음이 기본음으로 오인됨
- 최종 실험용 글리산도 WAV에서는 `2x/0.5x`뿐 아니라 `3.04x/0.33x`에 가까운 3배음 계열 점프도 반복적으로 관찰됨

따라서 문제는 단순히 “한 옥타브 높게 잡힌 값을 낮춘다”가 아니라, 마이크 입력의 배음 구조에서 detector가 잘못된 주기 후보를 선택하는 문제로 정의되어 있습니다.

## 2. 이미 시도하고 롤백한 방법

구현 메모 1.54에는 제품 경로에 후처리 보정을 추가했던 과정과 실패 이유가 정리되어 있습니다.

시도했던 방법은 다음과 같습니다.

- `f / 2`의 기본음·배음 에너지를 확인하는 harmonic verifier
- low-pass된 입력을 다시 검출하는 verifier
- `f`, `f/2`, `f/3`, `f/4`의 배음 에너지를 비교하는 저음 fundamental 후보 선택
- layout 범위 안에서 일정 에너지 이상을 가진 가장 낮은 주파수를 찾는 scan

합성 신호 테스트와 `test:game`, 빌드는 통과했지만 실제 마이크에서는 다음 부작용이 발생했습니다.

- A#3 → A#4 문제를 충분히 해결하지 못함
- 실제 고음도 잘못 저음으로 내리는 현상 발생
- 일부 음정이 불안정해지거나 검출되지 않음
- 단일 프레임 spectrum만으로 실제 고음과 저음의 배음을 완전히 구분하기 어려움
- 합성 테스트가 실제 마이크·악기·공간의 입력 특성을 대표하지 못함

그래서 이 보정들은 모두 제품 코드에서 제거되고, 현재 제품 경로는 다시 `pitchy` 원시 검출을 기준선으로 사용합니다. target 음정에 맞춘 옥타브 보정이나 hysteresis lock도 재도입하지 않는다는 판단입니다.

## 3. 현재 확정된 실험 원칙

3.4 명세가 Practice 피치 검출에 관해서는 3.2보다 우선합니다.

핵심 원칙은 다음과 같습니다.

- 새 detector를 곧바로 제품 코드에 교체하지 않음
- 같은 실제 WAV 입력을 여러 backend에 넣어 비교
- backend마다 다른 confidence 의미와 threshold를 분리
- 전처리, 후처리, resampling은 adapter 책임으로 제한
- `GamePitchFrame`, judge, renderer 계약은 유지
- 실패한 backend는 adapter 단위로 제거 가능해야 함
- target 음정이나 직전 프레임에 맞추어 결과를 강제하지 않음
- 배포 라이선스와 브라우저 실행 가능성까지 채택 기준에 포함
- 명확한 승자가 없으면 `pitchy` 제품 경로를 그대로 유지

즉, 현재 목표는 “무조건 다른 detector로 교체”가 아니라 “실제 입력에서 더 나은 backend가 있는지를 재현 가능하게 검증”하는 것입니다.

## 4. 현재 구현된 실험 기반

현재 브랜치는 `experiment/practice-pitch-detection`이며, 작업 트리는 아직 커밋되지 않은 실험 변경을 포함하고 있습니다.

완료된 기반은 다음과 같습니다.

- `GamePitchDetectorBackend` 공통 인터페이스
- 기본 `pitchy` backend adapter
- backend ID가 포함된 debug frame
- 동기·비동기 detector를 모두 처리하는 단일 실행 경로
- backend 오류·dispose·이전 비동기 결과 무효화 처리
- `?pitchDebug=1` 기반 디버그 도구
- 실제 detector 입력과 동일한 mono PCM을 저장하는 AudioWorklet
- 최대 60초 캡처
- PCM16 WAV 및 입력 metadata JSON 내보내기
- WAV decoder와 offline comparison runner
- frame timestamp, F0, confidence, RMS, gate 통과 여부, 처리 시간 기록
- `pitchy` WAV 비교 경로
- SwiftF0 ONNX offline contour backend
- Essentia.js pYIN offline contour backend

현재 의존성에는 `pitchy`, `essentia.js`, `onnxruntime-web`이 들어가 있으며 SwiftF0 모델도 로컬 public 경로에 준비되어 있습니다.

## 5. 공통 입력 검증 상태

최종 비교용 캡처는 explicit mono downmix가 적용된 `2026-07-11T11-55-28-143Z` WAV입니다.

확인된 정보는 다음과 같습니다.

- 48 kHz
- mono
- 1,309,952 samples
- 약 27.291초
- WAV header, metadata, comparison 입력 길이가 서로 일치
- echo cancellation, noise suppression, auto gain control 비활성화
- detector와 capture가 동일한 명시적 1채널 downmix를 사용

`pitchy` 기준 결과는 다음과 같습니다.

- 전체 2,551 frames
- confidence와 RMS gate를 통과한 frame 1,565개
- 평균 처리 시간 약 `0.583 ms/frame`

따라서 공통 비교 입력과 기준선 측정은 이미 확보된 상태입니다.

## 6. backend 진행 상황

현재 후보별 상태는 다음과 같습니다.

- `pitchy`: 제품 기준선 및 offline 비교 완료
- SwiftF0: ONNX Runtime Web 기반 offline adapter 구현 완료
- Essentia.js pYIN: offline contour adapter 구현 완료
- CREPE: 아직 미구현

Essentia.js는 라이선스 때문에 현재 단계에서는 비교 실험용 성격이 강합니다. 실제 제품 의존성 채택 여부는 성능과 별개로 배포 라이선스 검토가 필요합니다.

## 7. 바로 이어갈 작업

문서가 지정한 다음 작업은 명확합니다.

1. 최종 `143Z` WAV를 `pickAndComparePyinWav()`로 분석
2. pYIN 결과 JSON 생성
3. 동일 timestamp 기준으로 `pitchy`, SwiftF0, pYIN 결과 정렬
4. 다음 항목 비교

   - voiced coverage
   - 5 semitone 이상 large jump
   - 2배·3배 배음 오인식 구간
   - A#3와 A#4 구분
   - gliss와 vibrato 추적 안정성
   - 처리 시간 및 브라우저 적용 가능성

5. 결과가 의미 있게 좋아질 때만 실시간 제품 backend 적용을 검토

정리하면, 현재는 실패한 휴리스틱 보정들을 정리하고 실측 WAV·공통 backend 계약·비교 도구까지 갖춘 상태입니다. 다음 단계는 새로운 보정 코드 작성이 아니라, 준비된 최종 WAV에서 pYIN 결과를 생성하여 기존 `pitchy`와 SwiftF0 결과를 정량 비교하는 것입니다.

이번에는 현황 확인만 수행했으며 파일 변경이나 테스트 실행은 하지 않았습니다.

네. 현재는 CREPE와 aubio 구현을 미뤄도 괜찮습니다.

우선 이미 확보된 `pitchy`, pYIN, SwiftF0 결과를 동일 timestamp 기준으로 분석하는 편이 맞습니다. 이 세 backend만으로도 다음 판단이 가능합니다.

- 배음 오류가 실제로 줄어드는지
- voiced coverage와 gliss 추적이 개선되는지
- 정확도와 처리 비용 사이에 유의미한 우열이 있는지
- 현재 backend 후보군 자체가 충분한지

CREPE와 aubio는 다음 조건에서만 추가하는 것이 효율적입니다.

- pYIN과 SwiftF0 모두 `pitchy`보다 뚜렷하게 낫지 않은 경우
- 한 backend가 배음 오류는 줄이지만 coverage·gliss·latency가 부족한 경우
- DSP 계열의 실시간 대안이 추가로 필요하면 aubio
- DSP 후보들이 공통적으로 octave/배음 오류를 해결하지 못하면 CREPE

우선순위를 정하면 다음과 같습니다.

1. 기존 세 JSON 정량 비교
2. 후보가 있으면 실제 브라우저 실시간 적용 가능성 검증
3. 결론이 불충분하면 aubio
4. DSP 계열의 한계가 확인되면 CREPE

특히 CREPE는 모델 크기, 로딩, 추론 비용과 브라우저 통합 부담이 크므로 지금 바로 구현할 이유가 약합니다. aubio도 pYIN과 같은 계열의 DSP 후보를 이미 확보한 상황이므로 중복 실험이 될 가능성이 있습니다.

따라서 두 backend는 폐기하는 것이 아니라, 기존 세 결과로 결론을 내릴 수 없을 때 여는 2차 후보로 보류하는 것이 적절합니다.

## backend별 판단

### pitchy

장점:

- 가장 빠름
- coverage가 충분함
- 현재 실시간 제품 경로가 검증되어 있음

단점:

- 18회의 큰 점프 중 14회가 octave형

기준선으로는 적합하지만, 현재 문제를 그대로 가지고 있습니다.

### pYIN

장점:

- pitchy와 거의 동일한 coverage
- 5 semitone 이상 인접 점프가 0회
- 확인된 pitchy 배음 오류 구간을 안정적으로 낮은 기본음으로 추적
- 세 backend 동시 비교에서 독자적인 큰 이탈이 발견되지 않음

주의점:

- 현재 offline 구현은 27.291초 입력 처리에 38.763초가 걸려 약 `1.42×`
- 그대로는 실시간 backend가 될 수 없음
- contour 알고리즘의 smoothing이 빠른 gliss를 지연시키는지 별도 측정 필요
- Essentia.js 라이선스 때문에 제품 배포 의존성으로 채택할 수 있는지도 별도 판단 필요

현재 결과만 보면 정확도 후보 1순위입니다.

### SwiftF0

장점:

- pitchy의 2배음 오류를 pYIN과 함께 바로잡는 구간이 많음
- 처리 속도는 충분히 빠름
- 큰 인접 점프가 1회뿐임

단점:

- coverage가 46.1%로 다른 두 backend보다 약 15%p 낮음

- 실제 Practice 모드에서는 음정점이 자주 사라질 가능성이 있음

보조 비교군으로는 가치가 있지만, 현재 설정 그대로 기본 backend로 채택하기는 어렵습니다.

## 잠정 결론

현재 우선순위는 다음과 같습니다.

1. 정확도: pYIN
2. 실시간성과 기존 안정성: pitchy
3. 처리 속도가 빠른 ML 비교 후보: SwiftF0

- 브라우저 실시간 처리 및 배포 라이선스 조건을 만족할 수 있는가

이번 분석은 정답 음정 라벨이 없는 글리산도 녹음을 대상으로 했습니다. 따라서 pYIN을 절대적인 정답으로 확정한 것은 아니지만, 배음 오류 구간에서 두 독립 backend가 합의하고 pitchy만 이탈한 결과는 pYIN 쪽에 상당히 강한 근거를 제공합니다. 파일은 변경하지 않았습니다.

## 1. 글리산도 지연 측정

### 1단계: 기존 JSON으로 offline 지연 계산

먼저 추가 녹음 없이 `143Z`의 세 JSON을 이용합니다.

각 backend contour를 cents 단위로 변환한 뒤 다음 방법으로 비교합니다.

- pitchy–pYIN
- SwiftF0–pYIN
- pitchy–SwiftF0

정상 글리산도 구간만 대상으로 한쪽 contour를 시간축에서 앞뒤로 이동하면서 오차가 가장 작아지는 offset을 구합니다.

예를 들어 pYIN을 80ms 앞으로 옮겼을 때 pitchy와 가장 잘 맞는다면, pYIN contour에 약 80ms의 상대 지연이 있다고 해석할 수 있습니다.

함께 계산할 지표:

- 전체 글리산도 최적 time offset
- 상승 구간별 time offset
- 특정 음정 통과 시각 차이
- 50cent, 100cent 경계 통과 시각 차이
- 빠른 구간의 기울기 추종 오차
- vibrato 진폭 감소와 위상 지연
- 배음 오류 구간을 제외한 중앙값과 90백분위 지연

중요한 제한은 pitchy도 절대적인 정답 시각이 아니라는 점입니다. 따라서 이것은 상대 지연 측정입니다.

### 2단계: 정답 시각이 있는 테스트 입력 제작

상대 비교만으로 부족하면 정답 contour가 명확한 WAV를 만듭니다.

권장 corpus:

- 200Hz → 400Hz 선형 gliss: 2초
- 200Hz → 400Hz 선형 gliss: 500ms
- 400Hz → 200Hz 하강 gliss
- 일정음 250Hz 후 500Hz로 즉시 전환
- 250Hz 중심 ±50cent vibrato, 5Hz
- 기본음보다 2배음이 강한 합성 gliss

이 WAV는 각 sample의 기대 주파수를 알고 있으므로 다음을 직접 계산할 수 있습니다.

- 알고리즘 지연
- cents 오차
- gliss 기울기 손실
- step 응답 시간
- vibrato 위상 지연
- octave 오류율

다만 합성 신호 결과는 실제 악기 결과를 대체하지 않고, latency 원인을 분리하는 보조 자료로 사용해야 합니다.

### 3단계: 실제 연주에 기준 신호 추가

가장 정확한 실사용 측정은 녹음 시작과 동시에 클릭이나 전기적 기준음을 함께 기록하는 것입니다.

가능하다면 다음과 같이 구성합니다.

- 일정음 연주
- 메트로놈 클릭 시점에 gliss 시작
- 지정된 박자에 목표 음정 도달
- 동일 연주를 여러 번 반복

그 후 수동 라벨이나 별도 고정밀 offline 분석 결과를 기준 contour로 둡니다.

## 2. 실시간 처리 가능성 확인

현재 pYIN adapter는 실시간 adapter가 아닙니다. 전체 신호를 한 번에 분석하고, 모든 frame 후보를 모은 뒤 HMM smoothing을 수행합니다. Essentia 공식 설명도 `PitchYinProbabilistic`이 frame별 후보를 모은 다음 HMM으로 최종 track을 만든다고 명시합니다. [Essentia PitchYinProbabilistic 문서](https://essentia.upf.edu/reference/std_PitchYinProbabilistic.html)

따라서 먼저 실시간 의미를 두 종류로 나눠야 합니다.

### A. 누적 chunk 재분석

가장 구현이 쉬운 실험입니다.

- AudioWorklet에서 PCM을 수집
- 최근 0.5초, 1초, 2초 등 고정 window 유지
- window 전체에 pYIN 실행
- 마지막 안정 frame만 UI에 전달
- 일정 hop마다 반복

측정값:

- window 길이별 처리 시간
- 호출 간격
- UI가 멈추는 시간
- 마지막 frame의 안정성
- 실제 end-to-end 지연
- 메모리 증가
- main thread와 Web Worker 차이

합격 기준의 예:

- 한 번의 분석 시간이 호출 주기보다 짧음
- UI frame drop이 체감되지 않음
- 전체 표시 지연이 Practice 모드 허용 범위 이내
- 30초 이상 실행해도 처리 backlog가 누적되지 않음

하지만 HMM이 충분한 과거·미래 문맥을 요구한다면 window 끝부분 결과가 불안정할 수 있습니다. 마지막 frame 대신 일정 look-ahead만큼 이전 frame을 출력해야 할 가능성이 큽니다.

### B. 실제 streaming pYIN

Essentia C++에는 streaming 형태의 pYIN 관련 알고리즘이 문서화되어 있지만, 현재 사용 중인 Essentia.js API가 브라우저에서 동일한 streaming graph를 실용적으로 노출하는지는 별도 검증이 필요합니다. [Essentia 알고리즘 목록](https://essentia.upf.edu/algorithms_reference.html)

확인할 항목:

- Essentia.js가 `streaming::PitchYinProbabilistic` binding을 제공하는지
- frame별 후보 상태를 호출 사이에 유지할 수 있는지
- HMM이 결과를 확정하기 위해 미래 frame을 얼마나 요구하는지
- Web Worker 또는 AudioWorklet에서 WASM instance를 안전하게 실행할 수 있는지
- 매 frame 메모리 할당 없이 buffer를 재사용할 수 있는지

현재 패키지 API에 streaming binding이 없다면, Essentia.js pYIN을 제품 실시간 backend로 전환하는 작업은 작은 adapter 변경이 아니라 별도의 WASM binding 작업이 됩니다.

## 3. 성능 실험 구성

성능은 최소한 다음 환경에서 측정해야 합니다.

- 데스크톱 Chrome
- 데스크톱 Edge
- 개발자 도구를 닫은 일반 실행
- main thread
- Web Worker
- 30초 이상 연속 실행

각 설정에서 기록할 값:

- 첫 검출까지 걸린 시간
- chunk별 평균·p95·최대 처리 시간

현재 offline 결과는 오디오 길이의 약 `1.42×`가 걸렸기 때문에, 동일 구현을 main thread에서 반복 호출하는 방식은 실시간 합격 가능성이 낮습니다. frame size나 hop을 조금 조정하는 것만으로 해결될지 먼저 작은 benchmark로 확인해야 합니다.

## 4. 라이선스 확인 결과

현재 설치된 `essentia.js@0.1.3`의 `package.json`은 라이선스를 `AGPL-3.0`으로 명시합니다. 공식 저장소 역시 AGPL-3.0입니다. [Essentia.js 공식 저장소](https://github.com/MTG/essentia.js)

Essentia 공식 라이선스 페이지는 다음 경로를 제시합니다.

- 비상업적 공개 사용: AGPLv3
- 다른 조건으로 사용하는 제품: MTG에 상업용 라이선스 문의
- 일부 알고리즘의 제3자 의존성도 별도로 확인 필요

[Essentia 공식 라이선스 안내](https://essentia.upf.edu/licensing_information.html)

따라서 SPRED에 직접 번들링하려면 단순히 저작권 고지 파일만 추가하는 것으로 끝난다고 가정하면 안 됩니다.

확인해야 할 결정은 다음과 같습니다.

- SPRED 전체를 AGPLv3 조건에 맞춰 배포할 의향이 있는가
- 현재 프로젝트 또는 향후 서비스의 배포 정책과 AGPL이 호환되는가
- 호환되지 않는다면 MTG 상업용 라이선스를 취득할 것인가
- 어느 쪽도 아니라면 Essentia.js를 비교 실험 전용으로 제한할 것인가

법률 자문이 아니라 프로젝트 차원의 안전한 판단으로는, 현재 pYIN adapter를 제품 build에 포함하지 않고 debug용 동적 import와 실험 브랜치에만 유지하는 것이 맞습니다.

## 해석

이 결과를 “pYIN의 실시간 지연이 음수다”라고 해석하면 안 됩니다.

현재 pYIN은 전체 27초 신호를 한꺼번에 분석하고 HMM이 전체 시간 경로를 결정합니다. 미래 입력까지 참고할 수 있는 offline 처리이므로 contour가 실제 사건보다 앞서 배치되는 비인과적 효과가 생길 수 있습니다.

timestamp 정의 차이도 있습니다.

- pitchy 첫 frame 중심: 약 42.67ms
- pYIN 첫 frame 중심: 0ms
- SwiftF0 첫 frame 중심: 약 7.97ms

따라서 확인된 `-85ms`에는 다음이 섞여 있습니다.

- backend별 frame 중심 정의
- pYIN HMM의 미래 문맥 사용
- 실제 검출 특성 차이
- 글리산도에서의 smoothing

결론적으로 기존 JSON은 다음 사실을 보여줍니다.

- pYIN contour에는 pitchy와 약 85ms의 체계적인 시간 정렬 차이가 있음
- 정렬 후 정상 구간 오차는 약 7cent로 매우 작음
- 하지만 이 자료만으로 실시간 UI 지연을 판단할 수 없음

현악기형 배음을 추가해도 개선되지 않았으므로 단순히 “순음이라 pYIN이 실패했다”는 설명도 충분하지 않습니다. 짧은 전체 신호에 pYIN HMM을 적용할 때 서브하모닉 경로에 고정되는 특성일 가능성이 큽니다.

- 짧은 입력에서도 CPU 여유가 작고
- 현재 방식은 전체 신호가 끝나야 contour가 나오며
- 짧은 window에서는 한 옥타브 오류가 발생하고
- AGPL 배포 문제도 남아 있습니다.

따라서 고정 window를 반복 분석하는 실시간 pYIN 실험은 현재 중단하는 것이 타당합니다. pYIN은 실제 WAV를 분석하는 offline 비교 자료로만 유지하는 편이 안전합니다.

가장 유망한 다음 방향은 hybrid입니다.

- SwiftF0 confidence > 0.9이면 SwiftF0 사용
- SwiftF0가 confident하지 않으면 pitchy 사용

## Hybrid 방향

SwiftF0로 pitchy 값을 직접 교체하는 방식은 권장하지 않습니다. 충돌 frame의 약 42%에서는 SwiftF0 쪽이 잘못됐기 때문입니다.

가장 안전한 첫 hybrid는 “교체”가 아니라 “충돌 거부”입니다.

```text
SwiftF0가 없거나 stale:
    pitchy 결과 사용

SwiftF0가 fresh하고 pitchy와 근접:
    pitchy 결과 사용

SwiftF0가 fresh하고 5 semitone 이상 충돌:
    해당 pitch frame을 unvoiced로 거부
```

권장 실험 설정:

```text
SwiftF0 window: 500ms
update interval: 150ms
confidence: 0.9
maximum lookback: 150ms
agreement tolerance: 우선 100cent
hard conflict: 5 semitone 이상
```

이 방식은:

- pitchy의 2배음 오류가 화면에 잘못 표시되는 것을 차단하고
- SwiftF0의 617Hz·1660Hz 오류도 pitchy를 덮어쓰지 못하게 하며
- SwiftF0 coverage가 부족한 일반 구간에서는 기존 pitchy 동작을 유지합니다.

즉 이번 자료는 SwiftF0가 유용한 검증 신호이지만 대체 detector로는 불충분하다는 결론을 지지합니다. 다음 구현은 제품 기본값을 바꾸지 않는 debug 전용 실시간 conflict gate가 적절합니다.

SwiftF0 결과는 허용 한도 150ms에 상당히 가까운 상태로 들어왔습니다. 따라서 빠른 글리산도에서는 과거 음정과 현재 pitchy를 비교하게 됩니다.

## 점프 억제 효과

pitchy 원시 결과에는 5 semitone 이상 점프가 7회 있었습니다.

- conflict gate 적용 후: 1회

수치상 6회를 차단했지만 이것이 좋은 UX로 이어지지는 않았습니다.

실제로 잡은 pitchy 배음 오류:

- 15.760초: 231.8→469.8Hz, SwiftF0 232.4Hz
- 19.663초: 250.3→504.7Hz, SwiftF0 251.3Hz
- 24.913초: 251.9→508.4Hz, SwiftF0 252.0Hz
- 26.715초: 333.3→999.8Hz, SwiftF0 333.9Hz

즉 pitchy의 2배·3배음 오류를 탐지하기는 했습니다.

## 실사용이 나빠진 원인

### 1. 교정이 아니라 점을 제거함

현재 gate는 pitchy가 틀렸을 때 SwiftF0 음정으로 고치지 않고 `unvoiced`로 만듭니다.

따라서 사용자에게는:

- 배음으로 튐
- 초록 원이 사라짐
- 잠시 후 다시 나타남

형태로 보입니다. 배음 문제가 해결됐다기보다 끊김으로 바뀐 것입니다.

### 2. SwiftF0가 절반 정도만 사용 가능

pitchy voiced 후보 중 fresh SwiftF0가 있었던 비율은 54.4%입니다. 나머지 약 46%에서는 기존 pitchy 오류가 그대로 통과합니다.

실제로 다음 복귀 점프는 verifier가 stale이라 거부되지 않았습니다.

- 19.876초: 511.5→257.2Hz
- 24.978초: 510.9→256.2Hz

상승 오류는 거부했지만 복귀 순간은 통과하면서 표시가 불연속적으로 보일 수 있습니다.

### 3. SwiftF0 false positive

7.706초 부근:

- pitchy: 약 236Hz
- SwiftF0: 약 1661Hz
- verifier age: 약 120~141ms

이 구간에서는 SwiftF0가 잘못된 고배음을 냈고, 정상으로 보이는 pitchy frame을 conflict로 거부했습니다.

즉 SwiftF0는 pitchy 오류를 검출하기도 하지만 새로운 false rejection도 만듭니다.

### 4. 글리산도와 구조적으로 맞지 않음

verifier age 중앙값이 116ms이고 150ms 주기로 갱신됩니다. 지속음에서는 사용할 수 있지만 빠르게 변하는 글리산도에서는 서로 다른 시점의 음정을 비교하게 됩니다.

freshness를 더 짧게 하면:

- verifier coverage가 더 낮아짐
- pitchy 오류가 더 많이 그대로 통과

freshness를 길게 하면:

- 과거 SwiftF0 음정과 현재 pitchy가 더 자주 충돌
- gliss 인식이 더 나빠짐

단순 threshold 조정으로 해결하기 어려운 구조입니다.

## 결론

SwiftF0 conflict gate 실험은 제품 적용에 실패한 것으로 판단하는 것이 맞습니다.

- 배음 오류를 부분적으로 탐지함
- 하지만 교정하지 못하고 점을 끊음
- verifier coverage가 부족함
- 정상 pitchy를 거부하는 사례가 있음
- 지연 때문에 gliss 성능이 나빠짐

현재 기능은 별도 query에서만 활성화되므로 일반 경로에는 영향이 없습니다.

```text
일반 실행 또는 ?pitchDebug=1
→ 기존 pitchy 동작

?pitchDebug=1&pitchConflict=1
→ 실패한 conflict 실험
```

다음 방향은 SwiftF0 gate를 더 조정하는 것보다 aubio 계열 실시간 backend 비교로 넘어가는 것이 타당합니다. 지금까지의 결과로 보면:

- pYIN: offline 참고는 가능하지만 실시간·짧은 window 실패
- SwiftF0: 정확한 구간도 있지만 coverage·지연·false positive 문제
- pitchy 후처리: 실제 입력에서 안정적이지 않음

따라서 보류했던 aubio의 C/WASM 실시간 detector를 같은 backend 계약에 연결해 실제 WAV와 마이크에서 비교할 시점입니다. CREPE는 aubio까지 실패한 후 검토하는 순서가 적절합니다.

- 실제 마이크 입력에서는 정확한 2배·3배 주파수 오류
- pYIN과 SwiftF0도 일부 구간에서 독자적인 배음 오류
- smoothing·후처리·낮은 후보 선택은 다른 음정을 훼손
- detector 조합은 오류를 줄여도 지연·끊김·false rejection을 만듦

이는 단순히 알고리즘 하나가 나빠서라기보다, 특정 시점의 PCM이 여러 F0 가설 중 하나를 충분히 구분해주지 못할 가능성을 높입니다.

## 다음 최우선 분석

이미 무압축 WAV가 있으므로 새 녹음 전에 다음 분석을 할 수 있습니다.

대상 구간:

- `143Z`의 알려진 2배·3배음 오류 시점
- `370Z`의 pitchy 오류 구간
- pitchy는 정상인데 SwiftF0가 617Hz·1660Hz를 선택한 구간
- 정상 인식 직전·오류 중·정상 복귀 직후

각 구간에서 확인할 항목:

1. 기본음 주파수의 실제 에너지

   - F0 bin의 크기
   - 주변 noise floor 대비 SNR
   - 2F, 3F, 4F, 5F와의 상대 크기

2. 배음 간격

   - 2F만 존재하는지
   - 2F·3F·4F가 함께 있어 F0 간격을 추론할 수 있는지
   - 짝수 배음만 우세해 F0와 2F가 수학적으로 모호한지

3. 주기성 후보

   - autocorrelation/NSDF의 T와 T/2 peak
   - 두 peak의 높이와 안정성
   - 감쇠 중 어느 peak가 우세해지는지

4. cepstrum

   - F0에 해당하는 quefrency
   - 2F에 해당하는 절반 주기의 peak
   - 오류 순간에 F0 quefrency가 사라지는지

5. window 길이별 변화

   - 4096 samples: 약 85ms
   - 8192 samples: 약 171ms
   - 16384 samples: 약 341ms

긴 window에서만 F0가 드러난다면 정보는 존재하지만 실시간 latency와 trade-off가 있다는 뜻입니다.

## 판정 기준

분석 결과는 세 경우로 나눌 수 있습니다.

### F0 정보가 충분히 존재함

예:

- 기본음 에너지가 명확함
- 3F·5F 같은 배음이 있어 harmonic spacing이 F0를 지지함
- autocorrelation 또는 cepstrum에도 F0 후보가 안정적으로 존재함

이 경우 detector 구현·parameter·windowing 문제일 가능성이 있습니다. aubio는 독립적인 기준 구현으로 비교할 가치가 있습니다.

### F0는 약하지만 배음 간격으로 추론 가능함

예:

- F0 자체는 noise floor에 가까움
- 2F·3F·4F가 모두 존재해 공약 간격은 F0를 가리킴

이 경우 단일 주기 detector보다 multi-harmonic salience나 cepstral 계열을 검토할 수 있습니다. 그래도 실제 A#4와 A#3의 배음을 혼동할 위험을 별도로 측정해야 합니다.

### F0 식별 정보가 없음

예:

- F0가 noise floor 아래
- 2F·4F처럼 짝수 배음만 존재
- waveform 자체가 T/2 주기로 반복
- autocorrelation·cepstrum 모두 2F만 지지

이 경우 PCM만으로 실제 F0와 한 옥타브 높은 음을 구분하는 것은 원리적으로 어렵습니다. 다른 detector를 추가해도 안정적으로 해결되지 않습니다.

해결 방향은 알고리즘이 아니라 다음으로 이동해야 합니다.

- 마이크 위치·방향
- pickup 또는 오디오 인터페이스
- 입력 장치의 frequency response
- stereo→mono downmix 과정
- 브라우저/OS의 숨은 입력 처리
- 악기와 마이크 사이의 공진·위상 상쇄

## 특히 확인할 입력 경로

현재 장치가 mono 요청에도 2채널을 보고했고, SPRED는 이를 speaker 방식으로 explicit mono downmix했습니다. detector와 WAV가 동일 PCM을 사용한다는 점은 비교 공정성에는 좋지만, downmix 전에 두 채널의 기본음 위상이 달랐다면 합산 과정에서 F0가 약해질 가능성은 남아 있습니다.

따라서 mono WAV에서 F0 정보가 없다고 확인되면 다음 녹음은 다음처럼 해야 합니다.

- 원본 channel 1
- 원본 channel 2
- 현재 mono downmix

세 신호를 별도로 저장해 F0 에너지를 비교합니다.

다만 사용자 메시지에서 “알고리즘보다 먼저 확인해야 할 단서” 부분이 생략되어 있어 그 구체적인 단서는 아직 반영할 수 없습니다. 해당 대목이 입력 gain, stereo downmix, sample format, 마이크 처리 등에 관한 내용이라면 다음 분석 순서에 직접 영향을 줄 수 있으므로 나중에 원문을 알려주시는 것이 좋습니다.

현재 작업 우선순위는 다음처럼 수정하는 것이 타당합니다.

1. 실패 PCM의 F0 식별 가능성 분석
2. 입력/downmix 단계에서 기본음 손실 여부 확인
3. F0 정보가 존재할 때만 aubio를 독립 기준 구현으로 비교
4. F0 정보가 없다면 detector 교체 중단
5. CREPE는 정보가 존재하지만 고전 DSP가 반복해서 실패할 때만 검토

이 결론은 다음 구현 계획과 3.4 명세의 “다음 작업”에도 반영할 가치가 있습니다.

## 두 경우의 의미

### 경우 A: 기음은 약하지만 배음 간격이 남음

예를 들어 200·300·400·500Hz가 존재하면 100Hz 성분이 없어도 공통 간격으로 F0=100Hz를 추론할 수 있습니다.

이 상태에서 detector가 계속 200Hz를 고른다면 다음 가능성이 있습니다.

- 분석 window가 너무 짧음
- attack·gliss·vibrato 때문에 window 내부가 비정상적임
- 홀수차 배음의 세기가 detector threshold 아래임
- 비조화성이 후보 score를 흐림
- 전처리나 구현 parameter가 부적절함

이 경우 aubio 같은 독립 구현을 비교할 가치가 있습니다.

### 경우 B: 사실상 짝수배만 남음

200·400·600·800Hz만 남으면 파형의 실제 반복 주기는 200Hz입니다. 단일 frame만으로 100Hz가 의도된 F0인지, 200Hz가 실제 F0인지 판별할 수 없습니다.

이 경우 detector 교체는 근본 해결이 되지 않습니다. CREPE도 정보를 복원하는 것이 아니라 학습한 악기 prior를 이용해 더 그럴듯한 값을 추측하는 것입니다.

따라서 ML을 사용하더라도 “PCM에서 객관적으로 결정했다”가 아니라 “악기 음색 prior로 모호성을 해소했다”고 설명해야 합니다.

## SPRED에서 바로 적용할 분석 순서

### 1. 통제된 단일음 corpus

기존 긴 gliss WAV도 사용할 수 있지만, 식별 가능성을 판정하려면 의도한 F0와 구간이 명확한 짧은 녹음이 더 좋습니다.

우선 다음 두 파일이 필요합니다.

- A#3 약 233.08Hz: attack 포함 2~3초
- A#4 약 466.16Hz: attack 포함 2~3초

가능하면 각 조건을 반복합니다.

- 기존 거리
- 매우 가까운 거리
- 먼 거리
- 다른 마이크

프렛리스 악기라 정확한 주파수는 조금 변할 수 있으므로, 명목 음정보다 시간별 실제 F0 후보를 좁은 범위에서 추적해야 합니다.

### 2. 구간 분리

각 녹음을 다음으로 나눕니다.

- attack
- early sustain
- stable sustain
- decay

배음 오류가 sustain이나 decay에서만 발생한다면 전체 평균 spectrum으로 보면 원인이 가려질 수 있습니다.

### 3. frame별 측정

권장 초기 설정:

- sample rate: 48kHz
- Hann window
- window: 4096, 8192, 16384 samples
- hop: 256 또는 512 samples

기록할 값:

- DC 평균
- RMS와 peak amplitude
- clipping sample 수
- F0, 2F0, 3F0, 4F0, 필요하면 5F0 진폭
- 각 배음의 dBFS
- 주변 noise floor 대비 SNR
- 2F/F0, 3F/F0, 4F/F0 비율
- pitchy frequency와 clarity
- autocorrelation 또는 NSDF의 T·T/2 peak
- cepstrum의 F0·2F0 quefrency peak

고정 FFT bin 하나가 아니라 예상 주파수 주변의 작은 band peak를 사용해야 프렛리스 음정 변화와 vibrato를 견딜 수 있습니다.

## 가장 중요한 판정 그래프

```text
시간
├─ RMS / peak / clipping
├─ F0 amplitude 및 SNR
├─ 2F0 amplitude
├─ 3F0 amplitude
├─ 4F0 amplitude
├─ 2F/F0 및 3F/2F 비율
├─ Pitchy frequency / clarity
├─ autocorrelation T / T/2 score
└─ cepstrum F0 / 2F0 score
```

이 그래프를 attack·sustain·decay 경계와 함께 봐야 합니다.

## 2F notch 실험의 의미

2F0 notch는 진단용으로 가치가 있습니다.

- 2F를 제거하자 pitchy가 F0를 찾음
  → F0 정보는 PCM에 남아 있고, 강한 2F가 후보 선택을 압도했을 가능성

- 2F를 제거해도 F0를 찾지 못함
  → F0·홀수차 배음 정보가 부족하거나 다른 주기 성분이 문제일 가능성

- 2F 제거 후 3F 또는 다른 오검출로 이동
  → 단일 강한 배음 문제가 아니라 전체 후보 생성·score 문제일 가능성

notch는 제품 보정으로 바로 사용하면 안 됩니다. 실제 고음의 기본음을 제거할 수도 있기 때문입니다.

## Float32 PCM 필요성

현재 AudioWorklet 내부에서는 Float32 PCM을 받지만 WAV export는 PCM16입니다.

PCM16도 일반적인 spectrum 분석에는 충분하지만, F0가 noise floor 가까이에 있는지를 정밀하게 확인하려면 양자화 이전 Float32가 더 좋습니다. 다음 진단 도구에는 다음 중 하나가 필요합니다.

- raw Float32 binary와 metadata export
- IEEE float32 WAV export
- 브라우저 메모리에서 바로 분석하고 JSON만 export

세 번째 방식이 가장 편리합니다. PCM 전체를 저장하지 않고도 frame별 배음 amplitude와 score를 JSON으로 남길 수 있습니다. 다만 재분석 가능성을 위해 raw Float32 export도 함께 두는 편이 안전합니다.

## 기존 실험과의 관계

주의할 점은 SPRED에서 비교한 것이 Pitchy, pYIN, SwiftF0라는 것입니다. SWIPE는 아직 구현하지 않았습니다.

따라서 현재 결론은 다음과 같이 기록해야 정확합니다.

> Pitchy, pYIN, SwiftF0 및 SwiftF0 conflict 조합이 실제 마이크 입력에서 서로 다른 배음 오류 또는 실사용 회귀를 보였다. 다음 backend를 추가하기 전에 실패 PCM의 F0 식별 정보 존재 여부를 분석한다.

2. F0·3F0가 명확히 남아 있으면 새 녹음 없이 분석 창·후보 생성·전처리 문제를 조사합니다.
3. F0·3F0가 거의 없고 짝수차 배음만 지배적이면 입력 정보 부족 가능성을 확인합니다.
4. 결과가 경계선이거나 다운믹스 의심이 생길 때만 Float32 및 채널별 새 캡처를 수행합니다.

즉, 새 실사용 녹음은 지금 당장 필수는 아닙니다. 기존 WAV를 고정 회귀 자료로 삼아 배음 식별 가능성 분석기를 먼저 구현할 수 있습니다. CREPE나 aubio 도입 여부도 그 결과를 본 뒤 결정하는 편이 타당합니다.

기존 WAV 검증을 완료했습니다. 대표적인 2배 오인식 구간은 “기음 정보가 완전히 사라진 경우 B”보다는 “복원 단서가 남은 경우 A”에 가깝습니다.

두 실사용 WAV에서 약 2배 급상승 전이 16건을 찾았습니다.

- 낮은 F0 지지: 11건
- 경계 수준: 5건
- 짝수차만 남은 옥타브 식별 불가: 0건

특히 실제 F0 성분인 1F는 2F보다 30~44dB 약한 구간이 많았지만, 3F는 대체로 2F보다 약 6~14dB 낮은 수준으로 남아 있었습니다. 낮은 F0 주기의 자기상관도 대표 구간에서 약 `+0.2` 우세했습니다.

따라서 현재 판단은 다음과 같습니다.

- 마이크 PCM에 낮은 F0를 추론할 정보가 완전히 사라진 것은 아닙니다.
- Pitchy가 강한 2F를 선택하면서 3F와 낮은 주기 단서를 충분히 활용하지 못했을 가능성이 큽니다.
- aubio 같은 독립 DSP 구현을 비교할 가치는 여전히 있습니다.
- 동시에 CREPE 계열 최신 AI 모델도 aubio 실패 후로 미루지 않고 병렬 오프라인 비교 대상으로 격상하는 것이 타당합니다.
- 다만 AI도 정답을 객관적으로 복원한다기보다 학습된 악기 prior로 선택할 수 있으므로, 동일 WAV와 동일 오류 전이를 기준으로 평가해야 합니다.
