# 1.41 Performance Profiler Instrumentation

## 1. Purpose

긴 악보에서 병목 지점을 큰 범위부터 작은 범위로 좁히기 위해 runtime performance profiler를 추가했다.

이번 단계는 계측 기반 탐색 준비와 재생 follow UI의 1차 안정화까지 포함한다.

## 2. Implemented Files

- `regression-code/src/infra/perf_profiler.ts`
  - session 기반 측정
  - nested `measurePerf()` 기록
  - console summary/table 출력
  - `window.spredPerf` console API
- `regression-code/src/app/main.ts`
  - app render
  - score text edit orchestration
  - history patch
  - partial plan / render patch
  - playback reset
- `regression-code/src/app/app_runtime.ts`
  - rawText batch apply
  - runtime document rebuild
  - renderer base input creation
  - partial artifact build / full fallback
- `regression-code/src/renderer/canvas_score_renderer.ts`
  - layout calculation
  - visible item filtering
  - canvas resize
  - grid / marker / note draw
- `regression-code/src/app/playback/app_playback.ts`
  - playback runtime creation
  - audio schedule / queue / backend / scheduler / controller / tick mapper creation
- `regression-code/src/app/playback/app_playback_binding.ts`
  - play/pause toggle
  - after-play UI sync
  - playback scroll RAF frame
- `regression-code/src/audio/audio_schedule_builder.ts`
  - schedule mapper creation
  - active track filtering
  - track event conversion
  - overlap gain automation
- `regression-code/src/audio/playback_controller.ts`
  - play / seek / pauseAt controller steps
  - lookahead scheduling tick
- `regression-code/src/audio/audio_scheduler.ts`
  - lookahead window creation
  - event queue range queries
  - backend scheduling loops

## 3. How To Enable

URL query:

```text
http://127.0.0.1:5173/?perf=1
```

또는 browser console:

```js
window.spredPerf.enable()
window.spredPerf.disable()
window.spredPerf.isEnabled()
```

`enable()`은 `localStorage`에 `spred:perf = 1`을 저장하므로 새로고침 후에도 유지된다.

## 4. Output Policy

profiler가 꺼져 있으면 측정 함수는 callback만 실행하고 기록하지 않는다.

profiler가 켜져 있으면 action별로 console group을 출력한다.

대표 session:

- `app.render`
- `app.applyScoreTextEdits N edit(s)`
- `playback.toggle`
- `playback.toggle.afterPlay`
- `playback.raf.updateScroll`
- `playback.interval.lookahead`

각 session은 상세 단계와 label별 합산 summary를 `console.table()`로 출력한다.
0.25ms 미만 구간은 표시에서 제외해 작은 노이즈를 줄인다.
재생 중 RAF frame과 lookahead interval은 로그가 과도하게 쌓이지 않도록 느린 tick만 출력한다.

## 5. Current Measurement Strategy

첫 탐색 순서:

1. 긴 악보 로드 후 initial/full render 확인
2. 단일 note edit 측정
3. drag edit 측정
4. range paste/delete 측정
5. scroll 중 viewport redraw 체감 확인
6. playback start / seek / pause 측정
7. playback 중 `playback.raf.updateScroll` 또는 `playback.interval.lookahead` slow tick 확인

큰 구간에서 병목 후보가 보이면 다음 단계에서 해당 모듈 내부를 더 세분화한다.

예:

- `runtime.buildScoreTextEditPartialArtifacts`가 크면 parser/analyzer/renderer item builder 내부 계측
- `renderer.partial.note.drawNotes`가 크면 note/mute/global text draw 내부 계측
- `playback.preserve.createRuntime`이 크면 audio schedule builder 내부 계측
- `playbackRuntime.buildAudioSchedule`가 크면 schedule builder의 track별 변환 / overlap gain 내부 계측
- `playback.interval.lookahead`가 반복적으로 크면 audio event queue index 또는 backend schedule path 점검

## 7. Measurement Result

데스크톱 환경에서는 app render, edit, playback 모두 충분히 부드럽게 동작했다.

Chrome DevTools CPU throttling 4x에서는 대체로 frame budget 안에서 동작했고, 일부 render/playback tick이 튀는 정도였다.

CPU throttling 6x에서는 재생 중 `playback.raf.updateScroll`의 slow tick이 관찰되었다.
초기에는 `syncSeekUi`가 대부분의 시간을 차지했으므로 다음 조정을 적용했다.

- seek range / clock label / BPM label은 값이 바뀔 때만 DOM에 쓴다.
- `resolveBpmAtTick()`은 timing segment 선형 탐색 대신 이진 탐색을 사용한다.
- `syncSeekUi` 내부를 세분화해 다음 병목을 확인할 수 있게 했다.

이후 6x에서 follow scroll throttling을 30fps로 낮추는 실험도 했지만, 수치상 비용은 줄어도 화면이 계단식으로 움직여 눈에 피로했다.
따라서 30fps 제한은 제거하고, RAF마다 follow scroll을 갱신하는 정책을 유지했다.
대신 `scrollLeft`는 0.5px 이상 차이가 날 때만 쓰도록 했다.

실제 2019년형 저전력 노트북 환경에서도 악보 재생은 정상적으로 가능했다.
사양은 i5-10210U CPU, RAM 8GB였고, 약간의 frame drop은 있으나 재생 사용성은 유지되었다.

## 8. Stability Decision

현재 상태는 MVP 사용 테스트와 다음 기능 구현으로 넘어갈 수 있을 만큼 안정화된 것으로 본다.

단, 다음 조건은 유지한다.

- 저사양 노트북에서 완전한 60fps를 보장하지는 않는다.
- 재생 오디오가 끊기지 않고, edit 입력이 실사용 가능하며, 긴 악보가 조작 가능하면 통과로 본다.
- profiler는 유지하되 기본 상태에서는 꺼져 있어야 한다.
- 이후 큰 기능을 추가할 때는 `?perf=1`로 회귀 여부를 확인한다.
