## 2. 재생 위치 보존

편집, 볼륨 변경, wave 변경, column expand 같은 동작 뒤 playback runtime을 재생성할 때 무조건 처음으로 돌아가던 문제를 수정했다.

현재 방향:

- 기존 playback runtime에서 현재 score seconds를 읽는다.
- 기존 time mapper로 seconds를 tick으로 변환한다.
- 새 score/runtime을 만든 뒤 새 time mapper로 같은 tick의 seconds를 다시 계산한다.
- 새 controller를 `paused` 상태로 두고 해당 seconds에 위치시킨다.

이 방식은 BPM, column 수, layout 변화 뒤에도 사용자가 보고 있던 score tick을 기준으로 위치를 보존한다.
Trim Right, Clear All처럼 score 길이나 구조를 크게 줄이는 동작은 기존처럼 명시적 reset을 유지한다.

Space key는 브라우저의 focused button 재클릭 기본 동작을 막고, 재생/일시정지 단축키로 고정했다.
입력칸, select, textarea, contenteditable 안에서는 space 입력을 방해하지 않는다.

## 3. 입력 제한과 사용자 테스트 안전장치

공개 테스트 전에 큰 JSON이나 비정상 구조 입력으로 앱이 멈추는 문제를 줄이기 위해 client-side 제한을 보강했다.
정적 웹 앱 특성상 브라우저 콘솔로 제한을 우회할 수 있으므로 보안 경계라기보다 사용자 경험 보호 장치로 본다.

추가/정리된 제한:

- score JSON 파일 크기: 8 MiB
- local score save 크기: 3 MiB
- cell `rawText`: 100 characters
- score comment: 100 characters
- column count: 1..10000
- one expand action: max 960 columns
- row definitions: max 4096
- row height: 1..500
- layout preset JSON: 256 KiB
- YouTube video id: URL parser를 통과한 id만 저장/로드

또한 note/gap row의 string reference와 같은 구조 검증을 강화해 잘못된 row/string 관계가 runtime으로 들어오지 않게 했다.

## 4. 저장/로드 UX 방어

초기 앱은 빈 악보로 시작한다.
이 상태에서 `Local Load` 대신 `Local Save`를 누르면 저장된 악보가 빈 악보로 덮일 수 있고, 반대로 작업 중 `Local Save` 대신 `Local Load`를 누르면 현재 수정본을 잃을 수 있다.

현재 동작:

- `Local Save`: browser local storage의 기존 score를 덮어쓸지 확인한다.
- `Local Load`: 현재 score를 local storage score로 교체할지 확인한다.

이 확인은 JSON File Download/Load와 별개로, 브라우저 local storage 조작에만 적용된다.

## 5. Edit batch와 audio scheduler 안정화

리뷰에서 지적된 batch edit 정렬 비용과 audio schedule 누적 비용을 줄였다.

Edit batch:

- 기존에는 edit마다 cell 배열을 filter/push/sort하는 방식이었다.
- 현재는 대상 track/global별로 edit을 그룹화한다.
- 좌표 key Map에서 upsert/delete를 적용한 뒤 대상 배열별로 한 번만 sort한다.

Audio:

- overlap gain automation은 span pointer를 넘겨 이벤트마다 처음부터 span을 다시 훑는 비용을 줄였다.
- loop playback 중 scheduled key가 무한히 쌓이지 않도록 이전 loop cycle key를 pruning한다.
- oscillator backend에는 active voice cap과 tremolo pulse cap을 둬 비정상적으로 큰 score에서 browser audio work가 폭증하지 않게 했다.

## 6. Fullscreen, Fit Height, zoom

Fullscreen 진입/해제 시 현재 score 높이에 맞춰 자동 `Fit Height`를 수행한다.
브라우저 fullscreen layout이 한 frame 뒤 확정되므로 `fullscreenchange` 직후 `requestAnimationFrame`에서 fit을 실행한다.

Fit Height 기준은 다음으로 정리했다.

```text
target height = status-area top - score-area top
```

즉 layout/score 영역의 하단 기준은 status footer의 실제 윗면이다.
이렇게 해야 fullscreen에서 footer가 독립된 하단 영역으로 남고, score canvas가 footer 아래로 들어가지 않는다.

Zoom 정책:

- 수동 zoom 범위는 `50%`에서 `200%`로 완화했다.
- `Fit Height`도 같은 slider 범위 안에서 clamp된다.
- 과도한 확대/축소가 layout hit-test와 canvas bitmap 비용을 크게 흔들 수 있으므로 완전 무제한으로 열지는 않는다.

축소 시 score stage 아래에 흰 공백이 보이는 문제는 `--score-viewport-height`를 추가해 해결했다.
렌더 후 현재 `scoreArea.clientHeight`를 CSS 변수로 반영하고, layout/score stage의 `min-height`를 그 값으로 둔다.
layout area, score viewer, score area 배경은 `--score-roll-background` 진회색으로 맞췄다.

## 7. Status footer

`status-area`는 menu/edit/score viewer와 분리된 app shell의 마지막 grid row이다.
큰 layout label canvas가 footer 위로 넘어오지 않게 다음을 적용했다.

- `.layout-area { overflow: hidden; }`
- `.status-area`는 `--status-height`로 height/min-height/max-height를 고정
- `.status-area`에 `position: relative`와 z-index를 둬 footer가 독립 영역으로 보이게 함

Renderer status에는 row 수와 column 수를 함께 표시한다.

```text
renderer: 73 rows, 1000 cols
```

## 8. 배포 반영 방식

이번 사용자 테스트 버전은 별도 local staging copy인 `regression-code-test-publish/`에 소스 변경 파일을 복사한 뒤 빌드 확인하는 방식으로 반영했다.
배포 저장소에는 `base: "./"` 설정과 GitHub Pages workflow가 유지된다.

현재 운영 방식:

- main implementation: `regression-code/`
- publish staging copy: `regression-code-test-publish/`
- 변경 후 필요한 파일만 publish copy에 복사
- publish copy에서 `npm run build` 확인
- 실제 commit/push는 사용자가 해당 repository에서 수행

## 9. 검증

최근 변경 중 반복 확인한 명령:

```text

```

배포 staging copy에서는 변경 반영 뒤 `npm run build`를 확인했다.

## 10. 남은 작업 후보

- Fullscreen/Fit Height는 실제 브라우저 viewport와 OS fullscreen UI에 따라 미세 차이가 날 수 있으므로 사용자 테스트 피드백을 계속 받는다.
- `status-area`와 score viewer의 grid row 경계는 현재 CSS로 안정화했지만, Playwright screenshot 기반 회귀 테스트가 있으면 더 확실하다.
- 큰 score에서 base/layout/global marker canvas는 아직 full width 기반이다. note/note marker dynamic viewport 이후 남은 장기 성능 후보는 tile/chunk rendering이다.
- client-side 제한은 사용자 경험 보호 장치이며, 신뢰할 수 없는 입력의 보안 경계로 보지 않는다.
