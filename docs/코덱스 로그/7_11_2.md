**현재 작업 현황**

`regression-code/`가 안정판 구현 루트이고, 현재는 2차 구현 기능을 넓히기보다 실사용 테스트와 2차 구현 보고서 작성으로 넘어간 상태입니다. `HARNESS.md` 기준으로는 “구현 변경은 새 실사용 이슈가 발견될 때만” 하고, 보고서는 `docs/report-prep/`, `docs/4.0-second-extension-report-guide.md`, 실제 구현 파일과 git 기록을 대조해서 쓰는 흐름입니다.

2차 구현의 큰 축은 다음까지 진행되어 있습니다.

- parser/analyzer/renderer/audio/UI 기본 파이프라인 구축
- layout 편집, track layer, YouTube sync
- partial rebuild, viewport bounded rendering, 성능 profiler
- range selection, paste preview, undo/redo
- Examples, Manual, GitHub Pages 테스트 배포 준비
- Practice Mode: 마이크 입력, pitch dot, pitch/timing 판정, combo/result, gliss/vib/trem bonus, Easy/Standard/Pro 모드 일부

Practice Mode 쪽은 [docs/3.2-karaoke-game-mode-spec.md](./docs/3.2-karaoke-game-mode-spec.md:1)와 [docs/3.3-third-extension-candidate-spec.md](./docs/3.3-third-extension-candidate-spec.md:21)가 현재 맥락을 잘 보여줍니다. 구현은 `regression-code/src/app/game/` 아래에 분리되어 있고, 핵심은 `game_pitch_input.ts`, `game_judge.ts`, `game_effect_judge.ts`, `game_types.ts`입니다.

마이크 입력은 현재 `pitchy` 기반입니다. 흐름은 대략:

`getUserMedia` → `AudioContext` → `AnalyserNode.getFloatTimeDomainData()` → `PitchDetector.findPitch()` → `GamePitchFrame`

현재 필터는 `clarity >= 0.75`, `rms >= 0.01`, layout 기반 주파수 범위입니다. `game_pitch_input.ts`를 보면 FFT size는 `4096`이고, pitchy가 반환한 raw frequency를 `midi + centOffset`으로 바꾼 뒤 voiced 여부를 결정합니다.

중요한 점은, 과거에 target 기반 octave correction / harmonic correction / hysteresis lock 실험이 있었지만 롤백되었다는 것입니다. 현재 명세와 구현 방향은 “검출 pitch를 target에 맞춰 보정하지 않고 raw detector pitch를 쓴다”입니다. 다만 판정에서는 `game_judge.ts`가 pitch-class error, 즉 1200 cent modulo 비교를 사용합니다. 그래서 한 옥타브 위/아래로 검출된 경우에는 판정상 같은 음으로 취급될 수 있습니다. 반대로 3배음처럼 완전5도 쪽으로 튀면 `Bad`나 `Miss`가 날 가능성이 큽니다.

따라서 지금 말씀하신 “배음으로 인한 오인식”은 현재 구조상 아주 자연스러운 다음 실사용 이슈입니다. 해결 방향은 기존 결정과 충돌하지 않게 잡으려면, 곧바로 target에 끌어붙이는 보정보다는 다음 후보들이 더 잘 맞아 보입니다.

- raw frame의 안정성 판단 강화: clarity/RMS만이 아니라 최근 frame 연속성, jump 크기, voiced 지속시간 보기
- 배음 후보 진단: `f/2`, `f/3`, `2f` 등을 target pitch class와 비교하되, 판정 pitch를 무조건 바꾸지 않고 “의심 상태”로 다루기
- 시각화와 판정 분리 유지: pitch dot은 raw 입력을 보여주고, scoring은 별도 정책으로 안정화
- 실사용 로그/진단 UI 보강: raw frequency, clarity, rms, rejectReason은 이미 panel에 있으므로 배음 케이스를 재현하며 수치 확인 가능
- 필요시 `AudioWorklet`, median/smoothing, Meyda 같은 보조 feature 검토

한 줄로 정리하면: 현재 프로젝트는 2차 구현 안정화와 보고서 작성 단계에 있고, Practice Mode는 이미 상당히 구현되어 있습니다. 배음 오인식은 “새 기능 추가”라기보다 실사용 테스트에서 발견된 Practice Mode 안정화 이슈로 다루는 것이 현재 문서 체계에 가장 맞습니다.

> 어느 피치가 입력되는가는 초록 원을 통해서 검증이 가능합니다. 굳이 새로운 디버그 모드를 추가할 필요가 있을까요?

초록 원만으로도 “지금 어떤 높이로 인식되는가”는 확인 가능합니다. 그래서 단순히 A#3가 A#4로 보인다는 증상 확인만 목적이면 새 디버그 모드는 꼭 필요하지 않습니다.

다만 디버그 모드가 필요한 이유는 초록 원이 보여주지 못하는 것들이 있어서입니다.

- `pitchy`가 실제로 반환한 `rawFrequencyHz`가 얼마인지
- `clarity`가 높은 상태에서 틀리는지, 낮은 상태라 튀는지
- `rms`가 너무 낮거나 특정 음량에서만 문제가 생기는지
- A#3에서 A#4로 “항상” 가는지, 프레임별로 A#3/A#4를 오가는지
- gliss/vib 중에는 튐이 생기는지, sustain에서만 생기는지
- 보정 후보를 붙였을 때 왜 틀렸는지 비교할 근거가 있는지

초록 원은 최종 `midi + centOffset`의 시각화라서 좋지만, 원인 분석에는 약간 부족합니다. 특히 지금 문제는 “보이는 음이 틀렸다”가 아니라 “왜 detector가 그쪽으로 확신했는가”라서 `clarity`, `rms`, `rawFrequencyHz`, frame history가 있어야 다음 접근을 고르기 쉽습니다.

그래서 제 판단은 이렇습니다.

1. **새 디버그 UI는 당장 불필요**
   지금은 초록 원과 practice panel의 raw frequency / clarity / RMS 표시만으로 1차 관찰을 해도 됩니다.

2. **다만 기록 가능한 계측은 있으면 좋음**
   화면을 보며 수동으로 “A#4로 튄다”를 확인하는 것과, 5초간 frame을 모아서 “A#3 입력 중 78%가 A#4, clarity 평균 0.91, RMS 0.04”처럼 보는 건 다음 판단의 질이 다릅니다.

3. **최소 구현이라면 console/table 정도**
   제품 UI를 늘리지 않고 `?pitchDebug=1`일 때만 최근 frame을 console에 남기거나 copy 가능한 JSON으로 모으는 정도면 충분합니다. 초록 원을 대체하는 게 아니라, 초록 원으로 본 현상을 수치로 붙잡는 용도입니다.

즉 “검증”만이면 초록 원으로 충분하고, “해결책 선택”까지 가려면 최소한의 로그가 필요합니다. 지금 단계에서는 먼저 초록 원 + 기존 panel 수치로 실제 A#3/A#4 사례를 몇 번 관찰하고, 패턴이 애매하면 그때 아주 작은 계측을 붙이는 순서가 좋아 보입니다.

> 그렇다면 디버그 모드가 필요하겠네요. 디버그 모드를 구현하고 원인을 구체적으로 분석해 봅시다.

| pYIN | 한 프레임에서 여러 주기 후보와 확률을 만들고 HMM으로 추적 | octave error와 짧은 오검출 억제에 검증됨 | 시간 추적 때문에 지연 및 빠른 이동 완화 가능 | 비교 대상으로 적합 |
| CREPE | 악기·음성 데이터로 학습한 신경망이 파형에서 F0를 직접 추정 | 강한 배음과 잡음에 대체로 강함, MIT | 모델 크기, 초기 로딩, 추론 비용 | 가장 유력한 안정성 후보 |
| SwiftF0 | 스펙트로그램과 경량 CNN으로 F0 추정 | CREPE보다 빠른 브라우저용 WASM/ONNX 데모, MIT | 2025년 공개로 장기 검증은 상대적으로 부족 | 유력한 실험 후보 |
| comb/mcomb | 후보 F0의 정수배 위치에 실제 배음들이 정렬되는지 평가 | 문제 구조와 직접 부합 | 잘못 구현하면 이전 후보 보정과 동일한 실패 가능 | 직접 구현보다 검증 라이브러리 비교용 |
| YIN/YINFFT | 파형의 반복 주기를 차분 함수로 검출 | 고전적이고 가벼움 | MPM과 같은 계열이라 개선 폭이 불확실 | 보조 기준선 |

pYIN은 단일 결과를 즉시 확정하지 않고 여러 후보를 확률적으로 유지한 뒤 HMM으로 경로를 선택합니다. Essentia 구현도 이 구조를 명시하고 있습니다. [pYIN 논문](https://www.eecs.qmul.ac.uk/~simond/pub/2014/MauchDixon-PYIN-ICASSP2014.pdf), [Essentia pYIN 문서](https://essentia.upf.edu/reference/std_PitchYinProbabilistic.html)

CREPE는 시간영역 파형에서 F0를 직접 추정하는 학습 기반 검출기로, 논문에서 pYIN과 SWIPE보다 같거나 높은 성능을 보고했습니다. 악기 데이터도 학습에 포함돼 이번 용도와 잘 맞습니다. [CREPE 논문](https://arxiv.org/abs/1802.06182), [공식 구현](https://github.com/marl/crepe)

SwiftF0는 더 최근의 경량 모델입니다. 46.875~2093.75Hz를 지원하며 브라우저에서 ONNX/WASM으로 실행되는 데모와 MIT 구현이 있습니다. 다만 자체 벤치마크 비중이 크므로 CREPE보다 우선 채택하기보다는 병렬 비교가 적절합니다. [SwiftF0 구현](https://github.com/lars76/swift_f0), [비교 벤치마크](https://github.com/lars76/pitch-benchmark)

라이선스도 중요합니다. CREPE와 SwiftF0는 MIT이지만, Essentia.js는 AGPL-3.0이고 aubio는 GPL-3.0이라 GitHub Pages 제품에 직접 의존시키려면 배포 정책 검토가 필요합니다. 따라서 Essentia/pYIN은 우선 실험·보고서 비교 대상으로 사용하는 편이 안전합니다. [Essentia.js 저장소](https://github.com/mtg/essentia.js/), [aubio 저장소](https://github.com/aubio/aubio)

제 판단으로는 다음 순서가 좋습니다.

1. `pitchy`를 유지한 상태에서 원시 마이크 파형을 WAV 또는 PCM으로 함께 기록합니다.
2. 동일 녹음을 `pitchy`, pYIN, CREPE, SwiftF0에 입력합니다.
3. octave error, 정상 옥타브 이동, 빠른 글리산도, 비브라토, 지연을 비교합니다.
4. 결과가 좋을 때만 검출기 backend를 교체합니다.

현재 `pitchtest.json`에는 `pitchy`가 이미 선택한 주파수만 있어 다른 검출기를 공정하게 재실행할 수 없습니다. 다음 구현은 보정 로직이 아니라 디버그용 원시 오디오 녹음과 검출기 비교 장치가 되어야 합니다. 현 단계의 1순위 후보는 안정성이 축적된 CREPE, 2순위는 가벼운 SwiftF0, 비교 기준은 pYIN입니다.

**전통 DSP 계열**

pYIN, aubio 같은 구현은 주로 C/C++ 또는 Python API로 제공됩니다. Python 패키지는 실제 알고리즘을 감싼 인터페이스인 경우가 많습니다.

브라우저 적용 방법은 다음 중 하나입니다.

- C/C++ 구현을 WebAssembly로 컴파일
- 이미 만들어진 WASM wrapper 사용
- 논문을 바탕으로 TypeScript로 재구현
- Essentia.js처럼 기존 C++ 엔진의 공식 WASM 배포 사용

직접 TypeScript로 옮길 수도 있지만 pYIN 전체에는 후보 확률 계산과 HMM 추적이 포함되므로 단순 YIN보다 범위가 큽니다. 기존 구현을 보면서 번역하면 라이선스 의무도 따라올 수 있어, “논문만 참고한 독립 구현”과 “소스 이식”을 명확히 구분해야 합니다.

**CREPE·SwiftF0 같은 ML 계열**

여기서는 Python 코드 자체보다 학습된 모델 가중치가 핵심입니다.

```text
Python 학습 모델
  -> ONNX 또는 TensorFlow.js 형식으로 변환
  -> 브라우저용 runtime에서 로딩
  -> Float32 PCM 입력
  -> F0 및 confidence 출력
```

브라우저 실행 후보는 다음과 같습니다.

- TensorFlow.js
- ONNX Runtime Web
- WebAssembly backend
- 지원 환경에서는 WebGPU backend

CREPE는 TensorFlow/Keras 모델이므로 TensorFlow.js 또는 ONNX로 변환할 수 있습니다. SwiftF0는 이미 ONNX/WASM 기반 브라우저 데모가 있어 그 경로를 조사하는 편이 유리합니다. 이 경우 Python은 모델 학습과 변환 도구로만 사용되고, 실제 SPRED 배포판에는 Python runtime이 포함되지 않습니다.

**SPRED에 적합한 구조**

각 adapter가 언어와 runtime 차이를 숨기게 됩니다.

```ts
type GamePitchDetectorBackend = {
  detect(samples: Float32Array, sampleRate: number):
    GamePitchDetection | Promise<GamePitchDetection>;
  dispose(): void;
};
```

예를 들면:

```text
PitchyBackend
  -> TypeScript에서 직접 실행

EssentiaPyinBackend
  -> Essentia WASM 호출

CrepeBackend
  -> TensorFlow.js 또는 ONNX Runtime Web 호출

SwiftF0Backend
  -> ONNX Runtime Web 호출
```

모델이 16kHz 입력을 요구하면 adapter가 현재 48kHz 마이크 PCM을 16kHz로 resample합니다. 비동기 추론, 모델 초기화, confidence 해석도 adapter 내부 책임으로 둡니다.

주의할 점은 모델 파일도 별도 저작물이라는 사실입니다. 코드가 MIT이더라도 다음을 각각 확인해야 합니다.

- 원본 구현 라이선스
- browser port 또는 wrapper 라이선스
- model weight 라이선스
- ONNX/TensorFlow.js 변환본의 배포 조건
- runtime 라이선스

따라서 권장 순서는 Python 구현을 TypeScript로 무작정 번역하는 것이 아니라 다음과 같습니다.

1. 공식 브라우저/WASM 경로가 있는지 확인
2. 공식 또는 재현 가능한 모델 변환 경로 확인
3. 라이선스와 모델 출처 확인
4. 작은 adapter로 동일 PCM 비교
5. 브라우저 경로가 없을 때만 독립 재구현 검토

이번 후보 중에서는 `pitchy`는 순수 TypeScript, pYIN은 Essentia WASM, CREPE는 TensorFlow.js/ONNX, SwiftF0는 ONNX/WASM 경로가 현실적입니다. Python 서버를 별도로 운영하는 방식은 GitHub Pages 단독 배포 원칙과 맞지 않으므로 현재 범위에서는 제외하는 것이 좋습니다.

> 그렇네요. 이미 사람들이 제작해둔 변환 경로와 인터페이스가 존재하겠군요.
> 일반적인 코드는 웹어셈블리를 이용한 인터페이스를 통해 다른 언어에서도 사용할 수 있게 변환되고, AI 모델의 경우 ONNX라는 도구가 이용되는 것인가요? 보고서 작성용으로 그 개념과 실제 코드에의 적용 방법을 명세에 적어두는 것이 좋겠습니다.

[3.4 명세](./docs/3.4-practice-pitch-detector-backend-spec.md)에 보고서용 개념과 실제 적용 방법을 추가했습니다.

핵심 구분은 다음과 같습니다.

- WebAssembly: C/C++ 등을 브라우저 실행 형식으로 컴파일한 결과
- JavaScript glue/binding: WASM 초기화, 메모리와 함수 호출을 TypeScript에 연결
- ONNX: 모델 graph와 weight를 전달하는 공통 형식
- ONNX Runtime Web: ONNX 모델을 WASM CPU 또는 WebGPU에서 실제 실행
- TensorFlow.js: Keras/TensorFlow 모델을 직접 브라우저 형식으로 변환하는 대안

또한 ONNX 변환만으로 resampling, framing, STFT, normalization, confidence threshold, F0 decoding 같은 Python 전후처리가 자동 이식되지 않는다는 점을 명시했습니다. 이 부분은 각 backend adapter가 TypeScript로 재현하고 원 구현과 수치 비교해야 합니다.

명세에는 WASM·ONNX adapter 의사 코드, 원본 언어별 적용 판단표, 정적 model/WASM asset 배포, 비동기 초기화, warm-up, resource 해제 기준도 포함했습니다. Python server와 Pyodide는 현재 GitHub Pages 및 실시간 성능 범위에서 제외했습니다.

기술 근거는 [Emscripten 공식 문서](https://emscripten.org/docs/), [ONNX 개요](https://onnx.ai/onnx/repo-docs/Overview.html), [ONNX Runtime Web](https://onnxruntime.ai/docs/get-started/with-javascript/web.html), [TensorFlow.js 변환 가이드](https://www.tensorflow.org/js/guide/conversion)를 기록했습니다. 소스 코드를 가져온 것은 아니며 개념과 구조만 참고했습니다.

HARNESS에도 이 문서가 WASM/ONNX 이식 경계와 보고서 서술 기준을 담당한다고 연결했습니다. `git diff --check`도 통과했습니다.

> 다음 작업에 이어서 들어가면 되겠습니다. 실제 적절한 변환 경로가 있는지 탐색하고 적용하는 작업인가요?

네. 다만 바로 외부 backend를 붙이기 전에 명세의 Phase A를 먼저 마쳐야 합니다.

진행 순서는 다음과 같습니다.

1. 공통 인터페이스 보완
   - backend별 `isConfident`
   - 비동기 `detect()`
   - model 초기화 실패와 resource 해제
   - 서로 다른 frame size와 sample rate 지원

2. 원시 PCM/WAV 캡처
   - 실제 detector에 들어가는 mono PCM 저장
   - 마이크 설정과 sample rate metadata 기록
   - 최대 60초, debug 전용, 외부 전송 없음

3. 동일 입력 comparison runner
   - 저장한 WAV를 각 backend에 같은 frame/hop 조건으로 입력
   - F0, confidence, 처리 시간과 오류 구간을 JSON으로 출력
   - 실시간 연주를 반복하지 않고 공정하게 비교

4. 후보별 변환 경로 조사 및 적용
   - pYIN: Essentia.js/WASM API와 배포 파일 확인
   - aubio: 유지되는 browser WASM wrapper와 알고리즘 노출 범위 확인
   - CREPE: 검증 가능한 TensorFlow.js 또는 ONNX 변환본 확인
   - SwiftF0: 공식 ONNX model과 browser demo의 전처리·후처리 확인
   - 코드, wrapper, model weight, runtime 라이선스를 각각 기록

5. backend adapter 구현
   - 공식 경로가 있으면 그대로 감싸기
   - 비공식 변환본은 출처와 원 모델 일치 여부 검증
   - 지원되지 않는 operator나 누락된 전처리가 있으면 채택하지 않음
   - 브라우저 경로가 없을 때만 직접 변환 또는 독립 구현 검토

즉 다음 작업은 단순히 “Python 라이브러리를 설치해 본다”가 아니라, 먼저 비교 기반을 완성한 후 각 후보의 공식적이고 재현 가능한 브라우저 실행 경로를 조사하고 하나씩 adapter로 연결하는 작업입니다.

바로 이어갈 구현 대상은 공통 인터페이스의 비동기·confidence 계약 보완과 PCM/WAV 캡처입니다. 그 기반이 있어야 변환 경로가 실제로 정확한지 같은 입력으로 검증할 수 있습니다.

> 비동기가 의미하는 바를 잘 모르겠네요. javascript는 비동기 언어라 여러 작업을 엄밀하게 동시에 병렬 처리할 수는 없죠. 여기서의 비동기도 그걸 의미하나요? pitch dection을 각 백엔드 후보가 같이 수행하되 어떤 순서로 처리할 것인지를 말인가요?

여기서 말한 비동기는 “여러 pitch backend를 동시에 실행한다”는 의미가 아닙니다. 하나의 backend가 모델 로딩이나 추론 결과를 즉시 반환하지 못할 수 있으므로, 완료될 때까지 기다릴 수 있는 호출 형태를 뜻합니다.

현재 `pitchy`는 동기식입니다.

```ts
const result = backend.detect(samples, sampleRate);
// 이 줄이 끝났을 때 result가 이미 존재
```

ONNX Runtime Web 등은 보통 비동기식입니다.

```ts
const result = await backend.detect(samples, sampleRate);
// 추론이 완료된 뒤 다음 줄로 진행
```

`await`하는 동안 JavaScript 메인 스레드가 반드시 계산을 직접 수행하고 있다는 뜻은 아닙니다. 내부 구현에 따라 WASM worker, WebGPU, 브라우저 작업 큐 등이 계산을 수행할 수 있고, JavaScript는 완료 통지를 기다립니다.

다음 세 개념을 구분해야 합니다.

- 비동기: 호출 결과가 나중에 `Promise`로 도착
- 동시성: 여러 작업이 진행 중인 상태를 관리
- 병렬성: 여러 CPU thread나 GPU가 엄밀히 동시에 계산

비동기 API라고 자동으로 병렬 실행되는 것은 아닙니다. 반대로 JavaScript API가 하나여도 WebGPU나 multithread WASM 내부에서는 병렬 계산이 가능할 수 있습니다.

SPRED 제품 실행에서는 한 번에 선택된 backend 하나만 사용합니다.

```text
마이크 frame
  -> 선택된 backend 하나
  -> await detection
  -> GamePitchFrame 생성
  -> 다음 마이크 frame 처리
```

여러 backend를 같이 실행하는 것은 동일 입력 비교 도구에서만 필요합니다. 그 경우도 처음에는 순차 실행하는 것이 안전합니다.

```ts
for (const backend of backends) {
  const result = await backend.detect(samples, sampleRate);
  comparisonResults.push(result);
}
```

이렇게 하면 실행 순서는 `pitchy → pYIN → CREPE → SwiftF0`처럼 고정되지만, backend별 처리 시간이 합산됩니다. 오프라인 비교이므로 실시간일 필요는 없습니다.

제품의 실시간 루프에서는 느린 추론이 끝나기 전에 다음 frame을 겹쳐 실행하지 않습니다.

```ts
if (detectionInProgress) {
  return; // 이번 frame은 건너뜀
}

detectionInProgress = true;

try {
  const result = await backend.detect(samples, sampleRate);
  consume(result);
} finally {
  detectionInProgress = false;
}
```

이 제한이 없으면 60fps 입력에 40ms 추론 backend를 연결했을 때 처리되지 않은 Promise가 계속 쌓이고, 오래된 결과가 최신 결과보다 늦게 도착하는 문제가 생길 수 있습니다.

또한 모델은 사용 전에 별도로 비동기 초기화됩니다.

```ts
const backend = await createCrepeBackend();
// model 파일 다운로드와 ONNX session 생성 완료

startPitchInput(backend);
```

정리하면:

- 제품에서는 선택된 backend 하나만 실행
- `async`는 모델 로딩과 추론 완료를 기다리기 위한 계약
- 이전 frame 추론 중에는 새 추론을 겹치지 않음
- 여러 backend 비교는 저장된 오디오를 대상으로 순차 실행
- Web Worker, WASM thread, WebGPU를 사용해야 실제 병렬 계산이 발생할 수 있음

따라서 이번 인터페이스의 비동기 지원은 backend 실행 순서를 정하기 위한 기능이 아니라, ONNX/WASM 모델이 결과를 나중에 반환해도 마이크 루프가 안전하게 기다릴 수 있도록 하는 장치입니다.

> 이해했습니다. 백엔드간 작업 수행 속도 차이로 인해 생기는 순서 꼬임을 막기 위한 방어 장치를 둔다는 것이군요?

네, 정확합니다.

조금 더 엄밀히 말하면 두 상황을 방어합니다.

- 같은 backend에서 이전 프레임의 느린 추론이 끝나기 전에 다음 프레임 추론이 시작되어 결과 순서가 뒤집히는 문제
- backend마다 초기화·추론 시간이 달라 비교 결과가 서로 다른 입력 시점을 가리키는 문제

제품에서는 선택된 backend 하나에 대해 추론을 한 번만 진행하고, 완료 후 다음 프레임을 처리합니다. 오래 걸리면 중간 프레임은 건너뛰되 오래된 작업을 쌓지는 않습니다.

비교 도구에서는 동일한 녹음의 동일한 프레임 번호와 timestamp를 backend별 결과에 붙입니다. 처리 완료 순서가 달라도 나중에 같은 입력 시점끼리 정확하게 비교할 수 있습니다.

즉 비동기 지원의 목적은 병렬화를 적극적으로 하기보다, 느린 작업 때문에 결과의 시간 순서와 입력 대응 관계가 흐트러지는 것을 막는 데 있습니다.

> 그런데 이는 pitch detection 실험을 후보별로 따로따로 진행할 경우에도 필요한 조치인가요?

후보별로 완전히 따로 실험한다면, backend 사이의 완료 순서가 뒤섞이는 문제는 없습니다. 한 번에 하나만 실행하기 때문입니다.

그래도 비동기 대응은 필요합니다. 이유는 backend 간 경쟁이 아니라 같은 backend 내부의 프레임 순서 때문입니다.

예를 들어 CREPE가 한 프레임을 처리하는 데 40ms가 걸리는데 마이크 루프가 16ms마다 호출되면:

```text
frame 1 추론 시작
frame 2 추론 시작
frame 3 추론 시작
frame 2 완료
frame 1 완료
frame 3 완료
```

후보를 하나만 시험해도 결과가 입력 순서와 다르게 도착할 수 있습니다. 따라서 실시간 실험에서는 다음 중 하나가 필요합니다.

- 현재 추론이 끝날 때까지 다음 입력을 시작하지 않기
- 최신 프레임 하나만 대기시키고 오래된 프레임 버리기
- sequence 번호를 붙이고 늦게 도착한 과거 결과 무시하기

반면 저장된 WAV를 후보별로 오프라인 분석하면서 각 프레임을 `await`하고 다음 프레임으로 넘어간다면 별도의 순서 방어가 거의 필요 없습니다.

```ts
for (const frame of recordedFrames) {
  const result = await backend.detect(frame.samples, sampleRate);
  results.push(result);
}
```

정리하면:

- 후보별 오프라인 실험: 단순 순차 `await`면 충분
- 후보별 실시간 마이크 실험: 같은 backend 안에서 프레임 중첩 방지 필요
- 여러 backend 동시 비교: backend별 순서와 동일 timestamp 대응까지 추가 관리 필요

이번 계획은 먼저 동일 WAV를 후보별로 순차 분석하므로 복잡한 병렬 제어까지 만들 필요는 없습니다. 공통 인터페이스가 `Promise`를 받을 수 있게 하고 comparison runner에서 한 프레임씩 `await`하면 충분합니다. 실시간 backend로 채택할 후보가 나온 뒤에만 최신 프레임 우선 정책을 적용하는 것이 범위상 적절합니다.

> 마이크의 입력 속도와 백엔드의 연산 속도 차이도 고려해야 하는군요. 이해했습니다. 말씀하신대로 보완을 진행합시다.

- 공식 저장소와 ONNX 모델이 MIT

다만 구현 전에 확정할 구조 문제가 하나 발견됐습니다.

현재 SPRED backend는 다음 구조입니다.

```text
한 frame 입력 → pitch 결과 하나
```

SwiftF0와 pYIN은 다음 구조입니다.

```text
전체 또는 가변 길이 오디오 입력 → 자체 timestamp를 가진 pitch contour 배열
```

SwiftF0를 현재 계약에 억지로 넣으면 한 입력에서 나온 여러 pitch 중 무엇을 반환할지 임의로 골라야 합니다. 비교 결과도 원 알고리즘과 달라질 수 있습니다.

제가 권하는 방향은 두 계약을 분리하는 것입니다.

- `GamePitchDetectorBackend`: 제품의 실시간 단일 pitch 경로
- `GamePitchContourBackend`: 저장된 WAV를 분석하는 offline 배열 경로
