# 1.27. Step5 레이아웃 저장 제약조건과 Cleanup 메모

## 1. 목적

이 메모는 첫 layout preset UI 실전 테스트 이후 진행한 안정화 작업을 기록한다.

이 메모는 다음 문서를 보완한다.

- `docs/1.3-score-json-format.md`
- `docs/2.4-layout-edit-ui-spec.md`
- `docs/2.5-layout-preset-format-spec.md`
- `docs/3.1-extension-roadmap.md`

이 문서는 구현 해설용 메모이다.
저장 제한과 활성 동작 규칙은 active spec 문서를 따른다.

## 2. 구현된 안정화 범위

구현된 범위는 다음과 같다.

- layout preset 이름 길이 제한
- layout preset JSON byte size 제한
- score JSON file load byte size 제한
- local score save byte size 제한
- score cell `rawText` 길이 검증
- 유효하지 않은 `rawText`가 저장되기 전 edit input 단계에서 차단
- tuplet slot input과 조합된 tuplet `rawText` guard
- layout preset 파일명 규칙 단순화
- score-load-time layout snapshot 기반 Default Layout 복원
- whole-score deep clone 대신 layout apply 구조 공유 적용
- layout apply 또는 layout preset apply 이후 playback runtime reset
- 오래된 미사용 layout preset index와 generic JSON helper code 제거
- layout dialog binding을 `src/app/layout/layout_dialog_binding.ts`로 추출

## 3. 현재 제한값

현재 구현은 다음 제한값을 사용한다.

| 대상 | 제한 | 경계 |
| --- | ---: | --- |
| layout preset display name | 30글자 | UI 입력과 preset 생성/저장 검증 |
| layout preset JSON | UTF-8 기준 256 KiB | file load/save와 local slot save/load 검증 |
| score JSON file load | UTF-8 기준 8 MiB | file import parsing 경계 |
| local score save | UTF-8 기준 3 MiB | localStorage score save 경계 |
| score cell `rawText` | 1..100글자 | score validation과 edit composition |
| tuplet slot input | 30글자 | UI 입력 경계 |
| 조합된 tuplet `rawText` | 100글자 | edit composition 경계 |
| local layout preset slots | `instrumentPresetId`별 3개 | localStorage key 정책 |

Local score save 제한이 file-load 제한보다 낮은 이유는 브라우저 `localStorage` quota가 score data, layout preset, 같은 origin의 다른 data와 공유되기 때문이다.

큰 JSON 파일을 runtime memory에 load하는 것 자체가 `localStorage` quota를 줄이지는 않는다.
다만 앱이 score를 parse, validate, analyze, render하는 동안 runtime memory pressure는 증가할 수 있다.

## 4. 검증 책임

검증은 data boundary별로 의도적으로 분리한다.

- `src/core/score/score_limits.ts`
  - score/file byte helper와 score 중심 limit constant를 제공한다.
- `src/core/score/json_load.ts`
  - oversized score JSON text를 parse 전에 거부한다.
- `src/core/score/score_validate.ts`
  - 저장된 `ScoreFile` 구조와 저장된 cell `rawText`를 검증한다.
- `src/infra/score_local_storage.ts`
  - `localStorage.setItem()` 전에 oversized local score save를 거부한다.
- `src/app/edit/edit_apply.ts`
  - oversized `rawText`를 쓰게 되는 note/global edit command를 차단한다.
- `src/app/edit/edit_tuplet.ts`
  - 저장 cell limit을 초과하는 finalized tuplet text를 차단한다.
- `src/app/layout/layout_draft.ts`
  - preset 또는 score mutation이 만들어지기 전에 layout draft mutation rule을 검증한다.
- `src/app/layout/layout_preset.ts`
  - `UserLayoutPresetData`, preset display name length, preset JSON byte size, preset file name을 검증한다.
- `src/infra/layout_preset_storage.ts`
  - 고정 local slot key와 localStorage read/write error 변환을 담당한다.

이 분리는 score-file integrity check를 core score layer에 두고, layout-preset-specific rule을 app layout layer에 남긴다.

## 5. 재사용과 캡슐화 판단

Byte-length helper와 score 중심 constant는 `core/score`에 둔다.
Score file import, score local storage, edit write guard가 같은 저장 score limit에 의존하기 때문이다.

`app/layout`은 일관된 UTF-8 byte 측정이 필요할 때 이 작은 pure helper를 import할 수 있다.
반대 방향은 허용하지 않는다.
즉 `core`는 `app`, `infra`, DOM, layout dialog code에 의존하면 안 된다.

비슷해 보이는 검증을 항상 공유하지는 않는다.
예를 들어 `ScoreFile` row/cell validation과 `UserLayoutPresetData` row validation은 서로 다른 저장 포맷을 보호하므로 분리한다.
낮은 수준의 byte helper만 공유하면 domain contract 결합을 피할 수 있다.

## 6. Layout Apply 판단

Layout apply는 이제 구조 공유를 통해 next `ScoreFile`을 만든다.

Apply path가 교체하는 항목은 다음으로 제한한다.

- `instData`
- `layout.rowDefinitions`
- 삭제된 row reference를 제거해야 할 때 영향을 받는 track cell array

변경되지 않은 metadata, global lines, 영향을 받지 않은 tracks는 재사용한다.
이 방식은 immutable app-state replacement를 유지하면서 whole-score deep clone 비용을 피한다.

Layout apply 이후에도 app은 runtime artifact를 다시 만든다.

```text
ScoreFile
-> RuntimeDocument
-> ParsedScoreDocument
-> AnalysisResult
-> CanvasRenderInput
-> render
-> playback runtime reset
```

현재 단계에서는 이 방식이 적절하다.
Layout 변경은 renderer coordinate, hit testing, audio scroll reference behavior에 영향을 주기 때문이다.
세밀한 parse/analyze/render invalidation은 후속 성능 확장으로 남긴다.

## 7. Cleanup 완료

제거하거나 scope를 줄인 항목은 다음과 같다.

- 오래된 local preset display-name duplicate helper
- 오래된 index 기반 local preset type
- 미사용 generic `downloadJsonFile()` / `serializeJsonValue()` helper
- 미사용 layout row import
- 미사용 layout dialog parameter
- 미사용 playback local variable
- module boundary를 넘을 필요가 없었던 preset validation public export

Layout dialog와 toolbar preset binding은 이제 `layout_dialog_binding.ts`에 있다.
`app_view_binding.ts`는 general view control 용도로 남긴다.

## 8. 검증

안정화 작업은 다음 명령으로 검증했다.

```text
npx tsc --noEmit --noUnusedLocals --noUnusedParameters
npm run typecheck
npm run test:score
npm run test:edit
npm run test:layout
npm run build
```

이 메모 작성 시점에 모든 명령이 통과했다.

## 9. 다음 권장 작업 순서

다음 작업 순서는 다음과 같다.

1. 문서화와 보고서 정리
2. track layer UI/edit/render/playback filtering
3. 기본 audio verification checklist와 listening pass
4. local playback path가 안정된 뒤 YouTube mode

트랙 레이어 작업은 기존 고정 `basic`, `optional`, `extra` track 위의 UI와 filtering부터 시작하는 것이 좋다.
첫 구현에서 score track format을 바꿀 필요는 없다.
