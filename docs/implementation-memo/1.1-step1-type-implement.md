# 1.1. 1단계 타입 구현 메모
> 본 설계 메모 문서 1.1에서는 1.0에서 정한 로드맵 1단계에 대한 설명을 기록한다.
> 코드에 대한 설명은 코드 파일 자체에 JavaDoc 주석을 달아 그것을 중심으로 한다.
> 세부 설명과 판단 근거의 작성이 필요할 때만 본 문서에 기록한다. 

## 1. 개요
core 내부의 핵심 모듈 `score`, `parse`, `analyze`에서 사용하는 타입 파일에 대한 구현 메모이다.

현재 작성된 타입 파일은 다음과 같다.

- `regression-code/src/core/score/types.ts`
  - 악보 JSON 저장 구조
  - 로드 직후 런타임 인덱스 구조
- `regression-code/src/core/parse/types.ts`
  - note/global parser 산출물
  - parser 입력과 문서 단위 parsed 결과
  - parser 함수 시그니처와 캐시 키
- `regression-code/src/core/analyze/types.ts`
  - analyzer 이벤트
  - 전역 timing/dynamics timeline
  - analyzer 진단, 부분 분석 요청, 파생 캐시

이 문서는 active specification을 대체하지 않는다. 타입 구현 중 생긴 판단 근거와 구현 메모만 보충한다.


## 2. 필요한 JavaScript / TypeScript 개념

### 2.1. 변수, 클래스 선언

참고 : https://inpa.tistory.com/entry/TS-📘-타입스크립트-타입-선언-종류-💯-총정리

구현에 있어 필수적인 선언 방식을 짚고 넘어간다. 

- 기초
    - const와 let으로 기본적 상수와 변수를 선언할 수 있다.
    - JavaScript에서는 C와 유사한 각종 타입들을 지원한다.
        - number : 정수와 실수를 뭉뚱그려 취급한다.
        - string : 작은 따옴표도 인정하며, 백틱으로 표기시 템플릿 문자열 `...{변수}...`을 사용할 수도 있다.
        - array  : 기본적으로 동적으로 생성된다. 두가지 방법으로 선언 가능하다. 개인적으로 전자를 선호한다. 정적으로 생성하려면 new 연산자와 생성자를 이용해야 한다.

        ```ts
        let fruits: string[] = ['Apple', 'Banana', 'Mango'];
        // or
        let fruits: Array<string> = ['Apple', 'Banana', 'Mango'];

        // 정적 배열 생성
        const staticArray = new Array(10);
        ```

- 고급 타입
    - TypeScript는 객체 구조를 표현할 때 `type alias`와 `interface`를 모두 지원한다.
    - 현재 타입 파일에서는 `type`을 우선 사용한다. 이유는 union type, literal type, template literal type과 함께 쓰기 쉽고, 현재 설계 문서의 예시도 `type` 중심으로 작성되어 있기 때문이다.
    - `interface`는 객체 구조 확장에 장점이 있지만, 현재 단계에서는 discriminated union을 많이 쓰므로 `type`이 더 일관적이다.

    ```ts
    type ScoreFormat = {
      formatName: string;
      version: string;
    };
    ```

    - 리터럴 : 문자열이나 숫자 상수 자체를 타입으로 지정할 수도 있다.

    ```ts
    type TrackId = "basic" | "optional" | "extra";
    ```

### 2.2. 이번 타입 구현에서 사용한 주요 TypeScript 문법

이번 타입 파일은 단순한 객체 타입뿐 아니라, 파서와 analyzer의 분기 구조를 안전하게 표현하기 위해 몇 가지 TypeScript 문법을 사용한다.

#### 2.2.1. String literal union

정해진 문자열 값 중 하나만 허용하려고 할 때 사용한다.

```ts
type GlobalKind =
  | "bpm"
  | "beatsPerBar"
  | "stepsPerBeat"
  | "dynamics";
```

이렇게 두면 `"bpmm"` 같은 오타를 타입 단계에서 막을 수 있다.
`TrackId`, `GlobalKind`, `ParsedCellKind`, `AnalyzedEventKind`가 이 방식이다.

#### 2.2.2. Discriminated union

여러 종류 중 하나로 갈라지는 구조를 표현할 때 사용한다.
각 타입은 공통 구분자 필드를 가진다.

```ts
type RowDefinition =
  | GlobalRowDefinition
  | NoteRowDefinition
  | GapRowDefinition;

type NoteRowDefinition = {
  type: "note";
  rowId: RowId;
  midi: number;
};
```

후속 코드에서는 `row.type`을 검사하면 TypeScript가 해당 타입을 자동으로 좁혀준다.
`ParsedCell`, `ParsedGlobalCell`, `AnalyzedEvent`도 같은 패턴이다.

이 구조를 쓰면 `kind === "note"`인데 note 전용 필드가 없는 식의 불가능한 상태를 줄일 수 있다.

#### 2.2.3. Record

고정된 키 집합에 같은 종류의 값을 매핑할 때 사용한다.

```ts
type ScoreDifficulty = Record<TrackId, number>;
```

이는 아래 구조와 같은 의미이다.

```ts
type ScoreDifficulty = {
  basic: number;
  optional: number;
  extra: number;
};
```

트랙 세 종류가 고정되어 있고 각 트랙 난이도가 항상 존재해야 하므로 `Record`가 적합하다.

#### 2.2.4. Map

런타임에서 빠른 조회가 필요한 인덱스에는 `Map`을 사용한다.

```ts
type ScoreIndexes = {
  rowById: Map<RowId, RowDefinition>;
  trackById: Map<TrackId, Track>;
};
```

JSON 저장 구조는 배열 중심이지만, parser와 analyzer는 특정 row, track, col을 반복 조회한다.
따라서 로드 이후에는 `ScoreIndexes` 같은 Map 기반 구조를 따로 만든다.

#### 2.2.5. Template literal type

문자열 조합으로 만든 key의 형식을 타입으로 표현할 때 사용한다.

```ts
type CellCoordKey = `${RowId}|${number}`;
type ParsedCellCacheKey = `${TrackId}|${RowId}|${number}`;
```

이는 좌표 key가 어떤 순서로 만들어지는지 문서화하는 효과도 있다.
다만 실제 구현에서는 key 생성 함수를 하나로 고정하여 구분자나 순서가 섞이지 않게 해야 한다.

#### 2.2.6. import type

타입만 가져올 때는 `import type`을 사용한다.

```ts
import type { ScoreFile, ScoreIndexes } from "../score/types";
```

현재 타입 파일들은 런타임 값이 아니라 타입 계약만 공유한다.
따라서 `import type`을 쓰면 값 import와 타입 import의 의도가 분명해진다.

#### 2.2.7. optional과 null

필드가 아예 없을 수 있는 경우에는 `?`를 사용한다.

```ts
type InstrumentString = {
  openMidi?: number;
};
```

필드 자체는 항상 존재하지만 값이 없음을 명시해야 하는 경우에는 `null`을 사용한다.

```ts
type ParsedModifiers = {
  gliss: ParsedGliss | null;
};
```

현재 기준은 다음과 같다.

- 저장 JSON에서 조건에 따라 필드가 없을 수 있으면 optional
- parser/analyzer 결과에서 슬롯은 있지만 modifier가 없다는 식의 상태를 명시해야 하면 `null`

#### 2.2.8. Extract

이미 정의한 union type 중 일부만 골라서 새 타입처럼 쓸 때 사용한다.

```ts
type ParsedLinearGlobalCell = {
  globalKind: Extract<GlobalKind, "bpm" | "dynamics">;
};
```

`GlobalKind` 전체 중 선형 변화 토큰을 허용하는 것은 `bpm`, `dynamics`뿐이다.
이를 별도 문자열 union으로 다시 쓰는 대신 `Extract`로 기존 타입에서 뽑아 쓰면 원래 타입과의 관계가 분명해진다.

#### 2.2.9. Generic type

함수 결과 구조는 같지만 내부 값의 타입만 달라질 때 generic을 사용한다.

```ts
type ParsePartResult<T> =
  | { ok: true; value: T; nextIndex: number }
  | { ok: false; error: ParseError };
```

`parseGliss`, `parseTrem`, `parseMicroPitch`처럼 하위 parser들은 성공/실패 구조가 같고 성공 시 value 타입만 다르다.
따라서 `ParsePartResult<T>`로 묶어둘 수 있다.

## 3. 현재 구현 판단

### 3.1. 타입 파일 우선 작성

1단계에서는 알고리즘 구현 전에 `score`, `parse`, `analyze` 타입 파일을 먼저 작성한다.
이는 이후 구현될 모듈들이 동일한 입력/출력 계약을 공유해야 하기 때문이다.

### 3.2. common.types.ts 보류

`TrackId`, `GlobalKind`, `RowId`, `StringId` 같은 타입은 여러 모듈에서 사용될 가능성이 있다.
다만 현재는 `score/types.ts`가 저장 구조와 인덱스의 출발점이므로, 먼저 score 모듈에 두고 필요한 곳에서 import한다.

세 타입 파일 작성이 안정된 뒤 실제로 여러 모듈에서 반복 사용하는 원자 타입만 `common.types.ts`로 옮긴다.

### 3.3. parser와 analyzer 캐시 타입의 위치

parser 캐시 키는 `parse/types.ts`에 둔다.
analyzer 캐시는 공식 저장 구조가 아니라 `AnalysisResult`로부터 재생성 가능한 파생 구조이므로 `analyze/types.ts`에 둔다.
