**현재 작업 내용**

`회귀 코드`는 `regression-code/`를 중심으로 이미 상당 부분 구현이 진행된 상태입니다. 단순 파서 이식이 아니라, 기존 Google Sheets + Apps Script 기반 `이세계 코드`를 TypeScript/HTML/CSS 웹 앱으로 재구성하면서 다음 흐름이 연결되어 있습니다.

1. `ScoreFile` 로드/검증
2. runtime document/index 생성
3. note/global cell 파싱
4. analyzer event 생성
5. canvas renderer 표시
6. edit mode를 통한 score mutation
7. Web Audio 기반 재생/seek
8. layout edit/apply/preset 저장

현재 초점은 **사용자 커스텀 레이아웃 수정 기능**입니다. 관련 활성 문서는 `docs/2.4-layout-edit-ui-spec.md`, `docs/2.5-layout-preset-format-spec.md`이고, 구현 메모는 `docs/implementation-memo/1.26-step5-layout-draft-apply-preset.md`입니다.

현재 구현된 레이아웃 기능은 대략 다음 범위입니다.

- `Modify` 버튼으로 Layout dialog 열기
- 현재 `ScoreFile`에서 layout draft 생성
- note/gap row 목록 편집
- 공통 note height 수정
- gap height 수정 및 preview drag resize
- note/gap row 추가/삭제
- 삭제될 track cell 계산
- cell 삭제가 생기면 사용자 확인 후 Apply
- Apply 후 `ScoreFile -> runtime -> parse -> analyze -> render` full rebuild
- Local Slot 1..3 저장/불러오기
- layout preset JSON 파일 저장/불러오기
- 상단 layout toolbar에서 `Default Layout`, `Slot 1..3` 선택
- `npm run test:layout` 추가

현재 변경 파일 상태를 보면 이 작업은 아직 커밋되지 않은 진행 중 작업입니다. 특히 `app_view_binding.ts`에 layout dialog와 toolbar 관련 로직이 많이 들어가 있고, 새 파일로는 `layout_apply.ts`, `layout_preset.ts`, `layout_preset_storage.ts`, `test_layout_apply.ts`, `docs/2.5-layout-preset-format-spec.md` 등이 추가되어 있습니다.

**앞으로 예정된 작업**

가장 가까운 다음 단계는 기능 확장보다 **안정화, 리팩토링, 보고서화**로 보입니다.

1. 레이아웃 기능 검증
   `npm run typecheck`, `npm run test:layout`, `npm run test:score`, `npm run test:parse`, `npm run test:edit`, `npm run build` 기준으로 현재 상태를 다시 확인해야 합니다. 필요하면 브라우저에서 수동으로 Local/File preset round trip, Apply 후 삭제 확인, toolbar slot 적용까지 확인하는 것이 좋습니다.

2. `app_view_binding.ts` 리팩토링
   현재 layout dialog, preview, local/file preset, toolbar preset logic이 `app_view_binding.ts`에 커져 있습니다. 보고서 전후로 `regression-code/src/app/layout/layout_dialog_binding.ts` 같은 전용 모듈로 분리하는 것이 자연스러운 다음 리팩토링입니다.

3. 레이아웃 저장 정책 정리
   현재 정책은 “score file은 기본 레이아웃을 저장하고, 사용자 프리셋은 별도 local/file preset으로 저장”입니다. 이 구분을 보고서와 문서에서 명확히 써야 합니다. zoom/scroll/font size는 score layout과 별개입니다.

4. 보고서 작성
   보고서에는 다음 흐름을 중심으로 정리하면 됩니다.
   - 기존 `이세계 코드`의 한계: 스프레드시트 의존, UI/렌더/재생 확장성 한계
   - `회귀 코드`의 구조: parser/analyzer/renderer/audio/UI state 분리
   - 이번 작업의 핵심: 사용자 레이아웃 draft 편집, Apply 시 score rebuild, destructive cell deletion confirmation
   - 저장 방식: Score JSON과 Layout Preset JSON 분리
   - 검증: TypeScript build/test와 layout apply test
   - 남은 과제: layout UI binding 분리, undo, 더 정교한 import reconciliation, audio effect 품질 개선

5. 이후 기능 후보
   문서상 deferred 항목은 dynamics automation, tuplet timing refinement, loop playback, Tone.js 또는 sampled backend 검토, tuplet hit-test UX 개선, undo/pending edit grouping, GitHub Pages용 production build 정리입니다.

그런데 사용자가 layout editor에서 `family`, `instName`, row 구성을 바꿀 수 있는 상황에서는 `presetId`가 “현재 악기 설정의 고유 ID”처럼 동작하지 않습니다. 그렇다고 UI에 노출되는 식별자도 아니고요.

제 판단은 이렇습니다.

1. 지금 단계에서 `presetId`를 사용자 커스텀 레이아웃의 고유 식별자로 쓰면 안 됩니다.
2. `family` 변경에 따라 `presetId`를 자동 변경하는 것도 보류하는 편이 낫습니다.
3. 현재 v1 포맷에서는 `presetId`를 “원본/기준 악기 프리셋 ID” 정도로 축소 해석하는 게 안전합니다.
4. 사용자 레이아웃의 실제 식별자는 `layoutPresetId`와 `layoutPresetDisplayName`이 맡는 게 맞습니다.
5. 장기적으로는 `instData.presetId`를 `basePresetId` 또는 `instrumentProfileId` 같은 이름으로 바꾸는 포맷 v2 개편을 검토하는 게 좋습니다.

즉 지금은 제거보다는 **의미를 축소해서 고정 내부 필드로 남기는 것**을 추천합니다. 다만 문서와 코드에서는 `instrumentPresetId`라는 표현이 “현재 악기 설정의 ID”처럼 읽히므로, 설명을 `base instrument preset id` 또는 `source preset id` 쪽으로 정리하는 게 좋겠습니다.

구현상 바로 손볼 만한 부분은 다음입니다.

- layout editor에서 `family`를 바꿔도 `presetId`는 바꾸지 않는다.
- local slot key는 일단 기존 `presetId` 기준을 유지한다.
- 문서에는 `presetId`가 사용자 레이아웃 고유 ID가 아니며, 커스텀 레이아웃 식별은 `layoutPresetId`가 담당한다고 명시한다.
- file load에서 `instrumentPresetId` 불일치는 강한 오류가 아니라 참고 경고 정도로 유지한다.

> 아니면 layoutPresetId를 presetId에 저장하는 것은 어떻습니까?

가능은 하지만, 저는 권하지 않습니다.

`layoutPresetId`를 `instData.presetId`에 저장하면 두 개념이 섞입니다.

- `instData.presetId`: 악기 계열/기준 프리셋 식별자
- `layoutPresetId`: 사용자가 저장한 특정 레이아웃 프리셋 식별자

특히 현재 로컬 슬롯 key가 `layout-slot:{instrumentPresetId}:1..3` 구조라서, `instData.presetId`를 `slot-1` 같은 `layoutPresetId`로 바꾸면 문제가 생깁니다. 예를 들어 `otamatone-basic` 기준 slot을 적용한 뒤 `presetId`가 `slot-1`이 되면, 다음부터는 `layout-slot:slot-1:1..3`을 찾게 되어 기존 슬롯 목록이 끊길 수 있습니다.

더 안전한 방향은 이렇습니다.

1. `instData.presetId`는 당분간 `basePresetId`처럼 해석한다.
2. `layoutPresetId`는 layout preset 데이터 안에서만 유지한다.
3. 현재 score에 적용된 layout preset을 기억해야 한다면 별도 필드를 둔다.

예를 들면 후속 포맷에서:

```ts
instData: {
  presetId: "otamatone-basic",
  family: "otamatone",
  instName: "My Layout",
  ...
}

layout: {
  appliedLayoutPresetId?: "layout-...",
  rowDefinitions: [...]
}
```

또는:

```ts
instData: {
  basePresetId: "otamatone-basic",
  appliedLayoutPresetId: "layout-...",
  ...
}
```

지금 당장 큰 포맷 변경 없이 가려면, `presetId`는 그대로 두고 문서에서 의미를 “기준 악기 프리셋 ID”로 축소하는 편이 가장 안전합니다. `layoutPresetId`를 `presetId`에 덮어쓰는 방식은 나중에 저장소 key, 파일 호환성 경고, Default Layout 복귀 동작까지 혼란을 만들 가능성이 큽니다.

> 현재는 악보의 복사본을 생성하고 그 위에 draft를 덮어쓰는 방식이죠?
> 악보의 길이가 매우 길어지면 복사본 생성에도 오랜 시간이 걸릴 것 같습니다. 기능 특성상 사용자가 다소 기다리더라도 불만을 가지지는 않을 것 같지만 일단 검토해볼까 합니다. 원본에 바로 덮어쓰면 어떻게 되나요?

맞습니다. 현재는 `applyLayoutDraftToScore()`에서 `cloneScoreFile(score)`로 전체 `ScoreFile` 복사본을 만든 뒤, 그 복사본의 `instData`, `layout.rowDefinitions`, `tracks[].cells`를 교체하고 마지막에 `validateScoreFile()`을 통과하면 새 상태로 반영합니다.

원본에 바로 덮어쓰는 방식은 성능상 유리할 수 있지만, 현재 구조에서는 권하지 않습니다.

가장 큰 문제는 **실패 복구**입니다. layout apply는 다음 이유로 실패할 수 있습니다.

- 새 layout row 검증 실패
- rowId 중복
- note MIDI 범위 오류
- 삭제 확인 취소
- `validateScoreFile()` 실패
- runtime rebuild 중 예외

원본에 직접 덮어쓴 뒤 실패하면, 중간 상태의 `ScoreFile`이 앱 상태에 남을 수 있습니다. 특히 track cell 삭제까지 수행한 뒤 검증이 실패하면 복구하려면 결국 이전 상태 snapshot이 필요합니다. 그러면 성능 이점이 거의 사라집니다.

현실적인 선택지는 세 가지입니다.

1. **현재 방식 유지**
   - 가장 안전합니다.
   - layout apply는 빈번한 작업이 아니므로, 매우 긴 악보에서도 사용자가 약간 기다리는 건 UX상 허용 가능성이 큽니다.
   - 지금 구조와 테스트가 이 모델에 맞춰져 있습니다.

2. **부분 복사로 최적화**
   - 전체 deep clone 대신 바뀌는 부분만 구조 공유 방식으로 새 `ScoreFile`을 만듭니다.
   - 예: `musicData`, `globalLines`는 그대로 참조하고, `instData`, `layout`, `tracks`만 필요한 만큼 복사.
   - track cell 삭제가 필요 없는 경우 `tracks`도 그대로 재사용 가능.
   - 가장 좋은 개선 방향입니다.

3. **원본 직접 mutation + rollback snapshot**
   - 구현 복잡도만 커지고 장점이 약합니다.
   - 안전하게 하려면 결국 변경 전 `instData`, `layout.rowDefinitions`, `tracks[].cells` snapshot이 필요합니다.
   - 추천하지 않습니다.

제가 보기엔 지금 당장 바꾼다면 “원본에 바로 덮어쓰기”가 아니라 **부분 복사 방식**이 적절합니다.

예상 구조는 이런 느낌입니다.

```ts
const nextScore: ScoreFile = {
  ...score,
  instData: cloneInstrumentData(draft.instData),
  layout: {
    ...score.layout,
    rowDefinitions: [
      ...cloneGlobalRows(score.layout.rowDefinitions),
      ...cloneEditableRows(draft.rowDefinitions),
    ],
  },
  tracks: deletedCells.totalCount > 0
    ? score.tracks.map((track) => ({
        ...track,
        cells: track.cells.filter((cell) => nextNoteRowIds.has(cell.rowId)),
      }))
    : score.tracks,
};
```

이 방식이면 원본은 안전하게 유지하면서도 `musicData`, `globalLines`, 삭제 없는 `tracks[].cells` 같은 큰 구조는 복사하지 않습니다.

결론: 원본 직접 덮어쓰기는 피하고, 필요하면 `cloneScoreFile()` 전체 복사를 layout apply 전용 부분 복사로 바꾸는 게 좋습니다.

> 부분 복사의 구현 비용은 어떻게 되나요?
> 악보 원본 전체 복사의 병목이 심하지 않다면 일단 그대로 두고 추후 확장 고려 사항으로 둘 수도 있겠습니다.

부분 복사 구현 비용은 낮은 편입니다. `layout_apply.ts` 한 파일 중심으로 바꾸면 되고, 테스트도 기존 `test:layout` 대부분을 그대로 씁니다.

예상 변경 범위:
- `applyLayoutDraftToScore()`에서 `cloneScoreFile(score)` 제거
- `ScoreFile` 객체를 spread 기반으로 새로 구성
- `instData`, `layout.rowDefinitions`, 필요 시 `tracks[].cells`만 복사
- `cloneScoreFile` import 제거
- 기존 테스트 유지 + 원본 불변성 테스트는 그대로 통과해야 함

대략 30~50줄 수준의 변경입니다.

다만 지금 당장 꼭 할 필요는 없어 보입니다. layout apply는 매 입력마다 실행되는 경로가 아니라 사용자가 Apply를 누를 때만 실행되고, 이후 어차피 `createRuntimeDocument() -> parse -> analyze -> render` full rebuild가 뒤따릅니다. 악보가 매우 길어졌을 때 병목은 전체 clone보다 rebuild나 canvas/render 쪽에서 먼저 커질 가능성이 높습니다.

제 판단은:

- 지금은 그대로 둬도 됩니다.
- 보고서/메모에는 “현재는 안전성을 위해 full clone 후 apply, 추후 긴 악보 최적화 시 structural sharing 부분 복사로 개선 가능” 정도로 남기면 충분합니다.
- 실제 성능 문제가 보이면 그때 부분 복사로 바꾸는 게 좋습니다.

즉 현재 milestone에서는 리스크 없는 안정 구현을 유지하고, 부분 복사는 추후 확장 고려 사항으로 두는 쪽을 추천합니다.

네, 둘은 별도입니다.

JSON 파일 로드는 사용자가 선택한 파일을 브라우저 메모리로 읽고, `JSON.parse`로 JS 객체를 만든 뒤 앱 상태에 올리는 흐름입니다. 이때 쓰는 것은 주로 **런타임 메모리(RAM)** 이고, 사용 가능한 양은 브라우저/기기/OS/탭 상태에 영향을 받습니다.

반면 localStorage는 브라우저가 origin별로 제공하는 **영구 문자열 저장소**입니다. 파일을 로드했다고 해서 자동으로 localStorage 용량을 차지하지 않습니다. localStorage 용량은 `localStorage.setItem(...)`을 호출할 때만 사용됩니다.

현재 흐름으로 보면:

```text
File Load
-> 파일 텍스트를 메모리로 읽음
-> JSON parse
-> ScoreFile 검증
-> AppState에 반영
-> localStorage에는 저장하지 않음

Local Save
-> 현재 ScoreFile을 JSON 문자열로 직렬화
-> 3 MiB 제한 검사
-> localStorage.setItem(...)
-> localStorage 용량 사용
```

그래서 8 MiB 파일을 로드할 수 있어도, 같은 악보를 Local Save하려 하면 3 MiB 제한에 걸릴 수 있습니다. 이 정책은 의도적으로 다릅니다. 파일 로드는 외부 백업/공유용 큰 악보를 허용하고, localStorage는 프리셋과 함께 쓰는 제한된 브라우저 저장소라 더 작게 잡는 구조입니다.

단, 큰 JSON을 로드하면 localStorage와 무관하게 렌더링/분석/재생에서 RAM과 CPU 부담은 커집니다. 나중에 긴 악보 지원을 본격화하면 localStorage보다 먼저 virtualization, partial rebuild, IndexedDB 저장 전환을 검토하는 편이 맞습니다.

type TrackLayerState = {

의미:
