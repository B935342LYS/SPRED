# 1.15. 2단계 edit-render verification loop 구현 메모
> 본 설계 메모 문서 1.15에서는 canvas grid renderer 이후부터 edit mode 입력, score JSON mutation, analyzer 확장, renderer 시각화까지 이어진 구현 흐름을 기록한다.
> 본 문서는 active specification을 대체하지 않는다.
> 함수별 세부 설명은 코드 파일의 `/** ... */` 주석을 중심으로 하고, 본 문서는 흐름과 사용 문법, 설계 판단을 보충한다.

## 1. 개요

이번 구현 구간의 핵심 목표는 테스트케이스 제작을 위한 입력-분석-렌더링 확인 루프를 만드는 것이었다.

기존 1.13, 1.14 단계에서는 아래 경로가 준비되어 있었다.

```text
RuntimeDocument
-> CanvasRenderInput
-> CanvasScoreLayout
-> layout/base grid draw

RuntimeDocument
-> buildParsedDocument()
-> analyzeDocument()
-> default note / "-" hold NoteEvent
```

이번 작업에서는 이 흐름을 실제 UI 입력과 연결했다.

현재 전체 확인 루프는 아래와 같다.

```text
edit UI 입력
-> note rawText 합성
-> ScoreFile track cell upsert/delete
-> createRuntimeDocument()
-> buildParsedDocument()
-> analyzeDocument()
-> buildCanvasNoteRenderItems()
-> renderCanvasScore()
```

이 경로가 만들어지면서 parser가 만든 문자열이 실제 score JSON에 저장되고, analyzer가 이를 event로 바꾸며, renderer가 시각 결과를 보여주는 최소 검증 루프가 가능해졌다.

## 3. edit mode 입력 체계

Default 입력은 현재 아래 token들을 조합할 수 있다.

- defaultText
- long hold `-`
- vibrato hold `~`
- gliss modifier `@g(id,kind)`
- trem modifier `@t(div)`
- absolutePitch `@p(midi)`
- microPitch `@m(cent)`

CUSTOM defaultText는 사용자 입력 문자를 그대로 UI에 보여주되, parser 예약문자는 rawText에 저장할 때 escape한다.
custom text 길이는 테스트케이스 작성과 UI 안정성을 위해 10자로 제한했다.

absolutePitch는 MIDI 직접 입력 대신 계이름 dropdown으로 바꾸었다.
dropdown은 sharp 계열 계이름만 사용하고, 높은 음에서 낮은 음 순서로 표시한다.

AUTO sharp/flat defaultText는 현재 row의 MIDI를 기준으로 표시 문자열을 만든다.
다만 absolutePitch가 존재하면 표시 계이름도 absolutePitch를 우선한다.
microPitch가 양수이면 표시 문자열 뒤에 `+`, 음수이면 `-`를 붙인다.
`@p(0)`과 `@m(0)`은 효과가 없는 token이므로 rawText에 생성하지 않는다.

microPitch 입력은 parser 규칙에 맞춰 `-100..100`, 소수점 이하 최대 1자리로 정규화한다.

## 4. tuplet 입력 준비

tuplet UI는 현재 SELECT ROW 기반 slot 입력까지 준비되었다.

입력 흐름은 아래와 같다.

```text
1. tuplet On 상태로 전환한다.
2. slot input을 클릭해 active slot을 선택한다.
3. insert mode를 SELECT ROW로 둔다.
4. score 영역의 note row를 클릭한다.
5. 현재 Default token 조합과 선택 row의 @n(midi)를 slot에 기록한다.
6. Finalize Value를 누르면 /n(...) rawText가 준비된다.
7. insert mode가 SELECT MODE로 돌아간다.
8. score cell 클릭 시 전체 tuplet 문자열을 cell에 입력한다.
```

셋잇단 상태는 일반 rawText 입력을 잠그지 않는다.
이는 legacy `이세계 코드`의 유스케이스와 맞춘 판단이다.

아직 tuplet analyzer/render 연결은 구현하지 않았다.
현재 단계의 tuplet 구현은 테스트케이스 제작을 위한 rawText 입력 준비 단계이다.

## 5. JSON file / storage 경계

File panel에는 아래 기능이 연결되었다.

- JSON Load
- JSON Download
- Local Save
- Local Load

이 기능은 app/controller가 직접 storage API를 다루지 않도록 `src/infra/` 계층으로 분리했다.

```text
score_file_io.ts
- serializeScoreFile()
- downloadScoreJson()
- readTextFile()

score_local_storage.ts
- saveScoreToLocalStorage()
- loadScoreFromLocalStorage()
```

JSON Load는 `loadRuntimeDocument()`를 거쳐 score 구조를 다시 검증한다.
로드에 실패하면 현재 score state는 유지한다.

Local Save는 현재 `state.document.score`를 localStorage에 저장한다.
Local Load도 JSON Load와 같은 runtime document rebuild 경로를 사용한다.

## 6. analyzer 확장

기존 analyzer MVP는 default note와 `"-"` hold만 처리했다.
이번 구현에서는 gliss와 tuplet을 제외한 비교적 단순한 modifier를 먼저 연결했다.

현재 `analyze_track.ts`는 다음을 처리한다.

- 일반 note
- `"-"` hold
- `"~"` vibrato hold
- `@t(div)` tremolo
- `@p(midi)` absolutePitch
- `@m(cent)` microPitch

`@p`는 최종 발음 MIDI를 바꾼다.
renderer의 note fill 색상도 `event.sound.midi`를 기준으로 결정되므로, 사용자는 실제 연주될 음을 색으로 확인할 수 있다.

`@m`은 최종 발음 cent offset과 표시 cent offset에 모두 반영한다.
`+100cent`는 위쪽 note row 중심, `-100cent`는 아래쪽 note row 중심과 대응되도록 renderer에서 보간한다.

hold 병합은 최종 display position과 final sound pitch가 모두 같을 때만 일어난다.
따라서 `@p`나 `@m`이 있는 note를 이어 붙이려면 hold 셀 쪽도 같은 최종 조건을 만족해야 한다.

effect는 `NoteEffectSegment[]`로 기록한다.
`@t`가 시작한 tremolo는 plain `"-"` hold에서 이어지고, `"~"`가 나오면 tremolo가 끊긴 것으로 처리한다.

gliss와 tuplet은 이벤트 구조가 더 복잡하므로 이번 범위에서 제외했다.

## 7. renderer 확장

renderer는 analyzer 결과를 직접 rawText로 다시 해석하지 않는다.
`canvas_item_builder.ts`가 `AnalysisResult`를 renderer-owned DTO로 변환한다.

이번 구간에서 `CanvasNoteRenderItem`에는 아래 정보가 추가되었다.

- `displayCentOffset`
- `effects`
- `displayTextAnchors`

note layer는 이를 사용해 다음 시각 요소를 그린다.

- 발음 MIDI 기준 note rectangle 색상
- microPitch에 따른 y 위치 보간
- anchor별 displayText
- trem chop line
- vibrato sine wave

vibrato는 `"~"`가 여러 칸 이어져도 segment마다 wave를 따로 그리지 않는다.
renderer에서 연속된 vib segment를 하나의 draw range로 병합하고, 칸당 1주기 기준으로 하나의 path를 그린다.

tremolo는 chop line으로 표시한다.
레거시 구현과 같이 선 색상은 note row 배경색을 사용해, 노트 사각형이 끊기는 느낌을 주도록 했다.

## 8. renderer theme 정리

renderer 내부에서 여러 파일이 공유하는 색상과 치수 상수는 `canvas_theme.ts`로 분리했다.

```text
canvas_theme.ts
- CANVAS_COLORS
- CANVAS_METRICS
```

이 파일에는 아래 성격의 값만 둔다.

- roll background
- note row background
- grid line color
- label text color
- note stroke/text color
- vib/trem 시각 효과 색상
- layout label/padding/font 기준 크기
- note render height, inset, 최소 크기
- trem line width, vib sample count

pitch class 팔레트는 `canvas_note_colors.ts`에 유지했다.
이는 MIDI pitch class를 색으로 바꾸는 계산과 일반 renderer theme 상수를 분리하기 위해서이다.

## 10. 남은 작업

다음 구현 단위는 아래 순서가 적절하다.

1. 현재 renderer 시각 표현을 실제 JSON download/load round trip으로 다시 확인한다.
2. tuplet analyzer/render 연결을 구현한다.
3. gliss analyzer/render 연결을 구현한다.
4. 중앙 player group을 실제 score metadata와 연결한다.
5. Details dialog의 정보 수정 기능을 활성화한다.
6. audio generator와 playback controller 경계를 설계한다.
