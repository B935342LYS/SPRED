현재 이해한 진행 상황은 다음과 같습니다.

`회귀 코드`의 노래방 모드는 내부적으로 `game mode`, UI에서는 `practice mode`로 구현되고 있습니다. 방향은 기존 parser/analyzer/renderer/audio 의미를 바꾸지 않고, `AnalysisResult` 위에 런타임 판정 계층을 얹는 구조입니다. 점수, pitch frame, 판정 결과는 score JSON에 저장하지 않습니다.

구현은 이미 1차 UI 진입 단계를 넘었습니다. 현재 `regression-code/src/app/game/` 아래에 다음 축이 들어가 있습니다.

- `game_types.ts`: practice 상태, pitch frame, 점수 summary, Sync 설정 타입
- `game_binding.ts`: practice mode 버튼, 마이크 권한 요청, pitch runtime 연결, Sync 보정 팝업
- `game_pitch_input.ts`: `pitchy` 기반 마이크 pitch detection
- `game_pitch_math.ts`: frequency/MIDI/cent/RMS 계산
- `game_pitch_overlay.ts`, `game_judge_overlay.ts`, `game_ui.ts`: pitch dot, 판정 텍스트, practice panel/result UI
- `game_judge.ts`: active track `NoteEvent` 기반 pitch/timing 판정과 점수 누적
- `game_effect_judge.ts`: gliss/vib effect bonus 1차 판정

현재 완료된 핵심 기능은 다음입니다.

- practice mode 진입/종료, 마이크 권한 처리
- edit mode 자동 해제 및 edit/seek/loop/layout/file 조작 잠금
- active track 기반 판정 대상 생성
- `pitchy`로 실시간 pitch frame 생성
- 생 pitch 기반 pitch dot 표시
- target 기반 octave/harmonic/hysteresis 보정은 제거하고 raw detector pitch 사용
- pitch class 기준 판정: 같은 계이름이면 옥타브가 달라도 정답 처리
- `Perfect / Ok / Bad / Miss` 판정과 combo, score, accuracy 표시
- `Miss`는 0% sample로 평균에 포함
- timing onset 기반 `early / late / bad / miss` 보조 판정
- `+90ms` 기본 Sync 보정, `-200ms~+200ms`, 10ms 단위, localStorage 저장
- 결과 dialog에 pitch/timing summary 분리 표시
- gliss bonus 1차 구현
- vib bonus 1차 구현 및 짧은 vib target 완화
- trem bonus는 아직 follow-up으로 남아 있음

테스트 기준도 잡혀 있습니다. `package.json`에 `npm run test:game`이 있고, `dev/test_game.ts`에서 pitch math, Sync 보정, active track target, pitch-class 판정, timing downgrade, score summary, gliss/vib bonus까지 단위 검증합니다. HARNESS 기준 최신 검증 기록은 `npm run test:game`, `npm run typecheck`, `npm run build`가 통과된 상태로 정리되어 있습니다.

남은 주요 작업은 다음으로 보입니다.

- trem effect bonus 구현
- timing anti-cheat를 “no-onset이면 바로 Bad/Miss”가 아니라 attack credit / score eligibility 방식으로 재설계
- 실제 마이크 환경에서 pitch threshold, vib/trem 안정성 추가 조정
- 긴 악보/저성능 환경에서 practice playback 성능 확인
- YouTube 동시 사용과 practice Sync의 수동 검증 지속
- loop playback 연결은 별도 후속 작업이며, practice mode 중에는 loop를 금지하는 정책 유지

정리하면, 현재 노래방 모드는 “진입 스켈레톤” 단계가 아니라, 기본 음정/타이밍 채점과 결과 화면, gliss/vib 보너스까지 들어간 1차 실사용 검증 단계입니다. 남은 중심은 trem, timing 공정성, 실제 마이크 환경 튜닝입니다.

> 네. 타이밍 판정을 가만히 꾹 누르고 있어도 연속된 노트들이 모두 perfect로 인식되는 꼼수를 사용할 수 없도록 재설계할 필요가 있습니다.

네, `scoreEligible = false`인 경우에는 `Perfect` 자체도 안 보이게 하는 쪽이 더 낫습니다.

지금처럼 `Perfect`는 보이는데 점수/콤보가 안 오르면 사용자가 “왜 Perfect인데 콤보가 안 오르지?”라고 느낄 수 있습니다. 꼼수 방지 목적이라면 더 일관된 표시는 다음 중 하나입니다.

1. `scoreEligible = false` sample은 overlay를 아예 표시하지 않음
2. 또는 작은 보조 텍스트로 `No attack` / `Hold` 같은 중립 피드백 표시
3. live accuracy에는 포함하지 않거나, 포함하더라도 점수와 분리 설명 필요

현재 UX에서는 1번이 가장 깔끔합니다. “꾹 누르기 꼼수”를 사용해도 아무 판정 텍스트가 안 뜨고 점수/콤보도 안 오르는 방식입니다.

글리산도 판정 범위 완화는 합리적입니다. 특히 2tick처럼 짧은 gliss는 실제 연주와 pitch detector frame 타이밍이 조금만 어긋나도 실패하기 쉽습니다. 다만 `+= 1 tick` 같은 tick 기반 완화는 BPM에 따라 실제 시간이 달라지므로, 기본은 `min(글리산도 길이의 일부, 80~120ms)` 같은 seconds 기반 grace가 더 안정적입니다. 짧은 gliss에는 앞뒤 grace를 주고, 긴 gliss에는 과도하게 넓어지지 않게 cap을 두는 방식이 좋습니다.

Bad도 콤보를 잇는 완화는 가능하지만, 저는 신중하게 보겠습니다. 프렛리스 악기 특성상 `Ok`까지만 콤보 유지하면 너무 엄격할 수 있지만, `Bad`까지 모두 콤보 유지하면 콤보의 의미가 많이 약해집니다. 타협안은 `Bad`는 combo를 끊지 않지만 증가시키지는 않음입니다. 즉 `Perfect/Ok`는 combo +1, `Bad`는 combo 유지, `Miss`는 combo reset. 연습 보조 앱으로는 이쪽이 가장 현실적입니다.

Rules 설명은 추가하는 편이 좋습니다. 지금 점수 체계가 일반 리듬게임과 다르기 때문입니다. 특히 아래는 명시해야 합니다.

- 모든 note를 반드시 칠 필요는 없다.
- 아무 입력이 없으면 Miss로 세지 않고 점수도 없다.
- 틀린 pitch 입력이 들어온 경우에만 Miss sample이 생긴다.
- 반복 note는 새 attack이 확인되어야 점수/콤보를 받는다.
- pitch label과 score/combo eligibility는 분리될 수 있다.
- Gliss/Vib/Trem은 실패 감점이 아니라 성공 bonus다.

정리하면: `scoreEligible=false`일 때 Perfect 숨김은 바로 적용할 가치가 있고, gliss grace는 seconds 기반으로 설계하는 것이 좋습니다. Bad combo 완화는 “유지하되 증가 없음”을 추천합니다. Rules 보강은 필요합니다.

저는 지금 단계에서는 **콤보를 점수 공식에 넣지 않는 쪽**이 더 낫다고 봅니다. 대신 “성취감 표시”로만 강화하는 방향이 안전합니다.

이유는 간단합니다. 이 앱의 practice mode는 프렛리스/오타마톤류 악기의 연습 보조에 가깝고, pitch detection도 환경 영향을 받습니다. 여기에 콤보 배율까지 넣으면 점수가 “연주 실력”보다 “마이크 안정성, 판정 튐, 한 번의 실수”에 과하게 흔들릴 수 있습니다. 특히 100 combo 배율은 긴 악보에서 점수 차이를 크게 벌려서, 현재의 pitch/timing/effect 보너스보다 콤보가 점수의 주인이 될 위험이 있습니다.

Bad 판정 완화도 신중해야 합니다. `Bad`가 combo를 이어주면 사용자는 성취감을 얻지만, `Combo`의 의미가 “정확히 이어 연주했다”가 아니라 “대충 끊기지 않았다”로 바뀝니다. 이건 나쁜 선택은 아니지만, Rules 설명과 결과 해석이 더 복잡해집니다.

추천은 이렇습니다.

1. 지금은 `Perfect/Ok`만 combo 증가, `Bad/Miss` reset 유지
2. 점수 공식에는 combo 배율을 넣지 않음
3. 대신 결과 화면/라이브 UI에서 성취감 요소를 추가
   - `Max Combo`
   - `Full Combo` 뱃지: Miss 0, score-eligible sample 중 Bad 0 또는 정책에 따라 Miss 0
   - `No Miss`
   - `All Hit`
   - `Effect Bonus` count
4. 나중에 게임성을 강화하고 싶으면 별도 옵션으로 `Game scoring`을 추가

만약 완화한다면 제일 무난한 타협안은 **Bad는 combo를 끊지만, “No Miss” 업적은 유지**입니다. 즉 정확한 콤보는 엄격하게 두고, 성취감은 별도 뱃지로 줍니다.

예시:
- `Full Combo`: 모든 score-eligible sample이 `Perfect` 또는 `Ok`
- `No Miss`: `Miss`가 0개, Bad는 허용
- `Clean Run`: timing Bad/Miss 없이 완료
- `Articulation Bonus`: Gliss/Vib/Trem bonus 일정 수 이상

이렇게 하면 연습 프로그램의 신뢰성을 유지하면서도 게임적인 목표를 줄 수 있습니다. 지금 바로 점수 배율을 넣는 것보다, 결과 화면에 뱃지/칭호를 추가하는 쪽이 더 합리적입니다.

> 현재 점수의 최종 계산 공식을 정리해볼 수 있나요?

현재 Practice 점수 공식은 요약하면 다음입니다.

```text
Total Score
= sum(score-eligible pitch/timing sample scores)
+ sum(effect bonus scores)
```

**1. Pitch/Timing Sample Score**
각 scoring sample은 먼저 pitch 판정을 받습니다.

```text
pitchAccuracy:
Perfect = 1.0
Ok      = 0.666...
Bad     = 0.333...
Miss    = 0
```

timing 판정이 붙으면 최종 label이 강등될 수 있습니다.

```text
timing offset < 80ms      -> label 유지
80-149ms                 -> early/late 표시, label 유지
150-249ms                -> Perfect/Ok이면 Bad로 강등
250ms 이상               -> Miss로 강등
```

최종 sample 점수는 다음입니다.

```text
sampleScore =
  trackDifficulty * finalPitchAccuracy
```

단, 아래 조건이면 `0점`입니다.

```text
if label is Miss:
  sampleScore = 0

if scoreEligible is false:
  sampleScore = 0
```

`scoreEligible = false`는 예를 들어 반복 note에서 새 attack 없이 같은 pitch를 계속 누르고 있는 경우입니다. 이 경우 judge overlay도 안 뜨고 count/combo에도 반영되지 않습니다.

**2. Track Difficulty**
트랙별 기본값은 다음입니다.

```text
basic    = 1.0
optional = 1.2
extra    = 1.5
```

Music Info에 custom difficulty가 설정되어 있으면 그 값을 사용합니다. 값이 비어 있거나 `0`/비정상 값이면 기본값을 씁니다.

**3. Combo**
현재 combo는 점수 배율에 영향을 주지 않습니다.

```text
score-eligible Perfect / Ok / Bad -> combo +1
Miss                              -> combo reset
scoreEligible false / no input    -> combo 변화 없음
```

즉 combo는 성취감 표시용이고, 점수 공식에는 들어가지 않습니다.

**4. No Input**
무음, 불명확한 pitch, threshold 미달 입력은 sample을 만들지 않습니다.

```text
no input -> no score, no Miss, no combo change
```

**5. Effect Bonus**
현재 effect bonus는 성공 시에만 점수를 더합니다. 실패해도 감점은 없습니다.

```text
gliss interval bonus = trackDifficulty * 0.25
vib segment bonus    = trackDifficulty * 0.5
trem bonus           = 필드만 준비됨, 판정은 아직 follow-up
```

따라서 전체 공식은 더 구체적으로 쓰면:

```text
Total Score =
  Σ(scoreEligibleSample ? trackDifficulty * finalPitchAccuracy : 0)
  + Σ(glissBonus: trackDifficulty * 0.25)
  + Σ(vibBonus: trackDifficulty * 0.5)
```

현재 `Combo`와 `Timing Accuracy`는 표시/피드백용이며, 직접 점수 배율에는 들어가지 않습니다.

> 아직 완전히 정리된 것이 아니라 Trem까지 정리한 다음에 Rules에 추가하는 것이 낫겠네요.

네, 그 판단이 맞습니다. 지금 Rules에 최종 공식처럼 자세히 넣으면 Trem 구현 이후 다시 고쳐야 하고, 사용자가 “공식이 바뀌었다”고 느낄 수 있습니다.

현재 Rules는 지금처럼 짧게 유지하는 편이 좋습니다.

- `Score per note`
- `Total score`
- `Track difficulty`
- `Combo`
- `No input`
- `Effect bonus`

Trem까지 구현한 뒤에 그때 최종적으로 정리할 항목은 다음 정도가 적절합니다.

```text
Score per note = track difficulty x final pitch accuracy
Effect bonus = Gliss / Vib / Trem success bonus
No penalty for missed effects
Combo does not multiply score
No input does not count as Miss
```

즉 지금은 “상세 공식 문서화”보다 구현 안정화가 우선이고, Rules는 사용자가 플레이 중 헷갈리지 않을 정도만 유지하면 됩니다.

> 더 엄밀한 연주를 하고 싶은 사람을 위한 판정 강화 모드 또는 리듬게임의 체력 기능을 도입하는 것은 어떻습니까?

둘 다 가능하지만, 지금 우선순위는 **판정 강화 모드가 더 적합**하고, **체력 기능은 별도 Game Mode 옵션으로 나중에** 보는 게 맞습니다.

**판정 강화 모드**
연습 프로그램 성격과 잘 맞습니다. “점수 경쟁”보다 “내가 더 엄밀하게 연주하고 싶은가?”를 사용자가 선택하는 구조라 자연스럽습니다.

추천 형태:

```text
Practice Strictness
- Relaxed
- Standard
- Strict
```

예시 정책:

```text
Relaxed
- pitch class 허용
- Bad도 combo 유지
- gliss/vib grace 넓게
- no input은 Miss 아님

Standard
- 현재 기본값
- pitch class 허용
- Bad combo 유지
- attack credit 적용
- effect 실패 감점 없음

Strict
- pitch threshold 강화
- timing threshold 강화
- Bad는 combo reset
- gliss/vib grace 축소
- 필요하면 octave 정확도 요구 옵션
```

이 방식은 사용자에게 “내 실력 확인용으로 엄격하게 할 수 있다”는 선택권을 줍니다. 초보자 경험을 해치지 않으면서 고급 사용자도 만족시킬 수 있습니다.

**체력 기능**
리듬게임에는 익숙하지만, 연습 프로그램에는 부작용이 큽니다. 실수 몇 번으로 세션이 끝나면 반복 연습 흐름이 끊깁니다. 특히 마이크/pitch detector가 불안정한 상황에서는 사용자가 시스템 탓으로 느끼기 쉽습니다.

도입한다면 기본 Practice에는 넣지 말고, 별도 옵션이 좋습니다.

```text
Challenge Mode
- Health starts at 100
- Perfect/Ok 유지
- Bad 소폭 감소
- Miss 크게 감소
- no input은 감소하지 않거나, Strict에서만 감소
- health 0이어도 playback은 계속하고 result에 Failed 표시
```

중요한 점은 **체력이 0이 되어도 재생을 강제로 끊지 않는 것**입니다. 연습 앱에서는 중단보다 결과 표시가 낫습니다.

제 추천 순서:

1. Trem까지 구현
2. Rules/score formula 정리
3. `Standard / Strict` 판정 강화 모드 추가
4. 결과 화면에 strictness 표시
5. 이후 원하면 `Challenge Mode`로 health 추가

즉, “엄밀한 연주 욕구”는 판정 강화 모드로 해결하고, “게임성”은 별도 Challenge Mode로 분리하는 게 가장 깔끔합니다.

> 판정 강화를 위한 조건 분기 추가가 연산에 얼마나 영향을 미칠까요?

거의 영향 없습니다. 현재 구조에서는 판정 강화 모드가 성능 병목이 될 가능성은 낮습니다.

이유는 판정이 매 audio sample마다 도는 게 아니라, practice scoring cadence에서 대략 `100ms` 간격으로 도는 구조이기 때문입니다. 즉 초당 약 10회 수준입니다. 여기에 조건 분기 몇 개가 추가되는 비용은 canvas redraw, pitch detection, audio scheduling, DOM update에 비하면 무시해도 되는 수준입니다.

추가될 분기는 대략 이런 형태입니다.

```ts
const thresholds = getJudgeThresholds(strictness);

if (pitchError < thresholds.perfectCent) ...
if (timingOffset < thresholds.onTimeMs) ...
if (strictness === "strict") ...
```

이 정도는 초당 10회라 사실상 비용이 없습니다.

주의할 부분은 조건 분기 자체가 아니라, **모드별로 target lookup이나 frame scan을 더 많이 하게 되는 경우**입니다. 예를 들어 Strict에서:

- 전체 note coverage를 매 tick마다 다시 계산
- gliss/vib/trem target 전체를 매 frame마다 스캔
- 최근 pitch frame을 긴 window로 계속 filter/map/reduce
- DOM overlay를 더 자주 생성

이런 식으로 구현하면 비용이 생깁니다. 하지만 threshold만 바꾸고 현재 target/frame 구조를 재사용하면 괜찮습니다.

권장 구조는:

```ts
type PracticeStrictness = "relaxed" | "standard" | "strict";

type JudgeThresholds = {
  perfectCent: number;
  okCent: number;
  badCent: number;
  timingDisplayMs: number;
  timingBadMs: number;
  timingMissMs: number;
  glissGraceRatio: number;
  glissMaxGraceSeconds: number;
};
```

그리고 playback 시작 시 또는 UI 변경 시 한 번만 threshold를 정해 둡니다. scoring loop에서는 이미 만들어진 threshold 객체만 읽습니다.

정리하면:

- 단순 threshold 분기: 성능 영향 거의 없음
- 모드별 추가 스캔/DOM 작업: 주의 필요
- 현재 구조에 가장 맞는 방식: `strictness -> thresholds`를 미리 계산하고 기존 판정 함수에 전달

실제 체감 성능에는 거의 영향이 없을 가능성이 큽니다.

> 이미 판정 기준을 매우 후하게 잡고 있으므로 relax 모드는 필요하지 않을 것 같네요. 전반적인 판정을 더 엄밀하게 강화한 pro mode만 추가할까 합니다. 대략적인 계획은 아래와 같이 잡고 있습니다.
>  - sync 오른쪽에 하드 모드 버튼 추가 : 클릭시 전반적인 프랙티스 모드의 UI가 붉은색으로 변함
>  - 피치 : 반음 이상 피치 차이나면 바로 Miss
>  - 타이밍 기준 강화 : 평균적인 리듬게임 기준 참고
>  - 나머지는 유지
>
> 명세를 작성하면서 모호한 부분, 개선 가능한 부분을 점검해 봅시다.

좋습니다. `Relaxed / Standard / Strict` 3단계보다, 현재 기본이 이미 관대한 편이므로 **기본 Practice + Pro Mode** 2단계가 더 명확합니다.

제가 보기엔 방향은 합리적이고, 명세에서 먼저 정리해야 할 모호점은 아래입니다.

**1. 이름**
UI 버튼을 `Hard`로 할지 `Pro`로 할지 먼저 정하는 게 좋습니다.

추천은 `Pro`입니다.
`Hard`는 게임 난이도처럼 느껴지고, `Pro`는 “엄밀한 연습 모드”에 더 가깝습니다.


```text

```

**2. 버튼 위치와 상태**
`Sync` 오른쪽에 `Pro` toggle을 두는 건 좋습니다. 다만 practice 도중 켜고 끄면 점수 공정성이 흔들립니다.

- countdown / playing / paused / finished 중에는 disabled

**4. 피치 기준**
“반음 이상 피치 차이나면 바로 Miss”는 중요한 모호점이 있습니다.

```text
Bad     < 200 cent
Miss    >= 200 cent
```

```text
Ok      < 60 cent
Bad     < 100 cent

```

이게 가장 자연스럽습니다. “반음 이상은 Miss”도 만족하고, Bad도 유지됩니다.

**5. 타이밍 기준**
현재 기준은 꽤 관대합니다.

```text

80-149ms    early/late 표시
150-249ms   Bad 강등
>= 250ms    Miss 강등
```

Pro 후보는 이렇게 잡는 게 적당해 보입니다.

```text
50-99ms     early/late 표시
100-149ms   Bad 강등
>= 150ms    Miss 강등
```

이 정도면 일반적인 리듬게임 느낌에 가까워지지만, 마이크 latency와 pitch detector 지연을 고려해 너무 과격하지는 않습니다.

**6. Combo 정책**
“나머지는 유지”라면 Pro에서도 현재 정책을 유지하는 게 좋습니다.

```text
score-eligible Perfect / Ok / Bad -> combo continues
Miss -> combo reset
```

Pro mode에서 Bad까지 combo를 끊으면 Full Combo가 다시 어려워지고, 방금 정한 연습 친화 방향과 충돌할 수 있습니다.

**7. Score 공식**
Pro mode가 점수 배율을 주는지는 정해야 합니다.

추천은 일단 배율 없음입니다.

```text
Pro mode changes judge thresholds only.
Score formula stays the same.
```

이렇게 해야 “엄격한 기준으로 같은 점수 체계를 통과했다”는 의미가 명확합니다. 나중에 결과 화면에서 `Pro Clear` 같은 뱃지를 줄 수 있습니다.

**8. Effect 판정**
“나머지는 유지”라면 gliss/vib/trem grace와 bonus 기준은 그대로 유지하면 됩니다.

다만 나중에 원하면 Pro에서만:

```text
gliss grace 축소
vib 평균 pitch 기준 강화
trem envelope 기준 강화
```

를 추가할 수 있습니다. 지금은 scope를 줄이는 게 좋습니다.

> 3. 추가적으로 Bad에서 콤보를 끊는 것도 생각해볼 수 있겠습니다.
