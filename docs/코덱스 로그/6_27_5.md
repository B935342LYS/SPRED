정리하면, 현재 작업의 중심은 기존 `이세계 코드`를 보존/호환하는 것이 아니라, `regression-code/`의 TypeScript 웹 앱인 `회귀 코드` 위에 **노래방/practice mode를 런타임 평가 계층으로 얹는 것**입니다. 판정의 기준은 score JSON rawText가 아니라 parser/analyzer를 거친 `AnalysisResult`이며, 마이크 입력, pitch frame, 판정 결과, 점수는 score 파일에 저장하지 않는 일시 상태입니다.

현재 노래방 모드는 문서상 `game mode`, UI상 `practice mode`로 취급됩니다. 관련 초안은 [docs/3.2-karaoke-game-mode-spec.md](./docs/3.2-karaoke-game-mode-spec.md)이고, 실제 구현은 [regression-code/src/app/game](./regression-code/src/app/game) 아래에 모여 있습니다.

현재 구현 상태는 대략 이렇습니다.

- practice mode 진입/종료, 마이크 권한 요청, countdown, edit/seek/loop/layout/file 동작 차단은 이미 들어가 있음
- `pitchy` 기반 마이크 입력에서 `GamePitchFrame`을 만들고, clarity/RMS/frequency range로 유효 입력을 필터링함
- 악보 위에 pitch dot을 표시하며, active track의 현재 target note를 참고해 옥타브 단위 보정을 적용함
- active track의 `NoteEvent`를 현재 score time 기준 판정 대상으로 수집하고, pitch class 기준 오차로 `Perfect / Ok / Bad / Miss` sample을 만들 수 있음
- summary에는 accuracy, Perfect/Ok/Bad/Miss count, combo, score가 누적됨
- `npm run test:game`용 테스트도 있고, frequency 변환, RMS, pitch class folding, hysteresis, target lookup, scoring sample 쪽을 확인하고 있음

입력 음정 보정의 핵심은 [game_pitch_math.ts](./regression-code/src/app/game/game_pitch_math.ts)에 있습니다. 현재 정책은 “사용자가 낸 음을 무조건 target으로 끌어당기는 것”이 아니라, **현재 target 후보와 pitch class가 충분히 가까울 때만 옥타브를 접어 표시/비교**하는 방향입니다. 기준값은 `OCTAVE_CANDIDATE_MAX_ERROR_CENT = 100`이고, 짧은 target 공백이나 pitch detector 튐을 완화하기 위해 `TARGET_LOCK_GRACE_MS = 220`의 hysteresis 상태를 유지합니다.

즉 현재 보정 흐름은 다음처럼 이해했습니다.

```text
마이크 입력
-> pitchy frequency + clarity
-> RMS / clarity / layout range 필터
-> GamePitchFrame
-> 현재 playback 기준 +100ms 근처 active target pitch 수집
-> pitch class가 가까우면 target 옥타브 주변으로 fold
-> 한두 frame의 target 공백/검출 튐은 hysteresis로 안정화
-> overlay dot 표시 및 scoring sample 판정
```

중요한 구분점은, overlay의 보정은 [game_pitch_overlay.ts](./regression-code/src/app/game/game_pitch_overlay.ts)에서 쓰이고, 판정은 [game_judge.ts](./regression-code/src/app/game/game_judge.ts)에서 pitch class error 기준으로 다시 계산한다는 점입니다. 그래서 앞으로 “입력 음정 보정 기능 보강”을 할 때는 표시용 보정과 판정용 보정을 같은 정책으로 맞출지, 아니면 시각 안정화만 더 강하게 하고 점수 판정은 원 입력 기준을 유지할지 결정해야 합니다.

제가 보기엔 다음 작업 후보는 이 순서가 자연스럽습니다.

1. 현재 보정 정책이 overlay와 scoring에서 일관적인지 점검
2. hysteresis가 “틀린 음을 너무 오래 붙잡는” 부작용을 만들지 테스트 보강
3. raw frequency/clarity/RMS/rejectReason 진단 UI를 기준으로 threshold 조정
4. 필요하면 scoring 쪽에도 명시적인 correction state 또는 corrected pitch field를 도입할지 결정

한 가지 문서상 주의점도 있습니다. `docs/3.2-karaoke-game-mode-spec.md`는 아직 active spec이 아니라 extension-plan 성격이고, 실제 구현은 이미 그 초안을 일부 앞질러 진행되었습니다. 따라서 다음 구현에서는 활성 2.x 문서와 충돌하지 않는 범위에서 `3.2`와 현재 코드 상태를 task-local 기준으로 삼는 방식이 맞습니다.

> 좋습니다. 다음으로 진행할 사항이 무엇이었죠?

다음 큰 흐름은 **노래방/practice mode의 입력 음정 보정과 판정 안정화**입니다. 방금 한 것은 표시용 pitch dot 위치 보정이었고, 이어서 볼 만한 항목은 아래 순서입니다.

1. 표시용 보정과 판정용 보정의 일관성 점검
   현재 dot은 hysteresis 기반 보정을 쓰지만, scoring은 원 pitch frame을 pitch class error로 바로 판정합니다. 두 결과가 체감상 어긋나는지 확인해야 합니다.

2. hysteresis 부작용 테스트 보강
   한두 frame 튐은 잡되, 틀린 음을 너무 오래 이전 target에 붙잡아 두지 않는지 `test:game`에 케이스를 추가하는 작업입니다.

3. 실제 브라우저 마이크 입력 기준 threshold 조정
   clarity/RMS/raw frequency 진단 UI를 보고 `MIN_CLARITY = 0.75`, `MIN_RMS = 0.01`, frequency range가 너무 빡빡하거나 느슨하지 않은지 조정합니다.

4. 판정 UI/결과 안정화
   `Perfect / Ok / Bad / Miss`, accuracy, combo, score 표시가 재생 중/finished 상태에서 의도대로 갱신되는지 확인하고 부족한 부분을 보강합니다.

네, 몇 가지는 정하고 가는 편이 좋겠습니다.

제가 이해한 구현 방향은 이렇습니다.

- scoring sample이 생성될 때마다 score 영역 위에 판정 텍스트를 띄움
- 위치는 **선택된 target note의 위쪽 여백**
- 텍스트는 두 줄:
  ```text
  COMBO n
  Perfect
  ```
- `Perfect / Ok / Bad / Miss` 모두 표시
- 일정 시간 새 판정이 없으면 자동으로 사라짐
- combo는 현재 구현의 `currentCombo` 기준

모호한 부분과 제안 기본값입니다.

1. 표시 시간
   제안: 마지막 판정 후 `700ms` 동안 표시하고 사라지게 하겠습니다. 너무 길면 악보를 가리고, 너무 짧으면 확인이 어렵습니다.
