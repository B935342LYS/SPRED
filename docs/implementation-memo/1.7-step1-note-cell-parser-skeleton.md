# 1.7. 1단계 Note 셀 Parser 골격 구현 메모
> 본 설계 메모 문서 1.7에서는 1단계 구현 중 `parse_note_cell.ts`의 첫 구현 범위를 기록한다.
> 본 문서는 active specification을 대체하지 않는다.
> 함수별 세부 설명은 코드 파일의 `/** ... */` 주석을 중심으로 하고, 본 문서는 흐름과 현재 구현 범위를 보충한다.
> 이후 추가된 일반 note modifier 구현은 `1.8-step1-note-modifier-parser.md`에 기록한다.

## 1. 개요

전역 셀 parser가 fixture 기준으로 동작하므로, 다음 parser 작업으로 note 영역 셀 parser의 골격을 추가하였다.
이번 단계의 목표는 모든 note 문법을 한 번에 구현하는 것이 아니라, 최상위 셀 종류 판별과 fixture에 들어 있는 기본 note/hold 셀을 안정적으로 처리하는 것이다.

현재 note parser의 위치는 아래와 같다.

```text
RuntimeDocument
-> tracks[].cells
-> parseNoteCell()
-> ParsedCell
```

## 2. 현재 구현 범위

`parse_note_cell.ts`의 공개 진입점은 `parseNoteCell(input)`이다.

현재 처리하는 셀 종류는 다음과 같다.

- 빈 문자열
  - `empty_note` invalid 반환
- `//`로 시작하는 mute 셀
  - `ParsedMuteCell` 반환
  - `//` 뒤 문자열은 표시 문자열로만 취급
- 정확히 `/&`인 tuplet extend 셀
  - `ParsedPletExtendCell` 반환
- `/n(` 형태로 보이는 tuplet head 후보
  - 현재 단계에서는 미구현 invalid 반환
- 그 외 일반 note 셀
  - 맨 앞 `-` 또는 `~` hold 파싱
  - defaultText 파싱
  - 본 문서 작성 당시 modifier는 다음 단계 구현 대상으로 분리
  - 현재 modifier 구현 세부 내용은 `1.8-step1-note-modifier-parser.md` 참조

현재 fixture의 track 셀은 `E3`, `F3`, `F#3`, `-` 형태이므로 이번 구현으로 모두 valid하게 파싱된다.

## 3. 구현 판단

### 3.1. tuplet head를 placeholder로 둔 이유

`/n(...)` tuplet head는 slot 분리, rest slot, slot note, `@n(midi_num)` 위치 토큰, nested tuplet 금지 등 세부 규칙이 많다.
따라서 최상위 판별만 먼저 연결하고, 실제 tuplet head 구현은 다음 단계로 분리하였다.

현재는 tuplet head 후보가 들어오면 `more_than_expected` invalid를 반환한다.
이는 "해당 셀 종류를 판별할 수는 있지만 아직 구현하지 않았다"는 임시 상태이다.

### 3.2. modifier를 별도 단계로 분리한 이유

일반 note 셀의 modifier 순서는 `@g -> @t -> @p -> @m`으로 고정되어 있다.
각 modifier는 인수 파싱과 범위 검사가 필요하므로 별도 단계로 구현하는 편이 안전하다.

본 골격 구현 단계에서는 defaultText 스캔 중 `@`를 만나면 스캔을 멈춘 뒤, 상위 `parseNoteText()`에서 `unknown_modifier` invalid로 처리하도록 계획했다.
이후 `1.8-step1-note-modifier-parser.md` 단계에서 이 지점에 `@g`, `@t`, `@p`, `@m` 순서의 하위 parser를 연결했다.

### 3.3. escape 처리를 먼저 넣은 이유

defaultText와 mute 표시 문자열은 예약문자를 escape해서 표현할 수 있어야 한다.
따라서 modifier 구현 전이라도 escape 복원 유틸을 먼저 두었다.

현재 허용 escape 문자는 명세 기준과 같다.

```text
\  /  @  |  (  )  -  ~
```

unescaped 예약문자는 일반 note defaultText에 직접 들어올 수 없으므로 `unexpected_reserved_char`로 처리한다.

## 4. 이번 구현에서 사용한 주요 TypeScript 문법

### 4.1. 타입 가드

`isHoldKind()`는 문자열 후보가 `HoldKind`인지 확인하는 타입 가드이다.

```ts
function isHoldKind(value: string | undefined): value is HoldKind
```

`rawText[index]`는 TypeScript 입장에서 일반 `string | undefined`이다.
따라서 `value === "-" || value === "~"` 조건을 함수로 분리해 통과 후 `HoldKind`로 좁힌다.

### 4.2. Set

예약문자와 escape 허용 문자는 `Set`으로 관리한다.

```ts
const RESERVED_NOTE_CHARS = new Set(["/", "@", "|", "(", ")", "-", "~"]);
const ESCAPABLE_CHARS = new Set(["\\", "/", "@", "|", "(", ")", "-", "~"]);
```

문자 스캔 중 `has()`로 현재 문자가 예약문자인지, escape 허용 대상인지 빠르게 확인한다.

### 4.3. placeholder invalid

아직 구현하지 않은 큰 문법 단위는 placeholder invalid를 반환한다.
이는 조용히 note로 잘못 해석하는 것보다 안전하다.

```ts
if (isPletHeadCandidate(rawText)) {
  return parsePletHeadPlaceholder(rawText);
}
```

이 방식은 후속 구현 전까지 미지원 문법이 명확히 드러나게 한다.
