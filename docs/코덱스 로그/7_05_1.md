현재 작업 현황을 이렇게 이해했습니다.

`회귀 코드`의 활성 구현은 [regression-code](./regression-code)이고, 레거시 `이세계 코드`는 참고용 아카이브입니다. `regression-code-extend/`는 3차 확장 실험용, `regression-code-test-publish/`는 GitHub Pages 사용자 테스트 배포용 staging copy라서 기본 작업 대상은 아닙니다.

2차 구현은 안정 단계에 들어갔고, 현재 중심 작업은 추가 기능 구현보다 **2026-06-19 이후 2차 구현 보고서 작성 준비**입니다. [docs/HARNESS.md](./docs/HARNESS.md)에 따르면 주요 구현 흐름은 레이아웃 편집 안정화, 트랙 레이어, YouTube sync, partial rebuild와 viewport bounded rendering 성능 개선, range selection/undo/redo, Practice mode, Examples/Supabase, Manual, GitHub Pages 배포까지 이어졌습니다.

보고서 준비 자료는 [docs/report-prep](./docs/report-prep)에 이미 생성되어 있습니다. 특히 다음 파일이 출발점입니다.

- [README.md](./docs/report-prep/README.md): 추출 자료와 생성 스크립트 설명
- [daily-timeline.md](./docs/report-prep/session-summaries/daily-timeline.md): 날짜별 작업/커밋 연표
- [second-extension-report-outline.md](./docs/report-prep/second-extension-report-outline.md): 2차 구현 보고서 구조 초안

보고서의 핵심 서술 축은 “레거시 Google Sheets + Apps Script 구현을 독립 웹 앱으로 전환한 뒤, 2차 구현에서 실사용 가능한 편집, 재생, 연습, 배포 환경을 안정화했다”는 방향입니다. 특히 긴 악보 성능 개선은 단순 최적화가 아니라 partial rebuild, 렌더 레이어 분리, viewport bounded rendering, RAF batching 같은 구조 개선으로 설명하는 것이 맞아 보입니다.

현재 워크트리는 깨끗하지 않습니다. `docs/HARNESS.md`, 편집 관련 TypeScript 파일 일부, `docs/report-prep/`, `tools/`, 새 구현 메모가 커밋 전 상태입니다. 따라서 다음 작업을 이어갈 때는 이 변경들이 사용자가 남긴 현재 작업물이라고 보고 보존하면서 진행해야 합니다.

네, 코드 수정 없이 같은 커밋으로 다시 시도하려면 GitHub Actions 화면의 **Re-run jobs** 버튼을 쓰면 됩니다.

권장 순서:
1. 실패한 workflow run으로 들어갑니다.
2. 오른쪽 위 **Re-run jobs** 클릭
3. 전체 실패 원인을 모르겠으면 **Re-run all jobs**
4. 특정 job만 네트워크/캐시 문제로 실패한 게 확실하면 **Re-run failed jobs**

다만 먼저 실패 로그에서 원인이 `npm install`, `npm run build`, Pages upload/deploy 중 어디인지 한 번만 확인하는 게 좋습니다. 로컬에서는 배포 repo 기준 `npm run typecheck`와 `npm run build`가 통과했기 때문에, 일시적인 Pages/Actions 문제라면 re-run으로 해결될 가능성이 큽니다.

**보고서 구조**
1. 작업 개요
6월 19일 이후 작업은 1학기 최종 구현물을 “기능이 있는 프로토타입”에서 “사용자 테스트 가능한 안정판”으로 끌어올리는 과정이었다고 잡으면 됩니다.
여기서는 레거시 복원이 아니라, 편집성, 성능, 재생 안정성, 연습 기능, 배포 준비가 핵심이었다고 설명합니다.

2. 2차 구현 요약
여기는 너무 길게 쓰지 않고 기능 목록과 결과 중심으로 씁니다.
각 항목은 5~8줄 정도로 “무엇을 추가했고 현재 어떤 상태인지”만 정리하면 됩니다.

3. 주요 구현 내용
본문의 핵심입니다. 각 절을 다음 템플릿으로 통일하면 보고서 밀도가 좋아집니다.

```text
문제
- 기존 상태에서 무엇이 불편하거나 불안정했는가

설계 판단
- 어떤 구조로 해결하기로 했는가
- 왜 단순한 임시 처리 대신 이 방식을 택했는가

구현 방식
- 관련 모듈과 데이터 흐름
- UI / 상태 / 렌더러 / 오디오 / 저장소 중 어떤 경계를 건드렸는가

검증과 보정
- 어떤 테스트나 실사용 확인을 했는가
- 발견된 문제와 수정 방향
```

4. 구현상 난점
3장에서 기능별로 설명한 내용을 다시 반복하지 말고, “기술적으로 어려웠던 축”만 묶는 장으로 쓰는 게 좋습니다. 예를 들어 성능 문제는 2.4/3.x에 이미 나오겠지만, 4장에서는 “왜 전체 rebuild가 한계였고, 어떤 조건에서 병목이 드러났는가”를 더 분석적으로 씁니다.

5. 현재 완성도와 한계
여기는 결론에 가깝게 씁니다. “목표 달성”을 말하되, 완전한 제품 완성처럼 과장하지 않고 실사용 테스트, 저사양 환경, 마이크 다양성, 예제 악보 다양성 검증이 남았다고 정리하면 균형이 좋습니다.

**특히 강조하면 좋은 포인트**
