**현재 작업 상태**

`회귀 코드`는 `regression-code/`를 중심으로 구현 중이며, 지금 초점은 사용자 커스텀 레이아웃 수정 기능입니다. 이 기능은 단순 화면 설정이 아니라 현재 악보의 기본 레이아웃인 `ScoreFile.layout.rowDefinitions`와 악기 정보인 `ScoreFile.instData`를 수정하고, 적용 후 전체 파이프라인을 다시 빌드하는 기능입니다.

현재 완료된 범위는 “레이아웃 편집 UI shell” 단계입니다.

- [docs/2.4-layout-edit-ui-spec.md](</D:/오타마톤악보생성기/docs/2.4-layout-edit-ui-spec.md>)에 레이아웃 편집 명세가 추가됨
- `Modify` 버튼으로 `Layout` 다이얼로그를 여는 UI가 구현됨
- preset / instrument / string / row / add row / save-load 버튼 영역이 HTML/CSS로 구성됨
- 현재 `ScoreFile.instData`를 읽어 악기 family, preset 이름, string 목록 등을 다이얼로그에 표시함
- `0..127` MIDI 계이름 드롭다운 helper가 연결됨
- `src/app/layout/layout_types.ts`에 draft, preset, apply 결과 타입이 정의됨
- `Apply`, `Add Row`, `Local Save/Load`, `File Save/Load`는 아직 placeholder 안내만 표시함

실제 코드상으로는 `regression-code/src/app/layout/` 아래에 현재 [layout_types.ts](</D:/오타마톤악보생성기/regression-code/src/app/layout/layout_types.ts>)만 존재합니다. 다이얼로그 DOM 배선은 아직 별도 `layout_dialog_binding.ts`가 아니라 [app_view_binding.ts](</D:/오타마톤악보생성기/regression-code/src/app/app_view_binding.ts>) 쪽에 들어가 있습니다.

**아직 구현되지 않은 핵심 범위**

- 실제 `LayoutDraftBundle` 생성
- 선택된 `stringId`별 note/gap row 목록 렌더링
- 기존 note/gap row의 height 수정
- note/gap row 추가 및 삭제 draft mutation
- gap row의 `fromMidi/toMidi` 자동 계산
- draft 단계 검증
- `Apply` 시 `ScoreFile.instData`와 `layout.rowDefinitions` 반영
- 새 layout과 호환되지 않는 track cell 삭제 계산
- 삭제될 cell이 있을 때 사용자 확인 dialog

- 레이아웃 프리셋 JSON 파일 저장/불러오기

**앞으로의 권장 작업 순서**

1. `layout_draft.ts` 추가
   현재 score에서 `LayoutDraftBundle`을 만들고, 전역 행을 제외한 note/gap row만 draft로 복사합니다.

2. row 목록 표시 연결
   선택된 `stringId` 기준으로 note/gap row를 다이얼로그 row list에 표시하고, height 입력을 draft에 반영합니다.

3. draft row 조작 추가
   note row 추가, gap row 추가, row 삭제를 draft 안에서만 처리합니다. 이 단계에서는 아직 실제 score에 반영하지 않는 것이 안전합니다.

4. draft 검증 추가
   note MIDI 중복, row height 범위, string 범위, gap 삽입 가능 여부 등을 먼저 막습니다.

5. `layout_apply.ts` 추가
   draft의 `instData`와 note/gap rows를 현재 `ScoreFile` clone에 덮어쓰고, 기존 global rows는 유지합니다.

6. cell 삭제 확인 연결
   새 layout에 없는 note rowId를 참조하는 track cell을 삭제 대상으로 계산하고, 실제 적용 전에 사용자 확인을 받습니다.

7. full rebuild 연결
   apply 성공 후 기존 edit 흐름처럼 runtime document, parse result, analysis result, canvas render input을 다시 만들고 playback 상태를 reset합니다.

8. preset 저장/불러오기
   먼저 localStorage, 그 다음 JSON file save/load를 붙이는 순서가 적절합니다.

현재 git 상태상 변경된 문서는 `docs/2.4-layout-edit-ui-spec.md`, `docs/HARNESS.md`이고, 새 memo `docs/implementation-memo/1.25-step5-layout-editor-ui-shell.md`가 아직 untracked입니다. 즉 다음 작업은 “UI shell 이후 실제 draft state 연결”로 이어가면 됩니다.

> 사용자가 모든 행의 px를 직접 입력해야 한다는 것이 UX 면에서는 큰 흠이네요...
> 레이아웃 영역에서 직접 드래그를 통해 행간 높이를 조절하는 것은 구현 난이도가 어떻게 되나요?

구현 난이도는 **중간 이상**입니다. 다만 “레이아웃 편집 다이얼로그 안의 draft row 높이만 조절”로 시작하면 충분히 현실적입니다. “실제 악보 화면의 좌측 layout label 영역에서 바로 드래그”까지 가면 난이도가 한 단계 올라갑니다.

가장 쉬운 1차안은 다음입니다.

- `Layout` 다이얼로그 row 목록에 각 row 사이 resize handle 추가
- 사용자가 handle을 위아래로 드래그하면 해당 row의 `height` draft 값을 갱신
- row list의 height input도 같이 갱신
- 아직 `Apply` 전이므로 실제 score/render에는 반영하지 않음
- 적용은 기존 계획대로 `Apply`에서 처리

이건 현재 구현에 바로 붙일 수 있고 난이도는 **낮음~중간**입니다.

조금 더 좋은 2차안은 다음입니다.

- 다이얼로그 안에 mini layout preview를 추가
- note/gap row를 실제 높이 비율로 세로 표시
- row 경계선을 드래그해서 height 조절
- 오른쪽 row list의 height input과 동기화

이건 UX가 훨씬 낫지만, preview DOM/CSS와 drag 좌표 변환이 추가되어 난이도는 **중간**입니다.

가장 직접적인 3차안은 말씀하신 것처럼 실제 좌측 레이아웃 영역에서 드래그하는 방식입니다.

- `layoutStage` 또는 label canvas 위 pointer 좌표를 읽음
- 현재 `CanvasScoreLayout.rows`에서 어느 row 경계 근처인지 hit-test
- drag delta를 row height에 반영
- draft layout을 갱신하고 preview/render를 다시 그림
- edit mode, playback scroll, zoom, canvas 좌표계와 충돌하지 않게 모드 분기 필요

- gap row 개별 높이: 특정 음역 사이 간격 강조, 옥타브/포지션 구분, 시각적 여백

개별 note row height를 허용하면 생기는 문제도 있습니다.

- 같은 음표 박스 높이와 row 높이 관계가 불균형해짐
- pitch 간격이 “음정 간 거리”가 아니라 “각 행 두께”에 따라 뒤틀림
- microPitch나 gliss 표시가 시각적으로 해석하기 어려워짐
- 사용자가 어떤 note row를 얼마나 키웠는지 관리하기 어려움
- layout preset 비교/복구가 복잡해짐

그래서 추천 정책은 이렇습니다.

1. **note row height는 string별 또는 layout 전체 공통값으로 통일**
2. **gap row height만 개별 조절 허용**
3. preview drag에서 note row를 드래그하면 해당 string의 모든 note row height를 같이 변경
4. gap row를 드래그하면 해당 gap row만 변경
5. row list에서도 note row height input은 공통 height 입력처럼 동작하거나, note row에서는 개별 input을 숨김/비활성화

구현 난이도는 지금 상태에서 크지 않습니다. `updateLayoutDraftRowHeight()`를 둘로 나누거나 옵션을 추가하면 됩니다.

- `row.type === "note"`이면 같은 `stringId`의 모든 note row height를 같은 값으로 갱신
- `row.type === "gap"`이면 현재처럼 해당 row만 갱신

저장 포맷은 그대로 둬도 됩니다. `ScoreFile.layout.rowDefinitions`에는 여전히 각 note row마다 `height`가 들어가지만, UI가 통일된 값을 유지하도록 보장하는 방식입니다. 즉 저장 구조 변경 없이 UX 정책만 바꾸면 됩니다.

제안하는 1차 수정은 “string별 note row 높이 통일”입니다. 전체 악기 공통보다 string별이 나중에 다현 악기 확장에 유리하고, 현재 오타마톤처럼 string이 하나면 자연스럽게 전체 note row 통일이 됩니다.

> note row의 높이는 손으로 현을 짚는 점을 의미하므로 악기나 현에 따라 달라지지는 않을 것 같습니다. 손가락의 굵기는 일정하니까요. note row는 전체 공통값으로 통일하는 것이 가장 가볍겠습니다. Common Note Height 입력창을 어디에 두는 것이 무난할지만 정하면 되겠습니다.

그 해석이면 **전체 공통 note row height**가 맞습니다. 악기/현이 아니라 “사용자가 누르는 손가락 두께”에 해당하므로 string별로 나눌 이유도 약합니다.

`Common Note Height` 입력 위치는 저는 **Rows 섹션 상단, String 선택 옆**이 가장 무난하다고 봅니다.

현재 구조가 대략:

```text
Rows
[String select]
[row list]
```

여기에 이렇게 붙이면 자연스럽습니다.

```text

[String select] [Common Note Height]
[row list]
```

- Instrument 영역에 넣기에는 악기 속성이라기보다 layout 편집 속성임
- Preview 쪽에 넣기에는 조작과 표시가 섞여서 역할이 애매함
- Apply/Save 근처에 넣기에는 너무 자주 쓰는 값임

정책은 이렇게 가면 됩니다.

- Add Row에서 `gap`을 추가할 때만 Add Row의 height 입력을 사용

새 정책은 이렇게 잡는 게 좋겠습니다.

- gap row는 선택 string 안 어디든 추가 가능
- 단, gap row가 연속으로 두 개 올 수 없음

현재 방식의 문제:
- `fromMidi/toMidi`를 인접 note 두 개에서 계산함
- 끝부분 gap은 만들기 어려움
- 같은 두 note 사이 gap 중복을 막기 위한 제약이 있음

새 방식에서 추천하는 내부 규칙:
- gap row의 `rowId`는 위치 기반으로 생성
- 예: `{stringId}-gap-{index}` 또는 `{stringId}-gap-{serial}`
- `fromMidi/toMidi`는 가장 가까운 note row 기준으로 채움

  - 한쪽 note만 있으면 `fromMidi = midi`, `toMidi = midi`

- row 삽입 후 rowId 충돌이 있으면 다음 번호를 증가시키는 간단한 while만 사용

- Apply 전 최종 저장 시에도 그대로 사용 가능

“gap 연속 금지”는 다음 위치 기준으로 보면 됩니다.

- 삽입될 위치의 바로 위 row가 gap이면 금지
- 삽입될 위치의 바로 아래 row가 gap이면 금지

이렇게 하면 하단/상단 gap은 허용하면서도 “gap gap”은 막습니다.

UX상 의미도 충분합니다.
- 사용자는 top/bottom 여백을 만들 수 있음
- gap은 “두 note 사이 간격”이 아니라 “레이아웃 여백 row”로 더 일반화됨
- preview drag UX와도 잘 맞음
- row list에서 from/to를 안 보여주고 있으므로 사용자 혼란도 적음

다만 문서 관점에서는 `docs/2.4-layout-edit-ui-spec.md`의 기존 gap 정책과 충돌합니다. 구현에 들어간다면 해당 문서의 gap row 규칙도 나중에 업데이트하는 게 좋습니다. 지금 바로 구현해도 되지만, 이건 명세와 저장 rowId 규칙을 건드리는 변경이라 작업 후 문서 반영까지 같이 하는 편이 안전합니다.

> rowId를 바꾸는건 다소 무거운 결정이네요. 기존 rowId를 유지하되 빈 midi 변수 쪽에 fromMidi에서 + 또는 - 1 숫자를 자동으로 채우는 방식은 안될까요?

가능합니다. 그리고 지금 단계에서는 그쪽이 더 가볍습니다.

즉 `rowId` 규칙은 유지합니다.

```text
{stringId}-gap-{fromMidi}-{toMidi}
```

대신 끝부분 gap에서 한쪽 note만 있을 때 `fromMidi/toMidi`를 이렇게 채우는 방식입니다.

- 두 note 사이 gap:
  - 기존처럼 `fromMidi = 낮은 midi`
  - `toMidi = 높은 midi`
- 맨 위 gap:
  - 아래쪽 note가 `midi`
  - `fromMidi = midi`
  - `toMidi = midi + 1`
- 맨 아래 gap:
  - 위쪽 note가 `midi`
  - `fromMidi = midi - 1`
  - `toMidi = midi`
- note가 전혀 없으면 gap 추가 불가
- MIDI 범위는 `0..127` 안으로 clamp 또는 실패 처리

저는 clamp보다 **실패 처리**가 더 낫다고 봅니다. 예를 들어 `midi = 127` 위에 gap을 만들려고 하면 `toMidi = 128`이 되어 저장값 범위를 벗어납니다. 이 경우 “Cannot add gap beyond MIDI range.”처럼 막는 편이 rowId 의미를 더 깨끗하게 유지합니다.

이 방식의 장점:
- 저장 타입 변경 없음
- rowId 규칙 변경 없음
- 기존 validator와 renderer 영향이 작음
- gap row의 `fromMidi/toMidi`가 여전히 대략적인 인접 음역 범위를 표현함
- 끝부분 gap도 지원 가능
- “연속 gap 금지”만 추가하면 suffix/중복 문제도 거의 줄어듦

주의할 점:
- 맨 아래 gap이 이미 있고 그 아래에 또 gap을 추가하려는 경우는 연속 gap 금지로 막아야 합니다.
- 맨 위 gap도 마찬가지입니다.
- 같은 note 기준으로 위쪽 gap과 아래쪽 gap은 서로 다른 id가 됩니다.
  - 위쪽: `midi-midi+1`
  - 아래쪽: `midi-1-midi`
- 두 note 사이 gap은 기존과 동일합니다.

예시:
- `C4 = 60` 아래에 하단 gap 추가:
  - `fromMidi = 59`
  - `toMidi = 60`
  - `rowId = s1-gap-59-60`
- `C4 = 60` 위에 상단 gap 추가:
  - `fromMidi = 60`
  - `toMidi = 61`
  - `rowId = s1-gap-60-61`

이 정책이면 구현 비용은 작습니다. `addGapRow()`의 “두 note가 모두 있어야 함” 조건을 완화하고, 삽입 위치의 바로 위/아래 row가 gap인지 검사하는 연속 gap 금지만 추가하면 됩니다.

문서 충돌은 여전히 있지만 rowId 규칙 변경보다는 훨씬 작습니다. `2.4`의 gap 삽입 가능 위치만 나중에 수정하면 됩니다.

다만 rowId를 바꾸는 것이 여전히 무거운 이유는 있습니다.

- 기존 score cell들이 rowId를 참조합니다.
- layout apply 시 rowId가 바뀌면 기존 cell 참조와 호환성 문제가 생깁니다.
- 테스트 fixture와 문서 예시가 현재 형식을 전제로 합니다.
- 이벤트 id나 정렬 fallback에 rowId 문자열이 들어가므로, rowId 변경은 저장 데이터뿐 아니라 파생 id 안정성에도 영향을 줄 수 있습니다.

위험도는 이렇게 보면 됩니다.

- rowId 형식 변경: 중간~높음
- 기존 rowId 형식 유지 + 끝 gap 허용: 낮음~중간
- 두 note 사이 gap만 유지: 낮음

> 그렇게 합시다. 마지막으로, 0 아래의 갭 행과 127 위의 갭 행도 추가 자체는 가능해야 한다고 봅니다. rowId 생성 자체에 제약이 걸리지는 않았죠?

네, **rowId 생성 자체에는 `0..127` 제약이 걸려 있지 않습니다.** `rowId`는 그냥 문자열이라 `s1-gap--1-0`, `s1-gap-127-128` 같은 값도 만들 수는 있습니다.

- `GapRowDefinition.fromMidi`
- `GapRowDefinition.toMidi`

이 두 필드는 문서상 `0..127` 범위로 되어 있고, validator가 해당 범위를 강제하고 있다면 `-1`이나 `128`은 저장 검증에서 막힐 수 있습니다. 그래서 구현 전에 `score_validate.ts`의 gap row MIDI 검증을 확인해야 합니다.

정책적으로는 선택지가 둘입니다.

1. **끝 gap 허용을 위해 gap row의 `fromMidi/toMidi`만 `-1..128`까지 허용**

2. **저장값은 `0..127`로 유지하고 끝 gap은 `0-0`, `127-127` 같은 값으로 표현**

저는 1번이 더 낫다고 봅니다. gap row의 `fromMidi/toMidi`는 실제 연주 음정이 아니라 “인접 음역 경계 표현”에 가깝기 때문에, 경계값으로 `-1`과 `128`을 허용하는 것이 오히려 자연스럽습니다.

- note row `midi`: 계속 `0..127`

  - 127 위: `s1-gap-127-128`
- track cell은 gap row를 참조하지 못하므로 연주/분석에는 직접 영향 없음

`Local Save/Load`
- 개인 작업 환경용
- 같은 브라우저에서 자주 쓰는 악기별 layout preset을 빠르게 재사용

`File Save/Load`
- 백업, 공유, 다른 브라우저/기기 이동용

> ScoreFile 내부의 createAt과 updatedAt은 악보의 생성 및 수정 시각인데 UserLayoutPresetData를 통해 덧씌울 의미가 있나요?

아니요. `UserLayoutPresetData.createdAt / updatedAt`은 **ScoreFile의 `musicData.createdAt / updatedAt`을 덮어쓸 의미가 없습니다.**

둘은 의미가 다릅니다.

- `ScoreFile.musicData.createdAt`
  - 악보 자체의 생성 시각
- `ScoreFile.musicData.updatedAt`
  - 악보 내용의 마지막 수정 시각
- `UserLayoutPresetData.createdAt`
  - layout preset 자체가 만들어진 시각
- `UserLayoutPresetData.updatedAt`
  - layout preset 파일/local 저장본이 마지막으로 갱신된 시각

따라서 preset load/apply 시에는:

- `instData` 적용 가능
- note/gap `rowDefinitions` 적용 가능
- 삭제될 track cell 확인 후 정리 가능
- `ScoreFile.musicData.createdAt`은 절대 변경하지 않음
- `ScoreFile.musicData.updatedAt`도 layout apply 순간에 자동 변경하지 않는 편이 현재 edit apply 정책과 일관적임
- 실제 JSON 저장 시점에 기존 `touchScoreTimestampsForSave()` 경계에서 `updatedAt`을 갱신

즉 `UserLayoutPresetData.createdAt / updatedAt`은 preset 관리용 metadata일 뿐이고, ScoreFile metadata로 전파하지 않는 것이 맞습니다.
문서에 이 점을 더 명시해두는 게 좋겠습니다.

> 의미가 다르고 덮어쓰지도 않는군요. 그러면 괜찮습니다. 저장/불러오기 기능을 연결합시다.

> 레이아웃 프리셋 저장 개수에 제한을 두는 것이 좋겠습니다. 로컬스토리지를 프리셋이 과하게 차지하면 정작 중요한 악보 본편의 로컬 저장 가능 용량이 줄어듭니다. 5개 정도로 제한하면 적절할까요?

5개 제한은 적절합니다. 지금 용도라면 “자주 쓰는 개인 프리셋”이지 프리셋 라이브러리가 아니므로, 악기 프리셋별 local preset을 **최대 5개**로 제한하는 정책이 깔끔합니다.

추천 정책은 이렇습니다.

- 기준: `instrumentPresetId`별 최대 5개
- 기존 프리셋 덮어쓰기: 개수 증가 없음
- 새 프리셋 저장 시 이미 5개면 저장 차단
- 차단 메시지: “Local layout presets are limited to 5 per instrument. Save to file or overwrite an existing preset.”
- File Save는 제한 없음
- localStorage index가 손상됐거나 누락된 data가 있으면 가능한 항목만 표시하고, 저장 시 정상 index로 재정리

자동으로 가장 오래된 것을 밀어내는 방식은 비추천입니다. 레이아웃 프리셋은 사용자가 직접 만든 설정이라 조용히 삭제되면 손실 체감이 큽니다. 5개를 넘기려면 파일 저장을 쓰거나 기존 항목을 선택해서 덮어쓰게 하는 편이 안전합니다.

> 그렇습니다. 예를 들어 포켓몬의 기술 습득, RPG의 세이브 시스템처럼 슬롯을 미리 만들어두고 저장하게 하면 편할 것 같네요. 5개도 많고 3개만 되어도 괜찮을 것 같습니다.

그 방식이면 **3슬롯 고정형**이 더 맞습니다.

추천 정책:

- `instrumentPresetId`별 Local Preset Slot 1~3 고정
- 사용자는 슬롯을 선택한 뒤 `Local Save`로 덮어쓰기
- 빈 슬롯이면 새 저장
- 차 있는 슬롯이면 확인 후 덮어쓰기
- `Local Load`는 선택 슬롯을 draft로 불러오기
- `New Preset` 버튼은 local 저장 관점에서는 필요성이 줄어듦
- File Save/Load는 슬롯 제한 없음

이 구조의 장점은 명확합니다.

- localStorage 사용량 상한이 예측 가능함
- “이름 중복” 같은 규칙이 거의 필요 없어짐
- 사용자가 어디에 저장했는지 이해하기 쉬움
- 삭제 기능 없이도 빈 슬롯/덮어쓰기 중심으로 운영 가능
- UI도 `Preset Slot: [1 Empty] [2 My Tuning] [3 Compact]`처럼 단순해짐

구현 정책은 이렇게 잡으면 됩니다.

- key:
  - `layout-slot:{instrumentPresetId}:1`
  - `layout-slot:{instrumentPresetId}:2`
  - `layout-slot:{instrumentPresetId}:3`
- index는 없어도 됩니다. 팝업을 열 때 1~3번 key를 직접 읽어서 슬롯 상태를 만들면 됩니다.
- `layoutPresetId`는 `slot-1`, `slot-2`, `slot-3`처럼 고정해도 됩니다.
- 저장 시 `layoutPresetDisplayName`은 현재 입력값을 저장하고, select label에 표시합니다.
- load 시 해당 슬롯의 preset data를 draft로 복원합니다.
