# 1.1. 회귀 코드 계획서 초안

> DOCX 원본을 바탕으로 재정리한 Markdown 참조본이다.
> 본 문서는 초기 계획과 문제의식을 빠르게 파악하기 위한 상위 계획 문서이며, 세부 문법과 데이터 구조의 최신 구현 기준은 후속 active 문서인 `1.3`, `1.5`, `1.6`, `1.7`, `1.8`을 우선한다.

## 문서 정보

- 프로젝트명: 현악기용 피아노롤 에디터
- 소속: 홍익대학교 컴퓨터공학과
- 작성자: 이예서
- 제출일자: 2026년 3월 6일

## 프로그램 개요

- 본 프로그램은 초심자 또는 취미 연주자를 위한 현악기 입문용 악보 프로그램을 목표로 한다.
- 특히 해금, 바이올린류, 프렛리스 기타처럼 지판에 프렛이 없는 프렛리스 현악기의 연주 지원을 중점 목표로 둔다.
- 악보의 형태는 오선지가 아니라 피아노롤 기반이다.
- 단, 일반적인 피아노롤과 달리 현악기의 실제 물리적 음 간격을 반영한다.
- 즉 세로축의 음정 간격은 균일 격자가 아니라, 저음에서 넓고 고음으로 갈수록 좁아지는 현악기의 실제 음 간격을 시각적으로 반영하는 것을 목표로 한다.

## 프로젝트 목표

- 프렛리스 현악기 초심자가 음정 위치를 직관적으로 파악하며 연습할 수 있도록 한다.
- 현악기의 물리적 음 간격을 반영한 피아노롤 악보를 제공한다.
- 이를 시각화, 재생, 편집, 저장할 수 있는 웹 기반 편집기를 구현한다.
- 부차적으로 직관적인 악보 작성 기능을 통해 사용자의 음악 이해도를 높이는 것도 기대 효과로 본다.

### 해결하려는 문제

- 프렛리스 현악기는 반음 간격이 일정하지 않다.
- 저음일수록 반음 사이 간격이 넓고, 고음으로 갈수록 좁아지는 물리적 특성이 있다.
- 음 위치가 시각적으로 표시되지 않아 초심자가 정확한 위치를 익히기 어렵다.
- 바이올린 초심자가 지판에 스티커를 붙이는 보조 방식은 존재하지만, 악보와 지판 표시를 동시에 보기는 어렵다.
- 미세한 위치 차이가 음정 차이로 이어지므로 정확한 위치 인지가 매우 중요하다.
- 결국 많은 연주자가 기교 이전에 정확한 위치를 누르는 것 자체를 익히기 위해 장시간을 소모한다.

### 기대 효과

- 시각적 안내를 통해 프렛리스 현악기의 진입장벽을 낮춘다.
- 반복 연습에서 음 위치를 직관적으로 확인하게 해 숙달 시간을 줄이는 것을 기대한다.
- 연습 자체를 더 재미있고 접근하기 쉬운 경험으로 바꾸는 것을 목표로 한다.

## 프로그램 목표 기능

- 핵심 축은 `가변형 양식`, `시각화`, `재생`, `악보 작성`이다.
- 추가로 일반 소프트웨어로서 필요한 성능, 효율, 보안성을 확보해야 한다.
- 1차 목표는 단일 현의 안정적 구현이다.
- 다현 확장은 후속 단계에서 고려하되, 초기 구조는 확장을 염두에 두고 설계한다.

### 1. 악보 양식(템플릿)의 자유 변경

- 각 음정 사이 간격을 사용자가 개별적으로 조정할 수 있어야 한다.
- 현악기는 고음으로 갈수록 음 사이 간격이 좁아지므로, 그 간격 변화를 악기별로 다르게 설정할 수 있어야 한다.
- 악기별 튜닝과 음역 차이를 반영할 수 있어야 한다.
- 단일 현악기가 아니라 프렛리스 현악기 전반을 대상으로 하므로, 튜닝 차이와 악기 종류 차이에 모두 대응할 필요가 있다.
- 확장 기능으로 현 수에 맞춘 복수 악보 동시 전개를 고려한다.
- 다만 다현 동시 전개는 병렬 재생과 추가 렌더링 비용을 요구하므로 우선순위는 낮다.

### 2. 악보의 시각화

- 노트가 놓일 기본 배경 템플릿을 시각화해야 한다.
- 음정이 정확한 구간과 그렇지 않은 구간이 명확히 구분되어야 한다.
- 노트 길이, 바이브레이션, 글리산도, 트레몰로, 셋잇단, 빠르기, 셈여림 변화를 시각적으로 구별 가능해야 한다.
- 도형, 선, 기호, 색상을 사용하되 가독성을 유지해야 한다.
- 사용자는 시각화된 악보만 보고도 어떤 음을 어떤 방식으로 연주해야 하는지 판단할 수 있어야 한다.

### 3. 연주를 위한 재생 기능

- 피아노롤의 각 노트는 실제 음을 출력해야 한다.
- 출력되는 음은 단순 pitch뿐 아니라 음악적 articulation을 가능한 범위에서 반영해야 한다.
- 바이브레이션, 글리산도, 트레몰로, 셈여림, 빠르기 변화도 재생에 반영되어야 한다.
- 재생, 일시정지, 정지, 처음으로 이동, 특정 구간 이동, 루프 기능을 지원해야 한다.
- 확장 기능으로 반주 재생을 고려한다.
- 법적 문제와 비용 검토 후 유튜브 연동도 확장안으로 고려한다.
- 확장 기능으로 난이도별 멜로디 레이어를 고려한다.
- 같은 악보 안에서 초급, 중급, 고급용 멜로디 레이어를 켜고 끄는 방식도 검토 대상이다.

### 4. 악보 작성 기능(에딧 모드)

- 사용자가 직접 악보를 작성할 수 있어야 한다.
- 피아노롤 평면의 각 칸에 노트를 입력할 수 있어야 한다.
- 저장과 불러오기가 가능해야 한다.
- 복잡한 토큰 문법을 사용자가 직접 전부 외우지 않아도 되도록, 버튼 기반 조합 UI를 제공하는 방향을 목표로 한다.
- 셋잇단은 별도의 계산과 분할 처리가 필요하다.
- 따라서 셋잇단은 일반 셀과는 다른 편집 보조 UI와 내부 처리 로직이 필요하다.

### 5. 최적화

- 10여 분 길이의 악보도 감당 가능해야 한다.
- 목표는 일반적인 한 악장 수준의 길이를 안정적으로 처리하는 것이다.
- 에딧 모드에서는 전체 재계산 대신 변경 범위와 인접 영역만 부분 갱신해 비용을 줄여야 한다.

### 6. 보안

- 불러오는 파일이 악보 데이터 구조와 일치하는지 검증해야 한다.
- 과도하게 큰 입력 파일을 제한해야 한다.
- 사용자 입력 텍스트 출력 시 스크립트 삽입 같은 위험을 방어해야 한다.
- 보안은 현재 시점에서 설계가 완전히 끝난 주제가 아니므로, 구현과 학습을 병행하며 계속 보완해야 한다.

## 프로그래밍 구상

- 주 이용층을 고려할 때 웹 앱 형태가 가장 접근성이 높다고 판단한다.
- 개발 기술은 HTML, CSS, JavaScript를 상정한다.
- 직관적인 UI와 별도 설치가 필요 없는 방식이 중요하다.
- 구글의 `Chrome Music Lab`처럼 브라우저 기반의 직관적 음악 도구를 참고 방향 중 하나로 본다.
- 데스크톱 브라우저(Windows + Chrome)를 우선 지원 대상으로 둔다.
- 모바일은 필수 지원 범위에 포함하지 않는다.

## 전반적인 구조 구상

1. 부팅 및 UI 초기화
2. 악보 파일 로드
3. 각 셀의 노트 문자열 파싱
4. 파싱 결과를 분석하여 렌더/오디오 공통 데이터 생성
5. 피아노롤 배경 및 노트 렌더 데이터 생성
6. 오디오 데이터 생성
7. 런타임 루프에서 재생, 편집, 저장 요청 처리
8. 종료

### 핵심 구현 알고리즘

- 악보 테이블 셀 문자열을 규칙에 따라 파싱한다.
- 인접 셀 간 결합 규칙을 적용해 중간 산출물을 만든다.
- 그 산출물을 바탕으로 Canvas 렌더링과 Web Audio 재생 데이터를 생성한다.
- 에딧 모드에서는 변경 범위와 인접 영역만 부분 갱신하여 전체 비용을 줄인다.

### 초기 모듈 구상

- `Code.js`: 악보 로드, 캐시 관리 등의 중심 기능
- `index.html`: 하위 파일 호출
- `parts/UI`: 기본 UI 구성
- `parts/api`: 전역 변수, 공통 유틸, API
- `parts/renderBase`: 피아노롤 배경 구성
- `parts/parse`: 노트 문자열 파싱
- `parts/analyze`: 파싱된 데이터 분석
- `parts/analyze/renderNote`: 노트 시각화
- `parts/analyze/audio`: 노트 오디오 데이터 할당
- `parts/play`: 재생 관련 기능
- `parts/edit`: 편집, Undo/Redo, 영향 범위 재계산

위 구상은 초기 단계의 모듈 구상이며, 실제 구현에서는 TypeScript 기반 구조로 재편될 수 있다.

## 서버 및 배포 구상

- 웹앱 형태로 동작하며, 정적 리소스와 악보 데이터를 안정적으로 전달받을 수 있어야 한다.
- 기본 배포는 정적 호스팅 기반이며 GitHub Pages를 고려한다.
- 렌더링, 오디오, 재생 등 핵심 연산은 클라이언트에서 수행한다.
- 악보 데이터는 JSON 포맷을 기본으로 한다.
- 불러오기와 저장은 클라이언트 파일 입출력 방식으로 제공한다.
- 구글 스프레드시트 기반 입력을 JSON으로 변환하는 방식은 확장안으로 검토한다.
- 다만 스프레드시트 접근 권한, 인증, 수정 방식, 구현 복잡도는 별도 검토가 필요하다.
- 추후 필요시 저장/공유/검증용 별도 API 서버를 둘 수 있다.

### 반주 재생 확장 방향

- 반주 재생은 후순위 확장 기능으로 둔다.
- 초기 단계의 목표는 유튜브 직접 연동보다 일반 오디오 동기화 기능을 먼저 두는 것이다.
- 즉 1단계는 로컬 오디오 파일 또는 일반 URL 기반 오디오와의 동기화, 2단계는 유튜브 연동으로 나눈다.
- 이세계 코드에 유사 기능이 이미 존재하므로 알고리즘 흐름과 UX는 참고 가능하지만, 현재 웹앱 구조에 맞게 재설계가 필요하다.

## 악보의 형태

- 악보는 직관적이고 수정하기 쉬운 테이블 형태를 채택한다.
- 가로는 시간축이다.
- 세로 최상단에는 BPM, 박자, 스텝, 볼륨 행을 둔다.
- 하단에는 음역과 그 사이 간격을 표시하는 행을 둔다.
- 음 정보는 셀 내부의 텍스트로 기록한다.
- 실제 해석 규칙은 이후 노트 문자열 명세에서 더 자세히 다룬다.

### 스텝의 의미

- `step`은 박자를 다시 쪼갠 최소 시간 단위, 즉 1셀의 시간 길이를 결정하는 값이다.
- 이 초기 계획서에서는 `1셀 시간 = (60 / BPM) / step × 박 기준 보정`과 같은 개념으로 설명하고 있다.
- 이후 active 문서에서는 `stepsPerBeat`라는 용어와 데이터 구조로 정교화되었다.

## 설정 저장 원칙

- 악보 해석에 직접 영향을 주는 레이아웃 정보는 악보 파일에도 저장한다.
- 사용자별 편의 설정도 별도로 저장한다.
- 사용자 설정 중 줌 배율과 마지막 스크롤 위치는 저장 대상에서 제외한다.
- 글자 크기는 사용자 설정으로 분리한다.
- 레이아웃 커스텀 설정은 악기 프리셋별로 하나씩 저장하면 충분한 것으로 본다.
- 즉 동일 프리셋에 대해 "악보 기본 레이아웃"과 "사용자 로컬 커스텀 레이아웃"이 함께 존재할 수 있다.

### 레이아웃 충돌 처리 방침

- 악보 파일의 기본 레이아웃과 사용자 로컬 커스텀 레이아웃이 충돌하면, 먼저 사용자에게 커스텀 설정을 적용할지 확인한다.
- 사용자가 커스텀 설정 적용을 선택하면 악보 형태를 그 설정에 맞게 변형한다.
- 단순히 행 높이, 셀 너비, 열 너비 같은 시각적 차이만 있는 경우에는 렌더링 레벨 조정으로 처리할 수 있다.
- `rowId` 종류, 개수, 필수 전역 행 존재 여부처럼 구조 수준의 충돌은 별도의 보정 알고리즘 대상이다.
- 1차 구현에서는 이런 구조 충돌이 발생하면 import를 실패로 처리한다.
- 후속 단계에서는 보정 후 이식 알고리즘을 추가한다.
- 후속 보정 단계에서는 삭제 또는 변형 대상에 대한 경고 로그를 생성하고 사용자 확인을 거친 뒤 적용한다.

## 범위와 비범위

### 초기 구현 범위

- 단일 현 악기 우선
- 데스크톱 브라우저 우선
- 기본 오실레이터 기반 재생
- 편집, 저장, 불러오기, 최소 재생, 기본 시각화
- 모바일에서는 파일 불러오기, 재생, 일시정지, 정지, 스크롤 기반 열람 정도까지를 목표 범위로 둔다.

### 후순위 또는 확장 범위

- 다현 동시 전개
- 반주 재생
- 유튜브 연동
- 난이도별 레이어 멜로디
- 별도 서버 기반 저장/공유
- 모바일 에딧 모드
